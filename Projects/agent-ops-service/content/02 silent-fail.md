# 02호 — 티스토리 발행 완성본

### 제목
자는 동안 자동 커밋이 사흘간 멈췄는데, 에러는 단 한 줄도 없었습니다

### 본문

지난 편에서 약속했던 그 이야기입니다. "자는 동안 커밋이 조용히 멈췄는데 사흘을 몰랐던 이야기."

저는 작업 내용을 자동으로 커밋·푸시하는 훅을 깔아두고 삽니다. 세션이 끝날 때마다 vault의 변경분을 알아서 `git commit` 하고, 세션 종료 시 원격에 `push`까지 하는 훅이에요. 혼자 일하니까 "커밋하는 걸 깜빡한다"는 사람의 실수를 아예 기계에 맡겨버린 거죠. 잘 돌아갔습니다. 한동안은.

그러던 어느 날, 정리할 게 있어서 vault의 git 히스토리를 리셋했습니다. 그리고 사흘 뒤, 뭔가 이상하다는 걸 깨달았어요. **사흘치 작업이 단 한 줄도 커밋돼 있지 않았습니다.**

여기서 진짜 무서운 부분은 따로 있어요. **그 사흘 동안 에러는 단 한 번도 안 떴습니다.** 경고도 없었어요. 훅은 매 세션 끝날 때마다 멀쩡히 실행됐고, 매번 성공적으로 `exit 0`을 반환했습니다. 로그상으로는 완벽하게 동작하는 훅이었어요. 다만 **실제로는 아무 일도 안 하고 있었을 뿐**입니다.

---

원인을 찾는 데 한참 걸렸습니다. 에러 메시지가 가리켜주는 게 없으니까요. 한 줄씩 따라가다 마침내 범인을 잡았습니다.

제 훅에는 "안전장치"가 하나 있었어요. **"여기가 정말 내가 커밋해도 되는 vault가 맞나?"를 판정하는 로직**입니다. Claude Code 훅은 가끔 엉뚱한 디렉토리(예: 터미널의 현재 위치가 다른 git 저장소)에서 실행될 수 있는데, 그 상태로 `git add -A`를 해버리면 **남의 저장소에 내 vault 변경을 오염시킬** 수 있거든요. 그래서 "이 디렉토리가 내 vault인지" 확인하는 마커를 뒀습니다.

그런데 그 마커가 **딱 하나, 루트의 `CLAUDE.md` 파일이 존재하는지**였어요.

```python
if not os.path.isfile(os.path.join(root, "CLAUDE.md")):
    sys.exit(0)   # "여긴 내 vault가 아니다" — 조용히 종료
```

히스토리를 리셋하면서 그 `CLAUDE.md` 파일이 사라진 겁니다. 그러자 훅은 매번 이렇게 판단했어요. **"마커가 없네? 여긴 내 vault가 아니구나. 건드리면 안 돼."** 그리고 점잖게 `exit 0`으로 종료. 오염을 막으려고 넣은 안전장치가, 정작 자기 집을 못 알아보고 **영구적인 no-op**에 빠진 거죠.

가장 뼈아픈 건, 이게 **설계대로 정확히 동작한 것**이라는 점입니다. 코드는 시킨 대로 했어요. "마커 없으면 조용히 종료." 버그는 코드가 아니라 **"마커가 영원히 있을 것"이라는 제 전제**에 있었습니다.

---

여기서 제가 얻은 교훈이 이번 편의 전부입니다.

**무인 자동화에서 가장 무서운 건 터지는 에러가 아닙니다. "조용히 일을 안 하는 성공"입니다.**

에러는 차라리 친절해요. 빨간 글씨로 "나 고장났어"라고 알려주잖아요. 그런데 `exit 0`을 반환하면서 아무것도 안 하는 훅은, 모든 대시보드에서 초록불입니다. 로그도 깨끗하고, 종료 코드도 정상이고, 알림도 안 울려요. 사람이 "어? 왜 커밋이 안 됐지?"하고 **우연히 눈치챌 때까지** 계속 거짓 평화가 유지됩니다. 그게 사흘이었고, 운이 나빴으면 더 길었을 거예요.

이런 걸 silent-fail(조용한 실패)이라고 부릅니다. 자동화를 무인으로 돌릴수록 이게 진짜 적이에요. 지켜보는 사람이 없으니까요.

---

해법은 두 갈래로 갔습니다.

**1. 마커를 넓혔다.** 마커를 `CLAUDE.md` **또는** `.claude/` 디렉토리 둘 중 하나라도 있으면 "내 vault"로 인정하게 바꿨습니다. `.claude/`는 제 프레임워크 디렉토리라 리셋 정도로는 사라지지 않거든요. 단일 지점 의존을 없앤 거예요.

```python
# CLAUDE.md 또는 .claude/ — 둘 중 하나라도 있으면 이 vault다
if not (os.path.isfile(os.path.join(root, "CLAUDE.md"))
        or os.path.isdir(os.path.join(root, ".claude"))):
    sys.exit(0)
```

**2. 그 계약을 테스트로 못박았다.** 이게 더 중요합니다. 고치는 것보다 **"다시는 조용히 안 깨지게 만드는 것"**이요. 그래서 마커 판정의 계약을 테스트 두 개로 박제했습니다.

- `test_dot_claude_marker_commits` — 루트 `CLAUDE.md`가 **없어도** `.claude/`만 있으면 반드시 커밋된다. (이게 바로 그 버그의 회귀 가드입니다.)
- `test_no_marker_noop` — 마커가 **둘 다 없으면** 커밋 안 한다. (오염 방지가 살아있는지 확인하는 반대편 테스트.)

두 번째 테스트가 핵심이에요. silent-fail을 막겠다고 "무조건 커밋"으로 바꿔버리면, 이번엔 남의 저장소를 오염시키는 원래 위험이 되살아납니다. 그래서 **"이런 경우엔 동작해야 한다"와 "이런 경우엔 동작하면 안 된다"를 둘 다** 테스트로 고정했어요. 한쪽만 있으면 반쪽짜리 안전장치거든요.

silent-fail의 본질이 "성공한 척하는 실패"라면, 그걸 잡는 유일한 방법은 **"진짜 일을 했는지"를 기계가 검증**하게 만드는 것뿐입니다. `exit 0`을 믿지 말고, **결과물(커밋이 실제로 생겼는가)**을 검사하는 거죠. 사람이 우연히 눈치채는 데 의존하면 안 됩니다.

---

이 일을 겪고 나서 제 자동화들을 전부 다시 훑었습니다. "이 훅이 조용히 일을 안 하면 내가 알아챌 방법이 있나?"라는 질문 하나로요. 의외로 많더라고요 — 실패해도 `exit 0` 하는 cron, 결과를 검증하지 않고 "돌았으니 됐겠지"로 넘어가는 스크립트, 조건 하나가 어긋나면 통째로 건너뛰는데 아무도 모르는 훅.

혼자 무인으로 돌리는 사람일수록 이게 치명적입니다. 지켜보는 눈이 적으니까요.

**그래서 — Claude Code 훅이나 cron 같은 무인 자동화가 "조용히 실패(일을 안 하면서 성공한 척)"하고 있는 건 아닌지 걱정되시면, 무료로 30분 안전 진단해드립니다.** 제가 쓰는 12항목 체크리스트로 "이 자동화가 일을 안 하면 알아챌 수 있는가"를 항목별로 짚어드릴게요.
