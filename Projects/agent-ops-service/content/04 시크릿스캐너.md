# 04호 — 티스토리 발행 완성본

## 제목
시크릿 유출을 막으려고 단 스캐너가, 저장할 때마다 "유출이다!"라고 소리쳤다 — 보안 도구의 진짜 어려움

## 본문

저는 직원 대신 AI 에이전트들에게 일을 시켜 1인 회사를 굴리고 있습니다. 그런데 에이전트한테 코드를 맡기면서 가장 무서웠던 게 하나 있었어요. **에이전트가 실수로 API 키를 커밋해버리는 것.**

사람은 "어, 이거 토큰인데?" 하고 멈칫합니다. 에이전트는 그냥 파일에 적고 저장합니다. 한 번 git 히스토리에 박히면 그 키는 영구히 노출된 거고요. GitHub Push Protection이 막아준다지만, 그건 push 단계. 저는 더 앞에서, **저장하는 순간** 잡고 싶었습니다.

그래서 훅을 하나 달았어요. 파일을 Write/Edit할 때마다 그 파일에 크리덴셜 패턴(`ghp_...`, `sk-ant-...`, AWS 키, private key 등)이 들어있는지 스캔하는 PostToolUse 훅(`secret-scan.py`)이요.

**처음엔 감동이었습니다.** 테스트로 가짜 GitHub 토큰을 적어 저장하니 바로 빨간 경고가 떴거든요. "credential 의심 — 1건". 됐다, 이제 에이전트가 키를 흘려도 잡힌다.

그리고 다음 날부터, **저장할 때마다 경고가 떴습니다.**

---

### 스캐너가 자기 자신을 신고하기 시작했다

문제는 두 군데였어요.

**첫째, 테스트 코드.** 저는 이 스캐너가 진짜 토큰을 잡는지 확인하려고 테스트 픽스처에 **가짜 토큰을 일부러 박아뒀습니다.** `FAKE_PAT = "ghp_..."` 같은 식으로요. 당연하죠 — 스캐너가 토큰을 잡는지 테스트하려면 잡을 토큰이 있어야 하니까. 그런데 그 테스트 파일을 저장할 때마다 스캐너가 "여기 토큰 있다!"고 소리쳤습니다. 맞는 말이긴 한데, 그건 **테스트하라고 둔 미끼**잖아요.

**둘째, 더 황당한 건 스캐너 자기 자신이었어요.** `scrub-secrets.py`라는 파일에는 "어떤 게 토큰인가"를 정의하는 정규식 패턴들이 들어 있습니다. `ghp_[A-Za-z0-9]{36}` 같은 거요. 그런데 이 패턴 정의 자체가 또 다른 패턴에 걸려서, **토큰 탐지기를 수정할 때마다 "너 토큰 유출했어"라고 자기가 자기를 신고**했습니다.

이게 제가 나중에 안 이름으로 **나그루프(nag loop)** 입니다. 도구가 너무 잘 잡아서, 잡으면 안 되는 것까지 잡고, 매번 같은 잔소리를 반복하는 상태. 경고가 일상이 되면 사람은 경고를 안 봅니다. **늑대다 소년이 된 스캐너는, 진짜 늑대가 와도 못 막아요.**

---

### 깨달음: 보안 도구의 어려움은 '잡는 것'이 아니다

여기서 제가 배운 게 이번 글의 핵심입니다.

**시크릿 스캐너를 만드는 진짜 어려움은 '시크릿을 잡는 것'이 아니라 '무엇을 안 잡을지'를 정하는 거였어요.** 그리고 더 어려운 건, **그 '안 잡을 목록'이 새로운 보안 구멍이 되지 않게 하는 것.**

생각해보세요. "테스트 파일은 그냥 통과시켜"라고 너무 넓게 풀어버리면? 진짜 토큰을 실수로 테스트 디렉토리에 둔 경우를 놓칩니다. "토큰처럼 생긴 건 다 예시일 거야"라고 풀면? 진짜를 놓치고요. 제외(allowlist)는 편한 만큼 위험합니다. 모든 제외 한 줄이 잠재적 백도어예요.

그래서 저는 두 가지를 **좁고 명시적으로** 갈랐습니다.

---

### 해법 1 — '어디서 났나'로 거르기: 테스트 디렉토리와 스캐너 자신만 제외

스캐너에 딱 두 경로만 제외하도록 박았습니다. 넓은 와일드카드가 아니라, **정확히 이 두 개만.**

```python
# 자기경고 방지: 테스트 픽스처(.claude/tests/)와 secret 도구 자신은 패턴을 *데이터로* 보유.
if "/.claude/tests/" in _norm or os.path.basename(fp) in ("scrub-secrets.py", "secret-scan.py"):
    sys.exit(0)
```

`.claude/tests/` 안의 픽스처와, 패턴 정의를 데이터로 들고 있는 도구 파일 두 개(`scrub-secrets.py`, `secret-scan.py`)만요. 이 둘은 "토큰을 데이터로 보유하는 게 정상인" 유일한 곳이거든요. 그 외 모든 곳은 여전히 빠짐없이 검사합니다. **제외 목록을 일반 소스 코드로 새지 않게, 일부러 손에 꼽을 만큼만 열어둔 겁니다.**

### 해법 2 — '진짜냐 가짜냐'로 거르기: placeholder는 통과

두 번째 축이 더 중요합니다. 코드 예제나 문서에는 `YOUR_TOKEN`, `ghp_xxxxxxxx...`, `<your-key>` 같은 **명백한 자리표시자(placeholder)**가 흔합니다. 이건 토큰처럼 생겼지만 토큰이 아니에요. 이걸 잡으면 KB 문서를 저장할 때마다 경고가 뜹니다.

그래서 명백한 더미 패턴이 매칭 안에 있으면 실 토큰이 아니라고 보고 건너뜁니다.

```python
PLACEHOLDER = re.compile(
    r"(?i)(x{4,}|y{4,}|z{4,}|your[_-]|example|sample|redacted|placeholder|dummy|"
    r"\.\.\.|<[^>]*>|abc123|0{6,}|123456|test[_-]?(?:key|token|secret)|...)")
```

`ghp_xxxxxxxxxxxx...`나 `YOUR_TOKEN`은 통과. 하지만 `ghp_` 뒤에 진짜 36자 랜덤 문자열이 오면 **마스킹**합니다. 그리고 자동 수정은 안 해요 — 오탐으로 원본을 훼손하면 안 되니까, 사람한테 "이거 진짜면 마스킹하고 폐기·교체하세요"라고 알리기만 합니다(`exit 2`, 비차단).

핵심은 **"통과시킨다"와 "마스킹한다"의 경계를 보수적으로 그은 것**이에요. 애매하면 통과가 아니라, 애매한 건 애초에 고신뢰 prefix가 명확한 패턴만 본다는 거죠. 일반 `api_key=...` 같은 저신뢰 패턴은 아예 안 봅니다. 지식 문서엔 정상적인 예제로 흔하니까요.

---

### 그리고 — 이 제외들이 '말'로만 있으면 안 된다

여기가 제가 1호 때부터 계속 강조하는 부분이에요. **제외 규칙은 코드에 한 줄 적어둔다고 지켜지지 않습니다.** 누가 나중에 그 줄을 지우거나, 리팩토링하다 깨도 아무도 모릅니다. 스캐너는 조용히 exit 0을 뱉을 뿐이고요. 보안 도구의 침묵은 "안전"이 아니라 "고장"일 수도 있습니다.

그래서 제외 하나하나에 **계약 테스트**를 붙였습니다.

- `test_tests_dir_excluded` — `.claude/tests/` 안에 가짜 토큰을 둔 파일은 경고가 안 떠야 한다(rc 0).
- `test_scrub_tool_self_excluded` — `scrub-secrets.py` 자신은 패턴을 들고 있어도 경고가 안 떠야 한다(rc 0).
- `test_placeholder_ignored` — `ghp_xxxx...`, `YOUR_TOKEN`은 마스킹 목록이 비어야 한다.
- 그리고 **positive control** — 진짜처럼 생긴 가짜 토큰은 정확히 1건 잡혀서 `<REDACTED:github-pat>`로 마스킹돼야 한다(`test_masks_real_token`).

마지막 positive control이 핵심입니다. 제외를 너무 넓게 풀어서 **아무것도 안 잡는 스캐너**가 돼버리면, 위의 "경고 안 뜸" 테스트들은 전부 통과하면서 보안은 0이 됩니다. 그래서 "이건 안 잡아야 해"와 "이건 반드시 잡아야 해"를 **쌍으로** 둬야, 제외가 구멍이 됐을 때 테스트가 빨개집니다.

---

### 정리하면

- 잡는 건 쉽다. **안 잡을 걸 정하는 게 어렵다.**
- 제외 목록은 편한 만큼 위험하다. **좁고 명시적으로**(경로 두 개, placeholder 패턴만) 풀어라.
- "통과 vs 마스킹"의 경계는 보수적으로. 고신뢰 패턴만 보고, 자동 수정은 하지 마라(오탐이 원본을 훼손하면 안 되니까).
- 그리고 제외든 탐지든 **전부 테스트로 박아라.** "안 잡아야 함" 테스트 옆에 반드시 "반드시 잡아야 함"(positive control)을 둬라. 안 그러면 아무것도 안 잡는 스캐너가 모든 테스트를 통과한다.

저는 이걸 다 직접 깨먹어가며 배웠습니다. 스캐너가 사흘간 매 저장마다 잔소리를 하고 나서야 "아, 이건 잡는 문제가 아니라 안 잡는 문제구나"를 알았어요.
