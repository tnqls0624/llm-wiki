# 03호 — 티스토리 발행 완성본

### 제목
무인 에이전트의 자동 git push, 충돌하면 어떻게 멈추나 — blind push를 버리고 "발산하면 사람에게 넘긴다"로 바꾼 이야기

### 본문

저는 같은 vault(노트 + 자동화 프레임워크가 든 git 저장소)를 맥북, 데스크톱, 그리고 cron으로 자는 동안 도는 무인 세션, 이렇게 여러 곳에서 동시에 굴립니다. 세션이 끝나면 훅이 자동으로 커밋하고 push까지 합니다. 사람이 안 봐도 변경이 origin으로 올라가니 편하죠.

편한데, **제일 무서운 순간이기도 했습니다.**

문제는 이겁니다. 무인 세션이 새벽에 돌면서 노트 몇 개를 고치고 커밋한 뒤 `git push`를 때린다고 칩시다. 그런데 제가 낮에 데스크톱에서 다른 변경을 이미 origin에 올려둔 상태라면? push는 거부됩니다. "non-fast-forward, 먼저 pull 하세요." 사람이 보고 있으면 그냥 pull 받고 합치면 되는데, **새벽 3시 무인 세션엔 사람이 없습니다.**

### 내가 처음 한 짓 (틀린 버전)

부끄럽지만 처음 짠 코드는 그냥 이거였습니다.

```python
git("push")  # 끝. 잘 되겠지.
```

이게 왜 위험한지 한참 뒤에 알았어요. push가 거부되면 두 갈래로 갑니다. 운이 좋으면 그냥 실패하고 변경이 로컬에 고여 있다가 다음에 누가 정리하겠죠(2호에서 다룬 "조용히 멈춤"이 이쪽 친척입니다). 운이 나쁘면, "충돌 자동 해결" 같은 걸 욕심내다 무인 세션이 `git pull --rebase`나 `merge`를 자동으로 시도하게 됩니다. 그리고 **무인 세션은 충돌을 해결할 줄 모릅니다.** 충돌 마커가 박힌 파일을 그대로 커밋해버리거나, 한쪽 변경을 통째로 날리거나. 그게 자동 commit·push 훅을 타고 origin으로 올라가면 — **다른 모든 기기로 전파됩니다.** 자는 사이에, 조용히.

여기서 제가 잡은 원칙이 이번 글의 전부입니다.

> **무인이 못 푸는 건, 무인이 건드리지 않는 게 안전하다. 멈추고 사람에게 넘긴다.**

### blind push를 버리고 — fetch부터 하고 상태를 본다

그래서 세션 종료 훅의 push 로직을 통째로 갈아엎었습니다. push를 때리기 전에 먼저 `git fetch`로 원격 상태를 끌어와서, **지금 로컬과 원격이 어떤 관계인지** 세 갈래로 나눕니다. 핵심은 이 한 줄입니다.

```python
counts = git("rev-list", "--count", "--left-right", "@{u}...HEAD")
behind, ahead = (int(x) for x in counts.stdout.split()[:2])
```

`behind`는 원격이 나보다 앞선 커밋 수, `ahead`는 내가 원격보다 앞선 커밋 수예요. 이 두 숫자로 상태를 판별합니다.

**1. `behind == 0` (나만 앞섰다)** → 원격이 안 움직였으니 그냥 fast-forward push. 안전합니다. 평소엔 거의 이 케이스예요.

**2. `ahead == 0` (원격만 앞섰다, 나는 따라잡기만 하면 됨)** → 내가 새로 만든 게 없으니 `git merge --ff-only @{u}`로 원격을 그냥 끌어당깁니다. **fast-forward만 허용**이라 충돌이 구조적으로 불가능해요. 새 머지 커밋이 안 생깁니다.

**3. 둘 다 0이 아니다 (발산 — 양쪽이 다 앞섰다)** → 여기가 핵심입니다. 무인이 손대면 안 되는 유일한 케이스. **push를 멈춥니다.** merge도, rebase도 안 합니다. 대신 마커 파일 하나를 남깁니다.

```python
else:  # 발산 — 무인 처리 금지, 사람에게 알림
    with open(status, "w", encoding="utf-8") as f:
        f.write(f"⚠ git 발산: 로컬 {ahead} ahead · 원격 {behind} behind. "
                f"자동 push 보류됨 — `git pull --rebase` 후 충돌 해결 필요.\n")
```

`.claude/runtime/sync-status.txt`라는 파일에 "발산했다, 보류했다, 네가 해결해라"를 적어두고 그냥 끝냅니다. 강제 push 안 합니다. 자동 merge 안 합니다. **아무것도 안 하는 게 이 상황에선 가장 안전한 행동이에요.**

### 그런데 "멈췄다"를 사람이 모르면 의미가 없다

여기서 한 번 더 막혔습니다. push를 멈추고 마커를 남긴 건 좋은데, **그 마커를 제가 안 보면** 변경은 영영 로컬에 고이고, 며칠 뒤 다른 기기에서 또 발산이 쌓입니다. 2호의 "사흘을 몰랐다"가 그대로 재현되는 거죠.

그래서 마커를 그냥 두지 않고, **다음 세션을 켤 때 부팅 화면 맨 위에 강제로 띄우게** 했습니다. 세션 시작 훅(`session-context.py`)이 컨텍스트를 조립할 때, sync-status.txt가 있으면 다른 모든 안내를 제치고 **0번 위치에 끼워 넣습니다.**

```python
sync = read(... "sync-status.txt").strip()
if sync:
    parts.insert(0, "## ⚠ Git 동기화 경고\n" + sync + "(해결 후 다음 push 성공 시 사라집니다.)")
```

`append`가 아니라 `insert(0, ...)`인 게 의도예요. 최근 컨텍스트, 레이더 추천, 뭐가 됐든 그 위에 **"⚠ 발산: 직접 해결하세요"가 맨 먼저** 보입니다. 그리고 제가 `git pull --rebase`로 풀어주면, 다음 세션 종료 때 `behind == 0`이 되면서 정상 push가 나가고 — 마커가 자동으로 지워집니다. 경고가 사라지면 그게 해결됐다는 신호예요.

즉 흐름이 이렇게 닫힙니다. **무인이 발산을 감지 → 멈춤 + 마커 → 다음 부팅에 사람에게 경고 → 사람이 해결 → 마커 소멸.** 무인은 절대 충돌에 손대지 않고, 사람은 절대 놓치지 않습니다.

### "말로 한 안전장치"는 안전장치가 아니라서 — 테스트로 박았다

여기가 제가 1호부터 계속 강조하는 부분인데요. "발산하면 멈추게 했어요"는 **말일 뿐입니다.** 코드가 정말 그렇게 도는지 증명 못 하면, 어느 날 리팩터링하다 조용히 깨져도 모릅니다. 그래서 진짜 git 저장소 두 개(로컬 A, 다른 기기 흉내 B)를 실제로 만들어 4가지 시나리오를 돌리는 테스트를 박아뒀습니다.

- **ahead** — A만 커밋 → 정말 origin에 push됐고 그 커밋에 파일이 들어갔는지까지 확인(no-op면 HEAD가 안 움직이니 바로 잡힘).
- **behind** — B가 원격을 올림, A는 가만 → A가 ff-only로 따라잡았는지.
- **diverged** — B도 올리고 A도 커밋(양쪽 전진) → **원격이 안 바뀌었는지(push 보류) + 마커가 생겼는지.**
- **resolve** — 발산 후 사람이 `pull --rebase` → 다음 세션에 정상 push되고 **마커가 지워졌는지.**

특히 diverged 테스트는 "무인이 원격을 건드리지 않았다"를 `self.rhead()`(원격 HEAD)가 그대로인지로 못박습니다. 행여 누가 자동 merge를 슬쩍 넣으면 이 테스트가 빨개져요. 안전장치를 말이 아니라 **빨간불로** 지키는 거죠.

### 정리하면

무인 자동화에서 제일 위험한 건 "에이전트가 똑똑하게 충돌을 풀려고 시도하는 것"이었습니다. 똑똑한 척하다 다른 기기까지 오염시키는 것보다, **"나는 이건 못 풀어. 멈추고 너한테 넘길게"가 압도적으로 안전합니다.** 무인의 미덕은 자율이 아니라 **자기 한계를 아는 것**이더라고요.

- blind push 금지 → fetch 먼저, 상태를 셋으로 나눈다.
- 안전한 두 케이스(나만 앞섬 / 원격만 앞섬)만 자동.
- 발산(양쪽 앞섬)은 멈추고 마커 → 다음 부팅 최상단에 경고 → 사람이 해결 → 자동 소멸.
- 그리고 이 전부를 4 시나리오 테스트로 못박는다.
