# 컨테이너 안에서 리눅스 기본기 다시 익히기 — load average부터 journalctl까지

2. `2. top-load-average.png` — 컨테이너 안 top의 load average 출력 (shot: 직접 실행 화면 캡처)

최근 AI 인프라 공부를 시작하면서 모델 학습 코드를 짜는 것보다 그 코드가 실제로 돌아가는 서버 환경을 이해하는 게 더 급하다는 걸 느꼈다. 요즘 운영 환경은 대부분 컨테이너 위에서 돌아가는데, 나는 컨테이너를 "가벼운 서버" 정도로만 알고 있었지 안에 뭐가 있고 뭐가 없는지는 제대로 만져본 적이 없었다. 그래서 하루는 코드를 하나도 안 쓰고 `docker run -it ubuntu`로 컨테이너 하나만 띄워놓고, 그 안에서 서버 상태를 확인하는 기본 명령어들을 하나씩 돌려봤다. 이번 글에서는 그 과정에서 확인한 컨테이너와 서버의 차이, load average 읽는 법, 퍼미션 비트를 정리한다.

## 컨테이너란 무엇인가

컨테이너를 VM(가상머신)과 헷갈리기 쉬운데, 둘은 격리 단위가 다르다.

- **VM**: 하드웨어까지 가상화한다. 자체 커널을 갖고 부팅부터 따로 한다.
- **컨테이너**: 호스트의 커널을 그대로 공유하면서, 프로세스 하나를 namespace·cgroup으로 격리한 것이다.

> 🖼️ **[사진 1]** 컨테이너와 VM의 구조 비교 — VM은 게스트 OS(커널)를 층마다 따로 얹지만, 컨테이너는 호스트 커널 하나를 공유한다
> → 업로드: `1. container-vs-vm.png`

쉽게 말해 컨테이너는 "작은 서버"가 아니라 "격리된 프로세스 하나"에 더 가깝다. 이 정의가 중요한 이유는 뒤에 나올 journalctl 실패와 바로 연결된다.

리눅스가 부팅되면 커널이 제일 먼저 띄우는 프로세스가 PID 1이고, 요즘 대부분의 배포판에서는 이 자리를 systemd(init 시스템)가 맡는다. systemd는 다른 모든 서비스·로그·소켓을 관리하는 컨트롤타워 역할을 한다. 그런데 `docker run -it ubuntu`로 컨테이너를 띄우면 그 컨테이너의 PID 1은 내가 실행한 셸(bash) 그 자체다. systemd가 아예 없다.

## 자원 확인 — lscpu, top, free, df, vmstat

먼저 자원 확인부터 했다.

- `lscpu`: 코어 수 확인 — 11개였다.
- `top`: CPU/메모리 사용률과 load average 확인.
- `free -m`: 사용 가능한 메모리(MB 단위) 확인.
- `df -h`: 디스크 사용량 확인.
- `vmstat 1`: 1초 간격으로 CPU/메모리 상태를 반복 출력.

여기서 load average를 처음엔 숫자만 읽고 의미를 몰랐다. load average는 "실행 중이거나 CPU를 기다리는 프로세스 수"의 1분/5분/15분 평균이다. 절대값만 보면 의미가 없고 코어 수와 비교해야 한다. 코어가 11개면 load average도 최대 11 근처까지 올라갈 수 있다는 뜻이고, 11을 넘기기 시작하면 그때부터 CPU를 기다리는 프로세스가 쌓인다는 뜻이다. 코어 수 대비 비율로 봐야 부하 여부를 판단할 수 있다는 걸 여기서 알았다.

## 최소 이미지엔 서버 도구가 없다

`top`을 켜려고 했는데 컨테이너 안에 명령 자체가 없었다. `apt update && apt install`로 직접 설치해야 했다. `whoami; id`를 쳐보니 계정도 기본이 root였다.

두 가지를 한 번에 확인한 셈이다. 첫째, 컨테이너 기본 이미지는 필요한 실행 바이너리만 담아 크기를 최소화하는 게 원칙이라 top·htop 같은 운영 도구는 기본 포함되지 않는다. 둘째, 컨테이너는 기본적으로 root 계정으로 뜬다(운영 환경에서는 보안상 non-root로 바꾸는 게 원칙이지만, 방금 띄운 최소 이미지 컨테이너에서는 root가 기본값이었다).

## journalctl이 실패한 이유

`journalctl`을 실행해봤는데 실패했다. 왜 안 되는지 찾아봤다.

로그를 관리하는 journald는 systemd의 구성 요소 중 하나다. 그런데 앞서 확인했듯 이 컨테이너에는 systemd(init)가 없고, PID 1은 내가 띄운 셸이다. journald 프로세스 자체가 존재하지 않으니 journalctl로 조회할 로그도 없는 게 당연하다. 실패는 버그가 아니라 컨테이너 구조 자체의 결과였다. 컨테이너는 격리된 프로세스 하나일 뿐, 완전한 서버가 아니라는 걸 이 실패 하나로 확인했다.

## 퍼미션 비트 — chmod 400/600

```
root@1e001715902c:/# touch /tmp/a; chmod 400 /tmp/a; ls -l /tmp/a
-r-------- 1 root root 0 Jul  6 14:23 /tmp/a
root@1e001715902c:/# chmod 600 /tmp/a
root@1e001715902c:/# ls -l /tmp/a
-rw------- 1 root root 0 Jul  6 14:23 /tmp/a
```

`ls -l` 출력 앞 10자리 중 첫 글자는 파일 종류(`-`면 일반 파일)이고, 그 뒤 9자리가 소유자·그룹·기타의 rwx를 3자리씩 나눈 것이다. 각 자리는 r=4, w=2, x=1의 합으로 숫자 하나에 대응한다.

- `400` = 4+0+0 → 소유자만 읽기(`r--`), 그룹·기타는 권한 없음.
- `600` = 4+2+0 → 소유자 읽기+쓰기(`rw-`), 그룹·기타는 그대로 없음.

숫자 하나가 rwx 비트 조합에 그대로 매핑된다는 걸 출력으로 직접 확인했다.

## 정리

이번 실습에서 새로 안 것 세 가지.

1. 최소 이미지는 서버 관리 도구를 기본으로 갖고 있지 않다 — 필요하면 `apt`로 직접 설치해야 한다.
2. journalctl 실패는 이 컨테이너에 systemd(init)가 없고 PID 1이 셸이라는 구조 때문이다 — 컨테이너는 완전한 서버가 아니라 격리된 프로세스라는 뜻이다.
3. chmod 400/600 같은 숫자는 rwx 권한 비트에 그대로 대응한다.
