# 02 메모리 관리
허브: [[Infra]]

가상 메모리부터 OOM killer까지, 서버를 운영하면서 마주치는 메모리 문제의 전부를 다룬다. 핵심 메시지는 하나다 — **리눅스의 메모리 숫자는 직관과 다르게 읽어야 한다.** "메모리 90% 사용 중"이 정상일 수 있고, RSS 합계가 물리 메모리를 넘어도 정상일 수 있다. 왜 그런지 원리부터 이해하면 면접 답변과 장애 대응이 동시에 해결된다.

---

## 1. 가상 메모리 — 모든 프로세스는 자기만의 우주에 산다

### 1.1 왜 가상 메모리인가

물리 메모리를 프로세스에 직접 나눠주면 세 가지 문제가 생긴다. ① 프로세스끼리 서로의 메모리를 침범할 수 있고(보호 없음), ② 연속된 빈 공간을 찾아야 해서 단편화가 심해지고, ③ 물리 메모리보다 큰 프로그램을 못 돌린다. 가상 메모리는 이 셋을 한 번에 푼다: **각 프로세스에게 독립된 가상 주소공간을 주고, 커널이 page table로 가상 주소 → 물리 주소를 변환한다.**

비유: 모든 입주자(프로세스)에게 "1호부터 시작하는 자기만의 아파트 도면"을 나눠주고, 실제 어느 동 어느 호에 사는지는 관리사무소(커널 + MMU)만 아는 구조다. 입주자는 옆집 도면을 볼 수도, 침범할 수도 없다.

### 1.2 페이지, page table, TLB

- 매핑의 단위는 **페이지**이며 x86-64 기본값은 **4KB**다. 가상 주소공간도, 물리 메모리(이쪽은 frame이라 부른다)도 4KB 단위로 쪼개 관리한다.
- 변환 정보는 **page table**에 있다. x86-64는 4~5단계 트리 구조라, 매핑 하나를 찾으려면 메모리를 여러 번 읽어야 한다.
- 매번 트리를 타면 너무 느리므로 CPU 안에 변환 결과 캐시인 **TLB**(Translation Lookaside Buffer)를 둔다. TLB hit이면 변환이 사실상 공짜, miss면 page table walk 비용을 낸다. **TLB 엔트리 수는 한정돼 있어서, 메모리를 넓게 랜덤하게 만지는 워크로드(DB, 대형 인메모리 캐시)는 TLB miss가 성능 병목이 된다 — 이것이 뒤에 나올 hugepages의 존재 이유다.**

### 1.3 주소공간 레이아웃

프로세스 하나의 가상 주소공간은 대략 이렇게 생겼다 (`cat /proc/<pid>/maps`로 직접 확인 가능):

| 영역 | 내용 | 특징 |
|---|---|---|
| text | 실행 코드 | 읽기+실행 전용, 같은 바이너리를 띄운 프로세스끼리 물리 페이지 공유 |
| data / bss | 전역·정적 변수 | 초기값 있는 것(data) / 0으로 시작하는 것(bss) |
| heap | `malloc` 등 동적 할당 | `brk`/`sbrk`로 위로 자람 |
| mmap 영역 | 공유 라이브러리, 파일 매핑, 대형 `malloc` | glibc malloc은 큰 할당(기본 128KB 이상)을 `mmap`으로 처리 |
| stack | 지역 변수, 함수 호출 프레임 | 아래로 자람, 스레드마다 별도 스택 ([[01 프로세스와 스레드]]) |

### 1.4 page fault — minor와 major의 구분이 핵심

가상 페이지에 접근했는데 page table에 유효한 매핑이 없으면 CPU가 트랩을 걸고 커널의 page fault 핸들러가 실행된다. 여기서 두 종류를 구분하는 것이 실무 포인트다.

- **minor fault**: **디스크 I/O 없이** 해결되는 fault — 물리 페이지가 이미 메모리에 있어 매핑만 추가하면 되거나, 새 zero 페이지를 배정하면 되는 경우. 예 — `malloc` 직후 첫 접근(리눅스는 할당 시점이 아니라 **첫 접근 시점에** 물리 페이지를 배정한다, demand paging), 이미 page cache에 있는 파일을 `mmap`으로 접근. 디스크를 안 거치므로 싸다. 정상 동작의 일부.
- **major fault**: 페이지 내용을 **디스크에서 읽어와야** 하는 경우. swap-in, 캐시에 없는 파일의 mmap 접근 등. 디스크 I/O가 끼므로 저장장치에 따라 수십~수만 배 느리다. **major fault가 지속적으로 발생하면 메모리 압박 신호다** — `ps -o min_flt,maj_flt`, `sar -B`(majflt/s)로 본다.

> 면접 단골: "malloc이 성공했는데 실제 메모리는 언제 할당되나?" → 첫 접근 시 minor fault로. 이것이 곧 overcommit(5장)의 출발점이다.

## 2. 페이지 캐시 — "free 메모리가 없어요"의 90%는 오해

### 2.1 파일 I/O는 기본적으로 캐시를 경유한다

`read()`는 디스크에서 직접 읽지 않는다. 커널이 먼저 **page cache**를 확인하고, 없으면 디스크에서 읽어 캐시에 채운 뒤 사용자 버퍼로 복사한다. `write()`도 디스크에 바로 쓰지 않는다 — 캐시 페이지를 수정하고 **dirty** 표시만 한 채 리턴하고, 나중에 **writeback** 커널 스레드가 모아서 디스크로 내린다. 그래서 같은 파일을 두 번 읽으면 두 번째가 압도적으로 빠르고, write가 디스크 속도보다 빠르게 "끝난" 것처럼 보인다.

비유: 페이지 캐시는 도서관의 "최근 대출 도서 비치대"다. 비치대가 꽉 차 있는 건 도서관이 일을 잘하고 있다는 뜻이지, 공간 부족이 아니다. 새 책이 오면 비치대의 오래된 책을 서가로 돌려보내면(회수) 그만이다.

### 2.2 buff/cache가 커도 메모리 부족이 아니다

리눅스는 "남는 메모리는 낭비"라는 철학으로 빈 메모리를 적극적으로 캐시에 쓴다. `free -h`에서 `buff/cache`가 수십 GB여도, 그 대부분은 애플리케이션이 메모리를 요구하면 **즉시 회수 가능**하다(clean 페이지는 버리면 되고, dirty 페이지는 디스크에 쓴 뒤 회수). 그래서 보는 칼럼은 `free`가 아니라 `available`이다(3장).

### 2.3 운영자가 만지는 손잡이들

```bash
# dirty page가 dirtyable 메모리(free+회수 가능 페이지 — 전체 RAM이 아님)의 몇 %일 때 동작하는지
sysctl vm.dirty_background_ratio   # 이 비율 넘으면 백그라운드 writeback 시작 (기본 10)
sysctl vm.dirty_ratio              # 이 비율 넘으면 write하는 프로세스 자체가 블록됨 (기본 20)

sync                               # dirty page 전부 디스크로 강제 flush
echo 3 > /proc/sys/vm/drop_caches  # 캐시 강제 비우기 (1=pagecache, 2=slab, 3=둘 다)
```

- `vm.dirty_ratio`에 도달하면 write를 호출한 프로세스가 **동기적으로 writeback을 떠안으며 멈춘다** — 대량 쓰기 작업에서 갑자기 latency가 튀는 고전적 원인. 대용량 메모리 서버에서는 비율(%) 대신 절대값인 `vm.dirty_bytes`/`vm.dirty_background_bytes`로 제어하는 편이 예측 가능하다.
- `drop_caches`는 **운영 서버에서 함부로 쓰지 말 것.** 캐시를 비우면 그 직후 모든 파일 I/O가 디스크로 직행해 latency가 폭등한다. 벤치마크 전 초기화 같은 명확한 목적 외에는 쓸 일이 없고, "메모리 확보"용으로 쓰는 것은 항상 잘못이다 — 캐시는 어차피 필요하면 자동 회수된다.

writeback과 디스크의 관계는 [[03 파일시스템과 스토리지]]에서 이어진다.

## 3. 메모리 측정 제대로 — VSZ, RSS, PSS, USS

### 3.1 네 가지 지표

| 지표 | 의미 | 함정 |
|---|---|---|
| **VSZ** (Virtual Size) | 프로세스가 **예약한 가상 주소공간** 총량 | 실제 사용량과 무관. JVM이나 mmap 많이 쓰는 프로세스는 수십 GB로 보여도 정상 |
| **RSS** (Resident Set Size) | 실제 물리 메모리에 올라온 페이지 합 | **공유 페이지(라이브러리 등)를 프로세스마다 중복 계산** → 전체 합이 물리 메모리를 초과할 수 있음 |
| **PSS** (Proportional Set Size) | 공유 페이지를 공유 프로세스 수로 **나눠서 배분** | 시스템 전체 합산이 의미 있는 유일한 지표. `/proc/<pid>/smaps_rollup`에서 확인 |
| **USS** (Unique Set Size) | 그 프로세스만 단독으로 쓰는 페이지 | "이 프로세스를 죽이면 돌려받는 메모리"에 가장 가까움 |

```bash
ps aux                                    # VSZ, RSS 칼럼
grep -e Pss -e Rss /proc/<pid>/smaps_rollup
smem -P nginx                             # PSS/USS를 프로세스별로 보여주는 도구
```

> 면접 단골: "nginx worker 10개의 RSS가 각 100MB면 총 1GB를 쓰는 건가?" → 아니다. 공유 라이브러리·공유 텍스트가 중복 계산된 것이므로 PSS를 합산해야 실제에 가깝다.

### 3.2 free -h 제대로 읽기

```
              total        used        free      shared  buff/cache   available
Mem:           62Gi        20Gi       2.0Gi       1.0Gi        40Gi        39Gi
```

- `free` 2GB만 보고 "메모리 부족"이라고 판단하면 안 된다. 40GB의 buff/cache 대부분이 회수 가능하다.
- **`available`이 진짜 여유다.** 커널이 "swap 없이, 캐시 회수를 감안해서 새로 할당해 줄 수 있는 양"을 추정한 값(`/proc/meminfo`의 `MemAvailable`). 알람 기준도 `used`가 아니라 `available`에 걸어야 한다.

### 3.3 /proc/meminfo 주요 필드

| 필드 | 의미 |
|---|---|
| `MemTotal` / `MemFree` / `MemAvailable` | 전체 / 완전 빈 / 실질 가용 |
| `Buffers` / `Cached` | 블록 디바이스 메타 버퍼 / 페이지 캐시 |
| `Dirty` / `Writeback` | 아직 디스크에 안 쓴 페이지 / 지금 쓰는 중인 페이지 |
| `Active` / `Inactive` | 최근 접근됨 / 회수 후보 (LRU 리스트) |
| `AnonPages` | 익명 메모리(heap 등, 파일 배경 없음 → 회수하려면 swap 필요) |
| `Slab` (`SReclaimable`/`SUnreclaim`) | 커널 객체 캐시 (8장) |
| `SwapTotal` / `SwapFree` | swap 공간 |
| `HugePages_Total` 등 | explicit hugepages 현황 (6장) |

여기서 중요한 구분 하나 — **익명(anonymous) 페이지 vs 파일(file-backed) 페이지.** 파일 페이지는 디스크에 원본이 있어 그냥 버리면 되지만, 익명 페이지(heap)는 버릴 곳이 없어 **swap이 없으면 회수가 불가능하다.** 이 차이가 다음 장의 swap 논쟁을 만든다.

## 4. swap — 오해가 가장 많은 메커니즘

### 4.1 swap의 진짜 역할

swap은 "메모리가 모자랄 때 쓰는 비상 공간"이라기보다, **한 번 쓰고 다시 안 건드리는 콜드 익명 페이지를 디스크로 밀어내서, 그 자리를 더 가치 있는 페이지 캐시로 쓰게 해주는 장치**다. 데몬의 초기화 코드처럼 두 번 다시 실행 안 될 페이지가 RAM을 점유하는 것보다, 그 공간에 자주 읽는 파일 캐시를 두는 편이 시스템 전체 성능에 낫다.

### 4.2 vm.swappiness

```bash
sysctl vm.swappiness   # 기본 60
```

- 0~200(구버전 0~100). **익명 페이지를 swap-out하는 것 vs 파일 캐시를 회수하는 것 사이의 선호도**다. 높을수록 swap을 적극 사용.
- **`vm.swappiness=0`이어도 swap이 완전히 금지되는 게 아니다.** 메모리 압박이 충분히 심하면 커널은 여전히 swap한다. "swap 절대 금지"는 swap 파티션을 없애거나 `mlock`(8장)으로만 보장된다. 이 점을 정확히 말하면 면접에서 점수를 딴다.

### 4.3 zram / zswap, 그리고 "swap 없는 서버" 논쟁

- **zram**은 RAM 일부를 압축 블록 디바이스로 만들어 swap으로 쓰는 것(디스크 안 감), **zswap**은 디스크 swap 앞단의 압축 캐시다. 둘 다 "swap의 비용을 디스크 I/O에서 CPU 압축으로 바꾸는" 트레이드오프.
- **swap 반대론**: swap이 있으면 메모리 누수가 OOM으로 빨리 죽는 대신 시스템 전체가 thrashing(swap-in/out 반복)으로 몇 시간씩 기어가며 "죽지도 살지도 않는" 최악의 상태가 된다. latency 민감 서비스에서는 빠른 죽음 + 재스케줄이 낫다.
- **swap 찬성론**(커널 개발자 쪽 시각): 적당한 swap은 콜드 익명 페이지를 치워 캐시 효율을 높이고, 메모리 회수에 여유를 준다.
- **K8s가 노드 swap을 기본 비활성으로 둬 온 이유**: kubelet은 pod의 메모리 요청/한도를 기준으로 스케줄링하고 cgroup으로 격리하는데, swap이 있으면 "이 pod이 실제 RAM을 얼마나 점유하는가"가 비결정적이 되어 격리와 예측 가능성이 깨진다. 죽을 pod은 빨리 죽고(OOMKilled) 다른 노드에 재스케줄되는 편이 운영 모델에 맞다. (최근 버전에서 swap 지원이 단계적으로 열리고 있으나, 기본 정서는 여전히 "예측 가능성 우선"이다.)

## 5. overcommit과 OOM killer — 항공사 좌석 초과판매 모델

### 5.1 overcommit: 약속은 크게, 지불은 나중에

1장에서 본 대로 물리 페이지는 첫 접근 시에 배정된다. 그렇다면 `malloc` 시점에 커널이 약속만 해두고 물리 메모리보다 많이 약속해도 되지 않을까? — 이것이 **overcommit**이다. 항공사가 노쇼를 예상하고 좌석보다 많은 티켓을 파는 것과 같다. 대부분의 프로그램은 할당받은 메모리를 다 쓰지 않으므로(fork 후 exec, 큰 버퍼의 일부만 사용 등) 보통은 잘 돌아간다.

```bash
sysctl vm.overcommit_memory
```

| 값 | 모드 | 동작 |
|---|---|---|
| `0` | heuristic (기본) | 명백히 터무니없는 요청만 거절, 나머지는 허용 |
| `1` | always | 무조건 허용 (Redis 등 fork 기반 스냅샷을 쓰는 소프트웨어가 권장하기도 함) |
| `2` | never (strict) | `swap + RAM × vm.overcommit_ratio` 한도 내에서만 약속. malloc이 실제로 NULL을 반환할 수 있음 |

### 5.2 좌석이 진짜 모자라면 — OOM killer

승객이 예상보다 많이 탑승하면(모두가 약속된 페이지를 실제로 만지면) 누군가를 내리게 해야 한다. 메모리 회수(캐시 정리, swap)로도 해결이 안 될 때 커널은 **OOM killer**를 발동해 프로세스 하나를 강제 종료(SIGKILL)한다.

- 희생자 선정: 프로세스마다 **`oom_score`**(대략 "전체 메모리 대비 점유 비율" 기반)를 매기고 가장 높은 쪽을 죽인다. `/proc/<pid>/oom_score`로 확인.
- **`oom_score_adj`**(-1000 ~ +1000)로 조정 가능. `-1000`은 절대 죽이지 말 것(sshd 등에 흔히 설정), 양수는 우선 희생. K8s는 pod QoS 클래스(Guaranteed/Burstable/BestEffort)에 따라 이 값을 자동 설정한다.

### 5.3 dmesg에서 OOM 로그 읽기

```bash
dmesg -T | grep -i -A 20 "out of memory"
# 또는
journalctl -k | grep -i oom
```

로그에서 볼 것 세 가지: ① `invoked oom-killer`를 유발한 프로세스(범인이 아니라 마지막으로 할당 요청한 자일 수 있음), ② 프로세스별 RSS 테이블(누가 실제로 많이 먹었나), ③ `Killed process <pid> (<name>)` — 실제 희생자. 유발자 ≠ 최대 점유자 ≠ 희생자가 전부 다를 수 있다는 점에 주의.

### 5.4 cgroup OOM은 전역 OOM과 다르다

- **전역 OOM**: 노드 전체 메모리가 고갈. 시스템의 아무 프로세스나 희생될 수 있다.
- **cgroup OOM**: 특정 cgroup(= 컨테이너)의 `memory.max` 한도 초과. **그 cgroup 안에서만** 희생자를 고른다. 노드는 멀쩡한데 컨테이너만 죽는다.
- K8s에서 컨테이너가 **OOMKilled, exit code 137**(128+9, SIGKILL)로 죽었다면 대부분 cgroup 한도 OOM이다 → limits가 작은가, 누수인가를 따진다. 노드 `dmesg`에는 `Memory cgroup out of memory`로 찍혀 전역 OOM과 구분된다. 자세한 cgroup 구조는 [[06 컨테이너 내부 구조]] 참고.

## 6. hugepages — TLB를 위한 큰 페이지

### 6.1 왜 필요한가

4KB 페이지로 256GB를 매핑하면 페이지가 6천만 개를 넘는다. TLB 엔트리는 한정돼 있어 hit rate가 무너진다. 페이지를 **2MB**(또는 1GB)로 키우면 같은 TLB 엔트리로 512배 넓은 영역을 커버해 TLB miss가 급감하고, page table 자체도 작아진다.

### 6.2 두 가지 방식과 각각의 실무 위치

| 방식 | 동작 | 실무 |
|---|---|---|
| **THP** (Transparent Huge Pages) | 커널이 자동으로 4KB들을 2MB로 합쳐줌. 앱 수정 불필요 | 기본 활성(`madvise` 또는 `always`)인 배포판이 많음. **일부 DB 벤더는 비활성 권장** |
| **explicit hugepages** | `vm.nr_hugepages`로 부팅 시 예약, 앱이 명시적으로 사용 | DPDK, 일부 DB의 버퍼 풀, 가상화(KVM 게스트 메모리), GPU 핀 메모리 영역 등 |

### 6.3 THP가 DB에서 미움받는 이유

THP는 흩어진 4KB 페이지들을 2MB 연속 영역으로 모으는 **compaction/defragmentation**을 백그라운드(khugepaged) 또는 할당 시점에 수행한다. 메모리가 단편화된 상태에서 이 작업이 할당 경로에 끼어들면 **수십~수백 ms의 latency 스파이크**가 발생한다. 처리량보다 tail latency가 중요한 DB·캐시 서버(일부 DB 벤더 문서가 명시적으로 비활성을 권고한다)에서는 끄거나 `madvise`(요청한 영역만)로 두는 것이 정석이다.

```bash
cat /sys/kernel/mm/transparent_hugepage/enabled   # [always] madvise never
grep AnonHugePages /proc/meminfo                  # THP 사용량
grep HugePages /proc/meminfo                      # explicit hugepages 현황
```

> 면접 단골: "hugepages는 무조건 좋은가?" → TLB miss는 줄지만, THP는 compaction 스파이크·메모리 낭비(2MB 중 일부만 써도 통째 점유) 비용이 있다. "어떤 워크로드냐"로 답해야 한다.

## 7. NUMA — 메모리에도 거리가 있다

### 7.1 로컬 vs 원격

멀티소켓 서버에서 각 CPU 소켓은 자기에게 직결된 메모리(로컬 노드)와, 다른 소켓의 메모리(원격 노드)를 모두 쓸 수 있지만 **원격 접근은 소켓 간 인터커넥트를 거쳐 수십 % 이상 느리다.** 이것이 NUMA(Non-Uniform Memory Access)다. 커널은 기본적으로 "스레드가 실행 중인 노드에서 메모리를 할당"(local allocation)하려 하지만, 스케줄러가 스레드를 다른 소켓으로 옮기면 메모리는 그대로 남아 원격 접근이 늘어난다.

```bash
numactl --hardware        # 노드 수, 노드별 메모리 크기, 노드 간 거리 행렬
numastat                  # numa_hit / numa_miss / other_node 통계
numastat -p <pid>         # 프로세스의 노드별 메모리 분포
numactl --cpunodebind=0 --membind=0 ./app   # CPU와 메모리를 노드 0에 고정
```

`numa_miss`가 꾸준히 증가하면 원격 할당이 일어나고 있다는 뜻이다. 대형 DB는 한 노드 메모리보다 큰 버퍼를 쓰므로 `interleave` 정책(노드에 골고루 분산)을 택하기도 한다.

### 7.2 GPU 서버에서 NUMA가 더 중요한 이유

GPU는 PCIe(또는 NVLink)로 **특정 CPU 소켓에 물리적으로 연결**되어 있다. 호스트→GPU 데이터 복사(H2D)의 출발지인 핀 메모리(pinned/page-locked memory)가 GPU와 같은 NUMA 노드에 있어야 전송 대역폭이 온전히 나온다. 데이터 로더 프로세스가 반대쪽 소켓에 스케줄되면 모든 학습 배치가 소켓 간 인터커넥트를 한 번 더 건너게 된다. `nvidia-smi topo -m`으로 GPU-NUMA 토폴로지를 확인하고, 데이터 로더·전처리 프로세스를 GPU와 같은 노드에 `numactl`로 묶는 것이 AI 인프라의 기본기다 ([[AI-Infra]]).

## 8. 짧지만 알아야 하는 것들

- **mmap vs read/write**: `read()`는 "커널이 page cache → 내 버퍼로 복사"이고, `mmap()`은 "page cache 페이지를 내 주소공간에 직접 매핑"이라 복사 한 번이 사라진다. 대신 페이지마다 fault 비용이 있어 작은 순차 읽기에는 read가 낫다.
- **slab**: 커널이 자기 객체(task_struct, dentry, inode 등)를 할당하는 객체 캐시. `slabtop`으로 보면 보통 **dentry/inode 캐시**가 상위에 있는데, 파일을 많이 여닫는 워크로드의 정상 동작이다. `/proc/meminfo`의 `SReclaimable`은 압박 시 회수된다. 수백만 개 작은 파일을 다루는 서버에서 "프로세스 RSS 합계는 작은데 메모리가 없다"면 slab을 의심한다.
- **mlock**: `mlock()`/`mlockall()`은 지정한 페이지를 **swap-out 금지**로 못 박는다. 암호 키를 디스크에 남기지 않으려는 보안 용도, latency를 위해 swap을 원천 차단하려는 용도(일부 DB, 실시간 시스템). `ulimit -l`(RLIMIT_MEMLOCK)의 제한을 받는다.

## 9. 실무 시나리오

### 9.1 "서버 메모리 90% 사용 중이래요" (알람 대응)

```bash
free -h                          # 1. available을 본다 — 충분하면 상황 종료에 가깝다
cat /proc/meminfo | head -20     # 2. MemAvailable, Dirty, AnonPages 확인
vmstat 1 5                       # 3. si/so(swap in/out) 칼럼 — 0이 아니면 진짜 압박
sar -B 1 5                       # 4. majflt/s — major fault 지속 발생 여부
smem -t -k | tail -20            # 5. 정말 부족하다면: PSS 기준 누가 먹는지
```

판단 기준: `available`이 넉넉하고 si/so가 0이면 **buff/cache가 큰 정상 상태**다 — 알람 기준을 `available`로 바꾸라고 제안하는 것까지가 대응이다. `available`이 바닥이고 si/so가 움직이면 진짜 압박이므로 5번으로 범인을 찾는다. (분석 방법론 전반은 USE Method — [[08 성능 분석과 트러블슈팅]].)

### 9.2 OOM 사후 분석 (post-mortem)

```bash
dmesg -T | grep -i -B 2 -A 25 "out of memory"
```

1. **누가 죽었나** — `Killed process` 줄. K8s라면 `kubectl describe pod`에서 `OOMKilled` / exit 137 확인.
2. **전역인가 cgroup인가** — 로그에 `Memory cgroup out of memory`가 있으면 컨테이너 한도 OOM, 없으면 노드 전역 OOM. 대응이 완전히 다르다(한도 조정 vs 노드 증설/배치 조정).
3. **왜 죽었나** — 로그의 프로세스 RSS 테이블에서 점유 상위를 본다. 죽기 직전 메트릭(모니터링)의 RSS 추이가 ① 계단식 우상향이면 **누수 의심**, ② 평소엔 평평한데 스파이크로 죽었으면 **트래픽/배치 이벤트**, ③ 한도 바로 아래에서 늘 운영됐다면 **한도가 워크로드에 비해 작은 것**.
4. **재발 방지** — 누수면 힙 프로파일링으로 넘기고, 한도 문제면 requests/limits 조정, 이벤트성이면 해당 작업의 메모리 상한 설정.

### 9.3 메모리 누수 추적 입문

```bash
# RSS 추이 기록 (가장 단순하고 확실한 1차 증거)
while true; do
  ps -o rss= -p <pid> >> /tmp/rss.log; sleep 60
done
```

- 핵심은 **시간축**이다. 한 시점의 RSS는 누수를 증명하지 못한다 — GC 언어 런타임이나 캐시는 원래 우상향하다 안정된다. 트래픽이 일정한데 RSS가 **무한히** 우상향하면 누수다.
- `smem`으로 PSS/USS를 함께 보면 "공유 메모리 증가"와 "프로세스 고유 증가"를 구분할 수 있다. USS가 자라면 진짜 그 프로세스 문제다.
- 다음 단계는 언어별 힙 프로파일러(jemalloc profiling, pprof, JVM heap dump 등)와 `/proc/<pid>/smaps`로 어떤 매핑이 자라는지 좁히는 것이다.

## 10. 면접 단골 요약

| 질문 | 한 줄 답의 뼈대 |
|---|---|
| minor vs major fault? | 매핑만 추가 vs 디스크 I/O 발생. major 지속 = 메모리 압박 신호 |
| free의 buff/cache가 큰데 문제인가? | 아니다. 회수 가능한 페이지 캐시. 진짜 여유는 `available` |
| RSS 합이 물리 메모리를 넘는 이유? | 공유 페이지 중복 계산. 합산은 PSS로 |
| swappiness=0이면 swap 안 하나? | 아니다. 선호도일 뿐, 극한 압박에선 swap 함 |
| overcommit이 왜 기본인가? | demand paging 덕에 약속과 사용이 분리되고, 대부분 약속을 다 안 쓰므로 |
| OOM 희생자는 어떻게 정해지나? | `oom_score`(점유 비율 기반) + `oom_score_adj` 조정 |
| exit 137의 의미? | 128+9=SIGKILL. K8s에선 대부분 cgroup 한도 OOMKilled |
| THP를 끄라는 권고 이유? | compaction이 할당 경로에 끼며 tail latency 스파이크 유발 |
| NUMA가 GPU 서버에서 왜 중요? | GPU가 특정 소켓에 물리 연결 — 핀 메모리·로더를 같은 노드에 정렬해야 전송 대역폭 보존 |
