# 08 성능 분석과 트러블슈팅

허브: [[Infra]]

성능 분석에서 가장 흔한 실패는 도구가 없어서가 아니라 **순서 없이 도구를 휘두르는 것**이다. 이 노트는 Brendan Gregg 계보의 정수를 압축한다: 방법론(USE Method)으로 시작해, 자원별 도구를 익히고, 실제 장애 시나리오 5개를 플레이북으로 마무리한다. 장애 새벽 3시에 머리가 안 돌아갈 때 위에서부터 그대로 따라갈 수 있게 쓴다.

## 1. 방법론 — 도구보다 질문이 먼저

### 스트리트라이트 안티패턴

술 취한 사람이 가로등 밑에서 열쇠를 찾는다. 열쇠를 거기서 잃어버려서가 아니라 **거기가 밝아서**다. 성능 분석의 흔한 안티패턴이 정확히 이것이다 — 아는 도구(`top`)만 계속 들여다보고, 문제는 디스크 큐에 있는데 CPU 그래프만 새로고침한다. 두 번째 안티패턴은 **무작위 튜닝**: 원인 규명 없이 커널 파라미터를 하나씩 바꿔보며 "좋아진 것 같다"를 반복하는 것. 둘 다 해독제는 같다 — 체크리스트 기반 방법론.

### USE Method

모든 자원(resource)에 대해 세 가지를 순서대로 점검한다:

- **Utilization(사용률)** — 자원이 일한 시간의 비율. 예: CPU 90% busy.
- **Saturation(포화)** — 자원이 처리 못 해 **대기 큐에 쌓인 일**의 정도. 예: run queue 길이, swap 활동.
- **Errors(에러)** — 에러 카운트. 예: NIC 드롭, 디스크 I/O 에러.

핵심 통찰: **사용률 100%보다 포화가 먼저 체감 지연을 만든다.** 사용률은 "바쁘다"는 사실만 말하고, 포화는 "기다리는 일이 있다"는 사실을 말한다. 사용자가 느끼는 건 후자다. 또 에러는 종종 조용히 누적되므로(재시도로 가려짐) 명시적으로 카운터를 봐야 한다.

자원별 지표 매핑:

| 자원 | Utilization | Saturation | Errors |
|---|---|---|---|
| CPU | `mpstat -P ALL 1`의 busy %, per-core | run queue 길이(`vmstat 1`의 `r` > CPU 수) | MCE 로그(`dmesg`) |
| 메모리 | `free -h`의 available 대비 사용량 | swap in/out(`vmstat`의 `si`/`so`), OOM 발생 | OOM kill(`dmesg`), ECC 에러 |
| 디스크 | `iostat -xz 1`의 `%util` (주의사항 후술) | `await` 상승, `aqu-sz`(큐 길이) | I/O 에러(`dmesg`, smartctl) |
| 네트워크 | `sar -n DEV 1`의 처리량 vs 링크 대역폭 | NIC 송신 큐 드롭, retrans(`sar -n TCP,ETCP`) | `ethtool -S`의 errors/drops, conntrack 드롭 |

순서는 자원을 하나씩 돌며 U → S → E를 묻는다. "어느 자원이 병목인가"를 빠르게 좁히는 **소거법**이지, 근본 원인까지 주는 방법은 아니다 — 자원을 특정한 뒤 그 자원 전용 도구(perf, iostat 상세, eBPF)로 내려간다.

## 2. 60초 체크리스트 — Netflix 스타일

장애 콜 받고 서버에 SSH 접속한 직후 60초. 순서가 중요하다(싼 것부터, 넓은 것부터):

| # | 커맨드 | 무엇을 보는가 |
|---|---|---|
| 1 | `uptime` | load average 3개 — 추세(올라가는 중인가 내려가는 중인가) |
| 2 | `dmesg -T \| tail` | OOM kill, 디스크/네트워크 에러 등 커널이 남긴 마지막 비명 |
| 3 | `vmstat 1` | `r`(run queue), `si`/`so`(스왑), `us`/`sy`/`id`/`wa`, `cs`(컨텍스트 스위치) |
| 4 | `mpstat -P ALL 1` | 코어별 불균형 — 한 코어만 100%면 단일 스레드 병목 또는 IRQ 집중 |
| 5 | `pidstat 1` | 어느 프로세스가 CPU를 쓰는가 (top보다 롤링 출력이라 추적 좋음) |
| 6 | `iostat -xz 1` | 디스크별 `await`, `aqu-sz`, `%util` — I/O 병목 여부 |
| 7 | `free -h` | available 메모리 — 페이지 캐시 포함 실제 여유 |
| 8 | `sar -n DEV 1` | NIC별 rxkB/s·txkB/s — 대역폭 한계 근접 여부 |
| 9 | `sar -n TCP,ETCP 1` | active/passive(연결 생성률), `retrans/s`(재전송 = 네트워크 품질) |
| 10 | `top` | 종합 재확인 — 위에서 본 가설을 한 화면에서 교차검증 |

이 10개로 CPU/메모리/디스크/네트워크 네 자원의 U·S·E를 대략 한 바퀴 돈다. 끝나면 "병목은 X 자원이다" 가설이 서 있어야 하고, 안 서면 애플리케이션 레벨(락 경합, 외부 의존성 지연)을 의심한다.

## 3. load average 정확히 읽기

`uptime`의 세 숫자는 **1/5/15분 지수 가중 이동 평균**이다. 중요한 함정: **리눅스의 load는 CPU만의 지표가 아니다.** 실행 중/대기 중(R state) 태스크뿐 아니라 **uninterruptible sleep(D state)** 태스크까지 합산한다 — 주로 디스크 I/O나 NFS 응답을 기다리며 잠든 프로세스다. 이는 리눅스 고유의 설계 결정으로, "시스템 전체의 수요(demand)"를 표현하려는 의도다.

따라서:

- **load 높음 ≠ CPU 부족.** D state 프로세스가 쌓여도 load는 치솟는다. load 30인데 `mpstat`에서 CPU가 한가하면 거의 확실히 I/O(또는 NFS) 병목이다 — 플레이북 ⑤ 참조.
- **코어 수로 나눠 해석한다.** 16코어에서 load 12는 여유, 4코어에서 load 12는 3배 초과 수요.
- **절대값보다 추세.** 1분 > 5분 > 15분이면 악화 중, 반대면 회복 중. 1분 값만 보고 알람 걸면 노이즈에 시달린다.

면접 단골: "load average가 높은데 CPU 사용률이 낮다면?" → D state(I/O 대기) 프로세스를 의심하고 `ps -eo state,pid,comm | grep '^D'`로 수색한다고 답하면 된다.

## 4. CPU 분석

### top의 CPU 행 8개 필드

```
%Cpu(s): 12.3 us, 4.5 sy, 0.0 ni, 78.0 id, 4.0 wa, 0.1 hi, 1.0 si, 0.1 st
```

| 필드 | 의미 | 실무 해석 |
|---|---|---|
| `us` | user — 애플리케이션 코드 | 높으면 프로파일러(perf)로 핫스팟 추적 |
| `sy` | system — 커널 코드(syscall 처리 등) | 비정상적으로 높으면 syscall 폭풍·스핀락 경합 의심, [[05 커널과 시스템 콜]] |
| `ni` | nice — 우선순위 낮춘 프로세스의 user 시간 | 배치 작업 분리 확인용 |
| `id` | idle | 진짜 여유 |
| `wa` | iowait — CPU가 놀면서 I/O 완료를 기다린 시간 | **힌트일 뿐.** 다른 스레드가 CPU를 쓰면 wa는 줄어든다(idle의 부분집합). wa 낮다고 I/O 병목 없다는 증명이 아님 — 판정은 `iostat`으로 |
| `hi` | hardware interrupt | NIC 인터럽트 폭주 시 상승 |
| `si` | software interrupt(softirq) | 네트워크 패킷 처리(NET_RX)가 주범. 특정 코어만 si 높으면 IRQ 집중 |
| `st` | steal — **하이퍼바이저에게 뺏긴 시간** | 클라우드 VM에서 st가 지속적으로 보이면 noisy neighbor 또는 CPU 크레딧 소진(버스트형 인스턴스). 내 코드 잘못이 아니므로 인스턴스 교체/타입 변경이 처방 |

### 코어별로 봐야 하는 이유

`top`의 전체 평균은 거짓말을 한다. 32코어 평균 6%인데 1개 코어가 `si` 100%라면 — 그 코어에 NIC IRQ가 몰려 패킷 처리가 단일 코어 한계에 막힌 것이다. `mpstat -P ALL 1`로만 보인다. 처방은 RSS/RPS로 IRQ 분산([[04 리눅스 네트워킹]]). 마찬가지로 단일 스레드 애플리케이션 병목(한 코어만 `us` 100%)도 평균에선 안 보인다.

### 그 외

- `pidstat 1` — 프로세스별 CPU를 초 단위 롤링으로. `top`과 달리 스크롤백에 기록이 남아 "그 순간" 캡처에 유리.
- `vmstat 1`의 `cs`(context switches) — 평소 대비 폭증하면 락 경합(스레드들이 블로킹-웨이크업 반복)이나 과도한 스레드 수를 의심. `r`이 CPU 수를 지속 초과하면 CPU 포화(USE의 S).

## 5. 메모리 분석

- `free -h` — **`available`을 본다.** `free`가 작아 보여도 대부분 페이지 캐시다. 리눅스는 남는 메모리를 캐시로 쓰다가 필요하면 즉시 회수한다("unused memory is wasted memory"). `available`은 회수 가능분을 반영한 실질 여유 추정치. [[02 메모리 관리]] 참조.
- `vmstat 1`의 `si`/`so` — **스왑 in/out이 0이 아니면 적신호.** 메모리가 정말 부족해 anonymous 페이지를 디스크로 내쫓는 중이며, 성능은 절벽처럼 떨어진다. 사용률(U)보다 확실한 포화(S) 신호.
- `dmesg -T | grep -i oom` — OOM killer 발동 여부. 어떤 프로세스가 왜 죽었는지(`oom_score_adj`, 그 시점 메모리 맵) 커널 로그에 부검 기록이 남는다.
- `/proc/meminfo` — `MemAvailable`, `Dirty`(아직 디스크에 안 쓴 페이지), `Slab`(커널 자료구조) 등 세부. 컨테이너 환경에선 cgroup 한도가 따로 있으니 호스트 수치만 믿지 말 것([[06 컨테이너 내부 구조]]).
- `smem` — 공유 라이브러리를 비율로 안분한 **PSS** 기준 프로세스별 실사용량. RSS 합산은 공유 페이지를 중복 계산해 과대평가하므로, "프로세스들이 합쳐서 얼마나 쓰나"는 PSS로 답한다.
- 페이지 캐시 직관: 같은 파일 읽기가 두 번째부터 빠른 이유. 캐시 히트율이 떨어지면(working set > RAM) 디스크 I/O가 갑자기 늘어난다 — 메모리 문제가 디스크 증상으로 나타나는 전형적 패턴. `cachestat`(bcc)으로 히트율을 직접 본다.

## 6. 디스크 I/O 분석

`iostat -xz 1` 핵심 컬럼:

| 컬럼 | 의미 | 해석 |
|---|---|---|
| `r/s`, `w/s` | 초당 읽기/쓰기 요청 수(IOPS) | 디바이스 스펙 대비 |
| `rkB/s`, `wkB/s` | 초당 처리량 | 대역폭 한계 대비 |
| `await` (`r_await`/`w_await`) | **요청당 평균 지연(큐 대기 + 서비스 시간, ms)** | 애플리케이션이 체감하는 숫자. SSD에서 수십 ms면 명백한 이상 |
| `aqu-sz` | 평균 큐 길이 | 1을 크게 넘으면 포화(S) — 요청이 줄 서 있다 |
| `%util` | 디바이스가 바빴던 시간 비율 | **주의: NVMe처럼 내부 병렬성이 큰 디바이스에선 100%이 포화가 아니다.** "어느 순간이든 요청이 1개 이상 진행 중"이라는 뜻일 뿐, 수십 개를 동시에 처리할 수 있는 디바이스는 100%에서도 여유가 있다. 포화 판정은 `await`와 `aqu-sz`로 |

- `iotop` — **누가** 쓰는가. 디바이스 레벨(iostat)에서 이상을 보면 프로세스 레벨로 내려가 범인을 특정.
- **파일시스템 레벨 vs 블록 레벨 구분.** 애플리케이션이 느낀 지연은 파일시스템 레벨(페이지 캐시, 저널링, 락 포함)이고, `iostat`은 블록 레벨이다. 둘이 다를 수 있다 — 캐시 히트면 블록 I/O 자체가 안 생기고, 반대로 `fsync` 경합은 블록 지표가 멀쩡해도 앱을 멈추게 한다. 파일시스템 레벨 지연은 eBPF(`ext4slower` 등 bcc 도구)로 직접 잰다. [[03 파일시스템과 스토리지]] 참조.

## 7. 네트워크 분석

- `ss -s` — 소켓 총괄 요약. TIME-WAIT 폭증, orphan 소켓 등 한눈에.
- `sar -n DEV 1` — 인터페이스별 rxkB/s·txkB/s·rxpck/s·txpck/s. **대역폭(kB/s)만 아니라 PPS도 본다** — 작은 패킷 폭주는 대역폭 여유가 커도 CPU(softirq)를 먼저 고갈시킨다.
- `sar -n TCP,ETCP 1` — `active/s`(우리가 건 연결), `passive/s`(받은 연결)로 연결 생성률 파악, **`retrans/s`(재전송)가 핵심**. 재전송은 패킷 손실 또는 타임아웃의 결과로, 곧바로 수십~수백 ms 지연 스파이크로 나타난다. "가끔 느려요"의 단골 범인.
- `ping`/`mtr` — 경로 상 어느 홉에서 지연/손실이 생기는지. `mtr`은 traceroute + ping의 연속 측정판.
- `ethtool -S eth0` — NIC 하드웨어 통계. `rx_dropped`, ring buffer 오버플로 등 커널 위 도구들이 못 보는 드롭이 여기 있다.
- conntrack 포화 — NAT/방화벽을 거치는 모든 연결은 conntrack 테이블 엔트리를 먹는다. `conntrack -C`(현재) vs `sysctl net.netfilter.nf_conntrack_max`(한도). 가득 차면 신규 연결이 **조용히 드롭**된다 — 쿠버네티스 노드에서 의외로 자주 만나는 장애. [[04 리눅스 네트워킹]] 연결.

## 8. 프로세스 심층 분석

자원 레벨에서 프로세스를 특정했다면, 그 프로세스 내부로 들어간다.

- `strace -c -p PID` — syscall별 횟수·누적 시간 요약. 어떤 syscall에 시간이 가는지 한 방에. `-e trace=network`나 `-e trace=file`로 범위를 좁히고, `-p`로 실행 중 프로세스에 부착. **주의: ptrace 기반이라 오버헤드가 매우 크다(대상이 수십 배 느려질 수 있음). 프로덕션에서는 짧게, 혹은 eBPF 대안 사용.**
- `ltrace` — 라이브러리 함수 호출 추적(strace의 libc 버전). 마찬가지로 무겁다.
- `/proc/PID/status` — state, 메모리(VmRSS), 스레드 수. `/proc/PID/wchan` — 잠든 프로세스가 **커널 어느 함수에서** 자고 있는지. `/proc/PID/stack` — 커널 스택 전체(root 필요). D state 프로세스가 NFS에서 자는지 디스크에서 자는지 여기서 갈린다.
- `gdb -p PID` 후 `thread apply all bt` — 유저스페이스 백트레이스 한 방. 행(hang) 걸린 프로세스가 어느 락에서 멈췄는지. 단 attach 동안 프로세스가 정지하므로 프로덕션에선 신중히.
- **언어별 프로파일러로 넘어가는 기준**: 시스템 도구가 "CPU를 쓰는 건 이 프로세스의 user 코드다"까지 좁혀줬는데 perf 스택이 런타임 내부(JIT, 인터프리터 루프)로만 보인다면 — `async-profiler`(JVM), `py-spy`(Python), `pprof`(Go) 같은 런타임 인지 프로파일러로 전환한다.

## 9. perf — CPU 프로파일링의 표준

- `perf top` — 시스템 전체 실시간 핫스팟. 어떤 함수(커널 포함)가 CPU를 먹는지 즉시 확인. "일단 뭐가 도는지 보자" 단계.
- 기록 후 분석:

```bash
perf record -g -p PID -- sleep 30   # 30초간 콜스택 포함 샘플링
perf report                          # 트리 형태로 탐색
```

`-g`가 콜스택 수집 옵션. 컴파일 시 frame pointer가 생략된 바이너리는 스택이 끊겨 보일 수 있다(이때 `--call-graph dwarf` 시도).

- **플레임그래프**: `perf report`의 트리를 한 장의 그림으로. **가로 폭 = 샘플에서 차지한 비중(시간 비례), 세로 = 스택 깊이**(위가 leaf). 색은 의미 없다. 넓은 평지(plateau)를 찾으면 그게 핫스팟. 생성 흐름:

```bash
perf script | stackcollapse-perf.pl | flamegraph.pl > out.svg
```

(스크립트는 Brendan Gregg의 FlameGraph 저장소에서.)

- `perf stat <cmd>` — 하드웨어 카운터 요약: instructions, cycles, **IPC**(instructions per cycle), cache-misses. 경험칙: **IPC가 1보다 충분히 낮으면(예: 0.5 이하) CPU가 연산이 아니라 메모리를 기다리고 있을 가능성이 크다** — "CPU 100%"가 사실은 캐시 미스 대기라는 뜻. 알고리즘이 아니라 데이터 레이아웃·locality를 봐야 한다는 신호.

## 10. eBPF 도구 — 프로덕션 친화적 관측

strace류와 달리 eBPF는 커널 내부에서 검증된 바이트코드로 집계만 하고 결과를 넘기므로 **오버헤드가 낮아 프로덕션에서 상시 사용 가능**하다(원리는 [[05 커널과 시스템 콜]]). bcc 도구 모음에서 자주 쓰는 것들:

| 도구 | 용도 | 잡아내는 문제 |
|---|---|---|
| `execsnoop` | 새 프로세스 실행 추적 | **단명 프로세스** — top에 안 잡히는, 초당 수백 번 fork되는 스크립트가 CPU를 갉아먹는 패턴 |
| `opensnoop` | `open()` 호출 추적 | 어떤 파일을 누가 여는지 — 설정 파일 미스, 반복 open 낭비 |
| `biolatency` | 블록 I/O 지연 **히스토그램** | 평균(`await`)이 가리는 꼬리 지연(p99) — 이중 봉우리면 캐시/디바이스 두 경로 혼재 |
| `tcplife` | TCP 연결 수명·전송량 로그 | 누가 어디로 연결을 만들고 얼마나 주고받았나 — 커넥션 풀 누수, 의외의 외부 호출 |
| `cachestat` | 페이지 캐시 히트율 | 캐시 미스 급증 = working set이 RAM을 넘었다는 조기 경보 |

bpftrace 원라이너 감각:

```bash
# syscall을 가장 많이 부르는 프로세스 톱
bpftrace -e 'tracepoint:raw_syscalls:sys_enter { @[comm] = count(); }'
# 1초마다 초당 블록 I/O 수
bpftrace -e 'tracepoint:block:block_rq_issue { @ = count(); } interval:s:1 { print(@); clear(@); }'
```

요지: "무엇이든 계측 가능하되, 안전하게" — 기존 도구가 평균만 줄 때 eBPF는 분포(히스토그램)와 이벤트 단위 사실을 준다.

## 11. 실전 플레이북 5

### ① "서버가 느려요"

가장 막연한 신고. **증상**: 전반적 응답 지연. **진단**: 60초 체크리스트를 그대로 한 바퀴 → 어느 자원의 U/S가 튀는지로 분기. **분기와 처방**:
- CPU(`r` > 코어 수, us/sy 높음) → `perf top` → 플레임그래프 → 코드/쿼리 최적화 또는 스케일아웃.
- 디스크(`await`·`aqu-sz` 상승) → `iotop`으로 범인 프로세스 → 로그 폭주? 백업 잡? `biolatency`로 꼬리 확인.
- 메모리(`si`/`so` > 0) → 스왑 중. 메모리 누수 프로세스 특정(`smem`) → 재시작/한도 조정. 스왑이 시작된 시스템은 회복까지 한참 느리다 — 빠른 완화는 해당 프로세스 재시작.
- 다 멀쩡 → 외부 의존성(DB, 다운스트림 API) 또는 락 경합. `tcplife`·APM으로 외부 호출 지연 확인.

### ② "디스크가 가득 찼어요"

**진단 순서**: `df -h`(어느 마운트인가) → `du -xh --max-depth=1 / | sort -rh | head`(어디가 큰가, `-x`로 마운트 경계 고정) → **`du` 합계가 `df`와 안 맞으면** `lsof +L1`(삭제됐지만 프로세스가 잡고 있는 파일 — 로그 파일을 `rm` 했는데 데몬이 fd를 쥔 채라 공간이 안 돌아오는 고전 패턴). **처방**: fd를 쥔 프로세스 재시작(또는 `truncate`), 그리고 재발 방지로 logrotate 설정 + 디스크 사용률 알람. inode 고갈(`df -i`)도 같은 증상("No space left")을 낸다는 것 기억 — 작은 파일 수백만 개가 범인.

### ③ "프로세스가 죽었어요"

**진단 순서**: ① `dmesg -T | grep -i oom` — OOM kill인가? (가장 흔함) ② systemd 서비스면 `journalctl -u <svc> -e` — exit code와 마지막 로그. 137 = 128+9(SIGKILL, OOM 또는 외부 kill), 139 = 128+11(SIGSEGV). ③ 시그널사면 core dump 확인 — `coredumpctl list` 후 `coredumpctl gdb`. **분기**: OOM이면 메모리 한도(cgroup limit vs 실사용) 재산정, segfault면 코어 덤프로 개발팀행, 외부 kill이면 누가(감사 로그, 다른 오케스트레이터). 컨테이너라면 `kubectl describe pod`의 `OOMKilled` 여부가 첫 확인([[06 컨테이너 내부 구조]]).

### ④ "연결이 안 돼요"

**첫 분기가 절반이다 — refused vs timeout.**
- `Connection refused` = 패킷이 **도착했는데** 그 포트에 리스너가 없다(RST 회신). → 서버에서 `ss -ltnp | grep <port>`로 리슨 확인 → 프로세스가 안 떴거나, 127.0.0.1에만 바인드했거나, 다른 포트.
- `timeout` = 패킷이 **버려졌다**(응답 없음). → 경로상 방화벽/Security Group/NACL이 드롭 중이거나, 라우팅 문제. `mtr`로 경로, 클라우드면 SG 규칙부터.
- 둘 다 아니고 "간헐적으로" 안 되면 → conntrack 포화(`conntrack -C` vs max), SYN 백로그 오버플로(`ss -s`, `netstat -s | grep -i listen`), DNS를 의심.

### ⑤ "load는 높은데 CPU는 놀아요"

3장의 함정 그 자체. **진단**: `ps -eo state,pid,wchan:30,comm | grep '^D'` — D state 프로세스 수색. 여러 개가 같은 `wchan`에서 자고 있으면 그게 병목 지점. **분기**: wchan이 NFS 관련이면 NFS 서버/네트워크 확인(NFS 행은 D state 양산의 단골), 블록 I/O 관련이면 `iostat -xz 1`로 디바이스 상태(await 폭등? 디바이스 고장?). **처방**: I/O 대상의 회복이 우선이고, D state는 시그널을 받지 않아 kill로는 못 푸는 게 원칙이다(단, NFS 등 커널이 TASK_KILLABLE로 대기하는 경로는 `kill -9`에 반응하는 예외가 있다) — 일반적으로는 기다리던 I/O가 끝나거나 마운트를 풀어야 풀린다.

## 12. 기록 습관 — 기준선 없으면 진단도 없다

"`await` 8ms는 정상인가?"라는 질문에는 **평소 값**을 알아야만 답할 수 있다. 장애 시점 데이터만으로는 비교 대상이 없다.

- **sar 히스토리**: `sysstat` 패키지 활성화(데비안계는 `/etc/default/sysstat`의 `ENABLED="true"`) → 10분 간격 수집이 기본. `sar -q -f /var/log/sysstat/saXX`처럼 과거 날짜를 소급 조회 — "어제 새벽엔 어땠나"가 공짜로 나온다.
- **atop 로그**: atop을 데몬으로 돌리면 프로세스 단위 히스토리까지 남는다 — "그 시각에 누가 CPU를 썼나"를 사후에 재생.
- **장애 타임라인 기록**: 장애 중 친 커맨드와 출력, 시각을 그대로 남긴다(터미널 로깅이라도). 포스트모템의 원자재이자, 다음 장애 때 본인 플레이북이 된다.
- 현대적 기준선은 결국 **node_exporter + Prometheus + Grafana** 상시 수집이다 — 이 노트의 커맨드들이 재는 지표 대부분이 거기 다 있고, 장애 시 커맨드는 "대시보드가 가리킨 곳을 현장에서 확대"하는 용도가 된다. 관측성 스택은 [[AI-Infra]] 쪽에서 다룬다.

## 13. 면접 단골 정리

- **USE Method를 설명하라** → 자원별 Utilization/Saturation/Errors, 사용률보다 포화가 체감 지연을 만든다는 포인트까지.
- **load average에 무엇이 포함되나** → R + D state(리눅스 고유), 그래서 load 높음 ≠ CPU 부족.
- **%util 100%이면 디스크 포화인가** → 병렬 디바이스(NVMe)에선 아니다, `await`/`aqu-sz`로 판정.
- **wa(iowait)가 낮으면 I/O 병목이 없는 건가** → 아니다, wa는 idle의 부분집합이라 CPU가 바쁘면 가려진다.
- **strace를 프로덕션에서 써도 되나** → 오버헤드 커서 위험, eBPF(bcc/bpftrace)가 프로덕션 친화적 대안.
- **st(steal)가 높으면** → 하이퍼바이저에 뺏기는 중, noisy neighbor/크레딧 소진 — 인스턴스 차원 대응.
