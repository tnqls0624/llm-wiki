# 06 컨테이너 내부 구조
허브: [[Infra]]

> **한 문장 요약**: 컨테이너는 가상머신이 아니다. **호스트 커널을 공유하는 평범한 리눅스 프로세스**에 namespaces(보이는 것 격리) + cgroups(쓸 수 있는 양 제한) + capabilities/seccomp(할 수 있는 일 제한) + overlayfs(파일시스템 레이어)를 씌운 것이다. 이 사실 하나에서 컨테이너의 모든 장점(가벼움·빠른 기동)과 모든 함정(커널 공유 리스크·리소스 거짓말·PID 1 문제)이 따라 나온다.

## 1. VM vs 컨테이너 — 격리의 위치가 다르다

| | VM | 컨테이너 |
|---|---|---|
| 격리 주체 | 하이퍼바이저 (하드웨어 수준) | 호스트 커널의 기능 (namespaces, cgroups) |
| 커널 | 게스트마다 **자기 커널** | **호스트 커널 공유** — 게스트 커널 없음 |
| 부팅 | 펌웨어→커널→init 전체 부팅 (수십 초) | `clone()`/`fork()` 한 번 — 프로세스 시작 (수십 ms) |
| 오버헤드 | 게스트 OS 메모리 + 가상화 비용 | 프로세스 하나 수준 |
| 보안 경계 | 강함 (하이퍼바이저 탈출은 드묾) | 약함 — **커널 취약점 = 모든 컨테이너 공유 리스크** |

핵심 비유: VM은 "건물을 따로 지은 것", 컨테이너는 "한 건물 안에 칸막이를 친 것"이다. 칸막이(namespace)는 시야를 가리고, 전기·수도 계량기(cgroup)는 사용량을 제한하지만, **건물의 기둥(커널)은 모두가 공유**한다. 그래서 커널이 무너지면(커널 패닉, 커널 익스플로잇) 칸막이는 의미가 없다.

면접 단골: "컨테이너 안에서 `uname -r`을 치면 뭐가 나오는가?" → **호스트 커널 버전**이 나온다. 게스트 커널이 없으므로. 이 한 줄로 컨테이너의 본질을 이해했는지 판별한다. 같은 이유로 멀티테넌트 환경(서로 못 믿는 고객 워크로드)에서는 gVisor·Kata Containers 같은 추가 격리 계층이나 아예 VM을 쓴다.

## 2. namespaces — "무엇이 보이는가"를 격리

namespace는 커널 리소스에 대한 프로세스의 **시야**를 분리한다. 같은 namespace에 속한 프로세스끼리만 서로의 리소스가 보인다. `clone()`/`unshare()`에 플래그를 주어 생성하고, `setns()`로 기존 namespace에 합류한다. 현재 8종이다.

| Namespace | 격리 대상 | 실무 포인트 / 실습 커맨드 |
|---|---|---|
| **pid** | 프로세스 ID 공간 — 컨테이너 첫 프로세스가 **PID 1** | 컨테이너 안 `ps`엔 자기 프로세스만 보임. 실습: `sudo unshare --pid --fork --mount-proc bash` 후 `ps aux` → PID 1이 bash |
| **net** | 네트워크 스택 전체 — 인터페이스, 라우팅 테이블, iptables/nftables, 포트 공간 | 컨테이너마다 독립된 `lo`와 자기만의 80 포트. **veth pair**(가상 케이블)로 호스트 브리지에 연결 ([[04 리눅스 네트워킹]]). 실습: `sudo ip netns add test; sudo ip netns exec test ip a` |
| **mnt** | 마운트 트리 | 컨테이너의 `/`는 호스트와 다른 마운트 트리(이미지 rootfs). 최초의 namespace(역사적으로 가장 오래됨) |
| **uts** | hostname, domainname | 컨테이너마다 다른 hostname — `docker run -h` 가 바꾸는 것 |
| **ipc** | System V IPC, POSIX message queue | 공유메모리 세그먼트 격리. PostgreSQL 등 SysV 공유메모리 쓰는 앱에 영향 |
| **user** | UID/GID 매핑 — **컨테이너 root(UID 0) ≠ 호스트 root** | rootless 컨테이너의 핵심. 컨테이너 안 UID 0이 호스트에선 비특권 UID 100000 등으로 매핑. 탈출해도 권한 없음 |
| **cgroup** | cgroup 트리의 루트 시야 | 컨테이너가 자기 cgroup 서브트리만 `/`로 보게 함 — 호스트 cgroup 구조 은닉 |
| **time** | `CLOCK_MONOTONIC`, `CLOCK_BOOTTIME` 오프셋 | 가장 최근 추가(Linux 5.6). 체크포인트/복원(CRIU)용. **벽시계(CLOCK_REALTIME)는 격리 안 됨** — 컨테이너에서 시간을 바꿀 수 없는 이유 |

### 실습: unshare로 namespace 직접 체험

```bash
# 새 PID + mount namespace에서 bash 실행
sudo unshare --pid --fork --mount-proc bash
# --fork: unshare 자신이 아니라 자식이 새 pid ns의 PID 1이 되도록
# --mount-proc: 새 ns 기준으로 /proc을 다시 마운트 (안 하면 ps가 호스트 것을 보여줌)
ps aux        # PID 1 = bash. 호스트의 수천 개 프로세스가 안 보임
echo $$       # 1
```

`--mount-proc`이 필요한 이유가 중요하다. `ps`는 커널에 묻는 게 아니라 **`/proc`을 읽는다**. PID namespace만 새로 만들고 `/proc`을 다시 마운트하지 않으면 호스트의 `/proc`이 그대로 보여서 격리가 안 된 것처럼 보인다. "리소스 거짓말"(아래 9절)의 복선이기도 하다.

### 실습: nsenter — 컨테이너 디버깅 실전기

쉘도 `ss`도 없는 distroless 컨테이너의 네트워크를 디버깅해야 할 때, **호스트의 도구를 컨테이너의 namespace 안에서** 실행할 수 있다.

```bash
# 1) 컨테이너의 호스트 PID를 알아낸다
docker inspect -f '{{.State.Pid}}' mycontainer   # 예: 12345

# 2) 그 프로세스의 net namespace에만 들어가서 호스트의 ss를 실행
sudo nsenter -t 12345 -n ss -tlnp
# -n: net ns만 합류. mnt ns는 호스트 것이므로 호스트 바이너리 사용 가능

# 전부 들어가면 사실상 docker exec와 동일
sudo nsenter -t 12345 -a bash
```

면접 단골: "컨테이너에 디버깅 도구가 없을 때 어떻게 하나?" → `nsenter`로 namespace 선택 합류(또는 K8s라면 `kubectl debug`의 ephemeral container — 같은 원리다).

## 3. cgroups — "얼마나 쓸 수 있는가"를 제한

namespace가 시야라면 cgroup(control group)은 **계량기이자 차단기**다. 프로세스 그룹의 CPU·메모리·IO·프로세스 수 사용량을 측정하고 제한한다.

- **v1**: 컨트롤러(cpu, memory, …)마다 **별도의 계층 구조** — 한 프로세스가 cpu 계층과 memory 계층에서 다른 그룹에 속할 수 있어 관리가 지옥이었다.
- **v2**: **통합 단일 계층** — 현재 표준. 최신 배포판과 K8s가 기본으로 사용. `/sys/fs/cgroup` 아래 디렉토리 트리가 곧 그룹 트리이고, 파일 읽기/쓰기가 곧 API다.

### 핵심 컨트롤러 (v2 기준)

| 컨트롤러 | 핵심 파일 | 의미 |
|---|---|---|
| **cpu** | `cpu.max` | `"<quota> <period>"` — 예: `50000 100000` = 100ms마다 50ms만 실행 = **0.5 CPU**. `max`면 무제한 |
| | `cpu.weight` | 경합 시 상대적 배분 비율 (1~10000, 기본 100). **제한이 아니라 보장 비율** |
| | `cpu.stat` | `nr_throttled`, `throttled_usec` — 스로틀링 진단의 핵심 |
| **memory** | `memory.max` | 하드 리밋. 초과 + 회수 실패 → **cgroup 내 OOM kill** |
| | `memory.current` | 현재 사용량 (페이지캐시 포함 — RSS만이 아님!) |
| | `memory.events` | `oom`, `oom_kill` 카운터 — OOMKilled 사후 분석용 |
| **io** | `io.max` | 디바이스별 bps/iops 제한 |
| **pids** | `pids.max` | 프로세스/스레드 수 제한 — fork bomb 방어 |

### K8s requests/limits → cgroup 번역

K8s 매니페스트의 리소스 선언은 결국 kubelet이 cgroup 파일에 쓰는 값이다.

| K8s 선언 | cgroup v2 값 | 의미 |
|---|---|---|
| `requests.cpu: 500m` | `cpu.weight` (비례 환산) | 스케줄링 기준 + **경합 시** 보장 비율. 한가하면 더 쓸 수 있음 |
| `limits.cpu: "1"` | `cpu.max = "100000 100000"` | 하드 캡 — 한가해도 못 넘음 |
| `requests.memory` | (주로 스케줄링용) | 노드 배치 계산에 사용 |
| `limits.memory: 1Gi` | `memory.max` | 초과 시 cgroup OOM → **OOMKilled** |

### 함정 1: CPU throttling — 멀티스레드 앱의 지연 스파이크

`limits.cpu: "1"`은 "코어 1개에 묶임"이 아니라 "**100ms 주기마다 100ms어치의 CPU 시간**"이다. 16스레드 앱이 16코어에서 동시에 돌면 **6.25ms 만에 quota를 다 쓰고, 나머지 ~94ms 동안 전원이 멈춘다.** 평균 CPU 사용률은 낮은데 p99 지연이 치솟는 전형적 패턴이다.

```bash
# 컨테이너 안에서 (cgroup v2)
cat /sys/fs/cgroup/cpu.stat
# nr_throttled  ← 스로틀된 주기 수
# throttled_usec ← 스로틀로 멈춰 있던 총 시간
```

`nr_throttled`가 빠르게 증가하면: limit을 올리거나, 앱 스레드 수를 limit에 맞추거나(9절), 지연 민감 서비스라면 CPU limit 자체를 빼는 선택지를 검토한다(requests로 보장하고 limit은 미설정 — 조직 정책에 따라). 면접 단골: "CPU 사용률 40%인데 왜 느린가?" → throttling 통계부터 본다.

### 함정 2: memory limit → cgroup OOM → exit 137

컨테이너 메모리가 `memory.max`에 도달하고 회수(페이지캐시 drop 등)로도 해소가 안 되면, 커널 OOM killer가 **그 cgroup 안에서** 프로세스를 죽인다([[02 메모리 관리]]의 OOM 점수 로직이 cgroup 범위로 동작). 주 프로세스가 SIGKILL(9)을 받으므로 exit code는 **137 = 128 + 9**, K8s 상태는 `OOMKilled`.

주의: `memory.current`에는 **페이지캐시가 포함**된다. 파일 IO가 많은 앱은 힙은 작아도 limit에 빨리 닿을 수 있다(캐시는 압박 시 회수되긴 하지만, 더티 페이지 쓰기 속도가 못 따라가면 OOM). "내 앱 힙은 512Mi인데 왜 1Gi limit에서 죽지?"의 흔한 답이다.

## 4. capabilities — root 권한을 잘게 쪼개기

전통 유닉스는 "root 아니면 일반 유저"의 이분법이었다. capabilities는 root의 권한을 약 40개의 조각으로 분해해 **필요한 것만** 부여한다.

| Capability | 허용하는 것 | 메모 |
|---|---|---|
| `CAP_NET_BIND_SERVICE` | 1024 미만 포트 바인딩 | nginx가 80 포트를 여는 데 필요한 전부 |
| `CAP_NET_ADMIN` | 인터페이스·라우팅 설정 | 네트워크 디버깅 사이드카에 |
| `CAP_SYS_PTRACE` | 다른 프로세스 추적 | strace/gdb 디버깅 컨테이너에 |
| `CAP_SYS_ADMIN` | mount, swap, namespace 조작 등 잡다한 관리 작업 | **"사실상 root"** — man 페이지조차 과적재(overloaded)를 인정. 이거 주면 쪼갠 의미가 없음 |

도커 등 런타임은 기본적으로 **소수의 capability만 남기고 전부 드롭**한다. 그래서 컨테이너 안 root가 `mount`나 모듈 로딩을 못 하는 것이다. 반대로 `--privileged`는 **모든 capability 부여 + 모든 디바이스 노출 + seccomp/AppArmor 해제** — 사실상 호스트 root와 등가이며, 컨테이너 탈출이 "취약점"이 아니라 "기능"이 된다. 실무 수칙: `--privileged`를 요구하는 워크로드는 먼저 "정확히 어떤 capability가 필요한가"로 되물어 `--cap-add` 최소 집합으로 좁힌다.

## 5. seccomp / AppArmor·SELinux — 심층 방어 한 줄씩

- **seccomp**: 프로세스가 호출할 수 있는 **시스템 콜을 필터링**([[05 커널과 시스템 콜]]) — 도커 기본 프로파일은 300여 개 중 위험한 수십 개(`mount`, `kexec_load` 등)를 차단해 커널 공격면을 줄인다.
- **AppArmor/SELinux (MAC)**: 권한이 있어도 **정책상 허용된 경로/행위만** 가능하게 하는 강제 접근 제어 — capability·seccomp를 뚫어도 한 겹 더 막는다.

격리는 단일 벽이 아니라 namespace → capability → seccomp → MAC의 **겹겹의 레이어**라는 게 핵심이다.

## 6. 이미지와 레이어 — overlayfs

이미지는 마법의 디스크가 아니라 **tar로 묶인 파일시스템 레이어들의 스택 + 메타데이터(매니페스트)**다. `FROM ubuntu` 위에 `RUN apt install ...`을 하면 "그 명령으로 변경된 파일들"만 담은 레이어가 하나 쌓인다. 같은 베이스 레이어는 모든 이미지·컨테이너가 디스크에서 **공유**한다 — 이미지가 효율적인 이유.

실행 시 이 스택을 하나의 파일시스템으로 합쳐 보여주는 것이 **overlayfs**다([[03 파일시스템과 스토리지]]).

```
merged (컨테이너가 보는 통합 뷰 = 컨테이너의 /)
├── upperdir   ← 읽기/쓰기 레이어 (이 컨테이너 전용, 컨테이너당 1개)
└── lowerdir   ← 이미지 레이어들 (읽기 전용, 여러 컨테이너가 공유)
```

- **읽기**: 위에서부터 찾는다 — upper에 있으면 upper, 없으면 lower.
- **쓰기 (copy-up)**: lower의 파일을 수정하려 하면 **파일 전체를 upper로 복사한 뒤** 수정한다. 이미지에 든 10GB 파일의 1바이트를 처음 고치는 순간 10GB 복사가 일어난다 — **"큰 파일 첫 수정이 이상하게 느린" 이유**. 블록 단위가 아니라 파일 단위임에 주의.
- **삭제**: lower는 못 지우므로 upper에 whiteout 마커를 만들어 가린다. `RUN rm`을 해도 **이미지 크기는 안 줄어드는** 이유(아래 레이어는 그대로).

**컨테이너 삭제 = upperdir 소멸.** 컨테이너 안에 쓴 모든 것이 함께 사라진다. 따라서 영속 데이터(DB 파일, 업로드물)는 반드시 **볼륨/bind mount** — overlay를 우회해 호스트 파일시스템에 직접 마운트되는 경로 — 에 둔다. 면접 단골: "컨테이너 재시작했더니 데이터가 사라졌다, 왜?" 의 정답 구조가 이것이다.

## 7. 런타임 스택 — docker는 데몬, 컨테이너는 프로세스

**OCI(Open Container Initiative)** 가 표준을 정의한다: image-spec(이미지 포맷), runtime-spec(런타임이 받을 `config.json` — namespace·cgroup·mount·capability 설정의 명세), distribution-spec(레지스트리 API). 덕분에 도커로 빌드한 이미지가 containerd·CRI-O·podman 어디서든 돈다.

호출 체인:

```
docker CLI ──→ dockerd ──→ containerd ──→ containerd-shim ──→ runc
K8s kubelet ──(CRI gRPC)──→ containerd ──→ containerd-shim ──→ runc
                                                                └─ clone()으로 namespace 생성,
                                                                   cgroup 디렉토리 생성/설정,
                                                                   pivot_root로 rootfs 전환,
                                                                   capability/seccomp 적용 후
                                                                   사용자 프로세스 exec — 그리고 종료
```

- **runc**가 실제 노동자다: OCI runtime-spec의 `config.json`을 읽어 커널 기능을 조립하고 프로세스를 띄운 뒤 **자기는 사라진다**. 살아남는 건 사용자 프로세스와 그것을 지켜보는 가벼운 shim뿐.
- K8s는 dockerd를 거치지 않고 CRI로 containerd에 직접 말한다(dockershim 제거 이후). "K8s가 도커를 버렸다"는 말의 실체 — 이미지는 OCI 표준이라 아무 영향이 없었다.

검증 실습: 컨테이너는 그냥 프로세스이므로 **호스트에서 보인다.**

```bash
docker run -d --name web nginx
ps aux | grep nginx        # 호스트 ps에 nginx가 그대로 나온다 (호스트 PID로)
ls /proc/$(docker inspect -f '{{.State.Pid}}' web)/ns/   # 그 프로세스의 namespace들
```

## 8. PID 1 문제 — 컨테이너 첫 프로세스의 두 가지 의무

PID namespace 안의 첫 프로세스는 PID 1, 즉 그 세계의 **init**이 된다. init에게 커널은 두 가지 특별 취급을 한다([[01 프로세스와 스레드]]).

1. **시그널**: PID 1에게는 자신이 핸들러를 등록하지 않은 시그널의 **기본 동작이 적용되지 않는다**(커널의 init 보호). 보통 프로세스라면 SIGTERM에 죽지만, 핸들러 없는 PID 1은 SIGTERM을 **그냥 무시**한다. → `docker stop`이 SIGTERM을 보내고 10초 기다린 뒤 SIGKILL로 강제 종료하는, "내 컨테이너는 왜 항상 10초 걸려 죽지?"의 정체.
2. **좀비 reaping**: 고아가 된 자식은 PID 1에게 입양되고, PID 1이 `wait()`로 거둬야 한다. 안 거두면 좀비가 쌓여 PID(=`pids.max`)를 소모한다.

해법:
- **tini** 같은 초소형 init을 PID 1로 두고 실제 앱은 자식으로 — 시그널 전달과 reaping을 대행한다. `docker run --init`이 정확히 이것이다.
- **쉘 폼 ENTRYPOINT의 함정**: `ENTRYPOINT node app.js`(쉘 폼)는 실제로 `/bin/sh -c "node app.js"`가 PID 1이 되고, sh는 시그널을 자식에 **전달하지 않는다**. graceful shutdown이 영영 안 된다. → **exec 폼** `ENTRYPOINT ["node", "app.js"]`를 쓰면 node가 직접 PID 1이 된다(그래도 시그널 핸들러는 앱이 등록해야 함).

면접 단골: "컨테이너가 SIGTERM을 무시하고 10초 뒤 강제 종료된다, 원인 후보 둘?" → ① 쉘 폼 ENTRYPOINT ② 핸들러 없는 PID 1.

## 9. 리소스 거짓말 — /proc은 namespace를 모른다

`/proc/cpuinfo`, `/proc/meminfo`는 **cgroup을 반영하지 않는 호스트 전역 값**이다. 그래서 limit 1 CPU / 1Gi짜리 컨테이너 안에서:

```bash
nproc          # 64  ← 호스트 코어 수
free -g        # 256 ← 호스트 메모리
```

이 거짓말을 믿은 런타임이 사고를 친다:

- **JVM**: 힙 기본값을 "물리 메모리의 1/4"로 잡던 시절, 256Gi 호스트에서 64Gi 힙을 잡고 1Gi limit에 즉사(OOMKilled)하는 사고가 반복됐다. 이후 JVM이 cgroup을 직접 읽는 **컨테이너 인식**(`-XX:+UseContainerSupport`, JDK 10부터 기본·8 후기 버전에 백포트)이 도입되어 현대 JDK는 limit 기준으로 동작한다 — "JVM 컨테이너 인식의 역사"로 면접에 종종 나온다.
- **스레드풀**: `Runtime.availableProcessors()`·Go의 `GOMAXPROCS`·nginx `worker_processes auto`가 호스트 코어 수를 반환하면, 1-CPU limit에서 64개의 워커가 quota를 잘게 나눠 쓰며 **3절의 throttling을 정확히 유발**한다. 처방: 런타임의 cgroup 인식을 확인하고, 안 되면 env로 명시(예: Go 1.24 이하에 `GOMAXPROCS=1` 또는 uber의 automaxprocs 라이브러리 — Go 1.25부터는 기본값이 cgroup CPU limit을 반영한다).

진짜 값은 cgroup 파일에 있다: `/sys/fs/cgroup/cpu.max`, `/sys/fs/cgroup/memory.max`. "컨테이너 안에서 내 진짜 리소스 한도를 어떻게 아는가?"의 정답이다.

## 10. GPU 컨테이너 — namespace 바깥의 존재

GPU에는 namespace가 없다. 커널이 GPU를 가상화해 주지 않으므로, GPU 컨테이너는 격리가 아니라 **주입(injection)**으로 동작한다.

- **nvidia-container-toolkit**: 컨테이너 생성 시점에 훅으로 끼어들어 ① 디바이스 파일(`/dev/nvidia0`, `/dev/nvidiactl`, `/dev/nvidia-uvm` 등)을 컨테이너에 노출하고 ② 호스트의 드라이버 사용자공간 라이브러리(`libcuda.so` 등)를 컨테이너 안에 마운트한다. 이미지에 든 CUDA 툴킷과 **호스트 커널 드라이버의 버전 호환**이 맞아야 하는 이유가 여기서 나온다.
- **K8s device plugin**: kubelet에게 `nvidia.com/gpu` 리소스를 광고하고, 스케줄러가 GPU 단위로 파드를 배치한다. cgroup의 cpu.max 같은 **시분할 제한이 아니라 디바이스 통째 할당**이 기본이다(MIG·time-slicing은 그 위의 별도 메커니즘).

GPU 서빙 인프라([[AI-Infra]])에서 "컨테이너가 GPU를 못 본다" 트러블슈팅은 거의 항상 이 주입 체인(toolkit 설정, 드라이버 버전, device plugin 상태) 어딘가가 끊긴 것이다.

## 11. 실습 — 도커 없이 "컨테이너 비슷한 것" 만들기

원리 체화용 개략 흐름이다. 이 다섯 단계가 곧 runc가 하는 일의 뼈대다.

```bash
# 0) 재료: 어딘가에 rootfs 디렉토리 준비 (예: docker export로 추출)
mkdir rootfs && docker export $(docker create alpine) | tar -C rootfs -xf -

# 1) cgroup v2 그룹 만들고 한도 설정
sudo mkdir /sys/fs/cgroup/mybox
echo "50000 100000" | sudo tee /sys/fs/cgroup/mybox/cpu.max     # 0.5 CPU
echo "256M"         | sudo tee /sys/fs/cgroup/mybox/memory.max

# 2) namespace 격리된 쉘 시작
sudo unshare --pid --fork --mount-proc --uts --ipc --net --mount bash

# 3) (새 쉘에서) 자신을 cgroup에 등록
echo $$ > /sys/fs/cgroup/mybox/cgroup.procs

# 4) rootfs로 전환 + /proc 마운트  (실제 런타임은 chroot보다 안전한 pivot_root 사용)
chroot rootfs /bin/sh
mount -t proc proc /proc

# 5) 확인: hostname 바꿔도 호스트 영향 없음, ps는 자기들만, 메모리 256M에서 OOM
hostname mybox && ps aux
```

namespace(시야) + cgroup(한도) + rootfs 전환(파일시스템), 세 조각을 손으로 끼우면 컨테이너의 80%가 완성된다. 나머지 20%(veth 네트워킹, capability 드롭, seccomp, overlayfs 조립)가 runc와 containerd가 얹는 프로덕션 품질이다.

## 12. 한 장 요약 — 면접 답변 골격

| 질문 | 30초 답변 골격 |
|---|---|
| 컨테이너 vs VM? | 컨테이너는 호스트 커널 공유 프로세스(namespaces+cgroups), VM은 게스트 커널+하이퍼바이저. 가볍지만 커널이 공유 공격면 |
| namespace vs cgroup? | namespace = 무엇이 **보이는가**(pid·net·mnt…8종), cgroup = 얼마나 **쓰는가**(cpu.max·memory.max) |
| OOMKilled(137)? | memory.max 도달 → cgroup OOM → SIGKILL(128+9). memory.current엔 페이지캐시 포함 주의 |
| CPU 40%인데 느림? | cpu.stat의 nr_throttled 확인 — 멀티스레드가 quota 조기 소진하는 throttling 패턴 |
| 컨테이너 데이터 증발? | 쓰기는 upperdir(overlayfs), 삭제 시 함께 소멸 — 영속 데이터는 볼륨으로 |
| SIGTERM 무시? | 쉘 폼 ENTRYPOINT 또는 핸들러 없는 PID 1 — exec 폼 + tini/--init |
| 도구 없는 컨테이너 디버깅? | `nsenter -t <PID> -n`으로 호스트 도구를 대상 namespace에서 실행 |
