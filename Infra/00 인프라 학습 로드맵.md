# 00 인프라 학습 로드맵
허브: [[Infra]]

이 노트는 Infra KB 전체의 입구다. **"코딩은 잘하는데 OS 내부는 모른다"는 백엔드 개발자가, 인프라/AI-인프라 엔지니어 수준의 시스템 이해를 갖추기 위한 학습 허브**로 설계되었다. 각 노트가 무엇을 다루고, 어떤 순서로 읽고, 어떻게 공부하고, 어디까지 알면 "됐다"고 판단할지를 이 한 편에서 정한다.

## 1. 이 KB의 목적

6년차 백엔드 개발자라면 API 서버를 짜고, 분산 시스템의 일관성 모델을 논하고, 메시지 큐로 트래픽을 받아낼 수 있다. 그런데 정작 그 코드가 **어느 컴퓨터의, 어떤 커널 위에서, 어떻게 실행되는가**를 물으면 답이 흐려지는 경우가 많다. 프로세스가 정확히 뭔지, 메모리의 RSS가 왜 코드에서 할당한 양과 다른지, 커넥션이 왜 `TIME_WAIT`에 쌓이는지 — 이 층위를 모르면 장애 상황에서 추측에 의존하게 된다.

인프라 엔지니어와 백엔드 개발자의 결정적 차이가 여기에 있다. 백엔드 개발자는 추상화 **위에서** 일하고, 인프라 엔지니어는 추상화가 **새는 지점에서** 일한다. 컨테이너의 OOM kill, 디스크 I/O 병목, 커널 파라미터 튜닝, GPU 노드의 NUMA 배치 — 전부 OS 지식이 곧 무기인 영역이다.

이 KB는 그 토대를 8개의 본문 노트로 체계화한다:

| 노트 | 한 줄 정의 | 성격 |
|---|---|---|
| [[01 프로세스와 스레드]] | 실행의 단위 — fork/exec, 스케줄링, 시그널, 좀비 | 개념 체인 1 |
| [[02 메모리 관리]] | 가상 메모리, 페이지, RSS/VSZ, OOM killer, swap | 개념 체인 2 |
| [[03 파일시스템과 스토리지]] | VFS, inode, 페이지 캐시, I/O 스택, ext4/xfs | 개념 체인 3 |
| [[04 리눅스 네트워킹]] | 소켓, TCP 상태 머신, netfilter, 커널 네트워크 스택 | 개념 체인 4 |
| [[05 커널과 시스템 콜]] | 유저/커널 모드, syscall 경로, 인터럽트, /proc·/sys | 개념 체인 5 |
| [[06 컨테이너 내부 구조]] | namespace, cgroup, 컨테이너 = 격리된 프로세스 | 개념 체인 6 (종합) |
| [[07 실무 리눅스 커맨드]] | 관찰·조작 도구 모음 — ps, ss, vmstat, strace 등 | 사전 (수시 참조) |
| [[08 성능 분석과 트러블슈팅]] | USE 방법론, 병목 탐색, 장애 시나리오별 진단 루트 | 사전 (수시 참조) |

여기에 [[99 인프라 학습 리소스]]가 외부 교재·강의·블로그(OSTEP, TLPI, Brendan Gregg 등)를 정리한 부록으로 붙는다.

### AI-Infra와의 관계

이 KB는 [[AI-Infra]] KB(GPU 서빙, vLLM, Kubernetes, KServe)의 **하부 레이어**다. 관계를 명확히 하면:

- Kubernetes의 Pod 리소스 제한은 결국 **cgroup**이다 → [[06 컨테이너 내부 구조]]를 모르면 K8s OOMKilled를 표면적으로만 이해하게 된다.
- GPU 서빙의 데이터 로딩 병목은 결국 **페이지 캐시와 I/O 스택**의 문제다 → [[03 파일시스템과 스토리지]].
- vLLM의 PagedAttention이라는 이름 자체가 OS의 **페이징** 개념을 차용한 것이다 → [[02 메모리 관리]]를 알면 설계 의도가 한 번에 읽힌다.
- 노드 간 통신, 서비스 메시, CNI는 전부 **리눅스 네트워크 스택** 위의 구조물이다 → [[04 리눅스 네트워킹]].

즉 "K8s를 배우다 막히면 이 KB로 내려오고, 이 KB를 끝내면 AI-Infra로 올라간다"는 양방향 사다리를 의도했다.

## 2. 읽는 순서 — 왜 이 순서인가

01→06은 **개념 체인**이다. 임의 순서로 읽어도 되는 백과사전이 아니라, 앞 노트가 뒤 노트의 어휘를 공급하는 누적 구조다:

```text
01 프로세스 ──> 02 메모리 ──> 03 파일시스템 ──> 04 네트워킹 ──> 05 커널 ──> 06 컨테이너
   (실행 단위)    (그 단위가      (데이터가 머무는    (데이터가        (이 모든 걸     (이 모든 걸
                  차지하는 공간)    영속 공간)         흐르는 통로)      중재하는 자)     묶어 격리)
```

- **01이 처음인 이유**: 리눅스에서 거의 모든 것은 프로세스의 관점에서 기술된다. `/proc/<pid>/` 아래에 메모리 맵, 열린 파일, 네트워크 소켓이 전부 매달려 있다. 프로세스를 모르면 나머지를 서술할 언어가 없다.
- **02가 그다음인 이유**: 프로세스의 정체는 사실상 "자기만의 가상 주소 공간"이다. 01에서 만든 fork의 의미(주소 공간 복제, copy-on-write)는 02를 읽어야 완성된다.
- **03·04**: 메모리 위의 데이터가 디스크로(파일시스템), 네트워크로(소켓) 나가는 두 갈래 출구. 페이지 캐시가 둘을 잇는다 — "파일 읽기는 사실 메모리 읽기"라는 통찰이 여기서 나온다.
- **05**: 앞의 네 영역에서 반복 등장한 "커널이 해준다"의 실체. 시스템 콜이라는 단일 관문, 유저/커널 모드 전환 비용, 인터럽트를 여기서 한 번에 정리한다.
- **06이 마지막인 이유**: 컨테이너는 새 기술이 아니라 **01~05의 종합 응용**이다. namespace(프로세스·네트워크·마운트의 시야 격리) + cgroup(CPU·메모리의 양 제한) = 컨테이너. 앞을 다 읽고 오면 "Docker는 마법이 아니라 syscall 묶음"임이 자명해진다.

07·08은 체인이 아니라 **사전(reference)**이다. 01을 읽다가 `ps` 옵션이 궁금하면 [[07 실무 리눅스 커맨드]]를 펴고, 장애 시나리오를 연습하고 싶으면 [[08 성능 분석과 트러블슈팅]]을 편다. 처음부터 끝까지 통독하는 노트가 아니다 — 다만 8주 플랜의 후반부에 한 번씩 정독 주간을 배치했다.

## 3. 학습 방법론 — 읽기만 하면 안 남는다

이 분야의 함정은 "읽으면 이해되는데, 일주일 뒤에 설명을 못 한다"는 것이다. OS 지식은 선언적 지식이 아니라 **관찰 가능한 현상**이라서, 터미널에서 직접 재현한 것만 장기 기억에 남는다. 모든 노트를 다음 4단계 루프로 소화하라:

```text
① 개념 읽기 ──> ② 커맨드로 직접 관찰 ──> ③ 장애 시나리오와 연결 ──> ④ 면접 질문으로 자기 점검
   (노트 1편)      (/proc, ss, vmstat...)     (이게 터지면 어떻게 보이나)    (소리 내서 2분 설명)
```

**① 개념 읽기** — 노트 한 편을 읽되, "정의 암기"가 아니라 "왜 그렇게 설계됐나"를 자문하며 읽는다. 예: 좀비 프로세스는 버그가 아니라 "부모가 자식의 종료 코드를 회수할 권리를 보장하는 설계"다.

**② 직접 관찰** — 읽은 개념을 즉시 터미널에서 확인한다. 리눅스의 미덕은 커널 상태가 거의 전부 파일로 노출된다는 점이다:

```bash
# 프로세스: 내 셸의 정체를 들여다보기
cat /proc/$$/status        # 상태, 메모리, 스레드 수
ls -l /proc/$$/fd          # 열린 파일 디스크립터
cat /proc/$$/cgroup        # 어느 cgroup에 속하는가

# 메모리: 시스템 전체와 프로세스별
vmstat 1 5                 # 1초 간격 5회 — si/so(스왑), free, cache
cat /proc/meminfo | head -20

# 네트워크: 소켓 상태 분포
ss -s                      # TCP 상태별 카운트 요약
ss -tn state time-wait | head   # TIME_WAIT 소켓은 소유 프로세스가 없어 -p는 무의미

# 커널: 시스템 콜을 눈으로 보기
strace -c ls /tmp          # ls 한 번에 어떤 syscall이 몇 번 불리나
```

**③ 장애 시나리오와 연결** — "이 개념이 무너지면 운영 환경에서 어떤 증상으로 나타나는가"를 한 문장으로 만들어 본다. 예: "fd leak → `ls /proc/<pid>/fd | wc -l` 증가 → `Too many open files` → 신규 커넥션 수락 불가". 이 인과 사슬이 곧 트러블슈팅 능력이고, [[08 성능 분석과 트러블슈팅]]이 시나리오 창고 역할을 한다.

**④ 자기 점검** — 아래 6절의 면접 질문을 보고, 노트를 덮은 채 소리 내어 2분간 설명한다. 막히는 지점이 곧 다음 복습 지점이다. 페인만 기법의 인프라 버전이라고 생각하면 된다.

비유하자면: 노트 읽기는 해부학 교과서, 커맨드 관찰은 실제 해부 실습이다. 교과서만 본 의사에게 수술을 맡기지 않듯, `vmstat`을 한 번도 안 돌려본 사람의 메모리 지식은 면접관이 두 번째 꼬리 질문에서 바로 간파한다.

## 4. 8주 학습 플랜

주당 5~7시간(평일 1시간 + 주말 핸즈온) 기준. 핸즈온은 로컬 리눅스 VM이나 컨테이너(Docker/Lima 등)에서 수행한다 — macOS 호스트에서는 `/proc` 관찰이 불가능하므로 리눅스 환경을 반드시 하나 마련할 것.

| 주차 | 노트 | 핸즈온 과제 |
|---|---|---|
| 1주 | [[01 프로세스와 스레드]] | C나 Python으로 `fork()` 호출 후 부모가 `wait()` 하지 않는 코드를 짜서 **좀비를 직접 생성**, `ps aux \| grep Z`와 `/proc/<pid>/status`의 `State: Z`로 확인. `kill -STOP`/`-CONT`로 상태 전이 관찰. 데몬 프로세스의 PPID가 1(또는 systemd 유저 인스턴스)이 되는 과정 추적 |
| 2주 | [[02 메모리 관리]] | 메모리를 무한 할당하는 스크립트로 **OOM killer를 의도적으로 발동**시키고 `dmesg` / `journalctl -k`에서 oom-killer 로그(점수, 희생자 선정) 분석. 같은 프로세스의 `VSZ`와 `RSS`를 비교하고 차이를 설명하는 글 한 단락 작성 |
| 3주 | [[03 파일시스템과 스토리지]] | `dd`로 큰 파일을 만든 뒤 첫 읽기/재읽기 시간 차이를 측정해 **페이지 캐시 효과 체감** (`echo 3 | sudo tee /proc/sys/vm/drop_caches`로 캐시 비우고 재측정 — `sudo echo 3 > …` 형태는 리다이렉션이 유저 셸 권한으로 실행돼 실패한다). 파일을 연 채 삭제해도 디스크가 안 비는 현상(deleted-but-open)을 `lsof +L1`로 재현 |
| 4주 | [[04 리눅스 네트워킹]] | `python3 -m http.server`를 띄우고 `curl` 요청을 보내며 `ss -tnp`로 **TCP 상태 전이 추적**(LISTEN → ESTABLISHED → TIME_WAIT). `tcpdump -i lo port 8000`으로 3-way handshake 패킷을 직접 캡처해 SYN/SYN-ACK/ACK 확인 |
| 5주 | [[05 커널과 시스템 콜]] | `strace -f -e trace=network curl http://example.com`으로 HTTP 요청 한 번의 **syscall 시퀀스 전체를 해부**(socket → connect → sendto → recvfrom; `close`는 network 클래스가 아니라 이 필터에 안 잡힌다). `strace -c`로 syscall 횟수·소요 시간 통계 비교 |
| 6주 | [[06 컨테이너 내부 구조]] | Docker 없이 `unshare --pid --net --mount --fork bash`로 **맨손 컨테이너 만들기**. 호스트에서 `docker run --memory=64m`로 띄운 컨테이너의 cgroup 파일(`memory.max` 등)을 직접 읽고, 컨테이너 안 프로세스를 호스트의 `ps`에서 찾아 "컨테이너 = 프로세스"를 확인 |
| 7주 | [[07 실무 리눅스 커맨드]] 정독 | 자기만의 **장애 첫 5분 치트시트** 작성: 접속 직후 순서대로 칠 커맨드 10개를 고르고 각각 "무엇을 배제하기 위해 보는가"를 한 줄씩 단다. 평소 쓰는 서버(또는 VM)에 적용해 베이스라인 수치를 기록 |
| 8주 | [[08 성능 분석과 트러블슈팅]] 정독 | `stress-ng`(또는 `dd`, `yes`)로 CPU·메모리·I/O 부하를 각각 인위 주입하고, USE 방법론대로 `vmstat`/`iostat`/`pidstat`만 보고 **어느 자원이 병목인지 블라인드 진단**하는 셀프 모의 장애 훈련. 면접 질문 전체(아래 표) 음성 답변 녹음 후 셀프 리뷰 |

8주를 마치면 [[99 인프라 학습 리소스]]에서 OSTEP(이론 심화)·TLPI(시스템 프로그래밍)·Brendan Gregg(성능 분석)로 가지를 뻗는다. 이 KB가 지도, 그 책들이 영토다.

## 5. 면접 단골 주제 매핑

인프라/SRE/플랫폼 직군 면접에서 반복 출제되는 질문을 노트별로 매핑했다. ④ 자기 점검 단계에서 이 표를 체크리스트로 쓴다.

| 노트 | 대표 면접 질문 |
|---|---|
| [[01 프로세스와 스레드]] | • 좀비 프로세스란 무엇이고 왜 생기나? PID 1의 역할은? • 프로세스와 스레드의 차이를 커널 관점(공유하는 것 vs 안 하는 것)에서 설명하라 • `kill -9`와 `kill -15`의 차이는? 왜 -9를 함부로 쓰면 안 되나? |

## 6. 셀프 체크리스트 — 설명할 수 있어야 하는 것

노트를 덮은 상태에서, 각 항목을 화이트보드에 그림 그려가며 2분 설명할 수 있으면 체크. 15개 이상 체크되면 이 KB는 졸업이다.

- [ ] `fork()` → `exec()` 흐름과, 셸이 명령을 실행하는 과정
- [ ] 좀비/고아 프로세스의 생성 원리와 PID 1의 reaping 역할
- [ ] 프로세스 상태(R/S/D/Z/T) 각각의 의미 — 특히 D state가 위험한 이유
- [ ] 시그널 전달 메커니즘과 SIGTERM/SIGKILL/SIGSTOP의 차이
- [ ] 가상 주소 공간의 구조(text/data/heap/stack/mmap)와 페이지 테이블의 역할
- [ ] RSS·VSZ·shared memory의 차이, 그리고 copy-on-write
- [ ] 페이지 폴트(minor vs major)와 swap in/out이 성능에 미치는 영향
- [ ] OOM killer의 동작 조건과 oom_score 개념
- [ ] inode·dentry·파일 디스크립터의 관계 — "파일을 연다"의 실체
- [ ] 페이지 캐시와 dirty page writeback — `write()` 리턴 ≠ 디스크 기록
- [ ] 소켓 생성부터 `accept()`까지의 서버 측 흐름과 backlog 큐 2개(SYN/accept)
- [ ] TCP 상태 머신 — 특히 TIME_WAIT과 CLOSE_WAIT의 발생 위치와 책임 소재
- [ ] 시스템 콜의 호출 경로(유저 모드 → 트랩 → 커널 모드)와 비용
- [ ] `/proc`와 `/sys`가 무엇이고, 커널 상태를 어떻게 노출하는가
- [ ] 컨텍스트 스위치의 비용 구성(레지스터, TLB, 캐시)과 과다 발생 징후
- [ ] namespace 종류(pid, net, mnt, uts, ipc, user)와 각각의 격리 대상
- [ ] cgroup의 CPU/메모리 제한 동작과 K8s requests/limits의 연결 고리
- [ ] "컨테이너는 격리된 프로세스다"를 호스트의 ps 출력으로 증명하는 방법
- [ ] load average의 정의(리눅스는 D state 포함)와 코어 수 대비 해석
- [ ] USE 방법론(Utilization·Saturation·Errors)으로 병목을 좁히는 절차

## 7. 이 vault의 학습 루프와 연결

이 KB는 정적인 문서 더미가 아니라 vault의 자동화 루프 안에서 살아 있는 구조다:

- **복습**: 궁금한 게 생기면 노트를 직접 뒤지기 전에 `/kb-query`로 질문한다 — KB에서 답을 찾아주고, 답이 없으면 그 사실 자체가 KB의 빈틈을 알려준다. "CLOSE_WAIT 누가 잘못이더라?" 수준의 단발 질문에 적합하다.
- **박제**: 핸즈온 중 새로 알게 된 것(예: 특정 배포판의 cgroup v2 경로 차이), 외부에서 읽은 좋은 자료, 실무 장애에서 얻은 교훈은 `/kb-ingest`(외부 자료)나 `/kb-save`(세션 대화)로 노트화한다. 8주 플랜의 핸즈온 결과물을 박제하면 KB가 "내 경험이 묻은 교과서"로 진화한다.
- **건강 관리**: 노트를 추가·수정하면 `python3 .claude/kb-lint.py`로 frontmatter와 위키링크 무결성을 점검한다. 노트 갱신 시에는 vault 규칙의 3단계 의무(updated bump → MOC 반영 → hot.md 한 줄)를 따른다.

학습 루프를 한 문장으로: **읽고([[Infra]] 노트) → 재현하고(터미널) → 막히면 묻고(/kb-query) → 새로 알면 박제한다(/kb-ingest)**. 이 사이클이 돌기 시작하면 KB는 읽는 자료에서 쓰는 도구로 바뀐다.
