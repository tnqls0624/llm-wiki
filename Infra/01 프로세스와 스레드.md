# 01 프로세스와 스레드

허브: [[Infra]]

분산시스템을 다뤄온 백엔드 개발자에게 프로세스는 "JVM이 하나 떠 있는 것" 정도로 추상화돼 있기 쉽다. 하지만 인프라 엔지니어의 관점은 한 층 아래에서 시작한다. **프로세스는 커널이 자원을 회계(accounting)하는 단위**이고, 스레드는 그 회계 장부를 공유하는 실행 흐름이다. 이 노트는 커널이 프로세스를 어떻게 표현·생성·종료·스케줄링하는지, 그리고 운영 현장에서 좀비·D state·graceful shutdown 같은 문제를 어떻게 다루는지를 정리한다.

---

## 1. 프로세스란 — 커널의 회계 단위

커널 내부에서 프로세스 하나는 `task_struct`라는 C 구조체 하나로 표현된다. 교과서에서 말하는 PCB(Process Control Block)의 리눅스 구현체다. 이 안에 PID, 상태, 메모리 맵(`mm_struct` 포인터), 열린 파일 테이블, 시그널 핸들러, 스케줄링 정보, 자격 증명(uid/gid)이 전부 들어 있다. **"프로세스를 죽인다/만든다"는 말은 결국 커널이 task_struct를 해제/할당한다는 뜻**이라고 기억하면 많은 동작이 자연스럽게 설명된다.

프로세스를 식별하는 ID는 하나가 아니다:

| ID | 의미 | 실무에서 마주치는 곳 |
|---|---|---|
| PID | 프로세스 고유 번호 | `kill <pid>`, `/proc/<pid>/` |
| PPID | 부모 PID | 좀비 발생 시 "누구 책임인가" 추적 |
| PGID | 프로세스 그룹 — 보통 파이프라인 하나가 한 그룹 | `kill -- -<pgid>`로 그룹 전체에 시그널 |
| SID | 세션 — 보통 로그인 셸 하나가 한 세션 | 데몬화(setsid), 터미널 끊김(SIGHUP) 범위 |

이 계층(세션 ⊃ 프로세스 그룹 ⊃ 프로세스)이 존재하는 이유는 **시그널을 묶음 단위로 배달하기 위해서**다. 터미널에서 `Ctrl+C`를 누르면 SIGINT가 포그라운드 프로세스 그룹 전체에 가고, SSH 세션이 끊기면 SIGHUP이 세션 단위로 퍼진다.

### /proc/[pid]/ 둘러보기

`/proc`은 커널이 task_struct를 파일 형태로 노출한 가상 파일시스템이다. 디버깅의 출발점은 항상 여기다:

```bash
cat /proc/1234/status      # 상태(State), VmRSS, Threads 수, Uid 등 사람이 읽는 요약
cat /proc/1234/cmdline | tr '\0' ' '   # 실제 실행 커맨드라인 (NUL 구분)
ls -l /proc/1234/fd        # 열린 파일 디스크립터 — 소켓, 파일, 파이프가 다 보임
cat /proc/1234/limits      # ulimit이 실제로 어떻게 적용됐는지
ls -l /proc/1234/exe       # 실행 중인 바이너리의 실제 경로 (삭제됐으면 "(deleted)")
```

> 면접 단골: "ps 없이 프로세스 정보를 보려면?" — `ps` 자체가 `/proc`을 읽어서 출력하는 도구다. 컨테이너처럼 도구가 없는 환경에서 `/proc`을 직접 읽는 능력은 실무 변별력이 크다.

## 2. 프로세스 생성 — fork()와 exec()

유닉스의 프로세스 생성은 "새로 만든다"가 아니라 **"자신을 복제한 뒤(fork) 다른 프로그램으로 갈아탄다(exec)"**는 2단계다.

### fork()와 copy-on-write

`fork()`는 호출한 프로세스의 복제본을 만든다. 여기서 핵심은 **메모리를 즉시 복사하지 않는다**는 것. 커널은 부모의 페이지 테이블만 복사하고 쓰기 가능한 private 페이지들을 읽기 전용으로 표시한다(공유 매핑(MAP_SHARED)은 CoW 대상이 아니다). 부모나 자식 중 하나가 그 페이지에 **쓰기를 시도하는 순간** page fault가 발생하고, 그때서야 해당 페이지 하나만 복사한다. 이것이 copy-on-write(CoW)다.

왜 이렇게 설계됐나? fork 직후 대부분의 자식은 곧바로 `exec()`을 호출해 메모리를 통째로 버리기 때문이다. 10GB짜리 JVM이 `fork()`를 해도 즉시 10GB가 복사되지 않는 이유이고, 반대로 **Redis가 BGSAVE(fork 기반 스냅샷) 중에 쓰기가 몰리면 CoW로 인해 메모리 사용량이 치솟는** 이유이기도 하다 — 부모가 페이지를 건드릴 때마다 복사가 일어나니까. 이 메커니즘의 메모리 측면은 [[02 메모리 관리]]에서 페이지 테이블과 함께 더 깊게 다룬다.

```c
pid_t pid = fork();
if (pid == 0) {
    // 자식: fork()가 0을 반환
    execvp("ls", (char *[]){"ls", "-l", NULL});  // 성공하면 돌아오지 않음
    _exit(127);                                   // exec 실패 시에만 도달
} else if (pid > 0) {
    // 부모: fork()가 자식의 PID를 반환
    waitpid(pid, &status, 0);
}
```

`fork()`가 **한 번 호출되고 두 번 반환된다**(부모에게는 자식 PID, 자식에게는 0)는 점이 면접 단골이다.

### exec() 패밀리

`exec()`은 현재 프로세스의 주소공간을 새 프로그램으로 **완전히 교체**한다. PID는 그대로 유지된다. `execl/execlp/execv/execvp/execve` 등 변형은 인자 전달 방식(list vs vector)과 PATH 탐색 여부(p 접미사)의 조합일 뿐, 최종적으로는 모두 `execve(2)` 시스템 콜로 수렴한다. 셸이 커맨드를 실행하는 방식이 정확히 fork + exec 패턴이다: 셸이 fork하고, 자식이 exec으로 명령을 실행하고, 부모(셸)는 wait한다.

한 줄 보충: `vfork()`는 CoW조차 아끼려고 부모를 정지시키는 옛 최적화로 지금은 쓸 이유가 거의 없고, `posix_spawn()`은 fork+exec을 한 번에 묶은 라이브러리 인터페이스로 대형 프로세스에서 자식을 띄울 때 더 안전하고 효율적이다.

## 3. 종료와 회수 — zombie는 이미 죽어 있다

프로세스가 `exit()`하면 커널은 메모리·fd 등 자원을 즉시 해제하지만, **task_struct의 일부(종료 코드, 자원 사용 통계)는 남겨둔다**. 부모가 `wait()`/`waitpid()`로 그 종료 정보를 수거해 가야 비로소 task_struct가 완전히 사라진다. 이 "죽었지만 부모가 아직 수거하지 않은" 상태가 zombie(Z)다.

좀비에 대해 정확히 이해해야 할 것:

- **좀비는 이미 죽은 프로세스다.** CPU도 메모리도 거의 안 쓴다. 남은 건 종료 코드를 담은 메타데이터 한 줌뿐이다.
- **좀비에게 `kill -9`는 의미가 없다.** 이미 죽은 것을 또 죽일 수 없다. 시그널을 받아 처리할 실행 흐름 자체가 없다.
- **좀비는 부모의 버그다.** 부모가 `wait()`를 호출하지 않아서(SIGCHLD를 무시하거나 핸들러에서 reap을 빠뜨려서) 생긴다. 고치려면 좀비가 아니라 **부모를 손봐야** 한다.
- 좀비가 위험한 이유는 자원이 아니라 **PID 고갈**이다. 커널의 PID 공간(`/proc/sys/kernel/pid_max`)은 유한하고, 좀비도 PID를 점유한다.

부모가 먼저 죽으면 자식은 orphan이 되고, 커널이 PID 1(전통적으로 init, 현대에는 systemd)에게 입양시킨다. PID 1은 입양한 자식들을 성실하게 `wait()`해주는 것이 본분이다 — 이를 reaping이라 부른다.

**컨테이너에서 이 본분이 깨진다.** 컨테이너의 PID 1은 보통 애플리케이션 자신인데, 일반 애플리케이션은 입양된 자식을 reap할 줄 모른다. 자식 프로세스를 만드는 워크로드(셸 스크립트, 헤드리스 브라우저 등)에서 좀비가 누적되는 전형적 패턴이다. 해법은 `tini`나 `docker run --init`처럼 reaping을 해주는 경량 init을 PID 1로 두는 것 — 자세한 구조는 [[06 컨테이너 내부 구조]] 참고.

## 4. 프로세스 상태 — R, S, D, Z, T

`ps`의 STAT 컬럼과 `/proc/<pid>/status`의 State 줄에 나오는 상태들:

| 상태 | 이름 | 의미 | 실무 포인트 |
|---|---|---|---|
| R | running / runnable | CPU에서 실행 중이거나 런큐에서 대기 중 | R이 많은데 느리면 CPU 경합 |
| S | interruptible sleep | 이벤트(소켓 데이터, 타이머 등) 대기, 시그널로 깨울 수 있음 | 대부분의 프로세스가 평소 이 상태 — 정상 |
| D | uninterruptible sleep | 커널 내부에서 I/O 완료를 대기, **시그널로 깨울 수 없음** | `kill -9`도 안 먹음. NFS 행, 디스크 장애의 신호 |
| Z | zombie | 종료됐고 부모의 wait 대기 중 | kill 불가, 부모를 고쳐라 (3장) |
| T | stopped | SIGSTOP/SIGTSTP로 정지됨 | `Ctrl+Z` (디버거 추적 정지는 현대 `ps`에서 소문자 `t`로 따로 표시) |

### D state — kill -9가 안 먹는 유일한 살아있는 상태

D state는 프로세스가 커널 코드 깊숙한 곳(보통 디스크 I/O나 NFS 응답 대기)에 들어가 있어, **여기서 중단시키면 커널 자료구조가 깨질 수 있기 때문에** 시그널 배달 자체를 보류하는 상태다. SIGKILL조차 "프로세스가 커널에서 안전한 지점으로 돌아온 뒤"에야 처리된다. 그 안전한 지점이 영영 오지 않으면(NFS 서버가 죽었거나 디스크가 맛이 갔거나) 프로세스는 영원히 D에 머문다. 보완: 커널 2.6.25부터 많은 대기 지점이 TASK_KILLABLE(SIGKILL만은 받는 uninterruptible)로 구현됐다 — `ps`에는 똑같이 D로 보이므로, D라고 해서 전부 SIGKILL이 안 통하는 것은 아니다.

**함정: 리눅스의 load average는 R뿐 아니라 D 상태 태스크도 포함한다.** CPU는 한가한데 load가 치솟는다면, 십중팔구 D state 프로세스 무리가 스토리지나 NFS를 기다리고 있는 것이다. CPU 문제로 오진하기 쉬운 대표 사례 — 진단 흐름은 [[08 성능 분석과 트러블슈팅]]에서 이어진다.

```bash
ps -eo pid,stat,wchan:32,cmd | awk '$2 ~ /D/'   # D state 프로세스와 커널 대기 지점(wchan)
```

`wchan`은 프로세스가 커널의 어느 함수에서 잠들어 있는지를 보여줘 원인 추정(NFS인지 블록 I/O인지)에 유용하다.

## 5. 스레드 — 커널에게는 그냥 "공유를 많이 하는 task"

커널은 프로세스와 스레드를 별개 개념으로 구분하지 않는다. 둘 다 task_struct 하나씩이다. 차이는 생성 시 `clone()` 시스템 콜에 넘긴 **공유 플래그**뿐이다:

- `fork()` ≈ `clone()`에 거의 아무것도 공유하지 않는 플래그 → 새 프로세스
- `pthread_create()` ≈ `clone(CLONE_VM | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD ...)` → 주소공간·fd·시그널 핸들러를 공유하는 task = 스레드

같은 프로세스의 스레드들은 **TGID(thread group ID)**를 공유한다. 우리가 흔히 "PID"라 부르는 것은 사실 TGID이고, 각 스레드는 자기만의 task ID(커널 관점의 pid, 도구에서는 LWP/TID/SPID로 표기)를 따로 가진다. `getpid()`는 TGID를, `gettid()`는 자기 TID를 반환한다.

```bash
ps -eLf | head            # LWP 컬럼 = 스레드 ID, NLWP = 스레드 수
ls /proc/1234/task/       # 프로세스 1234의 스레드들이 디렉토리로 나열됨
cat /proc/1234/status | grep Threads
top -H -p 1234            # 스레드 단위로 CPU 사용량 보기 — "어느 스레드가 먹나" 추적
```

### 무엇을 공유하고 무엇이 따로인가

| 자원 | 스레드 간 | 프로세스 간 (fork 직후) |
|---|---|---|
| 가상 주소공간 (힙, 전역 변수) | **공유** | CoW 분리 |
| 파일 디스크립터 테이블 | **공유** | 복사본 (같은 open file은 가리킴) |
| 시그널 핸들러 (disposition) | **공유** | 복사본 |
| 스택 | 각자 | 각자 (CoW) |
| 레지스터·프로그램 카운터 | 각자 | 각자 |
| PID(=TGID) | 같음 | 다름 |
| TID | 다름 | — |

이 표가 "스레드는 가볍고 프로세스는 무겁다"의 실체다. 스레드 간 통신이 빠른 이유(같은 주소공간), 한 스레드의 segfault가 프로세스 전체를 죽이는 이유(시그널과 주소공간 공유), 멀티프로세스 모델(예: gunicorn prefork)이 격리에 유리한 이유가 전부 이 표에서 나온다.

한 줄 보충: 현대 리눅스의 pthread 구현은 NPTL(Native POSIX Thread Library)로, 유저 스레드 1개 = 커널 task 1개의 1:1 모델이다.

## 6. 스케줄링 — CFS에서 EEVDF로

리눅스의 일반(non-realtime) 스케줄러는 오랫동안 CFS(Completely Fair Scheduler)였다. 핵심 아이디어는 각 태스크의 **vruntime(가중 누적 실행 시간)**을 기록하고, 항상 vruntime이 가장 작은(=지금까지 가장 적게 실행된) 태스크를 다음에 실행하는 것이다. 타임슬라이스를 고정 배분하는 대신 "공정한 몫"을 추적한다는 점이 이름의 유래다. **커널 6.6부터는 EEVDF(Earliest Eligible Virtual Deadline First)가 CFS를 대체**했다 — 공정성 모델은 계승하되 가상 데드라인 개념을 추가해 지연(latency)에 민감한 태스크의 응답성을 개선한 설계다. 면접에서 "리눅스 스케줄러?"라고 물으면 "전통적으로 CFS, 6.6부터 EEVDF"가 현재 시점의 정답이다.

### nice와 priority

`nice` 값(-20 ~ 19, 기본 0)은 vruntime의 누적 **가중치**를 조절한다. nice가 낮을수록(공격적일수록) 같은 실제 시간을 실행해도 vruntime이 천천히 쌓여 CPU를 더 받는다. "nice하다 = 남에게 양보한다"로 기억하면 방향이 안 헷갈린다.

```bash
nice -n 10 ./batch-job.sh      # 양보하며 시작
renice -n 5 -p 1234            # 실행 중 조정 (낮추는 건 root 필요)
```

### 실시간 정책 — 한 줄과 경고

`SCHED_FIFO`/`SCHED_RR`은 실시간 정책으로, **어떤 일반 태스크보다도 무조건 먼저** 실행된다. FIFO는 스스로 양보할 때까지, RR은 같은 우선순위끼리 라운드로빈. 위험성: 실시간 태스크가 무한루프에 빠지면 해당 CPU에서 일반 태스크(심지어 SSH 데몬)가 실행 기회를 거의 얻지 못해 시스템이 사실상 잠긴다 — 기본 커널은 RT throttling(`sched_rt_runtime_us`, 기본적으로 RT에 CPU 시간의 95%까지만 허용)으로 완전 기아는 막아두지만, 남는 5%로 정상 운영은 불가능하다. `chrt`로 설정하며, 운영 서버에서는 정말 필요한 경우(오디오, 저지연 트레이딩)가 아니면 피한다.

### CPU affinity

태스크를 특정 CPU에 고정하는 것. 캐시 지역성 확보, NUMA 노드 정렬, 인터럽트와 워크로드 분리에 쓴다:

```bash
taskset -c 0-3 ./server        # CPU 0~3에서만 실행
taskset -cp 1234               # 기존 프로세스의 affinity 확인 (변경은 taskset -cp 0-3 1234처럼 CPU 목록을 앞에)
```

컨테이너 세계에서는 cgroup의 cpuset 컨트롤러가 같은 일을 하며, Kubernetes의 CPU Manager `static` 정책이 이를 이용해 파드에 전용 코어를 준다. GPU 노드에서 NUMA 정렬은 AI 인프라의 성능 기본기다 — [[AI-Infra]] 트랙에서 다시 만난다.

## 7. 컨텍스트 스위칭 — 진짜 비용은 캐시 오염

컨텍스트 스위치는 CPU에서 실행 중인 태스크를 바꾸는 일이다. 커널은 현재 태스크의 레지스터·프로그램 카운터·스택 포인터를 task_struct에 저장하고, 다음 태스크의 것을 복원하며, 주소공간이 다르면 페이지 테이블 포인터(x86의 CR3)를 교체한다.

레지스터 저장/복원 자체는 마이크로초도 안 걸린다. **진짜 비용은 간접 비용** — 새 태스크가 실행되면서 이전 태스크가 데워놓은 CPU 캐시(L1/L2/L3)와 TLB(가상→물리 주소 변환 캐시)가 무용지물이 되고, 새 태스크는 한동안 캐시 미스를 맞으며 달려야 한다는 점이다. "컨텍스트 스위치 비용의 본질은 캐시·TLB 오염"이 면접용 한 줄이다. 같은 프로세스의 스레드 간 스위치는 주소공간이 같아 TLB flush가 없으므로 상대적으로 싸다.

관측 도구:

```bash
vmstat 1            # cs 컬럼 = 초당 컨텍스트 스위치 수 (시스템 전체)
pidstat -w 1        # 프로세스별 cswch/s(자발적), nvcswch/s(비자발적)
```

- **자발적(voluntary)** 스위치: 태스크가 스스로 블록됨(I/O 대기, 락 대기). 높으면 I/O 바운드이거나 락 경합.
- **비자발적(non-voluntary)** 스위치: 타임슬라이스 소진이나 선점으로 밀려남. 높으면 CPU 경합 — 러너블 태스크가 코어 수보다 많다는 뜻.

이 구분 하나로 "느린데 왜 느린가"의 방향(I/O냐 CPU냐)을 가를 수 있다.

## 8. 시그널 — 프로세스에게 말 거는 표준어

시그널은 커널/다른 프로세스가 프로세스에게 보내는 비동기 통지다. 주요 시그널:

| 시그널 | 번호 | 기본 동작 | 핸들링 | 의미와 용례 |
|---|---|---|---|---|
| SIGTERM | 15 | 종료 | 가능 | **정중한 종료 요청.** `kill`의 기본값. 정리 작업 후 종료할 기회를 줌 |
| SIGKILL | 9 | 종료 | **불가** | 커널이 프로세스를 즉시 제거. 프로세스는 통보조차 못 받음 — 정리 작업 기회 없음 |
| SIGINT | 2 | 종료 | 가능 | 터미널 `Ctrl+C` |
| SIGHUP | 1 | 종료 | 가능 | 터미널/세션 끊김. 데몬 관례로는 "설정 리로드" (nginx 등) |
| SIGSEGV | 11 | 종료+코어 | 가능(비권장) | 잘못된 메모리 접근 — 커널이 보내는 사망 선고 |
| SIGSTOP | 19 | 정지 | **불가** | 무조건 정지(T state). SIGCONT로 재개 |
| SIGUSR1/2 | 10/12 | 종료 | 가능 | 사용자 정의 — 로그 로테이션 통지, 상태 덤프 등 앱 마음대로 |

SIGKILL과 SIGSTOP만 가로챌 수 없다는 것, 그리고 **SIGKILL이 "강력한 시그널"이 아니라 "프로세스를 거치지 않고 커널이 직접 집행하는 제거"**라는 관점이 중요하다. 그래서 `kill -9`를 맞은 프로세스는 임시파일 정리, 트랜잭션 마무리, 커넥션 정리를 전혀 못 한다.

### graceful shutdown 패턴

운영의 표준 종료 절차는 **SIGTERM → 유예 시간 → (그래도 살아있으면) SIGKILL**이다:

```bash
kill 1234            # SIGTERM — "정리하고 나가라"
sleep 30
kill -9 1234         # 유예 후에도 살아있으면 강제 제거
```

Kubernetes의 파드 종료가 정확히 이 패턴과 동형이다: kubelet이 컨테이너의 PID 1에 SIGTERM을 보내고, `terminationGracePeriodSeconds`(기본 30초)만큼 기다린 뒤 SIGKILL을 보낸다. 그래서 애플리케이션은 SIGTERM 핸들러에서 "새 요청 거부 → 진행 중 요청 드레인 → 커넥션 정리 → exit"를 구현해야 무중단 배포가 성립한다. **컨테이너에서 앱이 셸 스크립트 뒤에 실행되면(`sh -c "..."`) SIGTERM이 셸에서 멈추고 앱에 전달되지 않는 함정**도 같은 맥락 — exec 형식 ENTRYPOINT를 쓰는 이유다([[06 컨테이너 내부 구조]]).

대상 지정 도구: `kill <pid>`(PID로), `pkill -f <pattern>`(커맨드라인 패턴으로), `killall <name>`(프로세스 이름으로). `pkill`/`killall`은 패턴이 의도보다 넓게 잡힐 수 있으니 `pgrep -f <pattern>`으로 먼저 확인하는 습관이 안전하다.

## 9. 데몬과 서비스 — 더블 fork에서 systemd로

전통적 데몬화는 의식(ritual)에 가까웠다: fork 후 부모 종료(셸에서 분리) → `setsid()`로 새 세션 리더가 됨(제어 터미널과 절연) → **한 번 더 fork**(세션 리더가 다시 터미널을 획득하는 것을 원천 차단) → 작업 디렉토리·umask 정리 → 표준 입출력을 `/dev/null`로. 이것이 "더블 fork"다.

systemd 시대에는 이 의식이 불필요해졌다. 서비스는 그냥 포그라운드로 실행하고, 데몬화·로그·재시작은 systemd가 맡는다. 유닛의 `Type=`만 알면 된다: `simple`은 "프로세스 뜨면 시작된 것으로 간주"(기본값 — systemd 문서는 신규 서비스에 `exec`을 권장), `forking`은 "전통적 더블 fork 데몬을 감쌀 때", `notify`는 "프로세스가 sd_notify로 준비 완료를 알릴 때".

임시 백그라운드 실행의 비교:

| 방법 | 효과 | 한계 |
|---|---|---|
| `cmd &` | 백그라운드 실행 | 셸 종료(SIGHUP) 시 같이 죽음 |
| `nohup cmd &` | SIGHUP 무시 | 재시작·로그 관리 없음, 부모는 여전히 셸 |
| `disown` | 셸의 잡 테이블에서 분리 | 위와 동일한 한계 |
| `systemd-run --user cmd` | 즉석 transient 유닛으로 실행 | — cgroup 격리, journald 로그, systemctl로 관리까지 얻음 |

서버에서 "잠깐 띄워둘" 프로세스도 `systemd-run`이 사후 추적 가능성 면에서 우월하다. nohup으로 띄운 프로세스는 한 달 뒤 "이거 누가 띄웠지"의 주인공이 된다.

## 10. 실무 시나리오

### 시나리오 A — 좀비가 수백 개 쌓였다

```bash
ps -eo pid,ppid,stat,cmd | awk '$3 ~ /^Z/'      # 좀비 나열, PPID 확인
ps -o pid,cmd -p <ppid>                          # 부모가 누구인지 확인
```

대응 순서: ① 좀비들의 PPID가 한 프로세스로 모이는지 확인 — 대부분 그렇다. ② 그 **부모를 고치거나 재시작**한다(wait 누락 버그). 부모가 재시작되거나 죽으면 좀비들은 PID 1에 입양되어 즉시 reap된다. ③ 컨테이너 안이라면 PID 1이 reaping을 하는지 확인하고 `tini` 또는 `--init`을 도입한다. 좀비 자체에 `kill`을 보내는 것은 어떤 경우에도 효과가 없다.

### 시나리오 B — kill -9가 안 듣는 프로세스 (D state)

```bash
cat /proc/<pid>/stat | awk '{print $3}'          # 상태 확인 — D
cat /proc/<pid>/stack 2>/dev/null                # (root) 커널 스택 — 어디서 멈췄나
ps -eo pid,stat,wchan:32,cmd | awk '$2 ~ /D/'    # D state 전체와 대기 지점
```

받아들여야 할 사실: D state 프로세스는 **시그널로 제거할 수 없다.** 대응은 ① wchan/커널 스택으로 무엇을 기다리는지 확인(NFS? 특정 블록 디바이스?) ② **기다리는 I/O 자원을 회복**시킨다 — NFS 서버 복구, 마운트 옵션 점검(NFS는 `hard`/`soft` 여부에 따라 행 거동이 다름 — `intr` 옵션은 커널 2.6.25부터 무시된다), 디스크 상태 확인. ③ 자원 회복이 불가능하면 현실적인 마지막 수단은 재부팅이다. "kill -9도 안 죽는 프로세스가 있나요?"는 면접 단골이고, 답은 "D state — 시그널 배달이 보류되기 때문"이다.

### 시나리오 C — 프로세스 트리 읽기

```bash
ps auxf          # f = forest, 부모-자식 관계를 ASCII 트리로
pstree -p        # PID 포함 트리 — 구조 파악에 가장 빠름
pstree -ps 1234  # 특정 PID의 조상 경로만 추적
```

트리를 읽으면 "이 워커는 누가 띄웠나", "systemd 직속인가 누군가의 셸 아래인가", "컨테이너 런타임(containerd-shim) 아래에 뭐가 매달려 있나"가 한눈에 들어온다. 장애 시 영향 범위 판단(부모를 재시작하면 무엇이 같이 죽는가)의 기본 도구다. 더 많은 관측 커맨드는 [[07 실무 리눅스 커맨드]]에 모아져 있다.

## 핵심 요약 (면접 직전 복기용)

- 프로세스 = task_struct 하나. 스레드도 task_struct 하나 — 차이는 `clone()` 공유 플래그(주소공간·fd·시그널 핸들러 공유, TGID 동일)뿐.
- `fork()`는 CoW라 즉시 복사가 없다. 쓰는 순간 페이지 단위로 복사. fork+exec이 유닉스 실행 모델의 전부.
- zombie는 이미 죽었고 kill 불가 — **부모의 wait 누락**이 원인. orphan은 PID 1이 입양·reap. 컨테이너 PID 1이 reap 안 하면 좀비 누적.
- D state는 uninterruptible — SIGKILL도 보류됨. 리눅스 load average는 D를 포함하므로 "CPU 한가한데 load 높음 = I/O 의심".
- 스케줄러는 CFS(vruntime 공정 배분) → 커널 6.6부터 EEVDF. nice는 가중치, SCHED_FIFO/RR은 일반 태스크를 굶길 수 있어 위험.
- 컨텍스트 스위치의 본질 비용은 캐시·TLB 오염. `pidstat -w`의 자발/비자발 구분으로 I/O 경합 vs CPU 경합을 가른다.
- SIGTERM(핸들링 가능, 정리 기회) → 유예 → SIGKILL(커널 직접 제거, 통보 없음). K8s `terminationGracePeriodSeconds`가 이 패턴의 제도화.

이 노트의 시스템 콜들이 커널과 어떻게 만나는지는 [[05 커널과 시스템 콜]], 메모리 쪽 세부는 [[02 메모리 관리]]에서 이어진다.
