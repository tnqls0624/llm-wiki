# 03 파일시스템과 스토리지

허브: [[Infra]]

`write()` 한 줄이 실제 디스크 플래터/낸드 셀에 도달하기까지는 **VFS → 파일시스템 → page cache → block layer → I/O 스케줄러 → 디바이스 드라이버**라는 긴 스택을 통과한다. 이 노트는 그 스택을 위에서 아래로 종주한다. 운영 장애의 단골 손님 — "Too many open files", "rm 했는데 용량이 안 줄어요", "df는 남았다는데 쓰기가 실패해요" — 가 전부 이 스택의 특정 층에서 발생하므로, 층을 알면 진단이 빨라진다. 메모리 쪽 배경(page cache, dirty page)은 [[02 메모리 관리]]와 이어진다.

## 1. VFS — 모든 파일시스템 위의 추상 계층

VFS(Virtual File System)는 커널이 ext4, XFS, NFS, tmpfs, procfs를 **같은 시스템 콜(open/read/write/stat)로 다루게 해주는 인터페이스 계층**이다. 자바 개발자라면 "VFS는 인터페이스, ext4/XFS는 구현체"라고 이해하면 정확하다. 각 파일시스템은 `file_operations`, `inode_operations` 같은 함수 포인터 테이블을 등록하고, VFS가 디스패치한다.

### fd → open file description → inode, 3계층 구조

`open()` 한 번이 만들어내는 구조는 세 층이다. 면접 단골이므로 정확히 구분하자.

| 계층 | 소유 | 담는 것 | 공유 단위 |
|---|---|---|---|
| **fd (file descriptor)** | 프로세스별 fd table | open file description을 가리키는 정수 인덱스 | 프로세스 내 (fork 시 복제) |
| **open file description** | 커널 전역 open file table | **파일 오프셋**, 접근 모드(`O_RDONLY` 등), 상태 플래그(`O_APPEND`, `O_NONBLOCK`) | `fork()`/`dup()`한 fd끼리 공유 |
| **inode** | 파일시스템 | 메타데이터: 크기, 권한, 소유자, 타임스탬프, 데이터 블록 위치, **link count** | 같은 파일을 연 모든 fd가 공유 |

핵심 함의:

- **오프셋은 fd가 아니라 open file description에 산다.** `fork()` 후 부모·자식이 같은 fd를 쓰면 오프셋을 공유한다 — 둘이 동시에 같은 로그 파일에 쓰는데 줄이 안 겹치는 이유 중 하나가 `O_APPEND`(원자적 끝 추가)다. 반대로 같은 파일을 `open()` 두 번 하면 description이 두 개라 오프셋이 따로 논다.
- **inode에는 파일명이 없다.** 이름→inode 매핑은 **디렉토리 엔트리(dentry)**가 담당한다. 디렉토리는 "(이름, inode 번호) 쌍의 목록"인 특수 파일일 뿐이다. 그래서 같은 inode를 가리키는 이름이 여러 개일 수 있다(hard link). `ls -i`로 inode 번호 확인, `stat <file>`로 메타데이터 전체 확인.

### fd 누수와 "Too many open files"

프로세스당 fd 개수 제한은 `ulimit -n`(soft limit)이다. 전통적 기본값은 1024로, 커넥션이 많은 서버에는 턱없이 작다. 소켓도 fd이므로 **fd 누수의 대부분은 닫지 않은 소켓·파일 핸들**이다.

```bash
ulimit -n                          # 현재 셸의 soft limit
cat /proc/<pid>/limits             # 실행 중 프로세스의 limit (재시작 없이 확인)
ls /proc/<pid>/fd | wc -l          # 그 프로세스가 실제로 연 fd 개수
lsof -p <pid>                      # 무엇을 열고 있는지 목록
```

`EMFILE (Too many open files)`가 터지면: ① `ls /proc/<pid>/fd | wc -l`로 limit 대비 사용량 확인 → ② `lsof -p <pid>`로 어떤 종류(소켓? 파일? pipe?)가 다수인지 → ③ 누수면 코드 수정, 정상 부하면 limit 상향(systemd 서비스는 `LimitNOFILE=`, `/etc/security/limits.conf`는 로그인 세션용). 시스템 전체 한도는 `fs.file-max`(sysctl)로 따로 있다.

## 2. 링크 — hard link vs symlink, 그리고 "rm 해도 용량이 안 줄어요"

| | hard link (`ln`) | symlink (`ln -s`) |
|---|---|---|
| 실체 | 같은 inode를 가리키는 **또 하나의 이름** | 대상 **경로 문자열**을 담은 별도 inode |
| 파일시스템 경계 | 같은 fs 안에서만 가능 | fs를 넘어 어디든 가능 |
| 원본 삭제 시 | 다른 이름으로 여전히 접근 가능 (link count만 감소) | **끊어진(dangling) 링크**가 됨 |
| 디렉토리 대상 | 불가 (루프 방지) | 가능 |
| 확인 | `ls -li`에서 inode 번호 동일, link count ≥ 2 | `ls -l`에 `->` 표시 |

**inode의 데이터 블록이 실제로 해제되는 조건은 두 가지가 모두 0이 될 때다: link count == 0 AND 그 inode를 연 fd == 0.** `rm`은 dentry를 지워 link count를 줄일 뿐이다("unlink"라는 시스템 콜 이름이 정직하다).

여기서 운영 단골 장면이 나온다. 수십 GB 로그 파일을 `rm`했는데 `df`가 그대로다 — **어떤 프로세스가 그 파일을 아직 열고 있어서** 블록이 해제되지 않은 것이다.

```bash
lsof +L1                # link count < 1 인데 열려 있는 파일 (deleted)
lsof | grep deleted     # 같은 목적의 관용구
```

해결: 그 프로세스를 재시작하거나, 로그라면 시그널로 reopen시키거나(`logrotate`의 `copytruncate`/`postrotate` 참고), 응급으로는 `> /proc/<pid>/fd/<n>`으로 truncate할 수 있다. 교훈: **로그는 rm이 아니라 truncate(`: > file`) 또는 로테이션으로 비운다.**

## 3. 주요 파일시스템 — ext4 vs XFS vs Btrfs, 그리고 tmpfs

| | ext4 | XFS | Btrfs |
|---|---|---|---|
| 포지션 | 범용 기본값, 가장 성숙 | 대용량 파일·고병렬 I/O에 강함 | CoW 기반: 스냅샷·압축·체크섬 내장 |
| 채택 | 다수 배포판 기본 | RHEL 계열 기본 | 일부 배포판(openSUSE 등)·NAS |
| 축소(shrink) | 가능 (오프라인) | **불가** — 줄이려면 백업 후 재생성 | 가능 |
| 강점 | 검증된 안정성, 도구 생태계 | 메타데이터 병렬성, 큰 파일 처리량 | 스냅샷, 투명 압축, 데이터 체크섬 |
| 주의점 | 매우 큰 볼륨에서 한계 | 축소 불가, 작은 파일 다수엔 이점 적음 | 워크로드 따라 성능 편차, 운영 노하우 필요 |

선택 기준은 단순하게: **고민할 이유가 없으면 배포판 기본값**(ext4 또는 XFS), 대용량 스토리지·DB·고병렬이면 XFS, 스냅샷이 1급 요구사항이면 Btrfs 또는 LVM 스냅샷/ZFS 검토.

### 저널링이 보장하는 것과 안 하는 것

저널링은 **크래시 후 파일시스템 메타데이터가 일관된 상태로 복구됨**을 보장하는 장치다 — 변경을 본 위치에 쓰기 전에 저널(로그)에 먼저 기록하고, 크래시 시 저널을 재생(replay)한다. DB의 WAL과 같은 아이디어다.

중요한 한계: **기본 모드에서 데이터 본문의 내용까지 보장하지는 않는다.** ext4의 기본 `data=ordered`는 "메타데이터를 커밋하기 전에 해당 데이터 블록을 먼저 디스크에 내린다"는 순서만 보장한다(쓰레기 블록이 파일에 보이는 것은 막아줌). `data=journal`은 데이터까지 저널에 쓰지만 두 번 쓰는 비용 탓에 거의 안 쓴다. **애플리케이션 데이터의 내구성은 fsync의 몫이다** — 저널링이 있다고 fsync를 생략할 수 있는 게 아니다(6절).

### tmpfs — 메모리가 곧 파일시스템

tmpfs는 page cache 위에 직접 사는 파일시스템이다. 디스크 백엔드가 없으니 빠르고, **재부팅하면 사라지며**, 메모리 압박 시 swap으로 밀려날 수 있다. `/dev/shm`, `/run`이 tmpfs고, 배포판에 따라 `/tmp`도 tmpfs다(Fedora·Arch 등은 기본, RHEL 계열은 기본 아님). 쓰는 내용만큼만 메모리를 차지하지만 **크기 상한까지 차면 그만큼 RAM을 먹는다** — cgroup 메모리 한도 안에서도 집계되므로, K8s에서 `emptyDir.medium: Memory`를 쓰면 그 볼륨 사용량이 Pod 메모리 한도를 잠식해 OOM을 부를 수 있다([[06 컨테이너 내부 구조]]).

## 4. 마운트 — fstab, 옵션, bind mount

리눅스에는 드라이브 문자가 없다. 모든 파일시스템은 단일 트리의 어딘가에 **마운트**되어 합류한다. 부팅 시 마운트 목록은 `/etc/fstab`:

```
# <device>                                <mountpoint>  <fstype>  <options>            <dump> <fsck>
UUID=1234abcd-...                         /data         xfs       defaults,noatime     0      2
tmpfs                                     /dev/shm      tmpfs     defaults,size=2g     0      0
```

디바이스는 `/dev/sdb1` 같은 이름 대신 **UUID나 LABEL**로 적는다 — 디스크 추가/제거 시 커널이 부여하는 이름이 바뀔 수 있기 때문이다(클라우드에서 특히). fstab을 고친 뒤에는 재부팅 전에 `mount -a`로 문법 오류를 검증하는 습관이 부팅 실패를 막는다.

주요 옵션:

- `defaults` — `rw,suid,dev,exec,auto,nouser,async` 묶음.
- `relatime` — atime(접근 시각)을 mtime보다 오래된 경우 등에만 갱신. 현대 커널 기본값. 읽기마다 메타데이터 쓰기가 발생하던 고전적 낭비를 줄인다.
- `noatime` — atime 갱신 자체를 끔. DB·대량 읽기 워크로드에서 흔한 튜닝. atime에 의존하는 도구가 거의 없으므로 보통 안전.
- `ro` — 읽기 전용. 손상 의심 디스크의 포렌식, 불변 이미지 마운트에 사용.

**bind mount**는 디렉토리 트리의 한 부분을 다른 경로에 다시 비추는 마운트다: `mount --bind /data/app /var/lib/app`. 디바이스가 아니라 **이미 마운트된 트리의 서브트리**를 재마운트하는 것으로, **컨테이너 볼륨(`docker run -v /host:/ctr`)의 구현 기반**이다 — mount namespace와 결합하는 그림은 [[06 컨테이너 내부 구조]]에서 다룬다.

현재 마운트 상태 탐색은 `mount` 출력보다 트리로 보여주는 `findmnt`가 압도적으로 읽기 좋다. 특정 경로가 어느 파일시스템에 속하는지는 `findmnt -T /path/to/dir`.

## 5. 디스크 공간 진단 — df vs du, 그리고 inode 고갈

**`df -h`는 파일시스템(슈퍼블록) 관점의 집계, `du -sh`는 디렉토리 트리를 걸으며 합산한 값**이다. 둘이 다르면 보통 다음 중 하나다:

1. **삭제됐지만 열려 있는 파일** — du는 dentry가 없어 못 보지만 df의 사용량에는 남아 있다. `lsof +L1`로 확인 (2절).
2. **마운트가 가린 디렉토리** — `/data`에 파일을 쓴 뒤 그 위에 다른 fs를 마운트하면, 원래 파일은 가려져 du에 안 잡히지만 하위 fs 용량은 차지한다. `umount` 하거나 bind mount로 원본을 들춰 확인.
3. 파일시스템 예약 블록(ext4는 기본 5%를 root용으로 예약 — `tune2fs -m`으로 조정) 등 회계 차이.

### inode 고갈 — 용량이 남아도 쓰기가 실패한다

ext4는 파일시스템 생성 시 inode 개수를 고정한다. **수백만 개의 작은 파일**(세션 파일, 캐시 조각, 빌드 아티팩트, 메일 큐)이 inode를 다 쓰면, `df -h`는 수십 % 여유를 보여주는데도 `No space left on device`가 터진다. 면접 단골 "df는 남았는데 쓰기 실패?"의 대표 답이 **inode 고갈**이다(그 외 후보: 디스크 쿼터, Btrfs 메타데이터 고갈). 주의 — 열린 deleted 파일은 이 시나리오의 원인이 아니다: 그 블록은 df에 사용 중으로 **잡히므로**, 그쪽은 반대로 "df는 찼는데 du 합계가 작다"(2·5절)를 설명한다.

```bash
df -i                      # IUse% 확인 — 100%면 inode 고갈
# 범인 디렉토리 찾기: 파일 수가 많은 디렉토리 톱다운 탐색
du --inodes -x -d1 /var | sort -n | tail   # GNU du: inode 수 기준 집계
```

해결은 파일 삭제/아카이브가 기본. 근본적으로는 mkfs 시 `-i`(bytes-per-inode) 값을 **낮춰** inode 밀도를 높이거나, inode를 동적 할당하는 XFS/Btrfs로 옮긴다.

## 6. I/O 경로 — write()가 디스크에 닿기까지

```
write(fd, buf, n)
  → page cache에 복사, 페이지를 dirty로 마킹  ← 여기서 시스템 콜은 리턴 (빠른 이유)
  → writeback (커널 스레드가 주기/임계치 기반으로 플러시)
  → block layer: 요청 병합(merge)·정렬, blk-mq 멀티큐
  → I/O 스케줄러 (mq-deadline / bfq / kyber / none)
  → 디바이스 드라이버 → 디스크 (디스크 자체 캐시 → 매체)
```

- **write()의 리턴은 "디스크에 썼다"가 아니라 "page cache에 복사했다"는 뜻이다.** 내구성은 별도로 요구해야 한다. dirty 비율·writeback 튜닝(`vm.dirty_ratio` 등)은 [[02 메모리 관리]] 참고.
- **block layer**는 인접 섹터 요청을 병합하고 정렬해 디바이스에 유리한 패턴으로 만든다. 현대 커널은 blk-mq(멀티큐) 구조로 CPU 코어별 소프트웨어 큐 → 하드웨어 큐로 분산해 NVMe의 병렬성을 살린다.
- **I/O 스케줄러**: `mq-deadline`(읽기 기아 방지, 무난한 기본), `bfq`(공정성·데스크톱 반응성), `kyber`(지연 목표 기반, 빠른 디바이스), `none`(스케줄링 생략). **NVMe처럼 디바이스 자체가 깊은 큐로 재정렬을 잘하는 경우 `none`이 일반적**이다 — 소프트웨어 재정렬이 오히려 오버헤드. 확인/변경: `cat /sys/block/<dev>/queue/scheduler`.

### O_DIRECT와 fsync — 캐시 우회와 내구성

- **`O_DIRECT`**: page cache를 우회해 사용자 버퍼 ↔ 디바이스 직행. 자체 버퍼 풀을 가진 DB(예: 일부 DB 엔진의 데이터 파일 I/O)가 **이중 캐싱을 피하려고** 쓴다. 정렬 제약(버퍼·오프셋·길이)이 있고, 캐시의 이점도 포기하는 것이므로 일반 앱에는 비추.
- **`fsync(fd)`**: 그 파일의 dirty 데이터와 메타데이터를 **디바이스까지** 내려보내고 완료를 기다린다. **DB가 커밋마다 WAL에 fsync하는 이유** — 이것 없이는 "커밋 성공" 응답 후 전원이 나가면 데이터가 증발한다. `fdatasync`는 메타데이터 중 필수가 아닌 것(mtime 등)을 생략해 약간 싸다.
- 한 줄 주의: **fsync가 실패한 뒤의 세계는 어렵다** — 실패 시 dirty 페이지의 상태 처리가 파일시스템마다 미묘해, 과거 PostgreSQL이 fsync 실패를 재시도하다 데이터 손실 가능성이 드러나 패닉하도록 바꾼 사례(통칭 fsyncgate)가 유명하다. "fsync 에러는 재시도하면 된다"고 가정하지 말 것.

### io_uring — 비동기 I/O의 현재 표준

리눅스의 전통 AIO(`io_submit`)는 사실상 `O_DIRECT` 전용에 가깝고 조건이 안 맞으면 몰래 블로킹되는 한계로 외면받았다. **io_uring**(5.1+)은 사용자 공간과 커널이 공유하는 **제출 큐(SQ)/완료 큐(CQ) 링 버퍼 한 쌍**으로 시스템 콜 횟수 자체를 줄이고, 파일·소켓·fsync 등 광범위한 연산을 진짜 비동기로 처리한다. 고성능 스토리지/네트워크 엔진의 표준으로 자리잡는 중이며, 강력한 만큼 보안 관점에서 일부 환경(컨테이너 seccomp 기본 정책 등)이 제한하기도 한다는 점만 기억해두자.

## 7. LVM — 디스크와 파일시스템 사이의 유연한 층

LVM(Logical Volume Manager)은 물리 디스크와 파일시스템 사이에 추상 계층을 끼워 **온라인 확장, 스냅샷, 여러 디스크의 풀링**을 가능하게 한다.

```
PV (Physical Volume)  : 물리 디스크/파티션을 LVM 멤버로 등록   pvcreate /dev/sdb
VG (Volume Group)     : PV들을 묶은 용량 풀                    vgcreate data_vg /dev/sdb /dev/sdc
LV (Logical Volume)   : VG에서 잘라낸 가상 블록 디바이스        lvcreate -n app_lv -L 100G data_vg
                        → mkfs 후 마운트: /dev/data_vg/app_lv
```

왜 쓰나: ① 파티션과 달리 **서비스 무중단으로 LV를 키울 수 있고**(클라우드에서 EBS 볼륨 확장 후 fs까지 늘리는 시나리오), ② CoW **스냅샷**으로 백업 시점 고정이 가능하며, ③ 디스크 여러 개를 하나의 풀로 쓴다.

디스크 확장 절차 (예: EBS 볼륨을 키운 뒤):

```bash
# 0. 클라우드 콘솔/CLI에서 볼륨 크기 변경 (온라인 가능)
lsblk                                  # 커널이 새 크기를 인식했는지 확인
growpart /dev/nvme1n1 1                # 파티션을 썼다면 파티션부터 확장
pvresize /dev/nvme1n1p1                # PV에 새 용량 반영
lvextend -r -l +100%FREE /dev/data_vg/app_lv
# -r 이 핵심: LV 확장 + 파일시스템 resize까지 한 번에
# -r 없이 했다면 수동으로: resize2fs(ext4) 또는 xfs_growfs <mountpoint>(XFS)
df -h                                  # 결과 확인
```

기억할 비대칭: **확장은 ext4/XFS 모두 온라인 가능, 축소는 ext4만(오프라인) 가능하고 XFS는 불가.** 그래서 처음부터 과하게 크게 잡기보다 작게 시작해 늘려가는 운용이 정석이다.

## 8. RAID — 표 하나로 정리

| 레벨 | 구성 | 최소 디스크 | 용량 효율 | 내결함성 | 쓰기 성능 | 용도 |
|---|---|---|---|---|---|---|
| RAID 0 | 스트라이핑 | 2 | 100% | **없음** (1개 죽으면 전체 손실) | 좋음 | 스크래치·캐시 등 잃어도 되는 데이터 |
| RAID 1 | 미러링 | 2 | 50% | 1개 손실 허용 | 보통 (양쪽에 씀) | OS/부트 디스크 |
| RAID 5 | 스트라이프+패리티 1 | 3 | (N-1)/N | 1개 손실 | 쓰기 페널티 (패리티 갱신: read-modify-write) | 읽기 위주 대용량 |
| RAID 6 | 패리티 2 | 4 | (N-2)/N | 2개 손실 | RAID 5보다 더 느린 쓰기 | 대용량 디스크 어레이 (리빌드 중 추가 고장 대비) |
| RAID 10 | 미러 후 스트라이프 | 4 | 50% | 미러 쌍당 1개 | 좋음 | DB 등 쓰기 성능+안정성 |

소프트웨어 RAID는 `mdadm`으로 만든다: `mdadm --create /dev/md0 --level=10 --raid-devices=4 /dev/sd[b-e]`. 상태는 `cat /proc/mdstat`.

면접 단골 두 가지: ① 대용량 HDD 시대에 RAID 5가 기피되는 이유 — 디스크 1개 교체 후 **리빌드에 수십 시간**이 걸리고 그동안 남은 디스크들이 풀 읽기 부하를 받아 두 번째 고장(또는 읽기 에러) 확률이 올라간다. 그래서 RAID 6/10 선호. ② **RAID는 백업이 아니다** — RAID는 디스크 *하드웨어 고장*에 대한 가용성 장치일 뿐, `rm -rf`·랜섬웨어·버그에 의한 삭제는 모든 복제본에 충실히 반영된다. 백업은 별도 매체·별도 장소에.

## 9. 디바이스 계층 — HDD vs SATA SSD vs NVMe

| | HDD | SATA SSD | NVMe SSD |
|---|---|---|---|
| 랜덤 액세스 지연 | **~10ms** (헤드 탐색+회전 대기) | **~100µs** 수준 | **~10–20µs** 수준 |
| 랜덤 IOPS | 수백 | 수만 | 수십만 이상 |
| 인터페이스 | SATA/SAS | SATA (~600MB/s 한계) | PCIe 직결 (수 GB/s) |
| 큐 | 얕음 (NCQ 32) | NCQ 32 | **다수 큐 × 큐당 수만 깊이** |
| 적합 워크로드 | 순차 대용량·콜드 데이터 | 범용 | DB·고병렬·저지연 |

자릿수(order of magnitude)가 핵심이다: **HDD→SSD는 약 100배, SSD→NVMe는 다시 약 5~10배** 지연이 줄어든다. HDD의 10ms는 기계식 동작(암 이동+플래터 회전)에서 오는 물리적 하한이라 랜덤 I/O에서 도망칠 수 없다 — DB 인덱스가 B-tree로 랜덤 액세스를 줄이도록 설계된 역사적 배경이다.

**큐 깊이(queue depth)** 는 디바이스에 동시에 걸어둘 수 있는 미완료 요청 수다. NVMe가 빠른 이유의 절반은 매체(낸드), 절반은 **프로토콜의 병렬성** — 큐가 깊고 많아서 여러 CPU 코어가 락 경합 없이 동시에 요청을 밀어 넣을 수 있다(blk-mq가 이를 받쳐준다). 벤치마크에서 `fio --iodepth=1`과 `--iodepth=32`의 IOPS가 크게 다른 이유이기도 하다: 깊이 1은 지연(latency)을, 깊은 큐는 처리량(throughput) 한계를 측정한다.

## 10. 클라우드 디스크 — EBS와 인스턴스 스토어

- **EBS gp3**: 범용 SSD. 핵심은 **IOPS와 처리량을 용량과 독립적으로 프로비저닝**할 수 있다는 것 — 구세대 gp2가 "용량에 비례한 IOPS"라서 성능 때문에 용량을 사야 했던 문제를 해결했다. 기준 성능(3,000 IOPS/125MB/s)에서 시작해 필요한 만큼만 추가 구매.
- **io2**: 고IOPS·높은 내구성 보장이 필요한 DB급 워크로드용 프로비전드 IOPS 볼륨.
- **인스턴스 스토어(instance store)**: 호스트에 물리적으로 붙은 NVMe. 매우 빠르지만 **인스턴스 중지/하드웨어 장애 시 데이터가 사라진다** — 캐시, 스크래치, 복제본이 따로 있는 데이터(예: 분산 스토리지의 한 레플리카)에만 사용.
- 함정 한 줄: **볼륨 성능 한도와 별개로 인스턴스 타입 자체에 EBS 대역폭 한도가 있다** — 큰 io2를 붙여도 작은 인스턴스면 인스턴스 한도에서 막힌다. 스토리지 성능 문제는 볼륨과 인스턴스 양쪽 한도를 모두 본다.
- EBS는 네트워크 너머의 블록 디바이스이므로 로컬 NVMe보다 지연이 한 자릿수 이상 길다는 감각(밀리초 미만~밀리초대)을 갖고 설계할 것.

## 11. 실무 시나리오

### 시나리오 A: "디스크가 가득 찼어요" 표준 대응 절차

```bash
# 1. 어느 파일시스템이 찼는지
df -h

# 2. 그 마운트포인트에서 큰 디렉토리 톱다운 추적 (-x: 다른 fs로 안 넘어감)
du -xh -d1 /var 2>/dev/null | sort -rh | head
du -xh -d1 /var/log | sort -rh | head      # 단계적으로 내려감

# 3. du 합계가 df보다 한참 작다면 → 삭제됐지만 열린 파일
lsof +L1                                    # 범인 프로세스 식별 → 재시작/reopen/truncate

# 4. 재발 방지: 로그 로테이션 점검
cat /etc/logrotate.d/*                      # 해당 로그가 로테이션 대상인지, 주기·보존 개수
journalctl --disk-usage                     # systemd 저널이 범인인 경우도 흔함
journalctl --vacuum-size=1G
```

루트 fs가 100%면 셸 동작도 이상해지므로(임시 파일 생성 실패), 응급으로 큰 로그 하나를 `: > /var/log/big.log`로 truncate해 숨통을 틔운 뒤 차분히 진행한다.

### 시나리오 B: 용량은 남는데 "No space left on device"

```bash
df -i                                       # IUse% 100% → inode 고갈
du --inodes -x -d1 / | sort -n | tail       # 파일 수 폭탄 디렉토리 추적
```

흔한 범인: 세션 파일 디렉토리, CI 빌드 캐시, 컨테이너 레이어 잔해, 메일 큐, 수년치 작은 로그 조각. 삭제 시 파일이 수백만 개면 `rm *`은 인자 한도에 걸리므로 `find <dir> -type f -delete`를 쓴다.

### 시나리오 C: 디스크 늘렸는데 df가 그대로

클라우드에서 볼륨만 키우고 끝낸 경우다. 블록 디바이스 크기(`lsblk`)와 파일시스템 크기(`df`)는 별개의 층이다. 파티션 → (LVM이면 PV/LV) → 파일시스템 순으로 각 층을 늘려야 한다(7절 절차). 마지막 단계가 `resize2fs`(ext4) 또는 `xfs_growfs`(XFS, **마운트포인트를 인자로**)임을 기억.
