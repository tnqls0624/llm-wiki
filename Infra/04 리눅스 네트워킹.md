# 04 리눅스 네트워킹
허브: [[Infra]]

TCP/IP 스택이 커널 안에서 어떻게 동작하는지, 그리고 서버 네트워크 문제를 어디서부터 파고들어야 하는지를 다룬다. 백엔드 개발자에게 네트워크는 "라이브러리가 알아서 해주는 것"이지만, 인프라 엔지니어에게는 **드롭과 지연이 발생하는 모든 지점을 지도처럼 그릴 수 있어야 하는 영역**이다. 시스템 콜 경계와 커널 구조는 [[05 커널과 시스템 콜]], 컨테이너 네트워크 네임스페이스는 [[06 컨테이너 내부 구조]]에서 이어진다.

## 1. 큰 그림 — 패킷 하나의 수신 여정

패킷이 NIC에 도착해서 애플리케이션의 `read()`에 도달하기까지의 경로. **이 그림이 모든 네트워크 드롭 진단의 지도다.** 어느 단계든 큐가 가득 차면 드롭이 발생하고, 단계마다 드롭을 확인하는 커맨드가 다르다.

```
NIC ──DMA──> ring buffer ──하드 IRQ──> softirq(NAPI poll)
  ──> 프로토콜 스택(IP→TCP) ──> socket receive buffer ──> 앱의 read()
```

| 단계 | 무슨 일이 일어나나 | 드롭 확인 |
|---|---|---|
| NIC → ring buffer | NIC가 DMA로 패킷을 메모리의 ring buffer에 복사 | `ethtool -S eth0` (rx_dropped, rx_missed) |
| 하드 IRQ | NIC가 CPU에 인터럽트 — "패킷 왔다"는 신호만 | `/proc/interrupts` |
| softirq (NAPI) | 실제 처리는 softirq 컨텍스트에서 폴링으로 일괄 수행 | `/proc/net/softnet_stat` 2번째 컬럼 |
| 프로토콜 스택 | IP 라우팅 판단, netfilter 통과, TCP 상태 머신 처리 | `netstat -s`, `nstat` |
| socket buffer | 소켓별 수신 버퍼에 적재, 앱이 읽기를 기다림 | `ss -tnp`의 Recv-Q |
| 앱 read() | 앱이 시스템 콜로 데이터를 가져감 | 앱이 느리면 Recv-Q 누적 |

왜 IRQ와 softirq를 나눴나? 하드 IRQ는 다른 인터럽트를 막으므로 최소한의 일만 하고, 무거운 처리는 softirq로 미룬다. NAPI는 패킷이 몰릴 때 인터럽트를 끄고 폴링으로 전환해 **interrupt storm을 방지**한다 — 트래픽이 많을수록 인터럽트는 오히려 줄어드는 이유다. 송신은 정확히 역방향이다: `write()` → socket send buffer → 프로토콜 스택 → qdisc(트래픽 제어 큐) → ring buffer → NIC.

기억할 비유: **컨베이어 벨트의 연속**. 각 벨트(큐) 사이의 속도 차가 누적되는 지점이 병목이고, 벨트가 넘치면 물건(패킷)이 바닥에 떨어진다(드롭).

## 2. TCP 연결 수립과 종료

### 3-way handshake

```
클라이언트                          서버 (LISTEN)
   │ ── SYN (seq=x) ──────────────> │  SYN_RECV
   │ <─ SYN-ACK (seq=y, ack=x+1) ── │
   │ ── ACK (ack=y+1) ────────────> │  ESTABLISHED
ESTABLISHED
```

왜 3번인가? 양쪽 모두 "내 시퀀스 번호를 상대가 받았다"는 확인이 필요하기 때문이다. 2번이면 서버는 자신의 시퀀스 번호가 전달됐는지 알 수 없다. 시퀀스 번호는 패킷 순서 보장과 재전송 판단의 기준이므로, 이 합의 없이는 신뢰성 있는 스트림이 성립하지 않는다 (RFC 9293).

### 4-way teardown

TCP는 전이중(full-duplex)이라 **각 방향을 독립적으로 닫는다**. 그래서 FIN이 두 번 필요하다.

```
능동 종료 측 (active close)              수동 종료 측 (passive close)
   │ ── FIN ──────────────────────────> │  CLOSE_WAIT  ← 앱이 close() 할 때까지 유지
FIN_WAIT_1
   │ <─ ACK ─────────────────────────── │
FIN_WAIT_2
   │ <─ FIN ─────────────────────────── │  LAST_ACK   ← 앱이 close()를 호출한 시점
   │ ── ACK ──────────────────────────> │  CLOSED
TIME_WAIT (2MSL 대기 후 CLOSED)
```

상태 다이어그램 전체를 외울 필요는 없다. 실무에서 만나는 것은 6개다: **LISTEN**(접속 대기), **SYN_RECV**(handshake 진행 중 — 폭증하면 SYN flood 의심), **ESTABLISHED**(정상), **TIME_WAIT**(능동 종료 측의 대기), **CLOSE_WAIT**(수동 종료 측이 close()를 안 한 상태), **FIN_WAIT_2**(상대의 FIN을 기다리는 중 — 상대 앱이 안 닫고 있다는 뜻).

```bash
ss -s                                  # 상태별 소켓 개수 요약
ss -tan state time-wait | wc -l        # TIME_WAIT 개수
ss -tan state close-wait               # CLOSE_WAIT 목록 (있으면 앱 버그)
```

## 3. TIME_WAIT 제대로 이해하기 (면접 단골 1순위)

**누가 진입하나**: 먼저 `close()`를 호출한 쪽(능동 종료 측)이다. 서버가 연결을 먼저 끊는 구조(짧은 HTTP 응답 후 close)라면 서버에 쌓이고, 클라이언트가 끊으면 클라이언트에 쌓인다.

**왜 기다리나** (2MSL, 리눅스는 60초 고정 — 커널 상수 `TCP_TIMEWAIT_LEN`, sysctl로 조정 불가):
1. **지연 패킷 처리**: 네트워크에 떠돌던 옛 연결의 패킷이, 같은 4-tuple(src IP:port, dst IP:port)로 새로 맺은 연결에 끼어들어 데이터를 오염시키는 것을 방지.
2. **마지막 ACK 유실 대비**: 내가 보낸 마지막 ACK가 유실되면 상대는 FIN을 재전송한다. 내가 이미 CLOSED라면 RST로 응답해 상대를 비정상 종료시키므로, 재전송된 FIN에 다시 ACK해 줄 수 있도록 대기한다.

**TIME_WAIT가 수만 개라도 대개 무해하다.** 소켓 하나가 차지하는 건 소량의 커널 메모리뿐이고 fd도 이미 반환됐다. **진짜 문제는 클라이언트 측 ephemeral port 고갈**이다: 같은 목적지(IP:port)로 나가는 발신 연결은 로컬 포트를 하나씩 점유하는데, `net.ipv4.ip_local_port_range`(보통 약 28,000개)를 TIME_WAIT가 다 잡고 있으면 새 연결이 `EADDRNOTAVAIL`로 실패한다. 프록시/게이트웨이 서버에서 백엔드로 매 요청 새 연결을 맺을 때 전형적으로 발생한다.

| 대응 | 평가 |
|---|---|
| connection pool / HTTP keepalive | **정석.** 연결을 재사용하면 TIME_WAIT 자체가 생기지 않는다 |
| `net.ipv4.tcp_tw_reuse=1` | 보조 수단. **클라이언트 측 발신(outgoing) 연결에만** 적용, timestamp 기반으로 안전하게 TIME_WAIT 소켓 재사용 |
| `ip_local_port_range` 확대 | 보조 수단. 포트 풀 자체를 늘림 |
| `tcp_tw_recycle` | **커널에서 제거된 옵션.** NAT 환경에서 정상 연결을 끊는 부작용으로 악명 높았다. 오래된 블로그가 권해도 절대 쓰지 말 것 |

## 4. CLOSE_WAIT — 누적되면 무조건 앱 버그

CLOSE_WAIT는 "상대가 FIN을 보냈고 커널이 ACK까지 했는데, **우리 앱이 아직 `close()`를 호출하지 않은 상태**"다. 커널은 앱이 닫아줄 때까지 무한정 기다린다 — 타임아웃이 없다.

따라서 CLOSE_WAIT가 계속 누적된다는 것은 거의 확실히 애플리케이션 버그다(상대가 쓰기 방향만 닫는 의도적 half-close 중에는 일시적으로 정상 존재할 수 있다): 예외 경로에서 소켓을 안 닫거나, HTTP 클라이언트 응답 body를 안 닫거나, 커넥션 풀 반환 누락 등 **fd 누수**다. 방치하면 fd 한도(`ulimit -n`)에 도달해 `Too many open files`로 서비스가 죽는다.

**면접 단골 비교** — 이 표 하나로 답이 끝난다:

| | TIME_WAIT | CLOSE_WAIT |
|---|---|---|
| 누가 | 능동 종료 측(먼저 close한 쪽) | 수동 종료 측(FIN 받은 쪽) |
| 의미 | 프로토콜이 요구하는 **정상 대기** | 앱이 close()를 안 한 **비정상 상태** |
| 소멸 | 60초 후 자동 소멸 | 앱이 close()할 때까지 영원히 유지 |
| 대응 | 대개 무대응, 포트 고갈 시 pool/keepalive | **무조건 앱 코드 수정** (fd 누수 추적) |

```bash
ss -tanp state close-wait              # 어느 프로세스가 범인인지 (-p)
ls /proc/<pid>/fd | wc -l              # 해당 프로세스 fd 개수 추이 관찰
```

## 5. 두 개의 backlog — 연결이 거부되는 첫 번째 관문

서버의 `listen()` 뒤에는 큐가 **두 개** 있다. 이걸 하나로 알고 있으면 튜닝이 어긋난다.

| 큐 | 무엇이 들어가나 | 크기 결정 | 가득 차면 |
|---|---|---|---|
| SYN backlog | SYN 받고 handshake 진행 중(SYN_RECV)인 연결 | `net.ipv4.tcp_max_syn_backlog` | 새 SYN 드롭 (클라이언트는 재시도 후 timeout) |
| accept backlog | handshake 완료 후 앱의 `accept()`를 기다리는 연결 | `listen(fd, backlog)` 인자, 상한은 `net.core.somaxconn` | SYN-ACK에 대한 ACK 무시 또는 SYN 드롭 |

핵심 함정: 앱이 `listen(fd, 4096)`을 줘도 `somaxconn`이 128이면 128로 잘린다(조용히). redis가 기동 로그에 somaxconn 경고를 찍는 이유다. accept backlog가 차는 것은 **앱이 accept()를 충분히 빨리 못 한다**는 뜻 — 워커 스레드 고갈, GC pause, 이벤트 루프 블로킹을 의심한다.

```bash
ss -ltn                  # LISTEN 소켓: Recv-Q = 현재 accept 대기 수, Send-Q = backlog 최대치
netstat -s | grep -i -E "listen|overflow"   # "times the listen queue overflowed" 누적 카운트
sysctl net.core.somaxconn net.ipv4.tcp_max_syn_backlog
```

SYN flood 방어인 `net.ipv4.tcp_syncookies=1`(기본 활성)은 SYN backlog가 넘쳐도 상태 저장 없이 SYN-ACK를 응답하게 해 준다 — 정상 트래픽에서 syncookies 로그가 보이면 backlog가 작거나 공격 중이라는 신호다.

## 6. 소켓 버퍼, 흐름 제어, 혼잡 제어

TCP의 속도는 두 가지 제어의 **최솟값**으로 결정된다:
- **흐름 제어(flow control)**: 수신자의 사정. 수신 윈도우(rwnd) = 수신 버퍼의 남은 공간을 ACK에 실어 광고한다. 수신 앱이 느려서 버퍼가 차면 윈도우가 0이 되고 송신이 멈춘다 (zero window).
- **혼잡 제어(congestion control)**: 네트워크의 사정. 송신자가 패킷 손실/지연을 신호로 혼잡 윈도우(cwnd)를 조절한다. 리눅스 기본은 **cubic**(손실 기반). **BBR**은 손실 대신 대역폭·RTT를 직접 측정하는 모델 기반 알고리즘으로, 장거리·일부 손실 구간에서 처리량이 크게 개선될 수 있다 (`net.ipv4.tcp_congestion_control`).

버퍼 크기는 `net.ipv4.tcp_rmem` / `tcp_wmem` (min/default/max 3값)으로 제어되고 커널이 자동 튜닝한다. 최대 처리량의 이론적 한계는 **bandwidth-delay product(BDP)** — 대역폭 × RTT만큼의 데이터가 "비행 중"일 수 있어야 파이프가 꽉 찬다. 10Gbps × 50ms RTT면 약 62MB가 필요하므로, 장거리 고대역폭 전송에서 버퍼 max가 작으면 대역폭을 못 채운다.

재전송(retransmission)은 네트워크 품질의 가장 솔직한 지표다:

```bash
ss -ti dst <ip>          # 연결별 상세: cwnd, rtt, retrans 카운트, 사용 중인 CC 알고리즘
netstat -s | grep -i retrans          # 시스템 전체 재전송 누적치 — 추이로 판단
nstat -az TcpRetransSegs              # 같은 값, 카운터 형식
```

## 7. UDP — 단순함이 기능이다

UDP는 비연결·비신뢰: handshake도, 재전송도, 순서 보장도, 혼잡 제어도 없다. 그 대신 지연이 예측 가능하고 헤더가 8바이트로 가볍다. DNS, 메트릭(statsd), 게임, 그리고 **QUIC(HTTP/3)의 기반**이 UDP인 이유다 — QUIC은 재전송·혼잡 제어를 유저스페이스에서 직접 구현해 커널 TCP의 경직성을 우회한다.

UDP의 유일한 운영 포인트는 **수신 버퍼 오버플로 드롭**이다. 앱이 읽는 속도보다 패킷이 빨리 오면 소켓 버퍼가 넘치고 커널이 조용히 버린다.

```bash
netstat -su | grep -i -E "errors|RcvbufErrors"   # receive buffer errors 증가 = 드롭 발생 중
ss -unp                                          # UDP 소켓별 Recv-Q 확인
sysctl net.core.rmem_max                         # 앱이 SO_RCVBUF로 키울 수 있는 상한
```

## 8. DNS — 모든 장애의 단골 용의자

이름 해석 경로: 앱은 glibc의 resolver를 통해 ① `/etc/nsswitch.conf`의 `hosts:` 줄 순서를 따라(보통 `files dns`) ② `/etc/hosts`를 먼저 보고 ③ `/etc/resolv.conf`의 `nameserver`에 질의한다. 주의: `dig`는 이 경로를 **건너뛰고** resolv.conf의 서버에 직접 묻는다 — `dig`는 되는데 앱은 안 되면 nsswitch/hosts/캐시 문제다.

```bash
dig example.com +short                 # 답만 간단히
dig @8.8.8.8 example.com A             # 특정 서버에 직접 질의 (로컬 캐시 우회)
dig example.com AAAA                   # 타입은 명시한다 (ANY는 다수 서버가 거부, 의존 금지)
dig example.com +trace                 # 루트부터 위임 경로 추적
host example.com                       # 간단 조회
```

응답의 TTL은 캐시 유효 시간이다. 배포 전 DNS 전환을 계획한다면 TTL을 미리 낮춰 둬야 한다. 최근 배포판은 `systemd-resolved`가 127.0.0.53에서 로컬 스텁 캐시로 동작한다 (`resolvectl status`로 실제 업스트림 확인). Kubernetes에서는 resolv.conf의 `options ndots:5` 때문에 외부 도메인 조회 시 search 도메인을 붙인 무효 질의를 여러 번 먼저 보낸다 — 외부 호출 지연의 흔한 범인이며, FQDN 끝에 점을 붙이거나(`example.com.`) ndots를 낮춰 회피한다.

## 9. HTTP 계층의 진화 — 전송 계층 한계와의 싸움

| 버전 | 전송 | 핵심 | 남은 문제 |
|---|---|---|---|
| HTTP/1.1 | TCP | keepalive로 연결 재사용 | 연결당 요청 1개씩 직렬 처리 → **HTTP head-of-line blocking**, 브라우저는 도메인당 6연결로 우회 |
| HTTP/2 | TCP | 한 연결에서 스트림 멀티플렉싱, 헤더 압축(HPACK) | **TCP HOL은 남는다**: 패킷 1개 유실이 모든 스트림을 멈춤 (TCP는 바이트 순서를 보장하므로) |
| HTTP/3 | QUIC(UDP) | 스트림별 독립 재전송 → HOL 해소, TLS 내장, **connection migration**(IP가 바뀌어도 연결 ID로 유지 — 모바일 Wi-Fi↔LTE 전환) | UDP 차단 네트워크에서 폴백 필요 |

TLS handshake: TLS 1.2는 TCP handshake 후 2-RTT가 더 필요했지만, **TLS 1.3은 1-RTT**로 줄였고 세션 재개(session resumption) 시 0-RTT까지 가능하다(재전송 공격 고려로 멱등 요청에만). `curl -v https://...`로 handshake 단계별 진행을 그대로 볼 수 있다.

## 10. NAT와 conntrack — 보이지 않는 상태 테이블

netfilter의 **conntrack**은 통과하는 모든 연결을 추적하는 커널 해시 테이블이다. NAT는 이 테이블 위에서 동작한다: **SNAT**(출발지 변환 — 사설 IP를 공인 IP로, 마스커레이딩), **DNAT**(목적지 변환 — 포트 포워딩, 로드밸런서 VIP).

운영 포인트: 테이블이 `net.netfilter.nf_conntrack_max`에 도달하면 **새 연결의 패킷이 그냥 드롭**되고 dmesg에 `nf_conntrack: table full, dropping packet`이 찍힌다. **Kubernetes 노드에서 실제로 흔히 발생한다** — 모든 파드 트래픽이 노드의 conntrack을 공유하는데, 연결을 많이 맺는 워크로드(짧은 요청 폭주, DNS 폭주)가 테이블을 채운다.

```bash
conntrack -C                                   # 현재 엔트리 수 (conntrack-tools)
sysctl net.netfilter.nf_conntrack_max          # 상한
sysctl net.netfilter.nf_conntrack_count        # 현재 값
dmesg | grep conntrack                         # table full 드롭 흔적
```

## 11. iptables / netfilter — 패킷이 통과하는 검문소

netfilter는 커널 패킷 경로에 박힌 5개의 훅(hook)이고, iptables는 그 훅에 규칙을 거는 도구다. **체인 = 위치(언제), 테이블 = 목적(무엇을)**으로 기억한다.

| 체인 | 시점 | 주 용도 |
|---|---|---|
| PREROUTING | 라우팅 판단 전 | DNAT (목적지 바꿔치기) |
| INPUT | 이 호스트로 들어올 때 | 호스트 방화벽 |
| FORWARD | 이 호스트를 경유할 때 | 라우터/게이트웨이 필터링 |
| OUTPUT | 이 호스트가 생성한 패킷 | 발신 제어 |
| POSTROUTING | 송신 직전 | SNAT/MASQUERADE |

테이블은 **filter**(허용/차단), **nat**(주소 변환), **mangle**(헤더 수정) 순으로 기억하면 충분하다. `iptables -L -n -v --line-numbers`, NAT는 `iptables -t nat -L -n`. 생태계는 **nftables**로 이행 중이며(`nft list ruleset`), 다수 배포판에서 iptables 명령은 nftables 백엔드 위의 호환 레이어다. kube-proxy는 Service VIP를 iptables 규칙(또는 대규모에서 IPVS 모드 — 해시 기반이라 규칙 수에 둔감)으로 구현한다 — 상세는 [[06 컨테이너 내부 구조]].

## 12. 로드밸런싱 — L4 vs L7

| | L4 (전송 계층) | L7 (애플리케이션 계층) |
|---|---|---|
| 분배 단위 | **연결**(TCP/UDP 4-tuple) | **요청**(HTTP 경로·헤더·쿠키) |
| 동작 | 패킷 포워딩/NAT, 내용을 모름 | TLS 종료, 요청 파싱 후 재작성·라우팅 |
| 구현 | IPVS, AWS NLB, HAProxy(TCP 모드) | nginx, envoy, AWS ALB |
| 장점 | 빠르고 저렴, 프로토콜 불문 | 경로 기반 라우팅, 재시도, 카나리, 관측성 |
| 단점 | 요청 단위 제어 불가 | 비용·지연 추가, 자체가 장애점 |

운영 개념 두 가지: **헬스체크**는 L4면 포트 연결 가능 여부, L7이면 `/healthz` 응답 코드 — "프로세스 살아있음"과 "트래픽 받을 준비됨(readiness)"을 구분해야 배포 중 5xx가 안 샌다. **드레이닝(draining)**은 인스턴스 제거 전 새 연결을 끊고 기존 연결이 자연 종료될 때까지 기다리는 절차 — graceful shutdown과 짝을 이뤄야 무중단 배포가 성립한다.

## 13. 핵심 커맨드 치트시트

netstat/ifconfig는 레거시(net-tools)다. 현역은 `ss`와 `ip`(iproute2).

```bash
# ss — 소켓 관찰 (netstat 대체)
ss -tlnp                  # TCP LISTEN + 포트 숫자 + 프로세스 (서버 점검 1번 커맨드)
ss -tnp                   # ESTABLISHED 연결 + 프로세스
ss -s                     # 상태별 통계 요약
ss -ti                    # 연결별 TCP 내부 정보 (rtt, cwnd, retrans)
ss -tan state time-wait '( dport = :443 )'   # 상태·포트 필터

# ip — 인터페이스/라우팅 (ifconfig·route 대체)
ip addr                   # 인터페이스와 IP
ip link                   # 링크 상태 (UP/DOWN, MTU)
ip route                  # 라우팅 테이블
ip route get 10.0.0.5     # 이 목적지로 갈 때 어떤 인터페이스/게이트웨이를 타나

# tcpdump — 패킷 직접 보기 (최후의 진실)
tcpdump -nn -i eth0 port 443                 # 포트 필터, 이름 해석 끄기(-nn)
tcpdump -i any host 10.0.0.5 and port 8080   # 호스트+포트 조합
tcpdump -i eth0 -w /tmp/cap.pcap port 53     # 파일로 저장 → Wireshark 분석
tcpdump -i eth0 'tcp[tcpflags] & (tcp-syn|tcp-rst) != 0'   # SYN/RST만

# curl — HTTP 계층 디버깅
curl -v https://api.example.com/health       # handshake·헤더 전 과정
curl -o /dev/null -s -w 'connect=%{time_connect} ttfb=%{time_starttransfer} total=%{time_total}\n' https://api.example.com/
curl --resolve api.example.com:443:10.0.0.5 https://api.example.com/   # DNS 우회해 특정 서버 직접 테스트

# 연결성·경로
nc -zv db.internal 5432   # 포트 열림 확인 (z=스캔만, v=결과 출력)
mtr -rwzbc 50 api.example.com   # traceroute+ping 결합, 홉별 손실/지연 리포트
```

`curl -w`의 타이밍 분해가 특히 유용하다: `time_connect`가 크면 네트워크/handshake 문제, `time_starttransfer`(TTFB)만 크면 서버 처리 지연 — **네트워크 탓인지 앱 탓인지를 한 줄로 가른다.**

## 14. 실무 시나리오

### connection refused vs timeout — 첫 갈림길

| 증상 | 의미 | 원인 |
|---|---|---|
| `Connection refused` (즉시) | 패킷이 **도달했고**, 호스트가 RST로 거부 | 그 포트에 LISTEN 없음(프로세스 죽음/다른 포트), 또는 방화벽 REJECT |
| `Connection timed out` (수십 초) | SYN에 **아무 응답 없음** | 방화벽 DROP, 라우팅 문제, Security Group, 호스트 다운 |

refused면 대상 서버에서 `ss -tlnp`로 리슨 확인, timeout이면 경로의 방화벽/SG/라우팅을 의심하고 `mtr`로 어느 홉에서 끊기는지 본다. **즉시 실패는 오히려 좋은 신호다 — 네트워크는 통한다는 뜻이다.**

### TIME_WAIT 폭증
① `ss -s`로 규모 확인 → ② 대부분 무해. 실제 증상(`EADDRNOTAVAIL`, connect 실패)이 있는지 먼저 확인 → ③ 어느 목적지로의 연결인지 집계(`ss -tan state time-wait | awk '{print $4}' | sort | uniq -c | sort -rn | head`) → ④ 그 경로에 connection pool/keepalive 도입이 근본 해결 → ⑤ 급한 불은 `tcp_tw_reuse=1`(발신 측) + `ip_local_port_range` 확대.

### CLOSE_WAIT 누적
① `ss -tanp state close-wait`로 프로세스 특정 → ② `ls /proc/<pid>/fd | wc -l` 추이로 fd 누수 확증 → ③ 코드에서 close 누락 경로 수색: 예외 핸들러, HTTP 응답 body 미소비, 타임아웃 후 정리 누락 → ④ 재시작은 증상 완화일 뿐, 반드시 코드를 고친다.

### 간헐적 지연 (p99만 튐)
① `curl -w`로 connect vs TTFB 분리 → ② connect가 튀면 네트워크: `ss -ti dst <ip>`로 retrans/rtt 확인, `netstat -s | grep -i retrans`를 시간차로 두 번 떠서 증가율 확인 → ③ 재전송이 있다면 경로 품질 문제 — `mtr`로 손실 홉 특정, 클라우드라면 인스턴스 네트워크 한도(ENA allowance) 확인 → ④ 재전송이 없으면 DNS(ndots, 캐시 만료)·conntrack full·앱 GC를 순서대로 배제. 재전송 타임아웃(RTO)은 최소 200ms 수준이므로 **"평소 5ms인데 가끔 200ms+"는 재전송 냄새**다.

## 면접 단골 한 줄 정리

- **3-way handshake가 왜 3번?** 양방향 시퀀스 번호 동기화를 양쪽 모두 확인해야 하므로.
- **TIME_WAIT는 왜 존재?** 옛 연결의 지연 패킷이 새 연결을 오염시키는 것 방지 + 마지막 ACK 유실 시 FIN 재전송에 응답하기 위해. 능동 종료 측이 60초.
- **TIME_WAIT vs CLOSE_WAIT?** 전자는 정상(자동 소멸), 후자는 앱이 close() 안 한 버그(영구 잔류).
- **backlog가 2개인 이유?** handshake 진행 중(SYN)과 완료 후 accept 대기는 별개의 큐 — 후자가 차는 건 앱이 느린 것.
- **HTTP/2에서도 왜 HOL이 남나?** TCP가 바이트 스트림 순서를 보장하므로 패킷 1개 유실이 전 스트림을 막는다. QUIC은 스트림별 독립 전달로 해소.
- **refused vs timeout?** refused는 RST를 받은 것(도달함), timeout은 무응답(DROP/라우팅).

다음 노트: 이 모든 동작의 진입점인 시스템 콜과 커널 구조는 [[05 커널과 시스템 콜]], 네임스페이스로 격리된 컨테이너 네트워킹은 [[06 컨테이너 내부 구조]], 측정 도구의 체계적 사용은 [[08 성능 분석과 트러블슈팅]]에서 다룬다.
