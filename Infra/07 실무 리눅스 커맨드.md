# 07 실무 리눅스 커맨드

허브: [[Infra]]

현장에서 매일 손에 잡는 커맨드를 "언제 쓰나 + 바로 복붙 가능한 예시"로 정리한 레퍼런스 노트다. 개념 노트([[01 프로세스와 스레드]], [[02 메모리 관리]], [[03 파일시스템과 스토리지]], [[04 리눅스 네트워킹]])가 "왜 그렇게 동작하는가"를 다룬다면, 이 노트는 그 개념을 손끝으로 확인하는 도구 모음이다. 장애 상황에서 man 페이지를 읽을 시간은 없다 — 여기 있는 패턴을 근육 기억으로 만들어 두는 것이 목표다. 더 깊은 성능 진단 도구(strace, perf, vmstat 계열)는 [[08 성능 분석과 트러블슈팅]]에서 이어진다.

## 1. 파일·디렉토리 — 찾고, 재고, 훑기

`find`는 "조건으로 파일을 골라내 액션을 거는" 미니 쿼리 언어다. 디렉토리 트리를 재귀로 걷기 때문에 큰 트리에서는 비용이 있다는 것만 기억하면 된다.

```bash
# 이름으로 찾기 (대소문자 무시하려면 -iname)
find /var/log -name "*.log"

# 최근 7일 내 수정된 파일 (-mtime -7 = 7일 미만, +7 = 7일 초과)
find /app -type f -mtime -7

# 100MB 넘는 파일 — 디스크 풀 장애의 첫 번째 용의자 수색
find / -xdev -type f -size +100M 2>/dev/null

# 찾은 파일마다 명령 실행 ({} 가 파일명으로 치환)
find . -name "*.pyc" -type f -exec rm {} \;

# -delete 는 빠르지만 조건 순서가 틀리면 대참사 — 반드시 -delete 없이 먼저 실행해 목록 확인
find /tmp/cache -type f -mtime +30 -print   # 1차: 눈으로 확인
find /tmp/cache -type f -mtime +30 -delete  # 2차: 삭제
```

| 커맨드 | 언제 쓰나 | 예시 |
|---|---|---|
| `stat` | mtime/atime/ctime, inode, 권한을 정확히 볼 때 | `stat /etc/passwd` |
| `ls -lahtr` | "가장 최근에 바뀐 파일이 뭐지?" — 시간 역순 정렬이라 최신이 맨 아래 | `ls -lahtr /var/log` |
| `du -sh */` | 어느 디렉토리가 디스크를 먹는지 한 단계씩 추적 | `du -sh */ \| sort -rh \| head -10` |
| `tree` | 디렉토리 구조를 한눈에 (기본 미설치인 배포판 많음) | `tree -L 2 -d /opt/app` |

`du -sh */ | sort -rh | head`는 디스크 풀 장애의 표준 수색 루틴이다. 루트에서 시작해 가장 큰 디렉토리로 한 단계씩 내려가면 범인을 몇 분 안에 찾는다.

## 2. 텍스트 처리 — 로그 분석의 빵과 버터

인프라 엔지니어의 텍스트 처리 4종 세트는 `grep`(거르기) → `awk`(필드 뽑기) → `sort | uniq -c`(세기) → `sed`(바꾸기)다. 파이프로 잇는 순간 로그가 데이터가 된다.

```bash
# grep: 재귀(-r) + 정규식(-E) + 줄번호(-n)
grep -rEn "timeout|connection refused" /var/log/app/

# 매치 제외(-v), 앞뒤 문맥(-B/-A/-C), 파일명만(-l), 개수만(-c)
grep -v "healthcheck" access.log | grep -c " 500 "
grep -C3 "OutOfMemory" app.log          # 매치 앞뒤 3줄씩
grep -rl "TODO" src/                     # 매치된 파일명만

# awk: 공백 기준 필드 추출 — access 로그에서 IP($1)와 상태코드($9)
awk '{print $1, $9}' access.log

# 합계/평균 — 3번째 필드 합산
awk '{sum += $3} END {print sum}' usage.txt

# CSV는 -F',' 로 구분자 지정
awk -F',' '{print $2}' users.csv

# sed: 치환(s/old/new/g), 제자리 수정 전 백업 습관(-i.bak)
sed -i.bak 's/DEBUG/INFO/g' app.conf

# 특정 줄 범위만 출력 (-n 과 p 조합)
sed -n '100,120p' huge.log
```

빈도 분석은 외워 둘 가치가 있는 단 하나의 파이프라인이다.

```bash
# 어떤 IP가 가장 많이 때리는가 — DDoS/크롤러 식별의 기본기
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -20
# 상태코드 분포
awk '{print $9}' access.log | sort | uniq -c | sort -rn
```

`uniq`는 **인접한** 중복만 합치므로 반드시 `sort`가 먼저 와야 한다 — 면접에서도, 실무에서도 자주 틀리는 포인트다.

| 커맨드 | 언제 쓰나 | 예시 |
|---|---|---|
| `cut` | 단순 구분자 필드 추출 (awk보다 가벼움) | `cut -d: -f1 /etc/passwd` |
| `tr` | 문자 치환/삭제 | `echo $PATH \| tr ':' '\n'` |
| `wc -l` | 줄 수 세기 | `wc -l access.log` |
| `head` / `tail` | 앞/뒤 N줄 | `tail -n 100 app.log` |
| `tail -f` | 로그 실시간 추적 (로테이션까지 따라가려면 `-F`) | `tail -f /var/log/syslog` |

`xargs`는 stdin의 목록을 다른 명령의 인자로 바꿔 준다. 파일명에 공백이 있을 수 있으므로 `find -print0 | xargs -0` 조합이 안전한 기본형이다.

```bash
# 찾은 파일들을 한 번에 grep (공백 안전)
find . -name "*.conf" -print0 | xargs -0 grep -l "listen"

# -I{} 로 인자 위치 지정, -P 로 병렬 실행 (4개 동시)
cat hosts.txt | xargs -I{} -P4 ssh {} "uptime"
```

`jq`는 JSON 시대의 awk다. API 응답과 구조화 로그를 다룰 때 필수.

```bash
curl -s https://api.example.com/pods | jq '.items[].metadata.name'
jq '.field' data.json                          # 필드 추출
jq '.[] | select(.status == "Failed")' a.json  # 조건 필터
```

## 3. 프로세스·서비스 — 무엇이 돌고 있고, 누가 관리하는가

프로세스 모델의 배경은 [[01 프로세스와 스레드]] 참조. 여기서는 조회와 제어만.

```bash
ps aux            # BSD 스타일: %CPU, %MEM 포함 — 자원 사용 볼 때
ps -ef            # System V 스타일: PPID 포함 — 부모 추적할 때
ps auxf           # f = 포레스트(트리) — "이 프로세스 누가 띄웠지?"
pgrep -f "gunicorn"      # 명령행 전체(-f)로 PID 검색
pkill -f "stale-worker"  # 같은 패턴으로 시그널 전송
```

`kill -15`(SIGTERM)는 "정리하고 종료해 달라"는 요청이고, `kill -9`(SIGKILL)는 커널이 프로세스를 즉시 제거하는 명령이라 핸들러가 돌 기회조차 없다. **항상 -15 먼저, 몇 초 기다린 뒤 -9** — 면접 단골이자, -9부터 날리면 임시파일·락·좀비 커넥션이 남는 실무 이유까지 함께 답할 수 있어야 한다. 참고로 SIGKILL로도 안 죽는 D state(uninterruptible sleep) 프로세스는 I/O 대기 중이라 커널도 어쩌지 못한다 — [[05 커널과 시스템 콜]] 참조.

`top` 안에서의 단축키: `M`(메모리순 정렬), `P`(CPU순), `c`(전체 명령행 표시), `1`(코어별 CPU 표시). `htop`은 같은 정보를 컬러 트리로 보여주고 마우스도 된다(기본 미설치인 곳 많음).

백그라운드 실행은 두 갈래다. `nohup cmd &`는 터미널이 끊겨도 살아남는 임시방편이고, `systemd-run --unit=myjob cmd`는 일회성 작업조차 systemd 유닛으로 등록해 journald 로깅·자원 제한·상태 조회를 공짜로 얻는다. 서버에 남길 거라면 후자가 정석이다.

```bash
systemctl status nginx            # 상태 + 최근 로그 몇 줄
systemctl restart nginx
systemctl enable --now nginx      # 부팅 자동시작 + 즉시 시작을 한 번에
systemctl cat nginx               # 실제 적용 중인 유닛 파일 내용
systemctl edit nginx              # 오버라이드 드롭인 생성 (원본 안 건드림)
systemctl list-units --failed     # 죽어 있는 유닛 일괄 점검
```

로그는 journalctl로 본다. journald는 바이너리 저널이라 grep 전에 필터 옵션을 먼저 쓰는 게 빠르다.

```bash
journalctl -u nginx -f                      # 특정 유닛 follow
journalctl -u nginx --since "1 hour ago"    # 시간 범위
journalctl -p err -b                        # 이번 부팅의 에러 이상만
journalctl -k                               # 커널 메시지 (dmesg 대용) — OOM 킬 확인
journalctl --disk-usage                     # 저널이 디스크 얼마나 먹는지
```

OOM 킬러가 내 프로세스를 죽였는지 확인하는 `journalctl -k | grep -i "out of memory"`는 [[02 메모리 관리]]의 개념이 실무로 내려오는 지점이다.

`crontab -e`의 형식은 `분 시 일 월 요일 명령` — 앞의 다섯 개가 시간 필드이고 그 뒤에 명령이 온다. `0 3 * * * /opt/backup.sh`는 매일 03:00. 요일과 일을 동시에 지정하면 OR로 동작한다는 함정이 있으니 둘 중 하나만 쓰는 게 안전하다. 반복 관찰은 `watch -n1 'ss -s'`처럼 `watch`가 편하다.

## 4. 네트워크 — 연결을 눈으로 확인하기

배경 이론은 [[04 리눅스 네트워킹]] 참조. `netstat`/`ifconfig`는 레거시이고 현역은 `ss`/`ip`다.

```bash
ss -tlnp                # 리슨 중인 TCP 포트 + 프로세스 — "8080 누가 잡고 있어?"
ss -tnp                 # 현재 맺어진 연결
ss -s                   # 소켓 통계 요약 (TIME-WAIT 폭증 감지)
ss state time-wait      # TIME-WAIT 소켓만 — 커넥션 풀 미사용 서비스의 단골 증상
ip a                    # 인터페이스와 IP (ifconfig 대체)
ip r                    # 라우팅 테이블 — "이 패킷 어디로 나가지?"
ip link                 # 링크 상태 (UP/DOWN)
ip -s link              # RX/TX 에러·드롭 카운트 — NIC 레벨 문제 1차 확인
```

DNS와 HTTP 진단:

```bash
dig +short api.example.com           # A 레코드만 깔끔하게
dig @8.8.8.8 api.example.com         # 특정 리졸버에 직접 질의 — 내부 DNS 의심될 때
dig MX example.com +short            # MX/TXT 등 레코드 타입 지정
curl -v https://api.example.com/health        # TLS 핸드셰이크까지 전 과정 출력
curl -sS -I https://example.com               # 헤더만 (-I), 조용하되 에러는 표시(-sS)
curl -X POST -d '{"k":"v"}' -H "Content-Type: application/json" http://localhost:8080/api
curl --connect-timeout 3 http://10.0.0.5:8080/   # 행 방지 — 스크립트에선 필수
# 단계별 타이밍 — "느린 게 DNS냐 TLS냐 서버냐" 를 숫자로
curl -sS -o /dev/null -w 'dns=%{time_namelookup} tcp=%{time_connect} tls=%{time_appconnect} ttfb=%{time_starttransfer} total=%{time_total}\n' https://example.com
```

`wget`은 파일 다운로드(재시도·이어받기 `-c`)에, `curl`은 API 디버깅에 강하다.

```bash
# 패킷 캡처 — 마지막 수단이지만 가장 확실한 증거
tcpdump -i any -nn port 443            # 모든 인터페이스, 이름 해석 끄기(-nn)
tcpdump -i any -nn -w out.pcap host 10.0.0.5   # 파일로 떠서 Wireshark 분석 (옵션은 필터식보다 앞에)

# 포트 열림 확인 / 임시 리스너
nc -zv db.internal 5432    # 연결만 시도(-z) + 결과 출력(-v) — 방화벽 확인 1순위
nc -l 8080                 # 8080에서 리슨 — "방화벽이 문제냐 앱이 문제냐" 분리 실험

mtr --report google.com    # ping + traceroute 결합 — 경로상 어느 홉에서 손실되는지
```

`nc -zv`로 되는데 앱이 안 붙는다면 앱 설정 문제이고, `nc`조차 안 되면 네트워크/방화벽 문제다 — 이 이분법이 네트워크 트러블슈팅의 절반이다.

## 5. 디스크·열린 파일 — "Device full인데 큰 파일이 없어요"

파일시스템 내부는 [[03 파일시스템과 스토리지]] 참조.

```bash
df -h        # 파일시스템별 사용량 — 디스크 풀 1차 확인
df -i        # inode 사용량 — "용량은 남았는데 No space left" 의 정체
du --max-depth=1 -h /var | sort -rh    # /var 아래 한 단계씩 범인 추적
lsblk -f     # 블록 디바이스 트리 + 파일시스템 타입 + UUID
findmnt /data          # 마운트 옵션 확인 (mount 출력보다 깔끔)
blkid /dev/sdb1        # 파티션 UUID/타입 — fstab 작성할 때
fuser -v /mnt/data     # 마운트 해제를 막는 프로세스 색출
```

`lsof`(list open files)는 "리눅스에서 모든 것은 파일"이라는 원칙 덕분에 소켓·디바이스·일반 파일을 한 도구로 추적한다.

```bash
lsof -i :8080          # 8080 포트를 잡은 프로세스
lsof -p 1234           # PID 1234가 연 모든 파일
lsof +D /var/log       # 디렉토리 아래 열린 파일 전부
lsof | grep deleted    # 삭제됐지만 아직 열려 있는 파일 — 디스크 풀 미스터리의 고전
```

마지막 줄은 면접 단골 시나리오의 답이다: **로그 파일을 `rm`으로 지웠는데 `df`가 줄지 않는 이유**는, 디렉토리 엔트리만 사라졌을 뿐 프로세스가 파일 디스크립터를 쥐고 있어 inode와 블록이 해제되지 않았기 때문이다. 해법은 파일 삭제가 아니라 `truncate -s 0`으로 비우는 것이고, 이미 지운 뒤라면 열린 fd 경로(`truncate -s 0 /proc/<PID>/fd/<FD>`)로 비우거나 프로세스 재시작/로그 로테이션을 한다.

## 6. 시스템 정보 — 서버에 접속하면 처음 30초

낯선 서버에 들어가면 반사적으로 치는 정찰 세트다.

| 커맨드 | 알 수 있는 것 |
|---|---|
| `uname -a` | 커널 버전, 아키텍처 |
| `cat /etc/os-release` (또는 `lsb_release -a`) | 배포판과 버전 |
| `uptime` | 가동 시간 + load average 1/5/15분 |
| `free -h` | 메모리 — available을 봐야 함 ([[02 메모리 관리]]의 페이지 캐시 때문) |
| `nproc` / `lscpu` | CPU 코어 수 / 토폴로지·캐시·NUMA |
| `hostnamectl` | 호스트명 + 가상화 여부(KVM/컨테이너)까지 한 번에 |
| `timedatectl` | 시간대·NTP 동기화 — 분산시스템에서 시계 어긋남은 치명적 |
| `w` / `who` / `last` | 지금 누가 접속해 있나 / 최근 로그인 이력 |
| `id` | 현재 사용자의 uid/gid/그룹 — sudo 권한 의심될 때 |

load average가 코어 수(`nproc`)보다 큰 상태가 지속되면 포화다 — 단, 리눅스의 load는 CPU 실행 대기뿐 아니라 D state(디스크 I/O 대기)도 포함하므로 "load 높음 = CPU 부족"이라고 단정하면 안 된다. 이 구분은 [[08 성능 분석과 트러블슈팅]]에서 이어진다.

## 7. 권한·소유 — 8진수 세 자리의 의미

`chmod 755`는 rwxr-xr-x, 즉 소유자 7(4+2+1=rwx), 그룹 5(r-x), 기타 5(r-x)다. `644`는 rw-r--r-- — 일반 파일의 표준값. 디렉토리의 x는 "통과(진입) 권한"이라 디렉토리에는 보통 755, 파일에는 644를 준다.

```bash
chmod u+x deploy.sh            # 소유자에게 실행 권한만 추가 (8진수보다 의도가 명확)
chmod -R 755 /opt/app          # -R 재귀 — 파일까지 전부 실행권한이 붙으므로 신중히
chown -R appuser:appgroup /opt/app
umask                          # 새 파일의 기본 권한 마스크 (022면 파일 644, 디렉토리 755)
sudo -i                        # 루트 로그인 셸 (환경까지 루트)
sudo -u postgres psql          # 다른 사용자로 명령 실행
setfacl -m u:deploy:rx /var/log/app   # 소유자/그룹 구조를 못 바꿀 때 ACL로 구멍 내기
```

한 줄씩만: **setuid**(`chmod u+s`)는 실행 시 파일 소유자 권한으로 도는 비트(`passwd`가 /etc/shadow를 고칠 수 있는 이유), **스티키 비트**(`chmod +t`)는 /tmp처럼 모두가 쓰는 디렉토리에서 "자기 파일만 지울 수 있게" 막는 비트다.

## 8. 압축·전송 — tar와 rsync만 제대로

`tar`는 옵션 세 글자만 외우면 된다: **c**reate/e**x**tract/lis**t** + **z**(gzip) + **f**(파일명).

```bash
tar czf app-backup.tar.gz /opt/app --exclude='*.log' --exclude='node_modules'
tar xzf app-backup.tar.gz -C /restore/    # -C 로 풀 위치 지정
tar tzf app-backup.tar.gz | head          # 풀기 전에 내용 확인하는 습관
```

압축 알고리즘 한 줄 비교: gzip은 어디에나 있고, zstd는 비슷한 압축률에 압축·해제가 훨씬 빨라 새로 만드는 파이프라인이라면 zstd(`tar --zstd -cf a.tar.zst dir/`)가 합리적 기본값이다.

`rsync`는 델타 전송(바뀐 부분만)과 재개가 되기 때문에 scp를 사실상 대체했다. scp는 중단되면 처음부터, rsync는 이어서 간다.

```bash
rsync -avzP ./build/ deploy@web1:/opt/app/     # archive+verbose+압축+진행률/재개
rsync -avz --dry-run --delete ./dist/ web1:/srv/www/   # --delete 는 반드시 --dry-run 먼저
```

`--delete`는 대상에서 원본에 없는 파일을 지운다 — 원본·대상 경로를 뒤집어 쓰면 배포 디렉토리가 통째로 증발한다. **`--dry-run`을 먼저 치는 것을 습관이 아니라 규칙으로** 삼는다. 트레일링 슬래시도 의미가 다르다: `src/`는 "내용물을", `src`는 "디렉토리 자체를" 복사한다. 대화형 탐색이 필요하면 `sftp host`로 들어가 `ls/get/put`을 쓴다.

## 9. 패키지 — 데비안계와 레드햇계 대응표

| 작업 | Debian/Ubuntu (apt) | RHEL/Fedora (dnf/yum) |
|---|---|---|
| 인덱스 갱신 | `apt update` | `dnf makecache` (`check-update`는 갱신 겸 업데이트 목록 확인) |
| 설치 | `apt install -y nginx` | `dnf install -y nginx` |
| 설치 목록 | `apt list --installed` | `dnf list installed` |
| 패키지 검색 | `apt search nginx` | `dnf search nginx` |
| 버전·출처 확인 | `apt-cache policy nginx` | `dnf info nginx` |
| 파일 → 패키지 역추적 | `dpkg -S /usr/sbin/nginx` | `rpm -qf /usr/sbin/nginx` |
| 패키지 → 파일 목록 | `dpkg -L nginx` | `rpm -ql nginx` |

`dpkg -l | grep nginx`는 설치 여부 빠른 확인, `apt-cache policy`는 "어느 저장소의 어떤 버전이 설치 후보인가"를 보여줘 버전 고정/저장소 충돌 디버깅에 쓴다.

## 10. 셸 생산성 — 손가락을 아끼는 기술

| 기법 | 효과 | 예시 |
|---|---|---|
| `Ctrl-R` | 히스토리 역방향 검색 — 가장 투자 대비 효율 좋은 단축키 | `Ctrl-R` 후 `rsync` 입력 |
| `!!` | 직전 명령 재실행 | `sudo !!` (권한 없어 실패했을 때) |
| `!$` | 직전 명령의 마지막 인자 | `mkdir /opt/new` 실행 후 `cd !$` (한 줄로 `&&` 연결 시엔 `cd $_`) |
| `alias` | 자주 쓰는 조합 축약 | `alias ll='ls -lah'` |
| `tee` | 화면에 보면서 파일에도 저장 | `make 2>&1 \| tee build.log` |
| `time` | 명령 수행 시간 측정 | `time ./migrate.sh` |
| `timeout` | 행 걸리는 명령 강제 제한 | `timeout 30s curl http://flaky/` |
| `yes` | 확인 프롬프트 자동 응답 | `yes \| ./installer.sh` |
| `seq` | 숫자 시퀀스 생성 | `seq 1 10` |
| `date +%F` | YYYY-MM-DD — 백업 파일명에 | `tar czf "bk-$(date +%F).tar.gz" /etc` |

bash 루프 한 줄은 임시 자동화의 기본기다.

```bash
for h in web1 web2 web3; do ssh "$h" "systemctl is-active nginx"; done
for i in $(seq 1 5); do curl -sS -o /dev/null -w '%{http_code}\n' http://localhost:8080/health; sleep 1; done
```

`tmux`는 SSH 세션이 끊겨도 작업이 살아남게 한다. 최소 생존 커맨드 네 개: `tmux new -s work`(세션 생성), `Ctrl-b d`(분리 — 작업은 계속 돈다), `tmux ls`(목록), `tmux attach -t work`(복귀). 장시간 마이그레이션·배포는 반드시 tmux 안에서 돌린다. 작업 증적이 필요하면 `script session.log`로 터미널 입출력 전체를 기록하고 `exit`로 끝낸다.

## 11. 안전 수칙 — 사고는 항상 같은 자리에서 난다

> [!danger] 프로덕션 안전 수칙
> - **`rm -rf` 변수 확장 사고**: `rm -rf "$DIR/"` 에서 `$DIR`이 비어 있으면 `rm -rf /` 가 된다. 변수는 반드시 따옴표로 감싸고, `rm -rf "${DIR:?DIR is empty}/"` 처럼 `:?` 가드를 걸거나, 스크립트 머리에 `set -u` 를 둔다.
> - **파괴적 플래그는 리허설 먼저**: `rsync --delete`, `find -delete` 는 같은 명령에서 `--dry-run` / `-print` 로 먼저 목록을 확인한 뒤 실행한다. 대화식 확인이 되는 `rm -i`, `mv -i` 도 alias로 걸어 둘 가치가 있다.
> - **strace/tcpdump 오버헤드**: strace는 모든 시스템 콜마다 대상 프로세스를 멈추는 구조라 고부하 서비스에서 수십 배 느려질 수 있다([[05 커널과 시스템 콜]] 참조). tcpdump도 트래픽 많은 NIC에서 CPU와 디스크를 잡아먹는다. 프로덕션에서는 필터를 좁게, 시간은 짧게, 가능하면 샘플링 기반 도구([[08 성능 분석과 트러블슈팅]])를 먼저.
> - **sudo 파이프 함정**: `sudo echo "x" > /etc/protected` 는 실패한다 — 리다이렉션은 sudo 이전에 **내 셸**이 처리하기 때문이다. 정답은 `echo "x" | sudo tee /etc/protected` (덧붙이기는 `tee -a`).
> - **`kill -9` 는 마지막 수단**: SIGTERM → 대기 → SIGKILL 순서. 정리 핸들러를 건너뛰면 락 파일·하다 만 쓰기가 남는다.

## 면접 단골 한 줄 정리

- "지운 로그 파일인데 디스크가 안 비워져요" → 열린 fd가 inode를 붙잡고 있음. `lsof | grep deleted`, 해법은 truncate 또는 재시작.
- "용량 남았는데 No space left on device" → inode 고갈. `df -i`로 확인, 수백만 개 작은 파일이 범인.
- `kill -15` vs `-9` → graceful shutdown 기회의 유무. -9는 핸들러조차 안 돈다.
- `sudo echo > file`이 안 되는 이유 → 리다이렉션은 셸이 sudo보다 먼저 처리. `| sudo tee`로 해결.
- `sort` 없이 `uniq` → 인접 중복만 제거되므로 빈도 분석이 틀어진다.
- scp 대신 rsync → 델타 전송·재개·`--dry-run` 리허설이 가능.

컨테이너 안에서 이 커맨드들이 어떻게 달라 보이는지(네임스페이스로 격리된 `ps`, `ss` 출력)는 [[06 컨테이너 내부 구조]]에서 다룬다.
