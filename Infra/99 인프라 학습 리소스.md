# 99 인프라 학습 리소스
허브: [[Infra]]

인프라 학습이 실패하는 이유는 자원이 부족해서가 아니라 **너무 많아서**다. 좋은 책 일곱 권을 동시에 펼치면 일곱 권 다 1장에서 멈춘다. 이 노트는 "무엇이 좋은가"의 목록이 아니라 **"지금 단계에서 무엇 하나만 읽을 것인가"를 결정해 주는 필터**다. 각 자원에는 반드시 "왜/언제 읽는가"를 붙였고, 단계별로 한 번에 들고 가는 책은 1~2권을 넘지 않게 설계했다. 전체 커리큘럼의 맥락은 [[00 인프라 학습 로드맵]]을 먼저 보라.

핵심 원칙 세 가지. **첫째, 읽기와 손은 항상 1:1** — 책 한 챕터를 읽으면 VM에서 한 번은 만져본다. **둘째, 레퍼런스형 책은 통독하지 않는다** — TLPI를 1쪽부터 읽는 것은 사전을 A부터 읽는 것과 같다. **셋째, 같은 단계의 자원을 중복해서 사지 않는다** — OSTEP을 읽는 중이면 다른 OS 입문서는 필요 없다.

## 1. 책 — 우선순위 순

### 1순위 · OSTEP (Operating Systems: Three Easy Pieces)

- **무엇**: Remzi & Andrea Arpaci-Dusseau 부부가 쓴 OS 교과서. 공식 사이트에서 **챕터별 PDF 전부 무료**.
- **왜 이 책인가**: OS 전체를 virtualization(CPU·메모리) / concurrency / persistence 세 조각으로 나눈 구성이 멘탈 모델 형성에 압도적으로 유리하다. 각 챕터가 "문제 → 해법의 진화" 서사라서, 단순 암기가 아니라 "왜 이렇게 설계됐나"를 답할 수 있게 된다 — 이게 면접에서 주니어와 시니어를 가르는 차이다.
- **언제·어떻게**: 지금 당장. [[01 프로세스와 스레드]]와 [[02 메모리 관리]]를 공부할 때 해당 챕터(Virtualization 파트)를 병행하면 노트가 요약본, OSTEP이 본문 역할을 한다. 챕터가 짧아(평균 10~20쪽) 출퇴근 시간에도 한 챕터씩 끊긴다.
- **건너뛰어도 되는 것**: 숙제(homework) 시뮬레이터는 선택. 대신 본문의 "Crux of the problem" 박스는 절대 건너뛰지 말 것 — 면접 질문이 거의 그대로 거기서 나온다.

### 2순위 · How Linux Works 3판 (Brian Ward)

- **무엇**: 부팅 → init/systemd → 디스크·파일시스템 → 네트워크 → 셸까지, **운영자 시점에서 리눅스 한 대가 어떻게 굴러가는지** 전체 그림을 그려주는 책.
- **왜 이 책인가**: OSTEP이 "OS는 왜 이렇게 설계됐나"라면 이 책은 "내 앞의 이 리눅스 박스에서 그게 어디에 있나"다. 6년차 백엔드라면 단편 지식은 이미 많을 텐데, 이 책이 그 파편들을 한 장의 지도로 꿰어준다.
- **언제**: OSTEP virtualization 파트를 끝낸 직후, [[07 실무 리눅스 커맨드]] 실습과 병행. 읽으면서 모든 커맨드를 VM에서 직접 쳐보는 것이 전제다.

### 3순위(레퍼런스) · The Linux Programming Interface (TLPI, Michael Kerrisk)

- **무엇**: 리눅스 syscall과 glibc API의 사실상 표준 사전. 저자 Kerrisk는 오랫동안 리눅스 man-pages 프로젝트의 메인테이너였다.
- **왜**: `fork()` 뒤에서 정확히 무슨 일이 일어나는지, signal과 thread가 섞이면 왜 지옥이 되는지 — man 페이지가 "무엇"만 말할 때 TLPI는 "왜"와 함정까지 말해준다.
- **언제·어떻게**: **통독 금지.** 1500쪽이 넘는다. [[05 커널과 시스템 콜]]을 공부하다가 특정 syscall이 궁금할 때, 혹은 [[06 컨테이너 내부 구조]]에서 namespaces/cgroups 관련 호출을 파고들 때 해당 챕터만 펼치는 사전이다. 책상에 두고 평생 쓰는 책.

### 2부째 메인 책 · Systems Performance 2판 (Brendan Gregg)

- **무엇**: 성능 분석의 바이블. [[08 성능 분석과 트러블슈팅]]에서 다루는 USE method, 도구 지도, 방법론의 **원전**이다.
- **왜**: 도구 사용법 책이 아니라 "관찰 가능성(observability)을 어떻게 사고하는가"의 책. CPU·메모리·디스크·네트워크 각 챕터가 "아키텍처 → 방법론 → 도구" 순서라, 도구가 바뀌어도 썩지 않는 지식이다.
- **언제**: 운영 단계를 마치고 심화에 들어갈 때. 1~2장(방법론)과 자기 직무에 가까운 리소스 챕터부터 읽고 나머지는 필요할 때.

## 3. 강의

- **MIT The Missing Semester** — 셸, 도구 체인, dotfiles, 디버깅 같은 "아무도 안 가르쳐주는 생활 기술". 강의당 1시간, 전부 무료. **운영 단계 첫 주에** 몰아서 듣기 좋다. 6년차라면 절반은 알겠지만 나머지 절반이 생산성을 바꾼다.
- **MIT 6.S081 (Operating System Engineering)** — xv6라는 교육용 미니 커널을 **직접 고치는** 랩 중심 강의. 자료·랩 전부 공개. 시간이 많이 들어서 **하드코어 선택**이다. "커널 코드를 읽고 수정해 봤다"는 경험은 면접에서 강력하지만, 전환 일정이 빠듯하면 과감히 생략하고 OSTEP + 실습으로 대체해도 된다.
- **Linux Foundation LFS101 (Introduction to Linux)** — 무료 입문 강의. 리눅스를 거의 처음 만지는 동료에게 권할 때 쓰는 카드이고, 이 로드맵을 따라온 사람에게는 쉬울 수 있다. 자기 진단용으로 목차만 훑어도 좋다.

## 4. 연습 환경 — 읽기만 하면 늘지 않는다

머리로 아는 것과 손이 기억하는 것 사이의 간극이 인프라 직무의 전부다. 환경 구축에 하루 이상 쓰지 말고, 아래 중 하나로 빠르게 시작한다.

### 로컬 VM (macOS에서 리눅스)

| 도구 | 특징 | 추천 상황 |
|---|---|---|
| **Lima** | 무료·CLI 중심, `limactl`로 가벼운 리눅스 VM | 기본 선택. Homebrew로 설치 즉시 사용 |
| **OrbStack** | 빠르고 매끈한 상용 도구(개인 무료 티어), Docker+VM 통합 | Docker도 같이 쓸 때 체감 편의 최고 |
| **Multipass** | Canonical 제공, Ubuntu VM 특화 | Ubuntu 환경을 정확히 재현하고 싶을 때 |

```bash
# Lima로 1분 안에 리눅스 셸 얻기
brew install lima
limactl start default     # 기본 템플릿으로 VM 생성·기동
limactl shell default     # VM 셸 진입 — 여기서 마음껏 부순다
```

macOS 호스트에서 직접 `tc`나 cgroups를 만질 수 없으므로(커널이 다르다), **리눅스 실습은 무조건 VM 안에서** 한다는 원칙을 세워라. 이 원칙은 사고 방지 효과도 겸한다.

### 클라우드

프리티어(AWS/GCP/Oracle 등)나 **스팟 인스턴스**로 진짜 서버 환경을 경험한다. 로컬 VM과의 결정적 차이는 "ssh로만 접근 가능하고, 망가지면 손해인" 긴장감이다. 스팟은 시간당 비용이 매우 싸지만 회수될 수 있다 — 오히려 그 덕에 "언제든 죽는 서버"를 전제로 한 운영 습관(상태를 인스턴스에 남기지 않기)이 몸에 밴다.

### 일부러 장애 만들기 — 08 플레이북 셀프 훈련

장애 대응 능력은 장애를 겪은 횟수에 비례한다. 운에 맡기지 말고 **VM 안에서 직접 주입**한다. 절차: ① 장애 주입 → ② 주입 사실을 모른다고 가정하고 [[08 성능 분석과 트러블슈팅]] 플레이북대로 진단 → ③ 원인 특정까지 걸린 시간 기록.

```bash
# CPU 포화 — load average와 %usr가 어떻게 움직이는지 관찰
stress-ng --cpu 4 --timeout 120s

# 메모리 압박 — 페이지 회수, swap, 최악엔 OOM killer까지
stress-ng --vm 2 --vm-bytes 1G --timeout 120s

# 디스크 가득 채우기 — "No space left on device"의 실제 증상 학습
# /tmp은 tmpfs(RAM)인 배포판이 많으니 디스크 위 경로(/var/tmp)를 쓰고, count 없이 ENOSPC가 날 때까지 채운다
dd if=/dev/zero of=/var/tmp/fill.img bs=1M
df -h /var/tmp && rm /var/tmp/fill.img   # 관찰 후 반드시 정리

# 네트워크 지연 주입 — tc netem으로 100ms 추가 (인터페이스명은 VM에 맞게)
sudo tc qdisc add dev eth0 root netem delay 100ms
ping -c 3 8.8.8.8                # 지연 확인
sudo tc qdisc del dev eth0 root  # 원복 — 주입했으면 반드시 지운다
```

| 주입한 장애 | 진단에 쓸 도구 | 찾아야 할 시그널 |
|---|---|---|
| CPU 포화 | `top`, `vmstat`, `pidstat` | load average 상승, run queue 길이, 범인 프로세스 |
| 메모리 압박 | `free`, `vmstat`, `dmesg` | swap in/out, page reclaim, OOM kill 로그 |
| 디스크 풀 | `df`, `du`, `lsof` | 사용률 100%, 지워졌지만 열려 있는 파일 |
| 네트워크 지연 | `ping`, `ss`, `mtr` | RTT 증가, 재전송, 어느 홉에서 지연되는지 |

**경고**: 이 실험들은 반드시 일회용 VM에서만 한다. 프로덕션은 물론이고 개인 작업용 호스트에서도 금지 — 특히 `dd`와 `tc`는 손가락이 미끄러지면 복구 비용이 크다.

## 5. 자격증 — 수단이지 목표가 아니다

[[00 인프라 학습 로드맵]]에서 정리한 입장 그대로다: **자격증은 학습을 강제하는 장치이자 서류 필터를 통과하는 수단이고, 실력의 본질은 포트폴리오(직접 굴려본 시스템과 장애 기록)다.** 그 전제 위에서 두 개만 의미가 있다.

- **LFCS (Linux Foundation Certified System Administrator)** — 객관식이 아니라 **실제 터미널에서 과제를 수행하는 실기 시험**. 그래서 준비 과정 자체가 [[07 실무 리눅스 커맨드]] 훈련과 일치한다. "리눅스 운영 가능"을 증명할 가장 정직한 시그널.
- **CKA (Certified Kubernetes Administrator)** — 역시 실기. AI 인프라 직무의 사실상 공용 플랫폼이 K8s이므로 [[AI-Infra]] 로드맵과 직결된다. 단, [[06 컨테이너 내부 구조]]를 건너뛰고 CKA부터 가면 "kubectl은 치는데 컨테이너가 뭔지 모르는" 상태가 된다 — 순서를 지킬 것.

면접에서 자격증을 물어보면 "따는 과정에서 무엇을 굴려봤는지"로 답하라. 자격증 이름만 나열하는 답변은 오히려 역효과다.

## 6. 뉴스레터·커뮤니티 — 흐름을 놓치지 않기

학습 단계가 끝나도 인프라는 계속 움직인다. 다만 구독 전략이 중요하다: **다 읽으려 하면 3주 안에 전부 안 읽게 된다.** 제목만 스캔하고 일주일에 한두 편만 깊이 읽는 것이 지속 가능한 유일한 방식이다.

- **LWN 주간 에디션** — 커널에서 지금 무엇이 논쟁 중인지. 심화 단계 이후 최고의 "커널 감각" 유지 장치.
- **Cloudflare 블로그** — 대규모 네트워크 장애·최적화의 심층 부검 기사. [[04 리눅스 네트워킹]]에서 배운 개념이 실전에서 어떻게 터지는지 보여주는 최고의 사례집.
- **Netflix TechBlog** — 성능 엔지니어링과 대규모 운영 사례. Gregg가 오래 몸담았던 조직답게 성능 분석 글의 밀도가 높다.
- **GeekNews (국내)** — 해외 기술 뉴스의 한국어 큐레이션. 위 영문 소스들의 진입 장벽을 낮춰주는 1차 필터로 쓰기 좋다.

## 7. 추천 학습 경로 — 다시 한 줄로

> **입문**(OSTEP + jvns로 멘탈 모델) → **운영**(How Linux Works + [[07 실무 리눅스 커맨드]]·[[08 성능 분석과 트러블슈팅]] 실습으로 손에 붙이기) → **심화**(TLPI를 사전 삼아 Systems Performance·eBPF로 깊이) → **특화**([[AI-Infra]]에서 GPU·K8s로 재적용)

단계마다 들고 가는 메인 책은 한 권, 보조는 한 개. 막히면 더 쉬운 자료(jvns)로 내려갔다가 다시 올라오고, 궁금하면 더 깊은 자료(Linux Insides, 6.S081)로 내려가되 메인 트랙을 잃지 않는다. 기간은 사람마다 다르니 단정하지 않는다 — 다만 "읽은 챕터 수"가 아니라 **"VM에서 재현해 본 시나리오 수"를 진도의 단위**로 삼으면 속도가 정직해진다.
