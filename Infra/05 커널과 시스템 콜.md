# 05 커널과 시스템 콜

허브: [[Infra]]

애플리케이션 개발자에게 커널은 "보이지 않는 런타임"이다. 6년간 백엔드 코드를 짰다면 사실 매일 수백만 번의 시스템 콜을 호출해 온 셈인데, 그 경계가 어떻게 생겼는지 모르면 "왜 이 서버만 느리지", "왜 connection이 drop되지", "왜 Too many open files가 뜨지" 같은 질문 앞에서 막힌다. 이 노트는 유저공간 코드가 커널과 만나는 **모든 접점** — 시스템 콜, 인터럽트, `/proc`, sysctl, rlimits, eBPF — 을 한 바퀴 돈다.

## 1. 커널이 하는 일 — 추상화와 보호

커널의 역할은 두 단어로 요약된다. **자원 추상화**와 **보호(protection)**.

- **추상화**: 하드웨어의 날것(CPU 코어, DRAM 주소, 디스크 섹터, NIC 레지스터)을 프로그래머가 쓸 수 있는 개념(프로세스, 가상 메모리, 파일, 소켓)으로 바꾼다. `read()` 한 줄 뒤에는 페이지 캐시, 블록 레이어, 디바이스 드라이버, DMA가 숨어 있다.
- **보호**: 한 프로세스가 다른 프로세스의 메모리를 읽거나, 임의로 하드웨어를 건드리지 못하게 막는다. 이것이 없으면 멀티테넌시도, 컨테이너도, 클라우드도 성립하지 않는다.

보호의 하드웨어적 기반이 **CPU 특권 레벨**이다. x86에서 커널은 ring 0, 유저 코드는 ring 3에서 돈다 — 유저 코드는 특권 명령(페이지 테이블 조작, I/O 포트 접근 등)을 실행하면 즉시 fault가 난다. 그래서 메모리 주소 공간도 **user space**와 **kernel space**로 갈리고, 둘 사이를 건너는 유일한 정식 통로가 시스템 콜이다.

리눅스는 **모놀리식 커널**이다. 스케줄러, 파일시스템, 네트워크 스택, 드라이버가 전부 하나의 커널 주소 공간에서 함수 호출로 엮여 돈다(마이크로커널처럼 메시지 패싱으로 분리되어 있지 않다). 대신 **동적 모듈**(loadable kernel module)로 드라이버나 파일시스템을 부팅 후에 끼워 넣을 수 있어 유연성을 보완한다 — 뒤의 `lsmod`/`modprobe` 절에서 다룬다.

> 비유: 커널은 호텔의 프런트 데스크다. 투숙객(프로세스)은 객실(가상 주소 공간) 안에서는 자유지만, 다른 방 열쇠·전기실·보일러실(하드웨어)에 가려면 반드시 프런트(시스템 콜)를 거쳐야 한다.

## 2. 시스템 콜 — 경계를 건너는 비용

### 호출 경로

애플리케이션이 `read(fd, buf, n)`을 부르면 실제로 일어나는 일:

1. **glibc 래퍼 함수** 호출 — 대부분의 "시스템 콜"은 사실 C 라이브러리 함수다. 인자를 레지스터 규약에 맞게 배치하고 syscall 번호를 세팅한다.
2. **`syscall` 명령 실행** (x86-64 기준) — CPU가 ring 3 → ring 0으로 전환하고, 커널의 syscall 엔트리 포인트로 점프한다.
3. **커널 모드 처리** — syscall 번호로 핸들러 테이블을 찾아 해당 커널 함수 실행. 유저가 넘긴 포인터는 검증 후 복사(`copy_from_user`/`copy_to_user`).
4. **복귀** — 반환값을 레지스터에 싣고 ring 3로 돌아온다. 실패 시 커널은 음수 errno를 반환하고, glibc가 이를 `errno` 변수 + `-1` 반환으로 바꿔 준다.

### 왜 비용이 드는가

모드 전환 자체가 수십~수백 ns 수준의 직접 비용을 갖고, 그보다 더 아픈 건 간접 비용이다 — 레지스터 저장/복원, CPU 파이프라인 플러시, 캐시/TLB 오염, 그리고 Meltdown 이후 추가된 완화 기법(KPTI 등)의 오버헤드. 그래서 고성능 시스템 설계의 공통 원칙이 **"syscall 횟수를 줄여라"**다: 버퍼링된 I/O, `writev` 같은 벡터 I/O, batch 처리, 그리고 궁극적으로 io_uring 같은 비동기 인터페이스가 모두 이 원칙의 산물이다.

### vDSO — syscall 없는 syscall

`gettimeofday()`나 `clock_gettime()`은 초당 수만 번 불리는데도 빠르다. 비밀은 **vDSO(virtual dynamic shared object)** — 커널이 모든 프로세스의 주소 공간에 매핑해 주는 작은 공유 라이브러리다. 커널이 시간 정보를 공유 메모리 페이지에 계속 갱신해 두면, 유저 코드는 모드 전환 없이 그 페이지를 읽기만 하면 된다. `ldd`로 아무 바이너리나 보면 `linux-vdso.so.1`이 보이는데 실제 파일은 없다 — 커널이 직접 주입하는 것이기 때문이다.

> 면접 단골: "시스템 콜이 왜 비싼가? → 모드 전환 + 캐시/TLB 영향. 그럼 gettimeofday는 왜 빠른가? → vDSO로 유저공간에서 처리." 이 두 문장을 잇는 답변이면 충분히 합격선이다.

### 주요 시스템 콜 분류

전체 목록은 `man 2 syscalls`에 있다(수백 개). 실무에서 strace 출력을 읽을 때 필요한 핵심만 분류하면:

| 분류 | 대표 syscall | 한 줄 설명 |
|---|---|---|
| 프로세스 | `fork`/`clone`, `execve`, `wait4`, `exit_group` | 생성·이미지 교체·자식 수거 — [[01 프로세스와 스레드]] 참조 |
| 파일 | `open`/`openat`, `read`, `write`, `close`, `stat`/`fstat`, `lseek` | "모든 것은 파일" — fd 기반 I/O의 전부 |
| 네트워크 | `socket`, `bind`, `listen`, `accept`, `connect`, `sendto`/`recvfrom` | 소켓도 fd다 — [[04 리눅스 네트워킹]] 참조 |
| 메모리 | `mmap`, `munmap`, `brk`, `mprotect` | 힙 확장과 파일 매핑 — [[02 메모리 관리]] 참조 |
| 동기화·이벤트 | `futex`, `epoll_create1`/`epoll_ctl`/`epoll_wait`, `poll`, `select` | 락 대기와 I/O 멀티플렉싱 |
| 기타 | `ioctl`, `fcntl`, `pipe2`, `dup2`, `getpid` | 디바이스 제어·fd 조작 |

`futex`가 동기화 분류에 있는 게 포인트다 — pthread mutex는 경합이 없으면 유저공간 atomic 연산만으로 처리되고, **경합이 생겼을 때만** `futex` syscall로 커널에 잠든다. strace에서 `futex`가 도배되어 있으면 락 경합을 의심하라.

## 3. strace — "이 프로세스가 뭘 하는지 모르겠으면 strace"

strace는 ptrace로 대상 프로세스의 모든 syscall을 가로채 보여 준다. 소스 코드 없이도 프로그램의 실제 행동(어떤 파일을 열고, 어디로 접속하고, 어디서 멈춰 있는지)을 볼 수 있는 최강의 블랙박스 도구다.

```bash
strace -c ./app              # syscall별 횟수·시간 요약 통계 (프로파일링 입구)
strace -e trace=network curl example.com    # 네트워크 syscall만 필터
strace -e trace=open,openat,stat ./app      # "어떤 설정 파일을 읽으려다 실패하나" 추적
strace -f ./app              # fork/clone된 자식까지 추적 (멀티프로세스 필수)
strace -p 1234 -tt -T        # 실행 중 프로세스에 부착, 타임스탬프 + syscall 소요시간
```

자주 쓰는 패턴:

- **설정 파일을 못 찾는 미스터리**: `strace -e trace=openat ./app 2>&1 | grep ENOENT` — 어떤 경로를 시도하다 실패하는지 즉시 보인다.
- **멈춰 있는 프로세스**: `strace -p <pid>` 했을 때 `read(...)`나 `futex(...)`에서 멈춰 있으면 각각 I/O 대기, 락 대기다. 어떤 fd를 기다리는지는 `/proc/<pid>/fd`로 교차 확인.
- **느린 구간 찾기**: `-T`(각 syscall 소요시간)와 `-c`(요약)를 먼저 돌리고 의심 syscall만 좁혀 간다.

⚠️ **프로덕션 경고**: strace는 syscall마다 대상 프로세스를 두 번 멈추기 때문에 syscall이 많은 워크로드에선 수 배~수십 배 느려질 수 있다. 프로덕션에 `-p`로 붙일 땐 짧게, 필터(`-e`)를 걸고, 가능하면 eBPF 기반 도구(아래 §12)로 대체하라.

## 4. I/O 멀티플렉싱 — select에서 epoll, 그리고 io_uring

소켓 1만 개를 스레드 1만 개로 감당하기엔 메모리·컨텍스트 스위칭 비용이 너무 크니(C10K 문제), 하나의 스레드가 "준비된 fd"만 골라 처리하는 게 멀티플렉싱이다. `select`/`poll`은 호출할 때마다 **감시할 fd 목록 전체를 커널에 넘기고 전체를 스캔**하므로 fd 수에 비례해 O(n)으로 느려진다. `epoll`은 관심 fd 집합을 커널 안에 등록해 두고(`epoll_ctl`), `epoll_wait`은 **준비된 fd만** 돌려받는다 — 호출 비용이 활성 이벤트 수에만 비례하므로 유휴 연결 수만 개를 들고 있어도 싸다. 이것이 nginx, Redis, Node.js의 이벤트 루프가 전부 epoll 위에 서 있는 이유고, C10K의 표준 답안이다. epoll에는 **레벨 트리거**(LT, 기본값 — 데이터가 남아 있는 한 계속 알림, 안전)와 **엣지 트리거**(ET — 상태가 변할 때 한 번만 알림, fd를 반드시 EAGAIN까지 비워야 하지만 깨우기 횟수가 줄어 고성능) 두 모드가 있다. 다음 세대인 io_uring은 제출/완료 큐를 유저-커널이 공유해 syscall 자체를 줄이는 비동기 I/O 인터페이스다 — 디스크 I/O 맥락은 [[03 파일시스템과 스토리지]]에서 잇는다.

> 면접 단골: "select와 epoll의 차이" — ① 매 호출마다 fd 집합 복사+전수 스캔 vs 커널 내 등록+준비된 것만 반환, ② fd 수 제한(select의 FD_SETSIZE) 유무, ③ LT/ET 모드. 이 세 가지를 말하면 된다.

## 5. 인터럽트 — 하드웨어가 커널을 부르는 길

syscall이 유저공간 → 커널의 통로라면, **인터럽트는 하드웨어 → 커널의 통로**다. NIC에 패킷이 도착하거나 디스크 I/O가 끝나면 디바이스가 CPU에 IRQ를 걸고, CPU는 하던 일을 멈추고 인터럽트 핸들러로 점프한다.

핵심 설계는 **상/하반부 분리**다:

- **하드 IRQ(top half)**: 인터럽트 비활성 상태에서 도니까 최소한의 일만 한다 — "패킷 왔다"는 표시만 하고 즉시 복귀.
- **softirq / 워커(bottom half)**: 실제 무거운 처리(패킷을 네트워크 스택에 올리기 등)는 나중에 softirq 컨텍스트에서 수행한다. softirq가 밀리면 커널 스레드 `ksoftirqd/N`(CPU별 하나)이 이어받는다 — `top`에서 `ksoftirqd`가 CPU를 먹고 있으면 "인터럽트 후속 처리가 밀리고 있다"는 신호다.

관측 입구:

```bash
cat /proc/interrupts    # IRQ별 × CPU별 하드 인터럽트 카운트 (NIC 큐가 어느 CPU에 박혔는지)
cat /proc/softirqs      # softirq 종류별(NET_RX, NET_TX, TIMER...) × CPU별 카운트
mpstat -P ALL 1         # %irq, %soft 컬럼으로 CPU별 인터럽트 부하 확인
```

**NIC 멀티큐와 IRQ 분산**: 현대 NIC은 큐가 여러 개라(멀티큐) **RSS**(하드웨어가 플로우 해시로 패킷을 큐별로 분산)로 여러 CPU가 나눠 받을 수 있고, 큐가 부족하면 **RPS**(소프트웨어 분산)로 보완한다. IRQ↔CPU 매핑은 `irqbalance` 데몬이 조정하거나 `/proc/irq/<n>/smp_affinity`로 수동 고정한다. 실무 단골 증상: **한 코어만 `si`(softirq)가 100% 근처고 나머지는 놀고 있다** → NIC 인터럽트가 한 CPU에 몰려 있는 것. `/proc/interrupts`에서 특정 CPU 열만 숫자가 폭증하는지 확인하고 RSS/RPS/irqbalance를 점검하는 게 정석 절차다. 상세한 네트워크 수신 경로는 [[04 리눅스 네트워킹]]에서 다룬다.

## 6. 시간 — 커널은 시간을 어떻게 아는가

커널은 주기적 타이머 인터럽트(**tick**)로 스케줄링·타임아웃을 구동한다(요즘은 유휴 CPU에서 tick을 끄는 tickless 모드가 일반적). 부팅 후 tick 수를 세는 전통적 카운터가 `jiffies`다(커널 코드 읽을 때 한 번은 만난다). 고해상도 시각의 원천은 **clocksource**이고, x86에서는 보통 CPU의 **TSC**(Time Stamp Counter)다 — `cat /sys/devices/system/clocksource/clocksource0/current_clocksource`로 확인하며, 이게 `tsc`가 아니라 `hpet` 같은 느린 소스로 떨어져 있으면 `clock_gettime` 비용이 눈에 띄게 올라간다(가상화 환경에서 종종 발생).

하드웨어 시계는 반드시 드리프트하므로 **NTP 동기화는 선택이 아니라 필수**다(요즘은 보통 chrony 사용, `chronyc tracking`으로 상태 확인). 분산 시스템 개발자라면 체감했을 것이다 — 노드 간 시계가 어긋나면 TLS 인증서 검증 실패, 분산 락/리스 오동작, 로그 타임라인 붕괴, Kerberos 인증 실패까지 전방위로 터진다.

## 7. sysctl — 커널을 재부팅 없이 튜닝하기

`/proc/sys/` 아래의 파일들은 **커널 런타임 파라미터**다. 파일을 읽고 쓰는 것만으로 커널 동작이 바뀐다. `sysctl` 명령은 이 트리의 편의 인터페이스다 (`net.core.somaxconn` ↔ `/proc/sys/net/core/somaxconn`).

```bash
sysctl net.core.somaxconn              # 조회
sysctl -w net.core.somaxconn=4096      # 즉시 적용 — 단, 재부팅하면 사라짐 (임시)
echo 'net.core.somaxconn = 4096' > /etc/sysctl.d/90-app.conf   # 영속화
sysctl --system                        # /etc/sysctl.d/* 전체 재적용
```

실무 단골 파라미터 — "언제 왜"가 핵심이다:

| 파라미터 | 무엇 | 언제 만지나 |
|---|---|---|
| `fs.file-max` | 시스템 전체 열린 파일 수 상한 | 시스템 전역으로 fd 고갈 날 때 (프로세스별 한도는 rlimit, §9) |
| `fs.nr_open` | 프로세스 하나가 가질 수 있는 fd 수의 절대 상한 | `LimitNOFILE`을 이 값보다 크게 줄 수 없어서, 초대형 한도가 필요할 때 함께 상향 |
| `net.core.somaxconn` | `listen()` accept 큐 길이 상한 | 트래픽 스파이크에 accept 큐 overflow로 연결 드롭될 때 — 앱의 backlog 인자도 같이 키워야 효과 |
| `net.ipv4.ip_local_port_range` | outbound 연결의 에페메럴 포트 범위 | 프록시·클라이언트가 단일 목적지로 대량 연결 시 포트 고갈("Cannot assign requested address") |
| `net.ipv4.tcp_tw_reuse` | TIME_WAIT 소켓을 outbound 연결에 재사용 | 위 포트 고갈의 짝꿍 처방 (deprecated된 `tcp_tw_recycle`과 혼동 금지 — 그건 쓰면 안 된다) |
| `vm.swappiness` | 스왑 적극성 (기본 60; 0~100, 커널 5.8+는 200까지) | DB·캐시 서버에서 페이지 캐시 대신 anon 메모리가 스왑되어 지연 튈 때 낮춤 — [[02 메모리 관리]] |
| `vm.overcommit_memory` | 메모리 오버커밋 정책 | Redis 등이 fork 시 경고할 때(=1 권고), OOM 정책 설계 시 — [[02 메모리 관리]] |
| `vm.max_map_count` | 프로세스당 mmap 영역 수 상한 | **Elasticsearch 필수**(공식 요구 262144) — 안 올리면 부팅 거부 |
| `kernel.pid_max` | PID 최대값 | 컨테이너·스레드를 대량으로 띄우는 노드에서 PID 고갈 방지 |

> 운영 원칙: `sysctl -w`로 먼저 실험하고, 효과가 검증되면 `/etc/sysctl.d/`에 박는다. 영속화 없이 튜닝했다가 재부팅 후 "왜 또 느려졌지"를 겪는 건 흔한 신입 인프라 실수다.

## 8. /proc과 /sys — 리눅스 관측의 원천

**"리눅스 관측의 원천은 결국 /proc"** — top, ps, netstat, free 전부 `/proc`을 파싱해서 보여 주는 껍데기다. `/proc`은 디스크에 없는 가상 파일시스템으로, 읽는 순간 커널이 현재 상태를 텍스트로 만들어 준다. 도구가 없는 컨테이너/장애 상황에서도 `cat`만 있으면 관측이 된다.

프로세스별 (`/proc/<pid>/`):

| 파일 | 내용 | 실무 용도 |
|---|---|---|
| `status` | 상태, 메모리(VmRSS), 스레드 수, UID | "이 프로세스 메모리 얼마나 써?" 의 원천 |
| `fd/` | 열린 fd의 심볼릭 링크 디렉토리 | `ls /proc/<pid>/fd \| wc -l`로 fd 누수 확인 |
| `limits` | 적용 중인 rlimit 실제값 | "ulimit 올렸는데 적용 안 됨" 검증의 정답지 |
| `maps` | 가상 메모리 매핑 전체 | 어떤 .so가 로드됐나, 메모리 폭증 원인 영역 |
| `environ` | 시작 시 환경 변수 (`tr '\0' '\n'`로 출력) | "이 프로세스 무슨 env로 떴지" |
| `stack` | 커널 스택 트레이스 (root) | D state로 멈춘 프로세스가 커널 어디서 자는지 |

시스템 전역:

- `/proc/meminfo`, `/proc/cpuinfo`, `/proc/loadavg`, `/proc/uptime` — free/top/uptime의 원천.
- `/proc/net/` — `tcp`, `udp`, `dev` 등 소켓·인터페이스 통계 (ss/netstat의 원천).

`/sys`(sysfs)는 성격이 다르다 — **디바이스와 커널 객체의 트리**다. 블록 디바이스 큐 설정(`/sys/block/<dev>/queue/`), NIC 속성(`/sys/class/net/`), cgroup(마운트 위치는 `/sys/fs/cgroup`) 등 하드웨어/객체 중심 구조이며 쓰기 가능한 노브도 많다. 거칠게 구분하면 **proc = 프로세스·커널 상태 관측, sys = 디바이스·커널 객체 설정**이다. 컨테이너가 이 둘을 어떻게 보는지는 [[06 컨테이너 내부 구조]]에서 다시 만난다.

## 9. rlimits — "Too many open files"의 해부

커널은 프로세스별 자원 한도(rlimit)를 강제한다. 가장 자주 부딪히는 게 `RLIMIT_NOFILE`(열 수 있는 fd 수) — 소켓도 파일도 전부 fd라서, 연결 많은 서버는 기본값(보통 soft 1024)을 반드시 넘는다. 증상은 `EMFILE: Too many open files`.

대응 절차(순서대로):

```bash
# 1. 실제 적용된 한도 확인 — 쉘의 ulimit이 아니라 그 프로세스의 limits가 진실
cat /proc/<pid>/limits | grep "open files"

# 2. 현재 사용량 확인 — 한도 문제인가, fd 누수인가부터 가르기
ls /proc/<pid>/fd | wc -l
ls -l /proc/<pid>/fd | awk '{print $NF}' | sort | uniq -c | sort -rn | head   # 뭐가 많은지

# 3-a. systemd 서비스라면 — 쉘 ulimit은 무시된다. unit에 지정해야 함
#   /etc/systemd/system/app.service.d/override.conf:
#   [Service]
#   LimitNOFILE=65536
systemctl daemon-reload && systemctl restart app

# 3-b. 쉘/일반 세션이라면
ulimit -n 65536                      # 현재 쉘 임시 (soft를 hard까지)
# 영속: /etc/security/limits.d/*.conf 에 "appuser soft nofile 65536" 등
```

핵심 함정: **systemd 서비스에는 `/etc/security/limits.conf`도 쉘의 `ulimit`도 적용되지 않는다.** PAM을 거치지 않기 때문이다 — 반드시 unit의 `LimitNOFILE`로 줘야 한다. "ulimit 올렸는데 왜 안 돼요"의 답이 거의 항상 이것이다.

그 외 자주 쓰는 한도:

- `nproc`(`RLIMIT_NPROC`): 사용자당 프로세스/스레드 수 — "fork: retry: Resource temporarily unavailable"이 뜨면 의심. 스레드도 이 카운트에 들어간다.
- core dump(`RLIMIT_CORE`): 기본 `ulimit -c 0`이라 크래시해도 코어가 안 남는다. 디버깅하려면 `ulimit -c unlimited` + 코어 파일이 어디로 가는지는 `/proc/sys/kernel/core_pattern`이 결정한다(요즘 배포판은 보통 systemd-coredump로 파이프되어 `coredumpctl`로 조회).

## 10. 커널 모듈과 커널 로그

모놀리식 커널의 유연성 보완 장치인 동적 모듈:

```bash
lsmod                        # 로드된 모듈 목록 (의존 관계 포함)
modinfo nvme                 # 모듈의 파라미터·버전·설명
modprobe br_netfilter        # 의존성까지 풀어서 로드 (insmod는 의존성 처리 안 함)
modprobe -r <module>         # 언로드
echo br_netfilter > /etc/modules-load.d/k8s.conf   # 부팅 시 자동 로드 영속화
```

인프라 실무에서 모듈을 직접 만질 일은 주로: Kubernetes 노드 준비(`br_netfilter`, `overlay` 로드가 kubeadm 사전 요구), NVIDIA 드라이버(GPU 노드 — AI 인프라에서 일상), 특정 NIC/스토리지 드라이버 이슈.

커널이 남기는 로그는 ring buffer에 쌓이고 두 입구로 읽는다:

```bash
dmesg -T                     # 사람이 읽을 수 있는 타임스탬프로 커널 메시지 출력
dmesg -T | grep -iE 'error|oom|kill|segfault|I/O'   # 장애 1차 스캔 단골 패턴
journalctl -k                # journald 경유 커널 메시지 (부팅 넘어 보존 가능)
journalctl -k -b -1          # 직전 부팅의 커널 로그 — "갑자기 재부팅됐어요" 조사 입구
```

OOM kill, 디스크 I/O 에러, NIC link down, segfault — 애플리케이션 로그에 안 보이는 "커널 레벨 사건"은 전부 여기 있다. **원인 불명 장애의 1차 체크리스트에 `dmesg -T`를 넣어라.**

## 11. 부팅 과정 — 전원 버튼에서 PID 1까지

부팅이 어디서 멈췄는지 진단하려면 단계를 알아야 한다:

1. **펌웨어(UEFI)** — 하드웨어 초기화, EFI 시스템 파티션에서 부트로더를 찾아 실행.
2. **부트로더(GRUB)** — 커널 이미지(`vmlinuz`)와 **initramfs**를 메모리에 적재하고 커널 커맨드라인(`cat /proc/cmdline`으로 확인 가능)을 넘긴다.
3. **커널 + initramfs** — 커널이 자신을 초기화한 뒤, initramfs(임시 루트 파일시스템)에서 진짜 루트를 마운트하는 데 필요한 드라이버·도구(스토리지 드라이버, LVM, 암호화 등)를 돌린다. "루트 디스크 드라이버가 커널에 내장 안 돼 있어도 부팅되는" 닭-달걀 해결 장치다.
4. **루트 마운트 후 PID 1 실행** — 진짜 루트로 전환(pivot)하고 `/sbin/init` → 요즘은 **systemd**가 PID 1.
5. **systemd가 target 단위로 기동** — unit 의존성 그래프를 풀며 `multi-user.target`/`graphical.target`까지 서비스를 병렬로 올린다.

진단 입구:

```bash
journalctl -b                # 이번 부팅의 전체 로그 (부팅 문제 1순위)
journalctl -b -1 -e          # 직전 부팅 로그의 끝부분 — 죽기 직전 마지막 메시지
systemd-analyze blame        # 부팅에서 오래 걸린 unit 순위
systemctl list-jobs          # 부팅이 "멈춘 것 같을 때" 무엇을 기다리는 중인지
```

클라우드에선 펌웨어/GRUB을 만질 일이 드물지만, **initramfs에 스토리지 드라이버가 빠져 루트 마운트 실패**(커널 업그레이드 후), **fstab의 죽은 마운트 때문에 emergency mode 진입** 같은 사고는 여전히 만난다 — 단계를 알면 어느 로그를 볼지 바로 정해진다.

## 12. eBPF — 커널을 다시 컴파일하지 않고 확장하기

**eBPF**는 커널 안에서 샌드박스된 프로그램을 실행하는 기술이다. 핵심 안전장치는 **verifier** — 로드 시점에 프로그램이 종료를 보장하는지, 잘못된 메모리 접근이 없는지 정적으로 검증하고, 통과한 것만 JIT 컴파일되어 커널 이벤트에 부착(attach)된다. 커널 모듈처럼 커널을 죽일 수 있는 코드가 아니라, "검증된 안전한 코드만 들어간다"는 점이 게임 체인저다.

부착 지점은 **kprobe**(커널 함수 동적 후킹), **tracepoint**(커널이 미리 박아 둔 안정적인 계측점), 네트워크 경로(XDP, tc) 등이며, 용도는 크게 **관측(observability)·네트워킹·보안** 세 갈래다.

bpftrace 원라이너 맛보기:

```bash
# 어떤 프로세스가 어떤 파일을 여는지 시스템 전체 실시간 관측
bpftrace -e 'tracepoint:syscalls:sys_enter_openat { printf("%s %s\n", comm, str(args->filename)); }'

# read() syscall 지연 분포 히스토그램 — "느린 I/O가 정말 있나"를 분포로 확인
bpftrace -e 'kprobe:vfs_read { @start[tid] = nsecs; }
             kretprobe:vfs_read /@start[tid]/ { @us = hist((nsecs - @start[tid]) / 1000); delete(@start[tid]); }'
```

strace와의 결정적 차이: strace는 대상 프로세스를 멈추지만 eBPF는 커널 안에서 이벤트를 집계하므로 **프로덕션에서 상시 켜 둘 수 있는 수준의 오버헤드**다. 그래서 현대 관측 스택이 eBPF로 수렴 중이다 — bcc 도구군(`execsnoop`, `opensnoop`, `biolatency`, `tcplife` 등 즉시 쓰는 완성품 모음), Cilium(쿠버네티스 CNI·네트워크 정책), Katran(L4 LB), Falco/Tetragon(런타임 보안)이 대표 생태계다. 성능 분석 워크플로우에서의 활용은 [[08 성능 분석과 트러블슈팅]]에서 잇는다.

## 면접 체크리스트

- 유저 모드/커널 모드는 왜 나뉘는가? syscall 한 번에 무슨 일이 일어나는가? (ring 전환, 비용, vDSO)
- `select` vs `epoll` — 왜 epoll이 C10K의 답인가? LT와 ET의 차이는?
- 하드 IRQ와 softirq를 왜 나누는가? `ksoftirqd`가 CPU를 먹는다는 건 무슨 신호인가?
- 한 코어만 `si`가 높을 때 진단 순서는? (`/proc/interrupts` → RSS/RPS/irqbalance)
- "Too many open files" 대응 절차를 처음부터 끝까지 — systemd 서비스에서의 함정 포함.
- `/proc`과 `/sys`의 차이를 한 문장으로.
- eBPF가 커널 모듈보다 안전한 이유는? (verifier) strace 대신 eBPF를 쓰는 이유는?
