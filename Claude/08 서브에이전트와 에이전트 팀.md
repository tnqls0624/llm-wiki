# 08 서브에이전트와 에이전트 팀

Claude Code가 여러 작업을 동시에 처리하는 방법을 다룬다. 한 세션 안에서 곁다리 작업을 격리하는 **서브에이전트**, 백그라운드 세션을 한 화면에서 관리하는 **에이전트 뷰**, 서로 메시지를 주고받는 **에이전트 팀**, 스크립트로 수십~수백 개 서브에이전트를 조율하는 **다이내믹 워크플로우**까지. 허브는 [[Claude]].

> 핵심 분기: **누가 작업을 조율하는가**, **작업자들이 서로 통신해야 하는가**, **같은 파일을 건드리는가**. 이 세 질문이 어떤 방식을 쓸지 결정한다.

---

## 병렬 실행 방식 비교 (agents)

`agents` 페이지는 네 가지 병렬 방식을 한눈에 비교하는 라우팅 페이지다. 모든 작업자는 결국 Claude 세션이며, 다른 도구를 끌어들이려면 [[10 MCP]] 서버로 노출해야 한다.

| 방식 | 무엇을 주는가 | 언제 쓰나 |
|------|--------------|-----------|
| Subagents (`/en/sub-agents`) | 한 세션 내부의 위임 작업자. 자기 컨텍스트에서 곁다리 작업을 하고 요약만 반환 | 검색 결과·로그·파일 내용이 메인 대화를 오염시킬 때 |
| Agent view (`claude agents`) | 백그라운드 세션을 dispatch/모니터링하는 한 화면. **research preview** | 독립 작업 여러 개를 던져두고 상태만 보다가 필요할 때만 개입 |
| Agent teams (`/en/agent-teams`) | 공유 작업 목록 + 에이전트 간 메시징, lead가 관리. **실험적·기본 비활성** | Claude가 프로젝트를 쪼개 배분하고 작업자 동기화 |
| Dynamic workflows (`/en/workflows`) | 다수 서브에이전트를 돌리고 결과를 교차 검증하는 스크립트. **research preview** | 서브에이전트 몇 개로 안 되는 대규모 작업, 결과 상호 검증 필요 |

### 조율 주체로 고르기
- **누가 조율하나?**
  - Claude가 한 대화 안에서 위임·수집 → **subagents**
  - 내가 독립 작업을 던지고 나중에 확인 → **agent view**
  - Claude가 작업자 그룹을 계획·배분·감독 → **agent teams** (실험적)
  - 스크립트가 계획을 보유 (Claude의 턴별 판단 대신) → **dynamic workflows**
- **작업자들이 서로 대화해야 하나?** 서브에이전트는 spawn한 대화로만 보고, agent view 세션은 나에게만 보고. 에이전트 팀의 teammate만 서로 직접 메시징.
- **같은 파일을 건드리나?** worktree로 격리. 서브에이전트와 직접 돌리는 세션은 각자 worktree 가능. 에이전트 팀은 teammate를 worktree로 격리하지 않으니 파일을 분할 소유시켜야 함.

### 작업을 다른 surface로 보조하는 것들
- **Worktrees** ([[20 베스트 프랙티스와 워크플로우]] 참고 영역): 각 세션에 별도 git checkout. agent view는 dispatch한 세션을 자동으로 worktree로 옮기고, 서브에이전트도 각각 받을 수 있음.
- **`/batch`**: 하나의 큰 변경을 5~30개 worktree-격리 서브에이전트로 쪼개 각자 PR을 여는 [[07 스킬]]. 서브에이전트+worktree의 패키징된 사용일 뿐 별도 방식 아님.
- 헷갈리기 쉬운 것들: background bash command(에이전트 spawn 아님), forked subagent(서브에이전트 spawn 방식), routine(스케줄 기반 클라우드 실행, 병렬 아님 — [[13 자동화와 스케줄링]]).

### 실행 중 작업 확인 명령
| 명령 | 무엇을 보나 |
|------|------------|
| `claude agents` | agent view: 모든 백그라운드 세션 |
| `/agents` | 현재 세션의 Running 탭(라이브 서브에이전트) + Library 탭(생성/편집). `claude agents`와 별개 |
| `/tasks` | 현재 세션 백그라운드의 모든 항목 (확인·attach·stop) |
| `/workflows` | 다이내믹 워크플로우 실행/완료 목록과 각 phase |

## 서브에이전트 (sub-agents)

서브에이전트는 특정 작업을 처리하는 전문 AI 보조다. 곁다리 작업이 메인 대화를 검색 결과·로그·파일 내용으로 채울 때 사용한다. 서브에이전트는 자기 컨텍스트 창에서 그 작업을 하고 **요약만** 반환한다. 같은 종류 작업자를 같은 지시로 반복 spawn한다면 커스텀 서브에이전트를 정의하라.

각 서브에이전트는 자기 컨텍스트 창 + 커스텀 시스템 프롬프트 + 특정 도구 접근 + 독립 권한으로 동작한다. 효과:
- **컨텍스트 보존**: 탐색·구현을 메인 대화 밖으로
- **제약 강제**: 사용 가능 도구 제한
- **설정 재사용**: 사용자 레벨 서브에이전트로 프로젝트 간 공유
- **동작 특화**: 특정 도메인용 집중 시스템 프롬프트
- **비용 통제**: Haiku 같은 빠르고 저렴한 모델로 라우팅

> 서브에이전트는 단일 세션 안에서 동작한다. 여러 독립 세션을 병렬로 돌리고 한 곳에서 모니터링하려면 agent view, 서로 통신하는 세션은 agent teams.

### 내장 서브에이전트
각 내장 서브에이전트는 부모 대화의 권한을 상속하되 도구 제한이 추가된다. **Explore와 Plan은 CLAUDE.md와 부모 git status를 건너뛰어** 탐색을 빠르고 저렴하게 유지한다. 나머지 모든 내장·커스텀 서브에이전트는 둘 다 로드한다.

| 에이전트 | 모델 | 도구 | 용도 |
|----------|------|------|------|
| Explore | Haiku | 읽기 전용 (Write/Edit 거부) | 파일 탐색, 코드 검색, 코드베이스 탐색. thoroughness: **quick / medium / very thorough** |
| Plan | 메인 상속 | 읽기 전용 (Write/Edit 거부) | plan mode 중 컨텍스트 수집. 무한 중첩 방지(서브에이전트는 다른 서브에이전트 spawn 불가) |
| General-purpose | 메인 상속 | 모든 도구 | 탐색+행동 모두 필요한 복잡한 다단계 작업 |
| statusline-setup | Sonnet | — | `/statusline` 실행 시 상태줄 설정 ([[12 출력 스타일과 상태줄]]) |
| claude-code-guide | Haiku | — | Claude Code 기능 질문 시 |

### 빠른 시작: `/agents`로 생성
`/agents`는 서브에이전트 관리 탭 인터페이스를 연다. **Running** 탭은 라이브 서브에이전트를 보여주고 열기/중지, **Library** 탭은 모든 서브에이전트 보기, 가이드 설정 또는 Claude 생성으로 새로 만들기, 기존 설정·도구 접근 편집, 커스텀 삭제, 중복 시 활성 항목 표시.

워크플로우(사용자 레벨 코드 리뷰어 예시):
1. `/agents` 실행
2. **Library** 탭 → **Create new agent** → **Personal** (저장 위치 `~/.claude/agents/`)
3. **Generate with Claude** 선택 후 설명 입력 → Claude가 identifier·description·시스템 프롬프트 생성
4. 도구 선택 (읽기 전용 리뷰어라면 **Read-only tools**만)
5. 모델 선택 (예: **Sonnet**)
6. 색상 선택
7. 메모리 설정 (**User scope** → `~/.claude/agent-memory/`, 또는 **None**)
8. `s`/`Enter`로 저장, `e`로 저장+편집. 즉시 사용 가능

```text
Use the code-improver agent to suggest improvements in this project
```

### 서브에이전트 scope (저장 위치와 우선순위)
서브에이전트는 YAML frontmatter가 있는 Markdown 파일. 이름이 겹치면 높은 우선순위 위치가 이김.

| 위치 | Scope | 우선순위 | 생성 방법 |
|------|-------|----------|-----------|
| Managed settings | 조직 전체 | 1 (최고) | [[04 설정]] managed settings로 배포 |
| `--agents` CLI 플래그 | 현재 세션 | 2 | 실행 시 JSON 전달 |
| `.claude/agents/` | 현재 프로젝트 | 3 | 인터랙티브 또는 수동 |
| `~/.claude/agents/` | 모든 내 프로젝트 | 4 | 인터랙티브 또는 수동 |
| 플러그인 `agents/` 디렉터리 | 플러그인 활성 위치 | 5 (최저) | [[11 플러그인]]으로 설치 |

- **프로젝트 서브에이전트**(`.claude/agents/`)는 버전 관리에 체크인해 팀이 공유. 현재 작업 디렉터리에서 위로 올라가며 탐색. `--add-dir`로 추가된 디렉터리는 파일 접근만 부여하고 서브에이전트 스캔은 안 함.
- `.claude/agents/`와 `~/.claude/agents/`는 **재귀 스캔**되어 `agents/review/` 같은 하위 폴더 정리 가능. 정체성은 오직 `name` frontmatter 필드에서 나오므로 하위 경로는 식별에 영향 없음. 같은 scope 내 `name` 중복 시 경고 없이 하나만 남고 폐기됨.
- **플러그인** `agents/` 하위 폴더는 [scoped identifier]의 일부가 됨: `agents/review/security.md` (플러그인 `my-plugin`) → `my-plugin:review:security`.

#### `--agents` CLI 정의 (세션 한정, 디스크 미저장)
```bash
claude --agents '{
  "code-reviewer": {
    "description": "Expert code reviewer. Use proactively after code changes.",
    "prompt": "You are a senior code reviewer. Focus on code quality, security, and best practices.",
    "tools": ["Read", "Grep", "Glob", "Bash"],
    "model": "sonnet"
  },
  "debugger": {
    "description": "Debugging specialist for errors and test failures.",
    "prompt": "You are an expert debugger. Analyze errors, identify root causes, and provide fixes."
  }
}'
```
`--agents`는 파일 기반과 동일한 frontmatter 필드를 받음: `description`, `prompt`, `tools`, `disallowedTools`, `model`, `permissionMode`, `mcpServers`, `hooks`, `maxTurns`, `skills`, `initialPrompt`, `memory`, `effort`, `background`, `isolation`, `color`. `prompt`가 시스템 프롬프트(파일 기반의 markdown 본문에 해당).

> 플러그인 서브에이전트는 보안상 `hooks`, `mcpServers`, `permissionMode` 필드를 지원하지 않음(무시). 필요하면 `.claude/agents/`나 `~/.claude/agents/`로 파일을 복사.

### 서브에이전트 파일 작성
```markdown
---
name: code-reviewer
description: Reviews code for quality and best practices
tools: Read, Glob, Grep
model: sonnet
---

You are a code reviewer. When invoked, analyze the code and provide
specific, actionable feedback on quality, security, and best practices.
```
- 본문이 시스템 프롬프트가 됨. 서브에이전트는 이 시스템 프롬프트(+작업 디렉터리 등 기본 환경 정보)만 받고 전체 Claude Code 시스템 프롬프트는 받지 않음.
- 디스크에서 직접 추가/편집하면 **세션 재시작**해야 로드됨. `/agents` 인터페이스로 만든 것은 즉시 반영.
- 서브에이전트는 메인 대화의 현재 작업 디렉터리에서 시작. 내부 `cd`는 Bash/PowerShell 호출 간 유지되지 않고 메인 작업 디렉터리에도 영향 없음. 격리된 저장소 복사본을 주려면 `isolation: worktree`.

#### 지원 frontmatter 필드 (`name`, `description`만 필수)
| 필드 | 필수 | 설명 |
|------|------|------|
| `name` | Yes | 소문자+하이픈 고유 식별자. [[09 훅]]의 `SubagentStart`가 `agent_type`로 받음. 파일명과 일치 불필요 |
| `description` | Yes | Claude가 언제 위임할지 |
| `tools` | No | 서브에이전트가 쓸 도구. 생략 시 전체 상속. Skill 프리로드는 `skills` 필드 사용(여기 `Skill` 나열 X) |
| `disallowedTools` | No | 거부할 도구(상속/지정 목록에서 제거) |
| `model` | No | `sonnet`/`opus`/`haiku`/full ID(예 `claude-opus-4-8`)/`inherit`. 기본 `inherit` |
| `permissionMode` | No | `default`/`acceptEdits`/`auto`/`dontAsk`/`bypassPermissions`/`plan`. 플러그인 서브에이전트는 무시 |
| `maxTurns` | No | 중지 전 최대 agentic 턴 수 |
| `skills` | No | 시작 시 컨텍스트에 프리로드할 [[07 스킬]]. description이 아닌 전체 내용 주입 |
| `mcpServers` | No | 이 서브에이전트가 쓸 [[10 MCP]] 서버. 이름 참조 또는 인라인 정의. 플러그인 서브에이전트는 무시 |
| `hooks` | No | 이 서브에이전트에 한정된 라이프사이클 [[09 훅]]. 플러그인 서브에이전트는 무시 |
| `memory` | No | 영속 메모리 scope: `user`/`project`/`local` |
| `background` | No | `true`면 항상 백그라운드 태스크로 실행. 기본 `false` |
| `effort` | No | 활성 시 노력 수준(세션 노력 재정의): `low`/`medium`/`high`/`xhigh`/`max`. 모델별 가용 |
| `isolation` | No | `worktree`면 임시 git worktree에서 실행. 기본 default 브랜치에서 분기(부모 HEAD 아님). 변경 없으면 자동 정리 |
| `color` | No | `red`/`blue`/`green`/`yellow`/`purple`/`orange`/`pink`/`cyan` |
| `initialPrompt` | No | `--agent`/`agent` 설정으로 메인 세션 에이전트일 때 첫 user 턴으로 자동 제출. 커맨드·스킬 처리됨. 사용자 프롬프트 앞에 추가 |

### 모델 선택
`model` 필드는 [[04 설정]] 모델을 제어. alias(`sonnet`/`opus`/`haiku`), full ID, `inherit`, 생략(기본 `inherit`). 해석 순서:
1. `CLAUDE_CODE_SUBAGENT_MODEL` 환경변수 (설정 시)
2. 호출별 `model` 파라미터
3. 서브에이전트 정의의 `model` frontmatter
4. 메인 대화 모델

### 서브에이전트 능력 통제

#### 사용 가능 도구
서브에이전트는 기본적으로 메인 대화의 [[06 내장 도구 레퍼런스]] 도구와 MCP 도구를 상속. 단 UI/세션 상태에 의존하는 다음은 `tools`에 나열해도 불가: `Agent`, `AskUserQuestion`, `EnterPlanMode`, `ExitPlanMode`(`permissionMode: plan` 제외), `ScheduleWakeup`, `WaitForMcpServers`.

allowlist(`tools`) 예시 — 읽기만, MCP 도구도 못 씀:
```yaml
---
name: safe-researcher
description: Research agent with restricted capabilities
tools: Read, Grep, Glob, Bash
---
```
denylist(`disallowedTools`) 예시 — Write/Edit만 제외하고 나머지 전부 상속:
```yaml
---
name: no-writes
description: Inherits every tool except file writes
disallowedTools: Write, Edit
---
```
둘 다 설정 시 `disallowedTools`가 먼저 적용된 뒤 나머지 풀에서 `tools` 해석. 양쪽에 있으면 제거됨.

#### spawn 가능한 서브에이전트 제한 (`--agent` 메인 스레드)
`claude --agent`로 메인 스레드인 에이전트는 Agent 도구로 서브에이전트를 spawn할 수 있음. `Agent(agent_type)` 구문으로 제한:
```yaml
tools: Agent(worker, researcher), Read, Bash
```
> v2.1.63에서 Task 도구가 Agent로 이름 변경됨. 기존 `Task(...)` 참조는 alias로 동작.

`Agent`(괄호 없이)는 모든 서브에이전트 허용, `Agent` 자체를 `tools`에서 생략하면 spawn 불가. 이 제한은 `claude --agent` 메인 스레드에만 적용(서브에이전트는 어차피 다른 서브에이전트 spawn 불가).

#### MCP 서버 범위 지정
```yaml
---
name: browser-tester
description: Tests features in a real browser using Playwright
mcpServers:
  - playwright:
      type: stdio
      command: npx
      args: ["-y", "@playwright/mcp@latest"]
  - github
---

Use the Playwright tools to navigate, screenshot, and interact with pages.
```
인라인 서버는 서브에이전트 시작 시 연결되고 종료 시 끊김. 문자열 참조는 부모 세션 연결 공유. MCP 도구 설명이 메인 대화 컨텍스트를 먹지 않게 하려면 `.mcp.json` 대신 여기 인라인 정의. v2.1.153부터 `--strict-mcp-config`/`--bare`, 엔터프라이즈 managed MCP, `allowedMcpServers`/`deniedMcpServers` 정책이 서브에이전트 frontmatter 서버에도 적용됨. (`--strict-mcp-config`는 `--agents`/SDK로 직접 전달한 인라인 서버는 필터링 안 함 — 명시적 caller 입력이므로.)

#### Permission modes
서브에이전트는 메인 대화의 권한 컨텍스트를 상속하되 모드를 재정의 가능(아래 부모 우선 예외 제외).

| 모드 | 동작 |
|------|------|
| `default` | 표준 권한 체크(프롬프트) |
| `acceptEdits` | 작업 디렉터리/`additionalDirectories` 경로의 파일 편집·일반 파일시스템 명령 자동 승인 |
| `auto` | 백그라운드 분류기가 명령·보호 디렉터리 쓰기 검토 ([[05 권한]]) |
| `dontAsk` | 권한 프롬프트 자동 거부(명시적 허용 도구는 동작) |
| `bypassPermissions` | 권한 프롬프트 건너뜀 |
| `plan` | 읽기 전용 탐색 |

> `bypassPermissions`는 주의. `.git`, `.config/git`, `.claude`, `.vscode`, `.idea` 등으로의 쓰기까지 승인 없이 실행. `rm -rf /` 같은 루트/홈 삭제는 차단 장치로 여전히 프롬프트.

부모가 `bypassPermissions`/`acceptEdits`면 우선되어 재정의 불가. 부모가 auto mode면 서브에이전트도 auto 상속하고 frontmatter의 `permissionMode`는 무시됨. 자세히는 [[05 권한]].

#### 스킬 프리로드
```yaml
---
name: api-developer
description: Implement API endpoints following team conventions
skills:
  - api-conventions
  - error-handling-patterns
---
```
나열된 스킬 전체 내용이 시작 시 주입. 이는 프리로드 대상을 제어할 뿐 접근 가능 스킬을 제어하지 않음(없어도 Skill 도구로 발견·호출 가능). 완전히 막으려면 `tools`에서 `Skill` 생략하거나 `disallowedTools`에 추가. `disable-model-invocation: true` 스킬은 프리로드 불가. 자세히는 [[07 스킬]].

#### 영속 메모리
```yaml
---
name: code-reviewer
description: Reviews code for quality and best practices
memory: user
---
```
| Scope | 위치 | 사용 시기 |
|-------|------|-----------|
| `user` | `~/.claude/agent-memory/<name>/` | 모든 프로젝트에서 기억 |
| `project` | `.claude/agent-memory/<name>/` | 프로젝트 한정, 버전 관리로 공유 (**권장 기본**) |
| `local` | `.claude/agent-memory-local/<name>/` | 프로젝트 한정, 버전 관리 미체크인 |

메모리 활성 시: 시스템 프롬프트에 읽기/쓰기 지침 포함, `MEMORY.md`의 첫 200줄 또는 25KB(먼저 도달하는 쪽) 포함(초과 시 큐레이션 지침), Read/Write/Edit 도구 자동 활성화. [[03 메모리와 컨텍스트]]의 CLAUDE.md 메모리와는 별개 개념. 작업 전 "메모리에서 패턴 확인", 작업 후 "배운 것 메모리에 저장"을 요청하면 지식 베이스가 누적됨.

#### 훅으로 조건부 규칙
`tools` 필드보다 세밀한 통제가 필요할 때 `PreToolUse` 훅으로 실행 전 검증. 읽기 전용 DB 쿼리만 허용하는 예:
```yaml
---
name: db-reader
description: Execute read-only database queries
tools: Bash
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-readonly-query.sh"
---
```
검증 스크립트는 stdin JSON에서 명령 추출 후 SQL 쓰기 연산이면 **exit code 2**로 차단(stderr로 에러 메시지 반환). 자세히는 [[09 훅]].

#### 특정 서브에이전트 비활성화
```json
{
  "permissions": {
    "deny": ["Agent(Explore)", "Agent(my-custom-agent)"]
  }
}
```
또는 `claude --disallowedTools "Agent(Explore)"`. 내장·커스텀 모두 적용. [[05 권한]] 참고.

### 서브에이전트용 훅 정의
두 가지 방법:
1. **서브에이전트 frontmatter** — 그 서브에이전트 활성 중에만 실행, 종료 시 정리
2. **`settings.json`** — 메인 세션에서 서브에이전트 시작/중지 시 실행

frontmatter 훅에서 자주 쓰는 이벤트: `PreToolUse`(도구 이름 matcher), `PostToolUse`(도구 이름), `Stop`(런타임에 `SubagentStop`으로 변환).
```yaml
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-command.sh $TOOL_INPUT"
  PostToolUse:
    - matcher: "Edit|Write"
      hooks:
        - type: command
          command: "./scripts/run-linter.sh"
```
`settings.json` 레벨 이벤트: `SubagentStart`(에이전트 타입 이름 matcher, 시작 시), `SubagentStop`(완료 시). 자세히는 [[09 훅]].

### 서브에이전트와 함께 작업하기

#### 자동 위임 이해
Claude는 요청의 작업 설명, 서브에이전트 `description` 필드, 현재 컨텍스트를 보고 자동 위임. 적극적 위임을 유도하려면 description에 "use proactively" 같은 문구 포함.

#### 명시적 호출 (세 가지 escalation)
- **자연어**: 프롬프트에서 서브에이전트 이름 언급, Claude가 위임 여부 결정
  ```text
  Use the test-runner subagent to fix failing tests
  ```
- **@-mention**: 그 서브에이전트가 한 작업에 반드시 실행됨을 보장
  ```text
  @"code-reviewer (agent)" look at the auth changes
  ```
  플러그인 제공 서브에이전트는 typeahead에 scoped 이름(`my-plugin:code-reviewer`)으로 표시. 수동 입력: 로컬은 `@agent-<name>`, 플러그인은 `@agent-my-plugin:code-reviewer`.
- **세션 전체** — 메인 스레드가 그 서브에이전트의 시스템 프롬프트·도구·모델을 가짐:
  ```bash
  claude --agent code-reviewer
  ```
  시스템 프롬프트가 기본 Claude Code 시스템 프롬프트를 완전 대체(`--system-prompt`와 동일). CLAUDE.md·프로젝트 메모리는 정상 로드. 시작 헤더에 `@<name>` 표시, resume 시 유지. 플러그인 충돌 시 scoped 이름으로 구분: `claude --agent my-plugin:security-reviewer`. 프로젝트 기본으로:
  ```json
  { "agent": "code-reviewer" }
  ```
  (CLI 플래그가 설정보다 우선). [[02 CLI 레퍼런스]] 참고.

#### 포그라운드 vs 백그라운드 실행
- **포그라운드**: 완료까지 메인 대화 차단. 권한 프롬프트가 나에게 전달됨.
- **백그라운드**: 동시 실행. 이미 부여된 권한으로 동작하고 프롬프트할 작업은 자동 거부. 명확화 질문이 필요하면 그 호출은 실패하나 서브에이전트는 계속.

Claude가 작업에 따라 결정하거나, "백그라운드로 실행" 요청, **Ctrl+B**로 백그라운드 전환. 전체 비활성: `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1`. `CLAUDE_CODE_FORK_SUBAGENT=1`이면 `background` 필드와 무관하게 모든 spawn이 백그라운드.

#### 공통 패턴
- **고용량 출력 격리**: 테스트 실행·문서 fetch·로그 처리를 위임해 verbose 출력은 서브에이전트 컨텍스트에 두고 요약만 회수
  ```text
  Use a subagent to run the test suite and report only the failing tests with their error messages
  ```
- **병렬 리서치**: 독립 조사를 동시 spawn (서로 의존 안 할 때 최선)
  ```text
  Research the authentication, database, and API modules in parallel using separate subagents
  ```
  > 많은 서브에이전트가 각자 상세 결과를 반환하면 컨텍스트를 크게 소모. 지속 병렬성이나 컨텍스트 초과는 agent teams.
- **체이닝**: 순차 다단계 워크플로우, 각 결과를 다음으로 전달
  ```text
  Use the code-reviewer subagent to find performance issues, then use the optimizer subagent to fix them
  ```

#### 서브에이전트 vs 메인 대화 선택
- **메인 대화**: 잦은 왕복·반복 정제 필요, 여러 단계가 큰 컨텍스트 공유, 빠르고 좁은 변경, 지연 중요(서브에이전트는 fresh 시작이라 컨텍스트 수집 시간 필요)
- **서브에이전트**: verbose 출력이 메인에 불필요, 특정 도구 제한 강제, 자기완결적이고 요약 반환 가능
- 메인 대화 컨텍스트에서 도는 재사용 프롬프트/워크플로우는 [[07 스킬]] 고려. 대화에 이미 있는 것 빠른 질문은 `/btw` ([[01 시작하기]]/인터랙티브 모드) — 전체 컨텍스트를 보지만 도구 접근 없고 답은 히스토리에 미추가.
- 서브에이전트는 다른 서브에이전트 spawn 불가. 중첩 위임이 필요하면 스킬 또는 메인 대화에서 체이닝.

### 서브에이전트 컨텍스트 관리

#### 시작 시 로드되는 것 (non-fork 서브에이전트)
각 서브에이전트는 fresh·격리 컨텍스트 창에서 시작. 대화 히스토리·이미 호출한 스킬·이미 읽은 파일을 보지 못함. 초기 컨텍스트:
- **시스템 프롬프트**: 에이전트 자기 프롬프트 + 환경 정보(전체 Claude Code 시스템 프롬프트 아님)
- **Task 메시지**: Claude가 작성한 위임 프롬프트
- **CLAUDE.md와 메모리**: 메인이 로드하는 [[03 메모리와 컨텍스트]] 계층 전부 (Explore/Plan만 건너뜀)
- **Git status**: 부모 세션 시작 시 스냅샷 (git 저장소 아니거나 `includeGitInstructions: false`면 없음. Explore/Plan은 무조건 건너뜀)
- **프리로드 스킬**: `skills` 필드 스킬 전체 내용 (내장 에이전트는 프리로드 안 함)

규칙이 서브에이전트에 꼭 도달해야 하면(예 "`vendor/` 무시") 위임 시 프롬프트에 재진술.

#### 서브에이전트 재개 (resume)
각 호출은 fresh 컨텍스트의 새 인스턴스. 기존 작업을 이어가려면 Claude에게 resume 요청. 재개된 서브에이전트는 이전 도구 호출·결과·추론 전체 히스토리 유지. 완료 시 Claude가 agent ID를 받고, `SendMessage` 도구로 `to`에 ID를 넣어 resume(이 도구는 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`일 때만 사용 가능). 중지된 서브에이전트가 `SendMessage`를 받으면 새 `Agent` 호출 없이 백그라운드에서 자동 재개.
```text
Use the code-reviewer subagent to review the authentication module
[Agent completes]
Continue that code review and now analyze the authorization logic
```
transcript는 `~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.jsonl`. 메인 대화 compaction과 무관하게 별도 파일에 저장. `cleanupPeriodDays`(기본 30일)에 따라 자동 정리.

#### Auto-compaction
서브에이전트도 메인과 동일 로직으로 자동 compaction. 기본 약 95% 용량에서 트리거. `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`로 더 일찍(예 `50`). transcript에 `compact_boundary` 시스템 이벤트로 기록.

### 현재 대화 fork (forked subagent)
fork는 fresh 시작 대신 **지금까지의 전체 대화를 상속**하는 서브에이전트다. 입력 격리는 포기하지만(같은 시스템 프롬프트·도구·모델·메시지 히스토리) fork의 도구 호출은 대화 밖에 남고 최종 결과만 회수되어 메인 컨텍스트는 깨끗하게 유지. 이름 붙은 서브에이전트가 너무 많은 배경 설명을 필요로 하거나, 같은 출발점에서 여러 접근을 병렬 시도할 때.

> v2.1.117+ 필요. v2.1.161부터 `/fork` 명령이 기본 활성. 이전 버전은 `CLAUDE_CODE_FORK_SUBAGENT=1` 필요. fork를 모델 *기본* spawn 동작으로 만드는 것은 실험적.

fork 모드 활성 시 두 가지 변화: general-purpose 서브에이전트를 쓸 자리에 fork를 spawn(Explore 등 named는 그대로), 모든 spawn이 백그라운드(동기로 유지하려면 `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1`).
```text
/fork draft unit tests for the parser changes so far
```
실행 중 fork 패널 조작:
| 키 | 동작 |
|----|------|
| `↑`/`↓` | 행 이동 |
| `Enter` | 선택한 fork transcript 열고 후속 메시지 전송 |
| `x` | 완료된 fork 닫기 / 실행 중 중지 |
| `Esc` | 프롬프트 입력으로 포커스 복귀 |

#### fork vs named 서브에이전트
| | Fork | Named 서브에이전트 |
|--|------|-------------------|
| 컨텍스트 | 전체 대화 히스토리 | fresh + 전달한 프롬프트 |
| 시스템 프롬프트·도구 | 메인과 동일 | 정의 파일에서 |
| 모델 | 메인과 동일 | `model` 필드 |
| 권한 | 프롬프트가 터미널에 표시 | 백그라운드 시 자동 거부 |
| 프롬프트 캐시 | 메인과 공유 | 별도 캐시 |

fork는 시스템 프롬프트·도구 정의가 부모와 동일해 첫 요청이 부모의 [[19 비용과 성능]] 프롬프트 캐시를 재사용 → fresh 서브에이전트보다 저렴. `CLAUDE_CODE_FORK_SUBAGENT=1`은 인터랙티브·non-interactive·Agent SDK 모두에서 활성, `0`은 서버측 rollout 포함 모두 비활성. fork는 추가 fork를 spawn할 수 없음.

### 예시 서브에이전트
**Code reviewer** (읽기 전용, Edit/Write 없음):
```markdown
---
name: code-reviewer
description: Expert code review specialist. Proactively reviews code for quality, security, and maintainability. Use immediately after writing or modifying code.
tools: Read, Grep, Glob, Bash
model: inherit
---
You are a senior code reviewer ensuring high standards of code quality and security.
When invoked:
1. Run git diff to see recent changes
2. Focus on modified files
3. Begin review immediately
...
Provide feedback organized by priority:
- Critical issues (must fix)
- Warnings (should fix)
- Suggestions (consider improving)
```
**Debugger** (분석+수정, Edit 포함), **Data scientist** (`model: sonnet`, SQL/BigQuery), **Database query validator** (`tools: Bash` + `PreToolUse` 훅으로 SELECT만 허용)도 원문에 수록. 베스트 프랙티스: 집중된 서브에이전트 설계, 상세 description, 최소 도구 접근, 버전 관리 체크인.

## 에이전트 뷰 (agent-view)

`claude agents`로 여는 한 화면. 모든 백그라운드 세션의 실행 상태·입력 필요 여부·완료를 보여준다. 각 백그라운드 세션은 터미널 없이도 계속 도는 완전한 Claude Code 대화다. 버그 수정·PR 리뷰·flaky 테스트 조사를 세 행으로 dispatch하고 다른 창에서 작업하다가 입력이 필요하거나 결과가 나온 행만 확인.

> research preview, v2.1.139+ 필요. 인터페이스·단축키 변동 가능.

### 빠른 시작
1. `claude agents` 실행 (Esc로 shell 복귀, 세션은 계속)
2. 프롬프트 입력 후 `Enter` → 백그라운드 세션 시작(각 프롬프트가 새 세션, 후속 아님). 헤더 모델과 그 디렉터리 권한 모드 사용
3. 행 선택 후 `Space` → peek 패널(최근 출력 또는 대기 중 질문), 답장 후 `Enter`
4. `Enter`/`→` → attach(전체 대화), 빈 프롬프트에서 `←` → detach
5. 이미 연 세션을 가져오려면 `/bg` 또는 빈 프롬프트에서 `←`

### 세션 상태 읽기
| 상태 | 아이콘 | 의미 |
|------|--------|------|
| Working | 애니메이션 | 도구 실행/응답 생성 중 |
| Needs input | 노랑 | 특정 질문/권한 결정 대기 |
| Idle | 흐림 | 할 일 없음, 다음 프롬프트 준비 |
| Completed | 초록 | 성공 |
| Failed | 빨강 | 에러로 종료 |
| Stopped | 회색 | `Ctrl+X`/`claude stop`으로 중지 |

아이콘 **모양**은 프로세스 실행 여부: `✻`/`✽`(살아있음, 즉시 응답), `∙`(프로세스 종료됨, peek/reply/attach 시 이어서 재시작), `✢`(`/loop` 세션 sleep 중, run count+countdown 표시 — [[13 자동화와 스케줄링]]). 우측 `PR #N` 라벨은 세션이 연 PR(상태 아이콘 아님).

상태는 디스크에 영속, supervisor 재시작·머신 sleep을 견딤(wake 시 재연결). 셧다운은 실행 세션을 중지.

#### 행 요약 / PR 상태
한 줄 요약은 **Haiku-class 모델**이 생성(작업 중 최대 15초마다, 턴 종료 시 1회 새로고침). v2.1.161부터 병렬 작업 항목이 2개 이상이면 `2/5` 같은 `done/total` 카운트가 앞에 표시. 서드파티 프로바이더에서 Haiku 미설정 시 메인 모델로 fallback, `ANTHROPIC_DEFAULT_HAIKU_MODEL`로 지정 ([[17 클라우드 프로바이더]]).

PR 번호 색상: 노랑(체크/리뷰 대기 또는 실패), 초록(통과·차단 리뷰 없음), 보라(merged), 회색(draft/closed). 보통 이 컬럼에서 결과를 받음(번호 초록 시 리뷰·머지).

### 목록 조직
그룹: `Ready for review`·`Needs input`이 `Working`·`Completed` 위. (`Ready for review`=열린 PR 보유, `Completed`=완료·실패·중지 묶음.) `Ctrl+S`로 디렉터리별 그룹화.
- `Ctrl+T`: 상단 고정 + idle 중에도 프로세스 유지
- `Shift+↑`/`Shift+↓`: 재정렬
- `Ctrl+R`: 이름 변경
- 그룹 헤더에서 `Enter`: 접기
- `Ctrl+X` → 2초 내 `Ctrl+X` 다시: 중지 후 삭제. (Claude가 만든 worktree와 미커밋 변경까지 삭제되니 먼저 push/commit)

### 세션 필터
| 필터 | 표시 |
|------|------|
| `a:<name>` | 그 에이전트로 도는 세션 |
| `s:<state>` | 그 상태(예 `s:working`, `s:blocked`=대기 중 전부) |
| `#<number>` 또는 PR URL | 그 PR 작업 세션 |

| 단축키 | 동작 |
|--------|------|
| `↑`/`↓` | 행 이동 |
| `Enter` | 선택 세션 attach, 입력 있으면 dispatch |
| `Space` | peek 패널 열기/닫기 |
| `Shift+Enter` | dispatch 후 즉시 attach |
| `→` | attach |
| `Alt+1`..`Alt+9` | 포커스 디렉터리의 세션 1~9 attach |
| `Tab` | 빈 입력이면 전체 서브에이전트 탐색, 아니면 강조 제안 적용 |
| `Ctrl+S` | 상태/디렉터리 그룹화 전환 |
| `Ctrl+T` | 핀/해제 |
| `Ctrl+R` | 이름 변경 |
| `Ctrl+G` | `$VISUAL`/`$EDITOR`로 dispatch 프롬프트 열기 |
| `Ctrl+X` | 중지(2초 내 재누르면 삭제) |
| `Shift+↑`/`Shift+↓` | 재정렬 |
| `Esc` | peek 닫기/입력 비우기/종료 |
| `Ctrl+C` | 입력 비우기(2번 누르면 종료) |
| `?` | 모든 단축키 표시 |

attach 중에는 일반 세션처럼 모든 [[02 CLI 레퍼런스]] 명령·단축키 동작. 항상 [[14 IDE와 데스크톱]]/fullscreen 모드로 렌더(scrollback 없음). detach(`←`/`Ctrl+Z`/`/exit`/double `Ctrl+C`/`Ctrl+D`)는 세션을 중지하지 않음. 내부에서 끝내려면 `/stop`.

### 새 에이전트 dispatch
#### agent view에서
프롬프트 입력 후 `Enter`. 자동 이름 지정(`Ctrl+R`로 변경). 이미지 붙여넣기 가능.
| 입력 | 효과 |
|------|------|
| `<agent-name> <prompt>` | 첫 단어가 커스텀 서브에이전트 이름이면 그 frontmatter 설정으로 메인 에이전트 실행 |
| `@<agent-name>` | 어디서든 멘션해 메인 에이전트로 실행 |
| `@<repo>` | agent view를 연 디렉터리 하위 저장소를 멘션해 거기서 실행 |
| `/<command>` | [[07 스킬]]·커맨드 제안 |
| `! <command>` | Claude 세션 대신 shell 잡으로 실행(행으로 표시) |
| `#<number>` 또는 PR URL | 이미 그 PR 작업 세션이 있으면 선택 |
| `Shift+Enter` | dispatch 후 즉시 attach |

`/exit`·`/quit`은 agent view 종료, `/logout`은 로그아웃 — 이들만 agent view 자체에서 실행되고 나머지 명령·스킬은 새 백그라운드 세션 첫 프롬프트로 전송. 같은 `@name`이 서브에이전트·형제 저장소 모두 매치하면 서브에이전트 우선.

**디렉터리 지정**: 해당 디렉터리에서 `claude agents` 열기, 부모 디렉터리에서 열고 `@<repo>` 멘션, 또는 shell에서 `cd` 후 `claude --bg "<prompt>"`.

#### 세션 내부에서 (`/background` / `/bg`)
현재 대화를 백그라운드 세션으로 이동. `/bg run the test suite and fix any failures`처럼 지시 추가 가능. backgrounding은 저장된 대화에서 재개하는 fresh 프로세스 시작이라 실행 중 서브에이전트·monitor·백그라운드 명령은 이전 안 됨(있으면 확인 요청). 원래 실행의 설정 플래그는 전달됨: `--mcp-config`, `--strict-mcp-config`, `--settings`, `--add-dir`, `--plugin-dir`, `--fallback-model`, `--allow-dangerously-skip-permissions`.

#### shell에서
```bash
claude --bg "investigate the flaky SettingsChangeDetector test"
claude --agent code-reviewer --bg "address review comments on PR 1234"
claude --bg --name "flaky-test-fix" "investigate the flaky SettingsChangeDetector test"
```
backgrounding 후 출력:
```text
backgrounded · 7c5dcf5d · flaky-test-fix
  claude agents             list sessions
  claude attach 7c5dcf5d    open in this terminal
  claude logs 7c5dcf5d      show recent output
  claude stop 7c5dcf5d      stop this session
```
**shell 명령 잡** — agent view 입력에 `!` 첫 문자, 또는 shell에서 `claude --bg --exec 'pytest -x'`. PTY-backed 잡으로 모델 미호출, 출력은 디스크 미기록(종료 약 5분 후 정리).

### 파일 편집 격리
모든 백그라운드 세션은 작업 디렉터리에서 시작하고, 파일 편집 전 Claude가 `.claude/worktrees/` 아래 격리된 git worktree로 이동. worktree를 건너뛰는 경우: 이미 linked git worktree 안, 작업 디렉터리가 git 저장소 아니고 `WorktreeCreate` 훅 없음, 쓰기가 작업 디렉터리 밖. 끄려면:
```json
{ "worktree": { "bgIsolation": "none" } }
```
(v2.1.143+). git 외 VCS는 `WorktreeCreate` 훅 ([[09 훅]]). 서브에이전트는 세션 작업 디렉터리를 상속하니 편집이 세션 worktree에 떨어짐. 별도 worktree를 주려면 frontmatter `isolation: worktree`.

### 모델 / 권한 모드 / effort
헤더 모델이 dispatch 기본(user settings의 `model`). agent view 전체 재정의는 `--model`. 세션별 재정의: `claude --bg --model`, attach 후 `/model`에서 `s`, 또는 서브에이전트 frontmatter `model`.

권한 모드는 시작 방식에 의존: `/bg`/`←` backgrounding은 현재 모드 유지, agent view 입력/`claude --bg`는 디렉터리 `defaultMode` 또는 서브에이전트 frontmatter `permissionMode`. supervisor 재시작 시 permission mode·model·effort·설정 플래그 모두 유지.

dispatch 기본 설정 플래그:
```bash
claude agents --permission-mode plan --model opus --effort high
```
`--agent`는 프롬프트가 named하지 않을 때 쓸 서브에이전트(기본 `agent` 설정 또는 내장 `claude`). `--dangerously-skip-permissions`=`--permission-mode bypassPermissions` 약칭, `--allow-dangerously-skip-permissions`는 각 세션 `Shift+Tab` 사이클에서 가용하게(시작 모드 아님).

| 플래그/설정 | 최소 버전 |
|-------------|-----------|
| `--permission-mode`, `--model`, `--effort`, `--dangerously-skip-permissions` | v2.1.142 |
| `--allow-dangerously-skip-permissions` | v2.1.143 |
| `--agent`, dispatched 세션의 `agent` 설정 존중 | v2.1.157 |

`bypassPermissions`/`auto`는 한 번 인터랙티브로 수락해야 사용 가능. 자세히는 [[05 권한]].

### 설정·플러그인·MCP (agent view 통과)
| 플래그 | 효과 |
|--------|------|
| `--settings <file-or-json>` | agent view·dispatched 세션 설정 재정의 |
| `--add-dir <path>` | 추가 디렉터리 파일 접근 ([[05 권한]]) |
| `--plugin-dir <path>` | 로컬 디렉터리에서 [[11 플러그인]] 로드 |
| `--mcp-config <file-or-json>` | 파일/JSON에서 [[10 MCP]] 서버 로드 |
| `--strict-mcp-config` | `--mcp-config` 서버만 사용 |

v2.1.142+. `--add-dir`/`--plugin-dir`/`--mcp-config`는 값당 한 번씩 반복(공백 구분 미지원).

### shell에서 세션 관리
짧은 ID는 `~/.claude/jobs/` 하위 디렉터리 이름.
| 명령 | 목적 |
|------|------|
| `claude agents` | agent view 열기 |
| `claude agents --cwd <path>` | 그 경로 하위 세션으로 범위 한정 (v2.1.141+) |
| `claude agents --json` | 라이브 세션을 JSON 배열로 출력(`pid`,`cwd`,`kind`,`startedAt`,`sessionId`,`name`,`status`,`waitingFor`) |
| `claude attach <id>` | 이 터미널에서 attach |
| `claude logs <id>` | 최근 출력 |
| `claude stop <id>` | 중지 (`claude kill`도 가능) |
| `claude respawn <id>` | 대화 유지하며 재시작(바이너리 업데이트 적용 등) |
| `claude respawn --all` | 모든 실행 세션 재시작 |
| `claude rm <id>` | 목록에서 제거. 미커밋 변경 없는 Claude-생성 worktree 제거, 있으면 경로 출력 |
| `claude daemon status` | supervisor 상태·버전·소켓 디렉터리·worker 수 |
| `claude daemon stop --any` | supervisor와 호스팅 세션 중지. `--keep-workers`로 세션 유지 |

### 백그라운드 세션 호스팅 (supervisor)
모든 agent view 세션은 백그라운드 세션. per-user **supervisor 프로세스**가 호스팅(터미널·agent view와 분리, 처음 backgrounding/agent view 열 때 자동 시작). 인터랙티브와 같은 자격증명 사용, 모델 API 외 추가 연결 없음. 작업 중·입력 대기·attach·실행 중 shell/서브에이전트/워크플로우/monitor가 있으면 프로세스 유지. idle·unattached 약 1시간 후 프로세스 중지(`Ctrl+T` 핀은 면제). transcript·상태는 디스크 유지, 다음 attach/peek/reply 시 이어서 재시작. 메모리 부족 시 idle non-pinned 먼저 중지. 바이너리 업데이트(로컬 파일 watch) 시 새 버전으로 재시작·재연결.

상태 저장 위치 (`CLAUDE_CONFIG_DIR` 설정 시 그 디렉터리):
| 경로 | 내용 |
|------|------|
| `~/.claude/daemon.log` | supervisor 로그 |
| `~/.claude/daemon/roster.json` | 실행 세션 목록(재시작 재연결용) |
| `~/.claude/jobs/<id>/state.json` | per-session 상태 |
| `~/.claude/jobs/<id>/tmp/` | per-session scratch(권한 미프롬프트, 삭제 시 제거) |

각 세션은 `CLAUDE_JOB_DIR` 환경변수가 `~/.claude/jobs/<id>`로 설정됨. `claude daemon status`/`/doctor`로 점검.

**agent view 끄기**: `disableAgentView: true` 설정 또는 `CLAUDE_CODE_DISABLE_AGENT_VIEW` 환경변수(관리자는 [[16 CI-CD와 팀 통합]] managed settings로 강제).

### 트러블슈팅 ([[25 트러블슈팅]] 참고)
- `claude agents`가 서브에이전트 목록만 출력 후 종료: 환경에서 agent view 미가용. `claude update`. Bedrock/Vertex AI/Foundry에서 구버전 미동작.
- 세션 없음: dispatch 전 온보딩 힌트. 프롬프트 입력 후 `Enter`.
- `Cannot open agents — N background task(s) running`: in-flight 작업 존재. `/tasks`로 확인 후 `/bg`로 확인.
- `Too short`: 4자 미만 프롬프트 거부.
- 셧다운 후 failed: attach/peek/reply 시 재시작. (sleep만으로는 발생 안 함.)
- background service did not respond: supervisor 정체. `claude daemon stop --any --keep-workers`. Windows는 PID 출력 → `taskkill /PID <pid>`.
- macOS 보호 폴더(Desktop/Documents/Downloads) `Operation not permitted`: System Settings > Privacy & Security > Files and Folders에서 허용.
- attach 후 느림: 1시간 idle 후 프로세스 중지 탓. `Ctrl+T` 핀.
- `.claude/worktrees/` 누적: `git worktree list`/`git worktree remove <path>`.

### 한계 (research preview)
- **Rate limit 적용**: 백그라운드 세션도 구독 사용량 소비. 10개 병렬 = 쿼터 약 10배 ([[19 비용과 성능]]).
- **로컬 세션**: 머신에서 실행. sleep 견디나 셧다운 시 중지.
- **Claude-생성 worktree는 세션 삭제 시 함께 삭제**. 먼저 머지/push.

## 에이전트 팀 (agent-teams)

여러 Claude Code 인스턴스가 팀으로 협업. 한 세션이 **team lead**가 되어 작업 조율·할당·결과 종합, **teammate**들은 각자 컨텍스트 창에서 독립 작업하며 **서로 직접 통신**한다. 서브에이전트(메인에만 보고)와 달리 lead를 거치지 않고 개별 teammate와 직접 상호작용 가능.

> **실험적·기본 비활성**. v2.1.32+ 필요. 세션 재개·작업 조율·셧다운에 알려진 한계.

### 활성화
```json
{ "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } }
```

### 언제 쓰나 / 서브에이전트와 비교
강력한 사용처: **리서치/리뷰**(여러 측면 동시 조사 후 공유·도전), **새 모듈/기능**(겹치지 않게 분담), **경쟁 가설 디버깅**(병렬 검증·수렴), **계층 횡단 조율**(frontend/backend/test). 조율 오버헤드와 토큰이 단일 세션보다 훨씬 큼. 순차 작업·같은 파일 편집·의존성 많은 작업은 단일 세션이나 서브에이전트가 낫다.

| | Subagents | Agent teams |
|--|-----------|-------------|
| Context | 자기 컨텍스트, 결과를 caller에 반환 | 자기 컨텍스트, 완전 독립 |
| Communication | 메인 에이전트에만 보고 | teammate끼리 직접 메시징 |
| Coordination | 메인 에이전트가 전부 관리 | 공유 작업 목록 + 자기 조율 |
| Best for | 결과만 중요한 집중 작업 | 논의·협업 필요한 복잡 작업 |
| Token cost | 낮음(요약 반환) | 높음(각 teammate가 별도 인스턴스) |

### 팀 시작
활성화 후 자연어로 작업과 팀 구조 설명:
```text
I'm designing a CLI tool that helps developers track TODO comments across
their codebase. Create an agent team to explore this from different angles: one
teammate on UX, one on technical architecture, one playing devil's advocate.
```
공유 작업 목록 생성 → teammate spawn → 탐색 → 종합 → 정리. lead 터미널에 teammate와 작업 나열. **Shift+Down**으로 teammate 순환·직접 메시징(마지막 후 lead로 wrap).

### 팀 제어
#### 디스플레이 모드
- **In-process**: 모든 teammate가 메인 터미널 내부. Shift+Down 순환, 입력해 메시지. 모든 터미널 가능, 추가 설정 불필요.
- **Split panes**: 각 teammate가 자기 pane. 동시 출력 보기·클릭 상호작용. **tmux 또는 iTerm2** 필요.

기본 `"auto"`(이미 tmux 안이면 split, 아니면 in-process). `"tmux"`는 split 활성·tmux/iTerm2 자동 감지.
```json
{ "teammateMode": "in-process" }
```
또는 `claude --teammate-mode in-process`. tmux는 시스템 패키지 매니저, iTerm2는 `it2` CLI + Python API 활성(Settings → General → Magic).

#### teammate·모델 지정
```text
Create a team with 4 teammates to refactor these modules in parallel.
Use Sonnet for each teammate.
```
teammate는 기본적으로 lead의 `/model`을 상속 안 함. 프롬프트 미지정 시 모델은 `/config`의 **Default teammate model**(또는 **Default (leader's model)**)로 설정.

#### teammate 계획 승인 요구
```text
Spawn an architect teammate to refactor the authentication module.
Require plan approval before they make any changes.
```
teammate가 읽기 전용 plan mode로 작업하다 계획 완료 시 lead에 승인 요청. lead 승인 시 plan mode 종료·구현 시작, 거부 시 피드백 반영·재제출. lead는 자율 결정하니 프롬프트로 기준 제공("only approve plans that include test coverage").

#### teammate와 직접 대화
각 teammate는 완전 독립 Claude Code 세션. In-process: Shift+Down 순환 후 입력, Enter로 세션 보기·Escape로 인터럽트, **Ctrl+T**로 작업 목록 토글. Split: pane 클릭.

#### 작업 할당·claim
공유 작업 목록이 조율. 상태: pending / in progress / completed. 작업 간 의존 가능(미해결 의존 있는 pending은 claim 불가). lead가 명시 할당 또는 teammate가 self-claim(완료 후 다음 미할당·미차단 작업 선택). claim은 파일 락으로 race condition 방지.

#### 셧다운·정리
```text
Ask the researcher teammate to shut down
```
lead가 셧다운 요청 → teammate 승인(우아 종료) 또는 거부(설명).
```text
Clean up the team
```
공유 팀 리소스 제거. **항상 lead로 정리**(teammate는 팀 컨텍스트가 안 풀려 inconsistent 상태 위험). 활성 teammate 있으면 실패하니 먼저 셧다운.

#### 훅으로 품질 게이트 ([[09 훅]])
- `TeammateIdle`: teammate가 idle 직전 실행. exit 2로 피드백 보내고 계속 작업시킴.
- `TaskCreated`: 작업 생성 시. exit 2로 생성 방지·피드백.
- `TaskCompleted`: 완료 표시 시. exit 2로 완료 방지·피드백.

### 작동 방식 (아키텍처)
| 컴포넌트 | 역할 |
|----------|------|
| Team lead | 팀 생성·teammate spawn·조율하는 메인 세션 |
| Teammates | 할당 작업을 하는 별도 인스턴스 |
| Task list | claim·완료할 공유 작업 항목 |
| Mailbox | 에이전트 간 메시징 시스템 |

의존성 자동 관리(완료 시 차단 작업 자동 해제). 로컬 저장: 팀 config `~/.claude/teams/{team-name}/config.json`, 작업 목록 `~/.claude/tasks/{team-name}/`. config는 런타임 상태(세션 ID·tmux pane ID) 보유 — 수동 편집/사전작성 금지(다음 상태 업데이트 때 덮어씀). config의 `members` 배열로 teammate가 다른 멤버 발견. 프로젝트 레벨 등가물 없음.

#### 서브에이전트 정의를 teammate로 재사용
spawn 시 어느 [scope](프로젝트/사용자/플러그인/CLI)의 서브에이전트 타입이든 참조 가능. 역할 한 번 정의 후 위임 서브에이전트·팀 teammate 양쪽 재사용.
```text
Spawn a teammate using the security-reviewer agent type to audit the auth module.
```
teammate는 그 정의의 `tools` allowlist·`model`을 따르고, 본문은 teammate 시스템 프롬프트에 **추가**됨(대체 아님). `SendMessage`·작업 관리 도구는 `tools`가 제한해도 항상 사용 가능.
> 서브에이전트 정의의 `skills`·`mcpServers` 필드는 teammate로 실행될 때 적용 안 됨. teammate는 프로젝트·사용자 설정에서 로드(일반 세션처럼).

#### 권한·컨텍스트·통신
teammate는 lead의 권한 설정으로 시작(lead가 `--dangerously-skip-permissions`면 전부). spawn 후 개별 변경 가능하나 spawn 시 per-teammate 모드 설정 불가. 각 teammate는 자기 컨텍스트 창 + spawn 시 일반 세션처럼 프로젝트 컨텍스트(CLAUDE.md·MCP·스킬) 로드(lead 대화 히스토리는 미전달). 정보 공유: 자동 메시지 전달(lead가 poll 불필요), idle 알림, 공유 작업 목록, 이름 기반 메시징(전체에 보내려면 수신자당 한 통). 예측 가능한 이름을 원하면 spawn 지시에서 명명.

#### 토큰 사용
단일 세션보다 훨씬 많은 토큰. 활성 teammate 수에 비례. [[19 비용과 성능]]의 agent team token costs 참고.

### 사용 사례
**병렬 코드 리뷰**:
```text
Create an agent team to review PR #142. Spawn three reviewers:
- One focused on security implications
- One checking performance impact
- One validating test coverage
Have them each review and report findings.
```
**경쟁 가설 조사** (적대적 토론):
```text
Users report the app exits after one message instead of staying connected.
Spawn 5 agent teammates to investigate different hypotheses. Have them talk to
each other to try to disprove each other's theories, like a scientific
debate. Update the findings doc with whatever consensus emerges.
```

### 베스트 프랙티스
- **충분한 컨텍스트 제공**: teammate는 대화 히스토리 미상속. spawn 프롬프트에 작업 세부 포함.
- **적절한 팀 크기**: 대부분 **3~5명**으로 시작. teammate당 5~6 작업이 적절. 15개 독립 작업이면 3명. 토큰 선형 증가·조율 오버헤드·diminishing returns.
- **작업 크기**: 너무 작으면 오버헤드 초과, 너무 크면 체크인 없이 오래 → 자기완결 단위(함수·테스트 파일·리뷰).
- **teammate 완료 대기**: lead가 직접 구현 시작하면 "Wait for your teammates to complete their tasks before proceeding".
- **리서치·리뷰로 시작**: 경계 명확하고 코드 미작성 작업(PR 리뷰·라이브러리 조사·버그 조사)부터.
- **파일 충돌 회피**: 같은 파일 편집은 덮어씀. teammate별 다른 파일 셋 소유.
- **모니터·조정**: 진행 점검·접근 재조정·종합. 방치하면 낭비 위험.

### 트러블슈팅 ([[25 트러블슈팅]])
- teammate 미표시: in-process는 Shift+Down 순환, 작업이 팀을 정당화할 만큼 복잡한지, split은 `which tmux`·iTerm2 `it2`·Python API.
- 권한 프롬프트 과다: spawn 전 [[05 권한]]에서 공통 작업 사전 승인.
- 에러로 중지: Shift+Down/pane으로 출력 확인 후 추가 지시 또는 교체 teammate spawn.
- lead 조기 셧다운: 계속하라고 지시.
- 고아 tmux: `tmux ls` → `tmux kill-session -t <session-name>`.

### 한계 (실험적)
- in-process teammate는 `/resume`·`/rewind`로 복원 안 됨(없는 teammate에 메시징 시도 가능 → 새로 spawn 지시).
- 작업 상태 지연 가능(완료 미표시 → 의존 차단). 수동 업데이트 또는 lead nudge.
- 셧다운 느림(현재 요청/도구 호출 완료 후).
- **한 번에 한 팀**(새 팀 전 정리), **중첩 팀 없음**(teammate는 팀/teammate spawn 불가), **lead 고정**(생성 세션이 평생 lead, 승격·이전 불가), **권한 spawn 시 설정**, split pane은 tmux/iTerm2 필요(VS Code 통합 터미널·Windows Terminal·Ghostty 미지원).
> `CLAUDE.md`는 정상 동작 — teammate가 작업 디렉터리의 CLAUDE.md 읽음 ([[03 메모리와 컨텍스트]]).

## 다이내믹 워크플로우 (workflows)

다이내믹 워크플로우는 **서브에이전트를 대규모로 조율하는 JavaScript 스크립트**다. Claude가 작업에 맞는 스크립트를 작성하고, 런타임이 백그라운드에서 실행하는 동안 세션은 응답 가능 상태를 유지한다. 한 대화로 조율할 수 없을 만큼 에이전트가 많거나, 조율을 읽고 재실행 가능한 스크립트로 코드화하고 싶을 때. 예: 코드베이스 전체 버그 스윕, 500파일 마이그레이션, 소스 교차 검증 리서치, 여러 각도에서 초안 작성하는 어려운 계획.

> research preview, v2.1.154+, 모든 유료 플랜·Anthropic API·Bedrock·Vertex AI·Microsoft Foundry. Pro는 `/config`의 Dynamic workflows 행에서 켬.

### 언제 워크플로우인가 (vs subagents/skills/teams)
차이는 **누가 계획을 보유하는가**:

| | Subagents | Skills | Agent teams | Workflows |
|--|-----------|--------|-------------|-----------|
| 무엇 | Claude가 spawn하는 작업자 | Claude가 따르는 지시 | peer 세션 감독 lead | 런타임이 실행하는 스크립트 |
| 다음 실행 결정 | Claude, 턴별 | Claude, 프롬프트 따름 | lead, 턴별 | 스크립트 |
| 중간 결과 위치 | Claude 컨텍스트 | Claude 컨텍스트 | 공유 작업 목록 | 스크립트 변수 |
| 재사용 가능 | 작업자 정의 | 지시 | 팀 정의 | 조율 자체 |
| 규모 | 턴당 몇 개 | subagents와 동일 | 소수 장기 peer | run당 수십~수백 에이전트 |
| 인터럽트 | 턴 재시작 | 턴 재시작 | teammate 계속 | 같은 세션 내 재개 가능 |

워크플로우는 계획을 코드로 옮긴다. 스크립트가 루프·분기·중간 결과를 보유해 Claude 컨텍스트엔 최종 답만 남음. 또 반복 가능한 품질 패턴 적용: 독립 에이전트가 서로의 발견을 적대적으로 리뷰, 여러 각도 계획을 비교 → 단일 패스보다 신뢰도 높은 결과.

### 번들 워크플로우 실행 (`/deep-research`)
| 명령 | 동작 |
|------|------|
| `/deep-research <question>` | 여러 각도로 웹 검색 fan-out, 소스 fetch·교차 검증, 각 주장에 투표, 교차 검증 못 통과한 주장 제거 후 인용 리포트 반환. [[06 내장 도구 레퍼런스]]의 WebSearch 도구 필요 |

```text
/deep-research What changed in the Node.js permission model between v20 and v22?
```
1. 실행 → 2. 워크플로우 허용 프롬프트 **Yes** → 3. `/workflows`로 진행 보기(각 phase의 에이전트 수·토큰·경과 시간, drill-down) → 4. 완료 시 리포트가 세션에 도착(주장별 소스 인용). 입력 박스 아래 task 패널에서도 한 줄 요약 watch 가능. 저장한 워크플로우도 같은 방식으로 `/` 자동완성에 명령으로 등장.

#### run 보기 (`/workflows` 진행 뷰)
| 키 | 동작 |
|----|------|
| `↑`/`↓` | phase/에이전트 선택 |
| `Enter`/`→` | phase drill-in, 에이전트의 프롬프트·최근 도구 호출·결과 |
| `Esc` | 한 레벨 뒤로 |
| `j`/`k` | 에이전트 상세 오버플로 스크롤 |
| `p` | run 일시정지/재개 |
| `x` | 선택 에이전트 중지, 포커스가 run이면 전체 중지 |
| `r` | 선택 실행 중 에이전트 재시작 |
| `s` | run 스크립트를 명령으로 저장 |

### Claude가 워크플로우 작성하게 하기
#### 프롬프트에서 요청
세션 effort를 안 바꾸고 단일 작업을 워크플로우로: 프롬프트에 `ultracode` 키워드 포함. 자기 말("use a workflow"/"run a workflow")도 동작. (v2.1.160 이전엔 트리거 키워드가 `workflow`였음.)
```text
ultracode: audit every API endpoint under src/routes/ for missing auth checks
```
키워드 강조됨. 의도 아니었으면 macOS `Option+W`/Win·Linux `Alt+W`로 강조 해제, 또는 키워드 직후 커서에서 backspace. 트리거 자체를 끄려면 `/config`의 Ultracode keyword trigger 끄기. 다른 방식으로 만든 오케스트레이터(서브에이전트 프롬프트 폴더, fan-out 스킬)를 가리키며 같은 일을 하는 워크플로우 요청 가능.

#### ultracode로 Claude가 결정
`xhigh` [[19 비용과 성능]]/reasoning effort + 자동 워크플로우 조율을 결합한 설정.
```text
/effort ultracode
```
켜면 모든 substantive 작업에 워크플로우 계획(한 요청이 이해→변경→검증 여러 워크플로우로). 토큰·시간 증가. 현재 세션 한정·새 세션 리셋. 루틴 작업 복귀 시 `/effort high`. `xhigh` 지원 모델에서만 가용.

#### 실행 전 계획 승인
CLI per-run 프롬프트 옵션: **Yes, run it** / **Yes, and don't ask again for `<name>` in `<path>`** / **View raw script** / **No**. `Ctrl+G`로 에디터에서 스크립트 열기, `Tab`으로 프롬프트 조정.

| 권한 모드 | 프롬프트 시점 |
|-----------|---------------|
| Default, accept edits | 매 run (해당 워크플로우에 "don't ask again" 선택 전까지) |
| Auto | 첫 launch만. **Yes**가 user settings에 동의 기록, 이후 무프롬프트. ultracode 시 완전 생략 |
| Bypass permissions, `claude -p`, Agent SDK | 없음. 즉시 시작 |

권한 모드는 launch 프롬프트만 제어. 워크플로우가 spawn하는 서브에이전트는 **항상 `acceptEdits` 모드**로 실행, 세션 모드 무관하게 [[05 권한]] 도구 allowlist 상속, 파일 편집 자동 승인. allowlist에 없는 shell 명령·web fetch·MCP 도구는 mid-run 프롬프트 가능(긴 run 전 미리 추가). Desktop 앱은 승인 카드(**Once**/**Always**/**Deny**), 진행 뷰는 Background tasks 사이드 페인. `claude -p`·Agent SDK는 프롬프트 없이 권한 규칙 따름.

#### 재사용 저장 / 입력 전달
`/workflows`에서 run 선택 후 `s`. Tab으로 저장 위치 토글: `.claude/workflows/`(프로젝트 공유) 또는 `~/.claude/workflows/`(개인 전체). Enter로 저장 → `/<name>`으로 실행. 이름 충돌 시 프로젝트 우선.

저장 워크플로우는 `args` 파라미터로 입력 받음(스크립트가 전역 `args`로 읽음):
```text
> Run /triage-issues on issues 1024, 1025, and 1030
```
Claude가 구조화 데이터로 전달하니 스크립트가 파싱 없이 배열·객체 메서드 직접 호출. `args` 생략 시 전역은 `undefined`.

### 워크플로우 실행 방식
런타임이 대화와 분리된 격리 환경에서 스크립트 실행. 중간 결과는 스크립트 변수에 머묾(Claude 컨텍스트 미진입). 매 run은 스크립트를 `~/.claude/projects/` 하위 세션 디렉터리에 기록(Claude가 경로 받음 — 읽기·diff·편집 후 relaunch 가능). 런타임이 각 에이전트 결과를 추적 → 같은 세션 내 재개 가능.

#### 동작·한계
| 제약 | 이유 |
|------|------|
| mid-run 사용자 입력 없음 | 에이전트 권한 프롬프트만 run 일시정지. 단계 간 sign-off는 단계별 워크플로우로 |
| 워크플로우 자체의 직접 파일시스템/shell 접근 없음 | 에이전트가 read/write/명령 실행, 스크립트는 조율만 |
| 동시 에이전트 최대 16개(CPU 제한 시 적게) | 로컬 리소스 제한 |
| run당 총 1,000 에이전트 | runaway 루프 방지 |

### run 관리
`/workflows` 뷰 또는 입력 박스 아래 진행 라인 확장.
- **재개**: 중지 후 재개 가능(완료 에이전트는 캐시 결과 반환, 나머지 live). `/workflows`에서 선택 후 `p`, 또는 같은 스크립트로 relaunch 요청. **같은 세션 내**만. 워크플로우 실행 중 Claude Code 종료 시 다음 세션은 fresh 시작.
- **비용**: 많은 에이전트 spawn으로 토큰이 훨씬 많음. 큰 작업 전 작은 슬라이스(한 디렉터리·좁은 질문)로 테스트. `/workflows` 뷰가 에이전트별 토큰 표시, 완료 작업 손실 없이 중지 가능. 에이전트 cap이 runaway 비용 제한. 큰 run 전 `/model` 확인, 일부 단계에 더 작은 모델 요청. [[19 비용과 성능]] 참고.
- **끄기**: `/config`의 Dynamic workflows 토글, `~/.claude/settings.json`의 `"disableWorkflows": true`, `CLAUDE_CODE_DISABLE_WORKFLOWS=1`. 조직 전체는 managed settings의 `"disableWorkflows": true` 또는 admin settings 페이지 ([[16 CI-CD와 팀 통합]]). 비활성 시 번들 명령 미가용·`ultracode` 키워드 미트리거·`/effort` 메뉴에서 제거.

워크플로우는 CLI·Desktop 앱·[[14 IDE와 데스크톱]] 확장·non-interactive(`claude -p`)·[[22 Agent SDK — 시작]]에서 가용.
