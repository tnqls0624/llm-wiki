# 20 베스트 프랙티스와 워크플로우

Claude Code를 실전에서 잘 쓰기 위한 패턴 모음이다. 환경 설정부터 병렬 세션 확장, 복사-붙여넣기 프롬프트, 클라우드 기반 계획/리뷰, 목표 기반 자동 진행까지 다룬다. 허브는 [[Claude]].

핵심 전제 하나로 거의 모든 베스트 프랙티스가 파생된다: **Claude의 컨텍스트 윈도우는 빠르게 채워지고, 채워질수록 성능이 떨어진다.** 컨텍스트에는 대화 메시지 전부, Claude가 읽은 모든 파일, 모든 명령어 출력이 들어간다. 디버깅 한 번이나 코드베이스 탐색 한 번으로도 수만 토큰이 쌓인다. 컨텍스트가 차면 Claude가 앞선 지시를 "잊거나" 실수가 늘기 때문에, 컨텍스트는 관리해야 할 가장 중요한 자원이다. 토큰 절감 전략은 [[19 비용과 성능]]을 참고.

## 베스트 프랙티스 (best-practices)

### Claude에게 스스로 검증할 수단을 줘라

Claude는 "끝난 것처럼 보일 때" 멈춘다. 실행 가능한 체크가 없으면 "보기에 됐다"가 유일한 신호가 되고, 사람이 검증 루프가 되어 모든 실수를 직접 잡아야 한다. **pass/fail 신호를 돌려주는 체크**를 주면 루프가 스스로 닫힌다 — Claude가 작업하고, 체크를 돌리고, 결과를 읽고, 통과할 때까지 반복한다.

체크는 대화에 신호로 돌아오는 것이면 무엇이든 된다: 테스트 스위트, 빌드 exit code, 린터, 출력을 fixture와 diff하는 스크립트, 디자인과 비교하는 [브라우저 스크린샷](https://code.claude.com/docs/en/chrome)([[21 브라우저와 컴퓨터 사용]]).

| 전략 | Before (모호) | After (검증 포함) |
| --- | --- | --- |
| 검증 기준 제공 | "implement a function that validates email addresses" | "write a validateEmail function. example test cases: user@example.com is true, invalid is false, user@.com is false. run the tests after implementing" |
| UI 변경 시각 검증 | "make the dashboard look better" | "[paste screenshot] implement this design. take a screenshot of the result and compare it to the original. list differences and fix them" |
| 증상이 아닌 근본 원인 | "the build is failing" | "the build fails with this error: [paste error]. fix it and verify the build succeeds. address the root cause, don't suppress the error" |

체크가 존재하면, **멈춤을 얼마나 강하게 게이팅할지** 결정한다. 설정 비용과 사람의 주의력을 트레이드오프하는 사다리다:

| 강도 | 방법 | 특징 |
| --- | --- | --- |
| 한 프롬프트 | 같은 메시지에서 체크 실행+반복 요청 | 오늘 당장 모든 작업에 적용 가능 |
| 세션 전체 | `/goal` 조건 설정 | 별도 평가자가 매 턴 재확인 (아래 goal 섹션 / [[13 자동화와 스케줄링]]) |
| 결정적 게이트 | Stop hook이 체크 스크립트 실행, 통과 전까지 턴 종료 차단 | 8회 연속 차단 시 Claude Code가 오버라이드해 종료. [[09 훅]] |
| 제2의 의견 | verification subagent / dynamic workflow가 신선한 모델로 결과를 반박 시도 | 작업한 에이전트가 채점하지 않게 함. [[08 서브에이전트와 에이전트 팀]] |

성공을 단언하지 말고 **증거를 보여주게** 하라: 테스트 출력, 실행한 명령과 반환값, 결과 스크린샷. 증거를 검토하는 게 직접 재검증하는 것보다 빠르고, 지켜보지 않은 세션에도 통한다.

### 먼저 탐색, 그다음 계획, 마지막에 코딩

바로 코딩에 뛰어들면 엉뚱한 문제를 푸는 코드가 나올 수 있다. plan mode([[05 권한]])로 탐색과 실행을 분리하라. 권장 4단계:

| 단계 | 모드 | 예시 프롬프트 |
| --- | --- | --- |
| Explore | plan mode | `read /src/auth and understand how we handle sessions and login. also look at how we manage environment variables for secrets.` |
| Plan | plan mode | `I want to add Google OAuth. What files need to change? What's the session flow? Create a plan.` |
| Implement | default mode | `implement the OAuth flow from your plan. write tests for the callback handler, run the test suite and fix any failures.` |
| Commit | default mode | `commit with a descriptive message and open a PR` |

Plan 단계에서 `Ctrl+G`를 누르면 계획을 텍스트 에디터로 열어 직접 편집한 뒤 진행할 수 있다.

> [!note] 계획의 트레이드오프
> Plan mode는 유용하지만 오버헤드도 있다. 오타 수정·로그 한 줄 추가·변수명 변경처럼 범위가 명확하고 작은 수정은 그냥 직접 시키는 게 낫다. 계획은 접근 방식이 불확실할 때, 여러 파일을 건드릴 때, 수정 대상 코드에 익숙하지 않을 때 가장 유용하다. **diff를 한 문장으로 설명할 수 있으면 계획을 건너뛰어라.**

### 프롬프트에 구체적 컨텍스트를 담아라

Claude는 의도를 추론할 수 있지만 마음을 읽지는 못한다. 구체적 파일을 참조하고, 제약을 명시하고, 예시 패턴을 가리켜라. 정밀할수록 교정 횟수가 줄어든다.

| 전략 | Before | After |
| --- | --- | --- |
| 작업 범위 한정 | "add tests for foo.py" | "write a test for foo.py covering the edge case where the user is logged out. avoid mocks." |
| 출처 지목 | "why does ExecutionFactory have such a weird api?" | "look through ExecutionFactory's git history and summarize how its api came to be" |
| 기존 패턴 참조 | "add a calendar widget" | "look at how existing widgets are implemented... HotDogWidget.php is a good example. follow the pattern to implement a new calendar widget..." |
| 증상 기술 | "fix the login bug" | "users report that login fails after session timeout. check the auth flow in src/auth/, especially token refresh. write a failing test that reproduces the issue, then fix it" |

다만 모호한 프롬프트가 유용할 때도 있다 — 탐색 중이고 방향 수정이 가능할 때. `"what would you improve in this file?"` 같은 프롬프트는 미처 묻지 못한 것을 끌어낸다.

#### 풍부한 콘텐츠 제공

- **`@`로 파일 참조**: 위치를 설명하는 대신 직접 참조. Claude가 응답 전에 읽는다.
- **이미지 붙여넣기**: 복사-붙여넣기 또는 드래그앤드롭.
- **URL 제공**: 문서/API 레퍼런스. `/permissions`로 자주 쓰는 도메인을 allowlist.
- **데이터 파이프**: `cat error.log | claude`로 파일 내용 직접 전달.
- **Claude가 직접 가져오게**: Bash, MCP 도구, 파일 읽기로 스스로 컨텍스트를 끌어오게 지시.

### 환경 설정

#### 효과적인 CLAUDE.md 작성

CLAUDE.md는 매 대화 시작 시 Claude가 읽는 특수 파일이다. Bash 명령, 코드 스타일, 워크플로우 규칙을 담아 코드만으로 추론 못 하는 영속 컨텍스트를 준다. `/init`으로 코드베이스를 분석해 스타터 파일을 생성한 뒤 다듬어라.

```markdown CLAUDE.md
# Code style
- Use ES modules (import/export) syntax, not CommonJS (require)
- Destructure imports when possible (eg. import { foo } from 'bar')

# Workflow
- Be sure to typecheck when you're done making a series of code changes
- Prefer running single tests, and not the whole test suite, for performance
```

매 세션 로드되므로 **넓게 적용되는 것만** 넣어라. 가끔만 필요한 도메인 지식/워크플로우는 [[07 스킬]]로. 간결하게 유지하라 — 각 줄에 "이 줄을 지우면 Claude가 실수할까?"를 물어 아니면 잘라라. **비대한 CLAUDE.md는 실제 지시를 무시하게 만든다.**

| ✅ 포함 | ❌ 제외 |
| --- | --- |
| Claude가 추측 못 하는 Bash 명령 | 코드를 읽어 알 수 있는 것 |
| 기본과 다른 코드 스타일 규칙 | Claude가 이미 아는 표준 언어 관례 |
| 테스트 지침/선호 러너 | 상세 API 문서 (링크로 대체) |
| 레포 에티켓 (브랜치 네이밍, PR 관례) | 자주 바뀌는 정보 |
| 프로젝트 특유 아키텍처 결정 | 파일별 코드베이스 설명 |
| 개발 환경 quirk (필수 env var) | "clean code 작성" 같은 자명한 관행 |

"IMPORTANT" 또는 "YOU MUST" 강조로 준수도를 높일 수 있다. git에 체크인해 팀과 공유하면 가치가 누적된다. `@path/to/import` 문법으로 다른 파일을 import할 수 있다:

```markdown CLAUDE.md
See @README.md for project overview and @package.json for available npm commands.
- Git workflow: @docs/git-instructions.md
- Personal overrides: @~/.claude/my-project-instructions.md
```

배치 위치별 적용 범위:

| 위치 | 적용 |
| --- | --- |
| `~/.claude/CLAUDE.md` | 모든 Claude 세션 |
| `./CLAUDE.md` | 프로젝트 루트, git 체크인해 팀 공유 |
| `./CLAUDE.local.md` | 개인 노트, `.gitignore`에 추가 |
| 상위 디렉터리 | 모노레포에서 자동으로 끌어옴 |
| 하위 디렉터리 | 해당 디렉터리 파일을 읽을 때 on-demand |

자세한 메모리 관리는 [[03 메모리와 컨텍스트]]에서 다룬다.

#### 권한 / CLI 도구 / MCP / 훅 / 스킬 / 서브에이전트 / 플러그인

기본적으로 Claude Code는 시스템 변경 가능 동작(파일 쓰기, Bash, MCP 도구 등)마다 권한을 묻는다. 인터럽트를 줄이는 3가지 방법:

| 방법 | 설명 | 참고 |
| --- | --- | --- |
| Auto mode | 별도 분류 모델이 명령을 검토해 위험한 것(권한 상승, 미지의 인프라, 적대적 콘텐츠 유발 동작)만 차단 | [[05 권한]] |
| Permission allowlist | `npm run lint`, `git commit` 등 안전한 도구만 허용 | [[05 권한]] |
| Sandboxing | OS 수준 격리로 파일시스템/네트워크 접근 제한 | [[18 보안과 샌드박스]] |

- **CLI 도구**: `gh`, `aws`, `gcloud`, `sentry-cli` 등을 쓰라고 지시. 외부 서비스와 상호작용하는 가장 컨텍스트 효율적인 방법. `gh` 미설치 시 무인증 API 요청은 rate limit에 걸리기 쉽다. 모르는 도구도 `Use 'foo-cli-tool --help' to learn about foo tool, then use it to solve A, B, C.` 식으로 학습시킬 수 있다.
- **MCP 서버**: `claude mcp add`로 Notion, Figma, DB 등 연결. [[10 MCP]]
- **훅**: 예외 없이 매번 일어나야 하는 동작에. CLAUDE.md 지시는 권고적이지만 훅은 결정적이다. `"Write a hook that runs eslint after every file edit"` 식으로 Claude에게 작성시키거나 `.claude/settings.json` 직접 편집, `/hooks`로 조회. [[09 훅]]
- **스킬**: `.claude/skills/<name>/SKILL.md`로 도메인 지식·재사용 워크플로우 추가. 자동 적용 또는 `/skill-name`으로 직접 호출. 부수효과가 있어 수동 트리거하고 싶으면 frontmatter에 `disable-model-invocation: true`. [[07 스킬]]
- **서브에이전트**: `.claude/agents/`에 자체 컨텍스트·허용 도구를 가진 전문 어시스턴트 정의. `"Use a subagent to review this code for security issues."` 식으로 명시 호출. [[08 서브에이전트와 에이전트 팀]]
- **플러그인**: `/plugin`으로 마켓플레이스 탐색. 스킬·훅·서브에이전트·MCP를 한 단위로 번들. 타입 언어면 code intelligence 플러그인 권장. [[11 플러그인]]

```markdown .claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: Fix a GitHub issue
disable-model-invocation: true
---
Analyze and fix the GitHub issue: $ARGUMENTS.
1. Use `gh issue view` to get the issue details
...
```

### 효과적으로 소통하기

- **코드베이스 질문**: 시니어 엔지니어에게 물을 법한 것을 그대로 물어라 — "How does logging work?", "What does `async move { ... }` do on line 134 of `foo.rs`?". 온보딩 시간을 줄이고 다른 엔지니어 부하도 줄인다. 특별한 프롬프트 불필요.
- **Claude가 너를 인터뷰하게**: 큰 기능은 최소 프롬프트로 시작해 `AskUserQuestion` 도구로 인터뷰시켜라. 기술 구현·UI/UX·엣지케이스·트레이드오프를 묻는다.

```text
I want to build [brief description]. Interview me in detail using the AskUserQuestion tool.
Ask about technical implementation, UI/UX, edge cases, concerns, and tradeoffs. Don't ask obvious questions, dig into the hard parts I might not have considered.
Keep interviewing until we've covered everything, then write a complete spec to SPEC.md.
```

스펙이 완성되면 **새 세션**을 열어 실행하라 — 깨끗한 컨텍스트가 구현에만 집중한다. 좋은 스펙은 자기완결적이다: 관련 파일/인터페이스를 명시하고, 범위 밖을 진술하고, 기능 작동을 증명하는 end-to-end 검증 단계로 끝난다.

### 세션 관리

대화는 영속적이고 되돌릴 수 있다. 빨리 자주 교정하라.

| 동작 | 단축키/명령 | 효과 |
| --- | --- | --- |
| 중단 | `Esc` | 동작 중 멈춤. 컨텍스트 보존, 재지시 가능 |
| 되감기 | `Esc + Esc` 또는 `/rewind` | rewind 메뉴 — 이전 대화/코드 상태 복원 또는 선택 메시지부터 요약 |
| 되돌리기 | `"Undo that"` | Claude가 변경 되돌림 |
| 컨텍스트 리셋 | `/clear` | 무관한 작업 사이 컨텍스트 완전 초기화 |
| 부분 압축 | `/compact <instructions>` | 예: `/compact Focus on the API changes` |
| 사이드 질문 | `/btw` | 답이 dismissible 오버레이에 떠 대화 기록에 안 들어감 |

**한 이슈를 두 번 넘게 교정했다면** 컨텍스트가 실패한 접근으로 오염된 것 — `/clear`하고 배운 것을 반영한 더 구체적 프롬프트로 새로 시작하라. 컨텍스트 한계 근처에서 자동 compaction이 코드 패턴·파일 상태·핵심 결정을 보존하며 요약한다. CLAUDE.md에 `"When compacting, always preserve the full list of modified files and any test commands"`처럼 compaction 동작을 커스터마이즈할 수 있다.

**조사는 서브에이전트에 위임**하라 (`"use subagents to investigate X"`). 별도 컨텍스트에서 탐색하고 요약만 보고하므로 메인 대화가 깨끗하게 유지된다. 컨텍스트가 근본 제약이므로 서브에이전트는 가장 강력한 도구 중 하나다.

**체크포인트**: 매 프롬프트가 체크포인트를 만든다. Claude가 변경 전 파일을 스냅샷하므로 대화/코드/둘 다를 임의 시점으로 복원 가능. 위험한 시도를 시키고 안 되면 되감으면 된다. 세션 간에도 유지된다. 자세히는 [[03 메모리와 컨텍스트]].

> [!warning] 체크포인트의 한계
> 체크포인트는 *Claude가 한* 변경만 추적하고 외부 프로세스는 추적하지 않는다. git 대체재가 아니다.

**대화 재개**: `claude --continue`로 가장 최근 세션 이어가기, `claude --resume`으로 목록에서 선택. `/rename`으로 `oauth-migration` 같은 이름을 붙여 워크스트림별 영속 컨텍스트로 다뤄라. 전체는 [[03 메모리와 컨텍스트]].

### 자동화와 확장

| 패턴 | 명령 | 용도 |
| --- | --- | --- |
| 비대화형 모드 | `claude -p "prompt"` | CI, pre-commit hook, 스크립트. `--output-format json` 또는 `stream-json --verbose` |
| 병렬 세션 | worktree / desktop / web / agent teams | 격리 실험, 복잡 워크플로우 |
| 파일 across fan-out | 루프로 `claude -p` 반복 + `--allowedTools` | 대규모 마이그레이션/분석 |
| 자율 실행 | `claude --permission-mode auto -p "fix all lint errors"` | 백그라운드 안전 체크와 함께 무중단 실행 |

```bash
# 비대화형 출력 포맷
claude -p "Explain what this project does"
claude -p "List all API endpoints" --output-format json
claude -p "Analyze this log file" --output-format stream-json --verbose
```

**Writer/Reviewer 패턴** — 병렬 세션의 신선한 컨텍스트는 코드 리뷰 품질을 높인다(방금 쓴 코드에 편향되지 않음):

| Session A (Writer) | Session B (Reviewer) |
| --- | --- |
| `Implement a rate limiter for our API endpoints` | |
| | `Review the rate limiter implementation in @src/middleware/rateLimiter.ts. Look for edge cases, race conditions, and consistency with our existing middleware patterns.` |
| `Here's the review feedback: [Session B output]. Address these issues.` | |

**Fan out across files** — 2,000개 파일 마이그레이션 같은 대규모 작업:

```bash
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

먼저 2~3개 파일로 테스트해 프롬프트를 다듬은 뒤 전체에 실행하라. `--allowedTools`는 무인 실행 시 Claude가 할 수 있는 것을 제한한다. 파이프라인 통합: `claude -p "<your prompt>" --output-format json | your_command`.

**적대적 리뷰 단계**: 무인 실행이 길어질수록 완료 판정 전 독립 체크가 중요하다. 신선한 서브에이전트는 diff와 기준만 보고 그것을 만든 추론은 못 보므로 결과를 독립적으로 평가한다. 정확성 체크는 번들 `/code-review` 스킬, 계획 대비 체크는 직접 프롬프트 작성:

```text
Use a subagent to review the rate limiter diff against PLAN.md. Check that
every requirement is implemented, the listed edge cases have tests, and
nothing outside the task's scope changed. Report gaps, not style preferences.
```

> [!warning] 과잉 엔지니어링 주의
> 갭을 찾으라고 시킨 리뷰어는 작업이 멀쩡해도 보통 갭을 보고한다 — 그게 요청받은 일이라서다. 모든 발견을 좇으면 불필요한 추상화·방어 코드·일어날 수 없는 케이스 테스트로 과잉 엔지니어링된다. **정확성이나 명시된 요구에 영향 주는 갭만 표시하게** 하고 나머지는 선택으로 취급하라.

### 흔한 실패 패턴

| 패턴 | 증상 | 해결 |
| --- | --- | --- |
| The kitchen sink session | 무관한 작업을 섞어 컨텍스트가 잡동사니로 참 | 작업 사이 `/clear` |
| Correcting over and over | 반복 교정으로 실패 접근이 컨텍스트 오염 | 2회 실패 후 `/clear` + 더 나은 초기 프롬프트 |
| The over-specified CLAUDE.md | 너무 길어 중요 규칙이 묻힘 | 가차없이 prune, 또는 훅으로 전환 |
| The trust-then-verify gap | 그럴듯하지만 엣지케이스 미처리 구현 | 항상 검증 제공(테스트/스크립트/스크린샷). 검증 못 하면 배포 금지 |
| The infinite exploration | 범위 없는 "조사"로 수백 파일 읽어 컨텍스트 채움 | 조사 범위를 좁히거나 서브에이전트 사용 |

### 직관을 길러라

위 패턴은 불변의 법칙이 아니라 일반적으로 잘 통하는 출발점이다. 복잡한 한 문제에 깊이 빠져 히스토리가 가치 있으면 컨텍스트를 쌓는 게 맞고, 탐색적 작업이면 계획을 건너뛰는 게 맞고, 어떻게 해석하는지 보고 싶으면 모호한 프롬프트가 정답일 수 있다. Claude가 좋은 결과를 낼 때와 헤맬 때 무엇을 했는지 관찰해 직관을 길러라.

## 일상 워크플로우 레시피 (common-workflows)

각 레시피는 모든 Claude Code 표면(surface)에서 작동한다. 프로젝트에 맞게 문구를 조정하라.

### 새 코드베이스 이해

- **개요 파악**: `cd /path/to/project` → `claude` → `give me an overview of this codebase`. 이어서 `explain the main architecture patterns used here`, `what are the key data models?`, `how is authentication handled?`. 넓은 질문에서 좁게. 모노레포 설정은 별도 large-codebases 문서 참고.
- **관련 코드 찾기**: `find the files that handle user authentication` → `how do these authentication files work together?` → `trace the login process from front-end to database`. 도메인 언어를 쓰고, 언어별 code intelligence 플러그인을 설치하면 정밀 "go to definition / find references" 가능.

### 버그 효율적으로 수정

`I'm seeing an error when I run npm test` → `suggest a few ways to fix the @ts-ignore in user.ts` → `update user.ts to add the null check you suggested`. 재현 명령과 스택 트레이스를 알려주고, 재현 절차와 (간헐적/일관적) 여부를 명시하라.

### 코드 리팩터링

`find deprecated API usage in our codebase` → `suggest how to refactor utils.js to use modern JavaScript features` → `refactor utils.js to use ES2024 features while maintaining the same behavior` → `run tests for the refactored code`. 작고 테스트 가능한 증분으로, 필요하면 하위 호환 유지를 요청.

### 테스트 작업

`find functions in NotificationsService.swift that are not covered by tests` → `add tests for the notification service` → `add test cases for edge conditions...` → `run the new tests and fix any failures`. Claude는 기존 테스트 파일을 살펴 스타일·프레임워크·assertion 패턴을 맞춘다. 놓친 엣지케이스(에러 조건, 경계값, 예상 밖 입력) 식별도 요청 가능.

### Pull Request 생성

직접 `create a pr for my changes`라고 하거나 단계별로: `summarize the changes I've made to the authentication module` → `create a pr` → `enhance the PR description with more context about the security improvements`. `gh pr create`로 PR을 만들면 세션이 자동으로 그 PR에 연결된다 — 나중에 `claude --from-pr <number>` 또는 `/resume` picker 검색에 PR URL을 붙여 돌아간다. 제출 전 생성된 PR을 검토하고 리스크/고려사항을 강조하게 하라.

### 문서 처리

`find functions without proper JSDoc comments in the auth module` → `add JSDoc comments to the undocumented functions in auth.js` → `improve the generated documentation...` → `check if the documentation follows our project standards`. 스타일(JSDoc, docstring 등)을 지정하고 예시를 요청하라.

### 노트/비코드 폴더에서 작업

Claude Code는 어느 디렉터리에서나 작동한다 — 노트 vault, 문서 폴더, 마크다운 모음에서 코드처럼 검색·편집·재구성. `.claude/`와 `CLAUDE.md`는 다른 도구의 설정 디렉터리와 충돌 없이 공존한다. Claude는 매 도구 호출마다 파일을 새로 읽으므로 다른 앱에서 한 편집을 다음 읽기 때 본다.

### 이미지 작업

이미지 추가 방법: (1) 창에 드래그앤드롭, (2) 복사 후 `ctrl+v`로 붙여넣기(**cmd+v 아님**), (3) 경로 제공 `Analyze this image: /path/to/your/image.png`. 활용: `What does this image show?`, `Describe the UI elements in this screenshot`, `Here's a screenshot of the error. What's causing it?`, `Generate CSS to match this design mockup`. 다이어그램·스크린샷·목업 모두 가능. Claude가 `[Image #1]`을 참조할 때 `Cmd+Click`(Mac)/`Ctrl+Click`(Win/Linux)으로 뷰어에서 연다.

### 파일/디렉터리 참조 (`@`)

| 형태 | 예시 | 효과 |
| --- | --- | --- |
| 단일 파일 | `Explain the logic in @src/utils/auth.js` | 파일 전체 내용을 대화에 포함 |
| 디렉터리 | `What's the structure of @src/components?` | 디렉터리 리스팅 제공(내용 아님) |
| MCP 리소스 | `Show me the data from @github:repos/owner/repo/issues` | `@server:resource` 형식으로 MCP 데이터 fetch |

경로는 상대/절대 모두 가능. `@` 참조는 해당 파일 디렉터리와 상위의 `CLAUDE.md`를 컨텍스트에 추가한다. 한 메시지에 여러 파일 참조 가능(`@file1.js and @file2.js`).

### 스케줄로 Claude 실행

| 옵션 | 실행 위치 | 적합 |
| --- | --- | --- |
| Routines | Anthropic 관리 인프라 | 컴퓨터 꺼져 있어도 실행. API 호출/GitHub 이벤트 트리거도 가능 |
| Desktop scheduled tasks | 데스크톱 앱(내 머신) | 로컬 파일/도구/미커밋 변경에 직접 접근 필요 시 |
| GitHub Actions | CI 파이프라인 | repo 이벤트 연동, 워크플로우 설정 옆에 둘 cron |
| `/loop` | 현재 CLI 세션 | 세션 열린 동안 빠른 폴링 |

스케줄 프롬프트는 자율 실행되어 질문 못 하므로 **성공 기준과 결과 처리를 명시**하라: "Review open PRs labeled `needs-review`, leave inline comments on any issues, and post a summary in the `#eng-reviews` Slack channel." 상세는 [[13 자동화와 스케줄링]] / [[16 CI-CD와 팀 통합]].

### Claude에게 자기 기능 묻기

Claude는 자신의 문서에 내장 접근하여 기능·한계를 답한다: `can Claude Code create pull requests?`, `how does Claude Code handle permissions?`, `what skills are available?`, `how do I use MCP with Claude Code?`, `how do I configure Claude Code for Amazon Bedrock?`, `what are the limitations of Claude Code?`. 버전과 무관하게 항상 최신 문서에 접근. 핸즈온 데모는 `/powerup`. 자세히는 [[25 트러블슈팅]].

### 그 외 핵심 레시피 요약

| 작업 | 명령 |
| --- | --- |
| 이전 대화 재개 | `claude --continue` (최근), `claude --resume`(목록), `/resume`(세션 내) — 없으면 `No conversation found to continue` |
| worktree 병렬 세션 | `claude --worktree feature-auth` (다른 터미널에 다른 이름) |
| 편집 전 계획 | `claude --permission-mode plan` 또는 세션 중 `Shift+Tab` |
| 서브에이전트 조사 위임 | `use a subagent to investigate how our auth system handles token refresh` |
| 스크립트 파이프 | `git log --oneline -20 \| claude -p "summarize these recent commits"` |

## 프롬프트 라이브러리 (prompt-library)

태스크와 역할(role)별로 태깅된 복사-붙여넣기 프롬프트 모음. 프롬프트의 `{slot}`은 자신의 값으로 치환한다. 출처(`src`)는 Common workflows / Best practices / How Anthropic teams use Claude Code 등이다.

### SDLC 단계(phase)와 카테고리

프롬프트는 5개 소프트웨어 수명주기 단계로 묶인다: **Discover → Design → Build → Ship → Operate.**

### "Start here" 추천 프롬프트

처음 시작할 때 좋은 5개 (startN 순):

| # | 단계·카테고리 | 프롬프트 |
| --- | --- | --- |
| 1 | Discover · Onboard | `give me an overview of this codebase: architecture, key directories, and how the pieces connect` |
| 2 | Discover · Understand | `explain what {path} does and how data flows through it. write it up as {format}` |
| 3 | Operate · Debug | `the {test} test is failing, find out why and fix it` |
| 4 | Build · Test | `write tests for {path}, run them, and fix any failures` |
| 5 | Build · Review | `review my uncommitted changes and flag anything that looks risky before I commit` |

### Discover — 탐색

| 카테고리 | 프롬프트 | 핵심 |
| --- | --- | --- |
| Onboard | `give me an overview of this codebase: architecture, key directories, and how the pieces connect` | 어떤 파일을 읽을지가 아니라 알고 싶은 걸 기술. 이어 `/init`으로 CLAUDE.md 세팅 |
| Understand | `explain what {path} does and how data flows through it. write it up as {format}` | 파일명+원하는 출력 형식 지정 (`format: an HTML page with a diagram, then open it in my browser`) |
| Understand | `where do we {behavior}?` | 파일명 대신 동작으로 검색 |
| Understand | `what would break if I deleted {target}?` | 삭제 전 호출자/다운스트림 영향 확인 |
| Understand | `look through the commit history of {path} and summarize how it evolved and why` | why를 물을 땐 commit history/blame을 가리킴 |
| Understand (pm/design) | `which files would I need to touch to {change}?` | 작업 규모 산정 |
| Understand (pm) | `I am a {role}. walk me through what happens when a user {action}, from the UI down to the result` | 역할 명시로 답 눈높이 조정 |

### Design — 설계

| 카테고리 | 프롬프트 | 핵심 |
| --- | --- | --- |
| Plan (pm/design) | `plan how to refactor the {target} to {goal}. list the files you would change, but don't edit anything yet` | "don't edit yet"로 탐색과 변경 분리 |
| Plan (pm) | `I want to build {feature}. interview me about implementation, UX, edge cases, and tradeoffs until we have covered everything, then write the spec to SPEC.md` | 스펙을 직접 쓰지 말고 인터뷰받기 |
| Plan (pm) | `read {input} and write up the action items, then create a {tracker} ticket for each with acceptance criteria` | 이슈 트래커(MCP/connector) 필요 |
| Plan (design/pm) | `list the error states, empty states, and edge cases for {feature} that the design needs to cover` | 디자인이 다룰 상태 열거 |
| Prototype (design/pm/marketing) | `here is a mockup. build a working prototype I can click through, matching the layout and states shown` | 목업 이미지 붙여넣기 |
| Prototype (design) | `implement this design, then take a screenshot of the result, compare it to the original, and fix any differences` | 디자인 이미지 + 브라우저 스크린샷 필요 |

### Build — 구현/테스트/리팩터/리뷰/조정

**Implement:**

| 프롬프트 | 핵심 |
| --- | --- |
| `look at how {example} is implemented to understand the pattern, then build {new} the same way` | 기존 패턴 따라가기 |
| `find {scope} without {format} comments and add them, matching the style already used in the file` | 문서 생성(docs) |
| `add a {endpoint} endpoint that returns {payload}` | 작고 잘 정의된 추가 |
| `create a {tool} using HTML, CSS, and vanilla JavaScript, then open it in my browser` | 사내 작은 도구 |
| `read issue #{issue}, implement the fix, and run the tests` | 이슈 end-to-end |
| `find every place we say "{copy}" or a close variant, show me each one in context, then update them all to "{new}". leave tests and the changelog alone` | 문구 일괄 변경 |
| `read the {examples} in {folder} to learn the structure and voice, then draft a new one for {topic}` | 구조·톤 학습 후 작성 |

**Test:**

| 프롬프트 | 핵심 |
| --- | --- |
| `write tests for {path}, run them, and fix any failures` | 기본 테스트 작성 |
| `write tests for {feature} first, then implement it until they pass` | TDD |
| `read {report} and add tests for the lowest-covered files until each is above {target}%` | 커버리지 리포트 기반 |

**Refactor:**

| 프롬프트 | 핵심 |
| --- | --- |
| `migrate everything from {from} to {to}: identify every place that needs to change, then make the changes` | 전면 마이그레이션 |
| `port {source} to {target}, keeping the same {keep}` | 포팅 |
| `optimize {target} to bring {metric} from {current} down to under {goal}` | 성능 최적화 |
| `the {element} extends {amount} beyond the {container} on {viewport}. fix it.` | UI 레이아웃 수정(design) |

**Review:**

| 프롬프트 | 핵심 |
| --- | --- |
| `review my uncommitted changes and flag anything that looks risky before I commit` | 커밋 전 리뷰 |
| `review PR #{pr} and summarize what changed, then list any concerns` | PR 리뷰 |
| `here is my Terraform plan output. what is this going to do, and is anything here going to cause problems?` | Terraform plan 검토 |
| `use a subagent to review {path} for security issues and report what it finds` | 보안 리뷰(서브에이전트) |
| `review {file} for {concerns} and list anything I should fix before it goes to {reviewer}` | 일반 리뷰 |

**Steer (조정):**

| 프롬프트 | 핵심 |
| --- | --- |
| `that is not right: {feedback}. try a different approach` | 방향 재설정 |
| `that is too much. keep only the changes to {scope} and undo your other edits` | 변경 범위 축소 |
| `you keep {mistake}. add a rule to CLAUDE.md so this stops happening` | 반복 실수를 CLAUDE.md 규칙으로 |

### Ship — 출하

| 카테고리 | 프롬프트 | 핵심 |
| --- | --- | --- |
| Git | `resolve the merge conflicts in this branch and explain what you kept from each side` | 머지 충돌 해결 |
| Git | `commit these changes with a message that summarizes what I did` | 커밋 |

| Release | `write a GitHub Actions workflow that {steps} on every push to {branch}` | CI 워크플로우. [[16 CI-CD와 팀 통합]] |

### Operate — 운영

### 프롬프트별 사전 조건(Needs)과 입력(Paste)

## worktree로 병렬 세션 실행 (worktrees)

git worktree는 자체 파일과 브랜치를 가지면서 같은 레포 히스토리·리모트를 공유하는 별도 작업 디렉터리다. 각 Claude Code 세션을 자기 worktree에서 돌리면 한 세션의 편집이 다른 세션 파일을 건드리지 않는다 — 한 터미널에서 기능을 만들면서 다른 터미널에서 버그를 고칠 수 있다. worktree는 **파일 편집을 격리**하고, 서브에이전트·에이전트 팀은 **작업 자체를 조율**한다([[08 서브에이전트와 에이전트 팀]]).

### worktree에서 Claude 시작

| 명령 | 동작 |
| --- | --- |
| `claude --worktree feature-auth` 또는 `-w` | `.claude/worktrees/feature-auth/`에 `worktree-feature-auth` 새 브랜치로 격리 worktree 생성 후 시작 |
| `claude --worktree bugfix-123` (다른 터미널) | 두 번째 격리 세션 |
| `claude --worktree` (이름 생략) | `bright-running-fox` 같은 이름 자동 생성 |
| 세션 중 "work in a worktree" | `EnterWorktree` 도구로 생성. 다른 worktree로 직접 전환도 가능, 이전 것은 디스크에 그대로 |

처음 `--worktree`를 쓰기 전에 해당 디렉터리에서 `claude`를 한 번 실행해 workspace trust 다이얼로그를 수락해야 한다(`-p` 조합 포함). 위치를 바꾸려면 [`WorktreeCreate` hook](https://code.claude.com/docs/en/hooks#worktreecreate) 설정.

> [!tip] gitignore
> `.claude/worktrees/`를 `.gitignore`에 추가해 worktree 내용이 메인 체크아웃에서 untracked로 안 보이게 하라.

### base 브랜치 선택

worktree는 기본적으로 `origin/HEAD`(리모트 매칭 clean tree)에서 분기한다. 리모트가 없거나 fetch 실패 시 로컬 `HEAD`로 폴백. 항상 로컬 `HEAD`에서 분기하려면 settings에 `worktree.baseRef`를 `"head"`로 설정 — 미푸시 커밋과 feature 브랜치 상태를 가져온다(진행 중 작업으로 서브에이전트 격리할 때 유용). `"fresh"` 또는 `"head"`만 허용([[04 설정]]):

```json
{ "worktree": { "baseRef": "head" } }
```

특정 PR에서 분기하려면 `#` 접두 PR 번호 또는 GitHub PR URL을 넘긴다. `pull/<number>/head`를 fetch해 `.claude/worktrees/pr-<number>`에 생성:

```bash
claude --worktree "#1234"
```

### gitignore된 파일을 worktree로 복사

worktree는 신선한 체크아웃이라 `.env` 같은 untracked 파일이 없다. 프로젝트 루트에 `.worktreeinclude`를 두면 자동 복사된다. `.gitignore` 문법을 쓰며, **패턴 매치 + gitignore된** 파일만 복사하므로 tracked 파일은 절대 중복되지 않는다:

```text .worktreeinclude
.env
.env.local
config/secrets.json
```

`--worktree`, 서브에이전트 worktree, 데스크톱 앱 병렬 세션에 적용된다([[14 IDE와 데스크톱]]).

### 서브에이전트를 worktree로 격리

"use worktrees for your agents"라고 하거나 커스텀 서브에이전트 frontmatter에 `isolation: worktree`를 추가하면 각 서브에이전트가 임시 worktree를 받고, 변경 없이 끝나면 자동 제거된다. base 브랜치는 `--worktree`와 동일.

### worktree 정리

종료 시 변경 여부에 따라:

| 상태 | 동작 |
| --- | --- |
| 미커밋 변경·untracked·새 커밋 없음 | worktree와 브랜치 자동 제거. 세션에 이름이 있으면 보존 여부를 물음 |
| 변경/untracked/새 커밋 존재 | 보존/제거 프롬프트. 제거 시 디렉터리·브랜치와 미커밋 변경 모두 삭제 |
| 비대화형(`-p`) 실행 | 종료 프롬프트가 없어 자동 정리 안 됨 → `git worktree remove`로 직접 제거 |

서브에이전트·백그라운드 세션용 worktree는 `cleanupPeriodDays`([[04 설정]])보다 오래되고 미커밋/untracked/미푸시가 없으면 자동 정리된다. `--worktree`로 직접 만든 것은 이 sweep에서 제거되지 않는다.

### 수동 관리 (git 직접)

```bash
git worktree add ../project-feature-a -b feature-a   # 새 브랜치로
git worktree add ../project-bugfix bugfix-123          # 기존 브랜치에서
cd ../project-feature-a && claude                       # worktree에서 시작
git worktree list                                       # 목록
git worktree remove ../project-feature-a                # 제거
```

새 worktree마다 개발 환경(의존성 설치, 가상환경 등)을 초기화해야 함.

### 비-git 버전 관리

SVN·Perforce·Mercurial 등은 `WorktreeCreate`/`WorktreeRemove` 훅으로 커스텀 생성·정리 로직을 제공한다. 훅이 기본 git 동작을 대체하므로 `--worktree` 사용 시 `.worktreeinclude`는 처리되지 않는다 — 로컬 설정 파일 복사를 훅 스크립트 안에서 하라. SVN 예시 (이름을 stdin에서 읽고 체크아웃 후 디렉터리 경로 출력):

```json
{
  "hooks": {
    "WorktreeCreate": [{
      "hooks": [{
        "type": "command",
        "command": "bash -c 'NAME=$(jq -r .name); DIR=\"$HOME/.claude/worktrees/$NAME\"; svn checkout https://svn.example.com/repo/trunk \"$DIR\" >&2 && echo \"$DIR\"'"
      }]
    }]
  }
}
```

세션 종료 정리는 `WorktreeRemove` 훅과 페어링. 훅 입력 스키마는 [[09 훅]].

## ultraplan — 클라우드 계획 (ultraplan)

> [!note] 리서치 프리뷰
> Ultraplan은 리서치 프리뷰이며 Claude Code v2.1.91 이상 필요. 동작/기능이 바뀔 수 있다.

Ultraplan은 로컬 CLI의 계획 작업을 [Claude Code on the web]([[15 웹과 모바일]])의 plan mode 세션으로 넘긴다. Claude가 클라우드에서 계획을 작성하는 동안 터미널은 다른 작업에 쓸 수 있다. 계획이 준비되면 브라우저에서 특정 섹션에 코멘트·수정 요청을 하고 실행 위치를 고른다. 터미널보다 풍부한 리뷰 surface가 필요할 때 유용하다(섹션별 타깃 피드백, hands-off 작성, 유연한 실행).

[Claude Code on the web] 계정과 GitHub 레포가 필요하다. Anthropic 클라우드 인프라에서 돌므로 **Amazon Bedrock / Google Cloud Vertex AI / Microsoft Foundry에서는 사용 불가**([[17 클라우드 프로바이더]]). 클라우드 환경이 없으면 첫 실행 시 자동 생성된다.

### CLI에서 실행하는 3가지 방법

| 방법 | 사용 |
| --- | --- |
| Command | `/ultraplan migrate the auth service from sessions to JWTs` |
| Keyword | 일반 프롬프트 아무 데나 `ultraplan` 단어 포함 |
| 로컬 계획에서 | 로컬 계획 승인 다이얼로그에서 **No, refine with Ultraplan on Claude Code on the web** 선택 |

Command/Keyword 경로는 실행 전 확인 다이얼로그를 띄운다(로컬 계획 경로는 선택 자체가 확인이라 생략). [Remote Control]이 활성화돼 있으면 ultraplan 시작 시 끊긴다(둘 다 claude.ai/code 인터페이스를 점유, 동시 연결 불가).

CLI 프롬프트 입력에 상태 표시기가 뜬다:

| 상태 | 의미 |
| --- | --- |
| `◇ ultraplan` | 코드베이스 조사 및 계획 작성 중 |
| `◇ ultraplan needs your input` | 명확화 질문 있음, 세션 링크 열어 응답 |
| `◆ ultraplan ready` | 브라우저에서 리뷰 준비됨 |

`/tasks`에서 ultraplan 항목을 선택하면 세션 링크·에이전트 활동·**Stop ultraplan** 액션이 있는 상세 뷰가 열린다. 중단하면 클라우드 세션이 아카이브되고 터미널에는 아무것도 저장되지 않는다.

### 리뷰·수정·실행 위치 선택

브라우저 리뷰 뷰: **인라인 코멘트**(구절 강조 후 코멘트), **이모지 반응**, **아웃라인 사이드바**. 코멘트 반영을 요청하면 계획을 수정해 갱신 초안을 제시 — 원하는 만큼 반복 가능.

실행 위치:

| 선택 | 동작 |
| --- | --- |
| **Approve Claude's plan and start coding** | 같은 클라우드 세션에서 구현. 완료 후 웹에서 diff 리뷰 → PR 생성 |
| **Approve plan and teleport back to terminal** | 로컬에서 구현(환경 전체 접근). CLI에서 시작했고 터미널이 polling 중일 때만 표시. 웹 세션은 아카이브 |

터미널로 보내면 **Ultraplan approved** 다이얼로그에 3옵션: **Implement here**(현재 대화에 계획 주입), **Start new session**(현재 대화 비우고 계획만 컨텍스트로 새로 시작 — 상단에 `claude --resume` 명령 출력), **Cancel**(실행 없이 파일로 저장, 경로 출력).

관련: [[15 웹과 모바일]](클라우드 인프라), [[05 권한]](plan mode).

## ultrareview — 클라우드 버그 리뷰 (ultrareview)

> [!note] 리서치 프리뷰
> Ultrareview는 리서치 프리뷰(v2.1.86+). 명령은 이제 `/code-review ultra`로 호출하고 `/ultrareview`는 별칭으로 남아 있다. 기능·가격·가용성이 바뀔 수 있다.

Ultrareview는 Claude Code on the web 인프라의 원격 샌드박스에서 **리뷰어 에이전트 fleet**을 띄워 브랜치나 PR의 버그를 찾는 깊은 멀티에이전트 코드 리뷰다. 로컬 `/review` 대비:

- **높은 신호**: 모든 발견을 독립적으로 재현·검증해 스타일 제안이 아닌 실제 버그에 집중
- **넓은 커버리지**: 다수 리뷰어 에이전트가 병렬 탐색해 단일 패스가 놓치는 이슈 발견
- **로컬 자원 미사용**: 원격 샌드박스에서 돌아 터미널이 자유로움

Claude.ai 계정 인증 필요(API 키만이면 `/login`). **Bedrock/Vertex AI/Foundry 불가**, Zero Data Retention 조직 불가.

### CLI에서 실행

| 명령 | 범위 |
| --- | --- |
| `/code-review ultra` | 현재 브랜치 vs 기본 브랜치 diff (미커밋·staged 포함). 레포 상태를 번들해 원격 샌드박스로 업로드 |
| `/code-review ultra 1234` | GitHub PR. 원격 샌드박스가 호스트에서 PR 직접 clone. github.com 및 GitHub Enterprise Server 지원 |

레포가 너무 커서 번들 불가하면 PR 모드 사용을 권유한다(브랜치 push 후 draft PR 열고 `/code-review ultra <PR-number>`). 실행 전 리뷰 범위(파일·라인 수)·남은 무료 실행·예상 비용 확인 다이얼로그를 띄운다. **Claude가 스스로 시작하지 않는다** — `/code-review ultra`로 직접 호출해야만 실행.

### 가격과 무료 실행

usage credit으로 청구되는 프리미엄 기능이다(플랜 포함 사용량이 아님).

| 플랜 | 무료 실행 | 이후 |
| --- | --- | --- |
| Pro | 3회 | usage credits 청구 |
| Max | 3회 | usage credits 청구 |
| Team / Enterprise | 없음 | usage credits 청구 |

Pro/Max의 3회는 계정당 일회성이며 갱신되지 않는다. 소진 후 리뷰당 보통 $5~$20(변경 크기에 따라). 원격 세션이 시작되면 1회 카운트 — 일찍 중단하거나 실패해도 무료 실행을 소비한다(유료 리뷰는 실행된 부분만 청구). 무료 실행 밖에서는 항상 usage credit으로 청구되므로 계정/조직에 usage credit이 켜져 있어야 한다. 꺼져 있으면 launch가 차단되고 billing 설정으로 안내한다. `/usage-credits`로 확인·변경. 비용은 [[19 비용과 성능]].

### 진행 추적

보통 5~10분 소요, 백그라운드 태스크라 계속 작업·다른 명령·터미널 종료 가능. `/tasks`로 실행 중/완료 리뷰 조회·상세 뷰·중단. 중단하면 클라우드 세션 아카이브되고 부분 발견은 반환 안 됨. 완료 시 검증된 발견이 세션 알림으로 표시되며 각 발견은 파일 위치+이슈 설명을 포함해 바로 고치도록 요청 가능.

### 비대화형 실행

`claude ultrareview` 서브커맨드로 CI/스크립트에서 인터랙티브 세션 없이 시작. 원격 리뷰 완료까지 블록하고 발견을 stdout에 출력, 성공 exit 0 / 실패 1:

```bash
claude ultrareview              # 현재 브랜치 vs 기본 브랜치
claude ultrareview 1234         # PR
claude ultrareview origin/main  # 특정 base 브랜치 대비
```

| 플래그 | 설명 |
| --- | --- |
| `--json` | 포맷된 발견 대신 raw `bugs.json` 페이로드 출력 |
| `--timeout <minutes>` | 최대 대기 분(기본 30) |

진행 메시지·라이브 세션 URL은 stderr로 가 stdout이 파싱 가능하게 유지된다. exit code: 0(완료, 발견 유무 무관), 1(launch 실패·원격 에러·타임아웃), 130(Ctrl-C 인터럽트 — 원격 리뷰는 계속 진행되니 stderr의 세션 URL로 추적). 서브커맨드 호출 자체가 billing/terms 프롬프트 동의로 간주된다. GitHub PR 자동 리뷰는 별도 Code Review 통합([[16 CI-CD와 팀 통합]]).

### `/review` vs `/code-review ultra` 비교

| | `/review` | `/code-review ultra` |
| --- | --- | --- |
| 실행 | 로컬 세션 | 원격 클라우드 샌드박스 |
| 깊이 | 단일 패스 | 멀티에이전트 fleet + 독립 검증 |
| 시간 | 초~수 분 | 약 5~10분 |
| 비용 | 일반 사용량 | 무료 실행 후 $5~$20/회 usage credits |
| 적합 | 반복 중 빠른 피드백 | 큰 변경의 머지 전 확신 |

## /goal — 목표 기반 자동 진행 (goal)

> [!note] 버전
> `/goal`은 Claude Code v2.1.139 이상 필요.

`/goal`은 완료 조건을 설정하면 Claude가 매 단계 프롬프트 없이 그 조건을 향해 계속 작업한다. 매 턴 후 작고 빠른 모델이 조건 충족 여부를 체크하고, 안 됐으면 제어권을 돌려주는 대신 다음 턴을 시작한다. 조건이 충족되면 목표가 자동으로 해제된다. 검증 가능한 종료 상태가 있는 큰 작업에 쓴다 — 모든 호출처가 컴파일·테스트 통과할 때까지 API 마이그레이션, 모든 수용 기준이 성립할 때까지 디자인 doc 구현, 각 모듈이 크기 예산 이하가 될 때까지 큰 파일 분할, 라벨된 이슈 백로그 비우기 등.

### 세션 유지 방식 비교

세 접근이 프롬프트 사이 현재 세션을 유지한다. 다음 턴이 무엇으로 시작되는지에 따라 고른다:

| 접근 | 다음 턴 시작 | 종료 |
| --- | --- | --- |
| `/goal` | 이전 턴 종료 시 | 모델이 조건 충족 확인 |
| `/loop` | 시간 간격 경과 | 직접 중단 또는 Claude가 끝났다고 판단 |
| Stop hook | 이전 턴 종료 시 | 자체 스크립트/프롬프트 결정 |

`/goal`은 세션 한정 단축키(현재 세션에만 활성), Stop hook은 settings 파일에 살며 스코프 내 모든 세션에 적용되고 결정적 스크립트 또는 모델 평가 프롬프트를 돌린다. Auto mode는 한 턴 *내* 도구 호출만 승인하고 새 턴을 시작하진 않는다 — `/goal`은 매 턴 후 신선한 모델로 조건을 체크하는 별도 평가자를 더한다(상보적: auto mode는 도구별 프롬프트 제거, `/goal`은 턴별 프롬프트 제거). 자세히는 [[13 자동화와 스케줄링]].

### 사용

세션당 목표 하나. 같은 명령이 인자에 따라 설정·확인·해제한다.

```text
/goal all tests in test/auth pass and the lint step is clean
```

목표 설정은 조건 자체를 디렉티브로 즉시 한 턴을 시작한다(별도 프롬프트 불필요). 활성 동안 `◎ /goal active` 표시기가 경과 시간을 보여준다. 매 턴 후 평가자가 충족/미충족 이유를 짧게 반환해 상태 뷰와 transcript에 표시된다.

### 효과적인 조건 작성

평가자는 **Claude가 대화에 surface한 것**으로만 조건을 판단한다 — 명령을 직접 실행하거나 파일을 독립적으로 읽지 않는다. 따라서 Claude의 출력으로 증명 가능한 것으로 조건을 써라("All tests in `test/auth` pass"는 Claude가 테스트를 돌려 결과가 transcript에 남아 동작). 견고한 조건은:

- **하나의 측정 가능한 종료 상태**: 테스트 결과, 빌드 exit code, 파일 수, 빈 큐
- **명시된 체크**: 증명 방법 — "`npm test` exits 0", "`git status` is clean"
- **중요한 제약**: 도중에 바뀌면 안 되는 것 — "no other test file is modified"

조건은 최대 4,000자. 실행 시간을 제한하려면 `or stop after 20 turns` 같은 턴/시간 절을 넣어라.

### 상태·해제·재개

| 명령 | 동작 |
| --- | --- |
| `/goal` (인자 없음) | 현재 상태(조건, 경과 시간, 평가된 턴 수, 토큰 소비, 평가자 최근 이유) |
| `/goal clear` | 충족 전 활성 목표 제거. `stop`/`off`/`reset`/`none`/`cancel` 별칭. `/clear`도 목표 제거 |

세션 종료 시 활성이던 목표는 `--resume`/`--continue`로 재개하면 조건이 복원된다(단 턴 수·타이머·토큰 baseline은 리셋). 이미 충족·해제된 목표는 복원 안 됨.

### 비대화형 실행

[[02 CLI 레퍼런스]]의 `-p`로 목표를 설정하면 한 호출에서 루프를 완료까지 돌린다:

```bash
claude -p "/goal CHANGELOG.md has an entry for every PR merged this week"
```

Ctrl+C로 충족 전 중단. 데스크톱 앱·Remote Control에서도 작동.

### 평가 원리와 요구사항

`/goal`은 세션 한정 **prompt-based Stop hook**의 래퍼다([[09 훅]]). 매 턴 종료 시 조건과 지금까지의 대화가 설정된 small fast model(기본 Haiku, [[04 설정]] model-config)로 전송되고, 모델이 yes/no와 짧은 이유를 반환한다. "no"는 계속 작업하라며 이유를 다음 턴 가이드로 포함, "yes"는 목표를 해제하고 achieved 항목을 transcript에 기록. 평가자는 세션이 설정된 프로바이더에서 돌고 도구를 호출하지 않으므로 이미 surface된 것만 판단한다. 평가 토큰은 small fast model로 청구되며 메인 턴 대비 보통 미미하다.

`/goal`은 trust 다이얼로그를 수락한 워크스페이스에서만 동작한다(평가자가 훅 시스템 일부). 어느 settings 레벨에서든 `disableAllHooks`가 설정됐거나 managed settings에 `allowManagedHooksOnly`가 설정되면 사용 불가하며, 이 경우 조용히 무시하지 않고 이유를 알려준다. 보안 설정은 [[18 보안과 샌드박스]].
