# 04 설정

Claude Code의 설정 시스템 전체 레퍼런스. `settings.json` 키, 환경변수, 모델 구성, auto mode 분류기, 설정 디버깅, 서버 관리 설정, 조직 배포까지 다룬다. 허브: [[Claude]]. 권한 규칙 상세는 [[05 권한]], 환경변수가 제어하는 도구 동작은 [[06 내장 도구 레퍼런스]], 메모리(CLAUDE.md) 로딩은 [[03 메모리와 컨텍스트]] 참고.

`/config` 명령(인터랙티브 REPL)으로 탭형 설정 인터페이스를 열어 상태 확인과 옵션 수정이 가능하다.

## 설정 스코프 (Configuration scopes)

Claude Code는 **스코프 시스템**으로 설정이 어디에 적용되고 누구에게 공유되는지를 결정한다. 개인용, 팀 협업용, 엔터프라이즈 배포용을 구분하는 핵심 개념이다.

### 사용 가능한 스코프

| Scope | Location | 영향 범위 | 팀 공유? |
|:------|:---------|:---------|:--------|
| **Managed** | 서버 관리 설정, plist/registry, 또는 시스템 레벨 `managed-settings.json` | 머신의 모든 사용자 | Yes (IT 배포) |
| **User** | `~/.claude/` 디렉토리 | 본인, 모든 프로젝트 | No |
| **Project** | 저장소의 `.claude/` | 이 저장소의 모든 협업자 | Yes (git 커밋) |
| **Local** | `.claude/settings.local.json` | 본인, 이 저장소에서만 | No (gitignore됨) |

Windows에서 `~/.claude`는 `%USERPROFILE%\.claude`로 해석된다.

### 스코프별 용도

- **Managed**: 조직 전역 강제 보안 정책, 재정의 불가한 컴플라이언스 요구사항, IT/DevOps 표준 배포.
- **User**: 어디서나 적용할 개인 선호(테마, 에디터), 전 프로젝트 공통 도구·플러그인, API 키·인증.
- **Project**: 팀 공유 설정(권한, 훅, MCP 서버), 팀 전체 플러그인, 협업자 간 도구 표준화.
- **Local**: 특정 프로젝트의 개인 오버라이드, 팀 공유 전 설정 테스트, 머신 종속 설정.

### 스코프 우선순위 (How scopes interact)

같은 설정이 여러 스코프에 있으면 다음 우선순위로 적용된다(높은 순):

1. **Managed** (최상위) — 무엇으로도 재정의 불가
2. **Command line arguments** — 세션 임시 오버라이드
3. **Local** — project, user 재정의
4. **Project** — user 재정의
5. **User** (최하위) — 다른 곳에서 지정 안 됐을 때

> 주의: 권한 규칙(permission rules)은 다르게 동작한다. 스칼라 값은 높은 스코프가 재정의하지만, **배열 값(permission 규칙, sandbox 경로 등)은 스코프 간 병합(concat + dedup)** 된다. 즉 낮은 스코프가 높은 스코프의 항목을 지우지 못하고 추가만 가능하다.

### 기능별 스코프 위치

| 기능 | User 위치 | Project 위치 | Local 위치 |
|:-----|:----------|:-------------|:-----------|
| **Settings** | `~/.claude/settings.json` | `.claude/settings.json` | `.claude/settings.local.json` |
| **Subagents** | `~/.claude/agents/` | `.claude/agents/` | None |
| **MCP servers** | `~/.claude.json` | `.mcp.json` | `~/.claude.json` (per-project) |
| **Plugins** | `~/.claude/settings.json` | `.claude/settings.json` | `.claude/settings.local.json` |
| **CLAUDE.md** | `~/.claude/CLAUDE.md` | `CLAUDE.md` 또는 `.claude/CLAUDE.md` | `CLAUDE.local.md` |

## 설정 파일 (Settings files)

`settings.json`이 Claude Code 구성의 공식 메커니즘이다.

- **User**: `~/.claude/settings.json` — 모든 프로젝트 적용
- **Project**:
  - `.claude/settings.json` — 소스 컨트롤에 체크인, 팀 공유
  - `.claude/settings.local.json` — 체크인 안 함, 개인 선호·실험용. 생성 시 Claude Code가 자동으로 git ignore 설정.
- **Managed settings** (조직 중앙 통제용, 여러 전달 메커니즘, 동일 JSON 포맷, user/project가 재정의 불가):
  - **Server-managed**: Claude.ai 어드민 콘솔에서 Anthropic 서버 통해 전달 → [server-managed settings 섹션](#서버-관리-설정-server-managed-settings)
  - **MDM/OS-level policies**:
    - macOS: `com.anthropic.claudecode` managed preferences 도메인 (plist). Jamf, Kandji(Iru) 등 MDM 구성 프로파일로 배포
    - Windows: `HKLM\SOFTWARE\Policies\ClaudeCode` 레지스트리 키의 `Settings` 값(REG_SZ/REG_EXPAND_SZ, JSON 내용). Group Policy 또는 Intune 배포
    - Windows (user-level): `HKCU\SOFTWARE\Policies\ClaudeCode` (최하위 정책 우선순위)
  - **File-based**: 시스템 디렉토리의 `managed-settings.json`, `managed-mcp.json`:
    - macOS: `/Library/Application Support/ClaudeCode/`
    - Linux/WSL: `/etc/claude-code/`
    - Windows: `C:\Program Files\ClaudeCode\`

> [!warning] 레거시 경로 폐기
> Windows 레거시 경로 `C:\ProgramData\ClaudeCode\managed-settings.json`은 v2.1.75부터 지원 중단. `C:\Program Files\ClaudeCode\managed-settings.json`으로 마이그레이션 필요.

**Drop-in 디렉토리**: 같은 시스템 디렉토리의 `managed-settings.d/`에 정책 조각을 분리 배포 가능(systemd 컨벤션). `managed-settings.json`이 베이스로 먼저 병합되고, drop-in 디렉토리의 `*.json`이 알파벳 순으로 위에 병합. 스칼라는 나중 파일이 재정의, 배열은 concat+dedup, 객체는 deep-merge. `.`로 시작하는 숨김 파일 무시. 병합 순서 제어는 숫자 접두사 사용(예: `10-telemetry.json`, `20-security.json`).

**기타 구성**은 `~/.claude.json`에 저장: OAuth 세션, user/local 스코프 MCP 서버 구성, 프로젝트별 상태(허용 도구, trust 설정), 각종 캐시. (project 스코프 MCP는 `.mcp.json`에 별도 저장.)

> Claude Code는 구성 파일의 타임스탬프 백업을 자동 생성하고 최근 5개를 보관한다.

### settings.json 예시

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": ["Bash(npm run lint)", "Bash(npm run test *)", "Read(~/.zshrc)"],
    "deny": ["Bash(curl *)", "Read(./.env)", "Read(./.env.*)", "Read(./secrets/**)"]
  },
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_METRICS_EXPORTER": "otlp"
  },
  "companyAnnouncements": ["Welcome to Acme Corp! Review our code guidelines at docs.acme.com"]
}
```

`$schema` 라인을 추가하면 VS Code, Cursor 등 JSON schema 지원 에디터에서 자동완성·인라인 검증이 켜진다. 단 공개 스키마는 주기적으로만 갱신되므로 최신 CLI에 추가된 필드는 검증 경고가 떠도 실제론 유효할 수 있다.

### 편집 적용 시점 (When edits take effect)

Claude Code가 설정 파일을 감시하며 변경 시 리로드한다. 대부분의 키(`permissions`, `hooks`, `apiKeyHelper` 등 credential helper 포함)는 재시작 없이 실행 중 세션에 적용. user/project/local/managed 모두 커버하며, 감지된 변경마다 [[09 훅]]의 `ConfigChange` 훅이 발화한다.

세션 시작 시 한 번만 읽혀 다음 재시작에 적용되는 키:
- `model`: 세션 중 전환은 `/model` 사용 → [모델 구성](#모델-구성-model-config)
- `outputStyle`: 시스템 프롬프트 일부, `/clear` 또는 재시작 시 재빌드 → [[12 출력 스타일과 상태줄]]

### 주요 settings.json 키 (Available settings)

`settings.json`이 지원하는 옵션 일부(전체 중 대표). 자세한 권한·sandbox·플러그인 키는 아래 전용 섹션 참고.

| Key | 설명 | Example |
|:----|:-----|:--------|
| `agent` | 메인 스레드를 named subagent로 실행, `claude agents`의 기본 에이전트 설정. 해당 subagent의 시스템 프롬프트·도구 제한·모델 적용 → [[08 서브에이전트와 에이전트 팀]] | `"code-reviewer"` |
| `alwaysThinkingEnabled` | 모든 세션에서 [extended thinking](#extended-thinking) 기본 활성화. 보통 `/config`로 설정. 강제 off는 `env`에 `CLAUDE_CODE_DISABLE_THINKING` | `true` |
| `apiKeyHelper` | `/bin/sh`에서 실행할 인증값 생성 스크립트. `X-Api-Key`, `Authorization: Bearer` 헤더로 전송. refresh 간격은 `CLAUDE_CODE_API_KEY_HELPER_TTL_MS` | `/bin/generate_temp_api_key.sh` |
| `attribution` | git commit/PR 어트리뷰션 커스터마이즈 → [어트리뷰션 설정](#어트리뷰션-설정-attribution) | `{"commit": "...", "pr": ""}` |
| `autoMemoryEnabled` | [[03 메모리와 컨텍스트]] auto memory 활성화. 기본 `true`. `/memory`로 토글 가능. env: `CLAUDE_CODE_DISABLE_AUTO_MEMORY` | `false` |
| `autoMode` | [auto mode](#auto-mode-구성-auto-mode-config) 분류기 커스터마이즈. `environment`, `allow`, `soft_deny`, `hard_deny` 산문 규칙 배열. shared project settings에서 안 읽음 | `{"soft_deny": ["$defaults", "Never run terraform apply"]}` |
| `autoUpdatesChannel` | 업데이트 릴리스 채널. `"stable"`(약 1주 지난 버전, 메이저 회귀 스킵) 또는 `"latest"`(기본). 완전 비활성은 env의 `DISABLE_AUTOUPDATER` | `"stable"` |
| `availableModels` | `/model`·`--model`·`ANTHROPIC_MODEL`로 선택 가능한 모델 제한. Default 옵션엔 영향 없음 → [모델 선택 제한](#모델-선택-제한-restrict-model-selection) | `["sonnet", "haiku"]` |
| `cleanupPeriodDays` | 이보다 오래된 세션 파일을 시작 시 삭제(기본 30일, 최소 1, `0`은 거부). 고아 subagent worktree 제거 컷오프도 제어. transcript 기록 완전 중단은 env `CLAUDE_CODE_SKIP_PROMPT_HISTORY` 또는 `-p` 모드의 `--no-session-persistence` | `20` |
| `companyAnnouncements` | 시작 시 표시할 공지(여러 개면 랜덤 순환) | `["..."]` |
| `defaultShell` | 입력창 `!` 명령의 기본 셸. `"bash"`(기본) 또는 `"powershell"`. powershell은 `CLAUDE_CODE_USE_POWERSHELL_TOOL=1` 필요 → [[06 내장 도구 레퍼런스]] | `"powershell"` |
| `disableAllHooks` | 모든 [[09 훅]]과 커스텀 상태줄 비활성화 | `true` |
| `disableAutoMode` | `"disable"`로 auto mode 활성화 차단. `Shift+Tab` 사이클에서 제거, `--permission-mode auto` 거부. managed에서 가장 유용 | `"disable"` |
| `disableSkillShellExecution` | user/project/plugin [[07 스킬]]·커스텀 명령의 인라인 셸 실행 비활성화. 번들·managed 스킬은 영향 없음 | `true` |
| `editorMode` | 입력 프롬프트 키 바인딩: `"normal"`(기본) 또는 `"vim"` | `"vim"` |
| `effortLevel` | [effort 레벨](#adjust-effort-level) 세션 간 유지. `"low"`/`"medium"`/`"high"`/`"xhigh"`. `--effort`·`CLAUDE_CODE_EFFORT_LEVEL`이 1회 오버라이드 | `"xhigh"` |
| `env` | 모든 세션과 spawn 서브프로세스에 적용되는 환경변수. (v2.1.143~ `NO_COLOR`/`FORCE_COLOR`는 서브프로세스엔 전달되나 자체 UI 색은 안 바뀜) | `{"FOO": "bar"}` |
| `includeCoAuthoredBy` | **Deprecated**: `attribution` 사용. commit/PR에 `co-authored-by Claude` 포함 여부(기본 `true`) | `false` |
| `includeGitInstructions` | 내장 commit/PR 워크플로 지침 + git status 스냅샷을 시스템 프롬프트에 포함(기본 `true`). env `CLAUDE_CODE_DISABLE_GIT_INSTRUCTIONS`가 우선 | `false` |
| `language` | Claude 응답 언어(`"japanese"`, `"korean"` 등). voice dictation 언어도 설정 | `"japanese"` |
| `model` | 기본 모델 오버라이드. `--model`·`ANTHROPIC_MODEL`이 1회 오버라이드 | `"claude-sonnet-4-6"` |
| `modelOverrides` | Anthropic 모델 ID → provider 전용 ID(Bedrock ARN 등) 매핑 → [버전별 모델 ID 오버라이드](#버전별-모델-id-오버라이드) | `{"claude-opus-4-6": "arn:..."}` |
| `outputStyle` | 시스템 프롬프트 조정 출력 스타일 → [[12 출력 스타일과 상태줄]] | `"Explanatory"` |
| `permissions` | 권한 구조 → [권한 설정](#권한-설정-permission) 및 [[05 권한]] | |
| `preferredNotifChannel` | 작업완료·권한프롬프트 알림 방식: `"auto"`, `"terminal_bell"`, `"iterm2"`, `"kitty"`, `"ghostty"`, `"notifications_disabled"` 등 | `"terminal_bell"` |
| `statusLine` | 커스텀 상태줄 → [[12 출력 스타일과 상태줄]] | `{"type": "command", "command": "~/.claude/statusline.sh"}` |
| `teammateMode` | [[08 서브에이전트와 에이전트 팀]] 팀메이트 표시: `auto`/`in-process`/`tmux` | `"in-process"` |
| `tui` | 터미널 UI 렌더러: `"fullscreen"`(alt-screen, 플리커 없음) 또는 `"default"`. `/tui`로 설정. env `CLAUDE_CODE_NO_FLICKER` | `"fullscreen"` |
| `skillOverrides` (v2.1.129+) | 스킬별 가시성 오버라이드: `"on"`/`"name-only"`/`"user-invocable-only"`/`"off"` → [[07 스킬]] | `{"deploy": "off"}` |
| `viewMode` | 시작 시 transcript 뷰: `"default"`/`"verbose"`/`"focus"`. `--verbose`가 1회 오버라이드 | `"verbose"` |
| `voice` | [voice dictation](https://) 설정: `enabled`, `mode`(`"hold"`/`"tap"`), `autoSubmit`. Claude.ai 계정 필요 | `{"enabled": true, "mode": "tap"}` |

#### 관리자(Managed-only) 전용 키 발췌

managed settings에서만 읽히는 키들. user/project에 두면 무시된다. 조직 정책 강제용 → [조직 배포](#조직-배포-admin-setup), [[05 권한]], [[10 MCP]], [[11 플러그인]].

| Key | 설명 |
|:----|:-----|
| `allowManagedHooksOnly` | managed/SDK 훅 + managed `enabledPlugins`로 force-enable된 플러그인 훅만 로드. 나머지 차단 |
| `allowManagedMcpServersOnly` | managed의 `allowedMcpServers`만 인정. `deniedMcpServers`는 전 소스 병합 |
| `allowManagedPermissionRulesOnly` | user/project가 `allow`/`ask`/`deny` 규칙 정의 못 하게 함 |
| `allowedMcpServers` / `deniedMcpServers` | MCP 서버 allow/deny 리스트. deny 우선 → [[10 MCP]] |
| `strictKnownMarketplaces` | 플러그인 마켓플레이스 소스 allowlist → [strictKnownMarketplaces](#strictknownmarketplaces) |
| `blockedMarketplaces` | 마켓플레이스 소스 blocklist |
| `strictPluginOnlyCustomization` | skills/agents/hooks/mcp를 plugin·managed에서만 허용 → [strictPluginOnlyCustomization](#strictpluginonlycustomization) |
| `claudeMd` | 조직 관리 메모리로 주입되는 CLAUDE.md 지침 → [[03 메모리와 컨텍스트]] |
| `forceRemoteSettingsRefresh` | 시작 시 remote managed settings 신규 페치 완료까지 블록(fail-closed) → [fail-closed](#fail-closed-시작-강제) |
| `requiredMinimumVersion` / `requiredMaximumVersion` | 허용 버전 범위 밖이면 시작 거부 |
| `minimumVersion` | 다운그레이드 방지 floor(시작은 막지 않음) |
| `policyHelper` (v2.1.136+) | managed settings를 동적 계산하는 어드민 배포 실행파일 → [policy helper](#policy-helper로-동적-managed-settings) |
| `parentSettingsBehavior` (v2.1.133+) | SDK/IDE 임베딩 호스트의 managed settings 적용 방식: `"first-wins"`(기본) 또는 `"merge"` |
| `wslInheritsWindowsSettings` | WSL이 Windows 정책 체인도 읽게 함(HKLM 또는 `C:\Program Files`에서만 유효) |

### 글로벌 config 설정 (~/.claude.json 저장)

이 키들은 `settings.json`이 아닌 `~/.claude.json`에 저장된다. `settings.json`에 넣으면 schema 검증 에러.

| Key | 설명 | Example |
|:----|:-----|:--------|
| `autoConnectIde` | 외부 터미널에서 시작 시 실행 중 IDE 자동 연결(기본 `false`). env `CLAUDE_CODE_AUTO_CONNECT_IDE`가 우선 → [[14 IDE와 데스크톱]] | `true` |

| `externalEditorContext` | `Ctrl+G` 외부 에디터 열 때 이전 응답을 `#` 주석으로 prepend | `true` |
| `teammateDefaultModel` | 팀메이트 기본 모델(별칭 또는 `null`로 리드 모델 상속) | `"sonnet"` |

### Worktree 설정

`--worktree`의 git worktree 생성·관리 방식 제어.

| Key | 설명 | Example |
|:----|:-----|:--------|
| `worktree.baseRef` | 새 worktree 분기 기준: `"fresh"`(기본, `origin/<default>`) 또는 `"head"`(현재 로컬 HEAD) | `"head"` |

## 권한 설정 (Permission)

전체 권한 규칙 문법·도구별 패턴·보안 한계는 [[05 권한]]에서 다룬다. 여기선 `settings.json`의 `permissions` 구조만 정리.

| Key | 설명 | Example |
|:----|:-----|:--------|
| `allow` | 도구 사용 허용 규칙 배열 | `["Bash(git diff *)"]` |
| `ask` | 사용 시 확인 요청 규칙 배열 | `["Bash(git push *)"]` |
| `deny` | 사용 거부 규칙 배열(민감 파일 제외용) | `["WebFetch", "Bash(curl *)", "Read(./.env)"]` |
| `additionalDirectories` | 파일 접근용 추가 작업 디렉토리 | `["../docs/"]` |
| `defaultMode` | 시작 시 기본 [permission mode]: `default`/`acceptEdits`/`plan`/`auto`/`dontAsk`/`bypassPermissions`. (v2.1.142~ project/local에서 `auto`는 무시 — 저장소가 스스로 auto 권한을 부여 못 하게) | `"acceptEdits"` |
| `disableBypassPermissionsMode` | `"disable"`로 bypassPermissions 모드·`--dangerously-skip-permissions` 차단 | `"disable"` |
| `skipDangerousModePermissionPrompt` | bypass 진입 전 확인 프롬프트 스킵(project settings에선 무시) | `true` |

### 권한 규칙 문법 빠른 참고

규칙 형식은 `Tool` 또는 `Tool(specifier)`. 평가 순서: **deny → ask → allow**, 첫 매치가 이김.

| Rule | 효과 |
|:-----|:-----|
| `Bash` | 모든 Bash 명령 매치 |
| `Bash(npm run *)` | `npm run`으로 시작하는 명령 |
| `Read(./.env)` | `.env` 파일 읽기 |
| `WebFetch(domain:example.com)` | example.com fetch 요청 |

### 민감 파일 제외

```json
{
  "permissions": {
    "deny": ["Read(./.env)", "Read(./.env.*)", "Read(./secrets/**)", "Read(./config/credentials.json)"]
  }
}
```

deprecated `ignorePatterns`를 대체. 매칭 파일은 검색·탐색에서 제외되고 읽기가 거부된다.

## Sandbox 설정

bash 명령을 파일시스템·네트워크에서 격리한다(macOS, Linux, WSL2). 상세 보안 모델은 [[18 보안과 샌드박스]].

| Key | 설명 | Example |
|:----|:-----|:--------|
| `enabled` | bash sandboxing 활성화(기본 false) | `true` |
| `failIfUnavailable` | sandbox 시작 불가 시 에러로 종료(기본 false면 경고 후 비격리 실행). managed 강제 게이트용 | `true` |
| `autoAllowBashIfSandboxed` | sandbox 시 bash 자동 승인(기본 true) | `true` |
| `excludedCommands` | sandbox 밖에서 실행할 명령 | `["docker *"]` |
| `allowUnsandboxedCommands` | `dangerouslyDisableSandbox` 탈출구 허용(기본 true). false면 모든 명령 강제 격리 | `false` |
| `filesystem.allowWrite` | sandboxed 명령 쓰기 허용 경로(전 스코프 병합, `Edit(...)` allow 규칙과도 병합) | `["/tmp/build", "~/.kube"]` |
| `filesystem.denyWrite` | 쓰기 금지 경로 | `["/etc", "/usr/local/bin"]` |
| `filesystem.denyRead` | 읽기 금지 경로 | `["~/.aws/credentials"]` |
| `filesystem.allowRead` | `denyRead` 안에서 재허용(denyRead보다 우선) | `["."]` |
| `network.allowedDomains` | 아웃바운드 허용 도메인(와일드카드 `*.example.com`) | `["github.com", "*.npmjs.org"]` |
| `network.deniedDomains` | 차단 도메인(allowedDomains보다 우선, 전 소스 병합) | `["sensitive.cloud.example.com"]` |
| `network.allowManagedDomainsOnly` | (managed only) managed의 allowedDomains·WebFetch allow만 인정 | `true` |
| `network.allowLocalBinding` | localhost 포트 바인딩 허용(macOS only) | `true` |
| `network.httpProxyPort` / `network.socksProxyPort` | 자체 프록시 포트 | `8080` |
| `enableWeakerNestedSandbox` | 비특권 Docker용 약한 sandbox(Linux/WSL2, **보안 저하**) | `true` |
| `bwrapPath` / `socatPath` | (managed only, Linux/WSL2) bwrap/socat 바이너리 절대 경로 오버라이드 | `/opt/admin/bwrap` |

### Sandbox 경로 prefix

| Prefix | 의미 | Example |
|:-------|:-----|:--------|
| `/` | 파일시스템 루트 절대경로 | `/tmp/build` → `/tmp/build` |
| `~/` | 홈 디렉토리 기준 | `~/.kube` → `$HOME/.kube` |
| `./` 또는 prefix 없음 | project settings는 프로젝트 루트, user settings는 `~/.claude` 기준 | `./output` → `<project-root>/output` |

> sandbox filesystem 경로는 표준 컨벤션을 쓴다. 이는 [[05 권한]] Read/Edit 규칙(`//path`=절대, `/path`=프로젝트 상대)과 다르니 주의. 레거시 `//path`도 동작.

### Sandbox 구성 예시

```json
{
  "sandbox": {
    "enabled": true,
    "autoAllowBashIfSandboxed": true,
    "excludedCommands": ["docker *"],
    "filesystem": { "allowWrite": ["/tmp/build", "~/.kube"], "denyRead": ["~/.aws/credentials"] },
    "network": {
      "allowedDomains": ["github.com", "*.npmjs.org"],
      "deniedDomains": ["uploads.github.com"],
      "allowLocalBinding": true
    }
  }
}
```

## 어트리뷰션 설정 (attribution)

git commit/PR에 추가되는 어트리뷰션. commit은 git trailer(`Co-Authored-By` 등), PR은 평문.

| Key | 설명 |
|:----|:-----|
| `commit` | commit 어트리뷰션(trailer 포함). 빈 문자열이면 숨김 |
| `pr` | PR 설명 어트리뷰션. 빈 문자열이면 숨김 |

```json
{ "attribution": { "commit": "Generated with AI\n\nCo-Authored-By: AI <ai@example.com>", "pr": "" } }
```

> `attribution`이 deprecated `includeCoAuthoredBy`보다 우선. 모두 숨기려면 `commit`·`pr`를 빈 문자열로.

## File suggestion 설정

`@` 파일 경로 자동완성용 커스텀 명령. 대형 모노레포에서 사전 빌드 인덱스로 가속할 때 유용.

```json
{ "fileSuggestion": { "type": "command", "command": "~/.claude/file-suggestion.sh" } }
```

명령은 [[09 훅]]과 동일한 환경변수(`CLAUDE_PROJECT_DIR` 등)로 실행되고, stdin으로 `{"query": "src/comp"}`를 받아 stdout에 줄바꿈 구분 파일 경로(최대 15개) 출력.

```bash
#!/bin/bash
query=$(cat | jq -r '.query')
your-repo-file-index --query "$query" | head -20
```

## Hook 설정 키 (Hook configuration)

[[09 훅]]에서 훅 자체를 다룬다. 여기선 어떤 훅이 실행 가능하고 HTTP 훅이 무엇에 접근할 수 있는지 제어하는 설정.

- `allowManagedHooksOnly: true` (managed only): managed 훅 + SDK 훅 + managed `enabledPlugins`로 force-enable된 플러그인 훅만 로드. user/project/그 외 플러그인 훅 차단. trust는 full `plugin@marketplace` ID로 부여.
- `allowedHttpHookUrls`: HTTP 훅이 타깃할 수 있는 URL allowlist(`*` 와일드카드). 호스트명 매칭은 대소문자 무시. 미매칭은 무음 차단.
  ```json
  { "allowedHttpHookUrls": ["https://hooks.example.com/*", "http://localhost:*"] }
  ```
- `httpHookAllowedEnvVars`: HTTP 훅이 헤더에 보간할 수 있는 환경변수 allowlist. 각 훅의 effective `allowedEnvVars`는 교집합.
  ```json
  { "httpHookAllowedEnvVars": ["MY_TOKEN", "HOOK_SECRET"] }
  ```

## policyHelper로 동적 managed settings

`policyHelper`는 시작 시 managed settings를 계산하는 실행파일을 가리킨다. 디바이스 상태·신원·원격 서비스에서 정책을 도출할 때 사용. **MDM 또는 시스템 `managed-settings.json`에서만 honored** (user/project/HKCU/server-managed 무시).

| Key | Type | 설명 |
|:----|:-----|:-----|
| `path` | string | 헬퍼 실행파일 절대경로 |
| `timeoutMs` | number | 헬퍼 대기 시간(초과 시 실패 처리) |
| `refreshIntervalMs` | number | 백그라운드 재실행 주기. `0`=비활성, 또는 최소 `60000` |

헬퍼는 stdout에 JSON envelope를 쓰되 설정을 `managedSettings` 키 아래 둔다(top-level에 두면 적용 안 됨):

```json
{
  "managedSettings": { "permissions": { "deny": ["Read(//etc/secrets/**)"] } },
  "claudeMd": "# Organization context\n...",
  "appendSystemPrompt": "Always cite the internal style guide."
}
```

헬퍼가 비정상 종료(non-zero)하면 Claude Code는 에러 출력 후 시작을 거부한다. 가용성이 필요하면 헬퍼가 자체 캐시에서 서빙하고 `0`으로 종료해야 한다.

## 설정 우선순위 (Settings precedence)

높은 순:

1. **Managed settings** (server-managed / MDM/OS-level / file-based)
   - managed 티어 내 우선순위: server-managed > MDM/OS-level > file-based(`managed-settings.d/*.json` + `managed-settings.json`) > HKCU 레지스트리(Windows only). **하나의 managed 소스만 사용**, 티어 간 병합 안 함(file-based 내부에선 drop-in과 base가 병합).
   - 임베딩 호스트(Claude Desktop 등)의 SDK `managedSettings`는 기본적으로 managed 티어 존재 시 무시. `parentSettingsBehavior: "merge"`로 opt-in 가능(정책 강화만, 완화 불가).
2. **Command line arguments** — `--settings <file-or-json>`로 전달, 다른 레이어와 동일 병합 규칙.
3. **Local project settings** (`.claude/settings.local.json`)
4. **Shared project settings** (`.claude/settings.json`)
5. **User settings** (`~/.claude/settings.json`)

CLI, [[14 IDE와 데스크톱]] VS Code 확장, JetBrains 모두 동일 우선순위.

> [!warning] 배열 설정은 병합됨
> `sandbox.filesystem.allowWrite`, `permissions.allow` 등 배열 값은 스코프 간 **concat + dedup**, 대체가 아니다. managed가 `["/opt/company-tools"]`, user가 `["~/.kube"]`면 최종은 둘 다 포함.

### 활성 설정 확인 (Verify active settings)

Claude Code 안에서 `/status` 실행 → Status 탭의 `Setting sources` 라인이 로드된 각 레이어를 나열(`User settings`, `Project local settings` 등). managed가 적용되면 전달 채널이 괄호로 표시: `Enterprise managed settings (remote)`, `(plist)`, `(HKLM)`, `(HKCU)`, `(file)`. 키가 1개 이상 로드된 소스만 표시되므로 빈 리스트 = 소스 없음. (단 `Setting sources`는 어떤 레이어가 각 키를 공급했는지는 안 보여준다.)

## 플러그인 설정 (Plugin settings)

[[11 플러그인]]에서 플러그인 시스템을 다룬다. 여기선 `settings.json`의 플러그인 키.

```json
{
  "enabledPlugins": {
    "formatter@acme-tools": true,
    "analyzer@security-plugins": false
  },
  "extraKnownMarketplaces": {
    "acme-tools": { "source": { "source": "github", "repo": "acme-corp/claude-plugins" } }
  }
}
```

### `enabledPlugins`

`"plugin-name@marketplace-name": true/false` 형식. 어느 스코프에도 항목이 없으면 플러그인의 `defaultEnabled` 값으로 폴백.

> project settings가 user settings보다 우선하므로, `~/.claude/settings.json`에서 `false`로 둬도 project의 `.claude/settings.json`이 enable한 플러그인은 비활성화 안 된다. 머신 단위 opt-out은 `.claude/settings.local.json`에서 `false`. (managed가 force-enable한 건 이 방법으로 못 끔.)

### `extraKnownMarketplaces`

저장소에 추가 마켓플레이스를 제공. 보통 repo-level에 둬 팀원이 필요한 플러그인 소스에 접근하게 한다. trust 시 마켓플레이스·플러그인 설치를 프롬프트하고, 사용자는 스킵 가능.

**마켓플레이스 source 타입**: `github`(uses `repo`), `git`(uses `url`, self-hosted GitLab/Bitbucket 포함), `url`, `npm`, `directory`(개발용, uses `path`), `hostPattern`, `settings`(별도 호스팅 없이 인라인 선언).

추가 옵션:
- `"skipLfs": true` (source 객체 내): Git LFS 다운로드 스킵 (v2.1.153+)
- `"autoUpdate": true`: 시작 시 마켓플레이스 새로고침·플러그인 업데이트. 공식 Anthropic 마켓플레이스는 기본 true, 그 외 false.

### strictKnownMarketplaces

**managed only**: 사용자가 추가·설치 가능한 플러그인 마켓플레이스를 통제. `managed-settings.json`에서만 구성, user/project가 재정의 불가(최상위). 네트워크/파일시스템 작업 **전에** 강제(차단 소스는 실행 안 됨).

- `undefined`(기본): 제한 없음
- `[]`(빈 배열): 완전 lockdown, 새 마켓플레이스 추가 불가
- 소스 리스트: 정확히 매치하는 것만 추가 가능

소스 타입은 GitHub(`repo`+optional `ref`/`path`), git(`url`), url(`url`+optional `headers`), npm(`package`), file(`path`), directory(`path`), hostPattern(regex), pathPattern(regex). git 계열은 `ref`·`path`까지 **정확 매치** 필요(둘 다 undefined거나 정확히 동일).

```json
{
  "strictKnownMarketplaces": [
    { "source": "github", "repo": "acme-corp/approved-plugins" },
    { "source": "github", "repo": "acme-corp/security-tools", "ref": "v2.0" },
    { "source": "hostPattern", "hostPattern": "^github\\.example\\.com$" }
  ]
}
```

`strictKnownMarketplaces` vs `extraKnownMarketplaces`: 전자는 **정책 게이트**(추가 통제, 등록 안 함), 후자는 **편의**(누락 마켓플레이스 자동 설치). 둘 다 managed에 두면 제한+사전등록 동시 가능.

### strictPluginOnlyCustomization

**managed only** (v2.1.82+): skills/agents/hooks/mcp를 user·project 소스에서 차단, plugin·managed에서만 허용. `true`는 4개 전부, 배열은 지정한 것만.

```json
{ "strictPluginOnlyCustomization": ["skills", "hooks"] }
```

| Surface | lock 시 차단 | 여전히 로드 |
|:--------|:-------------|:------------|
| `skills` | `~/.claude/skills/`, `.claude/skills/` | plugin/bundled/managed 스킬 |
| `agents` | `~/.claude/agents/`, `.claude/agents/` | plugin/built-in/managed 에이전트 |
| `hooks` | user/project/local settings.json의 훅 | plugin/managed 훅 |
| `mcp` | `~/.claude.json`, `.mcp.json`의 서버 | plugin/managed-mcp 서버 |

`/plugin` 명령으로 플러그인을 인터랙티브하게 관리(브라우즈, 설치/제거, 활성/비활성, 마켓플레이스 추가).

## 환경변수 (env-vars)

환경변수로 설정 파일 편집 없이 동작을 제어한다. 모든 변수는 `settings.json`의 `env` 키에도 넣어 모든 세션·팀에 적용 가능.

### 환경변수 설정 방법

셸에서(해당 터미널 세션만 유지):
```bash
# macOS/Linux/WSL
export API_TIMEOUT_MS="1200000"
claude
```
```powershell
# Windows PowerShell
$env:API_TIMEOUT_MS = "1200000"; claude
# 영구: [Environment]::SetEnvironmentVariable("API_TIMEOUT_MS", "1200000", "User")
```
```batch
:: Windows CMD
set API_TIMEOUT_MS=1200000 & claude
:: 영구: setx API_TIMEOUT_MS "1200000"
```

settings.json `env` 키에서(실행 방식 무관하게 매번 적용):
```json
{ "env": { "API_TIMEOUT_MS": "1200000", "BASH_DEFAULT_TIMEOUT_MS": "300000" } }
```

### 우선순위 (Precedence)

동일 동작에 환경변수와 설정 필드가 둘 다 있으면 **환경변수가 우선**(예: `ANTHROPIC_MODEL`이 `model` 설정을 오버라이드). CLI 플래그·세션 명령과의 관계는 기능별로 다르다: `--model`·`/model`은 `ANTHROPIC_MODEL`을 오버라이드하지만, `CLAUDE_CODE_EFFORT_LEVEL`은 `/effort`를 오버라이드. 환경변수는 시작 시 읽히므로 변경은 다음 실행부터 적용.

## 모델 구성 (model-config)

### 사용 가능한 모델

`model` 설정에 **모델 별칭** 또는 **모델 이름**(Anthropic API: full model name / Bedrock: inference profile ARN / Foundry: deployment name / Vertex: version name)을 쓸 수 있다.

> `ANTHROPIC_BASE_URL`은 요청을 어디로 보낼지만 바꾸고 어떤 모델이 답할지는 안 바꾼다. LLM 게이트웨이 라우팅은 [[17 클라우드 프로바이더]] 참고.

#### 모델 별칭 (Model aliases)

| Alias | 동작 |
|:------|:-----|
| `default` | 모델 오버라이드를 지우고 계정 유형의 권장 모델로 복귀(별칭 자체는 아님) |
| `best` | 가장 강력한 모델(현재 `opus`와 동일) |
| `sonnet` | 일상 코딩용 최신 Sonnet |
| `opus` | 복잡한 추론용 최신 Opus |
| `haiku` | 단순 작업용 빠르고 효율적인 Haiku |
| `sonnet[1m]` | 1M 토큰 컨텍스트 Sonnet |
| `opus[1m]` | 1M 토큰 컨텍스트 Opus |
| `opusplan` | plan 모드는 `opus`, 실행은 `sonnet` 자동 전환 |

provider별 해석: Anthropic API는 `opus`→Opus 4.8, `sonnet`→Sonnet 4.6. Claude Platform on AWS는 `opus`→Opus 4.7. Bedrock/Vertex/Foundry는 `opus`→Opus 4.6, `sonnet`→Sonnet 4.5(신 모델은 full name 또는 `ANTHROPIC_DEFAULT_*_MODEL`로 선택). 별칭은 시간이 지나면 갱신되므로 고정하려면 full name(`claude-opus-4-8`) 또는 env 사용. (Opus 4.8은 v2.1.154+ 필요.)

#### 모델 설정 방법 (우선순위 순)

1. **세션 중**: `/model <alias|name>` 즉시 전환, 인자 없으면 picker. picker에서 `Enter`=전환+기본 저장, `s`=세션만. (v2.1.153~ `/model`이 user settings의 `model` 필드에 기본값으로 저장. project/managed settings는 여전히 우선하고 다음 실행에 재적용.)
2. **시작 시**: `claude --model <alias|name>` (해당 세션만)
3. **환경변수**: `ANTHROPIC_MODEL=<alias|name>` (해당 세션만)
4. **Settings**: `model` 필드(영구)

```bash
claude --model opus      # Opus로 시작
/model sonnet            # 세션 중 Sonnet 전환
```
```json
{ "model": "opus" }
```

> 동시에 다른 터미널에서 다른 모델을 쓰려면 각각 `--model`로 실행. `--resume`/`--continue`/`/resume`로 재개한 세션은 transcript 저장 시 모델 유지(현재 `model` 설정 무시). 그 모델이 retire됐으면 정상 우선순위로 폴백.

### 모델 선택 제한 (Restrict model selection)

엔터프라이즈 어드민은 managed/policy settings의 `availableModels`로 선택 가능 모델을 제한한다. `/model`·`--model`·`ANTHROPIC_MODEL`로 리스트 밖 모델 전환 불가.

```json
{ "availableModels": ["sonnet", "haiku"] }
```

picker의 **Default 옵션은 `availableModels`에 영향 안 받음**(항상 가용, tier 런타임 기본 모델). `availableModels: []`라도 Default는 쓸 수 있다.

`model` 설정은 초기 선택이지 강제가 아니다. 모델 경험을 완전 통제하려면 세 가지 조합:
- `availableModels`: 전환 가능 모델 제한
- `model`: 시작 초기 모델
- `ANTHROPIC_DEFAULT_SONNET_MODEL`/`_OPUS_MODEL`/`_HAIKU_MODEL`: Default 옵션과 별칭이 해석되는 모델 제어

```json
{
  "model": "claude-sonnet-4-5",
  "availableModels": ["claude-sonnet-4-5", "haiku"],
  "env": { "ANTHROPIC_DEFAULT_SONNET_MODEL": "claude-sonnet-4-5" }
}
```
`env` 블록 없으면 Default 선택 사용자는 최신 Sonnet을 받아 버전 핀을 우회한다. `availableModels`가 여러 레벨에 있으면 병합+dedup. 엄격한 allowlist는 managed/policy에 둔다.

### `default` 모델 동작

| 계정 유형 | default 모델 |
|:----------|:-------------|
| Max, Team Premium, Enterprise pay-as-you-go, Anthropic API | Opus 4.8 |
| Claude Platform on AWS | Opus 4.7 |
| Pro, Team Standard, Enterprise subscription seats | Sonnet 4.6 |
| Bedrock, Vertex, Foundry | Sonnet 4.5 |

Opus 사용 임계치 도달 시 Sonnet으로 자동 폴백할 수 있다.

### `opusplan` 동작

plan 모드는 `opus`(복잡한 추론·아키텍처), 실행 모드는 `sonnet`(코드 생성). plan-mode Opus 구간은 표준 200K 컨텍스트로 동작하며, [Extended context](#extended-context)의 1M 자동 업그레이드는 `opusplan`엔 적용 안 됨.

### Adjust effort level

[Effort 레벨](https://platform.claude.com/docs/en/build-with-claude/effort)은 adaptive reasoning을 제어한다. 낮을수록 빠르고 저렴, 높을수록 복잡한 문제에 깊은 추론.

| Model | Levels |
|:------|:-------|
| Opus 4.8, Opus 4.7 | `low`, `medium`, `high`, `xhigh`, `max` |
| Opus 4.6, Sonnet 4.6 | `low`, `medium`, `high`, `max` |

지원 안 되는 레벨 설정 시 그 이하 최고 지원 레벨로 폴백(예: Opus 4.6에서 `xhigh`→`high`). 기본 effort: Opus 4.8/4.6·Sonnet 4.6은 `high`, Opus 4.7은 `xhigh`. `low`/`medium`/`high`/`xhigh`는 세션 간 유지, `max`는 토큰 제약 없는 최심 추론으로 세션만(단 `CLAUDE_CODE_EFFORT_LEVEL`로 설정 시 예외).

| Level | 사용 시점 |
|:------|:----------|
| `low` | 짧고 범위 좁고 지연 민감, 지능 비민감 작업 |
| `medium` | 비용 민감, 일부 지능 트레이드오프 가능 |
| `high` | 토큰·지능 균형(Opus 4.8/4.6·Sonnet 4.6 기본) |
| `xhigh` | 더 높은 토큰 비용의 깊은 추론(Opus 4.7 기본) |
| `max` | 까다로운 작업에 도움될 수 있으나 overthinking 위험, 도입 전 테스트 |
| `ultracode` | 실질 작업마다 [dynamic workflow]를 `xhigh` 추론으로 계획하는 Claude Code 설정(세션만) |

**ultracode**는 모델 effort 레벨이 아니라 Claude Code 설정이다: 모델엔 `xhigh`를 보내고 추가로 dynamic workflow를 오케스트레이트. `--settings`나 SDK control request로 `"ultracode": true`. `effortLevel` 설정·`--effort`·`CLAUDE_CODE_EFFORT_LEVEL`의 일부가 아니다.

**ultrathink**: 프롬프트 아무 데나 `ultrathink`를 넣으면 그 턴만 깊은 추론 요청(API로 보내는 effort 레벨은 불변). "think", "think hard" 등 다른 표현은 일반 텍스트로 통과.

#### effort 설정 방법

- `/effort` (인자 없으면 슬라이더, 레벨명 직접, `auto`=모델 기본)
- `/model`에서 좌우 화살표로 effort 슬라이더 조정
- `--effort <level>` 플래그(1회)
- `CLAUDE_CODE_EFFORT_LEVEL` 환경변수(레벨명 또는 `auto`)
- `effortLevel` 설정(`low`/`medium`/`high`/`xhigh`만, `max`·`ultracode`는 세션만이라 불가)
- skill/subagent frontmatter의 `effort` → [[07 스킬]], [[08 서브에이전트와 에이전트 팀]]

우선순위: 환경변수 > 설정 레벨 > 모델 기본. frontmatter effort는 해당 skill/subagent 활성 중 세션 레벨을 오버라이드(환경변수는 못 넘음).

#### Adaptive reasoning과 고정 thinking 예산

Opus 4.7 이상은 항상 adaptive reasoning 사용(고정 thinking 예산 모드·`CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING` 미적용). Opus 4.6·Sonnet 4.6은 `CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1`로 `MAX_THINKING_TOKENS` 기반 고정 예산 모드로 회귀 가능.

### Extended thinking

응답 전 Claude가 내보내는 추론. adaptive reasoning 모델은 effort 레벨이 주 제어이고, 아래는 on/off와 표시 제어.

| Control | 방법 |
|:--------|:-----|
| 현재 세션 토글 | macOS `Option+T`, Windows/Linux `Alt+T` |
| 글로벌 기본 | `/config`에서 thinking 토글(`alwaysThinkingEnabled`로 `~/.claude/settings.json`에 저장) |
| effort 무관 비활성 | `MAX_THINKING_TOKENS=0` (다른 값은 고정 예산일 때만) |

thinking 출력은 기본 접힘. `Ctrl+O`로 verbose 토글(회색 이탤릭). Anthropic API 인터랙티브 세션은 기본 redact된 thinking 블록을 받으므로 전체 요약을 보려면 `showThinkingSummaries: true`. 접히거나 redact돼도 thinking 토큰은 과금된다.

### 커스텀 모델 옵션 추가

`ANTHROPIC_CUSTOM_MODEL_OPTION`으로 `/model` picker에 커스텀 항목 1개 추가(내장 별칭 대체 X). 게이트웨이 배포는 `CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY=1`로 게이트웨이 `/v1/models`에서 자동 채움 가능.

```bash
export ANTHROPIC_CUSTOM_MODEL_OPTION="my-gateway/claude-opus-4-7"
export ANTHROPIC_CUSTOM_MODEL_OPTION_NAME="Opus via Gateway"
export ANTHROPIC_CUSTOM_MODEL_OPTION_DESCRIPTION="Custom deployment routed through the internal LLM gateway"
```
picker 하단에 추가. NAME/DESCRIPTION 생략 시 모델 ID와 `Custom model (<id>)` 기본값. 모델 ID 검증은 스킵된다.

### 별칭 매핑 환경변수

full model name(또는 provider 등가)이어야 한다.

| Environment variable | 설명 |
|:---------------------|:-----|
| `ANTHROPIC_DEFAULT_OPUS_MODEL` | `opus`, plan 활성 시 `opusplan`이 쓸 모델 |
| `ANTHROPIC_DEFAULT_SONNET_MODEL` | `sonnet`, plan 비활성 시 `opusplan`이 쓸 모델 |
| `ANTHROPIC_DEFAULT_HAIKU_MODEL` | `haiku`, 백그라운드 기능 모델 |
| `CLAUDE_CODE_SUBAGENT_MODEL` | 모든 subagent·agent team 모델. per-invocation `model`과 frontmatter `model`을 오버라이드. `inherit`면 정상 해석 |

(`ANTHROPIC_SMALL_FAST_MODEL`은 deprecated → `ANTHROPIC_DEFAULT_HAIKU_MODEL`.)

### 서드파티 배포 모델 핀 (→ [[17 클라우드 프로바이더]])

Bedrock/Vertex/Foundry/Claude Platform on AWS 배포 시 롤아웃 전 모델 버전을 핀해야 한다. 핀 안 하면 별칭이 provider 기본 ID로 해석되는데, 이는 최신보다 뒤처지거나 사용자 계정에 미활성일 수 있다.

| Provider | Example |
|:---------|:--------|
| Bedrock | `export ANTHROPIC_DEFAULT_OPUS_MODEL='us.anthropic.claude-opus-4-8'` |
| Vertex AI | `export ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-8'` |
| Foundry | `export ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-8'` |

`SONNET`·`HAIKU`도 동일 패턴. 1M 활성은 ID에 `[1m]` 추가:
```bash
export ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-8[1m]'
```
Claude Code가 provider로 보내기 전 suffix를 제거한다. suffix는 변수당 읽히므로 한 변수에 없으면 그 모델은 200K. `availableModels`는 alias(`opus`/`sonnet`/`haiku`)로 매치(provider-specific ID 아님).

#### 핀 모델 표시·기능 커스터마이즈

provider-specific ID는 picker에 그대로 보이고 Claude Code가 기능 지원을 인식 못 할 수 있다. 동반 변수로 표시명·기능 선언:

| Environment variable | 설명 |
|:---------------------|:-----|
| `ANTHROPIC_DEFAULT_OPUS_MODEL_NAME` | picker 표시명(기본=모델 ID) |
| `ANTHROPIC_DEFAULT_OPUS_MODEL_DESCRIPTION` | picker 설명(기본=`Custom Opus model`) |
| `ANTHROPIC_DEFAULT_OPUS_MODEL_SUPPORTED_CAPABILITIES` | 지원 기능 쉼표 구분 리스트 |

(`SONNET`·`HAIKU`·`CUSTOM_MODEL_OPTION`에도 동일 suffix.) `_NAME`·`_DESCRIPTION`은 `ANTHROPIC_BASE_URL`이 LLM 게이트웨이일 때도 적용. `api.anthropic.com` 직접 연결엔 무효.

| Capability | 활성화 |
|:-----------|:-------|
| `effort` | effort 레벨과 `/effort` |
| `xhigh_effort` | `xhigh` 레벨 (v2.1.111+) |
| `max_effort` | `max` 레벨 |
| `thinking` | extended thinking |
| `adaptive_thinking` | adaptive reasoning |
| `interleaved_thinking` | 도구 호출 사이 thinking |

```bash
export ANTHROPIC_DEFAULT_OPUS_MODEL='arn:aws:bedrock:us-east-1:123456789012:custom-model/abc'
export ANTHROPIC_DEFAULT_OPUS_MODEL_NAME='Opus via Bedrock'
export ANTHROPIC_DEFAULT_OPUS_MODEL_SUPPORTED_CAPABILITIES='effort,xhigh_effort,max_effort,thinking,adaptive_thinking,interleaved_thinking'
```

#### 버전별 모델 ID 오버라이드

같은 family 내 여러 버전을 distinct provider ID로 매핑하려면 family 변수 대신 `modelOverrides` 설정 사용.

```json
{
  "modelOverrides": {
    "claude-opus-4-7": "arn:aws:bedrock:us-east-2:123456789012:application-inference-profile/opus-prod",
    "claude-sonnet-4-6": "arn:aws:bedrock:us-east-2:123456789012:application-inference-profile/sonnet-prod"

}
```
키는 Anthropic 모델 ID. 미지정 키는 무시. Bedrock에선 자동 발견 inference profile보다 우선. `ANTHROPIC_MODEL`·`--model`·`ANTHROPIC_DEFAULT_*_MODEL`로 직접 준 값은 변형 안 됨. `availableModels`는 Anthropic ID로 평가하므로 ARN 매핑돼도 `"opus"` 항목은 계속 매치.

## auto mode 구성 (auto-mode-config)

[auto mode]는 권한 프롬프트 없이 실행하되, 각 도구 호출을 분류기에 통과시켜 비가역·파괴적·환경 외부 대상 작업을 차단한다. `autoMode` 설정 블록으로 분류기에 신뢰하는 repo·bucket·domain을 알려 루틴 내부 작업을 차단하지 않게 한다.

> auto mode는 Anthropic API에서 모든 사용자 가용. Bedrock/Vertex/Foundry는 `CLAUDE_CODE_ENABLE_AUTO_MODE` 먼저 설정 필요. 기본적으로 분류기는 작업 디렉토리와 현재 repo의 원격만 신뢰 — 회사 source-control org에 push, 팀 cloud bucket 쓰기 등은 `autoMode.environment`에 추가하기 전까지 차단.

### 분류기가 구성을 읽는 곳

분류기는 Claude가 로드하는 동일한 [[03 메모리와 컨텍스트]] CLAUDE.md 내용을 읽는다 — "never force push" 같은 지침이 Claude와 분류기를 동시에 조종. `autoMode`는 다음 스코프에서 읽는다:

| Scope | File | 용도 |
|:------|:-----|:-----|
| 한 개발자 | `~/.claude/settings.json` | 개인 신뢰 인프라 |
| 한 프로젝트·한 개발자 | `.claude/settings.local.json` | 프로젝트별 신뢰 bucket·서비스(gitignore) |
| 조직 전역 | [Managed settings](#서버-관리-설정-server-managed-settings) | 전 개발자 배포 |
| `--settings` 또는 SDK | 인라인 JSON | per-invocation 오버라이드 |

**shared project settings(`.claude/settings.json`)에선 안 읽음** — 체크인 repo가 자체 allow 규칙을 주입 못 하게. 각 스코프 항목은 결합되며, 개발자는 managed 항목을 제거 못 하고 추가만 가능. (allow가 soft block의 예외로 작동하므로 개발자가 추가한 allow가 조직 soft_deny를 오버라이드할 수 있다 — additive지 hard 경계 아님.)

> 분류기는 [[05 권한]] 시스템 **이후** 실행되는 두 번째 게이트다. 사용자 의도·분류기 구성과 무관하게 절대 실행 금지할 작업은 managed의 `permissions.deny`를 쓴다(분류기 이전 차단, 재정의 불가).

### 신뢰 인프라 정의

대부분 조직은 `autoMode.environment`만 설정하면 된다. 분류기가 무엇이 "외부"인지 판단하는 데 쓰며, 리스트에 없는 destination은 잠재적 exfiltration 타깃으로 본다. 기본 리스트는 working repo와 원격을 신뢰. 기본에 더하려면 배열에 리터럴 `"$defaults"`를 넣으면 그 위치에 기본 항목이 splice된다.

```json
{
  "autoMode": {
    "environment": [
      "$defaults",
      "Source control: github.example.com/acme-corp and all repos under it",
      "Trusted cloud buckets: s3://acme-build-artifacts, gs://acme-ml-datasets",
      "Trusted internal domains: *.corp.example.com, api.internal.example.com",
      "Key internal services: Jenkins at ci.example.com, Artifactory at artifacts.example.com"
    ]
  }
}
```

항목은 regex나 도구 패턴이 아닌 **산문**이다. 신입에게 인프라를 설명하듯 쓴다. 다룰 것: 조직(회사명·주 용도), source control(모든 push 대상 org), cloud providers·trusted buckets, trusted internal domains, key internal services(CI·아티팩트 레지스트리·인시던트 도구), additional context(규제·멀티테넌트·컴플라이언스). 한 번에 다 채울 필요 없이 기본+source control org+key services로 시작해 흔한 false positive를 해소하고 점진 추가.

### block·allow 규칙 오버라이드

세 필드로 분류기 내장 규칙 리스트를 교체: `autoMode.hard_deny`(무조건 보안 경계), `autoMode.soft_deny`(사용자 의도로 클리어 가능한 파괴적 작업), `autoMode.allow`(예외). 모두 산문 배열. 분류기 이전 hard block은 [[05 권한]]의 `permissions.deny`.

분류기 내부 4단계 우선순위:
- `hard_deny`: 무조건 차단(사용자 의도·allow 예외 미적용)
- `soft_deny`: 다음 차단(사용자 의도·allow가 오버라이드 가능)
- `allow`: 매치하는 `soft_deny`를 예외로 오버라이드
- 명시적 사용자 의도: 사용자 메시지가 Claude가 하려는 정확한 작업을 직접·구체적으로 기술하면 `soft_deny` 매치라도 허용

일반 요청은 명시적 의도가 아니다. "clean up the repo"는 force-push 권한이 아니지만 "force-push this branch"는 권한이다.

```json
{
  "autoMode": {
    "environment": ["$defaults", "Source control: github.example.com/acme-corp and all repos under it"],
    "allow": [
      "$defaults",
      "Deploying to the staging namespace is allowed: staging is isolated from production and resets nightly",
      "Writing to s3://acme-scratch/ is allowed: ephemeral bucket with a 7-day lifecycle policy"
    ],
    "soft_deny": [
      "$defaults",
      "Never run database migrations outside the migrations CLI, even against dev databases",
      "Never modify files under infra/terraform/prod/: production infrastructure changes go through the review workflow"
    ],
    "hard_deny": ["$defaults", "Never send repository contents to third-party code-review APIs"]
  }
}
```

> [!warning] `$defaults` 없이 두면 전체 기본 리스트 대체
> `environment`/`allow`/`soft_deny`/`hard_deny`를 `"$defaults"` 없이 설정하면 그 섹션의 전체 기본 리스트를 교체한다. `"$defaults"` 없는 `soft_deny`는 force push, `curl | bash`, production deploy 등 모든 내장 soft block을 버린다. `"$defaults"` 없는 `hard_deny`는 내장 데이터 exfiltration·auto-mode bypass 규칙을 버린다. 각 섹션은 독립 평가되므로 `environment`만 설정하면 나머지 기본은 유지된다. 전체를 소유하려면 `claude auto-mode defaults`로 내장 규칙을 출력→복사→검토 후 사용.

### 기본값·effective config 확인

| 명령 | 동작 |
|:-----|:-----|
| `claude auto-mode defaults` | 내장 `environment`/`allow`/`soft_deny`/`hard_deny` 규칙을 JSON 출력 |
| `claude auto-mode config` | 분류기가 실제 쓰는 effective 규칙(`$defaults` 확장 포함) JSON 출력 |
| `claude auto-mode critique` | 커스텀 규칙에 AI 피드백(모호·중복·false positive 유발 항목 플래그) |

저장 후 `claude auto-mode config`로 의도대로인지 확인. 내장 규칙을 제거·재작성하려면 `claude auto-mode defaults` 출력을 파일로 저장→편집→`"$defaults"` 대신 붙여넣기.

### 거부 리뷰

auto mode가 도구 호출을 거부하면 `/permissions`의 Recently denied 탭에 기록. 거부 항목에서 `r`을 누르면 재시도 표시 — 다이얼로그 종료 시 Claude Code가 모델에게 재시도 가능 메시지를 보내 대화 재개. 같은 destination 반복 거부는 보통 컨텍스트 누락이니 `autoMode.environment`에 추가 후 `claude auto-mode config`로 확인. 프로그램적 대응은 [[09 훅]]의 `PermissionDenied` 훅.

| `/agents` | 구성된 subagent와 설정 → [[08 서브에이전트와 에이전트 팀]] |
| `/hooks` | 활성 훅 구성 → [[09 훅]] |
| `/mcp` | 연결된 MCP 서버와 상태 → [[10 MCP]] |
| `/permissions` | 해석된 allow·deny 규칙 → [[05 권한]] |
| `/doctor` | 구성 진단: 잘못된 키, schema 에러, 설치 health |
| `/debug [issue]` | 세션 디버그 로깅 활성 + 로그·설정 경로로 진단 |
| `/status` | 활성 설정 소스(managed 적용 여부 포함) |

`/memory`에 메모리 파일이 없으면 위치를 [[03 메모리와 컨텍스트]] 로딩 규칙과 대조(서브디렉토리 CLAUDE.md는 Read로 그 디렉토리 파일을 읽을 때 on-demand 로드, 세션 시작 아님).

> CLAUDE.md와 permissions는 다른 문제를 푼다. CLAUDE.md는 Claude가 좋은 결정을 하도록 프로젝트 방식을 알려줌("우리는 이렇게 함"). [[05 권한]]·[[09 훅]]은 Claude 결정 무관하게 한계를 강제(보안 경계, 절대 일어나면 안 되는 것 — 보장이 필요할 때).

### 해석된 설정 확인

설정은 managed/user/project/local 스코프 병합. managed가 항상 우선. 나머지는 가까운 스코프가 넓은 것을 오버라이드(local→project→user). CLI 플래그·환경변수가 또 다른 오버라이드 레이어. 설정이 안 먹으면 보통 다른 스코프나 환경변수가 오버라이드 중이다. `/doctor`로 검증(이슈 보고 시 `f`로 Claude에 진단 전송), `/status`로 활성 소스 확인.

### MCP 서버 확인

`/mcp`로 모든 서버·연결 상태·프로젝트 승인 여부 확인. 정의는 맞아도 도구를 제공 안 하는 흔한 이유: project-scoped 서버의 일회성 승인 미완(프롬프트 dismiss 시 비활성 유지), 서버 시작 실패(`command`/`args`의 상대경로가 launch 디렉토리 기준 해석), connected인데 도구 0개(시작 성공했으나 도구 리스트 미반환 → Reconnect, 그래도 0이면 `claude --debug mcp`로 stderr 확인). → [[10 MCP]].

### 훅 확인

`/hooks`로 이벤트별 등록 훅 확인. 안 보이면 안 읽히는 것: 훅은 설정 파일의 `"hooks"` 키 아래여야 하고 standalone 파일 아님. 보이는데 발화 안 하면 보통 matcher 문제 — `matcher`는 `|`로 여러 도구를 매치하는 단일 문자열(예: `"Edit|Write"`). 오타는 무음 실패, 배열 값은 schema 에러로 드롭. settings.json 편집은 파일 안정 지연 후 실행 세션에 적용(재시작 불필요). 안 발화하면 `claude --debug hooks`로 라이브 평가 관찰 → [[09 훅]].

### 클린 구성과 비교

타깃 체크로 원인 격리가 안 되면, 평소 설정을 안 로드하는 세션과 비교한다. `CLAUDE_CONFIG_DIR`를 빈 디렉토리로 가리켜 `~/.claude`를 우회하고, `.claude`·`.mcp.json`·`CLAUDE.md` 없는 디렉토리에서 launch해 프로젝트 구성도 스킵.

```bash
cd /tmp && CLAUDE_CONFIG_DIR=/tmp/claude-clean claude
```

클린 세션엔 user/project 설정·훅·MCP·플러그인·메모리 없음. 단 managed settings는 시스템 경로(`~/.claude` 밖)라 여전히 적용. Linux/Windows는 재로그인 프롬프트(credential이 config 디렉토리에 저장), macOS는 Keychain이라 carry over. 문제가 사라지면 원인은 실제 `~/.claude`/프로젝트 `.claude`에 있으니 하나씩 재도입. 지속되면 user/project 밖이 원인 — `/status`로 managed 확인 후 환경변수 점검.

### 흔한 원인 (Check common causes)

| 증상 | 원인 | 해결 |
|:-----|:-----|:-----|
| 훅 안 발화 | `matcher`가 string 아닌 JSON 배열 | `\|`로 여러 도구 매치하는 단일 문자열(`"Edit\|Write"`) |
| 훅 안 발화 | `matcher`가 소문자(`"bash"`) | 대소문자 구분. 도구명 대문자: `Bash`, `Edit`, `Write`, `Read` |
| 훅 안 발화 | standalone 파일에 정의 | project/user config엔 standalone 훅 파일 없음. settings.json의 `"hooks"` 키에 정의 |
| 전역 permissions/hooks/env 무시 | `~/.claude.json`에 추가 | 그건 app state·UI 토글용. `permissions`/`hooks`/`env`는 `~/.claude/settings.json` |
| settings.json 값 무시됨 | 같은 키가 settings.local.json에 | `settings.local.json`이 `settings.json` 오버라이드, 둘 다 user settings 오버라이드 |
| 스킬이 `/skills`에 안 보임 | `.claude/skills/name.md`(폴더 아님) | 폴더+`SKILL.md`: `.claude/skills/name/SKILL.md` |
| 스킬은 보이나 Claude가 호출 안 함 | `disable-model-invocation: true` 또는 설명 불일치 | `/skills`의 "user-only" 배지 확인 → [[07 스킬]] |
| 서브디렉토리 CLAUDE.md 무시됨 | on-demand 로드(세션 시작 아님) | Read로 그 디렉토리 파일을 읽을 때 로드 |
| subagent가 CLAUDE.md 무시 | 내장 Explore/Plan은 CLAUDE.md 스킵 | Explore/Plan은 위임 프롬프트에 재기술, 커스텀은 agent body에 → [[08 서브에이전트와 에이전트 팀]] |
| 세션 끝에 cleanup 안 됨 | `SessionEnd` 훅 미구성 | settings.json에 `SessionEnd` 훅 추가 → [[09 훅]] |
| `.mcp.json` MCP 안 로드 | `.claude/` 안이거나 Claude Desktop 포맷 | repo 루트에 `.mcp.json` → [[10 MCP]] |
| settings.json `mcpServers` 안 보임 | settings.json은 `mcpServers` 키 안 읽음 | `.mcp.json` 또는 `claude mcp add --scope user` |
| MCP 서버가 환경변수 없이 시작 | settings.json `env`는 MCP 자식에 전파 안 됨 | `.mcp.json`의 per-server `env` 사용 |
| `Bash(rm *)` deny가 `/bin/rm`·`find -delete` 차단 안 함 | prefix 규칙은 리터럴 명령 문자열 매치 | 변형별 패턴 추가, 또는 PreToolUse 훅·[[18 보안과 샌드박스]] |

## 서버 관리 설정 (server-managed-settings)

서버 관리 설정은 어드민이 Claude.ai 웹 인터페이스로 Claude Code를 중앙 구성하게 한다. 사용자가 조직 credential로 인증하면 클라이언트가 자동 수신한다. **MDM 인프라가 없거나 비관리 디바이스 사용자를 관리해야 하는 조직용.**

> Claude for Teams / Enterprise 고객 전용.

### 요구사항

- Claude for Teams 또는 Enterprise 플랜
- Claude Code v2.1.38+ (Teams) 또는 v2.1.30+ (Enterprise)
- `api.anthropic.com` 네트워크 접근

### 서버 관리 vs 엔드포인트 관리

| 방식 | 적합 | 보안 모델 |
|:-----|:-----|:---------|
| **Server-managed** | MDM 없는 조직, 비관리 디바이스 사용자 | 인증 시 Anthropic 서버에서 전달 |
| **Endpoint-managed** | MDM/엔드포인트 관리 있는 조직 | MDM 프로파일·레지스트리·managed 파일로 디바이스 배포 |

디바이스가 MDM 등록돼 있으면 endpoint-managed가 더 강한 보장(OS 레벨에서 사용자 수정 방지).

### 구성 절차

1. **어드민 콘솔 열기**: Claude.ai → Admin Settings > Claude Code > Managed settings
2. **설정 정의**: JSON으로 추가. OS-level 정책 전용을 제외한 모든 `settings.json` 설정 지원([[09 훅]], 환경변수, `allowManagedPermissionRulesOnly` 같은 managed-only 포함)
3. **저장·배포**: 클라이언트가 다음 시작 또는 시간당 폴링 주기에 수신

권한 deny + bypass 차단 + managed 규칙만 강제 예시:
```json
{
  "permissions": {
    "deny": ["Bash(curl *)", "Read(./.env)", "Read(./.env.*)", "Read(./secrets/**)"],
    "disableBypassPermissionsMode": "disable"
  },
  "allowManagedPermissionRulesOnly": true
}
```
조직 전역 파일 편집 후 감사 스크립트 실행 훅:
```json
{ "hooks": { "PostToolUse": [{ "matcher": "Edit|Write", "hooks": [{ "type": "command", "command": "/usr/local/bin/audit-edit.sh" }] }] } }
```
auto mode 신뢰 인프라:
```json
{ "autoMode": { "environment": ["Source control: github.example.com/acme-corp and all repos under it", "Trusted internal domains: *.corp.example.com"] } }
```

훅은 셸 명령을 실행하므로 적용 전 [보안 승인 다이얼로그](#보안-승인-다이얼로그)가 뜬다.

### 검증·접근 제어·제한

- **검증**: 사용자에게 Claude Code 재시작 요청. 보안 다이얼로그 트리거 설정이면 시작 시 프롬프트. `/permissions`로 managed 권한 규칙 활성 확인.
- **접근 제어**: Primary Owner, Owner만 관리. 변경은 전 사용자에 적용되니 신뢰 인물로 제한.
- **managed-only 설정**: 대부분 키는 모든 스코프에서 작동하나 일부는 managed에서만 읽힌다 → [[05 권한]] managed-only 리스트.
- **현재 제한**:
  - 전 사용자에 동일 적용(per-group 미지원)
  - `managed-mcp.json`은 server-managed로 배포 불가 → `allowedMcpServers`/`deniedMcpServers` 정책 키로 전달 → [[10 MCP]]
  - OS-level 전용 설정(`policyHelper`, `wslInheritsWindowsSettings`)은 미적용 → MDM 또는 시스템 `managed-settings.json`

### 설정 전달·우선순위

server-managed와 endpoint-managed 모두 [설정 계층](#설정-우선순위-settings-precedence) 최상위. CLI 인자 포함 무엇도 오버라이드 불가. managed 티어 내: **server-managed 먼저 확인, 그다음 endpoint-managed**. 병합 안 함 — server-managed가 키를 하나라도 전달하면 endpoint-managed는 전부 무시. server-managed가 아무것도 안 주면 endpoint-managed 적용.

> 어드민 콘솔에서 server-managed를 지워 plist/registry로 폴백하려 해도, [캐시된 설정](#fetch--캐싱-동작)이 다음 성공 fetch까지 클라이언트에 잔존. `/status`로 활성 소스 확인.

### Fetch·캐싱 동작

시작 시 fetch + 활성 세션 중 시간당 폴링.
- **첫 launch(캐시 없음)**: 비동기 fetch, 실패 시 managed 없이 계속, 설정 로드 전 짧은 미강제 윈도우
- **이후 launch(캐시 있음)**: 캐시 즉시 적용, 백그라운드로 fresh fetch, 네트워크 실패에도 캐시 유지

OTel 같은 advanced 설정은 전체 재시작 필요, 나머지는 재시작 없이 자동 적용.

### Fail-closed 시작 강제

기본은 fetch 실패 시 managed 없이 계속. 미강제 윈도우가 허용 불가한 환경은 managed에 `forceRemoteSettingsRefresh: true`. 활성 시 시작을 블록하고 fresh fetch 완료까지 대기, 실패하면 종료(정책 없이 진행 안 함). self-perpetuating(서버 전달 후 로컬 캐시되어 다음 시작도 강제).

```json
{ "forceRemoteSettingsRefresh": true }
```
활성화 전 `api.anthropic.com` 연결 보장 필수(도달 불가면 사용자가 시작 못 함). v2.1.139~ `claude auth` 서브명령은 이 체크에서 면제(만료 credential 재인증 가능).

### 보안 승인 다이얼로그

보안 위험 가능 설정은 적용 전 명시적 승인 필요:
- 셸 명령 실행 설정
- known safe allowlist에 없는 커스텀 환경변수
- 훅 구성

사용자가 거부하면 Claude Code 종료. (단 `-p` 비인터랙티브 모드는 다이얼로그 스킵, 무승인 적용.)

### 플랫폼 가용성·보안 고려

server-managed는 `api.anthropic.com` 직접 연결 필요 — Bedrock/Vertex/Foundry/커스텀 엔드포인트(`ANTHROPIC_BASE_URL`·LLM 게이트웨이)에선 불가. **클라이언트 사이드 통제**라 비관리 디바이스에서 admin/sudo 사용자는 바이너리·파일시스템·네트워크 수정 가능.

| 시나리오 | 동작 |
|:---------|:-----|
| 캐시 설정 편집 | 변조 파일이 시작 시 적용, 다음 서버 fetch에 올바른 설정 복원 |
| 캐시 설정 삭제 | 첫 launch 동작(비동기 fetch + 짧은 미강제 윈도우) |
| API 불가 | 캐시 있으면 적용, 없으면 미강제. `forceRemoteSettingsRefresh`면 종료(`claude auth` 제외) |
| 다른 조직 인증 | managed 외 계정엔 미전달 |
| 서드파티 provider 구성 | server-managed 우회(`CLAUDE_CODE_USE_BEDROCK`/`_MANTLE`/`_VERTEX`/`_FOUNDRY` 또는 non-default `ANTHROPIC_BASE_URL`) |

런타임 변경 탐지는 [[09 훅]]의 `ConfigChange` 훅. 더 강한 보장은 MDM 등록 디바이스의 endpoint-managed. 감사 로그 이벤트는 compliance API/audit log export로 제공(Anthropic 계정팀 문의).

### 설정 전달 방식 결정

Claude Code는 4곳에서 managed settings를 찾고 디바이스에서 처음 발견한 것을 사용.

| 메커니즘 | 전달 | 우선순위 | 플랫폼 |
|:---------|:-----|:--------|:-------|
| Server-managed | Claude.ai 어드민 콘솔 | Highest | All |
| plist / registry policy | macOS: `com.anthropic.claudecode` plist / Windows: `HKLM\SOFTWARE\Policies\ClaudeCode` | High | macOS, Windows |
| File-based managed | macOS: `/Library/Application Support/ClaudeCode/managed-settings.json` / Linux·WSL: `/etc/claude-code/managed-settings.json` / Windows: `C:\Program Files\ClaudeCode\managed-settings.json` | Medium | All |
| Windows user registry | `HKCU\SOFTWARE\Policies\ClaudeCode` | Lowest | Windows only |

server-managed는 인증 시 도달+시간당 갱신, 엔드포인트 인프라 불필요(Teams/Enterprise 필요). provider 혼합 시 Claude.ai 사용자엔 server-managed + 그 외엔 file/plist/registry 폴백 병행. plist·HKLM은 admin 권한 필요라 변조 저항, HKCU는 비특권 쓰기 가능하니 편의 기본값으로 취급. WSL은 기본 `/etc/claude-code`만 읽으나 `wslInheritsWindowsSettings: true`로 Windows 정책 확장. 배열 설정(`permissions.allow`/`deny`)은 전 소스 병합 — 개발자는 확장만 가능, 제거 불가.

### 강제 대상 결정

| Control | 동작 | Key settings |
|:--------|:-----|:-------------|
| 권한 규칙 → [[05 권한]] | 도구·명령 allow/ask/deny | `permissions.allow`, `permissions.deny` |
| 권한 lockdown | managed 규칙만, `--dangerously-skip-permissions` 비활성 | `allowManagedPermissionRulesOnly`, `permissions.disableBypassPermissionsMode` |
| Sandboxing → [[18 보안과 샌드박스]] | OS-level 파일시스템·네트워크 격리+도메인 allowlist | `sandbox.enabled`, `sandbox.network.allowedDomains` |
| managed CLAUDE.md → [[03 메모리와 컨텍스트]] | 전 세션 org 지침(제외 불가) | managed policy 경로 파일 |
| MCP 서버 통제 → [[10 MCP]] | 추가·연결 가능 서버 제한 또는 고정 셋 배포 | `allowedMcpServers`, `deniedMcpServers`, `allowManagedMcpServersOnly`, `managed-mcp.json` |
| 플러그인 마켓플레이스 → [[11 플러그인]] | 추가·설치 가능 소스 제한 | `strictKnownMarketplaces`, `blockedMarketplaces` |
| 커스터마이즈 lockdown | skills/agents/hooks/mcp를 plugin·managed에서만 | `strictPluginOnlyCustomization` |
| 훅 제한 → [[09 훅]] | managed 훅만, HTTP 훅 URL 제한 | `allowManagedHooksOnly`, `allowedHttpHookUrls` |
| agent view 비활성 | `claude agents`/`--bg`/`/background` 끔 | `disableAgentView` |
| 버전 floor / 범위 | auto-update floor / 범위 밖 시작 거부 | `minimumVersion` / `requiredMinimumVersion`, `requiredMaximumVersion` |

> 권한 규칙과 sandboxing은 다른 레이어다. WebFetch deny는 Claude의 fetch 도구를 막지만 Bash가 허용되면 `curl`·`wget`은 어떤 URL이든 닿는다. sandboxing이 OS-level 네트워크 도메인 allowlist로 그 갭을 닫는다.

구성 후 개발자가 `/status` 실행 → `Enterprise managed settings (remote|plist|HKLM|HKCU|file)` 라인 확인. 로그인 이슈는 `/logout`→`/login`, `claude update`(엔터프라이즈 auth 옵션 누락 시), 터미널 재시작. ## 빠른 시작 예시

전형적인 팀 `.claude/settings.json`:
```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "model": "opus",
  "permissions": {
    "allow": ["Bash(npm run lint)", "Bash(npm run test *)"],
    "deny": ["Read(./.env)", "Read(./.env.*)", "Read(./secrets/**)", "Bash(curl *)"]

  "hooks": { "PostToolUse": [{ "matcher": "Edit|Write", "hooks": [{ "type": "command", "command": "npm run format" }] }] }
}
```

```bash
alias claude-work='CLAUDE_CONFIG_DIR=~/.claude-work claude'
```
