## 개요: Claude Code란 무엇인가 (overview)

Claude Code는 코드베이스를 읽고, 파일을 편집하고, 명령을 실행하고, 개발 도구와 통합되는 **에이전틱 코딩 도구**다. 터미널, IDE, 데스크톱 앱, 브라우저에서 모두 사용할 수 있다. 단일 파일만 보는 인라인 어시스턴트와 달리 프로젝트 전체를 보고 여러 파일·도구를 가로질러 작업한다.

대부분의 진입점은 [Claude 구독](https://claude.com/pricing)(Pro/Max/Team/Enterprise) 또는 Anthropic Console 계정을 요구한다. Terminal CLI와 VS Code는 서드파티 프로바이더([[17 클라우드 프로바이더]])도 지원한다.

### 진입 인터페이스 (surfaces)

모든 인터페이스는 동일한 Claude Code 엔진에 연결되므로, CLAUDE.md 파일·설정·MCP 서버가 전 인터페이스에서 그대로 동작한다.

| 인터페이스 | 특징 | 전용 노트 |
| --- | --- | --- |
| Terminal | 풀 기능 CLI. 파일 편집·명령 실행·프로젝트 관리 | (이 노트 / [[02 CLI 레퍼런스]]) |
| VS Code / Cursor | inline diff, @-mention, plan review, 대화 이력 | [[14 IDE와 데스크톱]] |
| Desktop app | diff 시각 검토, 다중 세션 병렬, 작업 스케줄링, 클라우드 세션 | [[14 IDE와 데스크톱]] |
| Web (`claude.ai/code`) | 로컬 설치 없이 브라우저에서. 장시간 작업·병렬 작업 | [[15 웹과 모바일]] |
| JetBrains (IntelliJ/PyCharm/WebStorm 등) | 대화형 diff, 선택 컨텍스트 공유 | [[14 IDE와 데스크톱]] |

### Claude Code로 할 수 있는 일 (예시)

원문 Accordion의 실전 명령 예시. 자연어를 그대로 프롬프트로 넘긴다.

```bash
# 미뤄둔 잡일 자동화 (테스트 작성/lint 수정/머지 충돌/의존성 업데이트)
claude "write tests for the auth module, run them, and fix any failures"

# 커밋/PR 생성 (git 직접 조작)
claude "commit my changes with a descriptive message"

# 유닉스 철학: 파이프/스크립트/자동화 (헤드리스 -p)
tail -200 app.log | claude -p "Slack me if you see any anomalies"
claude -p "translate new strings into French and raise a PR for review"
git diff main --name-only | claude -p "review these changed files for security issues"
```

## Quickstart: 5분 첫 사용 (quickstart)

이 절은 터미널 CLI 기준 첫 세션을 8단계로 안내한다. 사전 준비: 터미널, 코드 프로젝트, Claude 구독/Console/지원 클라우드 프로바이더 접근권.

### Step 1~3: 설치 → 로그인 → 세션 시작

```bash
# Step 1. 설치 (자세한 플랫폼별 옵션은 아래 "설치" 절)
curl -fsSL https://claude.ai/install.sh | bash    # macOS/Linux/WSL

# Step 2. 로그인 (최초 실행 시 브라우저 인증 프롬프트)
claude
# 계정 전환/재인증은 세션 안에서:
/login

# Step 3. 프로젝트 디렉터리에서 세션 시작
cd /path/to/your/project
claude
```

세션이 뜨면 상단에 버전·현재 모델·작업 디렉터리가 표시된다. `/help`로 명령 목록, `/resume`으로 이전 대화 이어가기.

### Step 4~8: 첫 질문부터 일반 워크플로우까지

Claude Code는 필요한 파일을 알아서 읽으므로 컨텍스트를 수동으로 추가하지 않아도 된다. 자연어를 그대로 입력한다.

```text
# Step 4. 코드베이스 이해
what does this project do?
where is the main entry point?
how do I create custom skills in Claude Code?

# Step 5. 첫 코드 변경 (제안 → 승인 → 편집)
add a hello world function to the main file

# Step 6. Git 조작
what files have I changed?
commit my changes with a descriptive message
create a new branch called feature/quickstart
help me resolve merge conflicts

# Step 7. 버그 수정 / 기능 추가
add input validation to the user registration form
there's a bug where users can submit empty forms - fix it

# Step 8. 기타 워크플로우
refactor the authentication module to use async/await instead of callbacks
write unit tests for the calculator functions
update the README with installation instructions
review my changes and suggest improvements
```

> [!note] 권한 안전장치
> Claude Code는 파일 수정 전 항상 허가를 요청한다. 개별 변경을 승인하거나, 세션 동안 "Accept all" 모드를 켤 수 있다. 권한 모델 전체는 [[05 권한]] 참조.

### 필수 명령어 (daily use)

| Command | 기능 | 예시 |
| --- | --- | --- |
| `claude` | 대화형 모드 시작 | `claude` |
| `claude "task"` | 일회성 작업 실행 | `claude "fix the build error"` |
| `claude -p "query"` | 일회성 쿼리 후 종료 (헤드리스) | `claude -p "explain this function"` |
| `claude -c` | 현재 디렉터리의 최근 대화 이어가기 | `claude -c` |
| `claude -r` | 이전 대화 재개(선택) | `claude -r` |
| `/clear` | 대화 이력 지우기 | `/clear` |
| `/help` | 명령 목록 | `/help` |
| `exit` 또는 `Ctrl+D` | 종료 | `exit` |

전체 명령·플래그는 [[02 CLI 레퍼런스]] 참조.

### 초보자 프로 팁

- **구체적으로 요청**: "fix the bug"보다 "fix the login bug where users see a blank screen after entering wrong credentials".
- **단계로 쪼개기**: 복잡한 작업은 1·2·3 단계로 나눠 지시.
- **먼저 탐색시키기**: 변경 전에 "analyze the database schema"처럼 이해를 먼저.
- **단축키 활용**: `/`로 명령/스킬 보기, Tab 자동완성, `↑` 명령 이력, `Shift+Tab`으로 권한 모드 순환.

> [!tip] 핵심 멘탈모델
> "동료에게 위임하듯" 대화하라. 목표를 설명하면 Claude가 거기까지 가는 길을 찾는다. (상세 워크플로우는 [[20 베스트 프랙티스와 워크플로우]])

## Claude Code의 동작 원리 (how-claude-code-works)

### 에이전틱 루프 (the agentic loop)

작업을 주면 Claude는 세 단계를 거친다 — **컨텍스트 수집(gather context) → 행동(take action) → 결과 검증(verify results)**. 세 단계는 섞여 있으며, Claude는 매 단계에서 도구를 쓰고 이전 단계에서 배운 것에 따라 다음 행동을 결정한다. 질문 하나는 컨텍스트 수집만으로 끝날 수도 있고, 버그 수정은 세 단계를 반복적으로 순환한다.

사용자도 루프의 일부다. 언제든 끼어들어 방향을 바꾸거나 컨텍스트를 더 주거나 다른 접근을 요청할 수 있다.

루프는 두 요소로 구동된다 — **모델(추론)**과 **도구(행동)**. Claude Code는 Claude 모델을 감싸는 **에이전트 하니스(agentic harness)**로서 도구·컨텍스트 관리·실행 환경을 제공해 언어 모델을 유능한 코딩 에이전트로 만든다.

#### 모델 (models)

여러 모델이 트레이드오프를 달리해 제공된다. Sonnet은 대부분의 코딩 작업을 잘 처리하고, Opus는 복잡한 아키텍처 결정에 강한 추론을 제공한다.

```text
/model            # 세션 중 모델 전환
claude --model <name>   # 특정 모델로 시작
```

모델 설정 상세는 [[19 비용과 성능]] 참조. "Claude가 선택한다/결정한다"는 표현은 모델의 추론을 가리킨다.

#### 도구 (tools)

도구가 Claude Code를 에이전틱하게 만든다. 도구 없이는 텍스트 응답만 가능하지만, 도구가 있으면 코드를 읽고 파일을 편집하고 명령을 실행하고 웹을 검색하고 외부 서비스와 상호작용할 수 있다. 내장 도구는 다섯 범주로 나뉜다.

| 범주 | Claude가 할 수 있는 것 |
| --- | --- |
| **File operations** | 파일 읽기, 코드 편집, 새 파일 생성, 이름 변경·재구성 |
| **Search** | 패턴으로 파일 찾기, regex로 내용 검색, 코드베이스 탐색 |
| **Execution** | 셸 명령 실행, 서버 시작, 테스트 실행, git 사용 |
| **Web** | 웹 검색, 문서 가져오기, 에러 메시지 조회 |
| **Code intelligence** | 편집 후 타입 에러/경고 확인, 정의로 점프, 참조 찾기 (언어별 [code intelligence 플러그인] 필요) |

이외에도 서브에이전트 스폰, 사용자 질문 등 오케스트레이션 도구가 있다. 전체 도구 목록과 동작은 [[06 내장 도구 레퍼런스]] 참조.

예: "fix the failing tests"라고 하면 Claude는 ① 테스트 실행 → ② 에러 출력 읽기 → ③ 관련 소스 검색 → ④ 파일 읽기 → ⑤ 수정 → ⑥ 재실행 검증 순으로 도구를 연쇄한다.

**기본 역량 확장**: 내장 도구가 토대다. 그 위에 [[07 스킬]](지식 추가), [[10 MCP]](외부 서비스 연결), [[09 훅]](워크플로우 자동화), [[08 서브에이전트와 에이전트 팀]](작업 위임)을 얹는다.

### Claude가 접근하는 것 (what Claude can access)

`claude`를 디렉터리에서 실행하면 다음에 접근한다.

- **프로젝트**: 현재 디렉터리·하위 파일 + 허가 시 다른 위치 파일.
- **터미널**: 빌드 도구·git·패키지 매니저·시스템 유틸리티·스크립트 등 명령줄로 할 수 있는 모든 것.
- **git 상태**: 현재 브랜치, 미커밋 변경, 최근 커밋 이력.
- **CLAUDE.md**: 프로젝트별 지시·관례·컨텍스트. ([[03 메모리와 컨텍스트]])
- **Auto memory**: 작업하며 자동 저장되는 학습(패턴·선호). 세션 시작 시 MEMORY.md의 **첫 200줄 또는 25KB 중 먼저 도달하는 한도**까지 로드. ([[03 메모리와 컨텍스트]])
- **설정한 확장**: MCP 서버, 스킬, 서브에이전트, Claude in Chrome.

### 실행 환경과 인터페이스

| 환경(Environment) | 코드 실행 위치 | 용도 |
| --- | --- | --- |
| **Local** | 내 머신 | 기본. 파일·도구·환경 풀 액세스 |
| **Cloud** | Anthropic 관리 VM | 작업 오프로드, 로컬에 없는 repo 작업 |
| **Remote Control** | 내 머신 (브라우저에서 제어) | 모든 것을 로컬에 두면서 웹 UI 사용 |

에이전틱 루프·도구·역량은 어디서나 동일하다. 바뀌는 것은 코드가 실행되는 위치와 상호작용 방식뿐이다. ([[15 웹과 모바일]], [[14 IDE와 데스크톱]])

### 세션 다루기 (work with sessions)

대화는 로컬에 JSONL 평문으로 `~/.claude/projects/` 아래에 저장된다. 모든 메시지·도구 사용·결과가 기록되어 rewind·resume·fork가 가능하다. 코드 변경 전에는 해당 파일을 스냅샷한다.

- **세션은 독립적**이다. 새 세션은 이전 대화 이력 없이 빈 컨텍스트 윈도로 시작한다. 세션 간 학습은 Auto memory로, 사용자 지시는 CLAUDE.md로 유지한다.

#### 브랜치를 가로질러 작업

각 대화는 현재 디렉터리에 묶인 세션이다. `/resume` 피커는 기본적으로 현재 worktree의 세션을 보여주고, 키 단축키로 다른 worktree/프로젝트로 범위를 넓힐 수 있다. 브랜치를 바꾸면 Claude는 새 브랜치 파일을 보지만 대화 이력은 그대로 유지된다. 디렉터리별로 세션이 묶이므로 **git worktrees**로 병렬 세션을 돌릴 수 있다.

#### 세션 재개/포크

```text
claude --continue   # 같은 세션 ID로 재개, 기존 대화에 새 메시지 추가
claude --resume     # 동일
--fork-session  /  /branch   # 이력을 새 세션 ID로 복사, 원본은 그대로
```

재개 플래그·`/resume` 피커·동일 세션을 두 터미널에서 연 경우의 동작 등 상세는 세션 관리 문서 참조. (CLI 플래그는 [[02 CLI 레퍼런스]])

### 컨텍스트 윈도 (the context window)

컨텍스트 윈도에는 대화 이력·파일 내용·명령 출력·CLAUDE.md·Auto memory·로드된 스킬·시스템 지시가 담긴다. 작업하며 컨텍스트가 차오르면 Claude가 **자동 컴팩션(auto-compact)**한다 — 단, 초기 대화의 지시가 유실될 수 있으니 지속 규칙은 CLAUDE.md에 둔다.

```text
/context          # 무엇이 공간을 차지하는지 확인
/compact          # 수동 컴팩션
/compact focus on the API changes   # 특정 포커스로 컴팩션
/mcp              # 서버별 토큰 비용 확인
```

#### 컨텍스트가 찰 때

한도에 다가가면 Claude Code가 오래된 도구 출력을 먼저 비우고, 필요하면 대화를 요약한다. 요청·핵심 코드 스니펫은 보존되지만 초기 상세 지시는 유실될 수 있다 → **지속 규칙은 CLAUDE.md로**. 컴팩션 보존 제어는 CLAUDE.md의 "Compact Instructions" 섹션이나 `/compact <focus>`로 한다.

단일 파일/도구 출력이 너무 커서 요약 직후 컨텍스트가 즉시 다시 차면, 몇 번 시도 후 auto-compact를 멈추고 thrashing 에러를 표시한다. 복구는 [[25 트러블슈팅]] 참조.

MCP 도구 정의는 기본 deferred 되어 [MCP tool search]로 필요 시 로드된다 — 특정 도구를 쓰기 전까지는 도구 이름만 컨텍스트를 차지한다.

#### 스킬·서브에이전트로 컨텍스트 관리

- **스킬**은 on-demand 로드. 세션 시작 시 설명만 보이고, 사용될 때 전체가 로드된다. 수동 호출 스킬은 `disable-model-invocation: true`로 호출 전까지 설명을 컨텍스트에서 빼둘 수 있다. 내가 작성하지 않은 스킬은 설정의 `skillOverrides`로 동일 처리. ([[07 스킬]])
- **서브에이전트**는 자체 신선한 컨텍스트를 가져 메인 대화를 오염시키지 않고, 끝나면 요약만 반환한다. 긴 세션에 유리하다. ([[08 서브에이전트와 에이전트 팀]])

### 체크포인트와 권한으로 안전 유지

두 안전장치 — **체크포인트**(파일 변경 되돌리기)와 **권한**(묻지 않고 할 수 있는 범위 제어).

#### 체크포인트로 변경 되돌리기

모든 파일 편집은 되돌릴 수 있다. 편집 전 현재 내용을 스냅샷하므로 **`Esc` 두 번**으로 이전 상태로 rewind하거나 Claude에게 되돌려 달라고 한다. 체크포인트는 세션 로컬이며 git과 별개고 파일 변경만 커버한다. 원격 시스템(DB·API·배포)에 영향을 주는 행동은 체크포인트 불가 → 그래서 외부 부수효과 명령 전 Claude가 묻는다.

#### Claude가 할 수 있는 것 제어 (권한 모드)

`Shift+Tab`으로 권한 모드 순환:

| 모드 | 동작 |
| --- | --- |
| **Default** | 파일 편집·셸 명령 전 항상 질문 |
| **Auto-accept edits** | 파일 편집과 `mkdir`/`mv` 등 일반 파일시스템 명령은 묻지 않음, 그 외 명령은 질문 |
| **Plan mode** | 읽기 전용 도구만 사용, 실행 전 승인할 plan 작성 |
| **Auto mode** | 모든 행동을 백그라운드 안전 점검으로 평가 (현재 research preview) |

`.claude/settings.json`에서 특정 명령(`npm test`, `git status` 등)을 허용해 매번 묻지 않게 할 수 있다. 설정은 조직 정책부터 개인 선호까지 스코프된다. 상세는 [[05 권한]], [[04 설정]].

### Claude Code로 효과적으로 작업하기

- **Claude Code에게 사용법을 물어라**: "how do I set up hooks?"처럼 물으면 설명한다. 내장 명령도 안내한다 — `/init`(CLAUDE.md 생성), `/agents`(커스텀 서브에이전트 구성), `/doctor`(설치 진단).
- **대화다**: 완벽한 프롬프트가 필요 없다. 시작 후 반복해서 다듬는다.
- **끼어들고 조종하기**: `Esc`로 즉시 정지(실행 중 도구 취소), 또는 교정 입력 후 `Enter`로 실행을 멈추지 않고 보내면 현재 행동 완료 즉시 반영.
- **앞단에서 구체적으로**: 특정 파일·제약·예시 패턴을 참조하면 교정이 줄고 첫 시도 성공률이 높다.
- **검증 대상을 줘라**: 테스트 케이스·기대 UI 스크린샷·원하는 출력을 주면 Claude가 스스로 점검한다.
- **구현 전 탐색**: 복잡한 문제는 plan mode(`Shift+Tab` 두 번)로 코드베이스를 먼저 분석시키고, plan을 검토·다듬은 뒤 구현시킨다 (2단계 접근).
- **지시가 아니라 위임**: 어떤 파일을 읽고 어떤 명령을 쓸지 지정하지 말고, 컨텍스트와 방향만 주고 맡긴다.

상세 베스트 프랙티스는 [[20 베스트 프랙티스와 워크플로우]].

## 설치·업데이트·제거 (setup)

### 시스템 요구사항

- **OS**: macOS 13.0+, Windows 10 1809+ / Server 2019+, Ubuntu 20.04+, Debian 10+, Alpine Linux 3.19+
- **하드웨어**: RAM 4GB 이상, x64 또는 ARM64
- **네트워크**: 인터넷 연결 필수 ([[18 보안과 샌드박스]]의 네트워크 설정 참조)
- **셸**: Bash, Zsh, PowerShell, CMD
- **위치**: Anthropic 지원 국가
- **추가 의존성**: ripgrep — 보통 Claude Code에 포함. 검색 실패 시 [[25 트러블슈팅]].

### 설치 방법

```bash
# Native Install (권장) — macOS/Linux/WSL: 백그라운드 자동 업데이트
curl -fsSL https://claude.ai/install.sh | bash

# Windows PowerShell
irm https://claude.ai/install.ps1 | iex

# Windows CMD
curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd

# Homebrew (자동 업데이트 안 됨)
brew install --cask claude-code           # stable 채널 (약 1주 지연, 대형 회귀 스킵)
brew install --cask claude-code@latest    # latest 채널

# WinGet (자동 업데이트 안 됨)
winget install Anthropic.ClaudeCode
```

설치 후 프로젝트 디렉터리에서 `claude` 실행. PowerShell/CMD 구분 팁: 프롬프트에 `PS C:\`가 보이면 PowerShell, `PS` 없이 `C:\`면 CMD.

> [!tip] GUI 선호 / 터미널 입문
> 데스크톱 앱([[14 IDE와 데스크톱]])으로 터미널 없이도 사용 가능. 터미널 입문자는 원문의 terminal guide 참조.

### Windows 설정 옵션

| 옵션 | 요구사항 | [[18 보안과 샌드박스]] 샌드박싱 | 권장 상황 |
| --- | --- | --- | --- |
| Native Windows | 없음; Git for Windows는 선택 | 미지원 | Windows 네이티브 프로젝트·도구 |
| WSL 2 | WSL 2 활성화 | 지원 | Linux 툴체인 또는 샌드박스 명령 실행 |
| WSL 1 | WSL 1 활성화 | 미지원 | WSL 2를 못 쓸 때 |

- **Git for Windows 없이**: 셸 명령을 PowerShell 도구로 실행.
- **Git for Windows 있으면**: Git Bash를 Bash 도구로 사용. 못 찾으면 settings.json에 경로 지정:

```json
{ "env": { "CLAUDE_CODE_GIT_BASH_PATH": "C:\\Program Files\\Git\\bin\\bash.exe" } }
```

Git for Windows 설치 시 PowerShell 도구는 점진 롤아웃 중. `CLAUDE_CODE_USE_POWERSHELL_TOOL=1`로 opt-in, `0`으로 opt-out. (도구 동작은 [[06 내장 도구 레퍼런스]])

### 업데이트

네이티브 설치는 시작 시·실행 중 주기적으로 백그라운드 자동 업데이트(다음 시작 시 적용). Homebrew/WinGet/Linux 패키지 매니저는 기본 수동 업데이트.

```bash
claude update   # 즉시 업데이트 (백그라운드 체크 대기 없이)
```

#### 릴리스 채널 / 버전 핀

```json
{ "autoUpdatesChannel": "stable" }   // "latest"(기본) 또는 "stable"(약 1주 지연)
```

- `/config` → **Auto-update channel**로도 설정. 엔터프라이즈는 [managed settings]로 조직 전체 강제 ([[05 권한]]).
- Homebrew는 이 설정 대신 cask 이름으로 채널 선택(`claude-code`=stable, `claude-code@latest`=latest).

```json
// minimumVersion: 이 값 미만 설치 거부. latest→stable 전환 시 다운그레이드 방지
{ "autoUpdatesChannel": "stable", "minimumVersion": "2.1.100" }
```

- `/config`로 latest→stable 전환 시 현재 버전 유지(=`minimumVersion`을 그 버전으로 설정) 또는 다운그레이드 허용을 선택. latest로 되돌리면 해제.
- `minimumVersion`은 업데이트만 제약. 버전 범위 밖에서 **실행 자체를 거부**하려면 managed settings의 `requiredMinimumVersion`/`requiredMaximumVersion` 사용. ([[04 설정]])

#### 자동 업데이트 비활성화 / 패키지 매니저 자동 업그레이드

```json
{ "env": { "DISABLE_AUTOUPDATER": "1" } }   // 백그라운드 체크만 중지 (claude update/install은 동작)
```

모든 업데이트 경로 차단은 `DISABLE_UPDATES` 사용. Homebrew/WinGet 자동 업그레이드 위임은 `CLAUDE_CODE_PACKAGE_MANAGER_AUTO_UPDATE=1` ([[04 설정]]의 환경변수). Homebrew는 업그레이드 후 구버전이 디스크에 남으니 `brew cleanup` 주기 실행.

### 고급 설치 옵션

```bash
# 특정 버전/채널 (설치 시 채널이 자동 업데이트 기본값이 됨)
curl -fsSL https://claude.ai/install.sh | bash -s stable
curl -fsSL https://claude.ai/install.sh | bash -s 2.1.89
```

#### Linux 패키지 매니저 (apt / dnf / apk)

서명된 저장소 제공. `stable`을 `latest`로 바꾸면 롤링 채널. 패키지 매니저 설치는 Claude Code를 통한 자동 업데이트가 아니라 시스템 업그레이드 흐름으로 갱신된다. 모든 저장소는 Claude Code 릴리스 서명 키로 서명됨.

```bash
# apt (Debian/Ubuntu)
sudo install -d -m 0755 /etc/apt/keyrings
sudo curl -fsSL https://downloads.claude.ai/keys/claude-code.asc -o /etc/apt/keyrings/claude-code.asc
echo "deb [signed-by=/etc/apt/keyrings/claude-code.asc] https://downloads.claude.ai/claude-code/apt/stable stable main" | sudo tee /etc/apt/sources.list.d/claude-code.list
sudo apt update && sudo apt install claude-code
# 키 지문 검증: gpg --show-keys → 31DD DE24 DDFA B679 F42D 7BD2 BAA9 29FF 1A7E CACE
# 업그레이드: sudo apt update && sudo apt upgrade claude-code
```

dnf는 `/etc/yum.repos.d/claude-code.repo` 작성 후 `sudo dnf install claude-code`(첫 설치 시 지문 확인), 업그레이드 `sudo dnf upgrade claude-code`. apk는 `claude-code.rsa.pub` 키를 받고 저장소 추가 후 `apk add claude-code`(키 sha256 검증: `395759c1...168b6`).

#### npm 설치

```bash
npm install -g @anthropic-ai/claude-code   # Node.js 18+ 필요
# 업그레이드: npm install -g @anthropic-ai/claude-code@latest  (npm update -g 는 피함)
```

npm 패키지는 standalone과 동일한 네이티브 바이너리를 플랫폼별 optional dependency(예 `@anthropic-ai/claude-code-darwin-arm64`)로 끌어와 postinstall로 링크한다. 설치된 `claude` 바이너리는 Node를 호출하지 않는다. 지원 플랫폼: `darwin-arm64`, `darwin-x64`, `linux-x64`, `linux-arm64`, `linux-x64-musl`, `linux-arm64-musl`, `win32-x64`, `win32-arm64`. 패키지 매니저가 optional dependencies를 허용해야 한다.

> [!warning] sudo npm install -g 금지
> 권한 문제·보안 위험을 유발한다. 권한 에러는 [[25 트러블슈팅]] 참조.

#### 바이너리 무결성 / 코드 서명

각 릴리스는 플랫폼별 SHA256 체크섬을 담은 `manifest.json`을 게시하고, 이 매니페스트는 Anthropic GPG 키로 서명된다(매니페스트 서명 검증 = 그 안의 모든 바이너리 검증). 매니페스트 서명은 `2.1.89`부터 제공.

```bash
# 1) 공개키 import 후 지문 확인 (31DD DE24 DDFA B679 F42D 7BD2 BAA9 29FF 1A7E CACE)
curl -fsSL https://downloads.claude.ai/keys/claude-code.asc | gpg --import
gpg --fingerprint security@anthropic.com
# 2) 매니페스트와 서명 다운로드
REPO=https://downloads.claude.ai/claude-code-releases; VERSION=2.1.89
curl -fsSLO "$REPO/$VERSION/manifest.json"; curl -fsSLO "$REPO/$VERSION/manifest.json.sig"
# 3) 서명 검증 (Good signature 이면 통과; "not certified" 경고는 정상)
gpg --verify manifest.json.sig manifest.json
# 4) 바이너리 체크섬 대조
sha256sum claude            # Linux
shasum -a 256 claude        # macOS
# PowerShell: (Get-FileHash claude.exe -Algorithm SHA256).Hash.ToLower()
```

플랫폼 네이티브 코드 서명: macOS는 "Anthropic PBC" 서명 + Apple notarize(`codesign --verify --verbose ./claude`), Windows는 "Anthropic, PBC"(`Get-AuthenticodeSignature .\claude.exe`), Linux은 개별 서명 없음(매니페스트 서명 또는 패키지 매니저로 검증).

### 제거 (uninstall)

설치 방법별로 제거. 제거 후에도 `claude`가 돈다면 두 번째 설치본이나 옛 셸 alias가 남은 것.

```bash
# Native (macOS/Linux/WSL)
rm -f ~/.local/bin/claude && rm -rf ~/.local/share/claude
# Homebrew
brew uninstall --cask claude-code   # 또는 claude-code@latest
# WinGet
winget uninstall Anthropic.ClaudeCode
# npm
npm uninstall -g @anthropic-ai/claude-code
```

apt/dnf/apk는 패키지 제거 + 저장소 설정 파일 제거.

```bash
# 설정/캐시 완전 제거 (모든 설정·허용 도구·MCP 구성·세션 이력 삭제됨!)
rm -rf ~/.claude && rm ~/.claude.json     # 사용자 설정·상태
rm -rf .claude && rm -f .mcp.json         # 프로젝트별 설정 (프로젝트 디렉터리에서)
```

> [!warning] 디렉터리 재생성 주의
> VS Code 확장·JetBrains 플러그인·Desktop 앱도 `~/.claude/`에 쓴다. 하나라도 설치돼 있으면 실행 시 디렉터리가 재생성된다. 완전 제거하려면 이들 통합을 먼저 제거하라.

## 인증 (authentication)

Claude Code는 셋업에 따라 여러 인증 방식을 지원한다. 개인은 Claude.ai 계정 로그인, 팀/조직은 Claude for Teams/Enterprise·Console·클라우드 프로바이더(Bedrock/Vertex/Foundry)를 쓴다.

### 로그인

설치 후 `claude` 실행 → 최초 실행 시 브라우저 로그인 창이 열린다.

- 브라우저가 자동으로 안 열리면 `c`를 눌러 로그인 URL을 클립보드에 복사 후 붙여넣기.
- 로그인 후 리다이렉트 대신 **로그인 코드**가 보이면 터미널의 `Paste code here if prompted`에 붙여넣기. (WSL2·SSH·컨테이너에서 로컬 콜백 서버에 못 닿을 때 흔함)
- 로그아웃·재인증: `/logout`.

지원 계정 타입: Claude Pro/Max 구독, Claude for Teams/Enterprise(관리자 초대 계정), Claude Console(관리자 초대 필요), 클라우드 프로바이더(Bedrock/Vertex/Foundry — `claude` 실행 전 환경변수 설정, 브라우저 로그인 불필요). 로그인 문제는 [[25 트러블슈팅]].

> [!note] 무료 플랜 제외
> Claude Code는 Pro/Max/Team/Enterprise/Console 계정이 필요하다. 무료 Claude.ai 플랜은 Claude Code 접근을 포함하지 않는다.

### 팀/조직 인증 설정

- **Claude for Teams**: 자가 서비스 플랜. 협업·관리·과금 관리. 소규모 팀에 적합.
- **Claude for Enterprise**: SSO, 도메인 캡처, 역할 기반 권한, 컴플라이언스 API, managed policy 설정 추가. 보안·컴플라이언스 요구가 큰 대규모 조직에 적합. (조직 정책·managed settings는 [[16 CI-CD와 팀 통합]], [[05 권한]])
- **Claude Console** (API 기반 과금): Console 계정 생성 → 멤버 초대(Bulk invite 또는 SSO) → 역할 부여(**Claude Code** 역할=Claude Code API 키만 생성 / **Developer** 역할=모든 종류 API 키 생성) → 각 사용자 셋업.
- **클라우드 프로바이더** (Bedrock/Vertex/Foundry): 프로바이더 셋업 → 환경변수·자격증명 생성 안내 배포 → 사용자 설치. ([[17 클라우드 프로바이더]])

> Console 최초 로그인 시 비용 추적용 "Claude Code" 워크스페이스가 자동 생성된다.

### 자격증명 관리 (credential management)

| 항목 | 내용 |
| --- | --- |
| **macOS 저장 위치** | 암호화된 macOS Keychain |
| **Linux** | `~/.claude/.credentials.json` (파일 모드 `0600`) |
| **Windows** | `%USERPROFILE%\.claude\.credentials.json` (사용자 프로필 디렉터리 접근 제어 상속) |
| **`CLAUDE_CONFIG_DIR` 설정 시** | Linux/Windows에서 `.credentials.json`이 그 디렉터리 아래로 |
| **커스텀 API 엔드포인트** | `ANTHROPIC_BASE_URL` 환경변수 |
| **지원 인증 타입** | Claude.ai, Claude API, Azure Auth, Bedrock Auth, Vertex Auth |
| **커스텀 자격증명 스크립트** | `apiKeyHelper` 설정 — API 키를 반환하는 셸 스크립트 실행 |
| **갱신 주기** | 기본 5분 후 또는 HTTP 401 시 `apiKeyHelper` 호출. `CLAUDE_CODE_API_KEY_HELPER_TTL_MS`로 커스텀 |
| **느린 helper 경고** | `apiKeyHelper`가 10초 초과 시 프롬프트 바에 경과 시간 경고 표시 |

`apiKeyHelper`·`ANTHROPIC_API_KEY`·`ANTHROPIC_AUTH_TOKEN`은 **터미널 CLI 세션에만** 적용된다. Claude Desktop과 원격 세션은 OAuth 전용으로, `apiKeyHelper`를 호출하거나 API 키 환경변수를 읽지 않는다.

#### 인증 우선순위 (precedence)

여러 자격증명이 있을 때 다음 순서로 선택한다.

1. **클라우드 프로바이더 자격증명** — `CLAUDE_CODE_USE_BEDROCK` / `CLAUDE_CODE_USE_VERTEX` / `CLAUDE_CODE_USE_FOUNDRY` 설정 시 ([[17 클라우드 프로바이더]]).
2. **`ANTHROPIC_AUTH_TOKEN`** — `Authorization: Bearer` 헤더로 전송. Bearer 토큰으로 인증하는 LLM 게이트웨이/프록시 경유 시.
3. **`ANTHROPIC_API_KEY`** — `X-Api-Key` 헤더. 직접 Anthropic API 접근. 대화형 모드에선 1회 승인/거부를 묻고 기억(이후 `/config`의 "Use custom API key" 토글로 변경). 비대화형(`-p`)은 키가 있으면 항상 사용.
4. **`apiKeyHelper` 스크립트 출력** — vault에서 가져오는 단명 토큰 등 동적·회전 자격증명.
5. **`CLAUDE_CODE_OAUTH_TOKEN`** — `claude setup-token`으로 생성한 장수 OAuth 토큰. 브라우저 로그인 불가한 CI·스크립트용.
6. **구독 OAuth 자격증명** — `/login`. Pro/Max/Team/Enterprise 기본값.

> [!warning] 구독 + API 키 충돌
> 활성 구독이 있어도 환경에 `ANTHROPIC_API_KEY`가 설정돼 있으면 (승인 후) API 키가 우선한다. 키가 비활성/만료 조직 소속이면 인증 실패를 일으킬 수 있다. `unset ANTHROPIC_API_KEY`로 구독으로 폴백하고 `/status`로 활성 방식을 확인하라. [Claude Code on the Web]은 항상 구독 자격증명을 쓰며, 샌드박스 환경의 `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN`은 이를 덮어쓰지 못한다.

#### 장수 토큰 생성 (long-lived token)

```bash
claude setup-token            # OAuth 인가 후 1년짜리 토큰을 터미널에 출력 (어디에도 저장 안 함)
export CLAUDE_CODE_OAUTH_TOKEN=your-token
```

CI·스크립트 등 대화형 브라우저 로그인이 불가한 환경용. 이 토큰은 Claude 구독으로 인증되며 Pro/Max/Team/Enterprise 플랜이 필요하다. **inference 전용**으로 스코프되어 [Remote Control] 세션은 만들 수 없다. Bare 모드(`--bare`)는 `CLAUDE_CODE_OAUTH_TOKEN`을 읽지 않으니 `ANTHROPIC_API_KEY`나 `apiKeyHelper`로 인증하라.

> [!note] 2026-06-15 변경 예정
> 구독 플랜에서 Agent SDK 및 `claude -p` 사용은 대화형 한도와 별개인 새 월간 Agent SDK 크레딧에서 차감된다. ([[22 Agent SDK — 시작]])

환경변수·설정 키 전체 레퍼런스는 [[04 설정]].

## 확장 기능 선택하기 (features-overview)

Claude Code는 코드를 추론하는 모델 + 파일/검색/실행/웹 내장 도구의 조합이다. 내장 도구로 대부분의 코딩이 되지만, 그 위에 **확장 레이어**를 얹어 Claude가 아는 것을 커스터마이즈하고, 외부 서비스를 연결하고, 워크플로우를 자동화한다.

> [!tip] 입문 순서
> CLAUDE.md(프로젝트 관례)부터 시작하고, 구체적 트리거가 생길 때마다 다른 확장을 추가하라. 모든 것을 미리 구성할 필요 없다.

### 확장이 에이전틱 루프에 끼어드는 지점

- **CLAUDE.md** — 매 세션 보는 지속 컨텍스트 ([[03 메모리와 컨텍스트]])
- **Skills** — 재사용 지식 + 호출 가능 워크플로우 ([[07 스킬]])
- **Code intelligence** — 언어 서버 연결, 심볼 단위 탐색·실시간 타입 에러 ([[06 내장 도구 레퍼런스]])
- **MCP** — 외부 서비스·도구 연결 ([[10 MCP]])
- **Subagents** — 격리 컨텍스트에서 자체 루프 실행 후 요약 반환 ([[08 서브에이전트와 에이전트 팀]])
- **Agent teams** — 공유 태스크와 P2P 메시징으로 독립 세션 다수 조율 ([[08 서브에이전트와 에이전트 팀]])
- **Hooks** — 라이프사이클 이벤트에 스크립트/HTTP/프롬프트/서브에이전트 실행 ([[09 훅]])
- **Plugins / marketplaces** — 위 기능들을 패키징·배포 ([[11 플러그인]])

스킬이 가장 유연한 확장이다. 스킬은 지식·워크플로우·지시를 담은 마크다운 파일로, `/deploy` 같은 명령으로 호출하거나 Claude가 관련 시 자동 로드한다. 현재 대화에서 실행하거나 서브에이전트로 격리 실행할 수 있다.

### 목표에 기능 매칭 (match features to your goal)

| Feature | 하는 일 | 언제 쓰나 | 예시 |
| --- | --- | --- | --- |
| **CLAUDE.md** | 매 대화 로드되는 지속 컨텍스트 | 프로젝트 관례, "항상 X" 규칙 | "Use pnpm, not npm. Run tests before committing." |
| **Skill** | Claude가 쓸 지시·지식·워크플로우 | 재사용 콘텐츠, 레퍼런스 문서, 반복 작업 | `/deploy` 배포 체크리스트; API 문서 스킬 |
| **Subagent** | 요약 결과를 반환하는 격리 실행 컨텍스트 | 컨텍스트 격리, 병렬 작업, 전문 워커 | 많은 파일을 읽되 핵심만 반환하는 리서치 |
| **Agent teams** | 독립 Claude Code 세션 다수 조율 | 병렬 리서치, 신규 기능 개발, 경쟁 가설 디버깅 | 보안·성능·테스트 리뷰어 동시 스폰 |
| **Code intelligence** | 언어 서버 탐색·진단 | 타입 언어, grep이 느리거나 부정확한 대형 코드베이스 | 파일 통독 대신 심볼 정의로 점프 |
| **MCP** | 외부 서비스 연결 | 외부 데이터·액션 | DB 쿼리, Slack 게시, 브라우저 제어 |
| **Hook** | 이벤트로 트리거되는 스크립트/HTTP/프롬프트/서브에이전트 | 매 매칭 이벤트마다 반드시 실행돼야 하는 자동화 | 파일 편집마다 ESLint 실행 |

**Plugins**는 패키징 레이어다. 스킬·훅·서브에이전트·MCP 서버를 하나의 설치 단위로 묶는다. 플러그인 스킬은 네임스페이스(`/my-plugin:review`)되어 공존 가능. 여러 repo에 같은 셋업을 재사용하거나 marketplace로 배포할 때 쓴다. ([[11 플러그인]])

### 시간이 지나며 셋업 쌓기 (트리거 → 추가)

| 트리거 | 추가할 것 |
| --- | --- |
| Claude가 관례·명령을 두 번 틀림 | CLAUDE.md에 추가 |
| 같은 시작 프롬프트를 반복 타이핑 | 사용자 호출 가능 스킬로 저장 |
| 같은 플레이북/다단계 절차를 세 번째로 붙여넣음 | 스킬로 캡처 |
| Claude가 못 보는 브라우저 탭에서 데이터를 반복 복사 | 그 시스템을 MCP 서버로 연결 |
| 심볼 정의·사용처를 찾으려 많은 파일을 읽음 | 언어용 code intelligence 플러그인 설치 |
| 부수 작업 출력이 대화를 범람시킴 (다시 안 볼 출력) | 서브에이전트로 라우팅 |
| 묻지 않고 매번 일어나길 원함 | 훅 작성 |
| 두 번째 repo가 같은 셋업 필요 | 플러그인으로 패키징 |

같은 트리거가 기존 것을 언제 갱신할지도 알려준다. 반복되는 실수·재발 리뷰 코멘트는 일회성 교정이 아니라 CLAUDE.md 수정 신호다.

### 비슷한 기능 구분 (compare similar features)

#### Skill vs Subagent

| 측면 | Skill | Subagent |
| --- | --- | --- |
| 정체 | 재사용 지시·지식·워크플로우 | 자체 컨텍스트를 가진 격리 워커 |
| 핵심 이점 | 컨텍스트 간 콘텐츠 공유 | 컨텍스트 격리, 요약만 반환 |
| 컨텍스트 윈도 영향 | 메인 윈도에 추가됨 | 별도 윈도(자체 입출력 토큰) |
| 적합 | 레퍼런스 자료, 호출 워크플로우 | 많은 파일 읽기, 병렬 작업, 전문 워커 |

스킬은 reference(세션 내내 쓰는 지식, 예 API 스타일 가이드) 또는 action(`/deploy`처럼 특정 작업 수행)일 수 있다. 컨텍스트 격리가 필요하거나 윈도가 차오를 때 서브에이전트를 쓴다. 둘은 조합 가능 — 서브에이전트가 특정 스킬을 preload(`skills:` 필드), 스킬이 `context: fork`로 격리 실행.

#### CLAUDE.md vs Skill

| 측면 | CLAUDE.md | Skill |
| --- | --- | --- |
| 로드 | 매 세션 자동 | on demand |
| 파일 포함 | 가능 (`@path` import) | 가능 (`@path` import) |
| 워크플로우 트리거 | 불가 | 가능 (`/<name>`) |
| 적합 | "항상 X" 규칙 | 레퍼런스 자료, 호출 워크플로우 |

Claude가 **항상 알아야 하면** CLAUDE.md(코딩 관례·빌드 명령·구조·"never do X"). **가끔 필요한 레퍼런스**나 `/<name>`로 트리거하는 워크플로우는 스킬. **규칙: CLAUDE.md는 200줄 이하 유지.** 커지면 스킬로 옮기거나 `.claude/rules/`로 분할.

#### CLAUDE.md vs Rules vs Skills

| 측면 | CLAUDE.md | `.claude/rules/` | Skill |
| --- | --- | --- | --- |
| 로드 | 매 세션 | 매 세션 또는 매칭 파일 열 때 | on demand (호출/관련 시) |
| 스코프 | 전체 프로젝트 | 파일 경로로 스코프 가능 | 작업 특정 |
| 적합 | 핵심 관례·빌드 명령 | 언어/디렉터리별 가이드라인 | 레퍼런스, 반복 워크플로우 |

`paths` frontmatter 있는 rules는 매칭 파일 작업 시에만 로드되어 컨텍스트를 절약한다. (rules·CLAUDE.md 상세는 [[03 메모리와 컨텍스트]])

#### Subagent vs Agent team

| 측면 | Subagent | Agent team |
| --- | --- | --- |
| 컨텍스트 | 자체 윈도; 결과는 호출자에게 반환 | 자체 윈도; 완전 독립 |
| 통신 | 메인 에이전트에만 결과 보고 | 팀원끼리 직접 메시지 |
| 조율 | 메인 에이전트가 모든 작업 관리 | 공유 태스크 목록 + 자기 조율 |
| 적합 | 결과만 중요한 집중 작업 | 토론·협업 필요한 복잡 작업 |
| 토큰 비용 | 낮음 (요약 반환) | 높음 (각 팀원이 별도 Claude 인스턴스) |

병렬 서브에이전트로 컨텍스트 한도에 부딪히거나 서브에이전트끼리 통신이 필요하면 agent team이 자연스러운 다음 단계. (Agent teams는 실험적이며 기본 비활성 — [[08 서브에이전트와 에이전트 팀]])

#### MCP vs Skill

MCP는 외부 서비스 연결(연결·인증을 서버가 처리하는 목적형 도구 제공), 스킬은 그 도구를 잘 쓰는 법을 포함한 지식 확장. 함께 잘 동작한다 — 예: MCP가 DB에 연결하고, 스킬이 데이터 모델·쿼리 패턴·작업별 사용 테이블을 가르친다.

#### Hook vs Skill

| 측면 | Hook | Skill |
| --- | --- | --- |
| 실행 | 셸 명령/HTTP/LLM 프롬프트/서브에이전트 | Claude가 읽고 따르는 지시 |
| 트리거 | 라이프사이클 이벤트(`PostToolUse`·`SessionStart` 등) | `/<name>` 입력 또는 Claude가 설명 매칭 |
| 결정성 | 이벤트마다 항상 발화(트리거 보장) | Claude가 해석, 결과 가변 |
| 컨텍스트 비용 | 출력 반환 안 하면 0 | 설명은 매 세션, 전체는 사용 시 |
| 적합 | 편집 후 lint, 위험 명령 차단, 로깅, 알림 | 추론 필요 워크플로우, 레퍼런스, 다단계 |

**가드레일은 훅에 넣어라.** CLAUDE.md나 스킬의 "never edit `.env`"는 요청이지 보장이 아니다. 편집을 차단하는 `PreToolUse` 훅이 강제다. 매번 반드시 지켜져야 할 규칙은 프롬프트 지시가 아니라 훅으로 만든다. 훅 출력은 컨텍스트에 들어간다 — `PostToolUse` 훅이 linter를 돌리면 결과가 텍스트로 Claude에 피드백된다.

### 기능 레이어링 (어떤 레벨이 이기나)

기능은 user-wide / per-project / 플러그인 / managed policy 등 여러 레벨에서 정의될 수 있다.

- **CLAUDE.md** — 가산적(additive). 모든 레벨이 동시에 컨텍스트에 기여. 충돌 시 Claude가 판단하되 더 구체적인 지시가 우선. 작업 디렉터리·상위는 시작 시, 하위는 작업하며 로드.
- **Skills/Subagents** — 이름으로 override. 스킬: `managed > user > project`. 서브에이전트: `managed > CLI flag > project > user > plugin`. 플러그인 스킬은 네임스페이스로 충돌 회피.
- **MCP 서버** — 이름으로 override: `local > project > user`.
- **Hooks** — 병합(merge). 출처 무관하게 매칭 이벤트마다 모두 발화.

### 기능 조합 (combine features)

| 패턴 | 작동 방식 | 예시 |
| --- | --- | --- |
| **Skill + MCP** | MCP가 연결 제공, 스킬이 잘 쓰는 법 교육 | MCP가 DB 연결, 스킬이 스키마·쿼리 패턴 문서화 |
| **Skill + Subagent** | 스킬이 병렬 작업용 서브에이전트 스폰 | `/audit` 스킬이 보안·성능·스타일 서브에이전트 가동 |
| **CLAUDE.md + Skills** | CLAUDE.md는 상시 규칙, 스킬은 on-demand 레퍼런스 | "follow our API conventions" + 전체 API 스타일 가이드 스킬 |
| **Hook + MCP** | 훅이 MCP로 외부 액션 트리거 | 편집 후 훅이 중요 파일 변경 시 Slack 알림 |

### 컨텍스트 비용 이해 (understand context costs)

모든 기능은 Claude의 컨텍스트를 일부 소비한다. 너무 많으면 윈도를 채울 뿐 아니라 노이즈가 되어 Claude가 덜 효과적이게 된다(스킬이 잘못 트리거되거나 관례를 놓침).

| Feature | 로드 시점 | 무엇이 로드 | 컨텍스트 비용 |
| --- | --- | --- | --- |
| **CLAUDE.md** | 세션 시작 | 전체 내용 | 매 요청 |
| **Skills** | 세션 시작 + 사용 시 | 시작 시 설명, 사용 시 전체 | 낮음 (설명만 매 요청)* |
| **MCP servers** | 세션 시작 | 도구 이름; 스키마는 on demand | 도구 사용 전까지 낮음 |
| **Code intelligence** | 편집 후·on demand | 편집 후 진단; 조회 시 심볼 위치 | 낮음; 다른 파일 읽기를 줄임 |
| **Subagents** | 스폰 시 | 지정 스킬 포함 신선 컨텍스트 | 메인 세션과 격리 |
| **Hooks** | 트리거 시 | 없음(외부 실행) | 0 (출력 반환 시 제외) |

*기본적으로 스킬 설명은 세션 시작 시 로드되어 Claude가 사용 시점을 판단한다. frontmatter에 `disable-model-invocation: true`를 두면 수동 호출 전까지 Claude에게 숨겨져 비용이 0이 된다. 내가 작성하지 않은 스킬은 설정의 `skillOverrides`로 동일 처리. ([[07 스킬]])

#### 기능별 로드 상세

- **CLAUDE.md**: 세션 시작. 모든 레벨(managed/user/project) 전체 로드. 작업 디렉터리→루트로 읽고, 하위 디렉터리는 접근 시 발견. *Tip: 200줄 이하 유지, 레퍼런스는 스킬로.*
- **Skills**: 기본은 시작 시 설명·사용 시 전체. `disable-model-invocation: true`면 호출 전까지 아무것도 로드 안 됨. Claude는 작업과 스킬 설명을 매칭해 선택하므로 설명이 모호하거나 겹치면 오선택/누락 가능. 번들 스킬 예: `/code-review`, `/batch`, `/debug`. **서브에이전트에선** on-demand가 아니라 `skills` 필드에 나열된 스킬이 시작 시 전체 preload(미나열 스킬은 Skill 도구로 발견·호출 가능). *Tip: 부수효과 있는 스킬은 `disable-model-invocation: true`.*
- **MCP servers**: 시작 시 도구 이름만, 스키마는 deferred. [Tool search]가 기본 on이라 유휴 도구는 최소 비용. *Tip: `/mcp`로 서버별 연결 상태·토큰 비용 확인.*
- **Code intelligence**: 편집 후 타입 에러/경고, 심볼 조회 시 정의·참조·타입. 심볼 조회가 광범위 파일 읽기를 대체해 순 비용이 줄 수 있음. LSP 도구는 언어용 code intelligence 플러그인 설치 전까지 비활성.
- **Subagents**: 스폰 시 신선 격리 컨텍스트 — 에이전트 자체 시스템 프롬프트(전체 CC 시스템 프롬프트 아님), `skills:` 나열 스킬 전체, CLAUDE.md·git status(단 내장 Explore·Plan 에이전트는 둘 다 생략), 리드가 프롬프트로 넘긴 컨텍스트. 대화 이력·호출 스킬을 상속하지 않음.
- **Hooks**: 트리거 시 발화(도구 실행·세션 경계·프롬프트 제출·권한 요청·컴팩션 등). 기본 컨텍스트 로드 없음. 출력이 메시지로 추가되지 않는 한 비용 0.

전체 비용·성능 관리는 [[19 비용과 성능]] 참조.
