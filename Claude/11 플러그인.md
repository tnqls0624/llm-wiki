# 11 플러그인

플러그인(plugin)은 [[07 스킬]], [[08 서브에이전트와 에이전트 팀]], [[09 훅]], [[10 MCP]] 서버, LSP 서버, 백그라운드 모니터를 하나의 자기완결적(self-contained) 디렉터리로 묶어 프로젝트·팀·커뮤니티 간에 **공유·버전관리·배포**할 수 있게 하는 패키징 단위다. 마켓플레이스(marketplace)는 이 플러그인들의 카탈로그다.

이 노트는 6개 원문 페이지(생성·레퍼런스·설치/발견·의존성·CLI 힌트·마켓플레이스)를 통합한 실전 레퍼런스다. 허브는 [[Claude]].

> [!tip] 핵심 멘탈 모델
> - **standalone(`.claude/`)** = 빠른 개인용 실험, 짧은 스킬 이름(`/hello`)
> - **plugin** = 공유·배포·버전관리·재사용, 네임스페이스 스킬(`/plugin-name:hello`)
> - **marketplace** = 앱스토어처럼 "스토어를 추가(add)"한 뒤 "앱을 개별 설치(install)"하는 2단계 구조

---

## 플러그인 vs 단독(standalone) 설정 — 언제 무엇을 쓰나

Claude Code는 커스텀 스킬·에이전트·훅을 추가하는 두 가지 방식을 지원한다.

| 방식 | 스킬 이름 | 적합한 경우 |
|:---|:---|:---|
| **Standalone** (`.claude/` 디렉터리) | `/hello` | 개인 워크플로우, 프로젝트 한정 커스터마이징, 빠른 실험 |
| **Plugins** (스킬/에이전트/훅 + `.claude-plugin/plugin.json` 매니페스트를 담은 디렉터리) | `/plugin-name:hello` | 팀·커뮤니티 공유, 버전 릴리스, 여러 프로젝트 재사용, 마켓플레이스 배포 |

판단 기준:
- **standalone을 쓸 때**: 단일 프로젝트 커스터마이징, 공유 불필요, 패키징 전 실험 단계, `/deploy` 같은 짧은 이름을 원할 때.
- **plugin을 쓸 때**: 팀/커뮤니티와 공유, 여러 프로젝트에서 동일 스킬 필요, 버전 관리·손쉬운 업데이트, 마켓플레이스 배포, 네임스페이스 스킬(`/my-plugin:hello`)을 수용할 때. 네임스페이싱은 플러그인 간 동일 이름 스킬 충돌을 방지한다.

> 권장 흐름: `.claude/`에서 빠르게 반복 → 공유 준비되면 플러그인으로 [전환](#기존-설정을-플러그인으로-변환).

## 플러그인 만들기 (Quickstart & 구조)

원문: `plugins`

### 최소 플러그인 만들기 (Quickstart)

전제: Claude Code가 설치·인증되어 있어야 한다. `/plugin` 명령이 안 보이면 최신 버전으로 업데이트하라([[25 트러블슈팅]] 참고).

```bash
# 1. 플러그인 디렉터리 생성
mkdir my-first-plugin

# 2. 매니페스트 디렉터리 + 파일 생성
mkdir my-first-plugin/.claude-plugin
```

`my-first-plugin/.claude-plugin/plugin.json`:

```json
{
  "name": "my-first-plugin",
  "description": "A greeting plugin to learn the basics",
  "version": "1.0.0",
  "author": {
    "name": "Your Name"
  }
}
```

| 필드 | 용도 |
|:---|:---|
| `name` | 고유 식별자이자 스킬 네임스페이스. 스킬은 이 접두사가 붙는다(`/my-first-plugin:hello`). |
| `description` | 플러그인 매니저에서 탐색·설치 시 표시. |
| `version` | 선택. 설정하면 이 필드를 올릴 때만 사용자가 업데이트를 받는다. 생략하고 git 배포 시 커밋 SHA가 버전이 되어 모든 커밋이 새 버전으로 취급됨. |
| `author` | 선택. 출처 표기용. |

스킬 추가 — 스킬은 `skills/` 디렉터리 안에 `<name>/SKILL.md` 폴더로 둔다. 폴더명이 스킬명이 되고 플러그인 네임스페이스가 붙는다.

```bash
mkdir -p my-first-plugin/skills/hello
```

`my-first-plugin/skills/hello/SKILL.md`:

```markdown
---
description: Greet the user with a friendly message
disable-model-invocation: true
---

Greet the user warmly and ask how you can help them today.
```

로컬 테스트 — `--plugin-dir` 플래그로 설치 없이 로드한다.

```bash
claude --plugin-dir ./my-first-plugin
```

```shell
/my-first-plugin:hello
```

`/help`을 실행하면 플러그인 네임스페이스 아래 스킬이 보인다. 네임스페이스 접두사를 바꾸려면 `plugin.json`의 `name`을 수정한다.

### 스킬 인자 받기

`$ARGUMENTS` 플레이스홀더가 스킬명 뒤 입력을 캡처한다.

```markdown
---
description: Greet the user with a personalized message
---

# Hello Skill

Greet the user named "$ARGUMENTS" warmly and ask how you can help them today. Make the greeting personal and encouraging.
```

```shell
/reload-plugins
/my-first-plugin:hello Alex
```

스킬 인자 상세는 [[07 스킬]]을 참고.

### 스킬 디렉터리에서 플러그인 개발 (skills-dir)

매번 `--plugin-dir`를 넘기는 대신, 스킬 디렉터리에 플러그인을 두면 Claude Code가 자동 로드한다. `claude plugin init`이 스캐폴딩한다.

```bash
claude plugin init my-tool
```

이는 `~/.claude/skills/my-tool/`에 `.claude-plugin/plugin.json` 매니페스트와 시작용 `SKILL.md`를 만든다. 다음 세션에 `my-tool@skills-dir`로 로드되며 마켓플레이스나 설치 단계가 없다. 자세한 규칙은 [skills-dir 플러그인](#skills-directory-플러그인) 절 참고.

### 플러그인 구조 개요

> [!warning] 흔한 실수
> `commands/`, `agents/`, `skills/`, `hooks/`를 `.claude-plugin/` **안에** 두지 말 것. `.claude-plugin/` 안에는 오직 `plugin.json`만. 나머지 디렉터리는 모두 플러그인 루트에 둔다.

| 디렉터리 | 위치 | 용도 |
|:---|:---|:---|
| `.claude-plugin/` | 플러그인 루트 | `plugin.json` 매니페스트 (컴포넌트가 기본 위치를 쓰면 선택) |
| `skills/` | 플러그인 루트 | `<name>/SKILL.md` 디렉터리 형태의 스킬 |
| `commands/` | 플러그인 루트 | 플랫 마크다운 파일 형태의 스킬. 신규 플러그인은 `skills/` 권장 |
| `agents/` | 플러그인 루트 | 커스텀 에이전트 정의 |
| `hooks/` | 플러그인 루트 | `hooks.json` 안의 이벤트 핸들러 |
| `.mcp.json` | 플러그인 루트 | MCP 서버 설정 |
| `.lsp.json` | 플러그인 루트 | 코드 인텔리전스용 LSP 서버 설정 |
| `monitors/` | 플러그인 루트 | `monitors.json` 안의 백그라운드 모니터 설정 |
| `bin/` | 플러그인 루트 | 플러그인 활성화 동안 Bash 도구 `PATH`에 추가되는 실행파일 |
| `settings.json` | 플러그인 루트 | 활성화 시 적용되는 기본 [[04 설정]] |

### 복잡한 플러그인 개발

**스킬 추가** — `skills/` 디렉터리에 `SKILL.md` 폴더들을 둔다. 스킬은 모델 호출형(model-invoked)으로 Claude가 작업 맥락에 따라 자동 사용한다. 설치 후 `/reload-plugins`로 로드. 상세는 [[07 스킬]].

```yaml
---
description: Reviews code for best practices and potential issues. Use when reviewing code, checking PRs, or analyzing code quality.
---

When reviewing code, check for:
1. Code organization and structure
2. Error handling
3. Security concerns
4. Test coverage
```

**LSP 서버 추가** — 흔한 언어(TS/Python/Rust 등)는 공식 마켓플레이스의 사전제작 LSP 플러그인을 설치하는 게 좋다. 미지원 언어만 직접 만든다. `.lsp.json`을 추가:

```json
{
  "go": {
    "command": "gopls",
    "args": ["serve"],
    "extensionToLanguage": {
      ".go": "go"
    }
  }
}
```

사용자는 해당 언어 서버 바이너리를 직접 설치해야 한다. 전체 옵션은 [LSP 서버](#lsp-서버) 절.

**백그라운드 모니터 추가** — 로그·파일·외부 상태를 백그라운드에서 감시하고 이벤트를 Claude에 알림으로 전달한다. 플러그인이 활성화되면 Claude Code가 각 모니터를 자동 시작하므로 별도 지시가 필요 없다. `monitors/monitors.json`:

```json
[
  {
    "name": "error-log",
    "command": "tail -F ./logs/error.log",
    "description": "Application error log"
  }
]
```

`command`의 각 stdout 줄이 세션 동안 알림으로 전달된다. 전체 스키마는 [모니터](#모니터) 절.

**기본 설정 동봉 (`settings.json`)** — 플러그인 루트의 `settings.json`이 활성화 시 기본 설정을 적용한다. 현재 `agent`와 `subagentStatusLine` 키만 지원. `agent`를 지정하면 플러그인의 커스텀 에이전트가 메인 스레드로 활성화되어 그 시스템 프롬프트·도구 제한·모델이 적용된다.

```json
{
  "agent": "security-reviewer"
}
```

`settings.json`의 설정은 `plugin.json`의 `settings`보다 우선한다. 알 수 없는 키는 조용히 무시된다.

### 로컬 테스트 (--plugin-dir / --plugin-url)

```bash
claude --plugin-dir ./my-plugin
```

`.zip` 아카이브도 받는다(Claude Code v2.1.128+):

```bash
claude --plugin-dir ./my-plugin.zip
```

`--plugin-dir` 플러그인이 이미 설치된 마켓플레이스 플러그인과 같은 이름이면, 해당 세션에서는 로컬 사본이 우선한다(설치 제거 없이 변경 테스트 가능). 단, 관리형 설정이 강제 활성/비활성한 플러그인은 `--plugin-dir`로 덮어쓸 수 없다.

변경 후 재시작 없이 반영하려면 `/reload-plugins` — 플러그인·스킬·에이전트·훅·플러그인 MCP/LSP 서버를 모두 다시 로드한다. 테스트 체크: 스킬은 `/plugin-name:skill-name`, 에이전트는 `/agents`에 보이는지, 훅은 기대대로 동작하는지 확인.

여러 플러그인 동시 로드:

```bash
claude --plugin-dir ./plugin-one --plugin-dir ./plugin-two
```

URL에 호스팅된 `.zip`(예: CI 빌드 산출물)을 테스트하려면 `--plugin-url`을 쓴다. 시작 시 가져와 그 세션 동안만 로드한다. 가져오기 실패/아카이브 무효 시 로드 에러를 보고하고 그 없이 시작한다. 신뢰하는 아카이브만 가리킬 것.

```bash
claude --plugin-url https://example.com/my-plugin.zip --plugin-url https://example.com/other.zip
# 또는 공백 구분 한 인자로
claude --plugin-url "https://example.com/my-plugin.zip https://example.com/other.zip"
```

### 디버그

1. **구조 확인**: 디렉터리가 `.claude-plugin/` 안이 아니라 플러그인 루트에 있는지.
2. **컴포넌트 개별 테스트**: 스킬·에이전트·훅 각각 점검.
3. **검증·디버깅 도구 사용**: [디버깅 및 개발 도구](#디버깅-및-개발-도구) 절의 CLI 명령 활용.

## 기존 설정을 플러그인으로 변환

원문: `plugins`

`.claude/`에 이미 스킬·훅이 있으면 공유·배포를 위해 플러그인으로 변환할 수 있다.

```bash
# 1. 플러그인 구조 생성
mkdir -p my-plugin/.claude-plugin
```

`my-plugin/.claude-plugin/plugin.json` 작성 후 기존 파일 복사:

```bash
cp -r .claude/commands my-plugin/
cp -r .claude/agents my-plugin/   # 있으면
cp -r .claude/skills my-plugin/   # 있으면
```

훅 마이그레이션 — `hooks/hooks.json`을 만들고 `.claude/settings.json`/`settings.local.json`의 `hooks` 객체를 그대로 복사한다(포맷 동일). 명령은 stdin으로 JSON 입력을 받으므로 `jq`로 파일 경로를 추출한다.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{ "type": "command", "command": "jq -r '.tool_input.file_path' | xargs npm run lint:fix" }]
      }
    ]
  }
}
```

### 변환 시 달라지는 점

| Standalone (`.claude/`) | Plugin |
|:---|:---|
| 한 프로젝트에서만 사용 가능 | 마켓플레이스로 공유 가능 |
| `.claude/commands/`의 파일 | `plugin-name/commands/`의 파일 |
| `settings.json`의 훅 | `hooks/hooks.json`의 훅 |
| 공유하려면 수동 복사 | `/plugin install`로 설치 |

마이그레이션 후 `.claude/`의 원본 파일은 제거해 중복을 피한다(로드 시 플러그인 버전이 우선).

### 커뮤니티 마켓플레이스에 제출

Anthropic은 두 개의 공개 마켓플레이스를 운영한다:
- **`claude-plugins-official`**: Anthropic이 큐레이션하는 공식 세트. 모든 설치본에서 자동 사용 가능.
- **`claude-community`**: 심사 후 서드파티 제출이 올라가는 공개 커뮤니티 마켓플레이스. `/plugin marketplace add anthropics/claude-plugins-community`로 추가하고 `@claude-community`로 설치.

제출 폼: Claude.ai는 `claude.ai/settings/plugins/submit`, Console은 `platform.claude.com/plugins/submit`. 제출 전 로컬에서 `claude plugin validate` 실행. 승인된 플러그인은 커뮤니티 카탈로그에서 특정 커밋 SHA에 고정되며 CI가 새 커밋 푸시 시 핀을 자동 갱신한다(공개 카탈로그는 야간 동기화라 지연 가능). 공식 마켓플레이스는 별도 큐레이션이며 신청 절차가 없다(제출 폼은 공식에 추가하지 않음).

## 플러그인 발견 및 설치

원문: `discover-plugins`

### 마켓플레이스 동작 방식 (2단계)

1. **마켓플레이스 추가(add)**: 카탈로그를 Claude Code에 등록(아직 설치 없음).
2. **개별 플러그인 설치(install)**: 카탈로그를 둘러보고 원하는 것만 설치.

### 공식 Anthropic 마켓플레이스

`claude-plugins-official`은 시작 시 자동 사용 가능. `/plugin` → **Discover** 탭에서 탐색하거나 `claude.com/plugins`에서 카탈로그 확인.

```shell
/plugin install github@claude-plugins-official
```

"마켓플레이스에서 플러그인을 찾을 수 없음"이라 나오면 마켓플레이스가 없거나 오래된 것:

```shell
/plugin marketplace update claude-plugins-official
# 또는 처음이면
/plugin marketplace add anthropics/claude-plugins-official
```

#### 공식 마켓플레이스 카테고리

**코드 인텔리전스 (LSP)** — Claude의 내장 LSP 도구를 켜서 정의로 점프, 참조 찾기, 편집 직후 타입 오류 확인 등을 가능케 한다. 언어 서버 바이너리가 시스템에 설치돼 있어야 한다.

| 언어 | 플러그인 | 필요 바이너리 |
|:---|:---|:---|
| C/C++ | `clangd-lsp` | `clangd` |
| C# | `csharp-lsp` | `csharp-ls` |
| Go | `gopls-lsp` | `gopls` |
| Java | `jdtls-lsp` | `jdtls` |
| Kotlin | `kotlin-lsp` | `kotlin-language-server` |
| Lua | `lua-lsp` | `lua-language-server` |
| PHP | `php-lsp` | `intelephense` |
| Python | `pyright-lsp` | `pyright-langserver` |
| Rust | `rust-analyzer-lsp` | `rust-analyzer` |
| Swift | `swift-lsp` | `sourcekit-lsp` |
| TypeScript | `typescript-lsp` | `typescript-language-server` |

설치 후 `/plugin` Errors 탭에 `Executable not found in $PATH`가 보이면 위 바이너리를 설치한다. 코드 인텔리전스 플러그인이 주는 두 가지: (1) **자동 진단** — 편집마다 언어 서버가 타입 오류·누락 import·구문 오류를 자동 보고, Claude가 같은 턴에 수정. `Ctrl+O`로 인라인 진단 확인. (2) **코드 네비게이션** — 정의 점프, 참조 찾기, 호버 타입정보, 심볼 목록, 구현 찾기, 호출 계층 추적(grep보다 정밀, 언어/환경별 가용성 차이).

**외부 연동 (사전 구성된 [[10 MCP]] 서버 번들)**:
- 소스 제어: `github`, `gitlab`
- 프로젝트 관리: `atlassian`(Jira/Confluence), `asana`, `linear`, `notion`
- 디자인: `figma`
- 인프라: `vercel`, `firebase`, `supabase`
- 커뮤니케이션: `slack`
- 모니터링: `sentry`

**자동 보안 리뷰** — `security-guidance` 플러그인은 Claude의 각 변경을 흔한 취약점에 대해 리뷰하고 같은 세션에서 수정하도록 지시한다([[18 보안과 샌드박스]] 관련).

**개발 워크플로우**:
- `commit-commands`: 커밋·푸시·PR 생성 등 Git 워크플로우
- `pr-review-toolkit`: PR 리뷰 전문 에이전트
- `agent-sdk-dev`: Claude Agent SDK 빌드 도구 (관련 [[22 Agent SDK — 시작]])
- `plugin-dev`: 플러그인 제작 툴킷

**출력 스타일** (관련 [[12 출력 스타일과 상태줄]]):
- `explanatory-output-style`: 구현 선택에 대한 교육적 통찰
- `learning-output-style`: 스킬 빌딩용 인터랙티브 학습 모드

`anthropics/claude-plugins-community`는 자동 검증·안전 심사를 통과한 서드파티 플러그인을 호스팅(각 플러그인은 특정 커밋 SHA에 고정). 수동 추가:

```shell

/plugin install <plugin-name>@claude-community
```

### 데모 마켓플레이스 체험

```shell
/plugin marketplace add anthropics/claude-code   # claude-code-plugins
```

`/plugin` 매니저는 **Tab**(역방향 **Shift+Tab**)으로 순환하는 4개 탭을 연다:
- **Discover**: 모든 마켓플레이스의 플러그인 탐색. 현재 작업 디렉터리에 관련된 플러그인은 상단에 "suggested for this directory" 라벨로 고정(v2.1.154+).
- **Installed**: 설치된 플러그인 보기/관리.
- **Marketplaces**: 마켓플레이스 추가/제거/업데이트.
- **Errors**: 플러그인 로딩 오류.

플러그인 상세 패널이 보여주는 것: **Context cost**(매 턴 컨텍스트 윈도우에 추가되는 토큰 추정, v2.1.143+), **Last updated**(v2.1.144+), **Will install** 섹션(설치할 commands/agents/skills/hooks/MCP·LSP 서버 목록, v2.1.145+).

설치 후 `/reload-plugins`로 활성화. 스킬은 플러그인명으로 네임스페이스됨(`/commit-commands:commit` 등).

### 마켓플레이스 추가하기

```shell
/plugin marketplace add <source>
```

> [!tip] 단축
> `/plugin marketplace` 대신 `/plugin market`, `remove` 대신 `rm` 가능.

지원 소스:
- **GitHub 저장소**: `owner/repo` 형식 (`anthropics/claude-code`)
- **Git URL**: 임의 git 저장소 URL(GitLab, Bitbucket, 셀프호스팅)
- **로컬 경로**: 디렉터리 또는 `marketplace.json` 직접 경로
- **원격 URL**: 호스팅된 `marketplace.json`의 직접 URL

```shell
# GitHub
/plugin marketplace add anthropics/claude-code

# 다른 Git 호스트 (.git 접미사로 클론 강제)
/plugin marketplace add https://gitlab.com/company/plugins.git
/plugin marketplace add git@gitlab.com:company/plugins.git
# 특정 브랜치/태그
/plugin marketplace add https://gitlab.com/company/plugins.git#v1.0.0

# 로컬 경로
/plugin marketplace add ./my-marketplace
/plugin marketplace add ./path/to/marketplace.json

# 원격 URL
/plugin marketplace add https://example.com/marketplace.json
```

URL 기반 마켓플레이스는 Git 기반보다 제약이 있다(상대 경로 플러그인 설치 시 "path not found" 발생 가능 — [트러블슈팅](#트러블슈팅) 참고).

### 플러그인 설치

```shell
/plugin install plugin-name@marketplace-name   # 기본 user 스코프
```

설치 스코프(상세는 [[04 설정]]·[설치 스코프](#플러그인-설치-스코프)):
- **User scope** (기본): 모든 프로젝트에서 본인만.
- **Project scope**: 이 저장소 협업자 전체(`.claude/settings.json`에 추가).
- **Local scope**: 이 저장소에서 본인만(협업자 비공유).
- **managed scope**: 관리자가 관리형 설정으로 설치(읽기 전용, 수정 불가).

> [!warning] 신뢰
> 설치 전 플러그인을 신뢰하는지 확인하라. Anthropic은 플러그인에 포함된 MCP 서버·파일·소프트웨어를 통제하지 않으며 의도대로 동작하는지 검증할 수 없다.

### 설치된 플러그인 관리

`/plugin` → **Installed** 탭. 목록은 스코프별로 묶이고 문제 우선 정렬(로드 에러·미해결 의존성이 상단, 그다음 즐겨찾기, 비활성은 하단 접힌 헤더). `f`로 즐겨찾기 토글, 타이핑으로 필터, Enter로 상세에서 활성/비활성/제거. 의존성을 선언한 플러그인 설치 시 함께 자동 설치된 의존성이 설치 출력에 표시됨.

직접 명령:

```shell
/plugin list                                  # --enabled / --disabled 필터 가능
/plugin disable plugin-name@marketplace-name
/plugin enable plugin-name@marketplace-name
/plugin uninstall plugin-name@marketplace-name
```

CLI에서 스코프 지정:

```shell
claude plugin install formatter@your-org --scope project
claude plugin uninstall formatter@your-org --scope project
```

### 재시작 없이 변경 적용 & 토큰 비용

```shell
/reload-plugins
```

모든 활성 플러그인을 다시 로드하고 플러그인·스킬·에이전트·훅·플러그인 MCP/LSP 서버 카운트를 표시한다. 다음 요청에 토큰 비용이 든다: 새로 로드된 컴포넌트가 대화에 추가되는 콘텐츠로 자기 자신을 announce하는 반면 기존 히스토리는 프롬프트 캐시에서 읽힌다. MCP 서버를 제공하는 플러그인은 도구가 tool search로 defer되지 않으면 캐시를 무효화해 전체 대화를 재독하므로 비용이 더 크다([[10 MCP]]·[[19 비용과 성능]] 관련).

### 마켓플레이스 관리 (요약)

```shell
/plugin marketplace list
/plugin marketplace update marketplace-name
/plugin marketplace remove marketplace-name   # 그 마켓플레이스에서 설치한 플러그인도 제거됨(주의)
```

### 자동 업데이트 설정

시작 시 마켓플레이스와 설치 플러그인을 자동 갱신할 수 있다. 갱신되면 `/reload-plugins`를 권하는 알림이 뜬다. UI에서 마켓플레이스별 토글: `/plugin` → Marketplaces → 마켓플레이스 선택 → **Enable/Disable auto-update**.

- 공식 Anthropic 마켓플레이스는 자동 업데이트 기본 **활성**.
- 서드파티·로컬 개발 마켓플레이스는 기본 **비활성**.
- 관리자는 관리형 설정의 [`extraKnownMarketplaces`](#팀-마켓플레이스-구성-extraknownmarketplaces) 항목마다 `"autoUpdate": true`를 설정해 조직 마켓플레이스 자동 업데이트를 강제할 수 있다.

전체 자동 업데이트(Claude Code + 플러그인) 비활성은 `DISABLE_AUTOUPDATER` 환경변수. Claude Code 자동 업데이트는 끄되 플러그인 자동 업데이트는 유지하려면:

```bash
export DISABLE_AUTOUPDATER=1
export FORCE_AUTOUPDATE_PLUGINS=1
```

### 팀 마켓플레이스 구성 (extraKnownMarketplaces)

팀 관리자는 `.claude/settings.json`에 마켓플레이스를 추가해 프로젝트 폴더를 신뢰할 때 설치를 유도할 수 있다.

```json
{
  "extraKnownMarketplaces": {
    "my-team-tools": {
      "source": {
        "source": "github",
        "repo": "your-org/claude-plugins"
      }
    }
  }
}
```

전체 옵션(`extraKnownMarketplaces`, `enabledPlugins`)은 [[04 설정]]의 플러그인 설정 참고.

### 보안

플러그인과 마켓플레이스는 사용자 권한으로 임의 코드를 실행할 수 있는 고신뢰 컴포넌트다. 신뢰하는 출처에서만 설치/추가하라. 조직은 [관리형 마켓플레이스 제한](#관리형-마켓플레이스-제한-strictknownmarketplaces)으로 추가 가능한 마켓플레이스를 통제할 수 있다. 상세는 [[18 보안과 샌드박스]].

### 트러블슈팅 (발견·설치)

`/plugin` 명령 미인식 시: `claude --version`으로 버전 확인 후 업데이트.
- Homebrew: `brew upgrade claude-code` (또는 `brew upgrade claude-code@latest`)
- npm: `npm install -g @anthropic-ai/claude-code@latest`
- 네이티브 설치본: Setup의 설치 명령 재실행

흔한 이슈:
- **마켓플레이스 미로딩**: URL 접근 가능 여부, `.claude-plugin/marketplace.json` 존재 확인.
- **설치 실패**: 소스 URL 접근성, 저장소 공개/접근권한 확인.
- **설치 후 파일 못 찾음**: 플러그인은 캐시로 복사되므로 디렉터리 밖 파일 참조 경로는 작동 안 함.
- **스킬 미표시**: `rm -rf ~/.claude/plugins/cache`로 캐시 비우고 재시작·재설치.

코드 인텔리전스 이슈:
- **언어 서버 미시작**: 바이너리가 `$PATH`에 있는지 확인, `/plugin` Errors 탭 확인.
- **높은 메모리 사용**: `rust-analyzer`·`pyright`는 대형 프로젝트에서 메모리를 많이 씀. `/plugin disable <plugin-name>`로 끄고 내장 검색 도구 사용.
- **모노레포 거짓 양성 진단**: 워크스페이스 미구성 시 내부 패키지 import 미해결 보고 가능(Claude 편집 능력에는 영향 없음).

## 플러그인 레퍼런스 (컴포넌트·스키마·CLI)

원문: `plugins-reference`

플러그인은 스킬·에이전트·훅·MCP 서버·LSP 서버·모니터를 담은 자기완결 디렉터리다.

### 컴포넌트: 스킬

- **위치**: 플러그인 루트의 `skills/` 또는 `commands/` 디렉터리, 또는 루트의 단일 `SKILL.md`.
- **형식**: 스킬은 `SKILL.md`가 있는 디렉터리, command는 단순 마크다운 파일.
- 설치 시 자동 발견, Claude가 작업 맥락에 따라 자동 호출, `SKILL.md` 옆에 보조 파일 동봉 가능. 상세 [[07 스킬]].

```text
skills/
├── pdf-processor/
│   ├── SKILL.md
│   ├── reference.md (선택)
│   └── scripts/ (선택)
└── code-reviewer/
    └── SKILL.md
```

### 컴포넌트: 에이전트

- **위치**: 플러그인 루트의 `agents/`. **형식**: 마크다운.

```markdown
---
name: agent-name
description: What this agent specializes in and when Claude should invoke it
model: sonnet
effort: medium
maxTurns: 20
disallowedTools: Write, Edit
---

Detailed system prompt for the agent describing its role, expertise, and behavior.
```

플러그인 에이전트가 지원하는 frontmatter: `name`, `description`, `model`, `effort`, `maxTurns`, `tools`, `disallowedTools`, `skills`, `memory`, `background`, `isolation`. `isolation`의 유일한 유효값은 `"worktree"`. **보안상 `hooks`, `mcpServers`, `permissionMode`는 플러그인 동봉 에이전트에서 미지원**. 에이전트는 `/agents`에 표시되고 자동/수동 호출 가능. 상세 [[08 서브에이전트와 에이전트 팀]].

### 컴포넌트: 훅

- **위치**: 플러그인 루트의 `hooks/hooks.json` 또는 `plugin.json` 인라인.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "\"${CLAUDE_PLUGIN_ROOT}\"/scripts/format-code.sh"
          }
        ]
      }
    ]
  }
}
```

플러그인 훅은 사용자 정의 훅과 동일 라이프사이클 이벤트에 반응한다([[09 훅]]). 주요 이벤트(발췌):

| 이벤트 | 발생 시점 |
|:---|:---|
| `SessionStart` | 세션 시작·재개 시 |
| `Setup` | `--init-only`, 또는 `-p` 모드의 `--init`/`--maintenance` 시작 시(CI·스크립트 1회 준비용) |
| `UserPromptSubmit` | 프롬프트 제출 시, Claude 처리 전 |
| `UserPromptExpansion` | 사용자 입력 명령이 프롬프트로 확장될 때(확장 차단 가능) |
| `PreToolUse` | 도구 호출 실행 전(차단 가능) |
| `PermissionRequest` | 권한 대화상자가 뜰 때 |
| `PermissionDenied` | auto 모드 분류기가 거부할 때. `{retry: true}` 반환 시 모델이 재시도 가능 |
| `PostToolUse` / `PostToolUseFailure` | 도구 호출 성공/실패 후 |
| `PostToolBatch` | 병렬 도구 호출 배치가 모두 끝난 후, 다음 모델 호출 전 |
| `Notification` / `MessageDisplay` | 알림 전송 시 / 어시스턴트 메시지 표시 중 |
| `SubagentStart` / `SubagentStop` | 서브에이전트 생성/종료 시 |
| `TaskCreated` / `TaskCompleted` | 태스크 생성/완료 표시 시 |
| `Stop` / `StopFailure` | Claude 응답 종료 시 / API 오류로 턴 종료 시 |
| `TeammateIdle` | 에이전트 팀 동료가 idle 직전 |
| `InstructionsLoaded` | CLAUDE.md나 `.claude/rules/*.md`가 컨텍스트에 로드될 때 |
| `ConfigChange` / `CwdChanged` / `FileChanged` | 설정 변경 / 작업 디렉터리 변경(cd 등) / 감시 파일 변경(`matcher`로 파일명 지정) |
| `WorktreeCreate` / `WorktreeRemove` | 워크트리 생성/제거 시 |
| `PreCompact` / `PostCompact` | 컨텍스트 압축 전/후 |
| `Elicitation` / `ElicitationResult` | MCP 서버가 입력 요청 시 / 응답 직전 |
| `SessionEnd` | 세션 종료 시 |

훅 타입: `command`(셸 명령/스크립트), `http`(이벤트 JSON을 POST), `mcp_tool`(MCP 서버 도구 호출), `prompt`(LLM으로 프롬프트 평가, `$ARGUMENTS` 플레이스홀더), `agent`(도구를 가진 에이전트형 검증기 실행).

### 컴포넌트: MCP 서버

- **위치**: 플러그인 루트의 `.mcp.json` 또는 `plugin.json` 인라인. 표준 MCP 서버 설정([[10 MCP]]).

```json
{
  "mcpServers": {
    "plugin-database": {
      "command": "${CLAUDE_PLUGIN_ROOT}/servers/db-server",
      "args": ["--config", "${CLAUDE_PLUGIN_ROOT}/config.json"],
      "env": {
        "DB_PATH": "${CLAUDE_PLUGIN_ROOT}/data"
      }
    },
    "plugin-api-client": {
      "command": "npx",
      "args": ["@company/mcp-server", "--plugin-mode"],
      "cwd": "${CLAUDE_PLUGIN_ROOT}"
    }
  }
}
```

플러그인 활성화 시 자동 시작되며 표준 MCP 도구로 Claude 툴킷에 통합된다.

### 컴포넌트: LSP 서버

- **위치**: 플러그인 루트의 `.lsp.json` 또는 `plugin.json` 인라인 `lspServers`.

```json
{
  "name": "my-plugin",
  "lspServers": {
    "go": {
      "command": "gopls",
      "args": ["serve"],
      "extensionToLanguage": { ".go": "go" }
    }
  }
}
```

**필수 필드**:

| 필드 | 설명 |
|:---|:---|
| `command` | 실행할 LSP 바이너리(PATH에 있어야 함) |
| `extensionToLanguage` | 파일 확장자 → 언어 식별자 매핑 |

**선택 필드**: `args`, `transport`(`stdio` 기본 / `socket`), `env`, `initializationOptions`, `settings`(`workspace/didChangeConfiguration`로 전달), `workspaceFolder`, `startupTimeout`(ms), `maxRestarts`.

> 언어 서버 바이너리는 별도 설치 필요. `Executable not found in $PATH` 시 해당 바이너리 설치.

사전제작 LSP 플러그인 예: `pyright-lsp`(`pip install pyright` 또는 `npm install -g pyright`), `typescript-lsp`(`npm install -g typescript-language-server typescript`), `rust-analyzer-lsp`(rust-analyzer 설치 안내). 바이너리 먼저 설치 후 마켓플레이스에서 플러그인 설치.

### 컴포넌트: 모니터

플러그인 활성화 시 Claude Code가 자동 시작하는 백그라운드 모니터. 세션 수명 동안 셸 명령을 돌리고 모든 stdout 줄을 Claude에 알림으로 전달한다. [Monitor 도구](#)와 동일 메커니즘([[06 내장 도구 레퍼런스]] 관련)이며 인터랙티브 CLI 세션에서만 동작하고 [훅](#컴포넌트-훅)과 같은 신뢰 수준으로 비샌드박스 실행. **Claude Code v2.1.105+ 필요**.

- **위치**: 플러그인 루트의 `monitors/monitors.json` 또는 `plugin.json` 인라인 `experimental.monitors`(실험적 컴포넌트).

```json
[
  {
    "name": "deploy-status",
    "command": "\"${CLAUDE_PLUGIN_ROOT}\"/scripts/poll-deploy.sh ${user_config.api_endpoint}",
    "description": "Deployment status changes"
  },
  {
    "name": "error-log",
    "command": "tail -F ./logs/error.log",
    "description": "Application error log",
    "when": "on-skill-invoke:debug"
  }
]
```

**필수**: `name`(플러그인 내 고유, 재로드/재호출 시 중복 프로세스 방지), `command`(세션 작업 디렉터리에서 지속 백그라운드 프로세스로 실행), `description`(태스크 패널·알림 요약에 표시). **선택**: `when` — `"always"`(기본, 세션 시작·재로드 시 시작) / `"on-skill-invoke:<skill-name>"`(해당 스킬이 처음 디스패치될 때 시작). `command`는 [환경 변수](#변수-치환-환경-변수) 치환 지원. 미세션 종료까지 비활성화해도 이미 실행 중인 모니터는 멈추지 않는다.

### 컴포넌트: 테마

`/theme`에 내장 프리셋·로컬 테마와 나란히 나타나는 컬러 테마. `themes/` 안의 JSON 파일로, `base` 프리셋과 sparse `overrides` 컬러 토큰 맵을 가진다(실험적 컴포넌트). 관련 [[12 출력 스타일과 상태줄]].

```json
{
  "name": "Dracula",
  "base": "dark",
  "overrides": {
    "claude": "#bd93f9",
    "error": "#ff5555",
    "success": "#50fa7b"
  }
}
```

플러그인 테마 선택 시 `custom:<plugin-name>:<slug>`가 사용자 설정에 영구화. 플러그인 테마는 읽기 전용이며 `/theme`에서 `Ctrl+E`로 `~/.claude/themes/`에 복사해 편집할 수 있다.

### 플러그인 설치 스코프

| 스코프 | 설정 파일 | 용도 |
|:---|:---|:---|
| `user` | `~/.claude/settings.json` | 모든 프로젝트의 개인 플러그인(기본) |
| `project` | `.claude/settings.json` | 버전관리로 공유되는 팀 플러그인 |
| `local` | `.claude/settings.local.json` | 프로젝트 한정, gitignore됨 |
| `managed` | 관리형 설정 | 관리형 플러그인(읽기 전용, 업데이트만) |

다른 Claude Code 설정과 동일 스코프 체계. 상세는 [[04 설정]].

### skills-directory 플러그인

스킬 디렉터리 하위의 어떤 폴더든 `.claude-plugin/plugin.json` 매니페스트를 가지면 다음 세션에 `<name>@skills-dir` 플러그인으로 로드된다(마켓플레이스·설치 단계 없음). 마켓플레이스 설치와 달리 캐시로 복사되지 않고 제자리에서 발견된다. `claude plugin init`으로 스캐폴딩.

| 가진 것 | 무엇인가 |
|:---|:---|
| `<skills-dir>/foo/SKILL.md` (매니페스트 없음) | 평범한 스킬 `foo` |
| `<skills-dir>/foo/.claude-plugin/plugin.json` | 플러그인 `foo@skills-dir` (자체 스킬·에이전트·훅 등 번들 가능) |
| `<plugin>/skills/bar/SKILL.md` | 플러그인 내부에 패키징된 스킬 `bar` |

로드 위치:

| 스킬 디렉터리 | 스코프 | 로드 조건 |
|:---|:---|:---|
| `~/.claude/skills/` | personal | 모든 프로젝트(본인 소유) |
| `<cwd>/.claude/skills/` | project | 해당 폴더 워크스페이스 신뢰 수락 후에만 |

프로젝트 스코프 플러그인은 저장소에 체크인되어 협업자에게 도달한다. 출처가 저장소이므로 `.claude/settings.json`과 같은 신뢰 게이트 후에만 로드되고, 코드 실행 컴포넌트는 추가 제한된다: MCP 서버는 프로젝트 `.mcp.json`과 동일한 서버별 승인, LSP 서버는 워크스페이스 신뢰 후에만 시작, 백그라운드 모니터는 로드 안 됨. personal 스코프는 이 제한이 없다.

> [!warning] 프로젝트 스코프 @skills-dir는 상위로 walk-up 안 함
> 프로젝트 스코프 `@skills-dir` 플러그인은 Claude Code를 시작한 디렉터리의 `.claude/skills/`에서만 로드된다. 평범한 스킬·command처럼 저장소 루트로 walk up 하지 않으므로 하위 디렉터리에서 실행하면 루트 플러그인을 놓친다. 저장소 루트에서 실행하거나 디렉터리 변경 후 `/reload-plugins`.

편집·재로드·비활성: 스킬의 `SKILL.md` 변경은 현재 세션에 즉시 반영. 그 외 컴포넌트(`hooks/`, `.mcp.json`, `agents/`, `output-styles/`)는 `/reload-plugins`나 재시작 필요. 중지하려면 폴더 삭제 또는 이름으로 비활성화(마켓플레이스 설치가 아니므로 uninstall 단계 없음):

```bash
claude plugin disable my-tool@skills-dir
```

### 플러그인 매니페스트 스키마 (plugin.json)

매니페스트는 **선택**. 생략하면 컴포넌트를 기본 위치에서 자동 발견하고 디렉터리명을 플러그인명으로 쓴다. 메타데이터나 커스텀 경로가 필요할 때 사용.

```json
{
  "name": "plugin-name",
  "displayName": "Plugin Name",
  "version": "1.2.0",
  "description": "Brief plugin description",
  "author": {
    "name": "Author Name",
    "email": "author@example.com",
    "url": "https://github.com/author"
  },
  "homepage": "https://docs.example.com/plugin",
  "repository": "https://github.com/author/plugin",
  "license": "MIT",
  "keywords": ["keyword1", "keyword2"],
  "skills": "./custom/skills/",
  "commands": ["./custom/commands/special.md"],
  "agents": ["./custom/agents/reviewer.md"],
  "hooks": "./config/hooks.json",
  "mcpServers": "./mcp-config.json",
  "outputStyles": "./styles/",
  "lspServers": "./.lsp.json",
  "experimental": {
    "themes": "./themes/",
    "monitors": "./monitors.json"
  },
  "dependencies": [
    "helper-lib",
    { "name": "secrets-vault", "version": "~2.1.0" }
  ]
}
```

**필수 필드**: 매니페스트를 포함한다면 `name`만 필수(kebab-case, 공백 없음). 네임스페이싱에 쓰임 — `plugin-dev` 플러그인의 `agent-creator` 에이전트는 `plugin-dev:agent-creator`로 표시.

**미인식 필드**: 최상위 미인식 필드는 무시된다. VS Code/Cursor 확장 매니페스트, npm `package.json`, MCPB/DXT 번들 매니페스트와 하나의 파일을 겸할 수 있다. `claude plugin validate`는 미인식 필드를 에러가 아닌 **경고**로 보고(1~2글자 오타면 추정 이름 제안). 타입이 틀린 필드는 여전히 실패(예: `keywords`가 배열 아닌 문자열이면 로드 에러). `--strict`로 경고를 에러로 취급(CI 권장).

```bash
claude plugin validate ./my-plugin --strict
```

**메타데이터 필드** (발췌):

| 필드 | 타입 | 설명 |
|:---|:---|:---|
| `$schema` | string | 에디터 자동완성/검증용 JSON Schema URL(로드 시 무시) |
| `displayName` | string | `/plugin` 피커 등 UI에 표시되는 사람용 이름. 공백·대소문자 허용, 네임스페이싱/조회엔 미사용. v2.1.143+ |
| `version` | string | 시맨틱 버전. 설정 시 그 문자열에 고정되어 올릴 때만 업데이트. 생략 시 git 커밋 SHA. 마켓플레이스 엔트리에도 있으면 `plugin.json`이 우선 |
| `description`/`author`/`homepage`/`repository`/`license`/`keywords` | — | 표준 메타데이터 |
| `defaultEnabled` | boolean | 사용자가 상태를 설정하지 않았을 때 활성으로 시작하는지(기본 `true`). v2.1.154+ |

**기본 활성화(defaultEnabled)**: `defaultEnabled: false`로 비활성 상태로 설치되게 한다(외부 서비스 연결 등 비용/스코프를 옵트인시키고 싶을 때). `claude plugin enable <plugin>`이나 `/plugin`으로 켠다. v2.1.154+. 우선순위: ① **사용자 설정**(어느 스코프든 `enabledPlugins` 항목 — 한번 쓰이면 업데이트/재설치에도 유지) ② **의존성 요구**(활성 플러그인이 요구하면 설치/활성 시 `true` 기록) > `defaultEnabled`. 마켓플레이스 엔트리의 같은 필드는 `plugin.json`보다 우선.

**컴포넌트 경로 필드** (발췌): `skills`(기본 `skills/`에 **추가**), `commands`/`agents`/`outputStyles`/`experimental.themes`/`experimental.monitors`(기본 디렉터리를 **대체**), `hooks`/`mcpServers`/`lspServers`(자체 병합 규칙), `userConfig`, `channels`, `dependencies`.

**실험적 컴포넌트(`experimental`)**: `themes`, `monitors`. 매니페스트 스키마가 안정화 동안 릴리스 간 바뀔 수 있음. 최상위 선언도 아직 동작하나 validate가 경고하며 향후 `experimental.*`가 강제될 예정.

#### 사용자 설정 (userConfig)

플러그인 활성화 시 사용자에게 값을 묻는다(`settings.json` 수동 편집 대체).

```json
{
  "userConfig": {
    "api_endpoint": {
      "type": "string",
      "title": "API endpoint",
      "description": "Your team's API endpoint"
    },
    "api_token": {
      "type": "string",
      "title": "API token",
      "description": "API authentication token",
      "sensitive": true
    }
  }
}
```

옵션 필드: `type`(필수: `string`/`number`/`boolean`/`directory`/`file`), `title`(필수), `description`(필수), `sensitive`(입력 마스킹+보안 저장소 저장), `required`, `default`, `multiple`(string에서 배열 허용), `min`/`max`(number 범위).

각 값은 MCP·LSP 서버 설정, 훅·모니터 명령에서 `${user_config.KEY}`로 치환되고(비민감 값은 스킬·에이전트 콘텐츠에서도 가능), 하위 프로세스에 `CLAUDE_PLUGIN_OPTION_<KEY>` 환경변수로 export된다. 비민감 값은 `settings.json`의 `pluginConfigs[<plugin-id>].options`에, 민감 값은 시스템 키체인(또는 `~/.claude/.credentials.json`)에 저장 — 키체인은 OAuth 토큰과 공유되고 약 2KB 한도라 작게 유지.

#### 채널 (channels)

대화에 콘텐츠를 주입하는 메시지 채널(Telegram/Slack/Discord 스타일)을 선언. 각 채널은 플러그인이 제공하는 MCP 서버에 바인딩.

```json
{
  "channels": [
    {
      "server": "telegram",
      "userConfig": {
        "bot_token": { "type": "string", "title": "Bot token", "description": "Telegram bot token", "sensitive": true },
        "owner_id": { "type": "string", "title": "Owner ID", "description": "Your Telegram user ID" }
      }
    }
  ]
}
```

`server`(필수)는 플러그인 `mcpServers`의 키와 일치해야 함. 채널별 `userConfig`는 최상위와 동일 스키마.

#### 경로 동작 규칙

- **기본 대체**: `commands`, `agents`, `outputStyles`, `experimental.themes`, `experimental.monitors`. 기본 디렉터리를 유지하며 추가하려면 명시적으로 나열(`"commands": ["./commands/", "./extras/"]`).
- **기본에 추가**: `skills`. 기본 `skills/`는 항상 스캔되고 나열한 디렉터리가 함께 로드.
- **자체 병합**: `hooks`, `mcpServers`, `lspServers`.

기본 폴더와 매니페스트 키가 둘 다 있으면 v2.1.140+에서 무시된 폴더를 `/doctor`·`claude plugin list`·`/plugin` 상세에 표시(매니페스트 경로로 로드). 모든 경로는 플러그인 루트 상대 + `./`로 시작. 루트에 `SKILL.md`가 직접 있고 `skills/` 하위와 `skills` 키가 없으면 v2.1.142+에서 단일 스킬 플러그인으로 자동 로드(`"skills": ["./"]` 불필요). 스킬 호출명은 frontmatter `name`, 없으면 디렉터리 basename.

#### 변수 치환 (환경 변수)

세 변수는 스킬·에이전트 콘텐츠, 훅·모니터 명령, MCP·LSP 서버 설정 어디서든 인라인 치환되고 하위 프로세스에 환경변수로 export된다.

| 변수 | 의미 |
|:---|:---|
| `${CLAUDE_PLUGIN_ROOT}` | 플러그인 설치 디렉터리 절대경로. 번들 스크립트·바이너리·설정 참조. 업데이트 시 경로 변함(이전 버전 디렉터리는 ~7일 후 정리, ephemeral로 취급하고 상태 저장 금지). 셸폼 훅·모니터에선 `"${CLAUDE_PLUGIN_ROOT}"`처럼 큰따옴표로 감쌀 것 |
| `${CLAUDE_PLUGIN_DATA}` | 업데이트를 넘어 지속되는 플러그인 상태 디렉터리. `node_modules`·venv·생성 코드·캐시 등. 처음 참조 시 자동 생성 |
| `${CLAUDE_PROJECT_DIR}` | 프로젝트 루트. 훅이 받는 `CLAUDE_PROJECT_DIR`와 동일. 공백 대응 위해 따옴표 |

```json
{
  "hooks": {
    "PostToolUse": [
      { "hooks": [{ "type": "command", "command": "\"${CLAUDE_PLUGIN_ROOT}\"/scripts/process.sh" }] }
    ]
  }
}
```

**지속 데이터 디렉터리**: `${CLAUDE_PLUGIN_DATA}`는 `~/.claude/plugins/data/{id}/`로 해석(`{id}`는 `a-zA-Z0-9_-` 외 문자를 `-`로 치환). 예: `formatter@my-marketplace` → `~/.claude/plugins/data/formatter-my-marketplace/`. 언어 의존성을 한번 설치해 세션·업데이트 간 재사용하는 패턴 — 번들 매니페스트와 데이터 디렉터리 사본을 비교해 다를 때 재설치:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "diff -q \"${CLAUDE_PLUGIN_ROOT}/package.json\" \"${CLAUDE_PLUGIN_DATA}/package.json\" >/dev/null 2>&1 || (cd \"${CLAUDE_PLUGIN_DATA}\" && cp \"${CLAUDE_PLUGIN_ROOT}/package.json\" . && npm install) || rm -f \"${CLAUDE_PLUGIN_DATA}/package.json\""
          }
        ]
      }
    ]
  }
}
```

데이터 디렉터리는 마지막 스코프에서 제거 시 자동 삭제(`/plugin`은 크기 표시 후 확인, CLI는 기본 삭제 — `--keep-data`로 보존).

### 플러그인 캐싱 및 파일 해석

플러그인은 두 방식으로 지정: ① `--plugin-dir`/`--plugin-url`(세션 한정), ② 마켓플레이스(이후 세션용 설치). 보안·검증을 위해 **마켓플레이스** 플러그인은 제자리 사용이 아니라 로컬 캐시(`~/.claude/plugins/cache`)로 복사된다. 각 설치 버전은 캐시 안 별도 디렉터리이며, 업데이트/제거 시 이전 버전은 orphan 표시 후 7일 뒤 자동 제거(동시 세션 보호용 유예). Glob·Grep은 orphan 디렉터리를 건너뜀.

**경로 탐색 제한**: 설치된 플러그인은 디렉터리 밖 파일을 참조할 수 없다(`../shared-utils` 등은 캐시로 복사 안 됨).

**심볼릭 링크로 마켓플레이스 내 파일 공유**: 심링크가 ① 플러그인 자체 디렉터리 내부면 상대 심링크로 보존, ② 같은 마켓플레이스 내 다른 곳이면 dereference되어 대상 콘텐츠가 복사(메타 플러그인의 `skills/`가 형제 플러그인 스킬을 링크 가능), ③ 마켓플레이스 밖이면 보안상 건너뜀. `--plugin-dir`/로컬 경로 설치는 자체 디렉터리 내부 심링크만 보존.

```bash
ln -s ../../shared-plugin/skills/foo ./skills/foo   # Windows는 mklink /D
```

│   └── plugin.json

├── agents/

### CLI 명령 레퍼런스

**plugin init** — `~/.claude/skills/<name>/`에 새 플러그인 스캐폴딩. 다음 세션에 `<name>@skills-dir`로 자동 로드.

```bash
claude plugin init <name> [options]
```

| 옵션 | 설명 | 기본 |
|:---|:---|:---|
| `--description <text>` | 매니페스트 description | |
| `--author <name>` | 작성자명 | `git config user.name` |
| `--author-email <email>` | 작성자 이메일 | `git config user.email` |
| `--with <components...>` | 컴포넌트 폴더 스캐폴딩: `skills`, `agents`, `hooks`, `mcp`, `lsp`, `output-style`, `channel` | |
| `-f, --force` | 기존 `.claude-plugin/` 덮어쓰기 | |

별칭 `new`. 관리자는 `strictKnownMarketplaces`나 `blockedMarketplaces`에 `{"source": "skills-dir"}` 추가로 이 소스를 차단할 수 있고, 차단되면 `plugin init`이 쓰기 전 실패.

```bash
claude plugin init my-helper
claude plugin init my-helper --with skills hooks
claude plugin init my-helper --force
```

**plugin install / uninstall / enable / disable / update**:

```bash
claude plugin install <plugin> [-s|--scope user|project|local]      # 기본 user
claude plugin uninstall <plugin> [--scope ...] [--keep-data] [--prune] [-y]   # 별칭 remove, rm
claude plugin enable <plugin> [--scope ...]
claude plugin disable <plugin> [--scope ...]
claude plugin update <plugin> [--scope user|project|local|managed]
```

- `install --scope project`는 `.claude/settings.json`의 `enabledPlugins`에 기록(클론한 모두에게 제공).
- `uninstall`은 기본적으로 마지막 스코프에서 제거 시 `${CLAUDE_PLUGIN_DATA}`도 삭제(`--keep-data`로 보존). `--prune`은 다른 플러그인이 안 쓰는 자동설치 의존성도 제거.

**plugin prune** — 어떤 설치 플러그인도 요구하지 않는 자동설치 의존성 제거(직접 설치한 플러그인은 손대지 않음). v2.1.121+. 별칭 `autoremove`.

```bash
claude plugin prune [-s|--scope ...] [--dry-run] [-y]
```

**plugin list** — 버전·소스 마켓플레이스·활성 상태 나열.

```bash
claude plugin list [--json] [--available]   # --available은 --json 필요
```

인터랙티브 `/plugin list`는 같은 목록을 인라인 출력하며 `--enabled`/`--disabled` 필터와 `ls` 약어 허용.

**plugin details** — 컴포넌트 인벤토리와 예상 토큰 비용 표시.

```bash
claude plugin details <name>
```

출력의 두 비용: **Always-on**(스킬·에이전트 설명, 명령명 등 리스팅 텍스트가 매 세션 추가하는 토큰 — 컴포넌트 발화 여부 무관), **On-invoke**(컴포넌트가 발화할 때 드는 토큰, 컴포넌트별 표시). Always-on 총합은 활성 모델의 `count_tokens` API로 계산(불가 시 문자 기반 추정), 컴포넌트별은 비례 환산. 관련 [[19 비용과 성능]].

```

  Skills (2)  scan-dependencies, review-changes
  Agents (0)

  ...

  component            always-on  on-invoke

```

**plugin tag** — 현재 디렉터리 플러그인의 릴리스 git 태그 생성(플러그인 폴더 안에서 실행).

```bash
claude plugin tag [--push] [--dry-run] [-f|--force]
```

### 버전 관리

Claude Code는 플러그인 버전을 캐시 키로 써서 업데이트 가용 여부를 판단한다. 버전 해석 순서:

1. `plugin.json`의 `version`
2. 마켓플레이스 엔트리의 `version`
3. git 커밋 SHA(`github`/`url`/`git-subdir`/git 호스팅 마켓플레이스의 상대경로 소스)
4. `unknown`(npm 소스, git 저장소 밖 로컬 디렉터리)

| 방식 | 방법 | 업데이트 동작 | 적합 |
|:---|:---|:---|:---|
| **명시 버전** | `plugin.json`에 `"version": "2.1.0"` | 이 필드를 올릴 때만 업데이트. 커밋만 푸시하면 `/plugin update`가 "already at the latest" | 안정 릴리스 사이클의 배포 플러그인 |
| **커밋 SHA 버전** | `plugin.json`·마켓플레이스 엔트리 모두 `version` 생략 | git 소스에 새 커밋마다 업데이트 | 활발히 개발 중인 내부/팀 플러그인 |

> [!warning] 버전 고정 주의
> `plugin.json`에 `version`을 설정하면 변경 시마다 올려야 한다. 커밋만 푸시하면 같은 버전 문자열이라 캐시 유지. 빠르게 반복 중이면 `version`을 비워 커밋 SHA를 쓰게 하라. 시맨틱 버저닝(`MAJOR.MINOR.PATCH`) 권장.

## 플러그인 의존성 버전 제약

원문: `plugin-dependencies` — **v2.1.110+ 필요**

플러그인은 `plugin.json`이나 마켓플레이스 엔트리에서 다른 플러그인에 의존할 수 있다. 기본적으로 의존성은 최신 버전을 추적하므로 업스트림 릴리스가 경고 없이 의존성을 바꿀 수 있다. 버전 제약으로 테스트한 범위에 고정할 수 있다.

설치 시 Claude Code가 의존성을 자동 해석·설치하고 설치 출력 끝에 추가된 의존성을 나열. 의존성이 사라지면 `/reload-plugins`·백그라운드 자동 업데이트가 재설치(그 마켓플레이스가 이미 구성돼 있으면). 추가하지 않은 마켓플레이스의 의존성은 미해결로 남음.

### 왜 제약하나

예: 플랫폼팀의 `secrets-vault`(MCP 서버)에 배포팀의 `deploy-kit`이 의존. `deploy-kit`은 `secrets-vault` v2.1.0에 테스트됨. 제약이 없으면 플랫폼팀이 MCP 도구명을 바꾸는 릴리스를 태그할 때 자동 업데이트가 모두를 신버전으로 옮겨 `deploy-kit`이 깨진다. `~2.1.0` 제약을 선언하면 최고 매칭 `2.1.x` 패치에 머물고, 배포팀은 넓은 제약의 새 `deploy-kit`을 배포해 자기 일정대로 올린다.

### 의존성 선언

`.claude-plugin/plugin.json`의 `dependencies` 배열. 각 항목은 플러그인명(bare 문자열) 또는 버전 제약 객체.

```json
{
  "name": "deploy-kit",
  "version": "3.1.0",
  "dependencies": [
    "audit-logger",
    { "name": "secrets-vault", "version": "~2.1.0" }
  ]
}
```

| 필드 | 타입 | 설명 |
|:---|:---|:---|
| `name` | string | 플러그인명. 선언 플러그인과 같은 마켓플레이스에서 해석. 필수 |
| `version` | string | [semver range](https://github.com/npm/node-semver#ranges): `~2.1.0`, `^2.0`, `>=1.4`, `=2.1.0` 등. 범위를 만족하는 최고 태그 버전을 가져옴 |
| `marketplace` | string | `name`을 해석할 다른 마켓플레이스. 크로스 마켓플레이스 의존성은 루트 마켓플레이스가 `allowCrossMarketplaceDependenciesOn`에 허용하지 않으면 차단 |

`version`은 Node `semver`가 지원하는 표현(caret/tilde/hyphen/comparator). 프리릴리스(`2.0.0-beta.1`)는 `^2.0.0-0`처럼 프리릴리스 접미사로 옵트인해야 포함.

### 다른 마켓플레이스 플러그인 의존

기본적으로 다른 마켓플레이스의 의존성 자동 설치를 거부(미검토 소스의 묵시적 끌어오기 방지). 허용하려면 루트 마켓플레이스(사용자가 설치하는 플러그인을 호스팅하는 곳)의 `marketplace.json`에 대상 마켓플레이스명을 `allowCrossMarketplaceDependenciesOn`에 추가. 그 마켓플레이스의 allowlist만 참조되므로 신뢰가 중간 마켓플레이스로 체이닝되지 않음.

```json
{
  "name": "acme-tools",
  "owner": { "name": "Acme" },
  "allowCrossMarketplaceDependenciesOn": ["acme-shared"],
  "plugins": [
    {
      "name": "deploy-kit",
      "source": "./deploy-kit",
      "dependencies": [
        { "name": "audit-logger", "marketplace": "acme-shared" }
      ]
    }
  ]
}
```

필드가 없거나 대상을 포함하지 않으면 `cross-marketplace` 에러로 실패. 사용자가 의존성을 먼저 수동 설치하면 allowlist 변경 없이 제약을 만족.

### 릴리스 태깅

버전 제약은 마켓플레이스 저장소의 git 태그에 대해 해석된다. 릴리스는 `{plugin-name}--v{version}` 규칙으로 태그(`{version}`은 그 커밋 `plugin.json`의 `version`과 일치).

```bash
claude plugin tag --push
```

`claude plugin tag`는 매니페스트·마켓플레이스 엔트리에서 태그명을 도출하고, 태그 전 플러그인 내용을 검증하고 `plugin.json`과 마켓플레이스 엔트리의 버전 일치를 확인하고 깨끗한 작업 트리를 요구하며 태그가 이미 있으면 거부. `--dry-run`으로 미리보기. `git tag secrets-vault--v2.1.0` 직접 실행과 동치. `--v` 구분자는 전체 플러그인명에 대한 prefix 매치로 파싱되어 하이픈 포함 이름도 처리.

설치 시 `~2.1.0`이면 `secrets-vault--v` 시작 태그를 필터해 만족하는 최고 버전을 가져옴. 없으면 의존 플러그인이 비활성화되고 가용 버전을 나열하는 에러. 해석된 태그의 semver는 `plugin.json` `version`과 별도 기록(stale 값이어도 실제 가져온 태그로 제약 검사). 태그 해석 설치 캐시 디렉터리명에 12자 커밋 SHA 접미사가 포함되어 태그 force-move 시 새 캐시 디렉터리를 받음.

> `npm` 소스는 태그 기반 해석이 적용 안 됨(제약이 가져올 버전을 통제 못 함). 로드 시 제약은 검사되며 미충족 시 `dependency-version-unsatisfied`로 비활성화.

### 제약 상호작용

여러 플러그인이 같은 의존성을 제약하면 범위를 교집합하고 모두를 만족하는 최고 버전으로 해석.

| A 요구 | B 요구 | 결과 |
|:---|:---|:---|
| `^2.0` | `>=2.1` | `2.1.0` 이상 최고 `2.x` 태그 1개 설치. 둘 다 로드 |
| `~2.1` | `~3.0` | B 설치가 `range-conflict`로 실패. A와 의존성은 그대로 |
| `=2.1.0` | 없음 | 의존성은 `2.1.0` 유지. A 설치 중 신버전 스킵 |

자동 업데이트는 마켓플레이스 최신이 아니라 모든 설치 플러그인 범위를 만족하는 최고 태그를 가져옴. 만족 태그가 없으면 스킵하고 `/doctor`·`/plugin` Errors에 제약 플러그인명과 함께 표시. 마지막 제약 플러그인을 제거하면 의존성은 더 이상 고정되지 않고 다음 업데이트에 마켓플레이스 엔트리 추적 재개.

### 의존성 있는 플러그인 활성/비활성

플러그인 활성화 시 의존성도 같은 스코프에서 함께 활성화되고, 다른 활성 플러그인이 여전히 필요로 하면 비활성화가 차단됨(둘 다 **v2.1.143+**, 이전 버전은 named 플러그인만 처리하고 다음 로드에 `dependency-unsatisfied`).

| 조건 | 결과 |
|:---|:---|
| 의존성 미설치 | 활성화 실패, 누락마다 `claude plugin install` 명령 출력 |
| 조직 정책으로 차단된 의존성 | 실패, 차단된 의존성명 표시 |
| 더 높은 우선순위 스코프에서 `false`인 의존성 | 실패. 그 스코프에서 활성화하거나 `--scope`로 거기에 쓰기 |
| 모두 설치·허용됨 | 성공, 플러그인과 각 의존성에 `true` 기록 |

이는 의존성이 `defaultEnabled: false`여도 동일(Claude Code가 명시적 `true`를 씀). 설치 시도 마찬가지. 비활성화 시 다른 활성 플러그인이 의존하면 거부하고 올바른 순서로 비활성화하는 체인 명령을 제시:

```text
secrets-vault is still required by deploy-kit. Disable that plugin first, or
disable everything together: claude plugin disable deploy-kit@acme-tools && claude plugin disable secrets-vault@acme-tools
```

### 고아 의존성 정리

자동 설치 의존성은 의존 플러그인 제거 후에도 디스크에 남는다. `claude plugin prune`(v2.1.121+)으로 정리:

```bash
claude plugin prune                 # 기본 user 스코프, 확인 프롬프트
claude plugin uninstall deploy-kit --prune   # 제거+정리 한번에
```

`--scope project|local`, `--dry-run`, `-y`(확인 생략, 비TTY 시 필요). 직접 설치한 플러그인은 절대 prune되지 않음(자동설치 의존성만).

### 의존성 에러 해결

| 에러 | 의미 | 해결 |
|:---|:---|:---|
| `dependency-unsatisfied` | 의존성 미설치 또는 비활성 | 에러의 `claude plugin install` 실행. 마켓플레이스 미구성 시 `claude plugin marketplace add`. 비활성이면 활성화 |
| `range-conflict` | 버전 요구 결합 불가(만족 버전 없음/잘못된 semver/너무 복잡) | 충돌 플러그인 제거·업데이트, 잘못된 `version` 수정, 긴 `\|\|` 체인 단순화, 업스트림에 제약 확대 요청 |
| `dependency-version-unsatisfied` | 설치된 의존성이 범위 밖 | `claude plugin install <dependency>@<marketplace>`로 재해석 |
| `no-matching-tag` | `{name}--v*` 태그가 범위를 만족 못 함 | 업스트림 태깅 확인 또는 범위 완화 |

`claude plugin list --json`의 각 플러그인 `errors` 필드로 프로그래밍적 확인 가능.

## CLI 힌트 — 자기 플러그인을 CLI에서 추천

원문: `plugin-hints`

공식 Anthropic 마켓플레이스에 플러그인이 있고 CLI/SDK를 유지보수한다면, CLI가 Claude Code 안에서 실행 중임을 감지해 stderr에 한 줄 마커를 써서 사용자에게 설치를 권할 수 있다. Claude Code가 마커를 읽고 출력에서 제거한 뒤 1회성 설치 프롬프트를 보여준다. 마커는 모델에 도달하기 전 제거되므로 대화에 나타나지 않고 토큰에도 안 잡힌다.

### 동작 방식

Claude Code는 Bash·PowerShell 도구로 실행하는 모든 명령과 [[09 훅]] 명령에 [`CLAUDECODE`](#) 환경변수를 `1`로 설정한다. CLI가 이를 보면 self-closing `<claude-code-hint />` 태그를 stderr에 쓴다. 훅 명령에서는 힌트 태그가 제거·무시되고, Bash·PowerShell 도구 출력만 설치 프롬프트를 트리거한다.

Claude Code는: ① 힌트 줄을 스캔해 모델 도달 전 제거 ② 힌트가 공식 마켓플레이스 플러그인을 가리키는지 확인 ③ 미설치이고 이전에 프롬프트한 적 없는지 확인 ④ 힌트를 낸 명령명을 적어 설치 프롬프트 표시. **자동 설치는 절대 없음 — 항상 사용자 확인.**

### 힌트 방출

`CLAUDECODE` 환경변수로 게이팅(사람 터미널엔 안 나타나게):

```javascript
// Node.js
if (process.env.CLAUDECODE) {
  process.stderr.write(
    '<claude-code-hint v="1" type="plugin" value="example-cli@claude-plugins-official" />\n',
  )
}
```

```python
# Python
import os, sys
if os.environ.get("CLAUDECODE"):
    print('<claude-code-hint v="1" type="plugin" value="example-cli@claude-plugins-official" />', file=sys.stderr)
```

```go
// Go
if os.Getenv("CLAUDECODE") != "" {
    fmt.Fprintln(os.Stderr, `<claude-code-hint v="1" type="plugin" value="example-cli@claude-plugins-official" />`)
}
```

```shell
# Shell
[ -n "$CLAUDECODE" ] &&
  printf '%s\n' '<claude-code-hint v="1" type="plugin" value="example-cli@claude-plugins-official" />' >&2
```

### 방출 위치 선택

Claude Code가 플러그인별로 dedupe하므로 매 호출 방출해도 무해. 잘 맞는 지점:

| 위치 | 이유 |
|:---|:---|
| `--help` 출력 | Claude가 낯선 CLI 탐색 시 자주 help 실행 |
| 알 수 없는 서브커맨드 에러 | Claude가 인터페이스에 혼란스러운 순간 |
| 로그인/인증 성공 | 사용자가 셋업 마인드셋 |
| 첫 실행 환영 메시지 | 자연스러운 온보딩 |

### 사용자가 보는 것 & 빈도

프롬프트는 힌트를 낸 명령명을 표시(도구-플러그인 불일치 식별 가능). 30초 무응답 시 **No**로 닫힘. 빈도 제한: **플러그인당 1회**(프롬프트 후 기록, 답변 무관 다시 안 물음), **세션당 1회**(머신의 모든 CLI 통틀어 세션당 최대 1개). **Yes**는 user 스코프 설치, **No, and don't show...**는 모든 향후 힌트 비활성화.

### 힌트 포맷

```text
<claude-code-hint v="1" type="plugin" value="example-cli@claude-plugins-official" />
```

| 속성 | 필수 | 설명 |
|:---|:---|:---|
| `v` | Yes | 프로토콜 버전. `1`만 지원 |
| `type` | Yes | 힌트 종류. `plugin`만 지원 |
| `value` | Yes | `name@marketplace` 형식 플러그인 식별자 |

값은 큰따옴표 또는 무인용(공백 불가) 가능. 이스케이프 시퀀스 미지원.

### 요구 조건 (강제)

- **자체 줄**: 태그는 자기 줄을 차지해야 함(줄 중간/로그문 안 태그는 무시). 앞뒤 공백은 허용.
- **공식 마켓플레이스**: `value`는 `claude-plugins-official` 같은 Anthropic 통제 마켓플레이스를 가리켜야 함. 다른 마켓플레이스 힌트는 조용히 폐기.

힌트 줄은 버전/타입이 미인식이어도 항상 모델 도달 전 제거(토큰 미계상). 권장(미강제): stderr에 쓰기(셸 파이프라인 보호, 단 Claude Code는 두 스트림 모두 스캔하므로 stdout도 동작), `CLAUDECODE`로 게이팅. 공식 마켓플레이스 등재는 Anthropic 재량(제출 폼은 커뮤니티 마켓플레이스로 가며 힌트 프로토콜은 그걸 검사 안 함). 환경변수 상세는 [[26 변경 이력과 용어집]] 관련.

## 마켓플레이스 생성 및 배포

원문: `plugin-marketplaces`

마켓플레이스는 플러그인을 배포하는 카탈로그다. 중앙 발견·버전 추적·자동 업데이트·다중 소스 타입(git 저장소·로컬 경로 등)을 제공.

흐름: ① 플러그인 생성 → ② `marketplace.json` 작성 → ③ GitHub/GitLab 등 호스팅 → ④ 사용자가 `/plugin marketplace add`로 추가·개별 설치. 이후 저장소에 변경을 푸시해 업데이트하면 사용자는 `/plugin marketplace update`로 갱신.

### 로컬 마켓플레이스 워크스루

```bash
mkdir -p my-marketplace/.claude-plugin
mkdir -p my-marketplace/plugins/quality-review-plugin/.claude-plugin
mkdir -p my-marketplace/plugins/quality-review-plugin/skills/quality-review
```

스킬 `SKILL.md`, 플러그인 매니페스트(`.claude-plugin/plugin.json`), 마켓플레이스 카탈로그(`.claude-plugin/marketplace.json`):

```json
{
  "name": "my-plugins",
  "owner": { "name": "Your Name" },
  "plugins": [
    {
      "name": "quality-review-plugin",
      "source": "./plugins/quality-review-plugin",
      "description": "Adds a quality-review skill for quick code reviews"
    }
  ]
}
```

```shell
/plugin marketplace add ./my-marketplace
/plugin install quality-review-plugin@my-plugins
/quality-review-plugin:quality-review
```

### marketplace.json 스키마

저장소 루트에 `.claude-plugin/marketplace.json` 생성.

```json
{
  "name": "company-tools",
  "owner": { "name": "DevTools Team", "email": "devtools@example.com" },
  "plugins": [
    {
      "name": "code-formatter",
      "source": "./plugins/formatter",
      "description": "Automatic code formatting on save",
      "version": "2.1.0",
      "author": { "name": "DevTools Team" }
    },
    {
      "name": "deployment-tools",
      "source": { "source": "github", "repo": "company/deploy-plugin" },
      "description": "Deployment automation tools"
    }
  ]
}
```

**필수 필드**: `name`(kebab-case, 공백 없음 — 공개, 사용자에게 `@name`으로 보임. 사용자당 이름별 마켓플레이스 1개만 등록, 같은 이름 추가 시 교체), `owner`(객체), `plugins`(배열).

> [!warning] 예약 이름
> 다음은 Anthropic 공식용으로 예약되어 서드파티가 못 씀: `claude-code-marketplace`, `claude-code-plugins`, `claude-plugins-official`, `anthropic-marketplace`, `anthropic-plugins`, `agent-skills`, `anthropic-agent-skills`, `knowledge-work-plugins`, `life-sciences`, `claude-for-legal`, `claude-for-financial-services`, `financial-services-plugins`. `official-claude-plugins`처럼 공식을 사칭하는 이름도 차단.

**owner 필드**: `name`(필수), `email`(선택). **선택 필드**: `$schema`, `description`, `version`, `metadata.pluginRoot`(상대 소스 경로에 prepend하는 기본 디렉터리 — `"./plugins"`면 `"source": "formatter"`로 축약 가능), `allowCrossMarketplaceDependenciesOn`. `description`·`version`은 하위호환으로 `metadata` 아래도 허용.

### 플러그인 엔트리

각 엔트리는 [매니페스트 스키마](#플러그인-매니페스트-스키마-pluginjson)의 모든 필드(`description`, `version`, `author`, `commands`, `hooks` 등) + 마켓플레이스 전용 필드 `source`, `category`, `tags`, `strict`를 포함. **필수**: `name`, `source`. **선택 메타데이터**: `displayName`(v2.1.143+), `description`, `version`, `author`, `homepage`, `repository`, `license`, `keywords`, `category`, `tags`, `strict`(기본 true), `defaultEnabled`(v2.1.154+, `plugin.json`보다 우선). **컴포넌트 설정**: `skills`, `commands`, `agents`, `hooks`, `mcpServers`, `lspServers`.

### 플러그인 소스 (source)

| 소스 | 타입 | 필드 | 비고 |
|:---|:---|:---|:---|
| 상대 경로 | `string` (`"./my-plugin"`) | 없음 | 마켓플레이스 저장소 내 로컬 디렉터리. `./`로 시작, 마켓플레이스 루트 기준 해석(`.claude-plugin/` 기준 아님) |
| `github` | object | `repo`, `ref?`, `sha?` | |
| `url` | object | `url`, `ref?`, `sha?` | Git URL 소스 |
| `git-subdir` | object | `url`, `path`, `ref?`, `sha?` | git 저장소 내 서브디렉터리. sparse 클론으로 모노레포 대역폭 절약 |
| `npm` | object | `package`, `version?`, `registry?` | `npm install`로 설치 |

> **마켓플레이스 소스 vs 플러그인 소스**: 마켓플레이스 소스는 `marketplace.json` 카탈로그 자체를 어디서 가져올지(`/plugin marketplace add`나 `extraKnownMarketplaces`에서 설정, `ref`만 지원, `sha` 미지원). 플러그인 소스는 카탈로그 안 개별 플러그인을 어디서 가져올지(엔트리 `source`, `ref`·`sha` 모두 지원). 둘은 서로 다른 저장소를 가리킬 수 있고 독립적으로 핀.

```json
// github (브랜치/태그/커밋 핀)
{ "name": "github-plugin", "source": { "source": "github", "repo": "owner/plugin-repo", "ref": "v2.0.0", "sha": "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0" } }

// url (Git)
{ "name": "git-plugin", "source": { "source": "url", "url": "https://gitlab.com/team/plugin.git", "ref": "main" } }

// git-subdir (모노레포 내 서브디렉터리, sparse 클론)
{ "name": "my-plugin", "source": { "source": "git-subdir", "url": "https://github.com/acme-corp/monorepo.git", "path": "tools/claude-plugin" } }

// npm (private registry 포함)
{ "name": "my-npm-plugin", "source": { "source": "npm", "package": "@acme/claude-plugin", "version": "^2.0.0", "registry": "https://npm.example.com" } }
```

상대 경로는 사용자가 Git(GitHub/GitLab/git URL)으로 마켓플레이스를 추가할 때만 동작. URL 직접 추가 시 해석 안 됨(GitHub/npm/git URL 소스 사용). `url` 필드는 `.git` 접미사 선택(Azure DevOps·AWS CodeCommit 대응), `git-subdir`의 `url`은 `owner/repo` 약어·SSH URL도 허용. `../`로 마켓플레이스 루트 밖 참조 금지.

### Strict 모드

`strict`는 `plugin.json`이 컴포넌트 정의의 권위인지 제어.

| 값 | 동작 |
|:---|:---|
| `true` (기본) | `plugin.json`이 권위. 마켓플레이스 엔트리가 컴포넌트를 보충하며 둘이 병합 |
| `false` | 마켓플레이스 엔트리가 전체 정의. 플러그인이 컴포넌트를 선언하는 `plugin.json`을 가지면 충돌·로드 실패 |

`strict: true`는 플러그인이 자체 `plugin.json`으로 컴포넌트를 관리하고 엔트리가 추가 보충(대부분의 경우). `strict: false`는 마켓플레이스 운영자가 완전 통제(플러그인이 원시 파일을 제공하고 엔트리가 어떤 파일을 스킬·에이전트·훅으로 노출할지 정의 — 작성자 의도와 다르게 재구성·큐레이션할 때).

### 호스팅 및 배포

- **GitHub (권장)**: 저장소 생성 → `.claude-plugin/marketplace.json` 추가 → `/plugin marketplace add owner/repo`.
- **다른 git**: `/plugin marketplace add https://gitlab.com/company/plugins.git`.
- **로컬 테스트**: `/plugin marketplace add ./my-local-marketplace` 후 `/plugin install test-plugin@my-local-marketplace`.

**비공개 저장소**: 수동 설치·업데이트는 기존 git 자격증명 헬퍼 사용(`gh auth login`, macOS Keychain, `git-credential-store`로 HTTPS, 또는 `known_hosts`·`ssh-agent` 기반 SSH). 백그라운드 자동 업데이트는 시작 시 헬퍼 없이 실행되므로 환경변수 토큰 필요:

| 프로바이더 | 환경변수 | 비고 |
|:---|:---|:---|
| GitHub | `GITHUB_TOKEN` 또는 `GH_TOKEN` | PAT 또는 GitHub App 토큰 |
| GitLab | `GITLAB_TOKEN` 또는 `GL_TOKEN` | PAT 또는 프로젝트 토큰 |
| Bitbucket | `BITBUCKET_TOKEN` | 앱 패스워드 또는 저장소 접근 토큰 |

```bash
export GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxx
```

[[16 CI-CD와 팀 통합]] 환경에선 시크릿 환경변수로 설정(GitHub Actions는 동일 조직 저장소에 `GITHUB_TOKEN` 자동 제공).

### 팀에 마켓플레이스 강제 (extraKnownMarketplaces / enabledPlugins)

`.claude/settings.json`에 마켓플레이스를 추가하면 팀원이 프로젝트 폴더 신뢰 시 설치 유도:

```json
{
  "extraKnownMarketplaces": {
    "company-tools": { "source": { "source": "github", "repo": "your-org/claude-plugins" } }
  },
  "enabledPlugins": {
    "code-formatter@company-tools": true,
    "deployment-tools@company-tools": true
  }
}
```

마켓플레이스 상태는 `~/.claude/plugins/known_marketplaces.json`에 사용자당 한 번 저장(프로젝트별 아님). worktree에서 실행해도 로컬 상대경로 소스는 메인 체크아웃을 가리켜 모든 worktree가 동일 위치 공유. 전체 옵션은 [[04 설정]].

### 컨테이너용 플러그인 사전 채우기 (CLAUDE_CODE_PLUGIN_SEED_DIR)

컨테이너 이미지·CI에서 빌드 타임에 플러그인 디렉터리를 사전 채워, 런타임 클론 없이 시작할 수 있다. `CLAUDE_CODE_PLUGIN_SEED_DIR`를 그 디렉터리로 지정. 여러 시드는 Unix `:`/Windows `;`로 구분(순서대로 검색, 첫 매치 우선). 시드 구조는 `~/.claude/plugins`를 미러:

```text
$CLAUDE_CODE_PLUGIN_SEED_DIR/
  known_marketplaces.json
  marketplaces/<name>/...
  cache/<marketplace>/<plugin>/<version>/...
```

빌드 시 Claude Code를 한번 실행해 플러그인 설치 후 `~/.claude/plugins`를 이미지로 복사하거나, 복사 생략하려면 빌드 동안 `CLAUDE_CODE_PLUGIN_CACHE_DIR`를 타깃 시드로 지정해 거기에 직접 설치:

```bash
CLAUDE_CODE_PLUGIN_CACHE_DIR=/opt/claude-seed claude plugin marketplace add your-org/plugins
CLAUDE_CODE_PLUGIN_CACHE_DIR=/opt/claude-seed claude plugin install my-tool@your-plugins
# 런타임:
export CLAUDE_CODE_PLUGIN_SEED_DIR=/opt/claude-seed
```

동작: **읽기 전용**(시드는 쓰여지지 않음, 시드 마켓플레이스 자동 업데이트 비활성), **시드 우선**(시작마다 사용자 설정의 매칭 엔트리를 덮어씀 — 옵트아웃은 `/plugin disable`), **경로 해석**(JSON 내 경로가 아니라 런타임에 `marketplaces/<name>/`를 probe하므로 다른 경로에 마운트돼도 동작), **변경 차단**(시드 관리 마켓플레이스에 `remove`·`update` 실패), **설정과 합성**(`extraKnownMarketplaces`/`enabledPlugins`가 시드에 있는 마켓플레이스를 선언하면 시드 사본 사용). 인터랙티브·`-p` 비인터랙티브 모두 동작. 관련 [[13 자동화와 스케줄링]]·[[17 클라우드 프로바이더]].

### 관리형 마켓플레이스 제한 (strictKnownMarketplaces)

조직은 관리형 설정의 `strictKnownMarketplaces`로 사용자가 추가 가능한 마켓플레이스를 통제.

| 값 | 동작 |
|:---|:---|
| undefined (기본) | 제한 없음 |
| 빈 배열 `[]` | 완전 잠금. 새 마켓플레이스 추가 불가 |
| 소스 목록 | allowlist와 정확히 일치하는 것만 추가 가능 |

```json
// 모든 추가 비활성
{ "strictKnownMarketplaces": [] }

// 특정 마켓플레이스만 허용
{ "strictKnownMarketplaces": [
    { "source": "github", "repo": "acme-corp/approved-plugins" },
    { "source": "github", "repo": "acme-corp/security-tools", "ref": "v2.0" },
    { "source": "url", "url": "https://plugins.example.com/marketplace.json" }
]}

// 내부 git 서버 전체 허용 (정규식 host 매칭 — GHES·셀프호스팅 GitLab 권장)
{ "strictKnownMarketplaces": [
    { "source": "hostPattern", "hostPattern": "^github\\.example\\.com$" }
]}

// 특정 디렉터리의 파일시스템 마켓플레이스 허용
{ "strictKnownMarketplaces": [
    { "source": "pathPattern", "pathPattern": "^/opt/approved/" }
]}
```

`pathPattern`에 `".*"`로 모든 경로 허용하며 네트워크는 `hostPattern`으로 통제 가능. 제한은 추가만 통제하고 등록은 안 하므로 자동 가용화는 `extraKnownMarketplaces`와 함께 사용. 검사는 네트워크/파일시스템 작업 전, add·install·update·refresh·auto-update에 실행. 정책 설정 전 추가됐고 더 이상 매칭 안 되는 마켓플레이스는 설치·업데이트 거부(`blockedMarketplaces`도 동일). 대부분 소스는 정확 매칭(GitHub은 `repo` 필수, `ref`/`path` 명시 시 일치 필요. URL은 전체 일치 — trailing slash·`.git`·`ssh://`vs`https://`를 다른 값으로 취급하므로 다중 URL 형태면 `hostPattern` 권장). 관리형 설정이라 사용자·프로젝트가 못 덮어씀. 상세 [[18 보안과 샌드박스]]·[[04 설정]].

### 버전 해석 및 릴리스 채널

버전 해석 순서: ① `plugin.json` `version` ② 마켓플레이스 엔트리 `version` ③ git 커밋 SHA. git 소스(`github`/`url`/`git-subdir`/git 마켓플레이스 상대경로)는 `version`을 완전히 생략하면 새 커밋마다 새 버전(내부·활발한 개발에 가장 간단).

> [!warning]
> `version` 설정 시 고정됨 — 커밋만 푸시하면 기존 사용자에 무효. `plugin.json`과 마켓플레이스 엔트리 양쪽에 `version` 설정 회피(`plugin.json`이 조용히 우선해 stale 매니페스트가 엔트리 버전을 가릴 수 있음).

**릴리스 채널**: 같은 repo의 다른 ref/SHA를 가리키는 두 마켓플레이스(stable/latest)를 만들어 관리형 설정으로 사용자 그룹별 할당.

```json
{ "name": "stable-tools", "plugins": [{ "name": "code-formatter", "source": { "source": "github", "repo": "acme-corp/code-formatter", "ref": "stable" } }] }
{ "name": "latest-tools", "plugins": [{ "name": "code-formatter", "source": { "source": "github", "repo": "acme-corp/code-formatter", "ref": "latest" } }] }
```

각 채널은 다른 버전으로 해석되어야 함(명시 버전이면 각 핀 ref의 `plugin.json`이 다른 `version` 선언, 생략 시 커밋 SHA가 구분). 같은 버전 문자열이면 동일로 취급해 업데이트 스킵. 그룹별로 `extraKnownMarketplaces`에 `stable-tools` 또는 `latest-tools` 할당.

**의존성 버전 핀**: [의존성 버전 제약](#플러그인-의존성-버전-제약) 절의 `{plugin-name}--v{version}` 태그 규칙 참고.

```bash
claude plugin validate .   # 또는 /plugin validate .
/plugin marketplace add ./path/to/marketplace
/plugin install test-plugin@marketplace-name
```

### CLI로 마켓플레이스 관리

```bash
claude plugin marketplace add <source> [--scope user|project|local] [--sparse <paths...>]
claude plugin marketplace list [--json]
claude plugin marketplace remove <name> [--scope ...]   # 별칭 rm
claude plugin marketplace update [name]
```

`add`의 `<source>`는 `owner/repo` 약어, git URL, 원격 URL, 로컬 경로. 브랜치/태그 핀은 GitHub 약어에 `@ref`, git URL에 `#ref`:

```bash
claude plugin marketplace add acme-corp/claude-plugins
claude plugin marketplace add acme-corp/claude-plugins@v2.0
claude plugin marketplace add https://gitlab.example.com/team/plugins.git
claude plugin marketplace add https://example.com/marketplace.json
claude plugin marketplace add ./my-marketplace
claude plugin marketplace add acme-corp/claude-plugins --scope project
claude plugin marketplace add acme-corp/monorepo --sparse .claude-plugin plugins   # 모노레포 sparse 체크아웃
```

`remove`의 `<name>`은 `list`가 보여주는 `marketplace.json`의 `name`(add에 넘긴 소스 아님). `--scope` 생략 시 모든 editable 스코프에서 제거.

> [!warning]
> 마지막 스코프에서 마켓플레이스 제거 시 거기서 설치한 플러그인도 제거됨. 플러그인 유지하며 갱신하려면 `update` 사용.

`remove`·`update`는 시드 관리(읽기 전용) 마켓플레이스에 실패(전체 업데이트 시 시드는 스킵하고 나머지는 갱신).

### 트러블슈팅 (마켓플레이스)

**검증 에러** (`claude plugin validate .`):

| 에러 | 원인 | 해결 |
|:---|:---|:---|
| `File not found: .claude-plugin/marketplace.json` | 매니페스트 누락 | 필수 필드로 생성 |
| `Invalid JSON syntax: ...` | JSON 구문 오류 | 쉼표·따옴표 확인 |
| `Duplicate plugin name "x" found in marketplace` | 이름 중복 | 고유 `name` 부여 |
| `plugins[0].source: Path contains ".."` | `..` 포함 경로 | `..` 없는 마켓플레이스 루트 상대경로 |
| `YAML frontmatter failed to parse: ...` | 스킬·에이전트·command frontmatter YAML 오류 | YAML 수정(플러그인 디렉터리 검증 시만 보고) |
| `Invalid JSON syntax: ...` (hooks.json) | `hooks/hooks.json` malformed | JSON 수정(malformed면 전체 플러그인 로드 실패) |

경고(비차단): plugins 비었음, description 없음, `Plugin name "x" is not kebab-case`(Claude Code는 허용하나 Claude.ai 마켓플레이스 sync는 거부). 마켓플레이스 디렉터리 검증은 `marketplace.json`만 검사(스키마·중복 이름·소스 경로 탐색·`plugin.json` 버전 불일치). 개별 플러그인 frontmatter·hooks 검사는 플러그인 디렉터리에 대해 실행(`claude plugin validate ./plugins/my-plugin`).

**기타 이슈**:
- **상대 경로 URL 마켓플레이스 실패**: URL 마켓플레이스는 `marketplace.json`만 다운로드(플러그인 파일 미다운로드). 상대 경로 소스를 github/npm/git URL로 바꾸거나 Git 기반 마켓플레이스 사용.
- **오프라인 업데이트 실패**: `git pull` 실패 시 기본적으로 stale 클론을 지우고 재클론 시도(오프라인이면 빈 디렉터리). `export CLAUDE_CODE_PLUGIN_KEEP_MARKETPLACE_ON_FAILURE=1`로 실패 시 기존 캐시 유지. 완전 오프라인은 `CLAUDE_CODE_PLUGIN_SEED_DIR` 사용.
- **Git 타임아웃**: 모든 git 작업은 120초 타임아웃. `export CLAUDE_CODE_PLUGIN_GIT_TIMEOUT_MS=300000`(ms)로 증가.
- **비공개 인증 실패**: 수동은 `gh auth status`·credential helper 확인, 자동 업데이트는 토큰(`GITHUB_TOKEN`은 private에 `repo` 스코프, GitLab은 `read_repository`) 확인.
- **설치 후 파일 못 찾음**: 캐시 복사 때문(`../shared-utils` 등 디렉터리 밖 참조 불가) — [플러그인 캐싱](#플러그인-캐싱-및-파일-해석)의 심링크 우회.
