# 07 스킬 (Skills)

스킬(Skill)은 Claude의 능력을 확장하는 가장 핵심적인 메커니즘이다. `SKILL.md` 파일에 지침을 작성하면 Claude가 이를 도구 모음에 추가하고, 관련 상황에서 **자동으로** 불러오거나 사용자가 `/skill-name`으로 **직접** 호출할 수 있다.

스킬을 만들어야 할 신호:
- 같은 지침·체크리스트·다단계 절차를 채팅에 반복해서 붙여넣고 있을 때
- CLAUDE.md의 한 섹션이 "사실"이 아니라 "절차"로 비대해졌을 때

> [!tip] 핵심 트레이드오프
> CLAUDE.md 내용과 달리 스킬 본문은 **사용될 때만 로드**되므로, 긴 레퍼런스 자료라도 필요해지기 전까지는 토큰 비용이 거의 0이다. 이것이 스킬을 CLAUDE.md보다 선호하는 가장 큰 이유다.

스킬은 [Agent Skills](https://agentskills.io) 오픈 표준을 따르며 여러 AI 도구에서 동작한다. Claude Code는 여기에 호출 제어, 서브에이전트 실행([[08 서브에이전트와 에이전트 팀]]), 동적 컨텍스트 주입 같은 기능을 더했다.

> [!note] 커스텀 명령어는 스킬로 통합됨
> `.claude/commands/deploy.md` 파일과 `.claude/skills/deploy/SKILL.md` 스킬은 둘 다 `/deploy`를 만들고 동일하게 동작한다. 기존 `.claude/commands/` 파일은 계속 작동한다. 스킬은 추가 기능(보조 파일 디렉터리, 호출 제어 frontmatter, 자동 로드)을 제공한다.

`/help`, `/compact` 같은 내장 명령어는 [[02 CLI 레퍼런스]]를 참고하라.

## 번들 스킬 (Bundled skills)

Claude Code는 모든 세션에서 사용 가능한 번들 스킬 세트를 포함한다: `/code-review`, `/batch`, `/debug`, `/loop`, `/claude-api` 등. 고정 로직을 직접 실행하는 대부분의 내장 명령어와 달리, **번들 스킬은 프롬프트 기반**이다 — Claude에게 상세 지침을 주고 도구로 작업을 오케스트레이션하게 한다. `/` 다음에 스킬 이름을 입력해 호출한다. 번들 스킬은 [[02 CLI 레퍼런스]]의 명령어 목록에서 Purpose 열에 **Skill**로 표시된다.

### 앱 실행 및 검증 (Run and verify)

세 개의 번들 스킬이 함께 작동해 테스트만이 아니라 실행 중인 앱에 대해 변경을 확인하게 해준다.

| Skill | Purpose |
| :--- | :--- |
| `/run` | 앱을 실행·구동하여 변경이 동작하는지 확인 |
| `/verify` | 앱을 빌드·실행하여 코드 변경이 의도대로 동작하는지 확인 (테스트·타입체크로 우회하지 않음) |
| `/run-skill-generator` | `/run`과 `/verify`에게 프로젝트 빌드·실행 방법을 학습시킴 |

세 스킬 모두 **Claude Code v2.1.145 이상** 필요.

`/run`과 `/verify`는 설정 없이 동작한다 — 프로젝트 타입(CLI, 서버, TUI, 브라우저 구동형)과 README, `package.json`, `Makefile` 내용에서 실행 방법을 추론한다. 다만 데이터베이스·env 파일·그래픽 세션·다단계 빌드 등 표준 실행을 넘어서는 프로젝트에서는 이 추론이 불안정해진다.

`/run-skill-generator`는 레시피를 기록한다 — 깨끗한 환경에서 앱을 실행시키고, 무엇이 통했는지(설치 명령, env 변수, 실행 스크립트)를 캡처한 뒤 `.claude/skills/run-<name>/`에 프로젝트별 스킬로 커밋한다. 이후 `/run`, `/verify` 및 레포의 다른 에이전트가 매번 재발견하는 대신 기록된 레시피를 따른다. 프로젝트당 한 번 실행하고, 빌드·실행 과정이 바뀌면 다시 실행하면 된다.

## 시작하기 (Getting started)

### 첫 스킬 만들기

다음 예시는 git 저장소의 커밋되지 않은 변경을 요약하고 위험한 부분을 표시하는 스킬이다. 라이브 diff를 프롬프트에 끌어와 응답을 실제 워킹 트리에 근거하게 만든다.

1. **스킬 디렉터리 생성** — 개인 스킬은 모든 프로젝트에서 사용 가능하다.

```bash
mkdir -p ~/.claude/skills/summarize-changes
```

2. **SKILL.md 작성** — 모든 스킬은 두 부분으로 구성된다: `---` 마커 사이의 YAML frontmatter(Claude가 언제 쓸지 알려줌)와 markdown 본문(스킬 실행 시 따르는 지침). 디렉터리 이름이 입력하는 명령어가 되고, `description`이 자동 로드 판단을 돕는다.

`~/.claude/skills/summarize-changes/SKILL.md`:

```yaml
---
description: Summarizes uncommitted changes and flags anything risky. Use when the user asks what changed, wants a commit message, or asks to review their diff.
---

## Current changes

!`git diff HEAD`

## Instructions

Summarize the changes above in two or three bullet points, then list any risks you notice such as missing error handling, hardcoded values, or tests that need updating. If the diff is empty, say there are no uncommitted changes.
```

`` !`git diff HEAD` `` 라인은 **동적 컨텍스트 주입**이다 — Claude Code가 명령을 실행해 그 출력으로 라인을 치환한 뒤 Claude가 스킬 내용을 보므로, 지침이 현재 diff와 함께 인라인되어 도착한다.

3. **테스트** — git 프로젝트를 열고 작은 편집을 한 뒤 `claude` 실행. 두 가지 방법:

```text
What did I change?
```
(description에 맞으면 Claude가 자동 호출)

```text
/summarize-changes
```
(직접 호출)

### 스킬이 저장되는 위치

저장 위치가 누가 사용할 수 있는지를 결정한다.

| Location | Path | Applies to |
| :--- | :--- | :--- |
| Enterprise | [managed settings](https://code.claude.com/docs/en/settings) 참조 | 조직 내 모든 사용자 |
| Personal | `~/.claude/skills/<skill-name>/SKILL.md` | 내 모든 프로젝트 |
| Project | `.claude/skills/<skill-name>/SKILL.md` | 이 프로젝트만 |
| Plugin | `<plugin>/skills/<skill-name>/SKILL.md` | 플러그인 활성 위치 |

**우선순위(이름 충돌 시):** enterprise > personal > project. 플러그인 스킬은 `plugin-name:skill-name` 네임스페이스를 쓰므로 다른 레벨과 충돌하지 않는다. `.claude/commands/` 파일도 같은 방식으로 동작하되, 스킬과 명령어가 같은 이름이면 **스킬이 우선**한다.

> [!note] 스킬 폴더를 플러그인으로
> 스킬 폴더에 `.claude-plugin/plugin.json`을 추가하면 `<name>@skills-dir`라는 [[11 플러그인]]으로 로드되어 에이전트·훅·MCP 서버를 번들할 수 있다. 프로젝트의 `.claude/skills/`에서는 워크스페이스 신뢰 다이얼로그 수락이 먼저 필요하다.

#### 라이브 변경 감지 (Live change detection)

Claude Code는 스킬 디렉터리의 파일 변경을 감시한다. `~/.claude/skills/`, 프로젝트 `.claude/skills/`, `--add-dir` 디렉터리 내의 `.claude/skills/`에서 스킬을 **추가·편집·삭제하면 재시작 없이 현재 세션 내에서 적용**된다. 단, 세션 시작 시 존재하지 않던 최상위 skills 디렉터리를 새로 만들면 그 디렉터리를 감시하려면 Claude Code 재시작이 필요하다.

> [!note]
> 라이브 변경 감지는 `SKILL.md` 텍스트만 다룬다. 스킬 폴더가 [[11 플러그인]]이기도 하면 `hooks/`, `.mcp.json`, `agents/`, `output-styles/` 변경은 `/reload-plugins`가 필요하다.

#### 상위·중첩 디렉터리에서 자동 발견

프로젝트 스킬은 시작 디렉터리와 레포 루트까지의 모든 상위 디렉터리의 `.claude/skills/`에서 로드된다 — 하위 디렉터리에서 Claude를 시작해도 루트의 스킬을 가져온다. 시작 디렉터리 아래의 하위 디렉터리 파일을 다룰 때는 중첩된 `.claude/skills/`도 필요 시 발견한다. 예: `packages/frontend/` 파일 편집 시 `packages/frontend/.claude/skills/`도 찾는다. 모노레포 구성을 지원하는 동작이다.

각 스킬은 `SKILL.md`를 진입점으로 하는 디렉터리다.

```text
my-skill/
├── SKILL.md           # Main instructions (required)
├── template.md        # Template for Claude to fill in
├── examples/
│   └── sample.md      # Example output showing expected format
└── scripts/
    └── validate.sh    # Script Claude can execute
```

`SKILL.md`는 필수이며 나머지 파일은 선택이다 — 템플릿, 예시 출력, Claude가 실행할 스크립트, 상세 레퍼런스 문서 등. 보조 파일은 `SKILL.md`에서 참조해 Claude가 무엇이 들었고 언제 로드할지 알게 하라.

#### 추가 디렉터리(`--add-dir`)에서의 스킬

`--add-dir` 플래그와 `/add-dir` 명령은 설정 발견이 아니라 **파일 접근만** 부여하지만, 스킬은 예외다 — 추가된 디렉터리 내의 `.claude/skills/`는 자동 로드된다. 이 예외는 `--add-dir`/`/add-dir`에만 적용된다. `settings.json`의 `permissions.additionalDirectories` 설정은 파일 접근만 부여하고 스킬을 로드하지 않는다. 서브에이전트·명령어·출력 스타일 등 다른 `.claude/` 설정은 추가 디렉터리에서 로드되지 않는다. ([[05 권한]] 참고)

> [!note]
> `--add-dir` 디렉터리의 CLAUDE.md는 기본적으로 로드되지 않는다. 로드하려면 `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1`을 설정한다. ([[03 메모리와 컨텍스트]])

## 스킬 구성 (Configure skills)

스킬은 `SKILL.md` 상단의 YAML frontmatter와 그 뒤의 markdown 본문으로 구성한다.

### 스킬 콘텐츠의 종류

**Reference content** — Claude가 현재 작업에 적용할 지식을 더한다(규칙, 패턴, 스타일 가이드, 도메인 지식). 인라인으로 실행되어 대화 컨텍스트와 함께 사용된다.

```yaml
---
name: api-conventions
description: API design patterns for this codebase
---

When writing API endpoints:
- Use RESTful naming conventions
- Return consistent error formats
- Include request validation
```

**Task content** — 배포·커밋·코드 생성 같은 특정 동작의 단계별 지침을 준다. 보통 Claude가 알아서 실행하기보다 `/skill-name`으로 직접 호출하고 싶은 것들이다. `disable-model-invocation: true`를 추가해 Claude의 자동 트리거를 막는다.

```yaml
---
name: deploy
description: Deploy the application to production
context: fork
disable-model-invocation: true
---

Deploy the application:
1. Run the test suite
2. Build the application
3. Push to the deployment target
```

> [!tip] 본문은 간결하게
> 스킬이 한 번 로드되면 그 내용은 **턴을 넘어 컨텍스트에 머문다**(아래 생명주기 참고). 모든 줄이 반복 토큰 비용이므로 "어떻게/왜"를 서술하기보다 "무엇을 할지"를 진술하라. [[20 베스트 프랙티스와 워크플로우]]의 CLAUDE.md 간결성 기준을 동일하게 적용.

### Frontmatter 레퍼런스

모든 필드는 선택이며, Claude가 언제 쓸지 알도록 `description`만 권장된다.

```yaml
---
name: my-skill
description: What this skill does
disable-model-invocation: true
allowed-tools: Read Grep
---

Your skill instructions here...
```

| Field | Required | Description |
| :--- | :--- | :--- |
| `name` | No | 스킬 목록에 표시되는 디스플레이 이름. 기본값은 디렉터리 이름. 호출 시 입력하는 이름과 다를 수 있음(아래 참고) |
| `description` | Recommended | 스킬이 하는 일과 언제 쓸지. 자동 로드 판단에 사용. 생략 시 본문 첫 문단 사용. **핵심 use case를 맨 앞에** — `description`+`when_to_use` 합산 텍스트가 목록에서 **1,536자**에서 잘림 |
| `when_to_use` | No | 언제 호출할지 추가 컨텍스트(트리거 문구, 예시 요청). `description`에 덧붙고 1,536자 캡에 포함 |
| `argument-hint` | No | 자동완성 시 보이는 인자 힌트. 예: `[issue-number]`, `[filename] [format]` |
| `arguments` | No | `$name` 치환용 명명된 위치 인자. 공백 구분 문자열 또는 YAML 리스트. 이름이 순서대로 위치에 매핑 |
| `disable-model-invocation` | No | `true`면 Claude의 자동 로드 차단. 수동(`/name`) 트리거 워크플로우용. 서브에이전트 프리로드도 막음. 기본 `false` |
| `user-invocable` | No | `false`면 `/` 메뉴에서 숨김. 사용자가 직접 호출할 게 아닌 배경 지식용. 기본 `true` |
| `allowed-tools` | No | 스킬 활성 시 허가 없이 쓸 수 있는 도구. 공백/쉼표 구분 문자열 또는 YAML 리스트 |
| `disallowed-tools` | No | 스킬 활성 동안 도구 풀에서 제거할 도구. 자율 스킬이 절대 호출하면 안 되는 도구(예: 백그라운드 루프의 `AskUserQuestion`)용. 다음 메시지 전송 시 제한 해제 |
| `model` | No | 스킬 활성 시 사용할 모델. 남은 턴 동안 적용되고 설정에 저장 안 됨. `/model` 값 또는 `inherit` |
| `effort` | No | 스킬 활성 시 effort 레벨. 세션 effort 재정의. `low`, `medium`, `high`, `xhigh`, `max`(모델별 가용) |
| `context` | No | `fork`로 설정 시 포크된 서브에이전트 컨텍스트에서 실행 |
| `agent` | No | `context: fork` 시 사용할 서브에이전트 타입 |
| `hooks` | No | 이 스킬 생명주기에 스코프된 훅. ([[09 훅]]) |
| `paths` | No | 스킬 활성화를 제한하는 glob 패턴. 설정 시 해당 패턴 파일 작업 시에만 자동 로드. 쉼표 구분 문자열 또는 YAML 리스트. path-specific rules와 동일 형식 ([[03 메모리와 컨텍스트]]) |
| `shell` | No | `` !`command` ``와 ` ```! ` 블록에 쓸 셸. `bash`(기본) 또는 `powershell`. `powershell`은 Windows에서 PowerShell로 실행, `CLAUDE_CODE_USE_POWERSHELL_TOOL=1` 필요 |

#### 스킬의 명령어 이름은 어떻게 정해지나

호출 시 입력하는 명령어는 **스킬 파일이 사는 위치**에서 온다. frontmatter `name` 필드는 스킬 목록에 표시되는 라벨을 설정하며, 플러그인 루트 `SKILL.md`를 제외하면 `/` 뒤에 입력하는 것을 바꾸지 않는다.

| Skill location | Command name source | Example |
| :--- | :--- | :--- |
| `~/.claude/skills/` 또는 `.claude/skills/` 하위 디렉터리 | 디렉터리 이름 | `.claude/skills/deploy-staging/SKILL.md` → `/deploy-staging` |
| `.claude/commands/` 하위 파일 | 확장자 제외 파일명 | `.claude/commands/deploy.md` → `/deploy` |
| 플러그인 `skills/` 하위 디렉터리 | 디렉터리 이름, 플러그인 네임스페이스 | `my-plugin/skills/review/SKILL.md` → `/my-plugin:review` |
| 플러그인 루트 `SKILL.md` | frontmatter `name`, 폴백은 플러그인 디렉터리 이름 | `my-plugin/SKILL.md` (`name: review`) → `/my-plugin:review` |

플러그인 루트 케이스가 `name`이 명령어 이름을 정하는 유일한 곳이다(가져올 스킬 디렉터리가 없기 때문). `name` 미설정 시 플러그인 디렉터리 이름 사용. ([[11 플러그인]])

#### 사용 가능한 문자열 치환 (String substitutions)

| Variable | Description |
| :--- | :--- |
| `$ARGUMENTS` | 호출 시 전달된 모든 인자. 본문에 없으면 `ARGUMENTS: <value>`로 끝에 덧붙음 |
| `$ARGUMENTS[N]` | 0-기반 인덱스로 특정 인자 접근. 예: `$ARGUMENTS[0]`은 첫 인자 |
| `$N` | `$ARGUMENTS[N]`의 약칭. `$0`은 첫 인자, `$1`은 둘째 |
| `$name` | `arguments` frontmatter 리스트의 명명 인자. 순서대로 매핑(`arguments: [issue, branch]`면 `$issue`=첫째, `$branch`=둘째) |
| `${CLAUDE_SESSION_ID}` | 현재 세션 ID. 로깅·세션별 파일 생성·출력 상관에 유용 |
| `${CLAUDE_EFFORT}` | 현재 effort 레벨: `low`/`medium`/`high`/`xhigh`/`max`. Ultracode는 별도 레벨이 아니라 `xhigh`로 보고됨 |
| `${CLAUDE_SKILL_DIR}` | 스킬의 `SKILL.md`가 있는 디렉터리. 플러그인 스킬은 플러그인 루트가 아닌 스킬 하위 디렉터리. 번들 스크립트/파일을 cwd와 무관하게 참조할 때 bash 주입에 사용 |

인덱스 인자는 셸 스타일 인용을 쓰므로 여러 단어 값은 따옴표로 감싼다. 예: `/my-skill "hello world" second` → `$0`=`hello world`, `$1`=`second`. `$ARGUMENTS`는 항상 입력한 전체 인자 문자열로 확장.

리터럴 `$`를 숫자·`ARGUMENTS`·선언된 인자명 앞에 두려면(예: 산문 내 `$1.00`) 백슬래시로 이스케이프: `\$1.00`. 다른 `$` 앞의 백슬래시는 그대로 유지. 토큰 바로 앞 단일 백슬래시만 이스케이프하며, `\\$1` 같은 이중 백슬래시는 둘 다 유지하고 치환은 그대로 진행.

치환 예시:

```yaml
---
name: session-logger
description: Log activity for this session
---

Log the following to logs/${CLAUDE_SESSION_ID}.log:

$ARGUMENTS
```

### 보조 파일 추가 (Add supporting files)

스킬 디렉터리에 여러 파일을 둘 수 있다. `SKILL.md`는 핵심에 집중시키고, 상세 레퍼런스는 필요할 때만 Claude가 접근하게 한다. 대형 레퍼런스·API 스펙·예시 모음은 매번 컨텍스트에 로드될 필요가 없다.

```text
my-skill/
├── SKILL.md (required - overview and navigation)
├── reference.md (detailed API docs - loaded when needed)
├── examples.md (usage examples - loaded when needed)
└── scripts/
    └── helper.py (utility script - executed, not loaded)
```

보조 파일은 `SKILL.md`에서 참조하라:

```markdown
## Additional resources

- For complete API details, see [reference.md](reference.md)
- For usage examples, see [examples.md](examples.md)
```

> [!tip] `SKILL.md`는 500줄 이하로 유지. 상세 레퍼런스는 별도 파일로 옮겨라.

### 누가 스킬을 호출하는지 제어

기본적으로 사용자와 Claude 모두 어떤 스킬이든 호출할 수 있다. 두 frontmatter 필드로 제한:

- **`disable-model-invocation: true`** — 사용자만 호출 가능. 부수효과가 있거나 타이밍을 통제하고 싶은 워크플로우(`/commit`, `/deploy`, `/send-slack-message`)용. 코드가 준비돼 보인다고 Claude가 멋대로 배포하면 안 된다.
- **`user-invocable: false`** — Claude만 호출 가능. 명령으로서 실행 가능하지 않은 배경 지식용. 예: `legacy-system-context` 스킬은 낡은 시스템 동작을 설명하지만 `/legacy-system-context`는 사용자가 취할 의미 있는 동작이 아니다.

| Frontmatter | You can invoke | Claude can invoke | When loaded into context |
| :--- | :--- | :--- | :--- |
| (default) | Yes | Yes | description은 항상 컨텍스트에, 전체 스킬은 호출 시 로드 |
| `disable-model-invocation: true` | Yes | No | description 컨텍스트에 없음, 전체 스킬은 사용자 호출 시 로드 |
| `user-invocable: false` | No | Yes | description은 항상 컨텍스트에, 전체 스킬은 호출 시 로드 |

> [!note]
> 일반 세션에서 스킬 description은 컨텍스트에 로드되어 Claude가 무엇이 있는지 알지만, 전체 내용은 호출 시에만 로드된다. [[08 서브에이전트와 에이전트 팀]]에 스킬을 프리로드하면 다르게 동작한다 — 전체 스킬 내용이 시작 시 주입된다.

### 스킬 콘텐츠 생명주기 (Skill content lifecycle)

사용자나 Claude가 스킬을 호출하면, 렌더링된 `SKILL.md` 내용이 단일 메시지로 대화에 들어가 **세션 나머지 동안 머문다**. Claude Code는 이후 턴에 스킬 파일을 다시 읽지 않으므로, 작업 전반에 적용돼야 할 지침은 일회성 단계가 아니라 **상시 지침**으로 작성하라.

**자동 압축(auto-compaction)** 시: 대화가 요약되어 컨텍스트를 비울 때, Claude Code는 각 스킬의 가장 최근 호출을 요약 뒤에 다시 붙이고 각각의 첫 **5,000 토큰**을 유지한다. 재첨부 스킬들은 합산 **25,000 토큰** 예산을 공유한다. 가장 최근 호출 스킬부터 채우므로, 한 세션에서 여러 스킬을 호출했다면 오래된 스킬은 압축 후 완전히 드롭될 수 있다. ([[03 메모리와 컨텍스트]])

> [!warning] 스킬이 첫 응답 뒤 영향력을 잃은 것처럼 보이면
> 보통 내용은 여전히 존재하고 모델이 다른 도구/접근을 택하는 것이다. `description`과 지침을 강화해 모델이 계속 선호하게 하거나, [[09 훅]]으로 결정론적으로 동작을 강제하라. 스킬이 크거나 이후 여러 개를 호출했다면 압축 후 재호출해 전체 내용을 복원하라.

### 스킬용 도구 사전 승인 (Pre-approve tools)

`allowed-tools` 필드는 스킬 활성 동안 나열된 도구에 권한을 부여해 Claude가 승인 프롬프트 없이 사용하게 한다. 도구 가용성을 **제한하지는 않는다** — 모든 도구는 여전히 호출 가능하고, 나열되지 않은 도구는 기존 [[05 권한]] 설정이 다스린다.

프로젝트의 `.claude/skills/`에 체크인된 스킬의 경우 `allowed-tools`는 해당 폴더의 워크스페이스 신뢰 다이얼로그를 수락한 후 적용된다(`.claude/settings.json`의 권한 규칙과 동일). 스킬이 스스로에게 광범위한 도구 접근을 부여할 수 있으므로 레포를 신뢰하기 전 프로젝트 스킬을 검토하라.

```yaml
---
name: commit
description: Stage and commit the current changes
disable-model-invocation: true
allowed-tools: Bash(git add *) Bash(git commit *) Bash(git status *)
---
```

스킬이 특정 도구를 쓰지 못하게 막으려면 [[05 권한]] 설정에 deny 규칙을 추가하라.

### 스킬에 인자 전달 (Pass arguments)

사용자와 Claude 모두 호출 시 인자를 전달할 수 있다. 인자는 `$ARGUMENTS` 플레이스홀더로 사용 가능하다.

```yaml
---
name: fix-issue
description: Fix a GitHub issue
disable-model-invocation: true
---

Fix GitHub issue $ARGUMENTS following our coding standards.

1. Read the issue description
2. Understand the requirements
3. Implement the fix
4. Write tests
5. Create a commit
```

`/fix-issue 123` 실행 시 Claude는 "Fix GitHub issue 123 following our coding standards..."를 받는다. 스킬에 `$ARGUMENTS`가 없는데 인자를 주면 Claude Code가 스킬 내용 끝에 `ARGUMENTS: <input>`을 덧붙인다.

위치별 개별 인자는 `$ARGUMENTS[N]` 또는 약칭 `$N`:

```yaml
---
name: migrate-component
description: Migrate a component from one framework to another
---

Migrate the $0 component from $1 to $2.
Preserve all existing behavior and tests.
```

`/migrate-component SearchBar React Vue` → `$0`=`SearchBar`, `$1`=`React`, `$2`=`Vue`.

## 고급 패턴 (Advanced patterns)

### 동적 컨텍스트 주입 (Inject dynamic context)

`` !`<command>` `` 구문은 스킬 내용이 Claude에게 전송되기 **전에** 셸 명령을 실행한다. 명령 출력이 플레이스홀더를 대체하므로 Claude는 명령 자체가 아닌 실제 데이터를 받는다.

```yaml
---
name: pr-summary
description: Summarize changes in a pull request
context: fork
agent: Explore
allowed-tools: Bash(gh *)
---

## Pull request context
- PR diff: !`gh pr diff`
- PR comments: !`gh pr view --comments`
- Changed files: !`gh pr diff --name-only`

## Your task
Summarize this pull request...
```

실행 순서:
1. 각 `` !`<command>` ``가 즉시 실행됨 (Claude가 아무것도 보기 전)
2. 출력이 스킬 내용의 플레이스홀더를 대체
3. Claude는 실제 데이터가 채워진 완성 프롬프트를 받음

이것은 Claude가 실행하는 것이 아니라 **전처리(preprocessing)**다. 치환은 원본 파일에 대해 한 번만 실행되고, 삽입된 명령 출력은 평문으로 들어가 다시 스캔되지 않으므로 명령이 다음 패스용 플레이스홀더를 내보낼 수 없다.

인라인 형식은 `!`가 줄 시작이거나 공백 바로 뒤에 올 때만 인식된다. `KEY=!`cmd`` 처럼 `!`가 다른 문자 뒤에 오면 리터럴 텍스트로 남고 명령은 실행되지 않는다.

여러 줄 명령은 ` ```! ` 로 여는 펜스 코드 블록을 사용:

````markdown
## Environment
```!
node --version
npm --version
git status --short
```
````

> [!warning] 셸 실행 비활성화
> 사용자·프로젝트·플러그인·추가 디렉터리 소스의 스킬·커스텀 명령에서 이 동작을 끄려면 [[04 설정]]에서 `"disableSkillShellExecution": true`를 설정한다. 각 명령은 실행되는 대신 `[shell command execution disabled by policy]`로 대체된다. 번들·관리형 스킬은 영향받지 않는다. 사용자가 재정의할 수 없는 managed settings에서 가장 유용하다.

> [!tip] 스킬 실행 시 더 깊은 추론을 원하면 스킬 내용 어디든 `ultrathink`를 포함하라. ([[19 비용과 성능]])

### 서브에이전트에서 스킬 실행 (Run skills in a subagent)

격리 실행이 필요하면 frontmatter에 `context: fork`를 추가한다. 스킬 내용이 서브에이전트를 구동하는 프롬프트가 되며, 대화 히스토리에 접근하지 못한다.

> [!warning]
> `context: fork`는 명시적 지침이 있는 스킬에만 의미가 있다. "이 API 규칙을 써라" 같은 가이드라인만 있고 태스크가 없으면 서브에이전트는 가이드라인만 받고 실행 가능한 프롬프트 없이 의미 있는 출력 없이 반환한다.

스킬과 [[08 서브에이전트와 에이전트 팀]]은 양방향으로 함께 작동한다.

| Approach | System prompt | Task | Also loads |
| :--- | :--- | :--- | :--- |
| `context: fork` 스킬 | 에이전트 타입에서 | SKILL.md 내용 | CLAUDE.md (에이전트가 Explore/Plan이면 제외) |
| `skills` 필드를 가진 서브에이전트 | 서브에이전트 markdown 본문 | Claude의 위임 메시지 | 프리로드 스킬 + CLAUDE.md |

`context: fork`에서는 스킬에 태스크를 쓰고 실행할 에이전트 타입을 고른다. 내장 Explore·Plan 에이전트는 CLAUDE.md와 git status를 건너뛰어 컨텍스트를 작게 유지하므로, `agent: Explore`를 쓴 포크 스킬은 SKILL.md 내용과 에이전트 자체 시스템 프롬프트만 본다.

#### 예시: Explore 에이전트를 쓰는 리서치 스킬

```yaml
---
name: deep-research
description: Research a topic thoroughly
context: fork
agent: Explore
---

Research $ARGUMENTS thoroughly:

1. Find relevant files using Glob and Grep
2. Read and analyze the code
3. Summarize findings with specific file references
```

실행 시: (1) 새 격리 컨텍스트 생성 → (2) 서브에이전트가 스킬 내용을 프롬프트로 받음 → (3) `agent` 필드가 실행 환경(모델·도구·권한) 결정 → (4) 결과가 요약되어 메인 대화로 반환.

`agent` 필드는 내장 에이전트(`Explore`, `Plan`, `general-purpose`) 또는 `.claude/agents/`의 커스텀 서브에이전트를 지정한다. 생략 시 `general-purpose` 사용.

### Claude의 스킬 접근 제한 (Restrict Claude's skill access)

기본적으로 Claude는 `disable-model-invocation: true`가 없는 모든 스킬을 호출할 수 있다. 일부 내장 명령(`/init`, `/review`, `/security-review`)도 Skill 도구로 사용 가능하지만 `/compact` 같은 다른 내장 명령은 아니다.

**모든 스킬 비활성화** — `/permissions`에서 Skill 도구 거부:

```text
# Add to deny rules:
Skill
```

**특정 스킬 허용/거부** — [[05 권한]] 규칙 사용:

```text
# Allow only specific skills
Skill(commit)
Skill(review-pr *)

# Deny specific skills
Skill(deploy *)
```

권한 구문: `Skill(name)`은 정확 매치, `Skill(name *)`은 임의 인자 접두 매치.

**개별 스킬 숨기기** — frontmatter에 `disable-model-invocation: true` 추가. Claude 컨텍스트에서 스킬을 완전히 제거한다.

> [!note]
> `user-invocable` 필드는 메뉴 가시성만 제어하고 Skill 도구 접근은 제어하지 않는다. 프로그램적 호출을 막으려면 `disable-model-invocation: true`를 사용하라.

### 설정에서 스킬 가시성 재정의 (skillOverrides)

`skillOverrides` 설정은 스킬 자체 frontmatter 대신 [[04 설정]]에서 스킬 가시성을 제어한다. SKILL.md를 편집하고 싶지 않은 스킬(공유 레포에 체크인된 것, MCP 서버 제공)에 사용. `/skills` 메뉴가 대신 써준다 — 스킬을 하이라이트하고 `Space`로 상태 순환, `Enter`로 `.claude/settings.local.json`에 저장.

각 키는 스킬 이름, 각 값은 네 상태 중 하나:

| Value | Listed to Claude | In `/` menu |
| :--- | :--- | :--- |
| `"on"` | 이름+description | Yes |
| `"name-only"` | 이름만 | Yes |
| `"user-invocable-only"` | 숨김 | Yes |
| `"off"` | 숨김 | 숨김 |

`skillOverrides`에 없는 스킬은 `"on"`으로 취급된다.

```json
{
  "skillOverrides": {
    "legacy-context": "name-only",
    "deploy": "off"
  }
}
```

플러그인 스킬은 `skillOverrides`의 영향을 받지 않는다 — `/plugin`으로 관리한다. ([[11 플러그인]])

## 스킬 공유 (Share skills)

대상 청중에 따라 다른 스코프로 배포한다.

- **Project skills**: `.claude/skills/`를 버전 관리에 커밋
- **Plugins**: [[11 플러그인]]에 `skills/` 디렉터리 생성
- **Managed**: managed settings로 조직 전체 배포 ([[16 CI-CD와 팀 통합]])

### 시각적 출력 생성 (Generate visual output)

스킬은 어떤 언어의 스크립트든 번들·실행할 수 있어, 단일 프롬프트로 불가능한 능력을 Claude에게 준다. 강력한 패턴 중 하나가 시각적 출력 생성이다 — 브라우저에서 열리는 인터랙티브 HTML로 데이터 탐색·디버깅·리포트 작성.

다음은 코드베이스 탐색기 예시다(확장/축소 트리 뷰, 파일 크기, 색상별 파일 타입).

```bash
mkdir -p ~/.claude/skills/codebase-visualizer/scripts
```

`~/.claude/skills/codebase-visualizer/SKILL.md` — 스크립트 경로에 `${CLAUDE_SKILL_DIR}`를 써서 personal·project·plugin 어디에 설치돼도 올바르게 해석되게 한다:

````yaml
---
name: codebase-visualizer
description: Generate an interactive collapsible tree visualization of your codebase. Use when exploring a new repo, understanding project structure, or identifying large files.
allowed-tools: Bash(python3 *)
---

# Codebase Visualizer

Generate an interactive HTML tree view that shows your project's file structure with collapsible directories.

## Usage

Run the visualization script from your project root:

```bash
python3 ${CLAUDE_SKILL_DIR}/scripts/visualize.py .
```

This creates `codebase-map.html` in the current directory and opens it in your default browser.
````

번들 Python 스크립트(`scripts/visualize.py`)는 디렉터리 트리를 스캔해 자체 완결형 HTML을 생성한다 — 요약 사이드바(파일/디렉터리 수, 총 크기, 파일 타입 수), 파일 타입별 막대 차트(크기 상위 8개), 색상 코딩된 확장/축소 트리. Python 3의 내장 라이브러리만 사용하므로 설치할 패키지가 없다. 테스트는 임의 프로젝트에서 Claude Code를 열고 "Visualize this codebase"라고 묻는다 — Claude가 스크립트를 실행하고 `codebase-map.html`을 생성해 브라우저에서 연다.

이 패턴은 의존성 그래프, 테스트 커버리지 리포트, API 문서, DB 스키마 시각화 등 모든 시각적 출력에 적용된다. **번들 스크립트가 작업을 하고 Claude가 오케스트레이션을 담당**한다.

## 트러블슈팅 (Troubleshooting)

자세한 진단은 [[25 트러블슈팅]] 참고.

### 스킬이 트리거되지 않을 때

1. description에 사용자가 자연스럽게 말할 키워드가 포함됐는지 확인
2. `What skills are available?`로 스킬이 나타나는지 검증
3. description에 더 가깝게 요청을 다시 표현
4. user-invocable이면 `/skill-name`으로 직접 호출

### 스킬이 너무 자주 트리거될 때

1. description을 더 구체적으로
2. 수동 호출만 원하면 `disable-model-invocation: true` 추가

### 스킬 description이 잘릴 때

스킬 description은 Claude가 무엇이 가용한지 알도록 컨텍스트에 로드된다. **모든 스킬 이름은 항상 포함**되지만, 스킬이 많으면 description이 문자 예산에 맞게 줄어들어 매칭에 필요한 키워드가 잘릴 수 있다. 예산은 **모델 컨텍스트 윈도우의 1%**로 스케일된다. 넘치면 가장 적게 호출한 스킬의 description부터 드롭되므로 실제로 쓰는 스킬은 전체 텍스트를 유지한다. `/doctor`로 예산 초과 여부와 영향받는 스킬을 확인하라.

예산을 늘리려면 `skillListingBudgetFraction` 설정(예: `0.02` = 2%) 또는 `SLASH_COMMAND_TOOL_CHAR_BUDGET` 환경변수(고정 문자 수)를 설정한다. 다른 스킬에 예산을 확보하려면 저우선 항목을 `skillOverrides`에서 `"name-only"`로 설정해 description 없이 나열한다. 소스에서 `description`·`when_to_use` 텍스트를 다듬을 수도 있다 — 핵심 use case를 맨 앞에 두면 좋다(각 항목 합산 텍스트는 예산과 무관하게 **1,536자** 캡). 이 캡은 `maxSkillDescriptionChars`로 설정 가능. ([[19 비용과 성능]])
