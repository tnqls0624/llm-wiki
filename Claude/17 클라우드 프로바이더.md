# 17 클라우드 프로바이더

Claude Code는 Anthropic이 직접 운영하는 Claude API 외에도 **클라우드 프로바이더**를 통해 Claude 모델을 호출할 수 있다. 이는 조직이 AWS / GCP / Azure의 기존 계약·과금·IAM·감사 체계 안에서 Claude Code를 배포하기 위한 경로다. 이 노트는 5개 배포 경로(Amazon Bedrock, Claude Platform on AWS, Google Vertex AI, Microsoft Foundry, 그리고 그 비교)와 두 가지 네트워크 인프라 토픽(LLM 게이트웨이, 엔터프라이즈 네트워크 설정)을 다룬다.

허브: [[Claude]] · 관련: [[04 설정]] · [[18 보안과 샌드박스]] · [[19 비용과 성능]] · [[16 CI-CD와 팀 통합]] · [[10 MCP]]

> [!tip] 핵심 멘탈 모델
> 각 프로바이더는 `CLAUDE_CODE_USE_*` 플래그로 활성화되고, 인증은 해당 클라우드의 자격증명 체계(AWS SigV4 / GCP ADC / Azure Entra ID)를 따른다. 멀티유저 배포 시에는 **반드시 모델 버전을 핀(pin)**해야 한다. `base URL`을 오버라이드하면 프록시·게이트웨이를 끼울 수 있고, `SKIP_*_AUTH` 플래그로 클라이언트 측 서명을 끌 수 있다.

## 배포 옵션 비교 (third-party-integrations)

조직은 Claude Code를 Anthropic 직접 경로 또는 클라우드 프로바이더 경로로 배포할 수 있다. 대부분의 조직에는 **Claude for Teams / Enterprise**가 가장 좋은 경험을 준다(Claude Code + 웹 Claude를 단일 구독으로, 중앙 과금, 인프라 셋업 불필요).

- **Claude for Teams**: 셀프서비스. 협업 기능·관리 도구·과금 관리 포함. 빠르게 시작해야 하는 소규모 팀에 적합.
- **Claude for Enterprise**: SSO·도메인 캡처, 역할 기반 권한, 컴플라이언스 API 접근, 조직 전체 Claude Code 설정을 위한 managed policy 추가. 보안·컴플라이언스 요구가 있는 대규모 조직에 적합.

특정 인프라 요구가 있는 경우 아래 표로 비교한다.

| Feature | Teams/Enterprise | Anthropic Console | Amazon Bedrock | Claude Platform on AWS | Google Vertex AI | Microsoft Foundry |
| :-- | :-- | :-- | :-- | :-- | :-- | :-- |
| Best for | 대부분의 조직(권장) | 개인 개발자 | AWS-native 배포 | AWS Marketplace 과금 + Claude API 기능 | GCP-native 배포 | Azure-native 배포 |
| Billing | Teams: $150/seat(Premium)+PAYG / Enterprise: Contact Sales | PAYG | PAYG through AWS | PAYG through AWS Marketplace | PAYG through GCP | PAYG through Azure |
| Regions | 지원 국가 | 지원 국가 | 다수 AWS 리전 | 다수 AWS 리전 | 다수 GCP 리전 | 다수 Azure 리전 |
| Prompt caching | 기본 활성 | 기본 활성 | 기본 활성 | 기본 활성 | 기본 활성 | 기본 활성 |
| Authentication | Claude.ai SSO / email | API key | API key or AWS credentials | API key or AWS credentials | GCP credentials | API key or Microsoft Entra ID |
| Cost tracking | Usage dashboard | Usage dashboard | AWS Cost Explorer | AWS Cost Explorer | GCP Billing | Azure Cost Management |
| Claude on web 포함 | Yes | No | No | No | No | No |
| Enterprise features | 팀 관리·SSO·사용량 모니터링 | 없음 | IAM policies, CloudTrail | IAM policies, CloudTrail | IAM roles, Cloud Audit Logs | RBAC policies, Azure Monitor |

각 경로의 셋업 진입점: Teams/Enterprise·Console은 인증 문서, Bedrock/Claude Platform on AWS/Vertex AI/Foundry는 아래 각 섹션.

### 프록시와 게이트웨이 구분 (함께 사용 가능)

대부분 조직은 추가 설정 없이 클라우드 프로바이더를 직접 쓸 수 있으나, 네트워크/관리 요구가 있으면 다음 두 가지를 (각각 또는 함께) 구성한다.

| 구분 | 용도 | 설정 변수 |
| :-- | :-- | :-- |
| **Corporate proxy** | 모든 아웃바운드 트래픽을 보안 모니터링·컴플라이언스·네트워크 정책상 프록시 서버로 통과 | `HTTPS_PROXY` / `HTTP_PROXY` → [엔터프라이즈 네트워크 설정](#엔터프라이즈-네트워크-설정-network-config) |
| **LLM Gateway** | Claude Code와 프로바이더 사이에서 인증·라우팅 담당. 팀 간 사용량 집계, 커스텀 rate limit/예산, 중앙 인증 관리 | `ANTHROPIC_BASE_URL` / `ANTHROPIC_BEDROCK_BASE_URL` / `ANTHROPIC_AWS_BASE_URL` / `ANTHROPIC_VERTEX_BASE_URL` → [LLM 게이트웨이](#llm-게이트웨이-구성-llm-gateway) |

프록시 + 게이트웨이 조합 예시(셸 또는 `.bashrc`/`.zshrc`에 export):

```bash
# Bedrock + corporate proxy
export CLAUDE_CODE_USE_BEDROCK=1
export AWS_REGION=us-east-1
export HTTPS_PROXY='https://proxy.example.com:8080'
```

```bash
# Bedrock + LLM gateway
export CLAUDE_CODE_USE_BEDROCK=1
export ANTHROPIC_BEDROCK_BASE_URL='https://your-llm-gateway.com/bedrock'
export CLAUDE_CODE_SKIP_BEDROCK_AUTH=1  # 게이트웨이가 AWS 인증을 처리하는 경우
```

```bash
# Foundry + LLM gateway
export CLAUDE_CODE_USE_FOUNDRY=1
export ANTHROPIC_FOUNDRY_BASE_URL='https://your-llm-gateway.com'
export CLAUDE_CODE_SKIP_FOUNDRY_AUTH=1  # 게이트웨이가 Azure 인증을 처리하는 경우
```

```bash
# Vertex AI + LLM gateway
export CLAUDE_CODE_USE_VERTEX=1
export ANTHROPIC_VERTEX_BASE_URL='https://your-llm-gateway.com/vertex'
export CLAUDE_CODE_SKIP_VERTEX_AUTH=1  # 게이트웨이가 GCP 인증을 처리하는 경우
```

> [!tip]
> `/status`를 실행해 프록시·게이트웨이 설정이 올바로 적용됐는지 확인하라.

### 조직 배포 베스트 프랙티스

- **문서/메모리에 투자**: Claude Code가 코드베이스를 이해하도록 `CLAUDE.md`를 여러 레벨로 배포. 조직 전체용은 시스템 디렉터리(macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`), 레포 레벨은 레포 루트의 `CLAUDE.md`(아키텍처·빌드 명령·기여 가이드)를 소스 컨트롤에 체크인. → [[03 메모리와 컨텍스트]]
- **배포 단순화**: 커스텀 개발 환경이라면 "원클릭" 설치 방법을 만들면 채택이 빨라진다.
- **가이드된 사용으로 시작**: 신규 사용자는 코드베이스 Q&A, 작은 버그·기능 요청부터. Claude에게 계획을 세우게 하고 피드백을 주다가 점차 에이전트적으로 풀어준다. → [[20 베스트 프랙티스와 워크플로우]]
- **클라우드 프로바이더는 모델 버전 핀**: `ANTHROPIC_DEFAULT_OPUS_MODEL` / `_SONNET_MODEL` / `_HAIKU_MODEL`로 명시 핀. 핀 없으면 alias가 프로바이더별 빌트인 기본값으로 풀리며 최신을 못 따라가거나 계정에 미활성일 수 있다.
- **보안 정책 구성**: 보안팀은 로컬 설정으로 덮어쓸 수 없는 managed permissions를 구성. → [[05 권한]] · [[18 보안과 샌드박스]]
- **MCP로 통합 강화**: 중앙 팀이 MCP 서버를 구성하고 `.mcp.json`을 레포에 체크인. → [[10 MCP]]

## Amazon Bedrock (amazon-bedrock)

AWS-native 배포 경로. Claude Code는 Bedrock **Invoke API**를 사용하며 **Converse API는 지원하지 않는다**.

### 사전 요구 사항

- Bedrock 접근이 활성화된 AWS 계정
- Bedrock에서 원하는 Claude 모델(예: Claude Sonnet 4.6) 접근 권한
- AWS CLI 설치·구성(선택 — 다른 자격증명 획득 수단이 있으면 불필요)
- 적절한 IAM 권한

### Sign in with Bedrock (로그인 위저드)

AWS 자격증명이 있으면 로그인 위저드가 Claude Code 측을 처리한다.

1. **AWS 계정에서 Anthropic 모델 활성화**: [Bedrock 콘솔](https://console.aws.amazon.com/bedrock/) → Model catalog에서 Anthropic 모델 선택 → use case 폼 제출(제출 즉시 접근 허용).
2. **`claude` 실행** → 로그인 프롬프트에서 **3rd-party platform** → **Amazon Bedrock** 선택.
3. **위저드 진행**: AWS 인증 방식 선택(`~/.aws`에서 감지된 AWS 프로파일, Bedrock API 키, access key + secret, 환경의 기존 자격증명). 위저드가 리전을 잡고 호출 가능한 모델을 검증·핀하며, 결과를 [user settings file](/en/settings)의 `env` 블록에 저장한다(직접 export 불필요).

로그인 후 언제든 `/setup-bedrock`으로 위저드를 다시 열어 자격증명·리전·모델 핀을 변경할 수 있다.

### 수동 설정 (CI/스크립트 롤아웃)

#### 1. Use case 폼 제출
Anthropic 모델 최초 사용자는 호출 전 use case 폼 제출 필요(계정당 1회). AWS Organizations 사용 시 관리 계정에서 [`PutUseCaseForModelAccess` API](https://docs.aws.amazon.com/bedrock/latest/APIReference/API_PutUseCaseForModelAccess.html)로 1회 제출(IAM 권한 `bedrock:PutUseCaseForModelAccess` 필요), 자식 계정에 자동 확장.

#### 2. AWS 자격증명 구성
Claude Code는 기본 AWS SDK 자격증명 체인을 사용한다.

| Option | 명령/변수 |
| :-- | :-- |
| A: AWS CLI | `aws configure` |
| B: 환경변수(access key) | `export AWS_ACCESS_KEY_ID=...` / `AWS_SECRET_ACCESS_KEY=...` / `AWS_SESSION_TOKEN=...` |
| C: SSO 프로파일 | `aws sso login --profile=<name>` 후 `export AWS_PROFILE=<name>` |
| D: 콘솔 자격증명 | `aws login` |
| E: Bedrock API 키 | `export AWS_BEARER_TOKEN_BEDROCK=your-bedrock-api-key` |

> Bedrock API 키는 전체 AWS 자격증명 없이 쓰는 더 간단한 인증 방식.

##### 고급 자격증명 구성 — `awsAuthRefresh` vs `awsCredentialExport`

settings 파일에 추가하는 두 설정은 트리거 조건이 다르다.

| 설정 | 트리거 | 출력 | 용도 |
| :-- | :-- | :-- | :-- |
| `awsAuthRefresh` | 자격증명이 **만료됐다고 감지**(타임스탬프 또는 Bedrock의 credential error)했을 때만 실행 후 재시도 | 사용자에게 표시(대화형 입력 불가) | `.aws` 디렉터리(자격증명·SSO 캐시·config)를 수정하는 명령. 브라우저 기반 SSO 흐름에 적합 |
| `awsCredentialExport` | 세션 시작 시 + 매 자격증명 reload 시(유효해도) 실행 | 조용히 캡처(미표시) | `.aws`를 수정할 수 없고 직접 자격증명을 반환해야 할 때만. 크로스 계정 자격증명에 사용 |

```json
{
  "awsAuthRefresh": "aws sso login --profile myprofile",
  "env": { "AWS_PROFILE": "myprofile" }
}
```

`awsCredentialExport`는 다음 JSON 형식으로 출력해야 한다.

```json
{
  "Credentials": {
    "AccessKeyId": "value",
    "SecretAccessKey": "value",
    "SessionToken": "value"
  }
}
```

#### 3. Claude Code 구성

```bash
# Bedrock 활성화
export CLAUDE_CODE_USE_BEDROCK=1
export AWS_REGION=us-east-1  # 또는 선호 리전

# 선택: small/fast 모델용 리전 오버라이드(Bedrock+Mantle).
# Bedrock에서는 ANTHROPIC_DEFAULT_HAIKU_MODEL(또는 deprecated ANTHROPIC_SMALL_FAST_MODEL) 없으면 무효.
export ANTHROPIC_SMALL_FAST_MODEL_AWS_REGION=us-west-2

# 선택: 커스텀 엔드포인트/게이트웨이용 Bedrock 엔드포인트 URL 오버라이드
# export ANTHROPIC_BEDROCK_BASE_URL=https://bedrock-runtime.us-east-1.amazonaws.com
```

주의사항:
- `AWS_REGION`은 **필수** 환경변수. Claude Code는 이 값을 `.aws` config에서 읽지 않는다.
- Bedrock 사용 시 `/logout` 명령은 사용 불가(인증이 AWS 자격증명으로 처리됨).
- **WebSearch 도구는 Bedrock에서 사용 불가**. → [[06 내장 도구 레퍼런스]]
- `AWS_PROFILE` 같은 변수는 다른 프로세스로 새지 않게 settings 파일에 둘 수 있다.

#### 4. 모델 버전 핀

> [!warning] 멀티유저 배포 시 반드시 핀
> 핀이 없으면 `sonnet`·`opus` alias가 Claude Code의 Bedrock 빌트인 기본값으로 풀리며, 최신 릴리스보다 뒤처지거나 계정에 미활성일 수 있다. 기본값이 없으면 시작 시 이전 버전으로 폴백하나, 핀을 두면 마이그레이션 시점을 제어할 수 있다.

`ANTHROPIC_DEFAULT_OPUS_MODEL` 없으면 `opus` alias는 Bedrock에서 Opus 4.6으로 풀린다. 최신을 쓰려면 Opus 4.8 ID로 설정:

```bash
export ANTHROPIC_DEFAULT_OPUS_MODEL='us.anthropic.claude-opus-4-8'
export ANTHROPIC_DEFAULT_SONNET_MODEL='us.anthropic.claude-sonnet-4-6'
export ANTHROPIC_DEFAULT_HAIKU_MODEL='us.anthropic.claude-haiku-4-5-20251001-v1:0'
```

위 값은 cross-region inference profile ID(`us.` 접두사). 다른 리전 접두사나 application inference profile을 쓰면 맞춰 조정. 핀 변수가 없을 때의 기본값:

| Model type | Default value |
| :-- | :-- |
| Primary model | `us.anthropic.claude-sonnet-4-5-20250929-v1:0` |
| Small/fast model | Primary model과 동일 |

배경 작업(세션 제목 생성 등)은 small/fast 모델(보통 Haiku급)을 쓰는데, Bedrock에서는 Haiku가 모든 계정/리전에 활성이 아닐 수 있어 기본적으로 primary 모델로 폴백한다. Haiku를 쓰려면 `ANTHROPIC_DEFAULT_HAIKU_MODEL`을 설정.

추가 커스터마이즈:

```bash
# inference profile ID 사용
export ANTHROPIC_MODEL='us.anthropic.claude-sonnet-4-6'
export ANTHROPIC_DEFAULT_HAIKU_MODEL='us.anthropic.claude-haiku-4-5-20251001-v1:0'

# application inference profile ARN 사용
export ANTHROPIC_MODEL='arn:aws:bedrock:us-east-2:your-account-id:application-inference-profile/your-model-id'

# 선택: 프롬프트 캐싱 비활성화
export DISABLE_PROMPT_CACHING=1

# 선택: 5분 기본 대신 1시간 캐시 TTL 요청(더 높은 요율 과금)
export ENABLE_PROMPT_CACHING_1H=1
```

> 프롬프트 캐싱이 모든 Bedrock 리전에서 가능하지 않을 수 있다. 캐시 토큰 카운트가 0이면 Bedrock 문서의 지원 모델·리전·한도를 확인. 비용 관련 → [[19 비용과 성능]]

##### 모델 버전별 inference profile 매핑 (`modelOverrides`)

`ANTHROPIC_DEFAULT_*_MODEL`은 모델 패밀리당 하나의 inference profile만 구성한다. 같은 패밀리의 여러 버전을 각각 다른 application inference profile ARN으로 `/model` picker에 노출하려면 settings 파일의 `modelOverrides`를 사용:

```json
{
  "modelOverrides": {
    "claude-opus-4-7": "arn:aws:bedrock:us-east-2:123456789012:application-inference-profile/opus-47-prod",
    "claude-opus-4-6": "arn:aws:bedrock:us-east-2:123456789012:application-inference-profile/opus-46-prod",
    "claude-opus-4-5-20251101": "arn:aws:bedrock:us-east-2:123456789012:application-inference-profile/opus-45-prod",
    "claude-opus-4-1-20250805": "arn:aws:bedrock:us-east-2:123456789012:application-inference-profile/opus-41-prod"
  }
}
```

`/model`에서 해당 버전 선택 시 매핑된 ARN으로 호출. 오버라이드 없는 버전은 빌트인 Bedrock 모델 ID 또는 시작 시 발견된 inference profile로 폴백.

### Startup model checks

Bedrock 구성으로 시작하면 Claude Code는 사용하려는 모델이 계정에서 접근 가능한지 검증한다(**v2.1.94 이상 필요**).

- 핀한 버전이 현재 기본값보다 오래됐고 새 버전 호출이 가능하면 → 핀 업데이트를 프롬프트. 수락 시 user settings에 새 ID를 쓰고 재시작. 거절은 다음 기본 버전 변경까지 기억. application inference profile ARN을 가리키는 핀은 관리자가 관리하므로 스킵.
- 핀하지 않았고 현재 기본값이 미활성이면 → 현 세션 동안 이전 버전으로 폴백하고 안내 표시(영구 저장 안 됨). 영구화하려면 모델을 활성화하거나 핀.

### IAM 구성

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowModelAndInferenceProfileAccess",
      "Effect": "Allow",
      "Action": [
        "bedrock:InvokeModel",
        "bedrock:InvokeModelWithResponseStream",
        "bedrock:ListInferenceProfiles",
        "bedrock:GetInferenceProfile"
      ],
      "Resource": [
        "arn:aws:bedrock:*:*:inference-profile/*",
        "arn:aws:bedrock:*:*:application-inference-profile/*",
        "arn:aws:bedrock:*:*:foundation-model/*"
      ]
    },
    {
      "Sid": "AllowMarketplaceSubscription",
      "Effect": "Allow",
      "Action": ["aws-marketplace:ViewSubscriptions", "aws-marketplace:Subscribe"],
      "Resource": "*",
      "Condition": { "StringEquals": { "aws:CalledViaLast": "bedrock.amazonaws.com" } }
    }
  ]
}
```

- `bedrock:GetInferenceProfile`은 application inference profile ARN을 backing foundation model로 해석해 올바른 요청 형태를 고르게 한다. 이 권한이 없으면 대체 형태로 1회 재시도해 요청은 성공하나 새 모델마다 추가 round-trip 발생. 특히 `AWS_BEARER_TOKEN_BEDROCK` 배포(토큰 정책이 좁음)에 해당.
- 비용 추적·접근 제어 단순화를 위해 Claude Code 전용 AWS 계정을 만들기를 권장.

### 1M 토큰 컨텍스트 윈도우

Claude Opus 4.6 이상, Sonnet 4.6은 Bedrock에서 1M 토큰 컨텍스트 윈도우를 지원. 1M 모델 변형 선택 시 자동 활성. 위저드는 모델 핀 시 1M 옵션 제공. 수동 핀에는 모델 ID에 `[1m]`을 덧붙임.

### Service tiers

[Bedrock service tiers](https://docs.aws.amazon.com/bedrock/latest/userguide/service-tiers-inference.html)로 비용 대 지연을 트레이드오프. `ANTHROPIC_BEDROCK_SERVICE_TIER`를 `default`/`flex`/`priority`로:

```bash
export ANTHROPIC_BEDROCK_SERVICE_TIER=priority
```

요청마다 `X-Amzn-Bedrock-Service-Tier` 헤더로 전송. 티어 가용성은 모델·리전마다 다름. 예약 용량(provisioned throughput)은 이 설정 대신 ARN을 모델 ID로 사용.

### AWS Guardrails

[Bedrock Guardrails](https://docs.aws.amazon.com/bedrock/latest/userguide/guardrails.html)로 콘텐츠 필터링. 콘솔에서 Guardrail 생성·버전 발행 후 settings 파일에 헤더 추가. cross-region inference profile을 쓰면 Guardrail에 Cross-Region inference를 활성화.

```json
{
  "env": {
    "ANTHROPIC_CUSTOM_HEADERS": "X-Amzn-Bedrock-GuardrailIdentifier: your-guardrail-id\nX-Amzn-Bedrock-GuardrailVersion: 1"
  }
}
```

### Mantle 엔드포인트 사용

Mantle은 Bedrock Invoke API가 아니라 **네이티브 Anthropic API 형태**로 Claude를 서빙하는 Bedrock 엔드포인트다. 동일한 AWS 자격증명·IAM 권한·`awsAuthRefresh`를 사용. **v2.1.94 이상 필요.**

#### Mantle 활성화

```bash
export CLAUDE_CODE_USE_MANTLE=1
export AWS_REGION=us-east-1
```

`AWS_REGION`에서 엔드포인트 URL 구성. 오버라이드는 `ANTHROPIC_BEDROCK_MANTLE_BASE_URL`. `/status`에서 provider 줄이 `Amazon Bedrock (Mantle)`이면 활성.

#### Mantle 모델 선택
Mantle 모델 ID는 `anthropic.` 접두사 + 버전 접미사 없음(예: `anthropic.claude-haiku-4-5`). 사용 가능 모델은 조직이 부여받은 범위에 따름(allowlist는 AWS 온보딩 자료에 있음).

```bash
claude --model anthropic.claude-haiku-4-5
```

#### Invoke API와 병행 실행
`CLAUDE_CODE_USE_BEDROCK`와 `CLAUDE_CODE_USE_MANTLE`을 둘 다 켜면 한 세션에서 두 엔드포인트를 호출. Mantle 형식 ID는 Mantle로, 그 외는 Bedrock Invoke API로 라우팅.

```bash
export CLAUDE_CODE_USE_BEDROCK=1
export CLAUDE_CODE_USE_MANTLE=1
```

`/model` picker에 Mantle 모델을 노출하려면 settings의 `availableModels`에 ID를 나열(이 설정은 picker를 나열 항목으로 제한하므로 유지할 alias 전부 포함):

```json
{ "availableModels": ["opus", "sonnet", "haiku", "anthropic.claude-haiku-4-5"] }
```

둘 다 활성이면 `/status`는 `Amazon Bedrock + Amazon Bedrock (Mantle)` 표시.

#### 게이트웨이 경유 Mantle
조직이 서버 측에서 AWS 자격증명을 주입하는 [LLM 게이트웨이](#llm-게이트웨이-구성-llm-gateway)로 라우팅하면, 클라이언트 측 인증을 끄도록(SigV4·`x-api-key` 미전송):

```bash
export CLAUDE_CODE_USE_MANTLE=1
export CLAUDE_CODE_SKIP_MANTLE_AUTH=1
export ANTHROPIC_BEDROCK_MANTLE_BASE_URL=https://your-gateway.example.com
```

#### Mantle 환경변수

| Variable | Purpose |
| :-- | :-- |
| `CLAUDE_CODE_USE_MANTLE` | Mantle 엔드포인트 활성(`1` 또는 `true`) |
| `ANTHROPIC_BEDROCK_MANTLE_BASE_URL` | 기본 Mantle 엔드포인트 URL 오버라이드 |
| `CLAUDE_CODE_SKIP_MANTLE_AUTH` | 프록시 셋업용 클라이언트 측 인증 스킵 |
| `ANTHROPIC_SMALL_FAST_MODEL_AWS_REGION` | Haiku급 모델용 AWS 리전 오버라이드(Bedrock과 공유) |

### Bedrock 트러블슈팅

- **SSO + 기업 프록시 인증 루프**: 브라우저 탭이 반복 생성되면 settings에서 `awsAuthRefresh` 제거. 기업 VPN/TLS 검사 프록시가 SSO 브라우저 흐름을 끊으면 Claude Code가 인증 실패로 보고 무한 재실행. 자동 흐름이 방해되면 시작 전 `aws sso login`을 수동 실행.
- **리전 이슈**: `aws bedrock list-inference-profiles --region your-region`로 가용성 확인 / `export AWS_REGION=us-east-1`로 전환 / cross-region inference profile 고려.
- **"on-demand throughput isn't supported"**: 모델을 inference profile ID로 지정.
- Claude Code는 Bedrock **Invoke API**만 사용, **Converse API 미지원**.
- **Mantle 엔드포인트 에러**: `/status`에 `Amazon Bedrock (Mantle)`이 안 보이면 변수가 프로세스에 안 닿은 것 — export 또는 settings `env` 확인. 유효 자격증명에 `403`은 모델 접근 미부여(AWS 계정팀 문의). 모델 ID를 명시한 `400`은 그 모델이 Mantle에 안 서빙됨 — Mantle 형식 ID를 쓰거나 양 엔드포인트 병행.

## Claude Platform on AWS (claude-platform-on-aws)

**Anthropic이 직접 운영하는 Claude API**를, AWS 인증·IAM 접근 제어·AWS Marketplace 과금과 결합한 형태. 요청이 Anthropic API에 직접 도달하므로 [Claude API](https://platform.claude.com/docs)와 **동일한 모델·기능·릴리스 일정**을 받는다. AWS 자격증명 또는 workspace API 키로 인증하고 AWS Marketplace로 결제.

> [!note] 별도 조직 생성
> AWS Marketplace 구독은 AWS 계정에 묶인 **새 Anthropic 조직**을 프로비저닝한다. 기존 Anthropic 조직과 별개이며 자격증명이 이전되지 않는다. **AWS-linked 조직의 workspace ID·API 키**를 쓰고, 기존 Claude Console 계정 것을 쓰지 말 것.

### 사전 요구 사항
- AWS Marketplace를 통한 활성 Claude Platform on AWS 구독
- AWS-linked Anthropic 조직 내 workspace와 그 workspace ID
- Anthropic 서비스 호출 권한이 있는 IAM principal, 또는 workspace 범위 API 키
- SigV4를 쓰려면 환경/`~/.aws/credentials`/IAM 롤의 AWS 자격증명. AWS CLI는 SSO 로그인 흐름에만 필요.

### Setup

#### 1. AWS 자격증명 구성 — 두 가지 인증

**Option A: AWS 자격증명 + SigV4**
표준 AWS 자격증명 체인(환경변수, `~/.aws/credentials`, IAM 롤, AWS SSO 세션 등)으로 SigV4 서명.

```bash
aws sso login --profile my-profile
export AWS_PROFILE=my-profile
```

CI/자동화는 러너에 Anthropic 서비스 호출 권한 IAM 롤을 주고 `AWS_REGION` 설정. SSO 자격증명이 세션 중 만료되면 `awsAuthRefresh`(→ [Bedrock 고급 자격증명](#고급-자격증명-구성--awsauthrefresh-vs-awscredentialexport))로 재로그인·재시도:

```json
{ "awsAuthRefresh": "aws sso login --profile my-profile" }
```

**Option B: Workspace API 키**
장수명 시크릿. AWS Console **Claude Platform on AWS → API keys**에서 생성 후 `ANTHROPIC_AWS_API_KEY`로:

```bash
export ANTHROPIC_AWS_API_KEY=sk-ant-xxxxx
```

이 키는 `x-api-key`로 전송되고 **SigV4보다 우선**한다(환경의 AWS 자격증명 무시). 별도 Claude Console 조직의 API 키는 여기서 작동하지 않음. → settings `env` 블록에 두면 전역 export 없이 머신에 한정 가능.

> `/login`·`/logout`은 Claude Platform on AWS 인증을 바꾸지 않는다(AWS 자격증명/workspace API 키로 인증).

#### 2. Claude Code 구성

```bash
export CLAUDE_CODE_USE_ANTHROPIC_AWS=1
export ANTHROPIC_AWS_WORKSPACE_ID=wrkspc_01ABCDEFGHIJKLMN
export AWS_REGION=us-east-1
```

`ANTHROPIC_AWS_WORKSPACE_ID`는 **필수**이며 매 요청에 `anthropic-workspace-id` 헤더로 전송. base URL은 `AWS_REGION`에서 `https://aws-external-anthropic.{region}.api.aws`로 계산. 직접 오버라이드는 `ANTHROPIC_AWS_BASE_URL`.

> 이 경로는 AWS 자격증명이 환경에 있어도 **opt-in**이다. provider 라우팅에서 Bedrock·Foundry가 우선하므로, 설정돼 있다면 `CLAUDE_CODE_USE_BEDROCK`·`CLAUDE_CODE_USE_FOUNDRY`를 unset할 것.

#### 3. 모델 버전 핀
직접 Claude API와 동일한 모델 ID 사용. 핀 없으면 alias가 빌트인 기본값으로(예: `opus`→Opus 4.7). 팀 배포 시 명시 핀:

```bash
export ANTHROPIC_DEFAULT_OPUS_MODEL=claude-opus-4-7
export ANTHROPIC_DEFAULT_SONNET_MODEL=claude-sonnet-4-6
export ANTHROPIC_DEFAULT_HAIKU_MODEL=claude-haiku-4-5
```

프롬프트 캐싱 자동 활성. 1시간 TTL은 `ENABLE_PROMPT_CACHING_1H=1`(더 높은 요율).

### Agent SDK 사용
[Agent SDK](/en/agent-sdk/overview)는 CLI와 같은 환경변수를 읽으므로, 서브프로세스를 띄우기 전 `CLAUDE_CODE_USE_ANTHROPIC_AWS`·`ANTHROPIC_AWS_WORKSPACE_ID`와 `ANTHROPIC_AWS_API_KEY`(또는 AWS 자격증명)를 export하면 된다. → [[22 Agent SDK — 시작]]

```typescript
import { query } from "@anthropic-ai/claude-agent-sdk";

process.env.CLAUDE_CODE_USE_ANTHROPIC_AWS = "1";
process.env.ANTHROPIC_AWS_WORKSPACE_ID = "wrkspc_01ABCDEFGHIJKLMN";
process.env.AWS_REGION = "us-east-1";

for await (const msg of query({ prompt: "What's in this repo?" })) {
  console.log(msg);
}
```

### 기업 프록시 경유
프록시/게이트웨이로 라우팅하려면 `ANTHROPIC_AWS_BASE_URL`을 프록시 주소로. 동일한 workspace·인증 헤더로 전송되므로 그대로 forward하는 게이트웨이면 동작.

```bash
export CLAUDE_CODE_USE_ANTHROPIC_AWS=1
export ANTHROPIC_AWS_WORKSPACE_ID=wrkspc_01ABCDEFGHIJKLMN
export ANTHROPIC_AWS_BASE_URL=https://anthropic-proxy.example.com
```

게이트웨이가 직접 서명하면 `CLAUDE_CODE_SKIP_ANTHROPIC_AWS_AUTH=1`(미서명 전송, 게이트웨이가 SigV4 추가). 게이트웨이가 자체 토큰을 요구하면 `ANTHROPIC_AUTH_TOKEN`에 설정.

```bash
export CLAUDE_CODE_USE_ANTHROPIC_AWS=1
export CLAUDE_CODE_SKIP_ANTHROPIC_AWS_AUTH=1
export ANTHROPIC_AWS_WORKSPACE_ID=wrkspc_01ABCDEFGHIJKLMN
export ANTHROPIC_AWS_BASE_URL=https://anthropic-proxy.example.com
```

### 트러블슈팅
`/status`로 해석된 provider·workspace ID·리전·base URL·auth-skip 설정을 확인(가장 빠른 진단).
- **모든 요청에 `403 Forbidden`/`AccessDenied`**: IAM principal에 Anthropic 서비스 호출 권한 부족. 롤에 `aws-external-anthropic` 액션이 있는지 확인. `ANTHROPIC_AWS_API_KEY`가 SigV4보다 우선하므로 stale 키도 동일 에러 — 재생성하거나 unset.
- **missing-workspace 에러**: `ANTHROPIC_AWS_WORKSPACE_ID`가 unset/빈 값. AWS 자격증명이 workspace ID를 암시하지 않으므로 명시 export 필요.
- **요청이 여전히 `api.anthropic.com`으로**: `CLAUDE_CODE_USE_ANTHROPIC_AWS`가 unset/비truthy. `1`로 설정 후 `/status` 확인. `CLAUDE_CODE_USE_BEDROCK`/`_FOUNDRY`가 우선함을 주의.

## Google Vertex AI (google-vertex-ai)

GCP-native 배포 경로.

### 사전 요구 사항
- 결제 활성화된 GCP 계정, Vertex AI API 활성 프로젝트
- 원하는 Claude 모델 접근, `gcloud` 설치·구성, 원하는 리전의 할당량(quota)

### Sign in with Vertex AI (로그인 위저드)
GCP 자격증명이 있으면 위저드가 Claude Code 측을 처리. **위저드는 v2.1.98 이상 필요.**

1. **GCP 프로젝트에서 Claude 모델 활성화**: Vertex AI API 활성화 후 [Model Garden](https://console.cloud.google.com/vertex-ai/model-garden)에서 접근 요청.
2. **`claude` 실행** → **3rd-party platform** → **Google Vertex AI** 선택.
3. **위저드 진행**: GCP 인증 방식 선택(`gcloud`의 ADC, 서비스 계정 키 파일, 환경의 기존 자격증명). 프로젝트·리전 감지, 호출 가능 모델 검증·핀, 결과를 user settings `env` 블록에 저장.

로그인 후 `/setup-vertex`로 재구성.

### 리전 구성
Vertex AI는 global/multi-region/regional 엔드포인트를 지원. `CLOUD_ML_REGION`을 `global`, multi-region(`eu`/`us`), 또는 특정 리전(`us-east5`)으로 설정. Claude Code가 각 형태에 맞는 호스트네임을 선택(multi-region은 `aiplatform.eu.rep.googleapis.com`·`aiplatform.us.rep.googleapis.com`).

> 모델 가용성은 특정 리전·multi-region·global 엔드포인트마다 다르다. 지원 위치로 전환하거나 지원 모델을 지정해야 할 수 있음.

### 수동 설정

#### 1. Vertex AI API 활성화
```bash
gcloud config set project YOUR-PROJECT-ID
gcloud services enable aiplatform.googleapis.com
```

#### 2. 모델 접근 요청
Model Garden에서 "Claude" 검색 → 원하는 모델 접근 요청 → 승인 대기(24-48시간 소요 가능).

#### 3. GCP 자격증명 구성
표준 Google Cloud 인증 사용. v2.1.121 이상은 동일 ADC 체인을 통한 [X.509 인증서 기반 Workload Identity Federation](https://cloud.google.com/iam/docs/workload-identity-federation-with-x509-certificates)을 지원 — `GOOGLE_APPLICATION_CREDENTIALS`를 credential configuration 파일 경로로 설정.

> 프로젝트 ID 우선순위: `GOOGLE_APPLICATION_CREDENTIALS`가 가리키는 파일 / `GCLOUD_PROJECT` / `GOOGLE_CLOUD_PROJECT`가 `ANTHROPIC_VERTEX_PROJECT_ID`보다 우선. 모두 없으면 `gcloud` 설정 또는 첨부 서비스 계정에서 해석.

##### 고급 자격증명 구성 — `gcpAuthRefresh`
GCP 자격증명이 만료/로드 불가로 감지되면 구성한 명령을 실행해 재시도.

```json
{
  "gcpAuthRefresh": "gcloud auth application-default login",
  "env": { "ANTHROPIC_VERTEX_PROJECT_ID": "your-project-id" }
}
```

명령 출력은 표시(대화형 입력 불가). 인증 미완료 시 3분 후 타임아웃. 프로젝트 설정(`.claude/settings.json`)에 두면 workspace trust 프롬프트 수락 후에만 실행.

#### 4. Claude Code 구성
```bash
export CLAUDE_CODE_USE_VERTEX=1
export CLOUD_ML_REGION=global
export ANTHROPIC_VERTEX_PROJECT_ID=YOUR-PROJECT-ID

# 선택: 커스텀 엔드포인트/게이트웨이용 Vertex 엔드포인트 URL 오버라이드
# export ANTHROPIC_VERTEX_BASE_URL=https://aiplatform.googleapis.com

# 선택: 프롬프트 캐싱 비활성화
export DISABLE_PROMPT_CACHING=1
# 선택: 1시간 캐시 TTL
export ENABLE_PROMPT_CACHING_1H=1

# CLOUD_ML_REGION=global일 때, global 엔드포인트 미지원 모델용 리전 오버라이드
export VERTEX_REGION_CLAUDE_HAIKU_4_5=us-east5
export VERTEX_REGION_CLAUDE_4_6_SONNET=europe-west1
```

대부분의 모델 버전에 대응하는 `VERTEX_REGION_CLAUDE_*` 변수가 있다(전체 목록은 환경변수 레퍼런스). Model Garden에서 어떤 모델이 global을 지원하는지 확인.

추가 메모:
- 프롬프트 캐싱 자동 활성. 비활성화 `DISABLE_PROMPT_CACHING=1`, 1시간 TTL `ENABLE_PROMPT_CACHING_1H=1`(높은 요율). rate limit 상향은 Google Cloud support에.
- Vertex AI에서 `/logout`은 사용 불가.
- **MCP tool search**는 Vertex AI에서 기본 비활성(MCP 도구 정의 upfront 로드). Sonnet 4.5+·Opus 4.5+에서만 지원하며 `ENABLE_TOOL_SEARCH=true`로 활성. 이전 모델은 필수 beta 헤더를 받지 못해 활성 시 요청 실패. → [[10 MCP]]

#### 5. 모델 버전 핀

> [!warning] 멀티유저 배포 시 반드시 핀 (Bedrock과 동일한 폴백 메커니즘)

`ANTHROPIC_DEFAULT_OPUS_MODEL` 없으면 `opus`는 Vertex에서 Opus 4.6으로 풀림:

```bash
export ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-8'
export ANTHROPIC_DEFAULT_SONNET_MODEL='claude-sonnet-4-6'
export ANTHROPIC_DEFAULT_HAIKU_MODEL='claude-haiku-4-5@20251001'
```

| Model type | Default value |
| :-- | :-- |
| Primary model | `claude-sonnet-4-5@20250929` |
| Small/fast model | Primary model과 동일 |

배경 작업은 Haiku급이지만 Vertex에서는 프로젝트/리전에 미활성일 수 있어 primary로 폴백. 추가 커스터마이즈:

```bash
export ANTHROPIC_MODEL='claude-opus-4-8'
export ANTHROPIC_DEFAULT_HAIKU_MODEL='claude-haiku-4-5@20251001'
```

### Startup model checks
Vertex 구성으로 시작하면 모델 접근성 검증(**v2.1.98 이상 필요**). 오래된 핀이고 새 버전 호출 가능하면 핀 업데이트 프롬프트(수락 시 settings에 기록·재시작, 거절은 다음 기본 변경까지 기억). 미핀+기본값 미활성이면 현 세션만 이전 버전 폴백(비영구). Model Garden에서 활성화하거나 핀으로 영구화.

### IAM 구성
`roles/aiplatform.user` 롤이 필요 권한 포함:
- `aiplatform.endpoints.predict` — 모델 호출·토큰 카운트에 필요

더 제한적으로는 위 권한만의 커스텀 롤. 비용 추적·접근 제어 단순화를 위해 Claude Code 전용 GCP 프로젝트 권장.

### 1M 토큰 컨텍스트 윈도우
Opus 4.6 이상, Sonnet 4.6은 Vertex AI에서 1M 지원. 1M 변형 선택 시 자동 활성. 위저드는 핀 시 1M 옵션 제공, 수동 핀에는 `[1m]` 덧붙임.

### Vertex AI 트러블슈팅
- **"Could not load the default credentials"**: `gcloud auth application-default login`으로 ADC 설정 / `GOOGLE_APPLICATION_CREDENTIALS`를 서비스 계정 키 경로로.
- **할당량 이슈**: Cloud Console에서 quota 확인/증액 요청.
- **"model not found" 404**: Model Garden에서 모델 Enabled 확인 / 지정 위치에서 가용한지 확인(일부는 `global`/`eu`/`us`만) / `CLOUD_ML_REGION=global` 시 global 지원 확인 후 미지원이면 `ANTHROPIC_MODEL`/`ANTHROPIC_DEFAULT_HAIKU_MODEL`로 지원 모델 지정하거나 `VERTEX_REGION_<MODEL_NAME>`로 리전 설정.
- **429 에러**: regional 엔드포인트에서 primary·small/fast 모델이 리전에 지원되는지 확인 / 가용성을 위해 `CLOUD_ML_REGION=global` 전환 고려.

## Microsoft Foundry (microsoft-foundry)

Azure-native 배포 경로. **다른 프로바이더와 달리 대화형 setup 위저드가 없고**, 환경변수 설정이 유일한 구성 경로다. **시작 시 모델 체크가 없어** 기본값이 미활성이면 요청이 실패한다.

### 사전 요구 사항
- Microsoft Foundry 접근 권한이 있는 Azure 구독
- Foundry 리소스·배포를 생성할 RBAC 권한
- Azure CLI(선택)

> 멀티유저 배포 시 롤아웃 전에 모델 버전 핀.

### Setup

#### 1. Microsoft Foundry 리소스 프로비저닝
[Foundry 포털](https://ai.azure.com/) → 새 리소스 생성(리소스 이름 기록) → Claude 모델 배포 생성(Opus / Sonnet / Haiku).

#### 2. Azure 자격증명 구성 — 두 가지 인증

**Option A: API 키 인증**
리소스 → **Endpoints and keys** → **API Key** 복사 → 환경변수:
```bash
export ANTHROPIC_FOUNDRY_API_KEY=your-azure-api-key
```

**Option B: Microsoft Entra ID 인증**
`ANTHROPIC_FOUNDRY_API_KEY`가 없으면 Azure SDK [default credential chain](https://learn.microsoft.com/en-us/azure/developer/javascript/sdk/authentication/credential-chains#defaultazurecredential-overview)을 자동 사용. 로컬에서는 보통:
```bash
az login
```

> Foundry 사용 시 `/logout`은 사용 불가.

#### 3. Claude Code 구성
```bash
export CLAUDE_CODE_USE_FOUNDRY=1

# Azure 리소스 이름 ({resource}를 리소스 이름으로)
export ANTHROPIC_FOUNDRY_RESOURCE={resource}
# 또는 전체 base URL 제공:
# export ANTHROPIC_FOUNDRY_BASE_URL=https://{resource}.services.ai.azure.com/anthropic
```

#### 4. 모델 버전 핀

> [!warning] 모든 배포에서 핀
> 핀이 없으면 alias가 빌트인 기본값으로 풀리며, **Foundry는 시작 시 모델 체크가 없어 기본값이 미활성이면 요청이 실패**한다. Azure 배포 생성 시 "auto-update to latest"가 아니라 특정 버전을 선택하라.

step 1에서 만든 배포 이름과 일치하게 설정. `ANTHROPIC_DEFAULT_OPUS_MODEL` 없으면 `opus`는 Opus 4.6으로:

```bash
export ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-8'
export ANTHROPIC_DEFAULT_SONNET_MODEL='claude-sonnet-4-6'
export ANTHROPIC_DEFAULT_HAIKU_MODEL='claude-haiku-4-5'
```

배경 작업은 Haiku급이나 모든 계정에 Haiku 배포가 있지 않아 primary로 폴백. Haiku를 쓰려면 위처럼 가용한 Haiku 배포로 설정. 프롬프트 캐싱 자동 활성, 1시간 TTL은 `ENABLE_PROMPT_CACHING_1H=1`.

#### 5. Claude Code 실행
```bash
claude
```
환경변수를 읽어 첫 프롬프트에서 Azure 리소스에 연결. (위저드 없음.)

### Azure RBAC 구성
`Azure AI User`와 `Cognitive Services User` 기본 롤이 Claude 모델 호출 권한 전부 포함. 더 제한적으로는 커스텀 롤:

```json
{
  "permissions": [
    { "dataActions": ["Microsoft.CognitiveServices/accounts/providers/*"] }
  ]
}
```

### Foundry 트러블슈팅
- **"Failed to get token from azureADTokenProvider: ChainedTokenCredential authentication failed"**: 환경에 Entra ID 구성, 또는 `ANTHROPIC_FOUNDRY_API_KEY` 설정.

## LLM 게이트웨이 구성 (llm-gateway)

LLM 게이트웨이는 Claude Code와 모델 프로바이더 사이의 중앙 프록시 레이어로, 흔히 다음을 제공한다: **중앙 인증**(단일 API 키 관리 지점), **사용량 추적**(팀·프로젝트별), **비용 제어**(예산·rate limit), **감사 로깅**(컴플라이언스), **모델 라우팅**(코드 변경 없이 프로바이더 전환).

### 게이트웨이 요구 사항

#### API 형식
게이트웨이는 다음 중 최소 하나를 클라이언트에 노출해야 한다.

| 형식 | 엔드포인트 | 헤더/바디 요구 |
| :-- | :-- | :-- |
| Anthropic Messages | `/v1/messages`, `/v1/messages/count_tokens` | 요청 헤더 forward: `anthropic-beta`, `anthropic-version` |
| Bedrock InvokeModel | `/invoke`, `/invoke-with-response-stream` | 요청 바디 필드 보존: `anthropic_beta`, `anthropic_version` |
| Vertex rawPredict | `:rawPredict`, `:streamRawPredict`, `/count-tokens:rawPredict` | 요청 헤더 forward: `anthropic-beta`, `anthropic-version` |

헤더 미forward/바디 필드 미보존 시 기능 저하 또는 사용 불가. Claude Code는 API 형식으로 어떤 기능을 켤지 판단한다. Bedrock/Vertex에 Anthropic Messages 형식을 쓸 때는 `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1`이 필요할 수 있다.

#### 요청 헤더 (프록시가 활용 가능)

| Header | 설명 |
| :-- | :-- |
| `X-Claude-Code-Session-Id` | 현 세션의 고유 식별자. 바디 파싱 없이 단일 세션의 모든 요청 집계에 사용 |
| `X-Claude-Code-Agent-Id` | 요청을 낸 서브에이전트/teammate 식별자. 세션 내 병렬 서브에이전트별 비용 귀속에. in-process 서브에이전트/teammate 요청에만 존재 |
| `X-Claude-Code-Parent-Agent-Id` | 요청 에이전트를 스폰한 에이전트의 식별자. 중첩 에이전트 비용 귀속에. 요청 에이전트가 다른 에이전트에 의해 스폰된 경우만 존재 |

두 agent ID 헤더는 spawn마다의 일시적 식별자(영속 user/device ID 아님). → [[08 서브에이전트와 에이전트 팀]]

Claude Code는 시스템 프롬프트에 클라이언트 버전·대화 fingerprint로 된 짧은 attribution 블록도 prepend한다. Anthropic API는 처리 전 이를 제거하므로 first-party 프롬프트 캐싱에 영향 없음. 게이트웨이가 전체 요청 바디 기준 자체 캐시를 쓴다면 `CLAUDE_CODE_ATTRIBUTION_HEADER=0`으로 생략.

### Configuration — 모델 선택
기본적으로 선택된 API 형식의 표준 모델 이름을 쓴다. `ANTHROPIC_BASE_URL`이 Anthropic Messages 형식 게이트웨이를 가리키면, 시작 시 게이트웨이의 `/v1/models`를 조회해 반환 모델을 `/model` picker에 추가할 수 있다. `CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY=1`로 활성(기본 비활성 — 공유 API 키 게이트웨이가 모든 모델을 노출하지 않게). **v2.1.129 이상 필요.**

- discovery는 Anthropic Messages 형식에만 적용(Bedrock/Vertex pass-through·미설정·`api.anthropic.com`에는 미적용).
- 인증은 추론 요청과 동일(`ANTHROPIC_AUTH_TOKEN` bearer, 없으면 `ANTHROPIC_API_KEY`를 `x-api-key`, 그리고 `ANTHROPIC_CUSTOM_HEADERS`).
- ID가 `claude` 또는 `anthropic`으로 시작하는 모델만 추가. 결과는 `~/.claude/cache/gateway-models.json`에 캐시되고 매 시작 시 갱신. 실패/미구현 시 직전 캐시 또는 빌트인 목록으로 폴백.
- discovery 필터에 안 맞는 이름은 [Model configuration](/en/model-config)의 환경변수로 수동 추가.

### LiteLLM 구성

> [!warning] LiteLLM 보안 경고
> LiteLLM PyPI **1.82.7·1.82.8 버전은 자격증명 탈취 악성코드에 감염**됐다. 설치 금지. 이미 설치했다면: 패키지 제거 → 영향 시스템의 모든 자격증명 로테이션 → [BerriAI/litellm#24518](https://github.com/BerriAI/litellm/issues/24518)의 조치 따르기. LiteLLM은 서드파티 프록시로 Anthropic이 보증·유지·감사하지 않는다.

## 엔터프라이즈 네트워크 설정 (network-config)

프록시 서버·커스텀 CA·mTLS 등 기업 네트워크/보안 구성. **이 페이지의 모든 환경변수는 [`settings.json`](/en/settings)에도 설정 가능**(단 `CLAUDE_CODE_CERT_STORE` 제외, 아래 참고).

### 프록시 구성
표준 프록시 환경변수를 존중한다. **SOCKS 프록시는 미지원.**

```bash
export HTTPS_PROXY=https://proxy.example.com:8080   # 권장
export HTTP_PROXY=http://proxy.example.com:8080     # HTTPS 불가 시

# 특정 요청에 프록시 우회 (공백 구분)
export NO_PROXY="localhost 192.168.1.1 example.com .example.com"
# 쉼표 구분도 가능
export NO_PROXY="localhost,192.168.1.1,example.com,.example.com"
# 모든 요청 우회
export NO_PROXY="*"
```

#### Basic 인증
프록시가 basic 인증을 요구하면 프록시 URL에 자격증명을 포함한다. 형식은 스킴 뒤에 사용자명, 콜론, 비밀번호, `@`, 그다음 호스트와 포트 순서다(즉 `http://` 스킴 다음에 `사용자명:비밀번호` 그리고 `@proxy.example.com:8080`을 이어 붙인다). 실제 값으로는 `사용자명`·`비밀번호` 자리에 자신의 프록시 계정 정보를 넣어 `HTTPS_PROXY`에 export한다.
> 스크립트에 비밀번호 하드코딩 금지 — 환경변수/안전한 자격증명 저장 사용. NTLM·Kerberos 등 고급 인증 프록시는 해당 인증을 지원하는 LLM Gateway 사용 고려.

### CA 인증서 저장소
기본적으로 Claude Code는 번들된 Mozilla CA 인증서 + OS 인증서 저장소를 모두 신뢰한다. CrowdStrike Falcon·Zscaler 같은 TLS-inspection 프록시는 루트 인증서가 OS trust store에 설치돼 있으면 추가 구성 없이 동작.

`CLAUDE_CODE_CERT_STORE`는 쉼표 구분 소스 목록. 인식 값: `bundled`(번들 Mozilla CA), `system`(OS trust store). 기본은 `bundled,system`.

```bash
export CLAUDE_CODE_CERT_STORE=bundled   # 번들 Mozilla만
export CLAUDE_CODE_CERT_STORE=system    # OS 저장소만
```

> `CLAUDE_CODE_CERT_STORE`는 전용 `settings.json` 스키마 키가 없다. `~/.claude/settings.json`의 `env` 블록 또는 프로세스 환경에서 설정.

### 커스텀 CA 인증서
```bash
export NODE_EXTRA_CA_CERTS=/path/to/ca-cert.pem
```

### mTLS 인증
클라이언트 인증서 인증이 필요한 환경:
```bash
export CLAUDE_CODE_CLIENT_CERT=/path/to/client-cert.pem
export CLAUDE_CODE_CLIENT_KEY=/path/to/client-key.pem
export CLAUDE_CODE_CLIENT_KEY_PASSPHRASE="your-passphrase"  # 암호화된 키일 때(선택)
```

### 네트워크 접근 요구 사항 (allowlist)
Claude Code는 다음 URL 접근이 필요하다. 프록시·방화벽(특히 컨테이너/제한 환경)에서 allowlist.

| URL | Required for |
| :-- | :-- |
| `api.anthropic.com` | Claude API 요청 |
| `claude.ai` | claude.ai 계정 인증 |
| `platform.claude.com` | Anthropic Console 계정 인증 |
| `downloads.claude.ai` | 플러그인 실행파일 다운로드; 네이티브 설치/자동 업데이트 |
| `storage.googleapis.com` | 2.1.116 이전 버전의 네이티브 설치/자동 업데이트(max-version 2.1.115) |
| `bridge.claudeusercontent.com` | Claude in Chrome 확장 WebSocket bridge |
| `raw.githubusercontent.com` | `/release-notes` changelog 피드·업데이트 후 릴리스 노트; 플러그인 마켓플레이스 install counts |

- npm 설치/자체 바이너리 배포면 `downloads.claude.ai`·`storage.googleapis.com`는 불필요할 수 있다.
- 선택적 운영 텔레메트리도 기본 전송됨 — 환경변수로 비활성 가능. allowlist 확정 전 [Telemetry services](/en/data-usage#telemetry-services) 참고.
- **Bedrock/Vertex AI/Foundry 사용 시** 모델 트래픽·인증은 프로바이더로 가고 `api.anthropic.com`/`claude.ai`/`platform.claude.com`에는 가지 않는다. 단 WebFetch 도구는 domain safety check 때문에 `api.anthropic.com`을 여전히 호출(settings에서 `skipWebFetchPreflight: true`로 끌 수 있음). → [[06 내장 도구 레퍼런스]]
- **Claude Code on the web**·**Code Review**는 Anthropic-managed 인프라에서 레포에 접속. GitHub Enterprise Cloud가 IP 제한이면 설치된 GitHub Apps의 IP allow list inheritance 활성(Claude GitHub App이 IP 범위 등록). 수동 추가나 GHES(self-hosted)는 [Anthropic API IP addresses](https://platform.claude.com/docs/en/api/ip-addresses)를 allowlist. → [[15 웹과 모바일]] · [[16 CI-CD와 팀 통합]]
