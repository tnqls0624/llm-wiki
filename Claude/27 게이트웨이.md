# 27 게이트웨이

**게이트웨이**는 조직이 Claude Code와 모델 프로바이더 사이에 두는 프록시다. Claude Code는 프로바이더로 직접 가지 않고 게이트웨이로 트래픽을 보내며, 게이트웨이는 조직이 보유한 자격증명으로 이를 프로바이더에 전달한다. 개발자는 프로바이더 자격증명을 들고 있지 않고 게이트웨이에만 인증하므로, 인증·사용량 추적·예산·감사 로깅이 한 곳에 모인다. 이 노트는 Anthropic이 `claude` 바이너리에 내장해 제공하는 **Claude apps gateway**(자체 셀프호스팅 게이트웨이)와, 조직이 이미 운영 중인 **서드파티/자체 운영 LLM 게이트웨이**를 Claude Code에 연결하는 프로토콜·개발자 접속·조직 롤아웃을 함께 다룬다. 서드파티 게이트웨이를 Claude Code 환경변수 레벨에서 붙이는 가장 흔한 설정(LiteLLM 등)은 [[17 클라우드 프로바이더]]에 있다.

허브: [[Claude]] · 관련: [[17 클라우드 프로바이더]](서드파티 게이트웨이 환경변수 설정·LiteLLM) · [[04 설정]](managed settings 파일 경로) · [[05 권한]](managed settings 정책) · [[18 보안과 샌드박스]] · [[19 비용과 성능]](OTLP·비용 추적) · [[09 훅]](managed 정책의 org-wide hooks)

> [!tip] 핵심 멘탈 모델
> 두 가지 credential이 관여한다 — **개발자 credential**(게이트웨이가 발급, 개발자를 인증·식별)과 **프로바이더 credential**(게이트웨이가 보유, 전달되는 모든 트래픽이 공유). Claude apps gateway는 조직 IdP로 개발자를 로그인시키고 IdP 그룹별로 모델 접근·managed settings를 강제하는 반면, 서드파티 게이트웨이는 Claude Code가 보내는 헤더·바디 필드를 프로토콜 레퍼런스대로 그대로 forward해야 기능이 깨지지 않는다.

## 게이트웨이 개요 (gateways)

각 개발자의 Claude Code는 게이트웨이 주소를 향하고 게이트웨이 발급 credential로 인증한다. 게이트웨이는 개발자를 인증하고, 설정된 접근·예산 규칙을 적용하고, 조직의 credential로 프로바이더(Anthropic API 또는 [[17 클라우드 프로바이더]]의 Amazon Bedrock·Google Cloud·Microsoft Foundry)에 요청을 전달한다. Claude apps gateway처럼 단일 Anthropic 형식 엔드포인트를 노출하는 게이트웨이라면, 프로바이더를 바꿔도 개발자 머신을 건드릴 필요가 없다.

### 게이트웨이 선택
- **Claude apps gateway**: Anthropic 자체 게이트웨이. `claude` 바이너리에 포함되어 매 릴리스와 함께 빌드·테스트되므로 헤더·필드 forwarding 목록을 따로 관리할 필요가 없다. 조직 IdP로 `/login` SSO, IdP 그룹별 모델 접근·[managed settings](/en/permissions#managed-settings), 자체 관측 스택으로 OTLP 사용량 전송을 제공. 로그인이 브라우저 SSO 단계라 서비스 토큰 흐름이 없으므로, 승인할 개발자가 없는 CI 파이프라인은 이 경로로 인증할 수 없다(프로바이더에 직접 구성).
- **다른 게이트웨이**: 조직이 이미 LLM/API 게이트웨이를 운영 중이면 그대로 쓸 수 있다. Anthropic은 서드파티 게이트웨이 제품을 보증·유지·감사하지 않으며, 게이트웨이를 통한 non-Claude 모델 라우팅은 지원하지 않는다.

### 구독과 게이트웨이
게이트웨이 credential로 연결하면 사용량은 조직의 프로바이더 계정에 API 요율로 과금되고 claude.ai 구독은 쓰이지 않는다. `ANTHROPIC_AUTH_TOKEN` 설정이나 Claude apps gateway `/login`은 해당 세션의 구독 로그인을 끈다. 예외는 게이트웨이 credential 없이 `ANTHROPIC_BASE_URL`만 설정한 경우 — 요청은 게이트웨이를 거치지만 저장된 claude.ai 로그인이 활성 credential로 남아 구독의 사용량 한도·과금이 적용된다.

### 게이트웨이와 별도로 설정되는 것
- **어떤 모델이 응답할지**: `/model` 커맨드나 [model 환경변수](/en/model-config)로 고른다. Claude apps gateway는 그룹별 `availableModels` allowlist로 선택 범위를 제한할 수 있지만 그 안에서는 개발자가 고른다.
- **다른 네트워크 트래픽**: 버전 체크·다운로드는 게이트웨이 경로와 별개로 Anthropic에 직접 간다. `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC`으로 선택적 스트림을 끌 수 있다.
- **기업 HTTP 프록시**: `HTTPS_PROXY`는 게이트웨이를 포함한 모든 서버와의 사이에 낀다. Claude apps gateway는 로그인 시 프록시 호스트도 사설망인지 검사하므로, 아니면 `NO_PROXY`에 게이트웨이 호스트를 추가해 직접 연결하게 한다.

## Claude apps gateway — 개요와 퀵스타트 (claude-apps-gateway)

> [!note] Claude for Enterprise와의 선택
> 데이터 레지던시 등으로 자체 인프라를 통해 추론을 라우팅해야 하는 조직을 위한 것이다. 그런 요구가 없고 SCIM 프로비저닝이나 Claude Code on web/mobile 같은 기능이 필요하면 Claude Enterprise가 더 맞을 수 있다 — [[17 클라우드 프로바이더]]의 기능 가용성 표 참고.

Claude apps gateway는 개발자의 Claude Code와 모델 프로바이더 사이에 있는 셀프호스팅 서비스다. 개발자는 API 키·클라우드 credential 대신 조직 IdP로 로그인한다. `claude gateway --config gateway.yaml`로 같은 바이너리가 서버로도 동작한다.

### 왜 쓰는가
- **Credential**: 상위 API 키/클라우드 credential은 조직 인프라에만 존재. 개발자는 SSO로 인증하고 짧은 수명의 bearer token을 받으므로 오프보딩이 IdP에서 끝난다(기본 세션 수명 1시간).
- **접근 제어**: IdP 그룹 → 모델 allowlist·managed settings 정책 매핑. 모델 접근은 서버 측에서 강제되고, 미허용 모델 요청은 거부된다.
- **설정 배포**: claude.ai admin 콘솔의 [server-managed settings](/en/server-managed-settings) 대신 게이트웨이가 직접 managed settings를 전달.
- **텔레메트리**: 각 목적지(Datadog·Splunk·ClickHouse 등)에 기본적으로 [OTLP 메트릭](/en/monitoring-usage)(토큰 수·모델·사용자 신원·지연), 로그·트레이스는 목적지별 opt-in.
- **업스트림 라우팅**: 클라이언트는 게이트웨이에 Anthropic Messages API로 말하고, 게이트웨이가 Bedrock/Claude Platform on AWS/Google Cloud/Foundry/Anthropic API로 번역·failover.

게이트웨이 자체 데이터 플레인은 Anthropic API가 업스트림으로 설정된 경우가 아니면 Anthropic 인프라로 아무것도 보내지 않는다.

### 퀵스타트
OIDC 클라이언트를 IdP에 등록(리다이렉트 URI `https://<gateway>/oauth/callback`) → PostgreSQL 14+ 준비(부팅 시 자체 스키마 마이그레이션, `CREATE TABLE` 권한 필요) → `gateway.yaml` 작성(5개 필수 섹션) → Docker Compose 또는 `claude gateway --config gateway.yaml`로 실행 → 인증 표면 3단계 확인(discovery document → device authorization → 브라우저 로그인) → 개발자 머신에 `forceLoginMethod: gateway` + `forceLoginGatewayUrl` managed settings 배포 후 `/login`.

> [!warning] 사설망에서만 배포
> Claude Code는 주소가 사설(RFC 1918·CGNAT·IPv6 ULA·loopback)로만 해석되는 게이트웨이에만 연결한다 — 신뢰된 게이트웨이가 개발자 머신에서 명령을 실행하는 설정을 푸시할 수 있기 때문이다. 내부 로드밸런서/VPN 뒤에 두고 사설 IP로만 풀리는 호스트네임을 부여할 것.

전제 조건: Claude Code v2.1.195+(`claude gateway` 서브커맨드), OIDC 준수 IdP(SAML·LDAP 미지원), PostgreSQL 14+, 모델 업스트림, HTTPS, 사설망 주소, **네이티브 Linux 바이너리**(macOS는 로컬 개발용만, Windows 서버 미지원).

### 개발자 연결
개발자는 자기 노트북에서 조직 계정으로 브라우저 SSO 1회로 연결하며 claude.ai 계정·API 키·구독이 필요 없다. CLI는 최초 연결 시 게이트웨이의 TLS leaf 인증서를 지문(fingerprint)으로 고정하며, `/login` 프롬프트가 SHA-256 다이제스트의 앞 16자를 보여준다. 인증서가 갱신되면 모든 개발자가 신뢰 프롬프트를 다시 본다.

`forceLoginMethod`/`forceLoginGatewayUrl`은 [managed settings 파일](/en/settings#settings-files)(MDM 배포)에만 있어야 하며 개발자 자신의 설정 파일에서는 무시된다. CI/원격 머신은 서비스 토큰 흐름이 없어 device flow(SSH로 `/login`, 브라우저에서 인증)를 쓰거나 프로바이더에 직접 구성해야 한다.

**신호인 세션에 강제되는 것**: 모델 접근(정책 밖 모델은 400, `/model` 피커도 필터링), 텔레메트리 목적지 고정, credential(게이트웨이 토큰이 유일 — `ANTHROPIC_AUTH_TOKEN`·`ANTHROPIC_API_KEY`·`apiKeyHelper`·기존 claude.ai 로그인 무시), managed settings(로컬 오버라이드 불가), 시작 시 게이트웨이 unreachable이면 약 10초 후 에러 종료, IdP에서 비활성화된 사용자는 다음 refresh 실패 시 `ttl_hours` 내 만료.

### 가용성과 제약
서버 측 웹 검색은 미지원(게이트웨이가 어느 업스트림으로 라우팅하는지 클라이언트가 알 수 없어 비활성), 표준 프롬프트 캐싱은 지원하나 1시간 TTL은 게이트웨이 세션에서 비활성(모든 업스트림이 지원하지 않을 수 있어 5분 TTL 사용), Auto mode는 opt-in(`CLAUDE_CODE_ENABLE_AUTO_MODE=1`), OTLP는 HTTP만(gRPC 미지원), SAML/LDAP/멀티테넌트(다중 OIDC issuer) 미지원, Windows 서버·Helm 차트·Admin UI 없음(설정은 YAML 파일).

## Claude apps gateway 설정 레퍼런스 (claude-apps-gateway-config)

`gateway.yaml` 하나가 전체 배포를 정의하며 부팅 시 1회 읽고 스키마로 검증한다(오타는 필드 단위 에러로 실패). 시크릿은 `${VAR}`(환경변수) 또는 `${file:/path}`(파일 내용) 형태로 참조하고 직접 쓰지 않는다.

### 필수 섹션

| 섹션 | 역할 |
| :-- | :-- |
| `listen` | 바인드 주소·포트(기본 `0.0.0.0:8080`), `public_url`(TLS 종료 프록시 뒤에서 필수 — IdP `redirect_uri`·discovery document·OTLP 엔드포인트 계산에 쓰임, `X-Forwarded-*` 미신뢰), `tls.cert`/`tls.key`, `trusted_proxies`(로드밸런서 CIDR — 설정해야 실제 클라이언트 IP로 rate limit·audit 기록) |
| `oidc` | IdP 연결: `issuer`·`client_id`/`client_secret`(필수), `allowed_email_domains`/`allowed_groups`, `groups_claim`(기본 `groups`, Entra는 `roles`), `google_groups`(Google은 groups claim이 없어 Admin SDK Directory API 필요), `email_claim`, `scopes`(기본 `[openid, profile, email, offline_access]`), `userinfo_fallback`(Okta org server 등), `use_pkce`(기본 true) |
| `session` | `jwt_secret`(32바이트+, 배열로 로테이션 가능 — index 0이 서명, 전체가 검증), `ttl_hours`(기본 1 — refresh token 없는 IdP면 8~12로) |
| `store` | `postgres_url`(device-grant rendezvous·rate-limit 카운터), `max_connections`(기본 5) |
| `upstreams` | 순서 있는 리스트. `5xx`/`429`/`401`/`403`/`404`/timeout에 다음 업스트림으로 failover(다른 `4xx`는 안 함). 프로바이더별 `auth` 블록(Anthropic `api_key`/`oauth_token`/federation_rule_id, Bedrock 빈 `{}`=AWS 기본 체인, Claude Platform on AWS `api_key`(x-api-key, SigV4보다 우선) 또는 SigV4, Vertex 빈 `{}`=ADC, Foundry `use_azure_ad: true`=DefaultAzureCredential) |

동일 프로바이더를 여러 번 나열하려면 `name:`으로 구분(리전·계정·provisioned throughput vs on-demand·크로스 프로바이더 fallback). `models.upstream_model`로 업스트림별 모델 ID를 매핑해 특정 모델만 특정 업스트림으로 라우팅할 수 있다.

### 선택 섹션
- **`admin`**: `/v1/organizations/spend_limits`와 요청별 지출 집행을 켠다. `write_keys`/`read_keys`(named API key, `x-api-key`)·`admin_groups`(IdP 그룹으로 풀 admin)·`blocked_message`·각종 retention 기간(`audit_retention_days` 365, `spend_retention_months` 13, `identity_retention_days` 90)·`group_limit_mode`(`min`/`max`). → [지출 한도](#지출-한도-claude-apps-gateway-spend-limits) 섹션.
- **`enforcement.fail_closed_on_error`**: Postgres 장애 시 지출 집행을 기본 fail-open에서 fail-closed로.
- **`models` / `auto_include_builtin_models`**: 비-US Bedrock 리전, provisioned-throughput ARN, Foundry 배포 이름처럼 빌트인 카탈로그로 안 되는 경우 admin이 큐레이션한 모델 목록.
- **`managed`**: IdP 그룹/이메일 도메인 기반 정책. `match: {}` catch-all이 베이스 레이어가 되고 나머지는 차이만 오버라이드(allow-list는 완전 교체, deny-list·hook 배열은 합집합, `env`/`modelOverrides`/`skillOverrides`는 shallow merge). `cli` 키는 완전한 `managed-settings.json` 문서 — `availableModels`(게이트웨이+CLI 양쪽 강제), `permissions`, `env`, `hooks` 등. 셀 실행 가능한 설정은 CLI가 1회 승인 다이얼로그를 보여준다(`-p` 헤드리스는 스킵). → [[05 권한]] · [[09 훅]]
- **`telemetry.forward_to`**: OTLP 목적지 배열, 목적지별 `metrics`/`logs`/`traces` opt-in(기본 메트릭만). 게이트웨이가 5개 env var(`CLAUDE_CODE_ENABLE_TELEMETRY` 등)를 클라이언트에 푸시해 OTEL 설정을 대신한다. → [[19 비용과 성능]]
- **HTTP 튜닝**: `access_control`(IP allow/deny), `limits`(요청 크기 상한), `timeouts`(업스트림 TTFB), `rate_limits`(device-authorization 엔드포인트 IP당 제한).

`managed`의 `cli` 정책은 다른 managed 소스(로컬 파일·MDM·policy helper)와 **병합되지 않고** 우선순위가 가장 높은 소스가 전부를 제공한다(정책 helper > 게이트웨이 > MDM > `managed-settings.json` > HKCU). `mcpServers`는 `cli` 블록에서 부팅 시 거부됨 — MCP는 파일 기반 `managed-mcp.json`으로 배포. → [[10 MCP]]

### 클라이언트 측 managed settings
게이트웨이가 스스로 밀어넣을 수 없는 두 키(`forceLoginMethod: "gateway"`, `forceLoginGatewayUrl`)는 각 OS의 managed settings 파일에 직접 배포해야 한다: macOS `/Library/Application Support/ClaudeCode/managed-settings.json`, Linux/WSL `/etc/claude-code/managed-settings.json`, Windows `C:\Program Files\ClaudeCode\managed-settings.json` 또는 HKLM 레지스트리. → [[03 메모리와 컨텍스트]]

## 배포와 운영 (claude-apps-gateway-deploy)

### IdP 설정
컨피덴셜 OAuth/OIDC 웹 애플리케이션을 리다이렉트 URI `https://<gateway>/oauth/callback` 하나로 등록한다. IdP는 discovery 문서 제공·authorization-code flow(PKCE 기본)·`email`/`groups` claim(또는 userinfo fallback)을 지원해야 한다.

- **Okta**: org authorization server(`https://example.okta.com`)는 thin id_token이라 `userinfo_fallback: true` 필요. `groups` scope를 명시 요청해야 그룹이 나온다.
- **Microsoft Entra ID**: `issuer` = `https://login.microsoftonline.com/<tenant-id>/v2.0`. 그룹은 이름이 아닌 GUID(App Roles로 사람이 읽을 이름 사용 가능). `roles` claim이면 `groups_claim: roles`.
- **Google Workspace**: `issuer` = `https://accounts.google.com`. id_token에 그룹이 없어 `oidc.google_groups`(Admin SDK Directory API, domain-wide delegation) 또는 `email_domain` 매칭. `offline_access`도 무시하므로 `scopes: [openid, profile, email]` + `extra_auth_params: { access_type: offline, prompt: consent }`로 refresh token 확보.

### 배포
게이트웨이는 상태 없는(stateless) 단일 Linux 바이너리라 수평 확장이 자유롭다. 비용은 별도 라이선스 없이 `claude` 바이너리에 포함(추론 비용 + 컴퓨트만). 게이트웨이가 유일한 경로임을 강제하려면 네트워크 정책(예: `api.anthropic.com` egress를 게이트웨이에서만 허용)이 별도로 필요 — 이는 [WebFetch domain safety check](/en/data-usage#webfetch-domain-safety-check)도 끊으므로 `skipWebFetchPreflight: true` 필요. → [[06 내장 도구 레퍼런스]] · [[18 보안과 샌드박스]]

- **컨테이너 이미지**: 표준 릴리스에서 native `claude` 바이너리(glibc 빌드 권장) 다운로드 → GPG 서명 `manifest.json`으로 검증 → 빌드 컨텍스트에 복사. 쓰기 가능한 상태 디렉터리(`CLAUDE_CONFIG_DIR`), 커맨드는 `claude gateway --config /etc/claude/gateway.yaml`.
- **Kubernetes**: 상태 없는 Deployment. ConfigMap/Secret 마운트, Ingress에서 TLS 종료 + `listen.public_url`을 Ingress 호스트네임으로, readiness `/readyz`·liveness `/healthz`. 워크로드 아이덴티티(IRSA/GKE Workload Identity/AKS)를 정적 키보다 우선.
- **Cloud Run**: `port: 8080`(또는 `${PORT}`), `min-instances: 1`(콜드 OIDC discovery 방지), config는 secret volume. `public_url`은 보통 내부 로드밸런서 호스트네임(`*.run.app`은 공개 주소라 `/login`의 사설망 검사에 걸림).

### 운영
- **로그**: stderr에 두 스트림 — audit 이벤트(단일 라인 JSON: `config.load`·`session.mint`·`auth.denied`·`inference`·`spend.blocked` 등)와 운영 로그(`[gateway]` 접두사, `CLAUDE_GATEWAY_LOG_LEVEL`로 verbosity).
- **헬스**: `GET /healthz`(liveness)·`GET /readyz`(store 도달성 포함 readiness), 둘 다 `access_control.allow_cidrs` 예외.
- **장애 시**: Postgres 다운 시 기존 세션은 계속 서빙되지만 신규 로그인은 실패(디바이스 플로우가 store 필요). `/readyz`가 not-ready를 보면 오케스트레이터가 replica를 로테이션에서 빼므로, 신호인 개발자를 계속 살리려면 readiness probe를 `/healthz`로 돌리는 트레이드오프도 가능. IdP 다운 시 기존 세션은 `ttl_hours`까지 동작, 신규 로그인·refresh는 실패.
- **JWT 시크릿 로테이션**: 새 시크릿을 배열 앞에 추가 → 롤 → `ttl_hours`+마진 후 이전 시크릿 제거. 시크릿을 통째로 교체하면 모든 세션이 즉시 무효화(개별 세션 revocation은 없음 — 오프보딩은 IdP에서).
- **Postgres 5개 테이블**: `kv`(device grant·rate-limit, TTL 만료), `spend`(기간별 지출 카운터), `spend_limits`(설정된 캡), `admin_audit`(admin API 변경 이력), `principal_emails`(PII — 이메일·이름·그룹, 짧은 retention). DDL 권한 없는 롤이면 스키마를 미리 만들고 DML만 부여.
- **업그레이드**: replica가 stateless라 롤링 재시작 안전. 마이그레이션은 append-only라 롤백도 안전(단 새 릴리스가 도입한 config 키를 쓰면 구버전 스키마 검증에서 부팅 실패).

### 보안
게이트웨이는 조직 경계 안에 있지만 개발자 랩톱은 신뢰하지 않는다: 개발자는 원시 상위 키 대신 단명 JWT를 들고(RFC 8628 device grant + PKCE), device-verification 페이지는 same-origin POST와 IP당 rate limit(8자 20진법 user code, 2.56×10¹⁰ 조합, 10분 만료)을 강제하며, 아웃바운드 요청은 SSRF 가드(링크-로컬·클라우드 메타데이터·loopback 차단, 해석된 IP에 pin)를 거친다. **범위 밖**: 게이트웨이 호스트 자체가 침해된 경우(상위 credential과 managed settings 배포권을 모두 가짐), 악성 OIDC 프로바이더(신원을 임의로 주장 가능). 데이터 흐름 표: 추론·텔레메트리·신원·managed settings·audit log 모두 "Anthropic으로 전송" 열이 "Anthropic API가 업스트림일 때만" 또는 "Never".

## GCP 배포 예시 (claude-apps-gateway-on-gcp)

Google Cloud에서 Vertex(Google Cloud's Agent Platform)를 업스트림으로 하는 워킹 예시(지원되는 프로덕션 아티팩트가 아니라 참고용). Cloud Run 또는 GKE + Cloud SQL for PostgreSQL(private IP) + Secret Manager + `roles/aiplatform.user` 서비스 계정 + 내부 로드밸런서(HTTPS)로 구성.

핵심 결정: 프론트엔드에 따라 `trusted_proxies`가 달라진다 — Cloud Run 직결이면 `169.254.0.0/16`, 내부 ALB + Cloud Run이면 그 값 + 프록시 서브넷 CIDR, GKE 내부 Ingress(`gce-internal`)면 프록시 서브넷 CIDR만. 외부 GKE Ingress(class `gce`)는 공개 주소를 프로비저닝하므로 `/login`의 사설망 검사에 걸려 쓸 수 없다.

Cloud Run 배포는 `--no-invoker-iam-check`(또는 `--allow-unauthenticated`)로 invoker 체크를 열어야 한다 — 게이트웨이 자체 OIDC가 인증을 담당하므로. `*.run.app` URL은 기본적으로 공개 주소로 풀리므로 내부 ALB를 앞에 두거나 Private Service Connect + 내부 DNS 존이 필요하다. GKE는 Workload Identity로 서비스 계정을 바인딩하고, `169.254.169.254`(메타데이터 서버)를 막는 egress NetworkPolicy는 적용하면 안 된다.

Terraform/`setup.sh` 참조 아티팩트가 [anthropics/claude-code 레포](https://github.com/anthropics/claude-code/tree/main/examples/gateway/gcp)에 제공된다.

## 지출 한도 (claude-apps-gateway-spend-limits)

Claude apps gateway는 공유 상위 credential 하나로 모든 추론을 전달하므로, 프로바이더 청구서는 개별 개발자가 아니라 그 credential 전체에 귀속된다. 지출 한도는 그 위에 얹는 개발자별 회로 차단기다.

`admin:` 블록을 켜면 `/v1/organizations/spend_limits` admin API가 열리고 `/v1/messages`마다 실시간 집행된다(캡 자체는 `gateway.yaml`이 아니라 API로 설정). Anthropic 공개 Admin API와 와이어 형태가 같아 그 계약으로 작성한 클라이언트가 base URL만 바꿔 게이트웨이를 타겟팅할 수 있다.

```bash
# 조직 전체 월 $500 기본 캡
curl -sS https://claude-gateway.internal.example.com/v1/organizations/spend_limits \
  -H "x-api-key: $GATEWAY_ADMIN_WRITE_KEY" -H "Content-Type: application/json" \
  -d '{"scope": {"type": "organization"}, "amount": "50000", "period": "monthly"}'
```

| 필드 | 값 |
| :-- | :-- |
| `scope.type` | `user`(OIDC `sub`) / `rbac_group`(IdP 그룹명) / `organization` |
| `amount` | USD 센트 정수 문자열, 또는 `null`(무제한), `"0"`(전면 차단) |
| `period` | `daily`/`weekly`/`monthly` — 스코프당 기간별 1개씩, 어느 하나라도 넘으면 차단 |

캡 해석 순서: 유저 오버라이드 → 가장 제한적인 그룹 캡(`group_limit_mode: max`면 반대) → 조직 기본값 → 무제한. 초과 시 `429 billing_error`(`spend limit reached` + 설정된 `blocked_message`). `count_tokens`는 과금 없어 캡과 무관하게 항상 동작.

집행은 Postgres 2초 타임아웃 조회 1회로 이뤄지며 **기본 fail-open**(store 장애 시 요청 통과, 경고 로그) — `enforcement.fail_closed_on_error: true`로 fail-closed 전환 가능. 가격은 CLI 자체 비용 표시와 같은 테이블을 쓰고, 인식 못 하는 모델 ID는 0원이 아니라 fallback 티어(입력/출력 백만 토큰당 $5/$25)로 과금해 무제한 우회를 막는다. 클라이언트가 스트림을 중단해도 종료 프레임이 없으면 문자 수 기반 보수적 추정치로 과금한다(무과금 우회 방지).

Admin API는 `GET`/`POST`/`DELETE /v1/organizations/spend_limits[/{id}]`, `GET .../effective`(원본별 해석된 캡+기간별 지출), `GET .../audit`(변경 이력)를 제공하며 모든 변경은 `admin_audit`에 트랜잭션으로 기록된다. 개발자 이탈 시 `principal_emails` 행을 직접 삭제(`DELETE FROM principal_emails WHERE principal = '<sub>'`)하면 PII를 즉시 제거할 수 있다(`spend`/`admin_audit`는 익명 OIDC `sub`만 참조하므로 남아도 됨).

## 게이트웨이 프로토콜 레퍼런스 (llm-gateway-protocol)

서드파티/자체 운영 게이트웨이 운영자를 위한 Claude Code ↔ 게이트웨이 계약. Claude apps gateway는 이 계약의 상위집합을 `GET /protocol`로 직접 제공한다.

### API 형식
게이트웨이는 최소 하나를 노출해야 한다.

| 형식 | 선택 변수 | 엔드포인트 | Forward unchanged |
| :-- | :-- | :-- | :-- |
| Anthropic Messages | `ANTHROPIC_BASE_URL` | `/v1/messages`, `/v1/messages/count_tokens`(선택) | `anthropic-beta`·`anthropic-version` 헤더 |
| Amazon Bedrock InvokeModel | `ANTHROPIC_BEDROCK_BASE_URL` + `CLAUDE_CODE_USE_BEDROCK=1` | `/model/{model}/invoke[-with-response-stream]` | `anthropic_beta`·`anthropic_version` 바디 필드 |
| Google Cloud rawPredict | `ANTHROPIC_VERTEX_BASE_URL` + `CLAUDE_CODE_USE_VERTEX=1` | `:rawPredict`, `:streamRawPredict`, `count-tokens:rawPredict`(선택) | 헤더 2종 + `anthropic_version` 바디 필드 |

Foundry·Claude Platform on AWS는 Anthropic Messages 형식을 쓰되 각자 변수(`ANTHROPIC_FOUNDRY_BASE_URL`/`ANTHROPIC_AWS_BASE_URL`)로 라우팅된다. Claude Platform on AWS는 `anthropic-workspace-id` 헤더도 forward 필수. **응답은 반드시 스트리밍**해야 한다(버퍼링하면 클라이언트가 멈춘다).

### 헤더·바디는 열린 목록으로
`anthropic-*` 요청 헤더·바디 필드를 오늘 관찰된 목록으로 allowlist하지 말고 통째로 forward한다 — Claude Code가 릴리스마다 새 beta 값·필드를 추가하므로, 고정된 allowlist는 다음 기능 출시 때 깨진다. 예외는 Bedrock/Vertex처럼 스키마 자체가 다른 업스트림(그건 게이트웨이가 브리징할 몫).

`x-claude-code-session-id`/`-agent-id`/`-parent-agent-id` 헤더는 소비(forward 불필요)해 세션·서브에이전트별 비용 귀속에 쓸 수 있다. → [[08 서브에이전트와 에이전트 팀]]

### 시스템 프롬프트 attribution 블록
Claude Code가 시스템 프롬프트 맨 앞에 클라이언트 버전 + 대화 fingerprint 블록을 붙인다. `api.anthropic.com`은 이를 처리 전 제거(첫 system 블록으로 도착·forward 시)하지만, 다른 업스트림은 그대로 프롬프트에 들어간다. 게이트웨이가 바디 자체 캐시를 쓴다면 `CLAUDE_CODE_ATTRIBUTION_HEADER=0`으로 생략(v2.1.181+는 대화 생애주기 동안 안정적이라 굳이 끌 필요는 줄었다).

### Feature pass-through
헤더-바디 필드 쌍이 함께 다녀야 한다 — 한쪽만 없으면 조용히 기능이 꺼지지만, 헤더 없이 바디만 통과하면 `400`이 난다. Context management·beta 도구 필드·`output_config`(effort·구조화 출력) 모두 이 규칙. `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1`로 프리릴리스 기능 전체를 끌 수 있다(adaptive reasoning은 모델별 선택이라 영향 없음).

### 모델 디스커버리
`CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY=1`(v2.1.129+)이면 시작 시 `GET /v1/models?limit=1000`(3초 타임아웃, 리다이렉트는 실패 처리)로 모델을 조회해 `/model` 피커에 "From gateway"로 추가한다. `id`가 `claude`/`anthropic`으로 시작하는 항목만 채택, 결과는 `~/.claude/cache/gateway-models.json`에 캐시.

## 개발자 접속 (llm-gateway-connect)

개별 개발자가 조직의 게이트웨이에 Claude Code를 붙이는 절차.

### 이미 설정됐는지 확인
`claude` 실행 → 로그인 화면 없이 세션이 열리면 credential이 배포된 상태. `/status`의 **Status** 탭에서 `Anthropic base URL` 줄(게이트웨이 주소)과 `Auth token`/`API key` 줄(어떤 변수가 활성인지)을 확인.

### 직접 설정
1. **credential 변수**: "bearer token"이라 들었으면 `ANTHROPIC_AUTH_TOKEN`, "API key"/`x-api-key`면 `ANTHROPIC_API_KEY`, 로테이션/vault라면 [`apiKeyHelper`](#설정-키-요약)(5분 캐시, `CLAUDE_CODE_API_KEY_HELPER_TTL_MS`로 조정, 401 시 재실행).
2. **base URL**: `ANTHROPIC_BASE_URL`. 셀 export는 임시, `~/.claude/settings.json`(개인 전역) 또는 `.claude/settings.local.json`(프로젝트 — gitignore됨)의 `env` 블록이 영속. **프로젝트의 커밋되는 `.claude/settings.json`에는 credential을 넣지 않는다**(레포 클론한 모두에게 공유됨). → [[04 설정]]
3. **검증**: `curl`로 `$ANTHROPIC_BASE_URL/v1/messages`에 직접 1토큰 요청 — 모델명을 몰라도 인증만 통과하면 credential이 맞다는 뜻(모델 미인식 에러도 통과 증거). `401`이면 변수를 바꿔본다.

### 서피스별 설정
- **VS Code**: 확장 자체 로그인 체크가 `~/.claude/settings.json`이 아니라 VS Code User Settings의 `claudeCode.environmentVariables`를 본다.
- **Desktop**: `ANTHROPIC_BASE_URL`이 아니라 admin이 배포한 별도 구성을 읽는다.
- **GitHub Actions**: `ANTHROPIC_BASE_URL`은 워크플로 `env`에, credential은 액션의 `anthropic_api_key` 입력으로. bearer-token 게이트웨이면 같은 시크릿을 `ANTHROPIC_AUTH_TOKEN`으로도 중복 설정(런치 체크를 만족시키기 위함). → [[16 CI-CD와 팀 통합]]
- **Agent SDK**: SDK 자체엔 게이트웨이 전용 옵션이 없고 스폰되는 프로세스 환경변수로 전달(TypeScript는 `options.env`가 전체 교체라 `process.env` 스프레드 필요, Python은 병합). → [[22 Agent SDK — 시작]]
- **Slack·web·Remote Control**: Anthropic 호스팅 표면은 항상 Anthropic API를 쓰므로 게이트웨이 변수가 안 먹는다. Remote Control·음성 받아쓰기는 게이트웨이 credential이 활성이면 자체가 비활성화된다(claude.ai 신원이 필요해서).

### 추가 설정
- `ANTHROPIC_CUSTOM_HEADERS`로 테넌트/라우팅 헤더 추가.
- Bedrock/Vertex/Foundry/Claude Platform on AWS를 게이트웨이 경유로 붙일 때는 각 프로바이더 base URL 변수 + `CLAUDE_CODE_SKIP_*_AUTH=1`(게이트웨이가 서명 대신 처리) 조합 — 자세한 export 조합은 [[17 클라우드 프로바이더]]의 프록시/게이트웨이 조합 예시와 동일한 패턴.

트러블슈팅 핵심: `400`에 `context_management`/`thinking`류 필드명이 보이면 프리릴리스 기능이 업스트림에 안 맞는 것(`CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1`), 게이트웨이가 좁힌 컨텍스트 한도로 자체 에러 문구를 쓰면 자동 compact-retry가 안 걸리므로 `/compact` 수동 실행 또는 `CLAUDE_CODE_AUTO_COMPACT_WINDOW`로 미리 맞춤, `403`이 게이트웨이 로그에 안 남으면 WAF가 프롬프트의 XSS 패턴(XML 태그·코드)에 걸려 바디 검사를 차단한 것.

## 조직 롤아웃 (llm-gateway-rollout)

관리자가 이미 배포된 서드파티 게이트웨이를 조직에 굴리는 절차. 게이트웨이는 [API 형식](#api-형식) 중 하나를 받고, 스트리밍하고, 모델명을 라우팅하고, 헤더·바디를 그대로 forward하고, 업스트림 에러를 그대로 반환하고, `/v1/messages` 경로를 WAF 바디 검사에서 제외해야 한다.

### 5단계
1. **모델 라우팅 확인**: `<gateway-key>`로 각 모델명에 대해 `curl`로 200 확인(모델당 반복).
2. **개발자별 credential 발급**: 공유 키가 아니라 1인 1키라야 사용량 귀속·개별 오프보딩이 된다.
3. **Claude Code로 실측**: 셀에서 `ANTHROPIC_BASE_URL`+credential export 후 `claude -p "..."`로 게이트웨이 로그에 `POST /v1/messages 200`이 찍히는지 확인.
4. **배포**: [managed settings](/en/settings#settings-files)의 `env` 블록으로 중앙 배포하거나(가능하면 이쪽), 개발자에게 값을 직접 전달. `forceLoginMethod`/`forceLoginOrgUUID`를 게이트웨이 credential과 같은 managed settings에 넣지 않는다(v2.1.146+에서 둘이 공존 불가 — `This machine's managed settings require a first-party login` 에러).
5. **검증**: 개발자 머신에서 스트리밍 요청으로 엔드포인트·streaming·라우팅을 한번에 확인(`data:` 라인이 점진적으로 와야 함 — 한꺼번에 오면 버퍼링 중).

### 유지보수
새 릴리스가 `anthropic-beta`/바디 필드를 추가하면 allowlist 대신 통째 forward로 대응, 새 모델이 나오면 게이트웨이 라우팅 표에 추가, credential 로테이션은 `apiKeyHelper`로 재배포 없이 처리. per-key rate limit 산정 시 클라이언트가 `429`를 최대 10회 backoff 재시도한다는 점을 감안.
