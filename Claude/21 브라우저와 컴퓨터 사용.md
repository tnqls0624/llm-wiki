# 21 브라우저와 컴퓨터 사용

Claude Code가 터미널을 벗어나지 않고도 **브라우저**와 **네이티브 데스크톱 화면**을 직접 조작할 수 있게 해주는 두 가지 기능을 정리한다. 둘 다 "코드를 작성하고 → 같은 대화 안에서 실제로 실행/테스트/디버그"하는 워크플로우를 목표로 한다.

- **Chrome 통합** (`--chrome`): Claude in Chrome 브라우저 확장과 연동하여 웹 앱 테스트, 콘솔 로그 디버깅, 폼 자동 입력, 데이터 추출 등 **브라우저 자동화**를 수행한다.
- **Computer use** (`computer-use` 내장 MCP 서버): macOS에서 앱을 열고, 클릭/타이핑/스크롤하며 화면을 보는 **네이티브 GUI 제어** (research preview).

상위 허브는 [[Claude]]. 관련: 데스크톱 앱의 동일 기능은 [[14 IDE와 데스크톱]], 웹/모바일 환경은 [[15 웹과 모바일]], 샌드박스 비교는 [[18 보안과 샌드박스]], MCP 메뉴 조작은 [[10 MCP]] 참고.

---

## Chrome 통합 — 개요와 동작 방식

Claude Code는 [Claude in Chrome 브라우저 확장](https://chromewebstore.google.com/detail/claude/fcoeoabgfenejglbffodgkkbkcdhcgfn)과 통합되어, CLI 또는 [[14 IDE와 데스크톱]]의 VS Code 확장에서 브라우저 자동화 기능을 제공한다. 코드를 빌드한 뒤 컨텍스트 전환 없이 브라우저에서 바로 테스트/디버그할 수 있다.

핵심 동작 특성:

- Claude는 브라우저 작업을 위해 **새 탭을 열고**, 사용자의 **브라우저 로그인 상태를 공유**한다. 따라서 이미 로그인된 사이트라면 어디든 접근 가능하다 (별도 API 커넥터 불필요).
- 브라우저 액션은 **실시간으로 보이는 Chrome 창**에서 실행된다.
- 로그인 페이지나 CAPTCHA를 만나면 **일시 정지하고 사용자에게 수동 처리를 요청**한다.

> [!note] 베타 / 지원 범위 제한
> Chrome 통합은 베타이며 현재 **Google Chrome과 Microsoft Edge**에서만 동작한다. Brave, Arc 등 다른 Chromium 기반 브라우저는 아직 미지원이며, **WSL(Windows Subsystem for Linux)**도 지원하지 않는다.

### Chrome 통합으로 할 수 있는 일 (Capabilities)

브라우저 액션과 코딩 작업을 하나의 워크플로우로 묶을 수 있다.

| 활용 | 설명 |
| --- | --- |
| **Live debugging** | 콘솔 에러와 DOM 상태를 직접 읽고, 원인이 된 코드를 바로 수정 |
| **Design verification** | Figma 목업으로 UI를 만든 뒤 브라우저에서 열어 일치 여부 검증 |
| **Web app testing** | 폼 검증 테스트, 시각적 회귀 확인, 사용자 플로우 검증 |
| **Authenticated web apps** | Google Docs, Gmail, Notion 등 로그인된 앱을 API 커넥터 없이 조작 |
| **Data extraction** | 웹 페이지에서 구조화된 정보를 뽑아 로컬에 저장 |
| **Task automation** | 데이터 입력, 폼 작성, 다중 사이트 워크플로우 등 반복 작업 자동화 |
| **Session recording** | 브라우저 상호작용을 GIF로 녹화하여 문서화/공유 |

## Chrome 통합 — 사전 요구 사항 (Prerequisites)

## Chrome 통합 — CLI에서 시작하기

### 실행 플래그와 슬래시 명령

```bash
claude --chrome
```

기존 세션 안에서는 `/chrome`를 실행해 활성화할 수도 있다.

| 명령 / 플래그 | 동작 |
| --- | --- |
| `claude --chrome` | Chrome을 활성화한 채로 Claude Code 시작 |
| `/chrome` | 연결 상태 확인, 권한 관리, 확장 재연결, 사용할 브라우저 선택 |
| `/mcp` → `claude-in-chrome` 선택 | 사용 가능한 전체 브라우저 도구 목록 확인 |

`--chrome` 플래그는 [[02 CLI 레퍼런스]]에도 정리되어 있다. 브라우저가 둘 이상 연결되어 있고 브라우저 액션이 시작되면 Claude가 어느 것을 쓸지 묻는다.

브라우저 사용 요청 예시:

```text
Go to code.claude.com/docs, click on the search box,
type "hooks", and tell me what results appear
```

VS Code에서의 브라우저 자동화는 [[14 IDE와 데스크톱]]을 참고한다.

### 기본 활성화 (Enable Chrome by default)

매 세션마다 `--chrome`를 넘기지 않으려면 `/chrome`를 실행해 **"Enabled by default"**를 선택한다. VS Code 확장에서는 Chrome 확장이 설치되어 있으면 별도 플래그 없이 항상 사용 가능하다.

> [!note] 컨텍스트 사용량 트레이드오프
> CLI에서 Chrome을 기본 활성화하면 브라우저 도구가 항상 로드되어 컨텍스트 사용량이 증가한다. 컨텍스트 소비가 늘어난 것이 느껴지면 이 설정을 끄고 필요할 때만 `--chrome`를 쓰는 것이 좋다. 컨텍스트 관리는 [[03 메모리와 컨텍스트]], 비용 영향은 [[19 비용과 성능]] 참고.

### 사이트 권한 관리 (Manage site permissions)

사이트 단위 권한은 Chrome 확장에서 상속된다. Claude가 어떤 사이트를 탐색/클릭/타이핑할 수 있는지는 **Chrome 확장 설정**에서 관리한다. Claude Code 자체의 권한 모델은 [[05 권한]] 참고.

## Chrome 통합 — 예제 워크플로우

`/mcp` → `claude-in-chrome`로 전체 브라우저 도구 목록을 볼 수 있다. 아래는 브라우저 액션을 코딩 작업과 결합하는 대표 패턴이다.

### 로컬 웹 앱 테스트

```text
I just updated the login form validation. Can you open localhost:3000,
try submitting the form with invalid data, and check if the error
messages appear correctly?
```

Claude가 로컬 서버로 이동해 폼을 조작하고 관찰 결과를 보고한다.

### 콘솔 로그로 디버깅

로그가 장황할 수 있으므로 **전체 콘솔 출력을 요청하기보다 찾을 패턴을 명시**하는 것이 좋다.

```text
Open the dashboard page and check the console for any errors when
the page loads.
```

### 폼 자동 입력

```text
I have a spreadsheet of customer contacts in contacts.csv. For each row,
go to the CRM at crm.example.com, click "Add Contact", and fill in the
name, email, and phone fields.
```

로컬 파일을 읽어 웹 인터페이스를 탐색하며 각 레코드를 입력한다.

### Google Docs에 콘텐츠 작성

```text
Draft a project update based on the recent commits and add it to my
Google Doc at docs.google.com/document/d/abc123
```

문서를 열고 에디터를 클릭한 뒤 내용을 타이핑한다. Gmail, Notion, Sheets 등 로그인된 어떤 웹 앱에서도 동작한다.

### 웹 페이지 데이터 추출

```text
Go to the product listings page and extract the name, price, and
availability for each item. Save the results as a CSV file.
```

### 다중 사이트 워크플로우

```text
Check my calendar for meetings tomorrow, then for each meeting with
an external attendee, look up their company website and add a note
about what they do.
```

여러 탭에 걸쳐 정보를 모으고 워크플로우를 완수한다.

### 데모 GIF 녹화

```text
Record a GIF showing how to complete the checkout flow, from adding
an item to the cart through to the confirmation page.
```

상호작용 시퀀스를 녹화해 GIF 파일로 저장한다.

## Chrome 통합 — 트러블슈팅

전반적인 문제 해결은 [[25 트러블슈팅]]도 함께 참고한다.

### 확장이 감지되지 않을 때 (Extension not detected)

Claude Code의 setup-issues 줄에 `chrome`이 표시되면:

1. `chrome://extensions`에서 Chrome 확장이 설치/활성화되었는지 확인
2. `claude --version`으로 Claude Code가 최신인지 확인
3. Chrome이 실행 중인지 확인
4. `/chrome` 실행 후 **"Reconnect extension"** 선택
5. 그래도 안 되면 Claude Code와 Chrome 모두 재시작

Chrome 통합을 처음 활성화할 때 Claude Code는 **native messaging host 설정 파일**을 설치한다. Chrome은 시작 시 이 파일을 읽으므로, 첫 시도에 확장이 감지되지 않으면 Chrome을 재시작해 새 설정을 반영하라.

연결이 계속 실패하면 host 설정 파일이 아래 경로에 존재하는지 확인한다.

**Chrome:**

| OS | 경로 |
| --- | --- |
| macOS | `~/Library/Application Support/Google/Chrome/NativeMessagingHosts/com.anthropic.claude_code_browser_extension.json` |
| Linux | `~/.config/google-chrome/NativeMessagingHosts/com.anthropic.claude_code_browser_extension.json` |
| Windows | 레지스트리 `HKCU\Software\Google\Chrome\NativeMessagingHosts\` 확인 |

**Edge:**

| OS | 경로 |
| --- | --- |
| macOS | `~/Library/Application Support/Microsoft Edge/NativeMessagingHosts/com.anthropic.claude_code_browser_extension.json` |
| Linux | `~/.config/microsoft-edge/NativeMessagingHosts/com.anthropic.claude_code_browser_extension.json` |
| Windows | 레지스트리 `HKCU\Software\Microsoft\Edge\NativeMessagingHosts\` 확인 |

### 브라우저가 응답하지 않을 때 (Browser not responding)

1. **모달 다이얼로그**(alert, confirm, prompt)가 페이지를 막고 있는지 확인. JavaScript 다이얼로그는 브라우저 이벤트를 차단해 Claude가 명령을 받지 못하게 한다. 수동으로 닫은 뒤 계속 진행하라고 지시.
2. Claude에게 새 탭을 만들어 다시 시도하게 한다.
3. `chrome://extensions`에서 확장을 비활성화 후 재활성화하여 재시작.

### 긴 세션 중 연결이 끊길 때 (Connection drops)

확장의 service worker가 장시간 세션 중 idle 상태가 되면 연결이 끊긴다. 비활성 후 브라우저 도구가 멈추면 `/chrome` → **"Reconnect extension"**.

### Windows 특이 문제

- **Named pipe 충돌 (EADDRINUSE)**: 다른 프로세스가 같은 named pipe를 쓰고 있으면 Claude Code 재시작. Chrome을 쓰는 다른 Claude Code 세션을 닫는다.
- **Native messaging host 에러**: native messaging host가 시작 시 크래시하면 Claude Code를 재설치해 host 설정을 재생성.

### 자주 만나는 에러 메시지 (Common error messages)

| Error | Cause | Fix |
| --- | --- | --- |
| "Browser extension is not connected" | Native messaging host가 확장에 닿지 못함 | Chrome과 Claude Code 재시작 후 `/chrome`로 재연결 |
| "Extension not detected" | Chrome 확장이 미설치 또는 비활성화 | `chrome://extensions`에서 확장 설치/활성화 |
| "No tab available" | 탭이 준비되기 전에 Claude가 액션 시도 | 새 탭을 만들고 재시도하도록 요청 |
| "Receiving end does not exist" | 확장 service worker가 idle 상태로 전환됨 | `/chrome` → "Reconnect extension" |

## Computer use — 개요

> [!note] Research preview / 가용성
> Computer use는 **macOS 한정 research preview**이며 **Pro 또는 Max 플랜**이 필요하다. **Team이나 Enterprise 플랜에서는 사용 불가**. **Claude Code v2.1.85 이상**과 **인터랙티브 세션**이 요구되므로, `-p` 플래그를 쓰는 비인터랙티브 모드에서는 쓸 수 없다.

Computer use는 Claude가 앱을 열고, 화면을 제어하며, 사람이 하듯 머신에서 작업하게 해준다. CLI에서 Claude는 Swift 앱을 컴파일하고, 실행하고, 모든 버튼을 클릭하고, 결과를 스크린샷으로 남기는 일을 — 코드를 작성한 바로 그 대화 안에서 — 모두 처리할 수 있다.

이 노트는 **CLI에서의** computer use를 다룬다. macOS/Windows **데스크톱 앱**에서의 동일 기능은 [[14 IDE와 데스크톱]] 참고.

### Computer use로 할 수 있는 일

GUI가 필요한 작업, 즉 평소 터미널을 벗어나 손으로 해야 하는 일을 처리한다.

| 활용 | 설명 |
| --- | --- |
| **네이티브 앱 빌드/검증** | macOS 메뉴바 앱을 만들도록 요청 → Swift 작성, 컴파일, 실행, 모든 컨트롤 클릭하여 동작 확인 |
| **End-to-end UI 테스트** | 로컬 Electron 앱을 가리키며 "온보딩 플로우 테스트" → 앱을 열고 가입을 클릭하며 각 단계 스크린샷. Playwright 설정·테스트 하네스 불필요 |
| **시각/레이아웃 버그 디버깅** | "모달이 작은 창에서 잘린다" → 창 크기 조절로 버그 재현, 스크린샷, CSS 패치, 수정 검증 |
| **GUI 전용 도구 구동** | 디자인 툴, 하드웨어 제어판, iOS Simulator, CLI/API 없는 독점 앱 조작 |

## Computer use — 언제 적용되는가 (도구 우선순위)

Claude는 앱/서비스와 상호작용할 방법이 여러 개다. Computer use는 가장 **광범위하지만 가장 느린** 방식이므로, 가장 정밀한 도구를 먼저 시도한다.

1. 해당 서비스용 [[10 MCP]] 서버가 있으면 → MCP 사용
2. 셸 명령으로 되는 작업이면 → Bash 사용 ([[06 내장 도구 레퍼런스]])
3. 브라우저 작업이고 Claude in Chrome이 설정되어 있으면 → Chrome 사용 (위 Chrome 통합 섹션)
4. 위 어느 것도 해당 안 되면 → computer use 사용

즉 화면 제어는 **다른 무엇으로도 닿을 수 없는 것**(네이티브 앱, 시뮬레이터, API 없는 도구)을 위해 예약된다.

## Computer use — 활성화하기

Computer use는 `computer-use`라는 **내장 MCP 서버**로 제공되며 기본적으로 꺼져 있다.

| 단계 | 동작 |
| --- | --- |
| 1. MCP 메뉴 열기 | 인터랙티브 세션에서 `/mcp` 실행 → 서버 목록에서 `computer-use`(disabled로 표시) 찾기 |
| 2. 서버 활성화 | `computer-use` 선택 → **Enable**. 설정은 **프로젝트별로 영구 저장**되므로 프로젝트당 한 번만 하면 됨 |
| 3. macOS 권한 부여 | 처음 사용 시도 시 두 가지 macOS 권한 프롬프트 표시 |

```text
/mcp
```

### macOS 권한 (3단계 상세)

| 권한 | 용도 |
| --- | --- |
| **Accessibility** | Claude가 클릭, 타이핑, 스크롤 |
| **Screen Recording** | Claude가 화면 내용을 봄 |

프롬프트에는 해당 System Settings 창을 여는 링크가 포함된다. 둘 다 부여한 뒤 프롬프트에서 **Try again**을 선택한다. macOS는 **Screen Recording 부여 후 Claude Code 재시작**을 요구할 수 있다.

설정이 끝나면 GUI가 필요한 작업을 요청한다.

```text
Build the app target, launch it, and click through each tab to make
sure nothing crashes. Screenshot any error states you find.
```

## Computer use — 세션 단위 앱 승인 (Approve apps per session)

`computer-use` 서버를 켰다고 해서 머신의 모든 앱 접근 권한이 부여되는 것은 아니다. 세션에서 특정 앱이 처음 필요할 때 터미널에 프롬프트가 뜨며 다음을 보여준다.

- Claude가 제어하려는 앱
- 클립보드 접근 등 추가로 요청하는 권한
- Claude가 작업하는 동안 숨겨질 다른 앱의 개수

**Allow for this session** 또는 **Deny**를 선택한다. 승인은 **현재 세션 동안** 유지된다. Claude가 여러 앱을 함께 요청하면 한 번에 승인 가능하다.

### Sentinel 경고 — 광범위한 권한 앱

영향 범위가 큰 앱은 프롬프트에 추가 경고를 표시한다.

| Warning | Applies to |
| --- | --- |
| Equivalent to shell access | Terminal, iTerm, VS Code, Warp 등 터미널/IDE |
| Can read or write any file | Finder |
| Can change system settings | System Settings |

이 앱들이 차단되는 것은 아니며, 경고는 해당 수준의 접근이 작업에 합당한지 사용자가 판단하도록 돕는다.

### 앱 카테고리별 제어 수준

Claude의 제어 수준은 앱 카테고리에 따라 다르다.

| 카테고리 | 제어 수준 |
| --- | --- |
| 브라우저, 트레이딩 플랫폼 | **view-only** (보기 전용) |
| 터미널, IDE | **click-only** (클릭만) |
| 그 외 | **full control** (완전 제어) |

전체 티어 분류는 데스크톱 문서의 app permissions를 참고 ([[14 IDE와 데스크톱]]).

## Computer use — 화면에서 작동하는 방식

흐름을 이해하면 Claude의 행동을 예상하고 개입 시점을 잡기 쉽다.

### 한 번에 한 세션 (One session at a time)

Computer use는 활성 동안 **머신 전체 락(machine-wide lock)**을 잡는다. 다른 Claude Code 세션이 이미 컴퓨터를 쓰고 있으면 새 시도는 실패하며, 어느 세션이 락을 쥐고 있는지 알려준다. 그 세션을 끝내거나 종료해야 한다.

### 작업 중 앱이 숨겨짐 (Apps are hidden)

Claude가 화면 제어를 시작하면 다른 보이는 앱들이 숨겨져 Claude는 승인된 앱만 조작한다. **터미널 창은 계속 보이며 스크린샷에서 제외**되므로, 사용자는 세션을 지켜볼 수 있고 Claude는 자신의 출력을 보지 못한다. 턴이 끝나면 숨겨진 앱은 자동 복원된다.

### 스크린샷 자동 다운스케일 (Screenshots are downscaled automatically)

Claude Code는 모델에 보내기 전 모든 스크린샷을 다운스케일한다. Retina 등 고해상도 디스플레이에서도 해상도를 낮추거나 창을 줄일 필요가 없다. 예: 16인치 MacBook Pro 네이티브 Retina 해상도 **3456×2234** → 약 **1372×887**로 다운스케일(종횡비 유지).

목표 크기를 바꾸는 설정은 없다. 다운스케일 후 화면 텍스트/컨트롤이 너무 작아 Claude가 읽기 어렵다면, 디스플레이 해상도를 바꾸지 말고 **앱 안에서 크기를 키워라**.

### 언제든 중단 (Stop at any time)

Claude가 락을 획득하면 macOS 알림이 뜬다: "Claude is using your computer · press Esc to stop."

| 단축키 | 동작 |
| --- | --- |
| `Esc` (어디서든) | 현재 액션 즉시 중단 |
| `Ctrl+C` (터미널) | 동일하게 중단 |

어느 쪽이든 Claude는 락을 해제하고, 앱을 다시 보이게 하고, 제어권을 돌려준다. 작업이 끝나면 두 번째 알림이 뜬다.

## Computer use — 안전과 신뢰 경계 (Safety and the trust boundary)

> [!warning] 샌드박스 Bash와의 차이
> [[18 보안과 샌드박스]]의 샌드박스 Bash 도구와 달리, computer use는 **실제 데스크톱**에서 승인한 앱에 접근하며 동작한다. Claude가 각 액션을 점검하고 화면 콘텐츠로부터의 prompt injection 가능성을 표시하지만, 신뢰 경계가 다르다. 자세한 모범 사례는 [computer use 안전 가이드](https://support.claude.com/en/articles/14128542) 참고.

설정 없이 작동하는 내장 가드레일:

| 가드레일 | 설명 |
| --- | --- |
| **Per-app approval** | 현재 세션에서 승인한 앱만 제어 가능 |
| **Sentinel warnings** | 셸/파일시스템/시스템 설정 접근을 주는 앱은 승인 전에 표시 |
| **Terminal excluded from screenshots** | 터미널 창을 절대 보지 않아, 세션 내 화면 프롬프트가 모델로 되먹임되지 않음 |
| **Global escape** | `Esc`로 어디서든 중단하며, 키 입력이 소비되어 prompt injection이 이를 이용해 다이얼로그를 닫지 못함 |
| **Lock file** | 한 번에 한 세션만 머신 제어 |

## Computer use — 예제 워크플로우

### 네이티브 빌드 검증 (Validate a native build)

```text
Build the MenuBarStats target, launch it, open the preferences window,
and verify the interval slider updates the label. Screenshot the
preferences window when you're done.
```

Claude가 `xcodebuild`를 실행하고, 앱을 띄우고, UI를 조작한 뒤 결과를 보고한다.

### 레이아웃 버그 재현 (Reproduce a layout bug)

```text
The settings modal clips its footer on narrow windows. Resize the app
window down until you can reproduce it, screenshot the clipped state,
then check the CSS for the modal container.
```

### 시뮬레이터 플로우 테스트 (Test a simulator flow)

```text
Open the iOS Simulator, launch the app, tap through the onboarding
screens, and tell me if any screen takes more than a second to load.
```

XCTest 없이 마우스로 하듯 시뮬레이터를 조작한다.

## Computer use — 데스크톱 앱과의 차이 (Differences from the Desktop app)

CLI와 데스크톱은 동일한 computer use 엔진을 공유하지만 몇 가지 차이가 있다.

| Feature | Desktop | CLI |
| --- | --- | --- |
| Platforms | macOS and Windows | macOS only |
| Enable | **Settings > General** (under **Desktop app**) 토글 | `/mcp`에서 `computer-use` 활성화 |
| Denied apps list | Settings에서 설정 가능 | 아직 미지원 |
| Auto-unhide toggle | 선택적 | 항상 켜짐 |
| Dispatch integration | Dispatch가 생성한 세션도 computer use 사용 가능 | 해당 없음 |

## Computer use — 트러블슈팅

### "Computer use is in use by another Claude session"

다른 Claude Code 세션이 락을 쥐고 있다. 그 세션의 작업을 끝내거나 종료하라. 그 세션이 크래시한 경우, Claude가 프로세스가 더 이상 실행 중이 아님을 감지하면 락이 자동 해제된다.

### macOS 권한 프롬프트가 계속 다시 뜸

macOS는 Screen Recording 부여 후 요청 프로세스의 재시작을 요구할 때가 있다. Claude Code를 완전히 종료하고 새 세션을 시작하라. 그래도 계속되면 **System Settings > Privacy & Security > Screen Recording**을 열어 사용 중인 터미널 앱이 목록에 있고 활성화되어 있는지 확인하라.

### `/mcp`에 `computer-use`가 나타나지 않음

이 서버는 적격 환경에서만 나타난다. 다음을 확인하라.

- **macOS인지** — CLI computer use는 Linux/Windows 미지원. Windows에서는 데스크톱의 computer use 사용 ([[14 IDE와 데스크톱]]).
- **Claude Code v2.1.85 이상**인지 — `claude --version`으로 확인.
- **Pro 또는 Max 플랜**인지 — `/status`로 구독 확인.
- **claude.ai로 인증**되었는지 — Amazon Bedrock, Google Cloud Vertex AI, Microsoft Foundry 등 서드파티 프로바이더([[17 클라우드 프로바이더]])에서는 미지원. 서드파티로만 접근한다면 별도 claude.ai 계정 필요.
- **인터랙티브 세션**인지 — `-p` 플래그의 비인터랙티브 모드에서는 미지원.
