# 13 자동화와 스케줄링

Claude Code를 사람이 매 턴 지켜보지 않아도 동작하게 만드는 모든 방법을 모았다. 크게 네 갈래다.

- **헤드리스 실행** (`claude -p`): 스크립트/CI에서 한 번 호출해 결과만 받는다.
- **스케줄링** (`/loop`, Routines, Desktop scheduled tasks): 시간·이벤트 기반 반복 실행.
- **이벤트 푸시 / 원격 제어** (Channels, Remote Control): 외부 이벤트를 세션에 밀어넣거나, 실행 중인 로컬 세션을 다른 기기에서 조종.
- **딥링크** (`claude-cli://`): URL 클릭으로 미리 채워진 프롬프트와 함께 세션을 연다.

## Claude Code를 프로그래밍 방식으로 실행 (headless)

비대화형(non-interactive) 모드는 `-p`(또는 `--print`) 플래그로 켜며, 터미널 UI 없이 한 번 실행하고 결과를 stdout으로 반환한다. CLI를 통한 Agent SDK 사용에 해당하며, 구조화된 출력·도구 승인 콜백·네이티브 메시지 객체가 필요하면 Python/TypeScript SDK([[22 Agent SDK — 시작]] [[23 Agent SDK — 핵심 기능]])를 쓴다.

> [!note] 구독 플랜 크레딧 변경
> 2026년 6월 15일부터 구독 플랜에서 Agent SDK와 `claude -p` 사용량은 대화형 사용 한도와 분리된 **월간 Agent SDK 크레딧**에서 차감된다.

```bash
claude -p "Find and fix the bug in auth.py" --allowedTools "Read,Edit,Bash"
```

모든 [[02 CLI 레퍼런스]] 옵션이 `-p`와 함께 동작한다. 자주 쓰는 것:

| 옵션 | 용도 |
|------|------|
| `--continue` | 가장 최근 대화 이어가기 |
| `--resume <session_id>` | 특정 세션 이어가기 |
| `--allowedTools "..."` | 도구 자동 승인 |
| `--output-format text\|json\|stream-json` | 출력 형식 |
| `--bare` | 자동 디스커버리 생략으로 빠른 시작(아래) |

### bare 모드로 빠르게 시작

`--bare`는 hook·skill·plugin·MCP 서버·auto memory·CLAUDE.md 자동 디스커버리를 모두 건너뛰어 시작 시간을 줄인다. 이 플래그가 없으면 `claude -p`는 대화형 세션과 동일한 컨텍스트(작업 디렉터리나 `~/.claude`에 설정된 모든 것)를 로드한다.

bare 모드의 가치는 **결정성**이다. CI/스크립트에서 동료의 `~/.claude` hook이나 프로젝트 `.mcp.json`의 MCP 서버가 끼어들지 않으므로 어느 머신에서나 같은 결과가 난다. 명시적으로 넘긴 플래그만 적용된다.

```bash
claude --bare -p "Summarize this file" --allowedTools "Read"
```

bare 모드에서 Claude는 Bash·파일 읽기·파일 편집 도구만 갖는다. 필요한 컨텍스트는 플래그로 주입한다:

| 로드할 것 | 플래그 |
|-----------|--------|
| 시스템 프롬프트 추가 | `--append-system-prompt`, `--append-system-prompt-file` |
| 설정 | `--settings <file-or-json>` |
| MCP 서버 | `--mcp-config <file-or-json>` |
| 커스텀 에이전트 | `--agents <json>` |
| 플러그인 | `--plugin-dir <path>`, `--plugin-url <url>` |

bare 모드는 OAuth와 keychain 읽기도 건너뛴다. Anthropic 인증은 `ANTHROPIC_API_KEY` 또는 `--settings`로 넘긴 JSON 안의 `apiKeyHelper`에서 와야 한다(Bedrock/Vertex/Foundry는 각자의 프로바이더 자격증명 사용 — [[17 클라우드 프로바이더]]).

> [!tip] `--bare`는 스크립트/SDK 호출의 권장 모드이며, 향후 릴리스에서 `-p`의 기본값이 될 예정이다.

### 종료 시 백그라운드 작업

`claude -p` 실행 중 [[06 내장 도구 레퍼런스]]의 백그라운드 Bash 작업(dev 서버, watch 빌드 등)을 시작했다면, Claude가 최종 결과를 반환하고 stdin이 닫힌 뒤 **약 5초**가 지나면 그 작업이 종료된다. 이 유예 시간 덕분에 결과 직후 끝나는 작업도 출력을 전달할 수 있다. v2.1.163 이전에는 끝나지 않는 백그라운드 프로세스가 `claude -p` 호출을 무기한 붙잡았다.

### 실전 예시 모음

CI나 기타 스크립트 호출에서는 로컬 설정에 영향받지 않도록 `--bare`를 함께 쓰는 것이 좋다.

**데이터 파이프 통과** — 비대화형 모드는 stdin을 읽으므로 다른 CLI 도구처럼 파이프/리다이렉트가 된다:

```bash
cat build-error.txt | claude -p 'concisely explain the root cause of this build error' > output.txt
```

`--output-format json`을 쓰면 응답 페이로드에 `total_cost_usd`와 모델별 비용 분해가 포함되어, [[19 비용과 성능]] 대시보드 없이 호출당 지출을 추적할 수 있다.

> [!note] v2.1.128부터 파이프 stdin은 **10MB로 제한**된다. 초과 시 명확한 에러와 함께 0이 아닌 종료 코드로 끝난다. 더 큰 입력은 파일로 쓴 뒤 프롬프트에서 경로를 참조하라.

**빌드 스크립트에 Claude를 린터로 끼우기** — diff를 파이프로 넘기면 Claude가 diff를 읽기 위한 Bash 권한이 필요 없다:

```json
{
  "scripts": {
    "lint:claude": "git diff main | claude -p \"you are a typo linter. for each typo in this diff, report filename:line on one line and the issue on the next. return nothing else.\""
  }
}
```

### 구조화된 출력 받기

`--output-format`으로 응답 형식을 제어한다:

- `text`(기본): 평문
- `json`: result·session ID·메타데이터가 담긴 구조화 JSON
- `stream-json`: 실시간 스트리밍용 newline-delimited JSON

```bash
claude -p "Summarize this project" --output-format json
```

특정 스키마를 강제하려면 `--output-format json`에 `--json-schema`와 [JSON Schema](https://json-schema.org/) 정의를 함께 준다. 구조화 출력은 응답의 `structured_output` 필드에 담긴다:

```bash
claude -p "Extract the main function names from auth.py" \
  --output-format json \
  --json-schema '{"type":"object","properties":{"functions":{"type":"array","items":{"type":"string"}}},"required":["functions"]}'
```

`jq`로 파싱하기:

```bash
# 텍스트 결과 추출
claude -p "Summarize this project" --output-format json | jq -r '.result'
# 구조화 출력 추출
claude -p "Extract function names from auth.py" \
  --output-format json \
  --json-schema '{"type":"object","properties":{"functions":{"type":"array","items":{"type":"string"}}},"required":["functions"]}' \
  | jq '.structured_output'
```

### 응답 스트리밍

`--output-format stream-json`을 `--verbose`, `--include-partial-messages`와 함께 쓰면 토큰을 생성되는 대로 받는다. 각 줄은 이벤트를 나타내는 JSON 객체다:

```bash
claude -p "Explain recursion" --output-format stream-json --verbose --include-partial-messages
```

텍스트 델타만 골라 끊김 없이 스트리밍 출력하기(`-r` 원문 문자열, `-j` 개행 없이 결합):

```bash
claude -p "Write a poem" --output-format stream-json --verbose --include-partial-messages | \
  jq -rj 'select(.type == "stream_event" and .event.delta.type? == "text_delta") | .event.delta.text'
```

#### 스트림 이벤트 레퍼런스

API 요청이 재시도 가능한 에러로 실패하면 재시도 전에 `system/api_retry` 이벤트가 나온다:

| 필드 | 타입 | 설명 |
|------|------|------|
| `type` | `"system"` | 메시지 타입 |
| `subtype` | `"api_retry"` | 재시도 이벤트 식별자 |
| `attempt` | integer | 현재 시도 번호(1부터) |
| `max_retries` | integer | 허용된 총 재시도 횟수 |
| `retry_delay_ms` | integer | 다음 시도까지 밀리초 |
| `error_status` | integer\|null | HTTP 상태 코드, 연결 에러면 `null` |
| `error` | string | 에러 분류: `authentication_failed`, `oauth_org_not_allowed`, `billing_error`, `rate_limit`, `overloaded`, `invalid_request`, `model_not_found`, `server_error`, `max_output_tokens`, `unknown` |
| `uuid` | string | 이벤트 식별자 |
| `session_id` | string | 소속 세션 |

`system/init` 이벤트는 세션 메타데이터(모델·도구·MCP 서버·로드된 플러그인)를 보고하며, `CLAUDE_CODE_SYNC_PLUGIN_INSTALL`이 설정되지 않은 한 스트림의 첫 이벤트다. 플러그인 필드로 CI에서 플러그인 미로드를 실패 처리할 수 있다:

| 필드 | 타입 | 설명 |
|------|------|------|
| `plugins` | array | 성공적으로 로드된 플러그인(`name`, `path`) |
| `plugin_errors` | array | 로드 시점 에러(`plugin`, `type`, `message`). 미충족 의존성 버전, `--plugin-dir` 로드 실패 등 포함. 에러가 없으면 키 생략 |

`CLAUDE_CODE_SYNC_PLUGIN_INSTALL` 설정 시 첫 턴 전에 마켓플레이스 플러그인 설치 진행을 나타내는 `system/plugin_install` 이벤트가 나온다:

| 필드 | 타입 | 설명 |
|------|------|------|
| `subtype` | `"plugin_install"` | 식별자 |
| `status` | `"started"`/`"installed"`/`"failed"`/`"completed"` | started·completed가 전체를 감싸고, installed·failed가 개별 마켓플레이스 보고 |
| `name` | string, optional | 마켓플레이스 이름(installed·failed에 존재) |
| `error` | string, optional | 실패 메시지 |

### 도구 자동 승인 / 권한 모드

`--allowedTools`로 특정 도구를 프롬프트 없이 허용한다:

```bash
claude -p "Run the test suite and fix any failures" \
  --allowedTools "Bash,Read,Edit"
```

세션 전체 기준선을 세우려면 개별 도구를 나열하는 대신 [[05 권한]]의 권한 모드를 넘긴다. `dontAsk`는 `permissions.allow` 규칙이나 읽기 전용 명령 집합에 없는 것을 모두 거부 — 잠긴 CI 실행에 유용. `acceptEdits`는 파일 쓰기를 프롬프트 없이 허용하고 `mkdir`/`touch`/`mv`/`cp` 같은 흔한 파일시스템 명령도 자동 승인한다(다른 셸 명령·네트워크 요청은 여전히 `--allowedTools` 항목이나 `allow` 규칙 필요):

```bash
claude -p "Apply the lint fixes" --permission-mode acceptEdits
```

커밋 생성 예시 — `--allowedTools`는 [[04 설정]]의 permission rule 문법을 쓴다. 끝의 ` *`(공백 포함)는 prefix 매칭을 켠다. 공백 없는 `Bash(git diff*)`는 `git diff-index`까지 매치하므로 주의:

```bash
claude -p "Look at my staged changes and create an appropriate commit" \
  --allowedTools "Bash(git diff *),Bash(git log *),Bash(git status *),Bash(git commit *)"
```

> [!note] `/code-review` 같은 사용자 호출 [[07 스킬]]과 [[02 CLI 레퍼런스]]의 내장 명령은 **대화형 모드에서만** 쓸 수 있다. `-p` 모드에서는 원하는 작업을 자연어로 설명하라.

**시스템 프롬프트 커스터마이즈** — `--append-system-prompt`로 기본 동작을 유지하며 지시 추가, `--system-prompt`로 완전 대체:

```bash
gh pr diff "$1" | claude -p \
  --append-system-prompt "You are a security engineer. Review for vulnerabilities." \
  --output-format json
```

**대화 이어가기** — 멀티 호출 워크플로우:

```bash
session_id=$(claude -p "Start a review" --output-format json | jq -r '.session_id')
claude -p "Continue that review" --resume "$session_id"
```

CI 통합은 [[16 CI-CD와 팀 통합]]의 GitHub Actions / GitLab CI/CD 참조.

## 프롬프트를 스케줄에 따라 실행 (`/loop`와 세션 스코프 스케줄링)

> [!note] 스케줄 작업에는 Claude Code **v2.1.72 이상** 필요. `claude --version`으로 확인.

세션 스코프 스케줄링은 프롬프트를 일정 간격으로 자동 재실행한다. 배포 폴링, PR 베이비시팅, 장기 빌드 체크백, 세션 내 알림 등에 쓴다. 이벤트가 발생하는 즉시 반응하려면 폴링 대신 [Channels](#실행-중인-세션에-이벤트-푸시-channels)를, 조건 충족까지 매 턴 계속 작업하려면 `/goal`을 쓴다.

작업은 **세션 스코프**다 — 현재 대화에 살고, 새 대화를 시작하면 멈춘다. `--resume`/`--continue`로 재개하면 만료되지 않은 작업(7일 내 생성된 반복 작업, 실행 시각이 아직 안 지난 일회성)이 복원된다. 세션과 무관하게 살아남는 스케줄링은 [Routines](#routines로-작업-자동화-클라우드), [Desktop scheduled tasks](#desktop-scheduled-tasks-로컬-스케줄), 또는 GitHub Actions([[16 CI-CD와 팀 통합]])를 쓴다.

### 세 가지 스케줄링 옵션 비교 (CANONICAL)

|  | Cloud (Routines) | Desktop | `/loop` |
|--|------------------|---------|---------|
| 실행 위치 | Anthropic cloud | 내 머신 | 내 머신 |
| 머신 켜둬야 하나 | No | Yes | Yes |
| 열린 세션 필요 | No | No | Yes |
| 재시작 후 지속 | Yes | Yes | `--resume` 시 미만료면 복원 |
| 로컬 파일 접근 | No (fresh clone) | Yes | Yes |
| MCP 서버 | 작업별 커넥터 | 설정 파일 + 커넥터 | 세션에서 상속 |
| 권한 프롬프트 | No (자율 실행) | 작업별 설정 가능 | 세션에서 상속 |
| 커스텀 스케줄 | CLI의 `/schedule` 경유 | Yes | Yes |
| 최소 간격 | 1시간 | 1분 | 1분 |

> [!tip] **cloud**는 머신 없이도 안정적으로 돌아야 할 작업, **Desktop**은 로컬 파일·도구가 필요한 작업, **`/loop`**는 세션 중 빠른 폴링.

### `/loop`로 프롬프트 반복 실행

`/loop`는 세션을 열어둔 채 프롬프트를 반복 실행하는 번들 스킬이다. 간격과 프롬프트가 모두 선택적이며, 무엇을 주느냐로 동작이 결정된다:

| 제공한 것 | 예시 | 동작 |
|-----------|------|------|
| 간격 + 프롬프트 | `/loop 5m check the deploy` | 고정 스케줄로 실행 |
| 프롬프트만 | `/loop check the deploy` | 매 반복 Claude가 고른 간격으로 실행 |
| 간격만/없음 | `/loop` | 내장 유지보수 프롬프트(또는 `loop.md`) 실행 |

다른 명령을 프롬프트로 넘길 수도 있다: `/loop 20m /review-pr 1234`.

#### 고정 간격으로 실행

간격을 주면 Claude가 cron 표현식으로 변환·스케줄링하고 cadence와 job ID를 확인해준다:

```text
/loop 5m check if the deployment finished and tell me what happened
```

간격은 `30m`처럼 앞에 오거나 `every 2 hours`처럼 뒤에 올 수 있다. 단위: `s`(초)·`m`(분)·`h`(시)·`d`(일). 초는 cron의 1분 granularity 때문에 분 단위로 올림된다. `7m`·`90m`처럼 깔끔한 cron 스텝에 안 맞으면 가장 가까운 값으로 반올림되고 Claude가 무엇을 골랐는지 알려준다.

#### Claude가 간격을 고르게 하기 (self-paced)

간격을 생략하면 Claude가 매 반복마다 관찰한 것에 따라 1분~1시간 사이 지연을 동적으로 고른다(빌드/PR 활동 중이면 짧게, 잠잠하면 길게). 고른 지연과 이유가 각 반복 끝에 출력된다:

```text
/loop check whether CI passed and address any review comments
```

동적 `/loop`을 요청하면 Claude가 [[06 내장 도구 레퍼런스]]의 **Monitor 도구**를 직접 쓸 수 있다. Monitor는 백그라운드 스크립트를 돌려 각 출력 라인을 스트리밍해주므로 폴링을 아예 없애 토큰 효율적이고 반응이 빠르다.

> [!warning] 모순
> [[13 자동화와 스케줄링]] 본문은 self-paced `/loop`을 권장하지만, Bedrock·Vertex AI·Microsoft Foundry에서는 간격 없는 프롬프트가 **고정 10분 스케줄**로 돌고, 프롬프트 없는 `/loop`은 사용법 메시지만 출력한다(maintenance 프롬프트·`loop.md` 미동작). 프로바이더에 따라 동작이 갈리므로 [[17 클라우드 프로바이더]] 사용 시 확인 필요.

#### 내장 유지보수 프롬프트

프롬프트를 생략하면 Claude가 매 반복 다음을 순서대로 처리한다:

- 대화의 미완료 작업 이어가기
- 현재 브랜치 PR 돌보기: 리뷰 코멘트, 실패한 CI, 머지 충돌
- 그 외 아무것도 없으면 버그 헌트·단순화 같은 정리 패스

push·delete 같은 비가역 작업은 transcript가 이미 승인한 것을 이어갈 때만 진행한다.

```text
/loop
```

bare `/loop`은 동적 간격으로 돈다. `/loop 15m`처럼 간격을 주면 고정 스케줄.

#### `loop.md`로 기본 프롬프트 커스터마이즈

`loop.md`는 내장 유지보수 프롬프트를 사용자 지시로 대체한다. bare `/loop` 하나의 기본 프롬프트를 정의할 뿐 별도 작업 목록은 아니며, CLI에서 프롬프트를 직접 주면 무시된다. 두 위치를 순서대로 찾아 첫 번째를 쓴다:

| 경로 | 스코프 |
|------|--------|
| `.claude/loop.md` | 프로젝트 레벨. 둘 다 있으면 우선. |
| `~/.claude/loop.md` | 사용자 레벨. 자체 정의가 없는 모든 프로젝트에 적용. |

```markdown title=".claude/loop.md"
Check the `release/next` PR. If CI is red, pull the failing job log,
diagnose, and push a minimal fix. If new review comments have arrived,
address each one and resolve the thread. If everything is green and
quiet, say so in one line.
```

수정은 다음 반복부터 적용되므로 루프가 도는 중에도 다듬을 수 있다. 25,000바이트를 넘는 내용은 잘린다.

#### 루프 중지

다음 반복을 기다리는 동안 `Esc`를 누르면 pending wakeup이 지워져 다시 발화하지 않는다. Claude에게 직접 요청해 스케줄한 작업은 `Esc`에 영향받지 않으며 삭제할 때까지 유지된다. self-paced 모드에서는 작업이 명백히 완료되면 Claude 스스로 다음 wakeup을 잡지 않아 루프를 끝낼 수 있다. 고정 간격 루프는 사용자가 멈추거나 7일이 지날 때까지 돈다.

### 일회성 알림 설정

일회성 알림은 `/loop` 대신 자연어로 설명한다. Claude가 단발 작업을 스케줄하고 실행 후 스스로 삭제한다:

```text
remind me at 3pm to push the release branch
```
```text
in 45 minutes, check whether the integration tests passed
```

### 스케줄 작업 관리

자연어로 나열·취소하거나 내부 도구를 직접 참조한다:

```text
what scheduled tasks do I have?
cancel the deploy check job
```

내부적으로 쓰이는 도구:

| 도구 | 용도 |
|------|------|
| `CronCreate` | 새 작업 스케줄. 5-field cron 표현식, 실행할 프롬프트, 반복/단발 여부를 받음 |
| `CronList` | 모든 작업과 ID·스케줄·프롬프트 나열 |
| `CronDelete` | ID로 작업 취소 |

각 작업은 8자 ID를 가지며, 한 세션은 동시에 **최대 50개** 작업을 보유.

### 스케줄 작업 실행 방식

스케줄러는 매초 due 작업을 확인해 낮은 우선순위로 enqueue한다. 스케줄된 프롬프트는 응답 중간이 아니라 **턴 사이**에 발화한다. Claude가 바쁘면 현재 턴이 끝날 때까지 대기. 모든 시각은 **로컬 타임존**으로 해석된다 (`0 9 * * *`은 UTC가 아닌 실행 위치의 9am).

#### Jitter (지터)

모든 세션이 같은 wall-clock에 API를 때리지 않도록 작업 ID에서 유도한 결정적 오프셋을 더한다:

- 반복 작업: 스케줄 시각 후 최대 30분(시간당보다 자주 도는 작업은 간격의 절반까지). `:00` 시작 시간당 작업은 `:30`까지 발화 가능.
- 정시/반시 단발 작업: 최대 90초 일찍 발화.

정확한 타이밍이 중요하면 `:00`·`:30`이 아닌 분을 골라라(예: `0 9 * * *` 대신 `3 9 * * *` — 단발 지터 미적용).

#### 7일 만료

반복 작업은 생성 7일 후 자동 만료된다. 마지막으로 한 번 발화하고 스스로 삭제 — 잊힌 루프의 수명을 제한. 더 오래 가야 하면 만료 전 재생성하거나 Routines / Desktop scheduled tasks를 쓴다.

### Cron 표현식 레퍼런스

`CronCreate`는 표준 5-field cron을 받는다: `minute hour day-of-month month day-of-week`. 모든 필드는 와일드카드(`*`), 단일 값(`5`), 스텝(`*/15`), 범위(`1-5`), 콤마 리스트(`1,15,30`) 지원.

| 예시 | 의미 |
|------|------|
| `*/5 * * * *` | 5분마다 |
| `0 * * * *` | 매시 정각 |
| `7 * * * *` | 매시 7분 |
| `0 9 * * *` | 매일 9am local |
| `0 9 * * 1-5` | 평일 9am local |
| `30 14 15 3 *` | 3월 15일 2:30pm local |

day-of-week는 일요일 `0` 또는 `7`부터 토요일 `6`까지. `L`·`W`·`?` 확장 문법과 `MON`·`JAN` 같은 이름 별칭은 미지원. day-of-month와 day-of-week가 둘 다 제약되면 **둘 중 하나**만 매치해도 발화(표준 vixie-cron 시맨틱).

### 비활성화 및 제약

환경 변수 `CLAUDE_CODE_DISABLE_CRON=1`로 스케줄러를 완전히 끈다(cron 도구·`/loop` 사용 불가, 이미 스케줄된 작업 발화 중단). 전체 disable 플래그는 [[04 설정]] / [[25 트러블슈팅]]의 env-vars 참조.

세션 스코프 스케줄링의 본질적 한계:

- Claude Code가 실행 중이고 idle일 때만 발화. 터미널을 닫거나 세션이 종료되면 멈춤.
- **놓친 발화 catch-up 없음**. 장기 요청 중 시각이 지나면 idle이 될 때 한 번만 발화(놓친 간격당 한 번이 아님).
- 새 대화는 모든 세션 스코프 작업을 지움. `--resume`/`--continue`는 미만료 작업만 복원. **백그라운드 Bash와 monitor 작업은 resume 시 절대 복원 안 됨.**

## Routines로 작업 자동화 (클라우드)

> [!note] Routines는 research preview다. 동작·제한·API 표면이 바뀔 수 있다.

routine은 저장된 Claude Code 구성이다: 프롬프트 + 하나 이상의 리포지토리 + 커넥터 집합을 한 번 패키징해 자동 실행. **Anthropic-managed cloud**에서 돌아 노트북이 닫혀 있어도 동작한다. Pro·Max·Team·Enterprise 플랜에서 [Claude Code on the web]([[15 웹과 모바일]])을 켠 상태로 사용 가능.

각 routine은 하나 이상의 트리거를 가질 수 있다:

- **Scheduled**: 매시/매일/매주 등 반복 cadence, 또는 미래 특정 시각에 1회
- **API**: 라우틴별 엔드포인트에 bearer token으로 HTTP POST해 온디맨드 트리거
- **GitHub**: PR·릴리스 같은 리포지토리 이벤트에 반응

한 routine이 여러 트리거를 조합할 수 있다(예: PR 리뷰 routine을 매일 밤 + 배포 스크립트 + 새 PR마다 발화).

관리 위치:
- 웹: [claude.ai/code/routines](https://claude.ai/code/routines)
- CLI: `/schedule` (스케줄 트리거 routine만 생성)
- Desktop 앱: **Routines → New routine → Remote** (Local 선택 시 Desktop scheduled task가 됨)

세 표면 모두 같은 클라우드 계정에 쓰므로 한 곳에서 만든 routine이 다른 곳에 즉시 나타난다. Team·Enterprise 어드민은 [claude.ai/admin-settings/claude-code](https://claude.ai/admin-settings/claude-code)의 Routines 토글로 전체 멤버에 대해 비활성화 가능.

> [!note] routine은 완전 자율 클라우드 세션으로 돈다 — 권한 모드 선택기도, 실행 중 승인 프롬프트도 없다. 셸 명령, 클론된 리포지토리에 커밋된 [[07 스킬]], 포함한 커넥터를 모두 호출할 수 있다. 도달 범위는 선택한 리포지토리·브랜치 푸시 설정, 환경의 네트워크 접근·변수, 포함한 커넥터로 결정된다. 각각을 실제 필요한 만큼만 좁혀라. routine은 개인 claude.ai 계정 소속이며 팀원과 공유되지 않고, 연결된 GitHub/커넥터 신원으로 동작한다(커밋·PR이 본인 GitHub 유저로 표시).

### routine 생성 (웹 폼 흐름)

1. **생성 폼 열기** — [claude.ai/code/routines](https://claude.ai/code/routines)에서 **New routine**.
2. **이름 + 프롬프트** — 프롬프트가 가장 중요하다. 자율 실행이므로 self-contained하고 무엇을 할지·성공이 무엇인지 명시해야 한다. 입력에 모델 선택기 포함(매 실행에 그 모델 사용).
3. **리포지토리 선택** — 하나 이상의 GitHub 리포. 매 실행 시작 시 default branch에서 클론되며, Claude는 변경을 `claude/`-prefix 브랜치로 만든다.
4. **환경 선택** — 클라우드 환경이 접근 범위를 제어: **Network access**(인터넷 접근 수준), **Environment variables**(API 키·토큰), **Setup script**(의존성 설치, 결과 캐시됨). 기본 제공 **Default** 환경은 **Trusted** 네트워크 접근(패키지 레지스트리·클라우드 프로바이더 API·컨테이너 레지스트리·흔한 개발 도메인 허용, 그 외 차단).
5. **트리거 선택** — Schedule / GitHub event / API 중 하나 또는 조합.
6. **커넥터 & 권한 검토** — 연결된 모든 MCP 커넥터가 기본 포함(불필요한 것 제거). Permissions 탭의 **Allow unrestricted branch pushes**로 특정 리포에서 기존 브랜치 push 허용.
7. **Create** — 생성 후 트리거가 매치되면 실행. 즉시 실행은 detail 페이지의 **Run now**.

### CLI에서 생성 (`/schedule`)

```text
/schedule daily PR review at 9am
/schedule clean up feature flag in one week
```

`/schedule`은 대화형으로 정보를 모아 계정에 저장한다. **스케줄 트리거 routine만** 생성하며, API·GitHub 트리거는 웹에서 편집해 추가한다. 관리 명령도 지원: `/schedule list`, `/schedule update`, `/schedule run`.

### 스케줄 트리거

프리셋: hourly / daily / weekdays / weekly. 시각은 로컬 존으로 입력 후 자동 변환되어 클라우드 인프라 위치와 무관하게 그 wall-clock에 실행. stagger 때문에 몇 분 늦게 시작할 수 있고 오프셋은 routine마다 일정. 커스텀 간격(2시간마다, 매월 1일)은 폼에서 가장 가까운 프리셋을 고른 뒤 CLI `/schedule update`로 cron 표현식 지정. **최소 간격 1시간**(더 자주는 거부).

#### 일회성 실행

특정 timestamp에 1회 발화 후 자동 비활성화(웹 UI에 **Ran** 표시). CLI에서 자연어로:

```text
/schedule tomorrow at 9am, summarize yesterday's merged PRs
/schedule in 2 weeks, open a cleanup PR that removes the feature flag
```

일회성 실행은 **일일 routine 실행 cap에 카운트되지 않는다**. 대신 일반 구독 사용량을 다른 세션처럼 차감.

### API 트리거

라우틴에 전용 HTTP 엔드포인트를 부여한다. bearer token으로 POST하면 새 세션을 시작하고 session URL을 반환. 알림 시스템·배포 파이프라인·내부 도구 연동에 쓴다. (CLI는 현재 토큰 생성·취소 불가 — 웹에서만.)

흐름: routine 편집 → **Select a trigger → Add another trigger → API** → URL 복사 + **Generate token**(한 번만 표시되니 즉시 안전한 곳에 저장) → `Authorization: Bearer` 헤더로 POST. 토큰은 해당 routine 트리거에만 스코프되며 같은 모달에서 **Regenerate**/**Revoke** 가능.

#### routine 트리거 (curl 예시)

`/fire` 엔드포인트에 bearer token으로 POST. body의 선택적 `text` 필드로 실행별 컨텍스트(알림 본문, 실패 로그)를 저장된 프롬프트와 함께 전달. freeform 텍스트라 파싱되지 않으며 JSON을 보내도 리터럴 문자열로 받는다:

```bash
curl -X POST https://api.anthropic.com/v1/claude_code/routines/trig_01ABCDEFGHJKLMNOPQRSTUVW/fire \
  -H "Authorization: Bearer sk-ant-oat01-xxxxx" \
  -H "anthropic-beta: experimental-cc-routine-2026-04-01" \
  -H "anthropic-version: 2023-06-01" \
  -H "Content-Type: application/json" \
  -d '{"text": "Sentry alert SEN-4521 fired in prod. Stack trace attached."}'
```

성공 응답:

```json
{
  "type": "routine_fire",
  "claude_code_session_id": "session_01HJKLMNOPQRSTUVWXYZ",
  "claude_code_session_url": "https://claude.ai/code/session_01HJKLMNOPQRSTUVWXYZ"
}
```

> [!warning] `/fire`는 `experimental-cc-routine-2026-04-01` 베타 헤더 아래 제공된다. research preview 동안 요청·응답 형태, rate limit, 토큰 시맨틱이 바뀔 수 있다. 파괴적 변경은 새 날짜 베타 헤더로 나가며 직전 두 버전은 호환 유지. `/fire`는 claude.ai 유저 전용이며 Claude Platform API 표면이 아니다. 전체 API 레퍼런스는 Claude Platform 문서의 "Trigger a routine via API" 참조.

### GitHub 트리거

매치되는 이벤트가 연결된 리포에서 발생할 때마다 새 세션을 시작(이벤트마다 독립 세션, 세션 재사용 없음). 웹 UI에서만 설정. **Claude GitHub App**이 해당 리포에 설치돼 있어야 하며(트리거 설정이 미설치 시 설치 유도), `/web-setup`은 클론용 접근만 부여할 뿐 GitHub App을 설치하거나 webhook 전달을 켜지 않는다.

> [!note] research preview 동안 GitHub webhook 이벤트는 라우틴별·계정별 시간당 cap을 받고 초과분은 윈도우 리셋까지 drop.

#### 지원 이벤트

| 이벤트 | 발화 시점 |
|--------|-----------|
| Pull request | PR이 opened, closed, assigned, labeled, synchronized 등 |
| Release | 릴리스가 created, published, edited, deleted |

각 카테고리 안에서 특정 액션(`pull_request.opened`)을 고르거나 전체 액션에 반응.

#### PR 필터

모든 조건이 매치해야 발화. 필드: Author / Title / Body / Base branch / Head branch / Labels / Is draft / Is merged. 각 필드는 연산자와 짝: equals, contains, starts with, is one of, is not one of, matches regex.

`matches regex`는 전체 값을 테스트(부분 문자열 아님). `hotfix` 포함 제목 매치는 `.*hotfix.*`로. 리터럴 부분 문자열은 `contains` 사용. 예: base `main` + head contains `auth-provider`, is draft `false`, labels include `needs-backport`.

### routine 관리·실행·환경

detail 페이지에서 리포·커넥터·프롬프트·스케줄·API 토큰·GitHub 트리거·과거 실행을 본다. 각 실행을 클릭하면 풀 세션으로 열려 변경 검토·PR 생성·대화 이어가기 가능.

> [!warning] 실행 목록의 **green 상태**는 세션이 인프라 에러 없이 시작·종료됐다는 뜻일 뿐, 프롬프트 작업이 성공했다는 뜻이 아니다. 실제로 무엇을 했는지는 실행을 열어 transcript를 읽어 확인. 차단된 네트워크 요청, 누락 커넥터 도구, 작업 레벨 실패는 상태 표시기가 아니라 거기서 드러난다.

detail 페이지 조작: **Run now**, **Repeats** 토글(일시정지/재개), 연필 아이콘(편집), 삭제 아이콘.

- **리포지토리/브랜치 권한**: 매 실행 클론. 기본적으로 `claude/`-prefix 브랜치에만 push(보호·장수 브랜치 보호). 특정 리포에서 **Allow unrestricted branch pushes**로 해제.
- **커넥터**: [[10 MCP]]의 claude.ai integrations. `claude mcp add`로 로컬 추가한 서버는 머신에 저장돼 커넥터 목록에 안 나오므로, [claude.ai/customize/connectors](https://claude.ai/customize/connectors)에서 추가하거나 커밋된 `.mcp.json`에 선언.
- **환경/네트워크**: Default는 Trusted 접근(허용 목록 외 호스트는 `403` + `x-deny-reason: host_not_allowed`). 커넥터 트래픽은 Anthropic 서버 경유라 Allowed domains에 추가 불필요. 추가 도메인은 환경 설정에서 **Network access**를 **Custom**(또는 **Full**)으로 변경.

### 사용량과 제한

routine은 대화형 세션과 동일하게 구독 사용량을 차감한다. 추가로 **계정당 일일 실행 cap**이 있다. 현재 소비량은 [claude.ai/code/routines](https://claude.ai/code/routines) 또는 [claude.ai/settings/usage](https://claude.ai/settings/usage)에서 확인. cap 또는 구독 한도 도달 시 usage credits를 켠 조직은 metered overage로 계속 실행 가능(Settings → Billing). 일회성 실행은 일일 cap 면제. 비용 일반론은 [[19 비용과 성능]].

### 트러블슈팅

- **`/schedule` returns "Unknown command"**: 요건 미충족 시 CLI가 숨긴다. 원인 — (1) Console API 키나 Bedrock/Vertex/Foundry로 인증됨(`/schedule`은 claude.ai 구독 로그인 필요; `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN`/`apiKeyHelper`가 우선하니 먼저 제거), (2) `DISABLE_TELEMETRY`·`DO_NOT_TRACK`·`CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC`·`DISABLE_GROWTHBOOK`가 설정됨(feature-flag fetch 차단), (3) Claude Code on the web 세션 내부(웹 UI에서 관리), (4) CLI가 v2.1.81 미만(`claude update`).
- **"Routines are disabled by your organization's policy"**: Team/Enterprise 어드민이 Routines 토글을 끔. 서버 사이드 설정이라 로컬에서 못 덮음.

## Desktop scheduled tasks (로컬 스케줄)

Desktop 앱의 **Routines** 페이지는 로컬 scheduled task와 원격 [routine](#routines로-작업-자동화-클라우드)을 모두 만든다. 로컬 task는 내 머신에서 파일·도구에 직접 접근하되 **앱이 열려 있고 컴퓨터가 깨어 있을 때만** 발화한다. 데일리 코드 리뷰, 의존성 업데이트 체크, 캘린더·인박스를 끌어오는 아침 브리핑 등에 쓴다. (옵션 비교 표는 위 [세 가지 스케줄링 옵션 비교](#세-가지-스케줄링-옵션-비교-canonical) 참조 — Desktop 열.) 앱 전반은 [[14 IDE와 데스크톱]].

> [!note] 기본적으로 scheduled task는 작업 디렉터리의 현재 상태(커밋되지 않은 변경 포함)에 대해 돈다. 생성 시 **worktree 토글**을 켜면 각 실행이 자체 격리된 Git worktree를 가진다(병렬 세션과 동일).

### task 생성

**Routines → New routine → Local**. 필드:

| 필드 | 설명 |
|------|------|
| Name | 식별자. lowercase kebab-case로 변환돼 디스크 폴더명이 됨. task 간 고유해야 함. |
| Description | task 목록에 표시되는 짧은 요약. |
| Instructions | task 실행 시 할 일. 프롬프트 박스에 쓰듯 작성. 권한 모드·모델 선택기, 작업 폴더 선택, worktree 실행 여부 포함. |
| Schedule | 실행 빈도(아래). |

저장 전 폴더 필수(미신뢰 폴더면 trust 프롬프트). 세션에서 자연어로도 생성 가능: "set up a daily code review that runs every morning at 9am"(반복), "remind me at 3pm tomorrow to check the deploy"(발화 후 자동 비활성화 일회성).

### 스케줄 옵션

| 프리셋 | 동작 |
|--------|------|
| **Manual** | 스케줄 없음, **Run now**로만 실행. 온디맨드 프롬프트 저장용. |
| **Hourly** | 매시간 |
| **Daily** | 시각 선택기(기본 9:00 AM local) |
| **Weekdays** | Daily와 같되 토·일 제외 |
| **Weekly** | 시각 + 요일 선택기 |

선택기에 없는 간격(15분마다, 매월 1일, 미래 1회)은 세션에서 평문으로: "schedule a task to run all the tests every 6 hours."

### 실행 방식·놓친 실행

내 머신에서 실행. 앱이 열린 동안 매분 스케줄을 확인해 due면 fresh 세션 시작(수동 세션과 독립). 각 task는 API 트래픽 stagger를 위해 몇 분 결정적 지연. 발화 시 데스크톱 알림 + 사이드바 **Scheduled** 섹션에 새 세션. 세션은 파일 편집·명령 실행·커밋·PR을 할 수 있다.

앱이 돌고 컴퓨터가 깨어 있을 때만 실행. 스케줄 시각에 컴퓨터가 자면 스킵. idle-sleep 방지는 Settings → **Desktop app → General**의 **Keep computer awake**(노트북 뚜껑을 닫으면 여전히 sleep). 컴퓨터가 꺼져도 돌아야 하면 원격 routine 사용.

**놓친 실행 catch-up**: 앱 시작/컴퓨터 wake 시 지난 7일간 놓친 실행을 확인. 놓쳤으면 **가장 최근 놓친 시각에 대해 정확히 1회** catch-up하고 나머지는 폐기(6일 놓친 데일리 task도 wake 시 1회). 9am task가 종일 잠든 컴퓨터 탓에 11pm에 돌 수 있으니 프롬프트에 가드레일을 넣어라: "Only review today's commits. If it's after 5pm, skip the review and just post a summary of what was missed."

### 권한

각 task는 자체 권한 모드를 가진다([[05 권한]]). `~/.claude/settings.json`의 allow 규칙도 적용. Ask 모드에서 권한 없는 도구가 필요하면 승인할 때까지 실행이 멈추고 세션이 사이드바에 열린 채 대기. stall 회피: 생성 후 **Run now**로 권한 프롬프트마다 "always allow" 선택 → 이후 실행은 자동 승인. task detail에서 검토·취소 가능.

### task 관리

detail 페이지: **Run now**, **Status**(Active↔Paused), **Edit**, **Review history**(스킵 사유: 컴퓨터 sleep, 이전 실행 진행 중, 다른 task 실행 중), **Review allowed permissions**(Always allowed 패널), **Delete**(생성한 세션 아카이브; 확인 다이얼로그의 **Also delete files on disk** 체크 시 `SKILL.md`와 `~/.claude/scheduled-tasks/` 데이터 제거).

세션에서 자연어로도: "pause my dependency-audit task", "show me my scheduled tasks". 실행 중인 task가 `update_scheduled_task` MCP 도구로 **자기 스케줄/프롬프트를 수정**할 수 있다(예: 릴리스 브랜치 감지 시 코드 리뷰를 더 일찍 재스케줄).

디스크에서 프롬프트 편집: `~/.claude/scheduled-tasks/<task-name>/SKILL.md`(또는 `CLAUDE_CONFIG_DIR` 설정 시 그 아래). YAML frontmatter의 `name`·`description` + 본문 프롬프트. 다음 실행부터 적용. 스케줄·폴더·모델·enabled 상태는 이 파일에 없으니 Edit 폼이나 Claude에게 요청.

## 실행 중인 세션에 이벤트 푸시 (Channels)

> [!note] Channels는 research preview이며 v2.1.80 이상 필요. claude.ai 또는 Console API 키 인증 필요. Bedrock·Vertex AI·Foundry에서는 불가. Team·Enterprise 조직은 명시적으로 [활성화](#엔터프라이즈-제어)해야 한다.

channel은 **실행 중인 세션에 이벤트를 푸시하는 [[10 MCP]] 서버**다. 터미널 앞에 없을 때 일어난 일에 Claude가 반응할 수 있다. 양방향이 가능해 Claude가 같은 채널로 답신하는 채팅 브리지가 된다. 이벤트는 세션이 열려 있을 때만 도착하므로 always-on에는 백그라운드 프로세스/지속 터미널에서 Claude를 돌린다. plugin([[11 플러그인]])으로 설치하고 자기 자격증명으로 설정한다. preview에는 Telegram·Discord·iMessage 포함.

Claude가 채널로 답하면 터미널에는 inbound 메시지와 도구 호출·확인("sent")만 보이고 실제 답신 텍스트는 상대 플랫폼에 나타난다.

### 지원 채널

각 채널은 [Bun](https://bun.sh)을 요구하는 plugin이다.

#### Telegram

1. [BotFather](https://t.me/BotFather)에서 `/newbot` → 표시 이름과 `bot`으로 끝나는 username 부여 → 토큰 복사.
2. `/plugin install telegram@claude-plugins-official` (마켓플레이스 미발견 시 `/plugin marketplace update claude-plugins-official` 또는 `/plugin marketplace add anthropics/claude-plugins-official`). 설치 후 `/reload-plugins`.
3. `/telegram:configure <token>` → `~/.claude/channels/telegram/.env`에 저장(또는 `TELEGRAM_BOT_TOKEN` 셸 env).
4. 재시작: `claude --channels plugin:telegram@claude-plugins-official`
5. 봇에 아무 메시지 → 페어링 코드 수신 → `/telegram:access pair <code>` → `/telegram:access policy allowlist`로 잠금.

#### Discord

1. [Discord Developer Portal](https://discord.com/developers/applications) → New Application → Bot 섹션에서 username 생성 → Reset Token으로 토큰 복사.
2. **Privileged Gateway Intents**에서 **Message Content Intent** 활성화.
3. **OAuth2 → URL Generator**에서 `bot` scope + 권한(View Channels, Send Messages, Send Messages in Threads, Read Message History, Attach Files, Add Reactions) → 생성 URL로 서버에 초대.
4. `/plugin install discord@claude-plugins-official` → `/reload-plugins` → `/discord:configure <token>`(`~/.claude/channels/discord/.env`, 또는 `DISCORD_BOT_TOKEN`).
5. `claude --channels plugin:discord@claude-plugins-official` → 봇에 DM → 페어링 코드 → `/discord:access pair <code>` → `/discord:access policy allowlist`.

#### iMessage

macOS 전용, 봇 토큰·외부 서비스 불필요. Messages DB(`~/Library/Messages/chat.db`)를 직접 읽고 AppleScript로 답신.

1. **Full Disk Access** 부여(첫 읽기 시 macOS 프롬프트 → Allow; 거부했으면 System Settings → Privacy & Security → Full Disk Access에서 터미널 추가. 없으면 `authorization denied`로 종료).
2. `/plugin install imessage@claude-plugins-official`
3. `claude --channels plugin:imessage@claude-plugins-official`
4. 자기 Apple ID로 자신에게 문자 → 즉시 Claude 도달(self-chat은 접근 제어 우회). 첫 답신 시 macOS Automation 프롬프트 → OK.
5. 다른 발신자 허용: `/imessage:access allow +15551234567` (전화번호 `+country` 또는 Apple ID 이메일).

### Quickstart (fakechat)

fakechat은 localhost 데모 채널 — 인증·외부 서비스 불필요. 요건: Claude Code 설치·인증, Bun(`bun --version`), (Team/Enterprise/관리 Console org면) 어드민이 채널 활성화.

```text
/plugin install fakechat@claude-plugins-official
```
```bash
claude --channels plugin:fakechat@claude-plugins-official
```

`--channels`에 여러 plugin을 공백으로 구분해 넘길 수 있다. [http://localhost:8787](http://localhost:8787) UI에서 타이핑하면 `<channel source="fakechat">` 이벤트로 세션에 도착하고, Claude가 fakechat의 `reply` 도구를 호출해 답이 UI에 표시된다.

자리를 비운 사이 권한 프롬프트를 만나면 세션이 응답까지 멈춘다. permission relay capability를 선언한 채널 서버는 이를 원격으로 전달해 승인/거부할 수 있다(아래 reference). 무인 사용에는 [[18 보안과 샌드박스]]의 `--dangerously-skip-permissions`가 프롬프트를 완전히 우회하지만 신뢰하는 환경에서만. `-p` 비대화형 모드에서 채널을 돌리면 터미널 입력이 필요한 도구(객관식 질문, plan mode 승인)는 세션이 멈추지 않도록 비활성화된다.

### Security

승인된 모든 채널 plugin은 sender allowlist를 유지한다 — 추가한 ID만 메시지를 푸시할 수 있고 나머지는 조용히 drop. Telegram·Discord는 페어링으로 bootstrap(봇에 메시지 → 페어링 코드 → 세션에서 승인 → sender ID 추가). iMessage는 자신에게 보내면 게이트 자동 통과, 다른 연락처는 `/imessage:access allow`로 핸들 추가.

추가로 세션마다 `--channels`로 어떤 서버를 켤지 제어하고, 조직은 [`channelsEnabled`](#엔터프라이즈-제어)로 가용성 제어. **`.mcp.json`에 있는 것만으로는 메시지를 푸시할 수 없다 — `--channels`에 명명돼야 한다.** allowlist는 channel이 선언했다면 permission relay도 게이트하므로, 채널로 답신 가능한 사람은 누구나 세션의 도구 사용을 승인/거부할 수 있다. 그 권한을 믿는 sender만 allowlist에 넣어라.

### 엔터프라이즈 제어

어드민은 사용자가 덮을 수 없는 두 [managed settings]([[04 설정]])로 가용성을 제어한다. 기본값은 인증 방식에 달림: **claude.ai Team·Enterprise**는 어드민이 켤 때까지 차단, **Console API 키 인증**은 기본 허용(managed settings 배포 시에만 필요). 어떤 경우든 사용자가 `--channels`로 세션에 opt-in해야 채널이 돈다.

| 설정 | 용도 | 미설정 시 |
|------|------|-----------|
| `channelsEnabled` | 마스터 스위치. 모든 채널 전달에 `true` 필요. [Admin console](https://claude.ai/admin-settings/claude-code) 토글 또는 managed settings. off면 development 플래그 포함 모든 채널 차단. | Team/Enterprise: 차단. Console: managed settings 미배포면 허용, 배포 시 이 키 설정 전까지 차단 |
| `allowedChannelPlugins` | 채널 활성화 후 등록 가능한 plugin. 설정 시 Anthropic 목록 대체. `channelsEnabled: true`일 때만 적용. | Anthropic 기본 목록 |

Pro·Max 개인 유저는 이 체크를 건너뛴다 — 채널이 가용하며 세션마다 `--channels`로 opt-in.

```json
{
  "channelsEnabled": true,
  "allowedChannelPlugins": [
    { "marketplace": "claude-plugins-official", "plugin": "telegram" },
    { "marketplace": "claude-plugins-official", "plugin": "discord" },
    { "marketplace": "acme-corp-plugins", "plugin": "internal-alerts" }
  ]
}
```

`allowedChannelPlugins` 설정 시 Anthropic allowlist를 완전 대체(나열된 plugin만 등록). 빈 배열은 allowlist 채널을 모두 차단하되 `--dangerously-load-development-channels`는 로컬 테스트용으로 우회 가능. development 플래그 포함 완전 차단은 `channelsEnabled`를 미설정으로 둔다.

### research preview / 다른 연동과 비교

preview 동안 `--channels`는 Anthropic-maintained allowlist(또는 조직 allowlist)의 plugin만 받는다. 기본 승인 집합은 `claude-plugins-official`의 채널 plugin. 빌드 중인 채널 테스트는 `--dangerously-load-development-channels`(이 플래그는 allowlist만 건너뛰며 `channelsEnabled` 정책은 여전히 적용).

| 기능 | 하는 일 | 적합 |
|------|---------|------|
| Claude Code on the web | GitHub에서 클론한 fresh 클라우드 샌드박스에서 작업 | 나중에 확인할 self-contained 비동기 작업 위임 |
| Claude in Slack | 채널/스레드의 `@Claude` 멘션에서 웹 세션 생성 | 팀 대화 컨텍스트에서 바로 작업 시작 |
| 표준 MCP 서버 | 작업 중 Claude가 쿼리, 세션에 푸시 없음 | 온디맨드 읽기/쿼리 접근 |
| Remote Control | claude.ai/모바일 앱에서 로컬 세션 조종 | 자리를 비운 채 진행 중 세션 조종 |

Channels는 비-Claude 소스의 이벤트를 이미 실행 중인 로컬 세션에 푸시해 이 목록의 빈틈을 메운다 — 채팅 브리지(폰에서 Telegram/Discord/iMessage로 묻고 같은 채팅에 답신, 작업은 내 머신의 실제 파일에 대해 수행), webhook receiver(CI·에러 트래커·배포 파이프라인의 webhook이 Claude가 이미 컨텍스트를 가진 세션에 도착).

## Channels reference (자체 채널 빌드)

> [!note] Channels는 research preview, v2.1.80 이상. Permission relay는 v2.1.81 이상.

자체 채널을 빌드하면 plugin이 없는 시스템도 세션에 이벤트를 푸시할 수 있다. one-way(알림·webhook·모니터링 전달)와 two-way(채팅 브리지, [reply 도구](#reply-도구-노출) 노출)가 있고, 신뢰된 sender 경로를 가진 채널은 [permission relay](#permission-relay-원격-권한-승인)에 opt-in 가능.

### 개요와 요건

채널은 Claude Code와 같은 머신에서 도는 MCP 서버다. Claude Code가 subprocess로 spawn하고 stdio로 통신. 채팅 플랫폼(Telegram/Discord)은 로컬에서 폴링하고, webhook(CI/모니터링)은 로컬 HTTP 포트를 리슨한다.

유일한 하드 요건은 [`@modelcontextprotocol/sdk`](https://www.npmjs.com/package/@modelcontextprotocol/sdk)와 Node 호환 런타임(Bun·Node·Deno 모두 가능). 서버가 할 일:

1. `claude/channel` capability 선언(Claude Code가 notification listener 등록)
2. 일이 생기면 `notifications/claude/channel` 이벤트 emit
3. stdio transport로 연결

preview 동안 커스텀 채널은 allowlist에 없으므로 `--dangerously-load-development-channels`로 테스트한다(per-entry 우회). `--channels`와 조합해도 `--channels` 항목까지 우회가 확장되지 않는다.

```bash
# 개발 중인 plugin
claude --dangerously-load-development-channels plugin:yourplugin@yourmarketplace
# bare .mcp.json 서버
claude --dangerously-load-development-channels server:webhook
```

### 예제: webhook receiver (one-way)

```ts title="webhook.ts"
#!/usr/bin/env bun
import { Server } from '@modelcontextprotocol/sdk/server/index.js'
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'

const mcp = new Server(
  { name: 'webhook', version: '0.0.1' },
  {
    capabilities: { experimental: { 'claude/channel': {} } },
    instructions: 'Events from the webhook channel arrive as <channel source="webhook" ...>. They are one-way: read them and act, no reply expected.',
  },
)

await mcp.connect(new StdioServerTransport())

Bun.serve({
  port: 8788,
  hostname: '127.0.0.1',  // localhost 전용: 외부에서 POST 불가
  async fetch(req) {
    const body = await req.text()
    await mcp.notification({
      method: 'notifications/claude/channel',
      params: {
        content: body,                                       // <channel> 태그 본문
        meta: { path: new URL(req.url).pathname, method: req.method },  // 각 키 → 태그 속성
      },
    })
    return new Response('ok')
  },
})
```

`.mcp.json` 등록(프로젝트는 상대 경로, 사용자 레벨 `~/.claude.json`은 절대 경로):

```json title=".mcp.json"
{
  "mcpServers": {
    "webhook": { "command": "bun", "args": ["./webhook.ts"] }
  }
}
```

테스트:

```bash
claude --dangerously-load-development-channels server:webhook
# 별도 터미널에서:
curl -X POST localhost:8788 -d "build failed on main: https://ci.example.com/run/1234"
```

세션에 도착하는 형태:

```text
<channel source="webhook" path="/" method="POST">build failed on main: https://ci.example.com/run/1234</channel>
```

진단: `curl`은 성공했는데 Claude에 안 닿으면 `/mcp`로 서버 상태 확인("Failed to connect"는 보통 import 에러; `~/.claude/debug/<session-id>.txt`의 stderr 확인). `curl`이 "connection refused"면 포트 미바인딩 또는 stale 프로세스(`lsof -i :<port>` 후 `kill`). 트러블슈팅 전반은 [[25 트러블슈팅]].

### Server options

| 필드 | 타입 | 설명 |
|------|------|------|
| `capabilities.experimental['claude/channel']` | object | 필수. 항상 `{}`. 존재가 notification listener를 등록. |
| `capabilities.experimental['claude/channel/permission']` | object | 선택. 항상 `{}`. permission relay 수신 선언. |
| `capabilities.tools` | object | two-way 전용. 항상 `{}`. 표준 MCP 도구 capability. |
| `instructions` | string | 권장. Claude 시스템 프롬프트에 추가. 어떤 이벤트가 오는지, `<channel>` 속성 의미, 답신 여부와 도구, 되돌릴 속성(`chat_id` 등)을 알림. |

one-way는 `capabilities.tools`를 생략. 이벤트 푸시는 method `notifications/claude/channel`로 `mcp.notification()` 호출.

### Notification format

| 필드 | 타입 | 설명 |
|------|------|------|
| `content` | string | 이벤트 본문. `<channel>` 태그 본문으로 전달. |
| `meta` | `Record<string,string>` | 선택. 각 항목이 `<channel>` 태그 속성이 됨(chat ID, sender, severity 등). 키는 식별자(영문·숫자·언더스코어)만 — 하이픈 등은 조용히 drop. |

```ts
await mcp.notification({
  method: 'notifications/claude/channel',
  params: {
    content: 'build failed on main: https://ci.example.com/run/1234',
    meta: { severity: 'high', run_id: '1234' },
  },
})
```

`source` 속성은 서버 이름에서 자동 설정. notification은 ack되지 않으며, `await`는 transport에 쓰인 시점에 resolve(Claude가 처리한 시점 아님). 세션이 채널로 로드 안 했거나 조직 정책이 막으면 **조용히 drop**(에러 없음). 전달 확인이 필요하면 서버에서 상태를 추적하고 reply 도구로 보고. 이벤트는 순서대로 큐잉되며, Claude가 바쁠 때 여러 개 도착하면 다음 턴에 함께 묶여 처리. 독립 스트림 동시 처리는 별도 세션.

### reply 도구 노출 (two-way)

채팅 브리지는 Claude가 답신을 보내는 표준 MCP 도구를 노출한다. 채널 특화 요소는 없고 세 구성요소: ① 생성자 capabilities에 `tools: {}`, ② 도구 스키마·전송 로직 핸들러, ③ 언제·어떻게 호출할지 알리는 `instructions`.

```ts
import { ListToolsRequestSchema, CallToolRequestSchema } from '@modelcontextprotocol/sdk/types.js'

mcp.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [{
    name: 'reply',
    description: 'Send a message back over this channel',
    inputSchema: {
      type: 'object',
      properties: {
        chat_id: { type: 'string', description: 'The conversation to reply in' },
        text: { type: 'string', description: 'The message to send' },
      },
      required: ['chat_id', 'text'],
    },
  }],
}))

mcp.setRequestHandler(CallToolRequestSchema, async req => {
  if (req.params.name === 'reply') {
    const { chat_id, text } = req.params.arguments as { chat_id: string; text: string }
    send(`Reply to ${chat_id}: ${text}`)  // 실제로는 채팅 플랫폼에 POST
    return { content: [{ type: 'text', text: 'sent' }] }
  }
  throw new Error(`unknown tool: ${req.params.name}`)
})
```

`instructions`에 `<channel source="webhook" chat_id="...">`에서 `chat_id`를 받아 reply 도구로 라우팅하라고 명시. fakechat 서버가 파일 첨부·메시지 편집까지 보여주는 완전한 예시.

### 인바운드 메시지 게이팅

게이트 없는 채널은 prompt injection 벡터다. 엔드포인트에 닿는 누구나 Claude 앞에 텍스트를 둘 수 있다. `mcp.notification()` 호출 전에 sender를 allowlist로 검사:

```ts
const allowed = new Set(loadAllowlist())
// 핸들러 안, emit 전:
if (!allowed.has(message.from.id)) {  // room이 아닌 sender
  return  // 조용히 drop
}
await mcp.notification({ ... })
```

**chat/room id가 아닌 sender id로 게이트**하라(`message.from.id`, not `message.chat.id`). 그룹 채팅에서 둘이 다르며, room으로 게이트하면 allowlist 그룹의 누구나 주입할 수 있다. Telegram·Discord는 페어링으로 bootstrap, iMessage는 시작 시 사용자 자신의 주소를 감지해 통과.

### Permission relay (원격 권한 승인)

> [!note] v2.1.81 이상. 이전 버전은 `claude/channel/permission` capability를 무시.

Claude가 승인 필요한 도구를 호출하면 로컬 터미널 다이얼로그가 열리고 세션이 대기한다. two-way 채널은 같은 프롬프트를 병렬로 받아 다른 기기로 relay할 수 있다. 둘 다 살아있어 터미널 또는 폰에서 답할 수 있고, **먼저 도착한 답**이 적용되고 다른 쪽은 닫힌다. relay는 `Bash`·`Write`·`Edit` 같은 도구 승인을 커버하며, project trust와 MCP 서버 consent 다이얼로그는 relay되지 않는다(로컬 터미널만).

relay 루프 4단계: ① Claude Code가 짧은 request ID 생성·서버에 알림, ② 서버가 프롬프트+ID를 채팅 앱에 전달, ③ 원격 유저가 yes/no + ID로 응답, ④ 인바운드 핸들러가 verdict로 파싱, ID가 열린 요청과 매치하면 Claude Code가 적용. 그 사이 로컬 다이얼로그는 열려 있고 터미널에서 먼저 답하면 그것이 적용되고 원격 pending은 drop.

#### Permission request 필드 (`notifications/claude/channel/permission_request`)

| 필드 | 설명 |
|------|------|
| `request_id` | `l` 없는 소문자 `a`-`z` 5자(폰에서 `1`/`I`로 안 읽히게). 발신 프롬프트에 포함해 응답에 echo. Claude Code는 자신이 발급한 ID만 수용. 로컬 다이얼로그는 이 ID를 표시 안 하므로 outbound 핸들러가 ID를 아는 유일한 길. |
| `tool_name` | 도구 이름(`Bash`, `Write` 등). |
| `description` | 이 호출이 하는 일의 사람용 요약(로컬 다이얼로그와 동일 텍스트). Bash는 Claude의 명령 설명 또는 명령 자체. |
| `input_preview` | 도구 인자를 JSON 문자열로, 200자로 잘림. Bash는 명령, Write는 파일 경로+내용 prefix. |

verdict는 `notifications/claude/channel/permission`로 `request_id`(echo)와 `behavior`(`'allow'`/`'deny'`). allow는 진행, deny는 거부(로컬 No와 동일). 어느 verdict도 미래 호출에 영향 없음.

#### 채팅 브리지에 relay 추가

세 구성요소: ① `experimental`에 `claude/channel/permission: {}`, ② `permission_request` notification 핸들러(프롬프트 포맷·전송), ③ 인바운드 핸들러에서 `yes <id>`/`no <id>`를 인식해 verdict emit. **sender 인증([게이팅](#인바운드-메시지-게이팅))을 하는 채널만 capability를 선언하라** — 채널로 답신 가능한 누구나 도구 사용을 승인/거부할 수 있으므로.

```ts
import { z } from 'zod'

const PermissionRequestSchema = z.object({
  method: z.literal('notifications/claude/channel/permission_request'),
  params: z.object({
    request_id: z.string(),
    tool_name: z.string(),
    description: z.string(),
    input_preview: z.string(),
  }),
})

mcp.setNotificationHandler(PermissionRequestSchema, async ({ params }) => {
  send(
    `Claude wants to run ${params.tool_name}: ${params.description}\n\n` +
    `Reply "yes ${params.request_id}" or "no ${params.request_id}"`,
  )
})
```

인바운드 verdict 가로채기 — regex는 Claude Code가 생성하는 ID 형식(5자, `l` 없음)을 매치, `/i`로 폰 autocorrect 대문자 허용, 보내기 전 소문자화:

```ts
// "y abcde", "yes abcde", "n abcde", "no abcde" 매치
const PERMISSION_REPLY_RE = /^\s*(y|yes|n|no)\s+([a-km-z]{5})\s*$/i

async function onInbound(message: PlatformMessage) {
  if (!allowed.has(message.from.id)) return  // 먼저 sender 게이트
  const m = PERMISSION_REPLY_RE.exec(message.text)
  if (m) {
    await mcp.notification({
      method: 'notifications/claude/channel/permission',
      params: {
        request_id: m[2].toLowerCase(),
        behavior: m[1].toLowerCase().startsWith('y') ? 'allow' : 'deny',
      },
    })
    return  // verdict로 처리, 채팅으로 전달 안 함
  }
  await mcp.notification({  // verdict 형식 아니면 일반 채팅 경로
    method: 'notifications/claude/channel',
    params: { content: message.text, meta: { chat_id: String(message.chat.id) } },
  })
}
```

형식이 다르면(예: ID 없는 `approve it`) regex 미매치로 일반 메시지로 흘러가고, 형식은 맞지만 ID가 틀리면 서버가 verdict를 emit해도 Claude Code가 열린 요청을 못 찾아 조용히 drop — 두 경우 모두 다이얼로그는 열린 채 유지.

3-터미널 테스트: ① `claude --dangerously-load-development-channels server:webhook`, ② `curl -N localhost:8788/events`(outbound 스트림), ③ `curl -d "list the files in this directory" -H "X-Sender: dev" localhost:8788` → 로컬 다이얼로그가 열리고 `/events`에 5자 ID 프롬프트 등장 → `curl -d "yes <id>" -H "X-Sender: dev" localhost:8788`로 원격 승인.

### plugin으로 패키징

설치·공유 가능하게 하려면 [[11 플러그인]]으로 감싸 [marketplace]에 게시. `/plugin install` 후 `--channels plugin:<name>@<marketplace>`로 세션마다 활성화. 다만 자체 마켓플레이스 채널도 approved allowlist에 없으므로 여전히 `--dangerously-load-development-channels` 필요. 기본 allowlist는 `claude-plugins-official`의 채널 plugin뿐(Anthropic 큐레이션). 커뮤니티 마켓플레이스 제출 폼은 채널 allowlist에 추가하지 않는다. Anthropic 파트너 연락처가 있으면 official 등재를 조율하거나, Team·Enterprise 어드민이 조직 `allowedChannelPlugins`에 포함.

## Remote Control (다른 기기에서 로컬 세션 이어가기)

> [!note] Remote Control은 research preview, 모든 플랜 가용. v2.1.51 이상. Team·Enterprise는 어드민이 [Claude Code admin settings](https://claude.ai/admin-settings/claude-code)의 Remote Control 토글을 켜기 전까지 기본 off.

Remote Control은 [claude.ai/code](https://claude.ai/code) 또는 [Claude 모바일 앱]([[15 웹과 모바일]])을 내 머신에서 도는 세션에 연결한다. 책상에서 작업을 시작하고 소파의 폰이나 다른 컴퓨터 브라우저에서 이어받는다. Claude는 내내 로컬에서 돌아 아무것도 클라우드로 가지 않는다 — 풀 로컬 환경(파일시스템·[[10 MCP]] 서버·도구·프로젝트 설정)이 유지되고 `@`로 로컬 파일 경로 자동완성, 모든 기기에서 대화 동기화, 노트북 sleep·네트워크 끊김 후 자동 재연결. [[15 웹과 모바일]]의 Claude Code on the web과 달리 세션이 내 머신에서 실행되며 웹/모바일은 그 로컬 세션을 들여다보는 창일 뿐이다.

### 요건

- **구독**: Pro·Max·Team·Enterprise. API 키 미지원. Team·Enterprise는 어드민이 먼저 토글 활성화.
- **인증**: `claude` 실행 후 `/login`으로 claude.ai 로그인([[01 시작하기]]).
- **Workspace trust**: 프로젝트 디렉터리에서 `claude`를 한 번 실행해 trust 다이얼로그 수락.

### 세션 시작

#### Server mode

```bash
claude remote-control
```

터미널에 서버 모드로 머무르며 원격 연결 대기. session URL을 표시하고 spacebar로 QR 코드 토글.

| 플래그 | 설명 |
|--------|------|
| `--name "My Project"` | session 목록에 보이는 커스텀 제목 |
| `--remote-control-session-name-prefix <prefix>` | 자동 생성 이름 prefix(기본 호스트명, `myhost-graceful-unicorn`). env `CLAUDE_REMOTE_CONTROL_SESSION_NAME_PREFIX` |
| `--spawn <mode>` | 세션 생성 방식. `same-dir`(기본, 모두 같은 cwd 공유 — 같은 파일 편집 시 충돌 가능) / `worktree`(각 세션 자체 [git worktree], git repo 필요) / `session`(단일 세션, 추가 연결 거부, 시작 시에만). 런타임에 `w`로 same-dir↔worktree 토글 |
| `--capacity <N>` | 최대 동시 세션(기본 32). `--spawn=session`과 병용 불가 |
| `--verbose` | 상세 연결·세션 로그 |
| `--sandbox` / `--no-sandbox` | [[18 보안과 샌드박스]] 활성/비활성(기본 off) |

#### Interactive session

```bash
claude --remote-control            # 또는 --rc
claude --remote-control "My Project"
```

터미널의 풀 대화형 세션을 claude.ai/앱에서도 제어. server mode와 달리 로컬에서도 메시지 입력 가능.

#### 기존 세션에서

```text
/remote-control                    # 또는 /rc
/remote-control My Project
```

현재 대화 기록을 이어가는 Remote Control 세션 시작, session URL·QR 표시. `--verbose`·`--sandbox`·`--no-sandbox`는 이 명령에서 불가.

#### VS Code

[[14 IDE와 데스크톱]]의 VS Code 확장에서 `/remote-control` 또는 `/rc`(v2.1.79 이상). 프롬프트 박스 위 배너에 연결 상태, **Open in browser**로 바로 이동. 끊기는 배너 닫기 또는 `/remote-control` 재실행. CLI와 달리 name 인자·QR 없음.

### 다른 기기에서 연결

session URL을 브라우저에서 열거나, QR 스캔(server mode는 spacebar로 QR 토글), 또는 [claude.ai/code](https://claude.ai/code)·앱(모바일은 **Code** 탭)의 세션 목록에서 이름으로 찾기(온라인이면 컴퓨터 아이콘 + 초록 점). 제목 선택 순서: ① `--name`/`--remote-control`/`/remote-control` 인자, ② `/rename`, ③ 기존 대화의 마지막 의미있는 메시지, ④ `myhost-graceful-unicorn` 자동 생성. 앱이 없으면 `/mobile`로 다운로드 QR 표시.

### 모든 세션에 자동 활성화

기본은 명시적으로 켤 때만 활성. 매 대화형 세션에 자동 켜려면 `/config`에서 **Enable Remote Control for all sessions**를 `true`로([[12 출력 스타일과 상태줄]]의 `/config`; Desktop은 Settings → Claude Code → Enable remote control by default). 켜면 각 대화형 프로세스가 원격 세션 하나를 등록 — 한 프로세스에서 여러 동시 세션은 server mode를 쓴다.

### 연결과 보안

로컬 세션은 **outbound HTTPS 요청만** 하고 내 머신에 inbound 포트를 절대 열지 않는다. Anthropic API에 등록하고 work를 폴링하며, 다른 기기 연결 시 서버가 스트리밍 연결로 메시지를 라우팅. 모든 트래픽은 TLS 위 Anthropic API 경유(여느 세션과 동일). 연결은 각자 단일 목적에 스코프되어 독립 만료하는 여러 short-lived 자격증명 사용. 전반은 [[18 보안과 샌드박스]].

### 모바일 푸시 알림

> [!note] v2.1.110 이상.

Remote Control 활성 시 Claude가 폰에 푸시를 보낼 수 있다. 장기 작업 완료나 결정 필요 시 보내며 프롬프트로 요청 가능("notify me when the tests finish"). 설정: ① 앱 설치, ② 같은 계정·조직으로 로그인, ③ OS 알림 권한 수락, ④ `/config`에서 **Push when Claude decides** 활성. 안 오면 — `/config`가 **No mobile registered**면 앱을 열어 푸시 토큰 갱신; iOS는 Focus mode·notification summary; Android는 battery optimization에서 앱 제외.

### 제한

- **대화형 프로세스당 원격 세션 1개**(server mode 예외).
- **로컬 프로세스 유지 필수**: 터미널 닫기·VS Code 종료 등 `claude` 중지 시 세션 종료.
- **장기 네트워크 단절**: 머신이 깨어 있어도 약 10분 이상 네트워크 도달 불가면 timeout·종료(재시작 필요).
- **Ultraplan이 Remote Control 끊음**: 둘 다 claude.ai/code 인터페이스를 점유해 동시에 하나만 연결.
- **일부 명령 로컬 전용**: 터미널 인터랙티브 picker를 여는 `/mcp`·`/plugin`·`/resume`은 로컬 CLI에서만. 텍스트 출력 명령(`/compact`·`/clear`·`/context`·`/usage`·`/exit`·`/usage-credits`·`/recap`·`/reload-plugins`)은 모바일·웹에서 동작.

### 트러블슈팅 (요약)

- **"requires a claude.ai subscription"**: `claude auth login`으로 claude.ai 선택. `ANTHROPIC_API_KEY` 있으면 unset.
- **"requires a full-scope login token"**: `claude setup-token`/`CLAUDE_CODE_OAUTH_TOKEN`의 inference-only 토큰은 불가. `claude auth login`으로 full-scope 세션 토큰.
- **"Unable to determine your organization"**: 캐시된 계정 정보가 stale. `claude auth login`.
- **"not yet enabled for your account"**: `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC`/`DISABLE_TELEMETRY` 또는 `CLAUDE_CODE_USE_BEDROCK`/`_USE_VERTEX`/`_USE_FOUNDRY` 환경변수 영향. 해당 없으면 `/logout` 후 `/login`.
- **"disabled by your organization's policy"**: `/status`로 로그인 방식 확인. (1) API 키/Console 계정 → `/login` claude.ai, (2) 어드민 미활성 → 토글, (3) 토글 grayed out → 데이터 보존·컴플라이언스 비호환(Anthropic 지원 문의), (4) `disableRemoteControl` 언급 → IT 어드민이 [[04 설정]] managed settings로 비활성.
- **"Remote credentials fetch failed"**: `--verbose`로 전체 에러. 미로그인, 네트워크/proxy(443 차단), 구독 비활성.

## 링크에서 세션 시작 (Deep links)

> [!note] Deep links는 v2.1.91 이상.

deep link는 새 터미널 창에 Claude Code를 여는 `claude-cli://` URL이다. 작업 디렉터리와 미리 채울 프롬프트를 담을 수 있어, Claude Code가 설치된 누구나 클릭하면 프롬프트가 이미 입력된 세션이 열린다(프롬프트는 채워질 뿐 **Enter를 누를 때까지 전송 안 됨**). 인시던트 runbook, 모니터링 알림·대시보드, README/wiki 온보딩, CI 실패 알림 등 링크가 갈 수 있는 어디에나 둔다.

### 동작 방식

`claude-cli://`는 `mailto:`처럼 Claude Code가 OS에 등록하는 커스텀 URL 스킴이다. 클릭 시: ① 브라우저/앱이 URL을 OS에 넘김, ② OS가 prefix를 인식해 로컬 Claude Code 시작, ③ 링크가 지정한 디렉터리에서 Claude Code가 도는 새 터미널 창이 열리고 프롬프트가 입력 박스에 채워짐, ④ 읽고 편집한 뒤 Enter로 전송. 링크는 어디에 호스팅돼도 세션은 클릭한 컴퓨터에서 로컬로 열린다.

> [!note] 링크를 표시하는 플랫폼이 커스텀 URL 스킴을 허용해야 한다. GitHub 렌더 마크다운은 `http`/`https`만 허용하고 README·이슈·PR·wiki에서 `claude-cli://`를 strip(링크 텍스트만 보임). 우회는 아래 트러블슈팅.

deep link는 스스로 아무것도 실행하지 않는다 — 디렉터리를 고르고 프롬프트를 채울 뿐, 신뢰 안 하는 페이지의 링크여도 Enter 전까지 inert. 세션이 열리면 입력 박스 아래 `Prompt from an external link` 경고가 전송/클리어까지 표시되며, 1,000자 초과 프롬프트는 글자 수와 함께 스크롤해 전체 검토 후 Enter하라고 알림. 권한 규칙·`CLAUDE.md`·trust 프롬프트는 여느 세션과 동일 적용([[03 메모리와 컨텍스트]] [[05 권한]]).

### 링크 빌드

모든 링크는 `claude-cli://open`(핸들러가 받는 유일 경로)으로 시작. 최소형은 홈 디렉터리에 빈 프롬프트로 연다:

```text
claude-cli://open
```

| 파라미터 | 설명 |
|----------|------|
| `q` | 프롬프트 박스에 미리 채울 텍스트. URL-encode 필요. 멀티라인은 `%0A`. 최대 5,000자. |
| `cwd` | 작업 디렉터리 절대 경로. 네트워크·UNC 경로 거부. |
| `repo` | GitHub `owner/name` 슬러그. Claude Code가 이전에 본 로컬 클론으로 해석해 거기서 시작. 매칭 클론이 없으면 홈 디렉터리. |

`cwd`와 `repo`를 둘 다 주면 `cwd`가 우선(`cwd` 경로가 없어도 `repo` 무시).

```text
claude-cli://open?repo=acme/payments&q=Investigate%20the%20failed%20deploy%20of%20payments-api.%0ACheck%20recent%20commits%20to%20main%20and%20the%20last%20successful%20build.
```

#### `cwd` vs `repo` 선택

- `cwd`: 클릭하는 모두가 같은 절대 경로에 프로젝트를 가질 때(표준화된 devcontainer·VM 이미지).
- `repo`: 링크가 공유되고 각자 다른 위치에 클론할 때. 해석 규칙 — Git repo에서 `claude`를 돌릴 때마다 그 경로가 `owner/name`에 기록됨; deep link 도착 시 가장 최근 쓴 매칭 경로를 엶(클론·worktree 별도 추적); 한 번도 `claude`를 안 돌린 경로는 못 찾음; 브랜치는 안 바꿈(현재 상태 그대로). welcome 헤더가 고른 경로를 표시.

### 예시

runbook에 마크다운 링크로:

```markdown
## High 5xx rate on web-gateway

1. Acknowledge the page in PagerDuty.
2. [Open Claude Code in the gateway repo](claude-cli://open?repo=acme/web-gateway&q=5xx%20rate%20is%20elevated%20on%20web-gateway.%20Check%20recent%20deploys%2C%20error%20logs%20from%20the%20last%2030%20minutes%2C%20and%20open%20incidents%20in%20Linear.)
3. Post initial findings in #incident.
```

셸/스크립트/alias에서 OS의 URL 열기 명령으로:

```bash
# macOS
open "claude-cli://open?repo=acme/payments&q=review%20open%20PRs"
# Linux
xdg-open "claude-cli://open?repo=acme/payments&q=review%20open%20PRs"
```
```powershell
# Windows PowerShell
Start-Process "claude-cli://open?repo=acme/payments&q=review%20open%20PRs"
```
```cmd
:: Windows cmd.exe — 첫 인자가 창 제목으로 취급되니 빈 제목 먼저
start "" "claude-cli://open?repo=acme/payments&q=review%20open%20PRs"
```

### 등록과 지원 플랫폼

Claude Code는 macOS·Linux·Windows에서 처음 대화형 세션을 시작할 때 핸들러를 등록(별도 설치 명령 없음). 사용자 레벨에만 기록:

| 플랫폼 | 핸들러 위치 |
|--------|-------------|
| macOS | `~/Applications/Claude Code URL Handler.app` |
| Linux | `$XDG_DATA_HOME/applications`(기본 `~/.local/share/applications`)의 `claude-code-url-handler.desktop` |
| Windows | `HKEY_CURRENT_USER\Software\Classes\claude-cli` |

터미널 에뮬레이터 감지: macOS는 가장 최근 대화형 세션의 터미널 재사용(iTerm2·Ghostty·kitty·Alacritty·WezTerm·Terminal.app), Linux는 `$TERMINAL` → `x-terminal-emulator` → 흔한 목록, Windows는 Windows Terminal → PowerShell → `cmd.exe`. 등록을 막으려면 `settings.json`에서 `disableDeepLinkRegistration`을 `"disable"`로([[04 설정]]; 조직 강제는 managed settings).

### VS Code 탭으로 열기

VS Code 확장은 자체 핸들러 `vscode://anthropic.claude-code/open`을 등록해 터미널 창 대신 Claude Code 에디터 탭을 연다(파라미터는 [[14 IDE와 데스크톱]] 참조).

### 트러블슈팅

- **클릭해도 아무 일 없음**: 핸들러 미등록. 그 머신에서 대화형 `claude`를 한 번 실행·종료 후 재시도. 데스크톱 환경 없는 Linux는 `xdg-open`이 dispatch할 대상이 없을 수 있음.
- **링크가 평문으로 렌더**: 일부 렌더러가 `http`/`https`만 허용. GitHub README·이슈·PR·wiki에서 발생 — 코드 블록에 넣어 주소창에 붙여넣게 한다.
- **홈 디렉터리에서 열림**: `repo`는 이미 본 클론만 해석. 클론에서 `claude`를 한 번 돌리거나 `cwd` 절대 경로로 전환.
- **잘못된 터미널**: macOS는 선호 터미널에서 `claude`를 한 번 실행, Linux는 `$TERMINAL` 설정, Windows는 순서 고정(Windows Terminal 설치).

관련: [[07 스킬]](긴 runbook 프롬프트를 `/skill`로 저장해 `q`는 이름만 명명), [Non-interactive mode](#claude-code를-프로그래밍-방식으로-실행-headless)(터미널 없이 스크립트 실행).
