# 03 메모리와 컨텍스트

Claude Code는 매 세션을 빈 컨텍스트 윈도우로 시작한다. 세션 간 지식을 이어주는 두 축은 **사람이 쓰는 지시(CLAUDE.md)**와 **Claude가 스스로 쌓는 메모(auto memory)**이며, 작업 중에는 파일 읽기·룰·훅이 컨텍스트를 채운다. 이 노트는 메모리·컨텍스트 윈도우·세션·체크포인트·`.claude` 디렉터리·대형 코드베이스 운영을 사용법 레퍼런스로 정리한다. 허브는 [[Claude]], 권한 관련은 [[05 권한]], 설정 키는 [[04 설정]]을 참고하라.

---

## 메모리 개요: CLAUDE.md vs Auto memory

Claude Code에는 보완 관계인 두 메모리 시스템이 있고 둘 다 대화 시작 시 로드된다. 단, 이들은 **컨텍스트(맥락)이지 강제 설정이 아니다**. 무조건 차단하려면 메모리가 아니라 [[09 훅]]의 PreToolUse 훅을 써야 한다. 지시가 구체적·간결할수록 Claude가 더 일관되게 따른다.

|  | CLAUDE.md files | Auto memory |
| :-- | :-- | :-- |
| 작성 주체 | 사람(당신) | Claude |
| 내용 | 지시·규칙 | 학습된 패턴·인사이트 |
| 범위 | 프로젝트 / 사용자 / 조직 | 저장소(repo) 단위, worktree 간 공유 |
| 로드 | 매 세션 전체 | 매 세션 (`MEMORY.md` 첫 200줄 또는 25KB) |
| 용도 | 코딩 표준, 워크플로우, 아키텍처 | 빌드 명령, 디버깅 인사이트, Claude가 발견한 선호 |

서브에이전트도 자체 auto memory를 가질 수 있다 ([[08 서브에이전트와 에이전트 팀]] 참고).

## CLAUDE.md 파일

CLAUDE.md는 프로젝트·개인 워크플로우·조직 전체에 영속 지시를 주는 마크다운 파일이다. 사람이 평문으로 쓰고 Claude가 매 세션 시작 시 읽는다.

### 무엇을 CLAUDE.md에 넣을까

"다시 설명하게 될 것"을 적어두는 곳으로 취급하라. 다음 신호가 오면 추가한다.

- Claude가 같은 실수를 두 번째로 반복할 때
- 코드 리뷰가 이 코드베이스에서 Claude가 알았어야 할 것을 잡아낼 때
- 지난 세션에도 입력했던 정정/설명을 또 입력하고 있을 때
- 새 팀원도 동일한 맥락이 필요할 때

매 세션 들고 있어야 할 사실(빌드 명령, 컨벤션, 레이아웃, "항상 X 하라" 규칙)에 한정하라. 다단계 절차거나 코드베이스 일부에서만 의미 있으면 [[07 스킬]]이나 path-scoped 룰로 옮긴다.

### CLAUDE.md 위치와 범위 (로드 순서: 넓은 범위 → 좁은 범위)

| 범위 | 위치 | 목적 | 공유 대상 |
| :-- | :-- | :-- | :-- |
| **Managed policy** | macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`<br>Linux/WSL: `/etc/claude-code/CLAUDE.md`<br>Windows: `C:\Program Files\ClaudeCode\CLAUDE.md` | IT/DevOps가 관리하는 조직 전체 지시 | 조직 내 모든 사용자 |
| **User instructions** | `~/.claude/CLAUDE.md` | 모든 프로젝트 개인 선호 | 본인(전 프로젝트) |
| **Project instructions** | `./CLAUDE.md` 또는 `./.claude/CLAUDE.md` | 팀 공유 프로젝트 지시 | 팀(버전 관리) |
| **Local instructions** | `./CLAUDE.local.md` | 개인 프로젝트별 선호(`.gitignore`에 추가) | 본인(현 프로젝트) |

작업 디렉터리 상위 계층의 CLAUDE.md / CLAUDE.local.md는 시작 시 **전체 로드**되고, 하위 디렉터리의 파일은 그 안의 파일을 읽을 때 **온디맨드 로드**된다.

> [!tip] `/init`로 초기 생성
> `/init`를 실행하면 Claude가 코드베이스를 분석해 빌드/테스트 명령과 컨벤션이 담긴 CLAUDE.md를 자동 생성한다. 이미 있으면 덮어쓰지 않고 개선안을 제안한다. `CLAUDE_CODE_NEW_INIT=1`을 켜면 대화형 멀티페이즈 흐름(CLAUDE.md·skills·hooks 중 무엇을 세팅할지 묻고, 서브에이전트로 탐색 후 검토 가능한 제안 제시)이 된다. `/init` 시작은 [[01 시작하기]]도 참고.

### 효과적인 지시 작성

CLAUDE.md는 매 세션 컨텍스트에 로드되어 토큰을 소비한다. 강제 설정이 아니라 맥락이므로 작성 방식이 준수율에 직접 영향을 준다.

- **Size**: 파일당 200줄 미만 목표. 길수록 컨텍스트를 더 먹고 준수율이 떨어진다. 커지면 path-scoped 룰로 분리하라.
- **Structure**: 마크다운 헤더·불릿으로 그룹화. 조직화된 섹션이 dense paragraph보다 잘 따라진다.
- **Specificity**: 검증 가능할 만큼 구체적으로.
  - "Use 2-space indentation" > "Format code properly"
  - "Run `npm test` before committing" > "Test your changes"
  - "API handlers live in `src/api/handlers/`" > "Keep files organized"
- **Consistency**: 두 규칙이 모순되면 Claude가 임의로 하나를 고를 수 있다. 주기적으로 CLAUDE.md / 중첩 CLAUDE.md / `.claude/rules/`를 점검해 충돌·구식 지시를 제거하라.

### 다른 파일 import (`@path` 구문)

CLAUDE.md는 `@path/to/import` 구문으로 다른 파일을 가져올 수 있다. import된 파일은 launch 시 함께 펼쳐져 컨텍스트에 들어간다(조직화에는 좋지만 컨텍스트 절약은 안 됨). 상대/절대 경로 모두 가능하며 상대 경로는 import한 파일 기준. 재귀 import는 **최대 4 hop**.

```text
See @README for project overview and @package.json for available npm commands for this project.

# Additional Instructions
- git workflow @docs/git-instructions.md
```

git에 올리지 않을 개인 프로젝트 선호는 루트의 `CLAUDE.local.md`에 둔다(`.gitignore` 추가). 여러 worktree에서 개인 지시를 공유하려면 홈 디렉터리 파일을 import:

```text
# Individual Preferences
- @~/.claude/my-project-instructions.md
```

> [!warning] external import 승인
> 프로젝트에서 외부 import를 처음 만나면 승인 다이얼로그가 뜬다. 거부하면 import는 비활성 상태로 남고 다이얼로그도 다시 안 뜬다.

### AGENTS.md 호환

Claude Code는 `AGENTS.md`가 아니라 `CLAUDE.md`를 읽는다. 이미 `AGENTS.md`를 쓰는 저장소라면 이를 import하는 CLAUDE.md를 만들어 중복 없이 양쪽 도구가 같은 지시를 읽게 한다.

```markdown
@AGENTS.md

## Claude Code

Use plan mode for changes under `src/billing/`.
```

Claude 전용 내용이 필요 없으면 심볼릭 링크도 가능: `ln -s AGENTS.md CLAUDE.md` (Windows에선 관리자 권한/개발자 모드 필요 → `@AGENTS.md` import 권장). `/init`는 기존 `AGENTS.md`, `.cursorrules`, `.devin/rules/`, `.windsurfrules`도 읽어 반영한다.

### CLAUDE.md 로드 메커니즘

작업 디렉터리에서 루트까지 트리를 거슬러 올라가며 각 디렉터리의 `CLAUDE.md`/`CLAUDE.local.md`를 찾는다. 발견된 모든 파일은 **덮어쓰지 않고 연결(concatenate)**되며, 파일시스템 루트 → 작업 디렉터리 순으로 정렬된다. 즉 실행 위치에 가까운 지시가 더 나중에 읽힌다. 각 디렉터리 내에서는 `CLAUDE.local.md`가 `CLAUDE.md` 뒤에 붙는다.

- 블록 단위 HTML 주석(`<!-- ... -->`)은 컨텍스트 주입 전 제거된다(유지보수 노트용, 토큰 절약). 코드 블록 안 주석과 Read 도구로 직접 열 때는 유지.

#### 추가 디렉터리에서 메모리 로드

`--add-dir`로 준 디렉터리의 CLAUDE.md는 기본적으로 로드되지 않는다. 로드하려면 환경변수 설정:

```bash
CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1 claude --add-dir ../shared-config
```

이때 `CLAUDE.md`, `.claude/CLAUDE.md`, `.claude/rules/*.md`, `CLAUDE.local.md`가 로드된다(`--setting-sources`에서 `local`을 빼면 `CLAUDE.local.md`는 스킵). `--add-dir`는 [[02 CLI 레퍼런스]] 참고.

## `.claude/rules/` — 룰로 지시 조직화

큰 프로젝트에서는 지시를 `.claude/rules/`의 여러 파일로 모듈화할 수 있다. 룰은 특정 파일 경로에 **scoped** 가능해 매칭 파일을 읽을 때만 로드되어 노이즈·컨텍스트를 절약한다.

> [!note] 룰 vs 스킬
> 룰은 매 세션(또는 매칭 파일을 열 때) 컨텍스트에 들어간다. 항상 들고 있을 필요 없는 태스크별 지시는 [[07 스킬]]을 써라(호출/관련성 판단 시에만 로드).

### 룰 설정

`.claude/rules/`에 토픽별 `.md`를 둔다. 하위 디렉터리까지 재귀 발견된다.

```text
your-project/
├── .claude/
│   ├── CLAUDE.md           # Main project instructions
│   └── rules/
│       ├── code-style.md
│       ├── testing.md
│       └── security.md
```

`paths` frontmatter가 없는 룰은 launch 시 `.claude/CLAUDE.md`와 동일 우선순위로 로드된다.

### Path-specific rules

`paths` 필드로 특정 파일에 scoped. 매칭 파일을 읽을 때만 트리거된다(매 tool use가 아님).

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API Development Rules

- All API endpoints must include input validation
- Use the standard error response format
- Include OpenAPI documentation comments
```

| 패턴 | 매칭 |
| :-- | :-- |
| `**/*.ts` | 모든 디렉터리의 TS 파일 |
| `src/**/*` | `src/` 아래 모든 파일 |
| `*.md` | 프로젝트 루트의 마크다운 |
| `src/components/*.tsx` | 특정 디렉터리의 React 컴포넌트 |

brace expansion으로 여러 확장자도 가능:

```markdown
---
paths:
  - "src/**/*.{ts,tsx}"
  - "lib/**/*.ts"
  - "tests/**/*.test.ts"
---
```

### 룰 공유와 사용자 레벨 룰

- **심볼릭 링크**로 공유 룰을 여러 프로젝트에 연결 가능(순환 링크는 감지·처리됨).
  ```bash
  ln -s ~/shared-claude-rules .claude/rules/shared
  ln -s ~/company-standards/security.md .claude/rules/security.md
  ```
- **사용자 레벨 룰** `~/.claude/rules/`는 머신의 모든 프로젝트에 적용. 프로젝트 룰보다 먼저 로드되어 프로젝트 룰이 더 높은 우선순위를 갖는다.

## 대형 팀을 위한 CLAUDE.md 관리

### 조직 전체 CLAUDE.md 배포

managed policy 위치(위 표 참고)에 파일을 두면 머신의 모든 사용자·모든 저장소에 적용되며 개인 설정으로 제외 불가. MDM/Group Policy/Ansible로 배포한다. 별도 파일 대신 `managed-settings.json`의 `claudeMd` 키로 내용을 직접 넣을 수도 있다(user/project/local에 설정 시 무효).

```json
{
  "claudeMd": "Always run `make lint` before committing.\nNever push directly to main."
}
```

managed settings(기술적 강제)와 managed CLAUDE.md(행동 지침)는 역할이 다르다. 권한·샌드박스·env 강제는 [[05 권한]]·[[04 설정]]의 managed settings에서, 코드 스타일·컴플라이언스 알림 등 행동 지침은 managed CLAUDE.md에서.

### 특정 CLAUDE.md 제외 — `claudeMdExcludes`

모노레포에서 상위 CLAUDE.md가 관련 없을 때 path/glob으로 스킵. `.claude/settings.local.json`에 두면 머신 로컬.

```json
{
  "claudeMdExcludes": [
    "**/monorepo/CLAUDE.md",
    "/home/user/monorepo/other-team/.claude/rules/**"
  ]
}
```

절대 경로에 glob 매칭. user/project/local/managed 어느 [[04 설정]] 레이어에서도 설정 가능하며 배열은 레이어 간 **병합**된다. managed policy CLAUDE.md는 제외 불가.

## Auto memory

Auto memory는 당신이 아무것도 쓰지 않아도 Claude가 세션 간 지식(빌드 명령, 디버깅 인사이트, 아키텍처 노트, 코드 스타일 선호, 워크플로우 습관)을 누적하게 한다. 매 세션 저장하는 게 아니라, 미래 대화에 유용할지를 판단해 결정한다.

> [!note] 버전 요구
> Auto memory는 Claude Code **v2.1.59 이상** 필요. `claude --version`으로 확인.

### 켜기/끄기

기본 ON. 토글 방법:

- `/memory` 세션 내 auto memory 토글
- 프로젝트 설정 `autoMemoryEnabled`:
  ```json
  { "autoMemoryEnabled": false }
  ```
- 환경변수 `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`

### 저장 위치

프로젝트별 메모리 디렉터리: `~/.claude/projects/<project>/memory/`. `<project>`는 git 저장소에서 파생되므로 같은 repo의 모든 worktree·하위 디렉터리가 하나의 auto memory 디렉터리를 **공유**한다(git 밖이면 프로젝트 루트 사용).

다른 위치를 쓰려면 `settings.json`의 `autoMemoryDirectory`(절대 경로 또는 `~/`로 시작). 어느 [[04 설정]] 스코프에서도 읽힌다. 단, 프로젝트 `.claude/settings.json`/`settings.local.json`에 둔 값은 워크스페이스 신뢰 다이얼로그를 수락해야 적용된다(훅과 동일 게이트).

```json
{ "autoMemoryDirectory": "~/my-custom-memory-dir" }
```

디렉터리 구조:

```text
~/.claude/projects/<project>/memory/
├── MEMORY.md          # 간결한 인덱스, 매 세션 로드
├── debugging.md       # 디버깅 패턴 상세 노트
├── api-conventions.md # API 설계 결정
└── ...                # Claude가 만든 기타 토픽 파일
```

auto memory는 **머신 로컬**이며 머신/클라우드 환경 간 공유되지 않는다.

### 작동 방식

`MEMORY.md`의 첫 **200줄 또는 25KB**(먼저 도달하는 것)가 매 대화 시작 시 로드된다. 그 이상은 시작 시 로드 안 됨. Claude는 상세 노트를 별도 토픽 파일로 옮겨 `MEMORY.md`를 간결하게 유지한다. 이 한도는 `MEMORY.md`에만 적용되고, **CLAUDE.md는 길이와 무관하게 전체 로드**된다(짧을수록 준수율은 좋음). 토픽 파일은 시작 시 로드되지 않고 필요할 때 온디맨드로 읽힌다. UI에 "Writing memory"/"Recalled memory"가 보이면 메모리를 갱신/읽는 중이다.

### 감사·편집

auto memory 파일은 언제든 편집/삭제 가능한 평문 마크다운. `/memory`로 세션 내에서 열람한다.

## `/memory` — 메모리 보기·편집

`/memory`는 현재 세션에 로드된 모든 CLAUDE.md·CLAUDE.local.md·rules 파일을 나열하고, auto memory를 토글하며, auto memory 폴더 링크를 제공한다. 파일을 선택하면 에디터에서 열린다.

"always use pnpm, not npm" 같이 기억하라고 하면 auto memory에 저장된다. CLAUDE.md에 넣으려면 "add this to CLAUDE.md"처럼 명시하거나 `/memory`로 직접 편집한다.

## 메모리 트러블슈팅

### CLAUDE.md를 안 따를 때

CLAUDE.md 내용은 시스템 프롬프트가 아니라 **시스템 프롬프트 뒤 user message**로 전달된다. 따라서 엄격한 준수가 보장되지 않으며 모호·충돌 지시일수록 더 그렇다. 디버그:

- `/memory`로 파일이 실제 로드되는지 확인(목록에 없으면 Claude가 못 봄)
- 로드되는 위치인지 점검(위 위치 표)
- 더 구체적으로("Use 2-space indentation" > "format code nicely")
- 파일 간 충돌 지시 제거

특정 시점에 반드시 실행돼야 하는 것(매 커밋 전, 매 파일 편집 후)은 [[09 훅]]으로 작성하라. 시스템 프롬프트 레벨이 필요하면 `--append-system-prompt`(매 호출마다 전달해야 하므로 스크립트/자동화에 적합).

> [!tip] `InstructionsLoaded` 훅
> 어떤 지시 파일이 언제·왜 로드되는지 로깅하려면 `InstructionsLoaded` 훅을 사용. path-specific 룰이나 lazy-load 파일 디버깅에 유용. 자세한 건 [[09 훅]].

### auto memory가 무엇을 저장했는지 모를 때

`/memory`에서 auto memory 폴더를 열어 열람·편집·삭제(전부 평문 마크다운).

### CLAUDE.md가 너무 클 때

200줄 초과 시 컨텍스트를 더 먹고 준수율이 떨어질 수 있다. path-scoped 룰로 매칭 파일에서만 로드하거나 불필요한 내용을 다듬어라. `@path` import는 조직화에 도움이 되나 컨텍스트는 줄지 않는다(launch 시 로드되므로).

### `/compact` 후 지시가 사라진 것 같을 때

**프로젝트 루트 CLAUDE.md는 compaction을 견딘다**: `/compact` 후 Claude가 디스크에서 다시 읽어 재주입한다. 하위 디렉터리의 중첩 CLAUDE.md는 자동 재주입되지 않고 그 디렉터리 파일을 다시 읽을 때 reload된다. compaction 후 사라진 지시는 대화에서만 줬거나 아직 reload 안 된 중첩 CLAUDE.md에 있던 것이다. (아래 [What survives compaction](#what-survives-compaction-컴팩션-이후-생존표) 참고)

## 컨텍스트 윈도우

컨텍스트 윈도우는 Claude가 세션에 대해 아는 모든 것을 담는다: 당신의 지시, 읽은 파일, 자신의 응답, 그리고 터미널에 안 보이는 콘텐츠까지. 기본 한도는 약 **200K 토큰**. 공식 문서의 인터랙티브 타임라인이 startup → compaction까지 한 세션이 컨텍스트를 어떻게 채우는지 시뮬레이션한다.

### 시작 전에 자동 로드되는 것 (당신이 타이핑하기 전)

당신의 첫 프롬프트 전에 이미 많은 게 로드된다(대표 토큰값은 예시):

| 항목 | 대략 토큰 | 터미널 가시성 | 설명 |
| :-- | :-- | :-- | :-- |
| System prompt | ~4,200 | 안 보임 | 행동·도구 사용·응답 포맷 핵심 지시. 항상 첫 로드 |
| Auto memory (`MEMORY.md`) | ~680 | 안 보임 | 이전 세션의 자기 노트. 첫 200줄/25KB 로드 |
| Environment info | ~280 | 안 보임 | 작업 디렉터리·플랫폼·셸·OS·git 여부. git branch/status/최근 커밋은 시스템 프롬프트 맨 끝 별도 블록 |
| MCP tools (deferred) | ~120 | 안 보임 | MCP 도구 **이름**만. 스키마는 기본 deferred, 필요 시 tool search로 온디맨드 |
| Skill descriptions | ~450 | 안 보임 | 사용 가능 스킬의 한 줄 설명. 본문은 실제 사용 시 로드. **compaction 후 재주입 안 됨** |
| `~/.claude/CLAUDE.md` | ~320 | 안 보임 | 전역 개인 선호 |
| Project CLAUDE.md | ~1,800 | 안 보임 | 프로젝트 컨벤션·빌드·아키텍처. 가장 중요한 파일 |

> [!tip] MCP tool search
> 기본은 스키마 deferred. `ENABLE_TOOL_SEARCH=auto`면 컨텍스트의 10% 안에 맞을 때 미리 로드, `ENABLE_TOOL_SEARCH=false`면 전부 로드. 자세한 건 [[10 MCP]].

당신의 셋업(예: [[12 출력 스타일과 상태줄]]의 output style, `--append-system-prompt` 텍스트)은 같은 방식으로 시스템 프롬프트에 들어가 더 추가될 수 있다.

### Claude가 작업하는 동안

- **파일 읽기**가 컨텍스트의 큰 비중을 차지한다. 한 파일 Read에 수천 토큰. 터미널엔 "Read auth.ts" 한 줄만 보여도 내용 토큰은 Claude만 본다 → 프롬프트를 구체적으로("fix the bug in auth.ts") 써서 읽는 파일 수를 줄여라.
- **path-scoped 룰**은 매칭 파일을 읽을 때 자동 로드된다("Loaded .claude/rules/..." 한 줄 표시).
- **PostToolUse 훅**(예: prettier)은 매 편집 후 발동. 훅 출력은 `hookSpecificOutput.additionalContext` JSON 필드로만 컨텍스트에 들어간다. exit 0의 평문 stdout은 들어가지 않고 debug 로그로만 간다. PostToolUse는 exit code 2가 stderr를 에러로 표면화하나 도구가 이미 실행됐으므로 차단은 못 한다. 훅 상세는 [[09 훅]].
- 도구 출력(예: `npm test`)은 터미널에 요약만 보이고 전체 토큰은 컨텍스트에 들어간다.

### 가시성(terminal visibility) 3단계

| 표시 | 의미 |
| :-- | :-- |
| hidden | 터미널에 안 나타남 |
| brief | 한 줄 요약만 보임(전체 내용 아님) |
| full | 실제 내용이 터미널에 나타남 |

### 서브에이전트로 컨텍스트 절약

서브에이전트는 **별도의 새 컨텍스트 윈도우**에서 동작한다. CLAUDE.md와 MCP·skill 셋업은 로드하지만 **메인 세션의 대화 히스토리나 auto memory는 받지 않는다**. 서브에이전트가 6,100토큰의 파일을 읽어도 메인 컨텍스트에는 **최종 요약(예: 420토큰)과 작은 메타데이터 트레일러(토큰 수·소요시간)만** 돌아온다 — 이것이 컨텍스트 절약의 핵심이다.

- 서브에이전트는 자체(짧은) 시스템 프롬프트를 받고, 메인의 auto memory는 미포함. 커스텀 에이전트가 frontmatter에 `memory:`를 두면 자신의 별도 `MEMORY.md`를 로드.
- 부모 도구 대부분을 받되 plan-mode 컨트롤, background-task 도구, 그리고 재귀 방지를 위해 기본적으로 Agent 도구 자체는 제외된다.
- 내장 Explore/Plan 에이전트는 더 작은 컨텍스트를 위해 CLAUDE.md 로드를 건너뛴다.
- 리서치성 무거운 읽기는 서브에이전트에 위임하라. 자세한 건 [[08 서브에이전트와 에이전트 팀]].

### Bang 명령과 user-only 스킬

- `!git status`처럼 `!` 접두로 셸 명령을 실행하면 명령과 출력이 당신 메시지의 일부로 컨텍스트에 들어간다. Claude가 직접 실행하지 않고도 명령 결과로 grounding할 때 유용. [[02 CLI 레퍼런스]]의 interactive bash mode 참고.
- `disable-model-invocation: true` 스킬은 startup 스킬 인덱스에 설명이 없어 호출 전까지 **0 컨텍스트**다. 호출하는 순간에만 본문이 로드된다. commit/deploy/메시지 전송 등 부수효과가 있는 스킬에 권장. [[07 스킬]] 참고.

### `/compact` — 컴팩션

`/compact`는 대화를 구조화된 요약으로 치환한다("Conversation compacted" 표시, 요약 과정은 터미널에 안 보임). 요약은 당신의 요청·의도, 핵심 기술 개념, 검토/수정한 파일과 중요 코드 스니펫, 에러와 해결, 대기 작업, 현재 작업을 보존한다. 그러나 축어적 대화(전체 도구 출력·중간 추론)는 사라진다 — Claude는 작업을 참조할 수 있어도 앞서 읽은 정확한 코드는 갖지 못한다.

### What survives compaction (컴팩션 이후 생존표)

긴 세션이 컴팩션되면 대화 히스토리가 요약된다. 지시가 어떻게 로드됐는지에 따라 결과가 다르다.

| 메커니즘 | 컴팩션 이후 |
| :-- | :-- |
| System prompt, output style | 변경 없음(메시지 히스토리가 아님) |
| 프로젝트 루트 CLAUDE.md, unscoped 룰 | 디스크에서 재주입 |
| Auto memory | 디스크에서 재주입 |
| `paths:` frontmatter 룰 | 매칭 파일을 다시 읽을 때까지 소실 |
| 하위 디렉터리 중첩 CLAUDE.md | 그 디렉터리 파일을 다시 읽을 때까지 소실 |
| 호출된 스킬 본문 | 재주입(스킬당 5,000 / 총 25,000 토큰 상한, 오래된 것부터 드롭) |
| Hooks | 해당 없음(코드로 실행, 컨텍스트 아님) |

path-scoped 룰·중첩 CLAUDE.md는 트리거 파일을 읽을 때 메시지 히스토리에 들어가므로 컴팩션 때 요약돼 사라지고 매칭 파일을 다시 읽을 때 reload된다. compaction을 견디게 하려면 `paths:`를 빼거나 프로젝트 루트 CLAUDE.md로 옮겨라. 스킬 본문은 재주입되나 상한 초과 시 잘리므로(앞부분 유지) **중요 지시를 `SKILL.md` 앞쪽에** 두라.

### 컨텍스트가 찰 때

한도에 가까워지면 Claude Code가 **자동 컴팩션**하므로 윈도우가 가득 차도 세션이 끝나지 않는다(자동 패스는 `/compact`와 동일 방식). 자동 패스 전에 직접 행동할 수도 있다:

- **focus 컴팩션**: `/compact focus on the auth bug fix`처럼 긴 새 작업 전에 무엇을 남길지 지정
- **작업 간 clear**: 무관한 작업으로 전환할 때 `/clear`. 옛 대화가 다음에 필요한 파일을 밀어내고 매 메시지 토큰을 잡아먹음
- **큰 읽기 위임**: 리서치를 서브에이전트로 보내 파일 내용이 그쪽 컨텍스트에 머물게

작은 대화가 아니라 더 큰 윈도우가 필요하면 Opus 4.6 이상·Sonnet 4.6은 **100만 토큰** 컨텍스트를 지원한다(`[1m]` 모델 변형 선택). 컴팩션은 큰 한도에서도 동일하게 동작. 비용·성능은 [[19 비용과 성능]].

### 내 세션 점검

- `/context`: 현재 컨텍스트를 카테고리별로 실시간 분석하고 최적화 제안
- `/memory`: startup에 로드된 CLAUDE.md·auto memory 파일 확인

## 세션 관리

세션은 프로젝트 디렉터리에 묶인 저장된 대화다. 작업하는 동안 로컬에 계속 저장되어, 떠난 뒤 재개하거나 다른 접근을 위해 분기(branch)하거나 태스크 간 전환할 수 있다. 데스크톱 앱([[14 IDE와 데스크톱]]), 웹([[15 웹과 모바일]]), VS Code 확장은 각자 세션 히스토리를 유지한다. 아래는 CLI 기준.

### 세션 재개

| 명령 | 동작 |
| :-- | :-- |
| `claude --continue` | 현재 디렉터리의 가장 최근 세션 재개 |
| `claude --resume` | 세션 피커 열기 |
| `claude --resume <name>` | 이름으로 직접 재개 |
| `claude --from-pr <number>` | 해당 PR에 연결된 세션 재개 |
| `/resume` | 활성 세션 내에서 다른 대화로 전환 |

`claude -p`([[02 CLI 레퍼런스]]의 headless)나 Agent SDK([[22 Agent SDK — 시작]])로 만든 세션은 피커에 안 나오지만 session ID로 `claude --resume <session-id>` 재개 가능.

#### 피커가 보는 범위

세션은 프로젝트 디렉터리별 저장. 기본은 현재 worktree의 인터랙티브 세션 + `/add-dir`로 현 디렉터리를 추가한 다른 세션. `Ctrl+W`로 repo의 모든 worktree로, `Ctrl+A`로 머신의 모든 프로젝트로 확장. 다른 worktree 세션은 그 자리에서 재개되고, 무관한 프로젝트 세션은 `cd`+재개 명령이 클립보드에 복사된다.

이름 재개는 현 repo와 worktree를 가로질러 해석된다:

| 명령 | 정확 매칭 | 모호한 이름 |
| :-- | :-- | :-- |
| `claude --resume <name>` | 직접 재개 | 이름을 검색어로 채운 피커 열림 |
| `/resume <name>` | 직접 재개 | 에러 보고; 인자 없이 `/resume` 실행해 피커 |

### 세션 이름 짓기

병렬로 여러 태스크를 할 때 특히 중요.

| 시점 | 방법 |
| :-- | :-- |
| 시작 시 | `claude -n auth-refactor` |
| 세션 중 | `/rename auth-refactor` (이름이 프롬프트 바에 표시) |
| 피커에서 | 하이라이트 후 `Ctrl+R` |
| plan 수락 시 | plan mode에서 plan 수락 시 plan 내용으로 자동 명명(이미 설정했으면 유지). plan mode는 [[05 권한]] |

### 세션 피커 단축키

`/resume`(세션 내) 또는 `claude --resume`(인자 없이)로 연다.

| 단축키 | 동작 |
| :-- | :-- |
| `↑`/`↓` | 세션 간 이동 |
| `→`/`←` | 그룹 세션 펼치기/접기 |
| `Enter` | 하이라이트 세션 재개 |
| `Space` | 세션 내용 미리보기(`Ctrl+V`도 동작) |
| `Ctrl+R` | 하이라이트 세션 이름 변경 |
| `/` 또는 `Space` 외 인쇄 가능 문자 | 검색 모드. GitHub/GHE/GitLab/Bitbucket의 PR/MR URL을 붙여넣으면 그 세션을 찾음 |
| `Ctrl+A` | 머신의 모든 프로젝트 세션 표시(다시 누르면 현 repo) |
| `Ctrl+W` | 현 repo의 모든 worktree 세션(멀티 worktree repo에서만 표시) |
| `Ctrl+B` | 현재 git branch 세션만 필터(다시 누르면 전체) |
| `Esc` | 피커/검색 모드 종료 |

각 행은 세션 이름(없으면 요약/첫 프롬프트), 마지막 활동 이후 시간, 메시지 수, git branch를 보여준다. `/branch`·`/rewind`·`--fork-session`으로 만든 fork는 루트 세션 아래 그룹화(→로 펼침).

### 세션 분기 (`/branch`, `--fork-session`)

분기는 지금까지의 대화 복사본을 만들어 거기로 전환하며 원본은 그대로 둔다. 다른 접근을 시도할 때 사용.

```text
/branch try-streaming-approach
```

CLI에서는 `--continue`/`--resume`에 `--fork-session` 결합:

```bash
claude --continue --fork-session
```

`/branch`는 새 branch와 원본의 두 session ID를 출력한다. 원본 복귀는 그 ID를 `/resume`에 주거나 피커/`/resume <original-name>`. **"allow for this session"으로 승인한 권한은 새 branch로 이월되지 않는다**([[05 권한]]). fork 없이 같은 세션을 두 터미널에서 재개하면 메시지가 한 트랜스크립트에 섞인다. 단일 세션 내 체크포인트 되감기는 아래 체크포인팅 참고.

### 세션 내 컨텍스트 관리

| 명령 | 동작 |
| :-- | :-- |
| `/clear` | 빈 컨텍스트로 새 출발. 이전 대화는 저장·재개 가능 |
| `/compact [instructions]` | 히스토리를 요약으로 치환(선택적 포커스 지정) |
| `/context` | 현재 컨텍스트 소비 현황 표시 |

clear vs compact 전략은 [[20 베스트 프랙티스와 워크플로우]] 참고.

### 세션 데이터 export·위치

- `/export`: 현재 대화를 클립보드에 복사하거나 평문 파일로 저장(메시지·도구 출력을 읽기 좋게 렌더링). 파일명 주면 바로 기록.
- 트랜스크립트는 JSONL로 `~/.claude/projects/<project>/<session-id>.jsonl`에 저장(각 줄이 메시지/도구 사용/메타 JSON 객체). `~/.claude` 외에 저장하려면 `CLAUDE_CONFIG_DIR` 설정. 로컬 파일은 기본 **30일** 후 삭제 — `cleanupPeriodDays`로 변경([[04 설정]]).
- 트랜스크립트 기록을 완전히 막으려면 `CLAUDE_CODE_SKIP_PROMPT_HISTORY`, 또는 비대화 모드에서 `--no-session-persistence`.

관련: worktree로 격리 병렬 세션은 [[16 CI-CD와 팀 통합]]/대형 코드베이스 섹션, [[02 CLI 레퍼런스]]의 headless 참고.

## 체크포인팅 (Checkpointing)

Claude Code는 작업 중 Claude의 파일 편집을 자동 추적해, 무언가 어긋나면 빠르게 되돌리고 이전 상태로 rewind할 수 있게 한다.

### 자동 추적

Claude의 파일 편집 도구가 만든 모든 변경을 추적한다.

- 매 user 프롬프트가 새 체크포인트를 생성
- 체크포인트는 세션 간 지속되어 재개된 대화에서도 접근 가능
- 세션과 함께 **30일** 후 자동 정리(설정 가능)

### Rewind와 Summarize

`/rewind`를 실행하거나, 프롬프트 입력이 비었을 때 `Esc`를 두 번 눌러 rewind 메뉴를 연다. (입력에 텍스트가 있으면 더블 `Esc`는 입력을 지운다 — 지워진 텍스트는 입력 히스토리에 저장돼 `Up`으로 복구.) interactive mode 단축키는 [[02 CLI 레퍼런스]].

메뉴는 세션 중 보낸 각 프롬프트를 나열한다. 지점을 고른 뒤 행동 선택:

| 행동 | 효과 |
| :-- | :-- |
| **Restore code and conversation** | 코드·대화 모두 그 지점으로 되돌림 |
| **Restore conversation** | 대화만 되돌리고 현재 코드 유지 |
| **Restore code** | 파일 변경만 되돌리고 대화 유지 |
| **Summarize from here** | 선택 메시지 이후를 요약으로 압축(컨텍스트 공간 확보) |
| **Summarize up to here** | 선택 메시지 이전을 요약, 이후는 그대로 유지 |
| **Never mind** | 변경 없이 메시지 목록으로 복귀 |

conversation 복원/Summarize from here 후엔 선택 메시지의 원본 프롬프트가 입력 필드로 복원되어 재전송/편집 가능. Summarize up to here는 대화 끝에서 빈 입력으로 남는다.

#### Restore vs Summarize

restore는 상태를 되돌리고(코드·대화·둘 다), summarize는 디스크 파일을 바꾸지 않고 대화 일부를 AI 요약으로 압축한다.

- **Summarize from here**: 선택 메시지 이전은 그대로, 선택 메시지와 이후를 요약 치환. 곁가지 토론을 버리되 초기 컨텍스트는 상세 유지할 때.
- **Summarize up to here**: 선택 메시지 이전을 요약, 선택 메시지와 이후는 그대로(대화 끝에 머묾). 초기 셋업 토론을 압축하되 최근 작업은 상세 유지할 때.

두 경우 모두 원본 메시지는 트랜스크립트에 보존되어 Claude가 필요 시 참조 가능. 요약 포커스를 안내하는 선택적 지시를 타이핑할 수 있다. `/compact`와 유사하나 타깃형(전체가 아니라 선택 메시지의 어느 쪽을 압축할지 고름).

> [!note] Summarize vs Fork
> Summarize는 같은 세션에 머물며 컨텍스트를 압축한다. 원본 세션을 그대로 둔 채 다른 접근으로 갈라지려면 [fork](#세션-분기-branch---fork-session)(`claude --continue --fork-session`)를 써라.

### 활용 사례

- **대안 탐색**: 시작점을 잃지 않고 다른 구현 시도
- **실수 복구**: 버그를 낳거나 기능을 깬 변경을 빠르게 undo
- **기능 반복**: 동작하는 상태로 되돌릴 수 있다는 안전망으로 변형 실험
- **컨텍스트 공간 확보**: 장황한 디버깅 세션을 중간부터 요약, 초기 지시는 온전히 유지

### 한계 (중요)

- **Bash 명령 변경 미추적**: `rm`/`mv`/`cp` 등 bash로 수정한 파일은 rewind로 되돌릴 수 없다. Claude의 파일 편집 도구를 통한 직접 편집만 추적된다.
- **외부 변경 미추적**: 현재 세션에서 편집된 파일만 추적. Claude Code 밖 수동 변경이나 동시 다른 세션의 편집은(같은 파일을 건드리지 않는 한) 보통 캡처되지 않는다.
- **버전 관리 대체 아님**: 체크포인트는 세션 수준 빠른 복구용이다. 영구 이력·협업은 계속 Git을 써라. 체크포인트는 "local undo", Git은 "permanent history".

## 대형 코드베이스 / 모노레포 운영

대형 코드베이스(수백만 줄의 단일 트리 또는 다수 패키지 모노레포)에서는 작은 프로젝트 기준 디폴트가 컨텍스트를 무관한 지시·파일 읽기로 채워 토큰을 낭비하고 성능을 떨어뜨린다. 핵심은 Claude를 태스크가 건드리는 부분으로 **스코핑**하는 것. 각 설정은 독립적이며 서로 교체가 아니라 **레이어링**된다.

### 어디서 Claude를 시작할지

| 시작 위치 | 파일 접근 | launch 시 로드되는 CLAUDE.md | 적합한 경우 |
| :-- | :-- | :-- | :-- |
| 저장소 루트 | 모든 파일 | 루트만; 하위는 거기서 파일 읽을 때 온디맨드 | 여러 패키지/서브시스템에 걸친 태스크 |
| 하위 디렉터리 | 그 서브트리만(추가 권한 전까지) | 그 디렉터리 + 모든 상위 | 한 패키지/서브시스템에 스코프된 작업 |

`.claude/settings.json`은 **시작 디렉터리에서만** 로드되며 CLAUDE.md처럼 상위에서 상속되지 않는다(루트 settings는 루트에서 시작할 때만 적용).

### 디렉터리별 CLAUDE.md 레이어링

흔한 2단계 분할:

- **루트 `CLAUDE.md`**: 어디서나 적용(코딩 표준, 커밋 컨벤션, repo 레이아웃)
- **하위 디렉터리별 `CLAUDE.md`**: 해당 영역 스택 컨벤션(모노레포는 패키지당, 단일 트리는 `src/db/`·`src/api/` 등 서브시스템당)

루트 예시:

```markdown
This is a monorepo with three packages under packages/:

- packages/api: Node.js REST API with Express, TypeScript, and PostgreSQL
- packages/web: React frontend with Vite, TypeScript, and TailwindCSS
- packages/shared: shared TypeScript utilities used by both api and web

Run commands from the package directory, not the monorepo root.
Each package has its own tsconfig.json, package.json, and test suite.
```

`packages/api/CLAUDE.md`:

```markdown
This package is the REST API server.

- Run tests: `npm test` (uses Vitest)
- Run dev server: `npm run dev` (port 3001)
- Database migrations: `npm run migrate`
- Environment variables: copy `.env.example` to `.env`

API routes are in src/routes/. Each route file exports an Express router.
Database queries use Knex in src/db/. Never write raw SQL strings in route handlers.
```

`packages/api/`에서 시작하면 루트와 `packages/api/CLAUDE.md`만 로드되고 `packages/web/`는 컨텍스트에 없다. 파일 최신화: PR에서 CLAUDE.md 편집도 리뷰, 메이저 모델 릴리스 후 재검토(옛 모델 한계 우회 규칙은 신모델에선 오버헤드), Stop 훅으로 세션 트랜스크립트를 받아 CLAUDE.md 업데이트 제안([[09 훅]]).

#### 디렉터리별 CLAUDE.md vs path-scoped 룰

| 접근 | 파일 위치 | 로드 시점 | 사용 |
| :-- | :-- | :-- | :-- |
| 디렉터리별 `CLAUDE.md` | 디렉터리 안, 코드 옆 | 그 디렉터리에서 시작 시 launch, 또는 거기 파일 읽을 때 온디맨드 | 디렉터리 오너가 자기 컨벤션 유지, 코드와 함께 버전 관리 |
| path-scoped 룰(`.claude/rules/`) | repo 루트 중앙 `.claude/` | `paths:` glob 매칭 파일을 다룰 때 | 모든 컨벤션을 한 곳에, 또는 흩어진 여러 경로에 같은 룰 |

### 읽기 줄이기

#### 생성/벤더 코드 읽기 차단

검색은 기본적으로 `.gitignore`를 존중하므로 `node_modules/`·`dist/`·`build/`는 추가 설정 없이 결과에서 빠진다. 체크인된 벤더 SDK나 커밋된 생성 코드는 `permissions.deny`에 `Read` deny 룰 추가:

```json
{
  "permissions": {
    "deny": [
      "Read(./**/dist/**)",
      "Read(./**/build/**)",
      "Read(./**/*.generated.*)",
      "Read(./vendor/**)"
    ]
  }
}
```

deny 룰은 내장 파일 도구와 인식된 Bash 파일 명령(`cat`/`head`/`grep`/`find`)에 적용되나, 재귀 검색 출력에서 경로를 필터링하지는 않고 파일을 직접 여는 임의 서브프로세스도 못 막는다. 패턴 문법은 [[05 권한]]. 모든 세션에 강제하려면 managed settings에([[04 설정]]).

#### 코드 인텔리전스 플러그인

심볼 정의/사용처를 찾을 때 수많은 파일 읽기·grep 대신 language server로 점프·참조 검색·타입 에러 표면화. 공식 마켓플레이스에 TS·Python·Go·Rust 등 플러그인 존재:

```shell
/plugin install typescript-lsp@claude-plugins-official
```

repo 전체에 켜려면 `enabledPlugins` 프로젝트 설정에 추가. 각 머신에 language server 바이너리 필요. 플러그인 일반은 [[11 플러그인]]. `claudeMdExcludes`·`Read` deny와 잘 어울린다(무관 콘텐츠를 컨텍스트 밖으로, 코드 인텔리전스가 남은 것을 훑지 않게).

### worktree·파일 접근 스코핑

#### 필요한 디렉터리만 체크아웃 — `worktree.sparsePaths`

`--worktree`는 새 git worktree에서 세션을 시작해 변경을 격리한다(기본은 전체 repo 체크아웃). `worktree.sparsePaths`는 git sparse-checkout으로 나열한 디렉터리 + 루트 레벨 파일만 디스크에 써서 worktree가 더 빨리 시작하고 공간을 덜 쓴다.

```json
{
  "worktree": {
    "sparsePaths": [".claude", "packages/api", "packages/shared"]
  }
}
```

경로는 **repo 루트 기준**(어느 하위에서 시작하든). 디렉터리만 나열하고 개별 파일은 안 됨. `package.json`·lock 파일 등 루트 레벨 **파일**은 항상 함께 체크아웃되나 루트 레벨 **디렉터리**는 아니므로, 루트의 `.claude/settings.json`·`rules/`·`skills/`가 worktree 안에 필요하면 `.claude`를 목록에 넣어라. 서브에이전트 worktree 격리에 특히 유용(각 서브에이전트가 경량 체크아웃). 한 세션의 모든 worktree는 같은 `sparsePaths`를 공유하므로 서로 다른 패키지가 필요하면 모두 나열.

`node_modules` 같은 큰 디렉터리 중복을 피하려면 `symlinkDirectories`와 함께:

```json
{
  "worktree": {
    "sparsePaths": [".claude", "packages/api", "packages/shared"],
    "symlinkDirectories": ["node_modules"]
  }
}
```

> [!note] worktree 설정 로드 위치
> `sparsePaths`·`symlinkDirectories`는 worktree 생성 **전** 시작 디렉터리에서 읽힌다. 생성 후 작업 디렉터리는 worktree 루트(시작한 하위가 아님)이므로, worktree 안에서 필요한 다른 설정(권한 룰·훅 등)은 **repo 루트의 `.claude/settings.json`**에 두어야 한다.

#### 패키지/저장소 간 접근 — `additionalDirectories` / `--add-dir`

하위에서 시작했는데 형제 패키지나 다른 체크아웃을 읽고 편집해야 할 때:

```json
{
  "permissions": {
    "additionalDirectories": ["../shared", "../web"]
  }
}
```

상대 경로는 시작 디렉터리 기준. 런타임에 설정 없이 `claude --add-dir ../shared`로도 가능. CLAUDE.md·룰·스킬 로드 여부는 방식에 따라 다르다:

| 추가 방식 | CLAUDE.md·룰 로드 | 스킬 로드 |
| :-- | :-- | :-- |
| `additionalDirectories` 설정 | 절대 안 함 | 절대 안 함 |
| `--add-dir` 플래그 / `/add-dir` 명령 | 환경변수 설정 시에만 | 함 |

`--add-dir`로 추가한 디렉터리의 CLAUDE.md·룰을 로드하려면 `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1`(설정 방식엔 효과 없음).

### 디렉터리별 스킬

하위 디렉터리는 자기 스택에 스코프된 [[07 스킬]]을 정의할 수 있다(`.claude/skills/`). 관련성 판단 시 온디맨드 로드되어 API 스킬이 프론트 작업 중 컨텍스트를 안 먹는다.

```bash
mkdir -p packages/api/.claude/skills/api-testing
```

```markdown
---
name: api-testing
description: Testing patterns for the API package. Use when writing or modifying tests in packages/api/.
---

## Test structure
...
```

파일 패턴으로 스코프하려면 `paths` frontmatter(글로벌 `.claude/skills/`에 두되 `**/migrations/**` 같은 특정 파일에만 적용).

#### 스킬 발견성 유지

Claude는 모든 발견된 스킬의 name·description을 읽어 고른다(선택된 스킬 본문만 로드). 스코프는 시작 위치에 의존:

- **하위(`packages/api/`)에서**: 그 디렉터리 + 루트까지 모든 상위 + user/enterprise 레벨
- **루트에서**: 세션 중 건드린 모든 하위의 스킬(수백 개로 누적 가능)
- **`--add-dir`로 형제 추가 후**: 그 형제 스킬도 로드(`additionalDirectories`는 파일 접근만, 스킬 로드 안 함)

name은 항상 로드되나 description은 많을 때 잘릴 수 있어 Claude의 판단 키워드가 사라질 수 있다 → description을 짧게, 요청에 들어갈 단어를 앞에("writing or modifying tests in `packages/api/`"). 여러 디렉터리가 공유하는 스킬(PR 컨벤션, 배포 체크리스트)은 루트 `.claude/skills/`에. 버전 이력/저장소 간 사용이 필요하면 플러그인으로(`plugin-name:skill-name` 네임스페이스로 충돌 방지). 미사용 스킬 탐지는 OpenTelemetry 로그 익스포터 + `OTEL_LOG_TOOL_DETAILS=1`로 `skill_activated` 이벤트의 `skill.name`·`invocation_trigger` 기록([[19 비용과 성능]]).

### 레이어링이 한계에 다다르면 — 컨벤션 중앙화

디렉터리별 CLAUDE.md는 커지면 거버넌스가 어려워진다(드리프트, 구식화, 루트 오너 부재). 항상 로드되는 CLAUDE.md 밖, 온디맨드 로드 메커니즘으로 옮겨라:

- [[07 스킬]]: 태스크에 관련될 때만 로드하는 참조 자료
- [[11 플러그인]]: 플랫폼 팀이 중앙 소유하는 스킬·훅·명령 버전 번들
- [[10 MCP]]: 조직이 이미 코드 검색/RAG 인덱스를 운영하면 MCP 도구로 노출해 파일 직접 읽기 대신 질의

#### 세션 시작 시 올바른 플러그인 추천

컨벤션이 플러그인에 들어가면 낯선 영역에서 시작한 팀원은 어느 플러그인을 써야 할지 모른다. `SessionStart` 훅이 stdout에 출력한 것은 첫 프롬프트 전 컨텍스트에 추가되므로, 시작 디렉터리를 hook input에서 읽어 path-to-plugin 맵에 조회해 추천을 출력하게 할 수 있다([[09 훅]]).

### 종합 예시 (`packages/api/`에서 시작)

```json packages/api/.claude/settings.json
{
  "worktree": {
    "sparsePaths": [".claude", "packages/api", "packages/shared"],
    "symlinkDirectories": ["node_modules"]
  },
  "permissions": {
    "additionalDirectories": ["../shared"],
    "deny": ["Read(./**/dist/**)", "Read(./**/build/**)"]
  }
}
```

worktree 세션이 deny 룰을 집도록 repo 루트 `.claude/settings.json`에 deny 룰 복사본을 둔다(worktree 작업 디렉터리는 루트라 위 파일은 로드 안 됨):

```json .claude/settings.json
{
  "permissions": {
    "deny": ["Read(./**/dist/**)", "Read(./**/build/**)"]
  }
}
```

결과: `packages/api/`에서 시작 시 루트+`packages/api/CLAUDE.md` 로드(`web` 스킵), `api`·`shared` 읽기·편집 가능, `dist`/`build` 읽기 스킵, api-testing 스킬 온디맨드, worktree에 `.claude`/`api`/`shared`+루트 파일 + 루트 deny 룰 적용.

### 패키지 간 변경 스코핑·계획

설정이 Claude가 보는 것을 통제한다면, 여러 패키지를 건드리는 변경에선 스코프·시퀀싱도 결과에 영향을 준다.

- **한 세션에 전체 변경 넘기기**: 공유 편집과 호출처를 함께 줘 패키지마다 결정을 재유도하지 않게
- **편집 전 계획을 파일로 저장**: plan first 후 markdown으로 저장. 긴 패키지 간 세션은 도중 컴팩션되므로 저장된 계획이 대화 히스토리보다 오래 살아남음([[20 베스트 프랙티스와 워크플로우]])

조직 롤아웃은 [[16 CI-CD와 팀 통합]]도 참고.

## `.claude` 디렉터리 레퍼런스

Claude Code가 CLAUDE.md·settings·hooks·skills·commands·subagents·workflows·rules·auto memory를 읽는 곳. 프로젝트 스코프 파일은 repo의 `.claude/`(CLAUDE.md·`.mcp.json`·`.worktreeinclude`는 루트), 글로벌 스코프 파일은 `~/.claude/`(전 프로젝트 적용).

### 변경이 어디에 속하는지

| 하고 싶은 것 | 편집 | 범위 | 참고 |
| :-- | :-- | :-- | :-- |
| 프로젝트 컨텍스트·컨벤션 | `CLAUDE.md` | project/global | 메모리(이 노트) |
| 특정 도구 호출 허용/차단 | `settings.json` `permissions`/`hooks` | project/global | [[05 권한]], [[09 훅]] |
| 도구 호출 전후 스크립트 실행 | `settings.json` `hooks` | project/global | [[09 훅]] |
| 세션 환경변수 | `settings.json` `env` | project/global | [[04 설정]] |
| 개인 오버라이드를 git 밖에 | `settings.local.json` | project only | [[04 설정]] |
| `/name`으로 호출하는 프롬프트/기능 | `skills/<name>/SKILL.md` | project/global | [[07 스킬]] |
| 자체 도구를 가진 전문 서브에이전트 | `agents/*.md` | project/global | [[08 서브에이전트와 에이전트 팀]] |
| 스크립트로 다수 서브에이전트 오케스트레이션 | `workflows/*.js` | project/global | [[13 자동화와 스케줄링]] |
| MCP로 외부 도구 연결 | `.mcp.json` | project only | [[10 MCP]] |
| 응답 포맷 변경 | `output-styles/*.md` | project/global | [[12 출력 스타일과 상태줄]] |

### 파일 레퍼런스 (로드 시점 포함)

| 파일 | 범위 | 커밋 | 로드 시점 / 역할 |
| :-- | :-- | :-- | :-- |
| `CLAUDE.md` | project+global | ✓ | 매 세션 시작 시 컨텍스트 로드 |
| `rules/*.md` | project+global | ✓ | `paths:` 없으면 시작 시, 있으면 매칭 파일 진입 시 |
| `settings.json` | project+global | ✓ | 권한·훅·env·모델 기본값. global을 오버라이드 |
| `settings.local.json` | project only | (gitignore) | 사용자 편집 설정 중 최고 우선(CLI 플래그·managed는 우선) |
| `.mcp.json` | project only | ✓ | 세션 시작 시 서버 연결, 스키마는 deferred → tool search 온디맨드 |
| `.worktreeinclude` | project only | ✓ | worktree 생성(`--worktree`/`EnterWorktree`/서브에이전트 `isolation: worktree`) 시 복사할 gitignored 파일 |
| `skills/<name>/SKILL.md` | project+global | ✓ | `/name` 또는 Claude가 태스크 매칭 시 |
| `commands/*.md` | project+global | ✓ | `/command-name` 입력 시(스킬과 동일 메커니즘; 신규는 skills/ 권장) |
| `output-styles/*.md` | project+global | ✓ | `outputStyle` 설정 선택 시 세션 시작에 적용 |
| `agents/*.md` | project+global | ✓ | 호출/위임 시 자체 컨텍스트 윈도우에서 실행, @-멘션 가능 |
| `workflows/*.js` | project+global | ✓ | 시작 시 로드, 각 파일이 `/<name>` 명령이 됨 |
| `agent-memory/<name>/MEMORY.md` | project+global | ✓ | 서브에이전트 실행 시 첫 200줄(25KB 상한)이 서브에이전트 시스템 프롬프트에 로드 |
| `~/.claude.json` | global only | | 앱 상태·OAuth·UI 토글·개인 MCP 서버. `/config`·trust 승인 시 기록 |
| `projects/<project>/memory/` | global only | | auto memory(`MEMORY.md` 시작 시, 토픽 파일 온디맨드) |
| `keybindings.json` | global only | | 커스텀 단축키. 시작 시 읽고 편집 시 핫리로드 |
| `themes/*.json` | global only | | 커스텀 색 테마. `/theme`에 표시, 핫리로드 |

> [!note] 오버라이드 우선순위
> managed settings(조직 배포) > CLI 플래그(`--permission-mode`, `--settings`) > 일부 환경변수(설정별로 다름, env-vars 레퍼런스 확인) > settings 파일. 전체 순서는 [[04 설정]]의 settings precedence.

### 탐색기에 안 나오는 관련 파일

| 파일 | 위치 | 역할 |
| :-- | :-- | :-- |
| `managed-settings.json` | 시스템 레벨(OS별) | 오버라이드 불가 엔터프라이즈 설정([[04 설정]]) |
| `CLAUDE.local.md` | 프로젝트 루트 | 이 프로젝트 개인 선호(수동 생성 + `.gitignore`) |
| 설치 플러그인 | `~/.claude/plugins` | 클론 마켓플레이스·설치 버전·플러그인 데이터(`claude plugin`로 관리, 업데이트/제거 7일 후 고아 버전 삭제) |

## 애플리케이션 데이터 (`~/.claude`)

작성하는 설정 외에 `~/.claude`는 세션 중 Claude Code가 쓰는 데이터를 담는다. 이 파일들은 **평문**이며, 도구를 거치는 모든 것(파일 내용·명령 출력·붙여넣은 텍스트)이 디스크 트랜스크립트에 남는다.

### 자동 정리 (`cleanupPeriodDays`, 기본 30일)

시작 시 해당 일수보다 오래된 것 삭제.

| `~/.claude/` 하위 경로 | 내용 |
| :-- | :-- |
| `projects/<project>/<session>.jsonl` | 전체 대화 트랜스크립트(모든 메시지·도구 호출·결과) |
| `projects/<project>/<session>/subagents/` | 서브에이전트 트랜스크립트(부모와 함께 만료) |
| `projects/<project>/<session>/tool-results/` | 별도 파일로 분리된 큰 도구 출력 |
| `file-history/<session>/` | Claude가 바꾼 파일의 편집 전 스냅샷(체크포인트 복원용) |
| `plans/` | plan mode에서 작성된 계획 파일 |
| `debug/` | 세션별 디버그 로그(`--debug`/`/debug` 시에만) |
| `paste-cache/`, `image-cache/` | 큰 붙여넣기·첨부 이미지 내용 |
| `session-env/` | 세션별 환경 메타데이터 |
| `tasks/` | task 도구가 쓴 세션별 태스크 목록 |
| `shell-snapshots/` | Bash 도구용 캡처 셸 환경(정상 종료 시 제거, 크래시 잔여는 sweep) |
| `backups/` | config 마이그레이션 전 `~/.claude.json` 타임스탬프 복사본 |
| `feedback-bundles/` | `/feedback`이 만든 redacted 트랜스크립트 아카이브 |
| `todos/`, `statsig/`, `logs/` | 구버전 레거시(더 이상 안 쓰임, sweep이 비움) |

### 삭제 전까지 유지

| 경로 | 내용 |
| :-- | :-- |
| `history.jsonl` | 타이핑한 모든 프롬프트(타임스탬프·프로젝트 경로). up-arrow 복구용 |
| `stats-cache.json` | `/usage`가 보여주는 누적 토큰·비용 |
| `remote-settings.json` | 조직 server-managed settings 캐시(조직 설정 시에만, 매 launch 갱신) |

### 평문 저장 주의

트랜스크립트·히스토리는 **암호화되지 않는다**(OS 파일 권한이 유일한 보호). 도구가 `.env`를 읽거나 명령이 자격증명을 출력하면 그 값이 `projects/<project>/<session>.jsonl`에 기록된다. 노출 줄이기:

- `cleanupPeriodDays`를 낮춰 보관 기간 단축
- `CLAUDE_CODE_SKIP_PROMPT_HISTORY`로 트랜스크립트·프롬프트 히스토리 기록 스킵(비대화 모드는 `-p`와 `--no-session-persistence`, Agent SDK는 `persistSession: false`)
- 자격증명 파일 읽기를 [[05 권한]] deny 룰로 차단

### 로컬 데이터 정리 — `claude project purge`

v2.1.124 이상. 한 프로젝트의 상태 삭제(트랜스크립트·auto memory, 세션별 tasks/debug/file-history, `history.jsonl`의 매칭 프롬프트 줄, `~/.claude.json`의 프로젝트 엔트리). 삭제 전 전체 계획을 출력하고 확인을 받는다.

```bash
claude project purge ~/work/my-repo --dry-run   # 미리보기
claude project purge ~/work/my-repo             # 단일 확인
claude project purge ~/work/my-repo --yes        # 확인 스킵(스크립트용)
```

경로 생략 시 인터랙티브 목록. `--all`은 모든 프로젝트 일괄(이때 `history.jsonl` 통째 삭제), `-i`는 항목 하나씩. `shell-snapshots/`·`backups/`는 프로젝트 스코프가 아니라 건드리지 않고 경고만. 매칭 상태 없으면 status 1.

> [!warning] 삭제하지 말 것
> `~/.claude.json`, `~/.claude/settings.json`, `~/.claude/plugins/`는 auth·선호·설치 플러그인을 담으므로 삭제하지 마라.

설정·훅·파일이 안 먹으면 Debug your configuration 가이드를 보라([[25 트러블슈팅]]).
