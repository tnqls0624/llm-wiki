# 19 비용과 성능

Claude Code의 비용 추적·절감, 프롬프트 캐싱, 빠른 응답(fast mode), OpenTelemetry 기반 모니터링, 팀 단위 애널리틱스를 한 곳에 모은 레퍼런스. 허브는 [[Claude]]. 모델/effort 전환은 [[26 변경 이력과 용어집]]·설정 관점은 [[04 설정]], 권한 관점은 [[05 권한]]을 참고.

> [!info] 큰 그림
> Claude Code는 **API 토큰 소비량**으로 과금된다(구독 요금제는 별도). 비용을 줄이는 핵심은 "컨텍스트를 작게 유지"하는 것이고, 속도를 올리는 핵심은 fast mode/effort다. 둘 다 프롬프트 캐싱 동작과 얽혀 있어서 캐시 무효화 비용을 이해하는 것이 중요하다.

## 비용 관리 (costs)

Claude Code는 API 토큰 소비량으로 과금된다. 구독 요금제(Pro, Max, Team, Enterprise) 가격은 [claude.com/pricing](https://claude.com/pricing) 참고. 개발자당 비용은 모델 선택, 코드베이스 크기, 사용 패턴(여러 인스턴스 동시 실행, 자동화 등)에 따라 크게 달라진다.

엔터프라이즈 배포 기준 평균 비용은 **활성 사용일당 약 $13, 개발자당 월 $150–250** 수준이며, 90% 사용자는 활성일당 $30 미만에 머문다. 팀 도입 시 소규모 파일럿으로 베이스라인을 먼저 잡으라는 권고.

### 비용 추적 — `/usage`

`/usage` 상단의 Session 블록이 현재 세션의 상세 토큰 통계를 보여준다. 달러 수치는 로컬에서 토큰 수로 추정한 값이라 실제 청구액과 다를 수 있고, 정확한 청구는 [Claude Console의 Usage 페이지](https://platform.claude.com/usage)를 봐야 한다.

```text
Total cost:            $0.55
Total duration (API):  6m 19.7s
Total duration (wall): 6h 33m 10.2s
Total code changes:    0 lines added, 0 lines removed
```

> [!note] 구독자 주의
> `/usage`의 Session 비용 수치는 **API 사용자용**이다. Max/Pro 구독자는 사용량이 구독에 포함되므로 청구 목적으로는 무의미하며, 대신 플랜 사용량 막대·활동 통계·사용 분석을 같은 화면에서 본다.

Pro/Max/Team/Enterprise 플랜에서 `/usage`는 최근 사용량을 **skills, subagents, plugins, 개별 MCP 서버**별로 귀속해 각각 전체 대비 %로 보여준다. `d`/`w` 키로 최근 24시간 ↔ 7일 전환. 수치는 근사값이며 이 머신의 로컬 세션 히스토리에서만 계산되므로 다른 기기/claude.ai 사용량은 빠진다.

### 팀 비용 관리

| 환경 | 비용 관리 수단 |
| --- | --- |
| Claude API | workspace spend limit 설정(전체 Claude Code workspace 지출 한도), Console에서 cost/usage reporting 조회 |
| Pro / Max | `/usage-credits` 명령으로 usage credits 월 지출 한도 설정. 한도 도달 시 CLI 내에서 상향/해제 프롬프트. 변경에는 billing access 필요 |
| Bedrock / Vertex / Foundry | Claude Code가 클라우드에서 메트릭을 보내지 않음 → [LiteLLM](https://docs.litellm.ai/docs/proxy/virtual_keys#tracking-spend) 같은 오픈소스로 key별 지출 추적(비공식, 미감사) |

> [!note] "Claude Code" workspace
> Console 계정으로 최초 인증하면 "Claude Code" workspace가 자동 생성되어 조직 전체 사용량을 중앙 추적한다. 이 workspace에는 API key를 만들 수 없고 인증·사용 전용이다. 커스텀 rate limit 조직은 이 workspace의 Limits 페이지에서 [workspace rate limit](https://platform.claude.com/docs/en/api/rate-limits#setting-lower-limits-for-workspaces)을 걸어 Claude Code 트래픽이 다른 프로덕션 워크로드를 침범하지 않게 막을 수 있다.

#### Rate limit 권고치 (조직 규모별)

조직 단위로 적용되는 per-user TPM(Token Per Minute) / RPM(Request Per Minute) 권고:

| Team size | TPM per user | RPM per user |
| --- | --- | --- |
| 1-5 users | 200k-300k | 5-7 |
| 5-20 users | 100k-150k | 2.5-3.5 |
| 20-50 users | 50k-75k | 1.25-1.75 |
| 50-100 users | 25k-35k | 0.62-0.87 |
| 100-500 users | 15k-20k | 0.37-0.47 |
| 500+ users | 10k-15k | 0.25-0.35 |

예: 200명이면 사용자당 20k TPM × 200 = 총 4M TPM 요청. 규모가 클수록 동시 사용 비율이 낮아져 per-user TPM이 줄어든다. 한도는 조직 레벨이라 다른 사람이 안 쓸 때 개인이 일시적으로 더 쓸 수 있다. 라이브 트레이닝처럼 동시 사용이 몰리는 시나리오는 더 높은 할당이 필요할 수 있다.

#### Agent team 토큰 비용

[[08 서브에이전트와 에이전트 팀]]의 agent team은 여러 Claude Code 인스턴스를 띄우며, 각자 자기 컨텍스트 윈도를 가진다. 토큰 사용량은 활성 teammate 수와 실행 시간에 비례. 비용을 관리하려면:

- teammate는 **Sonnet** 사용(조정 작업에 능력/비용 균형이 좋음)
- 팀을 작게 유지(토큰은 팀 크기에 거의 비례)
- spawn 프롬프트를 집중적으로. teammate는 CLAUDE.md/MCP/skills를 자동 로드하지만 spawn 프롬프트의 모든 것이 시작부터 컨텍스트에 쌓임
- 작업 끝나면 팀 정리(idle teammate도 토큰 소비)
- agent teams는 기본 비활성. `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`로 활성화

### 토큰 사용 줄이기 (Reduce token usage)

토큰 비용은 컨텍스트 크기에 비례한다. Claude Code는 [프롬프트 캐싱](#프롬프트-캐싱-prompt-caching)과 auto-compaction으로 자동 최적화하지만, 다음 전략으로 컨텍스트를 작게 유지하는 게 핵심이다.

#### 컨텍스트를 능동적으로 관리

| 동작 | 명령/설정 | 효과 |
| --- | --- | --- |
| 작업 간 초기화 | `/clear` | 무관한 작업 전환 시 새 출발. `/rename` 후 `/clear`, 나중에 `/resume`으로 복귀 |
| 압축 지시 커스터마이즈 | `/compact Focus on code samples and API usage` | 요약 시 보존할 내용 지정 |
| 현재 사용량 확인 | `/usage` 또는 [[12 출력 스타일과 상태줄]]의 statusline | 토큰 사용 상시 표시 |

CLAUDE.md에 압축 동작을 박아둘 수도 있다:

```markdown
# Compact instructions

When you are using compact, please focus on test output and code changes
```

#### 올바른 모델 선택

Sonnet이 대부분의 코딩을 잘 처리하고 Opus보다 싸다. Opus는 복잡한 아키텍처 결정/다단계 추론에만. `/model`로 세션 중 전환, `/config`로 기본값 설정, 단순 subagent는 설정에 `model: haiku` 지정. (자세한 모델/effort는 [[26 변경 이력과 용어집]] 용어 참고)

#### MCP 서버 오버헤드 줄이기

[[10 MCP]] 도구 정의는 기본적으로 deferred라 Claude가 특정 도구를 쓰기 전까지 도구 이름만 컨텍스트에 들어온다. `/context`로 무엇이 공간을 차지하는지 확인.

- **가능하면 CLI 도구 선호**: `gh`, `aws`, `gcloud`, `sentry-cli` 등은 per-tool listing이 없어 MCP보다 컨텍스트 효율적
- **미사용 서버 비활성화**: `/mcp`로 확인 후 끄기

#### 타입 언어용 code intelligence 플러그인 설치

[[11 플러그인]]의 code intelligence 플러그인은 텍스트 검색 대신 정확한 심볼 내비게이션을 제공해 불필요한 파일 읽기를 줄인다. "go to definition" 한 번이 grep + 여러 후보 파일 읽기를 대체. 설치된 language server는 편집 후 타입 에러도 자동 보고.

#### 훅/스킬로 처리 오프로드

[[09 훅]]은 Claude가 보기 전에 데이터를 전처리할 수 있다. 1만 줄 로그를 Claude가 읽는 대신, 훅이 `ERROR`만 grep해서 매칭 줄만 반환 → 수만 토큰을 수백 토큰으로. [[07 스킬]]은 도메인 지식을 즉시 주입해 탐색 비용을 없앤다(예: "codebase-overview" 스킬).

예시 — 테스트 출력에서 실패만 보여주는 PreToolUse 훅. [[04 설정]]의 settings.json에 추가:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/filter-test-output.sh"
          }
        ]
      }
    ]
  }
}
```

`filter-test-output.sh`:

```bash
#!/bin/bash
input=$(cat)
cmd=$(echo "$input" | jq -r '.tool_input.command')

# If running tests, filter to show only failures
if [[ "$cmd" =~ ^(npm test|pytest|go test) ]]; then
  filtered_cmd="$cmd 2>&1 | grep -A 5 -E '(FAIL|ERROR|error:)' | head -100"
  echo "{\"hookSpecificOutput\":{\"hookEventName\":\"PreToolUse\",\"permissionDecision\":\"allow\",\"updatedInput\":{\"command\":\"$filtered_cmd\"}}}"
else
  echo "{}"
fi
```

| 전략 | 요점 |
| --- | --- |
| CLAUDE.md → skills 이동 | CLAUDE.md는 세션 시작 시 항상 로드되어 무관한 작업에도 토큰을 차지. 특정 워크플로 지시(PR 리뷰, DB 마이그레이션 등)는 [[07 스킬]]로 옮겨 on-demand 로드. CLAUDE.md는 **200줄 이하** 권장 ([[03 메모리와 컨텍스트]]) |
| extended thinking 조정 | thinking 토큰은 output으로 과금되고 기본 budget은 모델별로 수만 토큰. 단순 작업은 `/effort`나 `/model`로 effort level 낮추기, `/config`에서 thinking 비활성, 또는 `MAX_THINKING_TOKENS=8000` |
| 장황한 작업을 subagent에 위임 | 테스트/문서 fetch/로그 처리는 [[08 서브에이전트와 에이전트 팀]]에 위임 → 장황한 출력은 subagent 컨텍스트에 남고 요약만 메인으로 복귀 |
| agent team 비용 관리 | teammate가 plan mode로 돌면 표준 세션 대비 **약 7배** 토큰. 팀 작업을 작고 자기완결적으로 유지 |
| 구체적인 프롬프트 작성 | "improve this codebase"(광범위 스캔) 대신 "add input validation to the login function in auth.ts"(최소 파일 읽기) |

#### 복잡한 작업을 효율적으로

- **plan mode 사용**: Shift+Tab으로 [plan mode](/en/permission-modes) 진입 → Claude가 탐색·접근 제안 → 승인. 초기 방향이 틀렸을 때의 값비싼 재작업 방지
- **조기 교정**: 방향이 틀리면 Escape로 즉시 중단. `/rewind` 또는 Escape 더블탭으로 대화·코드를 이전 체크포인트로 복원
- **검증 타깃 제공**: 테스트 케이스/스크린샷/기대 출력을 프롬프트에 포함하면 Claude가 스스로 검증
- **점진적 테스트**: 한 파일 작성 → 테스트 → 계속. 싸게 일찍 잡기

### 백그라운드 토큰 사용

Claude Code는 유휴 상태에서도 일부 백그라운드 기능에 토큰을 쓴다:

- **Conversation summarization**: `claude --resume`용 이전 대화 요약 백그라운드 작업
- **Command processing**: `/usage` 같은 일부 명령의 상태 확인 요청

이런 백그라운드 프로세스는 활성 상호작용 없이도 소량(보통 **세션당 $0.04 미만**)을 소비한다.

### 동작 변경 이해

Claude Code는 비용 리포팅을 포함해 기능 동작이 바뀔 수 있다. `claude --version`으로 버전 확인. 청구 문의는 [Console 계정](https://platform.claude.com/login) 통해 Anthropic 지원으로.

## 프롬프트 캐싱 (prompt-caching)

프롬프트 캐싱은 Claude Code를 더 빠르고 비용 효율적으로 만든다. 캐싱이 없으면 API가 매 턴마다 전체 히스토리를 재처리하지만, 캐싱으로 이미 처리한 부분을 재사용하고 바뀐 부분만 새로 처리한다. Claude Code가 자동으로 관리하므로([비활성화](#캐싱-비활성화) 안 한 이상) 사용자가 직접 할 일은 없지만, **어떤 동작이 캐시를 무효화해 다음 턴을 느리고 비싸게 만드는지** 아는 것이 핵심이다.

### 캐시 구조와 prefix 매칭

매 메시지마다 새 API 요청이 나가고, 모델은 요청 간 기억이 없으므로 Claude Code는 매번 전체 컨텍스트(system prompt + 프로젝트 컨텍스트 + 모든 이전 메시지/도구 결과 + 새 메시지)를 재전송한다. 새 내용은 끝에 append되므로 각 요청의 대부분은 직전 요청과 동일하다. API는 요청의 시작 부분(**prefix**)을 최근 처리한 내용과 매칭해 캐싱한다. **매칭은 exact**라 prefix 어딘가가 바뀌면 그 뒤 전부 재계산. per-file/per-segment 캐싱은 없다.

Claude Code는 prefix 매칭을 극대화하려고 잘 안 바뀌는 내용을 앞에 두는 레이어 순서로 요청을 구성한다:

| Layer | Content | 변경 시점 |
| --- | --- | --- |
| System prompt | 핵심 지시, 도구 정의, output style | 로드된 도구 정의 집합이 바뀌거나 Claude Code 업그레이드 시 |
| Project context | CLAUDE.md, auto memory, unscoped rules | 세션 시작, `/clear` 또는 `/compact` 후 |
| Conversation | 내 메시지, Claude 응답, 도구 결과 | 매 턴 |

conversation 레이어 변경은 앞의 두 레이어 캐시를 유지하지만, system prompt 변경은 그 뒤 전부를 무효화한다. prefix 텍스트가 아니지만 **캐시 키의 일부**인 두 가지:

- **Model**: 모델마다 별도 캐시. 전환하면 내용이 같아도 전체 재계산
- **Effort level**: 같은 모델이라도 effort마다 별도 캐시. 세션 중 변경하면 전체 재계산하며 확인 다이얼로그를 띄움

> [!tip]
> 세션 시작 시 모델과 effort를 정하고, `/compact`는 작업 간 자연스러운 구분점에 아껴 쓰라. 작업 중 변경이 적을수록 cache hit rate가 높다.

[plan mode]와 [skill loading]은 지시를 conversation 메시지로 append하므로 캐시된 prefix가 그대로 유지된다.

#### 캐시가 사는 곳

캐싱은 모델을 서빙하는 인프라의 서버 측에서 일어나며, 인증 방식에 따라 위치가 다르다:

| 인증 방식 | 캐시 위치 |
| --- | --- |
| API key / Claude 구독 / Claude Platform on AWS | Anthropic 인프라(Claude API 경유) |
| Bedrock / Vertex AI | 클라우드 프로바이더 서빙 인프라 ([[17 클라우드 프로바이더]]) |
| Foundry | Anthropic 인프라로 라우팅 |
| 커스텀 `ANTHROPIC_BASE_URL` / LLM gateway | 요청이 포워딩되는 곳, 캐싱 동작 여부는 gateway에 의존 |

### 캐시를 무효화하는 동작

다음 동작은 다음 요청이 캐시를 일부/전부 미스하게 만든다. 일회성으로 느리고 비싼 턴 후 새 prefix가 캐시된다. 대부분 비용을 알면 작업 중 피할 수 있다.

| 동작 | 무효화 범위 / 메모 |
| --- | --- |
| **Switching models** (`/model`) | 전체. `opusplan`은 plan mode에서 Opus, 실행 시 Sonnet → plan 토글마다 모델 전환 = fresh 캐시 |
| **Changing effort level** (`/effort`) | 전체. 시작 후엔 확인 다이얼로그. 이미 적용 중인 레벨로 resolve되면 다이얼로그 스킵하고 캐시 유지 |
| **Turning on fast mode** (`/fast`) | 전체. uncached input을 [fast mode rate](#fast-mode-요금)로 과금 → **시작 시 켜는 게 깊은 대화 중 켜는 것보다 저렴**. 비용은 대화당 1회. v2.1.86+는 토글 간 헤더를 유지해 이후 on/off/rate-limit fallback이 캐시 유지(`/clear`·`/compact`는 리셋) |
| **MCP 서버 연결/해제** | deferred 도구면 영향 없음(append만). prefix에 로드된 도구면 전체 무효화 |
| **플러그인 활성/비활성** | skills/commands/agents/hooks/LSP/monitors/themes는 캐시 유지. **MCP 서버 제공 플러그인만** MCP 규칙 따름 |
| **전체 도구 deny** | bare 이름(`Bash`, `WebFetch`) 또는 `Bash(*)` deny는 도구를 컨텍스트에서 제거 → 무효화. scoped(`Bash(rm *)`)·allow·ask 규칙은 prefix 불변 |
| **Compaction** (`/compact`) | conversation 레이어. system prompt 재사용, project context는 CLAUDE.md/memory 불변이면 cache-hit |
| **Claude Code 업그레이드** | 보통 system prompt/도구 정의 변경 → 첫 요청 전체 재구축 |

#### Switching models / effort 세부

각 모델/effort마다 별도 캐시. effort 변경은 캐시를 무효화할 경우 확인 다이얼로그를 띄운다(같은 레벨로 resolve면 스킵).

#### MCP 서버 연결/해제 세부

도구 정의는 system prompt 레이어에 있다. tool search로 deferred(지원 모델 기본값)면 서버 연결/해제/도구목록 변경이 append만 한다. prefix에 로드되는 경우(tool search 불가/비활성: Haiku, Vertex AI, 커스텀 base URL gateway, `alwaysLoad` 표시 서버, threshold-based 로딩)는 어떤 변경도 무효화. stdio 프로세스 종료, HTTP 세션 만료, 자동 재연결, dynamic tool update 등 **사용자 행동 없이도** 일어날 수 있다. MCP config 편집 자체는 캐시를 바꾸지 않고 restart로 연결/해제될 때 적용된다.

#### 플러그인 세부

플러그인 변경은 `/reload-plugins` 실행이나 새 세션에서 적용되며 비용은 reload 후 첫 턴에 나타난다(`/plugin install`·`enable`·`disable` 시점 아님). v2.1.163부터 reload가 full re-read를 유발하면 `/reload-plugins`가 경고하며 보류; `--force`로 강제. 같은 세션에서 활성화했던 플러그인을 비활성화하면 이전 요청 형태로 복원되어, 그 prefix가 cache lifetime 내면 이전 캐시 엔트리를 읽는다.

#### Compaction 세부

요약 생성을 위해 Claude Code는 같은 system prompt/tools/history에 요약 지시를 마지막 user 메시지로 붙인 일회성 요청을 보낸다. prefix를 공유하므로 이 요청은 기존 캐시를 읽는다. compaction 시간의 대부분은 요약 생성이지 캐시 미스가 아니며, 이후 턴은 짧아진 요약만 캐시 재구축하므로 **post-compaction 턴이 느린 부분이 아니다**.

> [!tip]
> compaction은 버릴 컨텍스트가 더 이상 불필요할 때 유리. 오버헤드 시점을 고르려면 auto-compaction이 작업 중 터지기를 기다리지 말고 작업 간 구분점에서 `/compact`. 완전히 버릴 경로면 `/rewind`가 낫다(이미 캐시된 prefix로 truncate).

#### 업그레이드 세부

[Auto-update](/en/setup)는 백그라운드로 다운로드하고 다음 launch에 적용(세션 중 아님)하므로 restart 후 첫 턴이 uncached. `DISABLE_AUTOUPDATER=1`로 업그레이드 시점 제어.

> [!note] 업그레이드 후 resume
> 업그레이드 후 세션을 resume하면 히스토리가 다른 system prompt 뒤에 놓여 전체 히스토리를 cache hit 없이 재처리한다. 긴 세션으로 복귀하는 첫 턴이 가장 비싼 요청이 될 수 있다.

### 캐시를 유지하는 동작

다음은 끝에 append하거나 요청을 안 건드린다. 일부(CLAUDE.md 편집, output style 변경)는 설정 변경이 restart를 기다리는 이유이기도 하다.

| 동작 | 메모 |
| --- | --- |
| **레포 파일 편집** | 파일 내용은 Claude가 읽을 때만 컨텍스트 진입(append). 이전 read를 소급 변경하지 않고, 파일 변경 시 `<system-reminder>` append 후 필요하면 재읽기 |
| **CLAUDE.md 세션 중 편집** | project-root/user-level CLAUDE.md는 세션 시작 시 1회 읽어 메모리에 보관. 편집해도 무효화 안 되지만 **적용도 안 됨**(다음 `/clear`·`/compact`·restart에 로드). 단 하위 디렉토리 nested CLAUDE.md와 `paths:` frontmatter rules는 매칭 파일 첫 read 때 로드되므로 로드 전 편집은 적용됨 |
| **output style 변경** | system prompt의 일부. `/config`·`outputStyle`로 바꿔도 무효화 안 되고 적용도 안 됨(다음 `/clear`·restart). [[12 출력 스타일과 상태줄]] |
| **permission mode 변경** | system prompt/도구 정의 불변이라 cache-safe. 예외: `opusplan` + plan mode는 모델 전환을 유발 |
| **skills/commands 호출** | 지시를 호출 지점에서 user 메시지로 주입, 앞부분 불변 |
| **`/recap`** | 터미널 표시용 요약을 command 출력으로 append(`/compact`처럼 히스토리 교체 안 함) → prefix 유지 |
| **`/rewind`** | 이전 턴으로 truncate. 남은 히스토리가 그 시점 캐시 빌드 내용과 동일 → 이전 캐시 엔트리 hit. 파일 체크포인트 복원은 캐시에 별도 영향 없음 |
| **subagent spawn** | 아래 [Subagent와 캐시](#subagent와-캐시) 참고 |

### 캐시 수명 (Cache lifetime / TTL)

캐시된 prefix는 일정 비활동 시간 후 만료된다. cache hit하는 요청마다 타이머가 리셋되어 계속 작업하면 warm 유지. 오래 쉬면 다음 요청이 전체 재계산하므로 자리 비운 후 첫 턴이 눈에 띄게 느릴 수 있다.

API는 두 TTL을 제공: **5분**, **1시간**(긴 휴식에도 warm 유지하지만 cache write를 더 비싸게 과금). Claude Code가 인증 방식에 따라 골라준다:

| 환경 | 기본 TTL | 메모 |
| --- | --- | --- |
| Claude 구독 | **1시간** 자동 요청 | 플랜 포함이라 추가 비용 없음. 단 플랜 한도 초과로 usage credits를 쓰면 과금되므로 자동으로 5분으로 낮춤 |
| API key / Bedrock / Vertex / Foundry / Claude Platform on AWS | **5분** | per-token 과금이므로 저렴한 5분 유지. `ENABLE_PROMPT_CACHING_1H=1`로 1시간 opt-in |

Bedrock은 모델별로 캐싱 지원/최소 cacheable prefix 길이/1시간 TTL 가용성이 다르다. cache token이 0이면 [Bedrock 문서](https://docs.aws.amazon.com/bedrock/latest/userguide/prompt-caching.html#prompt-caching-models)의 지원 모델/리전 확인.

**TTL 강제 override**: `FORCE_PROMPT_CACHING_5M=1`로 인증과 무관하게 5분 강제(캐시 디버깅, 두 TTL 비교, managed settings의 `ENABLE_PROMPT_CACHING_1H` override 시 유용).

### 캐시 범위 (Cache scope)

Claude Code에서 캐시는 사실상 **1머신 + 1디렉토리** 단위. system prompt가 working directory, platform, shell, OS version, auto-memory paths를 박아넣으므로 다른 디렉토리의 두 세션은 다른 prefix를 만들어 서로의 캐시를 미스한다(같은 레포의 worktree 포함, 각 worktree가 자체 working directory).

- 같은 디렉토리에서 병렬 실행하는 세션들은 prefix가 일치해 서로의 캐시를 읽음
- 순차 세션은 시작 시점 git status 스냅샷(branch + recent commits)이 일치할 때만 prefix 공유

기저 API 캐시는 더 넓다. 조직 간 격리, 일부 프로바이더는 workspace 간 격리. 그 경계 내에서 같은 model + prefix면 같은 캐시를 읽는다. Agent SDK로 fleet을 돌리면 system prompt의 per-machine 섹션을 억제해 머신 간 캐시를 공유할 수 있다([[24 Agent SDK — 고급과 레퍼런스]] 참고).

### 캐시 성능 확인

API가 매 응답에 보고하는 두 토큰 카운트로 캐시 성능을 본다. 가장 직접적인 방법은 `current_usage` 객체를 읽는 [[12 출력 스타일과 상태줄]] statusline 스크립트다:

| Field | 의미 |
| --- | --- |
| `cache_creation_input_tokens` | 이번 턴에 캐시에 쓴 토큰, cache write rate로 과금 |
| `cache_read_input_tokens` | 이번 턴에 캐시에서 서빙된 토큰, 표준 input rate의 **약 10%**로 과금 |

read-to-creation 비율이 높을수록 캐싱이 잘 작동. creation이 턴마다 계속 높으면 prefix에 뭔가 바뀌고 있는 것(위 무효화 동작 참고). 조직 전체 가시성은 OpenTelemetry exporter가 per-user/session으로 cache read/creation을 보고([모니터링](#모니터링-monitoring-usage) 참고).

### Subagent와 캐시

[[08 서브에이전트와 에이전트 팀]]의 subagent는 부모와 별개의 자체 대화(자체 system prompt + tool set)를 시작한다. 자체 캐시를 빌드하므로 첫 호출은 cache hit 0이고 자기 턴들로 warm up. 구독에서도 subagent는 **5분 TTL** 사용(1시간 TTL은 메인 대화에만). 부모 캐시는 영향 없음(subagent 호출/결과가 부모 대화에 append, prefix 불변).

반면 **fork**는 부모의 system prompt/tools/history를 그대로 상속하므로 첫 요청이 부모 캐시를 읽는다. compaction 요약 호출도 같은 prefix-공유 방식.

### 캐싱 비활성화

특정 모델/프로바이더에서 캐싱 디버깅 시에만 유용. 다음 환경변수를 `1`로:

| Variable | Effect |
| --- | --- |
| `DISABLE_PROMPT_CACHING` | 전체 모델 비활성 |
| `DISABLE_PROMPT_CACHING_HAIKU` | Haiku만 |
| `DISABLE_PROMPT_CACHING_SONNET` | Sonnet만 |
| `DISABLE_PROMPT_CACHING_OPUS` | Opus만 |

조직 정책으로 적용하려면 이들 또는 TTL 변수를 [[04 설정]] managed settings의 `env` 블록에. 평상시엔 캐싱 켠 채로 두는 것이 정답.

## 빠른 응답 모드 (fast-mode)

> [!note] Research preview
> fast mode는 research preview 기능이다. 기능·가격·가용성·기저 API 구성이 피드백에 따라 바뀔 수 있다.

fast mode는 Claude Opus의 고속 구성으로, 모델을 **최대 2.5배 빠르게** 만들지만 토큰당 비용이 높다. 모델이 바뀌는 게 아니라 **속도를 비용 효율보다 우선하는 다른 API 구성**을 쓰는 것이며, 품질·능력은 동일하다. Opus 4.8/4.7/4.6에서 지원되고 Sonnet/Haiku/기타 모델에는 없다.

> [!warning] Opus 4.6 fast mode deprecated
> Opus 4.6의 fast mode는 deprecated이며 Opus 4.8 출시 약 30일 후 제거 예정. 제거 후엔 표준 속도·표준 가격으로 fallback. 속도를 유지하려면 Opus 4.8/4.7로 마이그레이션.

요약:

- `/fast`로 CLI에서 토글. **VS Code 확장에서는 미지원** ([[14 IDE와 데스크톱]])
- fast mode 요금(MTok input/output): Opus 4.8 = $10/$50, Opus 4.7·4.6 = $30/$150
- 구독 플랜(Pro/Max/Team/Enterprise) + Claude Console에 제공
- 구독 플랜에서는 **usage credits로만** 제공되며 구독 rate limit에 포함되지 않음
- Claude Code v2.1.36 이상 필요(`claude --version`)

### Fast mode 토글

| 방법 | 설명 |
| --- | --- |
| `/fast` + Tab | on/off 토글 |
| `"fastMode": true` | user settings file에 설정 |

기본적으로 세션 간 유지된다(관리자가 per-session opt-in으로 변경 가능). 비용 효율을 위해 **세션 중 전환보다 시작 시 활성화** 권장.

활성화 시:
- 다른 모델이면 자동으로 Opus로 전환
- "Fast mode ON" 확인 메시지
- 프롬프트 옆에 작은 `↯` 아이콘
- `/fast` 재실행으로 현재 상태 확인

`/fast`로 비활성화해도 **Opus에 머문다**(이전 모델로 안 돌아감). 다른 모델로 가려면 `/model`. Opus 4.8은 v2.1.154+에서 fast mode 기본값(v2.1.142–2.1.153은 Opus 4.7).

### Fast mode 요금 (비용 트레이드오프)

| Model | Input (MTok) | Output (MTok) |
| --- | --- | --- |
| Opus 4.8 | $10 | $50 |
| Opus 4.7 / Opus 4.6 | $30 | $150 |

fast mode 가격은 전체 1M 토큰 컨텍스트에 걸쳐 flat(표준 Opus와 비교하려면 [Claude pricing](https://platform.claude.com/docs/en/about-claude/pricing)).

> [!important] 대화당 1회 비용
> 대화에서 fast mode를 **처음 켜는 순간 전체 대화 컨텍스트에 대해 full fast-mode uncached input 가격**을 낸다. 대화가 깊을수록 비싸므로 **처음부터 켜는 게 저렴**하다. 이 비용은 대화당 1회라 껐다 다시 켜도 반복되지 않는다. 메커니즘은 [Turning on fast mode](#캐시를-무효화하는-동작) 참고.

### 언제 쓸까

| fast mode가 좋은 경우 | standard mode가 좋은 경우 |
| --- | --- |
| 코드 변경 빠른 반복 | 속도 덜 중요한 긴 자율 작업 |
| 라이브 디버깅 | 배치 처리 / CI-CD 파이프라인 ([[16 CI-CD와 팀 통합]]) |
| 마감 빠듯한 시간 민감 작업 | 비용 민감 워크로드 |

#### Fast mode vs effort level

| Setting | Effect |
| --- | --- |
| **Fast mode** | 동일 모델 품질, 낮은 latency, 높은 비용 |
| **Lower effort level** | thinking 시간 감소, 빠른 응답, 복잡 작업에서 품질 저하 가능 |

둘을 조합 가능: 간단한 작업에 fast mode + 낮은 effort level = 최대 속도.

### 요구사항

- **Anthropic API 또는 구독만**: Console API + 구독 플랜(usage credits). Bedrock/Vertex AI/Azure Foundry/Claude Platform on AWS에는 **없음** ([[17 클라우드 프로바이더]])
- **usage credits 켜짐**: 플랜 포함 사용량을 넘어 과금 허용. 개인은 [Console billing settings](https://platform.claude.com/settings/organization/billing), Team/Enterprise는 admin이 조직에 켜야 함
- **Team/Enterprise는 admin enablement 필요**: 기본 비활성

> [!note]
> fast mode 사용은 플랜 잔여량이 있어도 **usage credits에서 직접 차감**된다. 즉 fast mode 토큰은 플랜 포함 사용량에 안 잡히고 첫 토큰부터 fast mode rate로 과금. admin이 조직에 활성화하지 않았으면 `/fast`는 "Fast mode has been disabled by your organization." 표시.

#### 조직에 fast mode 활성화

| 환경 | 위치 |
| --- | --- |
| Console (API 고객) | [Claude Code preferences](https://platform.claude.com/claude-code/preferences) |
| Claude AI (Team/Enterprise) | [Admin Settings > Claude Code](https://claude.ai/admin-settings/claude-code) |

완전 비활성화: `CLAUDE_CODE_DISABLE_FAST_MODE=1` ([[18 보안과 샌드박스]]·env vars 참고).

#### Per-session opt-in 강제

기본은 세션 간 유지지만, Team/Enterprise admin은 managed settings 또는 server-managed settings에 `fastModePerSessionOptIn: true`로 매 세션 off로 시작하게 강제할 수 있다(사용자가 동시 세션을 여러 개 돌리는 조직의 비용 관리에 유용).

```json
{
  "fastModePerSessionOptIn": true
}
```

사용자는 여전히 `/fast`로 켤 수 있지만 새 세션 시작 시 리셋된다. 사용자 선호는 저장되므로 이 설정을 제거하면 기존 persistent 동작 복원.

### Rate limit 처리

fast mode는 표준 Opus와 별도 rate limit을 가지며, Opus 4.8/4.7/4.6의 fast mode는 같은 limit 풀을 공유한다. limit 도달 또는 usage credits 소진 시:

1. fast mode가 자동으로 표준 속도로 fallback
2. `↯` 아이콘이 회색(cooldown 표시)
3. 표준 속도·가격으로 계속 작업
4. cooldown 만료 시 fast mode 자동 재활성

cooldown을 기다리는 대신 수동 비활성화는 `/fast` 재실행.

## 모니터링 (monitoring-usage)

OpenTelemetry(OTel)로 텔레메트리를 export해 조직 전체의 Claude Code 사용량·비용·도구 활동을 추적한다. 메트릭은 표준 metrics 프로토콜(시계열), 이벤트는 logs/events 프로토콜, 선택적으로 distributed traces 프로토콜로 export.

### Quick start

```bash
# 1. Enable telemetry
export CLAUDE_CODE_ENABLE_TELEMETRY=1

# 2. Choose exporters (both optional)
export OTEL_METRICS_EXPORTER=otlp       # otlp, prometheus, console, none
export OTEL_LOGS_EXPORTER=otlp          # otlp, console, none

# 3. Configure OTLP endpoint
export OTEL_EXPORTER_OTLP_PROTOCOL=grpc
export OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4317

# 4. Authentication (if required)
export OTEL_EXPORTER_OTLP_HEADERS="Authorization=Bearer your-token"

# 5. Debugging: reduce export intervals
export OTEL_METRIC_EXPORT_INTERVAL=10000  # 10s (default 60000ms)
export OTEL_LOGS_EXPORT_INTERVAL=5000     # 5s (default 5000ms)

# 6. Run
claude
```

기본 export 간격은 메트릭 60초, 로그 5초. 디버깅 시 짧게 했다가 프로덕션엔 복원.

### 관리자 설정

[[04 설정]]의 managed settings file로 조직 전체 OTel 설정을 중앙 제어(MDM 배포 가능). managed settings의 env 변수는 우선순위가 높아 사용자가 override 불가.

```json
{
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_METRICS_EXPORTER": "otlp",
    "OTEL_LOGS_EXPORTER": "otlp",
    "OTEL_EXPORTER_OTLP_PROTOCOL": "grpc",
    "OTEL_EXPORTER_OTLP_ENDPOINT": "http://collector.example.com:4317",
    "OTEL_EXPORTER_OTLP_HEADERS": "Authorization=Bearer example-token"
  }
}
```

> [!note] 서브프로세스 미전파
> Claude Code는 `OTEL_*` 환경변수를 자신이 spawn하는 서브프로세스(Bash 도구, hooks, MCP 서버, language server)에 전달하지 않는다. Bash로 실행하는 OTel 계측 앱이 자체 텔레메트리를 export하려면 그 명령에 직접 변수를 설정해야 한다.

### Traces (beta)

distributed tracing은 각 user prompt를 그것이 유발한 API 요청·도구 실행과 연결하는 span을 export해, 전체 요청을 하나의 trace로 본다. 기본 off. 활성화: `CLAUDE_CODE_ENABLE_TELEMETRY=1` + `CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1` + `OTEL_TRACES_EXPORTER` 설정.

| Environment Variable | 설명 | Example |
| --- | --- | --- |
| `CLAUDE_CODE_ENHANCED_TELEMETRY_BETA` | span tracing 활성(필수, `ENABLE_ENHANCED_TELEMETRY_BETA`도 허용) | `1` |
| `OTEL_TRACES_EXPORTER` | traces exporter(쉼표 구분) | `console`, `otlp`, `none` |
| `OTEL_EXPORTER_OTLP_TRACES_PROTOCOL` / `_ENDPOINT` | traces 전용 override | `grpc` / `http://localhost:4318/v1/traces` |
| `OTEL_TRACES_EXPORT_INTERVAL` | span 배치 export 간격 ms(기본 5000) | `1000` |

span은 기본적으로 user prompt 텍스트/도구 입력/도구 내용을 redact(`OTEL_LOG_USER_PROMPTS=1`, `OTEL_LOG_TOOL_DETAILS=1`, `OTEL_LOG_TOOL_CONTENT=1`로 포함). tracing 활성 시 Bash/PowerShell 서브프로세스는 W3C trace context를 담은 `TRACEPARENT` 환경변수를 자동 상속해 end-to-end 분산 추적이 가능. Anthropic API 직결 시 각 모델 요청에 W3C `traceparent` 헤더가 붙고 API의 `traceresponse`가 span link로 기록된다. 기본적으로 `ANTHROPIC_BASE_URL`이 미설정/Anthropic API일 때만 헤더 전송(일부 proxy가 거부) — 커스텀 proxy에 전파하려면 `CLAUDE_CODE_PROPAGATE_TRACEPARENT=1`. Agent SDK·`-p` 세션은 자신의 환경에서 `TRACEPARENT`/`TRACESTATE`를 읽어 호출자 trace의 자식으로 표시(인터랙티브 세션은 무시).

#### Span 계층

```text
claude_code.interaction
├── claude_code.llm_request
├── claude_code.hook                    (requires detailed beta tracing)
└── claude_code.tool
    ├── claude_code.tool.blocked_on_user
    ├── claude_code.tool.execution
    └── (Agent tool) subagent claude_code.llm_request / claude_code.tool spans
```

각 user prompt가 `claude_code.interaction` root span을 시작하고 API/tool/hook 실행이 자식으로 기록된다. tool span은 permission 대기(`blocked_on_user`)와 실제 실행(`execution`) 두 자식을 가진다. 모든 span은 `span.type` 속성을 가지며, `llm_request`/`tool.execution`/`hook` span은 실패 시 OTel status `ERROR`, 나머지는 `UNSET`. 주요 span 속성: `llm_request`(model, gen_ai.*, query_source, agent_id, speed=fast/normal, ttft_ms, input/output/cache_read/cache_creation_tokens, request_id, attempt, stop_reason 등), `tool`(tool_name, duration_ms, result_tokens, tool_use_id, `OTEL_LOG_TOOL_DETAILS` 게이트로 file_path/full_command/skill_name/subagent_type). `claude_code.hook` span은 detailed beta tracing(`ENABLE_BETA_TRACING_DETAILED=1` + `BETA_TRACING_ENDPOINT`, 인터랙티브 CLI는 조직 allowlist 필요)에서만 방출된다.

### Dynamic headers

엔터프라이즈 동적 인증용. `http/protobuf`·`http/json` 프로토콜에만 적용(`grpc`는 static `OTEL_EXPORTER_OTLP_HEADERS`만). `.claude/settings.json`:

```json
{
  "otelHeadersHelper": "/bin/generate_opentelemetry_headers.sh"
}
```

스크립트는 string key-value HTTP 헤더의 유효한 JSON을 출력해야 한다:

```bash
#!/bin/bash
echo "{\"Authorization\": \"Bearer $(get-token.sh)\", \"X-API-Key\": \"$(get-api-key.sh)\"}"
```

helper 실패 시 `/doctor`, `--debug`/`/debug` 디버그 로그, `-p` 세션 stderr에 에러 보고. 스크립트는 시작 시 + 주기적(기본 29분, `CLAUDE_CODE_OTEL_HEADERS_HELPER_DEBOUNCE_MS`로 조정) 실행되어 토큰 갱신.

### 멀티팀 조직 지원

`OTEL_RESOURCE_ATTRIBUTES`로 그룹 구분 커스텀 속성 추가:

```bash
export OTEL_RESOURCE_ATTRIBUTES="department=engineering,team.id=platform,cost_center=eng-123"
```

이 속성은 모든 메트릭·이벤트에 포함되어 팀/부서 필터링, cost center별 추적, 팀별 대시보드, 알림에 쓰인다. 커스텀 키는 standard attributes(`user.id`, `session.id` 등)를 override하지 않는다. 각 커스텀 키는 모든 메트릭 series의 라벨이 되어 high-cardinality 값은 저장 비용을 늘리므로, resource block에만 담고 datapoint 라벨에서 빼려면 `OTEL_METRICS_INCLUDE_RESOURCE_ATTRIBUTES=false`.

> **v2.1.161 변경**: `OTEL_RESOURCE_ATTRIBUTES` 값이 metric datapoint의 label로 직접 포함된다. 이제 팀·레포·cost center 등 커스텀 차원으로 Prometheus/ClickHouse 등 백엔드에서 metric을 슬라이싱할 수 있다. 이전 버전은 resource 속성이 metric series에 붙지 않아 이벤트 로그에서만 조인 가능했다.

> [!warning] OTEL_RESOURCE_ATTRIBUTES 형식 주의
> 쉼표 구분 key=value, **공백 불가**, US-ASCII만(제어문자·공백·쌍따옴표·쉼표·세미콜론·백슬래시 제외), 특수문자는 percent-encode. 따옴표로 감싸도 공백이 escape되지 않고 따옴표가 리터럴로 포함된다. 예: `org.name=Johns_Organization` (O), `org.name="My Company"` (X, `"My Company"` 리터럴).

### 사용 가능한 메트릭

모든 메트릭·이벤트는 standard attributes를 공유한다: `session.id`, `app.version`, `app.entrypoint`(`cli`/`sdk-cli`/`sdk-ts`/`sdk-py`/`claude-vscode`), `organization.id`, `user.account_uuid`, `user.account_id`, `user.id`(설치별 익명 ID), `user.email`(OAuth 시), `terminal.type`, 그리고 `OTEL_RESOURCE_ATTRIBUTES` 키. 이벤트는 추가로 `prompt.id`(prompt 상관), `workspace.host_paths`를 가진다(메트릭엔 cardinality 때문에 미포함).

| Metric Name | 설명 | Unit |
| --- | --- | --- |
| `claude_code.session.count` | 시작된 CLI 세션 수 (`start_type`: fresh/resume/continue) | count |
| `claude_code.lines_of_code.count` | 수정된 코드 줄 수 (`type`: added/removed) | count |
| `claude_code.pull_request.count` | 생성된 PR 수 | count |
| `claude_code.commit.count` | 생성된 git commit 수 | count |
| `claude_code.cost.usage` | 세션 비용 | USD |
| `claude_code.token.usage` | 사용 토큰 수 (`type`: input/output/cacheRead/cacheCreation) | tokens |
| `claude_code.code_edit_tool.decision` | code 편집 도구 permission 결정 수 | count |
| `claude_code.active_time.total` | 총 활성 시간(idle 제외, `type`: user/cli) | s |

cost/token counter는 매 API 요청 후 증가하며 `model`, `query_source`(main/subagent/auxiliary), `speed`(fast), `effort`(low/medium/high/xhigh/max), 그리고 skill/plugin/agent/MCP 귀속 속성(`agent.name`, `skill.name`, `plugin.name`, `marketplace.name`, `mcp_server.name`, `mcp_tool.name`)을 가진다. user-defined/third-party 이름은 `custom`/`third-party`로 redact, official-marketplace·built-in은 그대로. code_edit_tool.decision은 `tool_name`(Edit/Write/NotebookEdit), `decision`(accept/reject), `source`, `language`를 가진다.

### 이벤트

`OTEL_LOGS_EXPORTER` 설정 시 logs/events로 export. 모든 이벤트는 standard attributes + `event.name`/`event.timestamp`(ISO 8601)/`event.sequence`를 공유. `prompt.id`(UUID v4)는 한 프롬프트 처리 중 발생한 모든 이벤트를 묶어준다(메트릭엔 미포함). 주요 이벤트:

| Event Name | 트리거 / 핵심 속성 |
| --- | --- |
| `user_prompt` | prompt 제출. prompt_length, prompt(redacted), command_name, command_source |
| `tool_result` | 도구 완료(거부 시 미방출). tool_name, tool_use_id, success, duration_ms, error_type, decision_source, size, `OTEL_LOG_TOOL_DETAILS` 시 tool_parameters/tool_input |
| `api_request` | 각 API 요청. model, cost_usd, duration_ms, tokens, request_id, speed, query_source, effort + 귀속 속성 |
| `api_error` | API 실패. error, status_code, attempt(>`CLAUDE_CODE_MAX_RETRIES` 기본 10이면 retry 소진), request_id |
| `api_refusal` | `stop_reason: refusal`(error 이벤트 안 뜸) |
| `api_request_body` / `api_response_body` | `OTEL_LOG_RAW_API_BODIES` 시. body(60KB) 또는 body_ref, body_length, body_truncated |
| `tool_decision` | permission 결정. decision(accept/reject), source(config/hook/user_permanent/user_temporary/user_abort/user_reject), tool_parameters |
| `permission_mode_changed` | from_mode, to_mode, trigger(shift_tab/exit_plan_mode/auto_gate_denied/auto_opt_in) |
| `auth` | `/login`·`/logout`. action, success, auth_method, error_category |
| `mcp_server_connection` | connect/disconnect/fail. status, transport_type, server_scope, is_plugin, plugin_id_hash |
| `internal_error` | error_name, error_code(Bedrock/Vertex/Foundry, `DISABLE_ERROR_REPORTING`엔 미방출) |
| `plugin_installed` / `plugin_loaded` | 설치/세션 시작 시 enabled 플러그인. plugin.name, marketplace.name, scope, enabled_via, has_hooks/has_mcp 등 |
| `skill_activated` | skill.name, invocation_trigger(user-slash/claude-proactive/nested-skill), skill.source, skill.kind |
| `at_mention` | `@`-mention 해석. mention_type(file/directory/agent/mcp_resource), success |
| `api_retries_exhausted` | retry 소진. total_attempts, total_retry_duration_ms |
| `hook_registered` / `hook_execution_start` / `hook_execution_complete` | hook 인벤토리/실행. hook_event, hook_name, num_hooks, num_success/blocking/cancelled |
| `hook_plugin_metrics` | official-marketplace 플러그인 hook의 per-invocation 메트릭(최대 20키) |
| `compaction` | trigger(auto/manual), success, pre_tokens/post_tokens, precompute_reuse(hit/miss_*) |
| `feedback_survey` | 세션 품질 설문. event_type, survey_type, response |

`tool_use_id`는 hook에 전달되는 값과 매칭되어 OTel 이벤트와 hook 캡처 데이터를 상관시킬 수 있다([[09 훅]]).

### 데이터 해석 / 분석

- **Usage monitoring**: `token.usage`를 type/user/team/model/skill.name/plugin.name/agent.name로 분해, `session.count`로 채택·참여 추이, `lines_of_code.count`로 생산성, `commit.count`/`pull_request.count`로 워크플로 영향
- **Cost monitoring**: `cost.usage`로 팀/개인 추이, 고사용 세션 식별, skill/plugin/subagent별 지출 귀속(근사값 — 공식 청구는 API 프로바이더)
- **Alerting**: cost spike, 비정상 토큰 소비, 특정 사용자 고세션 볼륨. 모든 메트릭은 user/org/session/model/version으로 segment 가능
- **Retry 소진 감지**: `api_error`의 `attempt`가 `CLAUDE_CODE_MAX_RETRIES`(기본 10)보다 크면 transient 에러로 retry 소진. session.id로 그룹핑해 이후 `api_request`가 있는지로 복구 여부 판별

### 보안 이벤트 감사

OTel 이벤트가 Claude Code 활동의 audit 데이터 소스다. 모든 이벤트가 identity 속성(`user.email`, `user.account_uuid`, `user.account_id`, `organization.id`, `user.id`, `session.id`)을 담아 도구 호출·MCP 활동·permission 결정을 트리거한 사용자에게 귀속시킨다. OTLP logs exporter로 SIEM이나 OTel Collector에 전달. API key/Bedrock/Vertex/Foundry 인증 시엔 Claude 계정이 없어 `user.id`·`session.id`만 채워지므로 `OTEL_RESOURCE_ATTRIBUTES`로 직접 identity를 붙인다:

```bash
export OTEL_RESOURCE_ATTRIBUTES="enduser.id=jdoe@example.com,enduser.directory_id=S-1-5-21-..."
```

MCP 활동 전체 디테일은 logs exporter + `OTEL_LOG_TOOL_DETAILS=1`로 캡처(server_name, tool_name, 인자가 `tool_result`/`tool_decision`/`mcp_server_connection`에 포함). 보안 질문 → 이벤트 매핑 예: 도구 허용/거부 → `tool_decision`(decision, source), permission 상승 → `permission_mode_changed`, 정책 hook 차단 → `hook_execution_complete`(num_blocking), 로그인 실패 → `auth`. Claude Code는 raw 이벤트 스트림만 방출하고 anomaly detection·correlation·alerting은 SIEM/observability backend 책임.

SIEM으로 보내기(이벤트만, MCP/Bash 감사용 tool detail 켬):

```json
{
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_LOGS_EXPORTER": "otlp",
    "OTEL_LOG_TOOL_DETAILS": "1",
    "OTEL_EXPORTER_OTLP_LOGS_PROTOCOL": "http/protobuf",
    "OTEL_EXPORTER_OTLP_LOGS_ENDPOINT": "https://siem.example.com:4318/v1/logs",
    "OTEL_EXPORTER_OTLP_HEADERS": "Authorization=Bearer your-siem-token"
  }
}
```

### Backend 선택 / 서비스 정보

| 신호 | 추천 backend |
| --- | --- |
| Metrics | 시계열 DB(Prometheus), 컬럼나(ClickHouse), observability 플랫폼(Honeycomb/Datadog) |
| Events/logs | 로그 집계(Elasticsearch/Loki), 컬럼나(ClickHouse), observability 플랫폼 |
| Traces | 분산 추적(Jaeger/Zipkin/Grafana Tempo), observability 플랫폼 |

DAU/WAU/MAU가 필요하면 효율적 unique 쿼리를 지원하는 backend 선택. 모든 export에 붙는 resource 속성: `service.name=claude-code`, `service.version`, `os.type`, `os.version`, `host.arch`, (WSL 시) `wsl.version`, Meter Name `com.anthropic.claude_code`.

### 보안/프라이버시 요점

- OTel export는 opt-in. Anthropic의 별도 운영 텔레메트리/비활성화는 [[18 보안과 샌드박스]]·data usage 참고
- raw 파일 내용·코드 스니펫은 메트릭/이벤트에 미포함(trace span은 별도 경로 — `OTEL_LOG_TOOL_CONTENT`)
- OAuth 인증 시 `user.email` 포함 → 우려되면 backend에서 필터/redact
- user prompt 내용 기본 미수집(길이만), 포함은 `OTEL_LOG_USER_PROMPTS=1`
- tool input 인자 기본 미로깅, 포함은 `OTEL_LOG_TOOL_DETAILS=1`(설정한 OTEL endpoint로만, Anthropic으로는 절대 안 감)
- raw API body 기본 미로깅, `OTEL_LOG_RAW_API_BODIES`로 포함(전체 히스토리 포함, extended-thinking은 항상 redact)

ROI 측정 가이드: [Claude Code ROI Measurement Guide](https://github.com/anthropics/claude-code-monitoring-guide)(Docker Compose/Prometheus/OTel 설정, Linear 통합 리포트 템플릿). Bedrock 모니터링: [Bedrock Monitoring](https://github.com/aws-solutions-library-samples/guidance-for-claude-code-with-amazon-bedrock/blob/main/assets/docs/MONITORING.md).

| Plan | Dashboard URL | 포함 |
| --- | --- | --- |
| Claude for Teams / Enterprise | [claude.ai/analytics/claude-code](https://claude.ai/analytics/claude-code) | usage 메트릭, GitHub 통합 contribution 메트릭, leaderboard, 데이터 export |

> per-user 토큰 수·비용 추정은 [모니터링](#모니터링-monitoring-usage)의 OpenTelemetry export로 구성.

### Team / Enterprise 애널리틱스

Admin·Owner가 대시보드를 본다. 포함: **Usage metrics**(수락된 코드 줄, suggestion accept rate, DAU·세션), **Contribution metrics**(GitHub 통합, CC로 작성된 PR·코드 줄), **Leaderboard**(상위 기여자), **Data export**(contribution CSV).

#### Contribution metrics 활성화

> public beta, Teams/Enterprise 플랜. claude.ai 조직 내 사용자만 커버(Console API·third-party 통합 제외). Owner 역할로 설정, GitHub admin이 앱 설치. **Zero Data Retention 조직은 불가**(usage 메트릭만 표시).

1. GitHub admin이 [github.com/apps/claude](https://github.com/apps/claude)에서 Claude GitHub 앱 설치
2. Claude Owner가 [claude.ai/admin-settings/claude-code](https://claude.ai/admin-settings/claude-code)에서 analytics 활성화
3. 같은 페이지에서 "GitHub analytics" 토글 켜기
4. GitHub 인증 완료 후 분석 대상 조직 선택

데이터는 활성화 후 보통 24시간 내 나타나고 매일 업데이트. 안 나타나면 "GitHub app required" 또는 "Data processing in progress" 메시지. GitHub Cloud + GitHub Enterprise Server 지원.

#### Summary 메트릭

> 의도적으로 보수적 — Claude Code 실제 임팩트의 **과소추정**. 높은 신뢰도 줄/PR만 카운트.

- **PRs with CC**: CC로 작성된 줄을 1줄 이상 포함한 merged PR 수
- **Lines of code with CC**: merged PR 중 CC 작성 줄. "effective lines"만(정규화 후 3자 초과, 빈 줄·괄호·사소한 구두점 제외)
- **PRs with Claude Code (%)**: CC 코드 포함 merged PR 비율
- **Suggestion accept rate**: Edit/Write/NotebookEdit 도구 suggestion 수락 비율
- **Lines of code accepted**: 사용자가 세션에서 수락한 CC 작성 줄(거부분 제외, 후속 삭제 미추적)

#### 차트

- **Adoption**: 일별 users(DAU) / sessions
- **PRs per user**: 일별 merged PR ÷ DAU, users
- **Pull requests**: 일별 PRs with CC vs without CC (Lines of code 뷰 토글 가능)
- **Leaderboard**: 상위 10명. Pull requests/Lines of code 토글, "Export all users"로 전체 CSV

#### PR attribution

contribution 메트릭 활성 시 merged PR을 분석해 어느 코드가 CC로 작성됐는지 판정. CC 세션 활동을 PR 코드와 매칭(보수적, 높은 신뢰도만).

- **Tagging**: CC 세션 중 작성된 줄을 1줄 이상 포함하면 "with Claude Code"
- **Process**: ① PR diff에서 added line 추출 ② 시간창 내 매칭 파일 편집 세션 식별 ③ multiple strategies로 매칭 ④ AI-assisted/total 줄 계산. 비교 전 정규화(공백 trim, 다중 공백 collapse, 따옴표 표준화, 소문자화). CC 줄 포함 merged PR은 GitHub에서 `claude-code-assisted` 라벨
- **Time window**: PR merge 날짜 기준 **21일 전 ~ 2일 후** 세션 고려
- **Excluded files**: lock 파일(package-lock.json, yarn.lock, Cargo.lock), 생성 코드(Protobuf 출력, build artifact, minified), build 디렉토리(dist/, build/, node_modules/, target/), 테스트 fixture(snapshot, cassette, mock), 1,000자 초과 줄
- **Attribution notes**: 개발자가 20% 초과 재작성한 코드는 CC에 미귀속, 21일 창 밖 세션 미고려, source/destination 브랜치 미고려

#### 애널리틱스 활용

ROI 입증, 채택 패턴 식별, 온보딩 도울 power user 찾기. Adoption 차트로 best practice 공유자·전체 추이·friction dip 모니터링. ROI는 PRs per user 변화·CC with/without 비교를 [DORA metrics](https://dora.dev/)·sprint velocity와 함께. Leaderboard로 prompting 기법 공유자 식별. GitHub에서 `claude-code-assisted` 라벨 검색으로 프로그래밍적 접근.

### API 고객 애널리틱스

Console 고객은 [platform.claude.com/claude-code](https://platform.claude.com/claude-code)에서 접근. **UsageView 권한** 필요(Developer/Billing/Admin/Owner/Primary Owner 역할에 부여). GitHub contribution 메트릭은 현재 미제공(usage·spend만).

대시보드: **Lines of code accepted**, **Suggestion accept rate**(Edit/Write/NotebookEdit), **Activity**(DAU·세션 차트), **Spend**(일별 API 비용 + user 수). **Team insights** 테이블 per-user 메트릭: Members(API key는 key 식별자, OAuth는 email), Spend this month, Lines this month. Console spend는 애널리틱스용 추정값 — 실제 비용은 billing 페이지.

## 비용 최적화 실전 — 계획 우선 에이전트

인터랙티브 모드의 비용이 높은 근본 원인은 **반복 오버헤드(iteration overhead)**다. Claude Code가 탐색·수정·재탐색을 반복하는 동안 매 턴마다 전체 컨텍스트를 재전송하며 토큰이 쌓인다. 동일한 작업을 계획 우선 에이전트(planning-first agent)로 처리하면 이 오버헤드를 크게 줄일 수 있다.

### 사례 비교 — 동일 작업, 62% 비용 절감

아래는 동일한 최적화 작업(CPU 추론 파이프라인 구축)을 두 가지 방식으로 실행한 실측 비교다:

| 방식 | 비용 | 방법 |
| --- | --- | --- |
| Claude Code 인터랙티브 단독 | $1.96 | 직접 코드 작성·수정 반복 |
| Claude Code + Neo(계획 우선 에이전트) | $0.74 | Neo가 먼저 조사 → 선형 실행 |

비용 차이(약 62%)는 능력 차이가 아니라 **컨텍스트 소비 패턴 차이**에서 온다.

### 계획 우선 에이전트의 동작 원리

Neo(MCP를 통해 Claude Code 내부에서 로컬로 실행되는 에이전트)의 접근:

1. **먼저 질문**: 실행 전 명확화가 필요한 부분을 한 번에 질문
2. **집중 리서치**: 모델 카드·CPU 벤치마크·ONNX vs PyTorch 처리량·빌드 요구사항 등 관련 문서를 2분간 집중 읽기
3. **최적 경로 선택**: 조사 결과 기반으로 최적 구현 방식 결정(예: HF Transformers 기본값 대신 ONNX Runtime AVX2 최적화 커널 선택 → RTF 37% 개선)
4. **선형 실행**: 10개 서브태스크를 순서대로 실행, 각 단계 후 자체 검증
5. **재탐색 없음**: 초기 조사가 충분하므로 맥락 재로드·반복 수정 불필요

### 인터랙티브 vs 계획 우선 — 언제 무엇을?

| 상황 | 권장 방식 | 이유 |
| --- | --- | --- |
| 빠른 단일 파일 수정 | 인터랙티브 | setup 오버헤드 없음 |
| 즉각 피드백이 필요한 실험 | 인터랙티브 | 방향 전환이 잦음 |
| 복잡한 다단계 구현(라이브러리 선택 포함) | 계획 우선 에이전트 | 리서치→선형 실행으로 반복 최소화 |
| 반복적 재작업이 예상되는 작업 | 계획 우선 에이전트 | 초기 계획으로 비용 급증 방지 |
| CI/CD 파이프라인 자동화 | 계획 우선 에이전트 | 무인 실행 시 비용 예측성 중요 |

계획 우선 에이전트의 핵심은 토큰을 아끼는 것이 아니라 **올바른 정보로 먼저 시작해 back-and-forth를 없애는 것**이다. [[08 서브에이전트와 에이전트 팀]]·[[09 훅]]을 활용해 사전 리서치 단계를 자동화하거나 MCP로 외부 에이전트를 Claude Code와 연결할 수 있다.

### 설정 예시 — MCP로 계획 에이전트 연결

`claude_desktop_config.json`의 `mcpServers` 블록에 로컬 에이전트를 MCP 서버로 등록하면 Claude Code에서 호출 가능:

```json
{
  "mcpServers": {
    "neo": {
      "command": "python3",
      "args": ["/path/to/neo/server.py"]
    }
  }
}
```

MCP 서버 설정 상세는 [[10 MCP]] 참고. 설정 파일 위치·스코프는 [[04 설정]] 참고.
