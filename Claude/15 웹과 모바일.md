# 15 웹과 모바일

Claude Code를 로컬 머신이 아닌 **Anthropic이 관리하는 클라우드 인프라**에서 실행하는 방식을 다룬다. 브라우저([claude.ai/code](https://claude.ai/code)) 또는 Claude 모바일 앱에서 작업(task)을 제출하면, Anthropic이 격리된 가상 머신(VM)에 저장소를 클론하고, 변경 작업을 수행한 뒤 리뷰용 브랜치를 푸시한다. 세션은 브라우저를 닫아도 유지되며 기기 간(노트북 → 휴대폰)에 이어서 작업할 수 있다.

> [!note] 리서치 프리뷰
> Claude Code on the web은 **research preview** 단계다. 대상: Pro / Max / Team 사용자, 그리고 premium seat 또는 Chat + Claude Code seat을 가진 Enterprise 사용자.

허브: [[Claude]] · 로컬 실행은 [[01 시작하기]], CLI는 [[02 CLI 레퍼런스]], 권한 모드는 [[05 권한]], 자기 머신에서 돌리며 웹에서 모니터링하는 Remote Control과 스케줄링은 [[13 자동화와 스케줄링]], 데스크톱 앱은 [[14 IDE와 데스크톱]] 참고.

## 시작하기 (web-quickstart)

웹 클라우드 세션은 로컬 설정이 전혀 없어도 동작한다. GitHub 저장소만 연결하면 Claude가 매 세션마다 새로 클론하므로 로컬에 체크아웃된 사본이 필요 없다.

### 잘 맞는 워크플로우

| 용도 | 설명 |
|------|------|
| **병렬 작업(Parallel tasks)** | 독립적인 여러 작업을 동시에 실행 — 각각 자체 세션과 브랜치. worktree 여러 개를 관리할 필요가 없다 |
| **로컬에 없는 저장소** | 매 세션 새로 클론하므로 로컬 체크아웃 불필요 |
| **잦은 개입이 필요 없는 작업** | 잘 정의된 작업을 제출하고 다른 일을 한 뒤 완료되면 리뷰 |
| **코드 질문/탐색** | 로컬 체크아웃 없이 코드베이스 이해, 기능 구현 추적 |

로컬 설정·도구·환경이 필요한 작업은 로컬 Claude Code([[01 시작하기]]) 또는 Remote Control([[13 자동화와 스케줄링]])이 더 낫다.

### 세션이 도는 방식

작업을 제출하면 다음 순서로 진행된다.

1. **Clone and prepare**: 저장소를 Anthropic VM에 클론하고, 설정한 setup script가 있으면 실행.
2. **Configure network**: 환경의 access level에 따라 인터넷 접근 설정.
3. **Work**: Claude가 코드 분석 → 변경 → 테스트 실행 → 자체 검증. 진행 중 관찰·조향(steer)하거나, 자리를 비웠다 완료 후 돌아와도 된다.
4. **Push the branch**: 멈춤 지점에 도달하면 브랜치를 GitHub에 푸시. diff 리뷰, 인라인 코멘트, PR 생성, 추가 메시지 전송이 가능하다.

브랜치를 푸시해도 세션은 닫히지 않는다. PR 생성과 추가 편집 모두 같은 대화 안에서 이뤄진다.

### 실행 방식 비교

Claude Code의 동작은 어디서나 동일하다. 달라지는 것은 **코드가 어디서 실행되는가**와 **로컬 설정을 쓸 수 있는가**다.

|  | On the web | Remote Control | Terminal CLI | Desktop app |
|--|-----------|----------------|--------------|-------------|
| **코드 실행 위치** | Anthropic cloud VM | 내 머신 | 내 머신 | 내 머신 또는 cloud VM |
| **채팅 위치** | claude.ai / 모바일 앱 | claude.ai / 모바일 앱 | 터미널 | Desktop UI |
| **로컬 설정 사용** | No (repo만) | Yes | Yes | 로컬 Yes / 클라우드 No |
| **GitHub 필요** | Yes (또는 `--remote`로 로컬 번들) | No | No | 클라우드 세션만 |
| **연결 끊겨도 계속 실행** | Yes | 터미널 열려 있는 동안 | No | 세션 타입에 따라 다름 |
| **권한 모드** | Auto accept edits, Plan | Ask, Auto accept edits, Plan | 모든 모드 | 세션 타입에 따라 다름 |
| **네트워크 접근** | 환경별 설정 가능 | 내 머신 네트워크 | 내 머신 네트워크 | 세션 타입에 따라 다름 |

> 권한 모드 전체는 [[05 권한]] 참고. 웹 클라우드 세션은 **Auto accept edits**와 **Plan** 두 가지만 제공한다 (Ask / Auto / Bypass 없음).

### GitHub 연결 및 환경 생성 (브라우저)

1회성 설정 과정이다. (GitHub CLI를 이미 쓴다면 [터미널에서 설정](#터미널에서-연결-web-setup) 가능.)

1. **claude.ai/code 방문**: [claude.ai/code](https://claude.ai/code)에서 Anthropic 계정으로 로그인.
2. **Claude GitHub App 설치**: 로그인 후 GitHub 연결 프롬프트를 따라 Claude GitHub App을 설치하고 저장소 접근 권한을 부여. 클라우드 세션은 **기존** GitHub 저장소에서 동작하므로, 새 프로젝트는 먼저 [빈 저장소 생성](https://github.com/new) 필요.
3. **환경 생성(Create environment)**: 환경은 세션 중 Claude의 네트워크 접근과 새 세션 시 실행 항목을 제어. 폼 필드는 아래와 같다.

| 필드 | 설명 |
|------|------|
| **Name** | 표시 라벨. 프로젝트/접근 수준별로 환경이 여러 개일 때 유용 |
| **Network access** | 세션이 인터넷에서 접근 가능한 범위. 기본값 `Trusted`는 npm·PyPI·RubyGems 등 공통 패키지 레지스트리는 허용하고 일반 인터넷은 차단 |
| **Environment variables** | 모든 세션에서 쓰이는 선택적 변수, `.env` 형식. 값에 따옴표를 쓰지 말 것(따옴표가 값의 일부로 저장됨). 이 환경을 편집할 수 있는 사람 누구나 볼 수 있음 |
| **Setup script** | Claude Code 실행 전에 도는 선택적 Bash 스크립트. VM에 없는 시스템 도구 설치용(예: `apt install -y gh`). 결과가 캐시되어 매 세션 재실행되지 않음 |

첫 프로젝트는 기본값을 그대로 두고 **Create environment** 클릭. 이후 편집·추가 가능.

### 터미널에서 연결 (`/web-setup`)

GitHub CLI(`gh`)를 이미 쓴다면 브라우저 없이 설정할 수 있다. [[01 시작하기]]의 Claude Code CLI가 필요하다. `/web-setup`은 로컬 `gh` 토큰을 읽어 Claude 계정에 연결하고, 환경이 없으면 기본 클라우드 환경을 만든다.

```bash
gh auth login            # 1. GitHub CLI 인증 (아직 안 했다면)
```
```text
/login                   # 2. Claude Code CLI 안에서 claude.ai 계정 로그인 (이미 했다면 생략)
/web-setup               # 3. gh 토큰을 Claude 계정에 동기화 + 환경 없으면 생성
```

`/web-setup` 완료 후엔 터미널에서 [`--remote`](#터미널-→-웹---remote)로 클라우드 세션을 시작하거나, `/schedule`([[13 자동화와 스케줄링]])로 반복 작업을 설정할 수 있다.

> [!note] 제약
> - **Zero Data Retention**이 켜진 조직은 `/web-setup` 등 클라우드 세션 기능을 쓸 수 없다.
> - GitHub CLI가 설치/인증되지 않았으면 `/web-setup`은 대신 브라우저 온보딩 흐름을 연다.

### 작업 시작 (Start a task)

1. **저장소·브랜치 선택**: claude.ai/code 또는 모바일 앱 Code 탭에서 입력 박스 아래 저장소 셀렉터로 저장소 선택. 브랜치 셀렉터로 기본 브랜치 대신 피처 브랜치에서 시작 가능. 여러 저장소를 추가해 한 세션에서 교차 작업도 가능.
2. **권한 모드 선택**: 입력창 옆 드롭다운 기본값은 **Auto accept edits**(승인 없이 변경·푸시). 편집 전 접근 방식을 제안받고 싶으면 **Plan mode**로 전환.
3. **작업 기술 후 제출**: 구체적으로 작성.
   - 파일/함수 명시: "Fix the failing auth test in `tests/test_auth.py`"가 "fix tests"보다 낫다
   - 에러 출력이 있으면 붙여넣기
   - 증상이 아닌 기대 동작을 서술

각 작업은 자체 세션과 브랜치를 갖는다 — 하나가 끝나길 기다릴 필요 없이 다음을 시작할 수 있다.

### 세션 사전 채우기 (Pre-fill sessions)

[claude.ai/code](https://claude.ai/code) URL에 쿼리 파라미터를 붙여 새 세션의 프롬프트·저장소·환경을 미리 채울 수 있다. 이슈 트래커에 "이슈 설명을 프롬프트로 하여 Claude Code 열기" 버튼 같은 통합을 만들 때 유용하다.

| 파라미터 | 설명 |
|----------|------|
| `prompt` | 입력 박스에 미리 채울 프롬프트 텍스트. 별칭 `q`도 허용 |
| `prompt_url` | 프롬프트가 쿼리 문자열에 담기엔 너무 길 때, 텍스트를 가져올 URL. cross-origin 요청 허용 필요. `prompt`가 함께 설정되면 무시됨 |
| `repositories` | 미리 선택할 `owner/repo` 슬러그의 쉼표 구분 목록. 별칭 `repo`도 허용 |
| `environment` | 미리 선택할 환경의 이름 또는 ID |

각 값은 URL 인코딩한다.

```text
https://claude.ai/code?prompt=Fix%20the%20login%20bug&repositories=acme/webapp
```

### 리뷰 및 반복 (Review and iterate)

1. **diff 뷰 열기**: 세션에 추가/삭제 라인 표시 인디케이터(예: `+42 -18`)를 선택하면, 좌측 파일 목록 / 우측 변경 내용의 diff 뷰가 열린다.
2. **인라인 코멘트**: diff의 라인을 선택해 피드백 입력 후 Enter. 코멘트는 다음 메시지를 보낼 때까지 큐에 쌓였다가 함께 묶여 전송된다. Claude는 메인 지시와 함께 "`src/auth.ts:47`에서 에러를 catch하지 말 것" 같은 위치 정보를 받는다.
3. **PR 생성**: diff가 만족스러우면 diff 뷰 상단 **Create PR** 선택. 정식 PR / 초안(draft) / GitHub compose 페이지로 이동(제목·설명 자동 생성) 중 선택.
4. **PR 이후에도 반복**: PR 생성 후에도 세션은 살아 있다. CI 실패 출력이나 리뷰어 코멘트를 채팅에 붙여넣고 처리를 요청. 자동 모니터링은 [Auto-fix](#auto-fix-pull-requests) 참고.

### 셋업 트러블슈팅

런타임 API 에러(`API Error: 500`, `529 Overloaded`, `429`, `Prompt is too long`)는 CLI·Desktop과 공통이므로 [[25 트러블슈팅]]의 Error reference를 본다. 클라우드 세션 고유 이슈는 다음과 같다.

| 증상 | 해결 |
|------|------|
| GitHub 연결 후 저장소가 안 보임 | 클라우드 세션은 App 설치 여부와 무관하게 연결 계정이 볼 수 있는 모든 저장소 사용 가능. 누락 시 연결된 GitHub 계정의 접근 권한 확인. Auto-fix를 원하면 github.com **Settings → Applications → Claude → Configure**에서 App을 해당 저장소에 설치 |
| GitHub 로그인 버튼만 표시 | 클라우드 세션은 GitHub 연결 필수. 브라우저 흐름으로 연결하거나 `/web-setup` 실행. GitHub 연결을 원치 않으면 Remote Control 사용 |
| "Not available for the selected organization" | Enterprise 조직은 관리자가 웹 기능을 활성화해야 함. Anthropic 계정 팀 문의 |
| `/web-setup` → "Unknown command" | `/web-setup`은 셸이 아닌 Claude Code CLI 안에서 실행. `claude` 먼저 실행 후 입력. 그래도 안 되면 CLI가 v2.1.80 미만이거나 API 키/서드파티로 인증된 상태 → `claude update` 후 `/login` |
| "Could not create a cloud environment" / "No cloud environment available" (`--remote`/ultraplan) | 전자는 자동 생성 실패, 후자는 CLI가 자동 생성 이전 버전. `/web-setup` 실행 또는 claude.ai/code에서 환경 수동 생성 |
| Setup script failed | 비-제로 종료로 세션 시작 차단. 원인: 레지스트리가 network access level에 없음 / 새 클론에 없는 파일·경로 참조 / 로컬과 다른 Ubuntu 호출 방식. 디버그: 스크립트 맨 위에 `set -x`, 비-치명 명령엔 `\|\| true` |
| 새 세션이 setup 중 멈춤/타임아웃 | setup script가 ~5분 캐시 빌드 예산 초과. 병렬화(`&` + `wait`), 큰 다운로드는 SessionStart hook으로 이동, 긴 retry sleep 제거 |
| 탭 닫아도 세션이 계속 돎 | 의도된 동작. 탭을 닫거나 이동해도 세션은 중지되지 않고 현재 작업을 끝낼 때까지 백그라운드 실행 후 idle. 사이드바에서 archive 또는 delete 가능 |

## 웹 클라우드 환경 레퍼런스 (claude-code-on-the-web)

### GitHub 인증 옵션

클라우드 세션은 코드 클론과 브랜치 푸시를 위해 GitHub 저장소 접근이 필요하다. 두 가지 방법이 있다.

| 방법 | 동작 방식 | 적합 대상 |
|------|----------|-----------|
| **GitHub App** | 웹 온보딩 중 Claude GitHub App 승인 | 브라우저 온보딩; Auto-fix를 원하는 팀 |
| **`/web-setup`** | 터미널에서 실행, 로컬 `gh` CLI 토큰을 Claude 계정에 동기화 | `gh`를 이미 쓰는 개별 개발자 |

> [!warning] 접근 범위 주의
> 어느 방법이든, 클라우드 세션은 연결된 GitHub 계정이 **볼 수 있는 모든 저장소**에 접근할 수 있다 — App이 설치된 저장소에 한정되지 않는다. App 설치는 Auto-fix용 PR webhook을 활성화할 뿐 세션 단위 접근 제어가 아니다. 팀이 클라우드 세션에서 닿을 수 있는 저장소를 제한하려면 **GitHub 자체에서** 팀/저장소 멤버십을 제한해야 한다.

- `/schedule`([[13 자동화와 스케줄링]])은 둘 중 하나라도 있는지 확인하고, 없으면 `/web-setup` 실행을 안내한다.
- **Auto-fix는 GitHub App 필수** (PR webhook 수신용). `/web-setup`으로 연결했다가 나중에 Auto-fix가 필요하면 해당 저장소에 App 설치.
- Team/Enterprise 관리자는 [claude.ai/admin-settings/claude-code](https://claude.ai/admin-settings/claude-code)의 Quick web setup 토글로 `/web-setup`을 비활성화 가능.
- **Zero Data Retention** 조직은 `/web-setup` 등 클라우드 세션 기능 사용 불가.

### 클라우드 환경에서 쓸 수 있는 것

각 세션은 저장소가 클론된 새 Anthropic VM에서 돈다. 저장소에 **커밋된 것**은 사용 가능하고, 내 머신에만 설치/설정된 것은 사용 불가다.

| 항목 | 클라우드 세션에서 사용 가능 | 이유 |
|------|:---:|------|
| 저장소의 `CLAUDE.md` | Yes | 클론의 일부 |
| 저장소의 `.claude/settings.json` 훅 | Yes | 클론의 일부 |
| 저장소의 `.mcp.json` MCP 서버 | Yes | 클론의 일부 |
| 저장소의 `.claude/rules/` | Yes | 클론의 일부 |
| 저장소의 `.claude/skills/`, `.claude/agents/`, `.claude/commands/` | Yes | 클론의 일부 |
| `.claude/settings.json`에 선언된 플러그인 | Yes | 세션 시작 시 선언한 marketplace에서 설치. marketplace 접근에 네트워크 필요 |
| 사용자 `~/.claude/CLAUDE.md` | **No** | 저장소가 아닌 내 머신에 존재 |
| 사용자 설정에만 활성화된 플러그인 | **No** | `enabledPlugins`가 `~/.claude/settings.json`에 있음 → 저장소 `.claude/settings.json`에 선언할 것 |
| `claude mcp add`로 추가한 MCP 서버 | **No** | 로컬 사용자 설정에 기록됨 → `.mcp.json`(project scope)에 선언할 것 |
| 정적 API 토큰·자격증명 | **No** | 전용 secrets store가 아직 없음(아래 참조) |
| AWS SSO 등 대화형 인증 | **No** | 브라우저 기반 로그인은 클라우드 세션에서 불가 |

> 설정을 클라우드 세션에서 쓰려면 **저장소에 커밋**한다. 전용 secrets store는 아직 없다. 환경 변수와 setup script 모두 환경 설정에 저장되어 그 환경을 편집할 수 있는 누구에게나 보인다 — 비밀 정보를 환경 변수로 넣을 땐 이 가시성을 감안하라. CLAUDE.md/메모리 동작은 [[03 메모리와 컨텍스트]], 설정 파일 구조는 [[04 설정]], 훅은 [[09 훅]], MCP는 [[10 MCP]], 플러그인은 [[11 플러그인]], 스킬은 [[07 스킬]] 참고.

#### GitHub 이슈/PR 작업

클라우드 세션엔 내장 GitHub 도구가 있어 별도 설정 없이 이슈 읽기·PR 목록·diff 조회·코멘트 작성이 가능하다. 이 도구들은 [GitHub proxy](#github-proxy)를 통해 인증하므로 토큰이 컨테이너에 들어가지 않는다.

`gh` CLI는 **사전 설치되어 있지 않다**. `gh release`, `gh workflow run` 등 내장 도구가 못 다루는 명령이 필요하면 직접 설치·인증한다.

1. setup script에 `apt update && apt install -y gh` 추가.
2. 환경 설정에 `GH_TOKEN` 환경 변수로 GitHub PAT 추가. `gh`는 `GH_TOKEN`을 자동으로 읽으므로 `gh auth login` 단계가 필요 없다.

#### 산출물을 세션으로 역링크

각 클라우드 세션엔 claude.ai 트랜스크립트 URL이 있고, 세션은 `CLAUDE_CODE_REMOTE_SESSION_ID` 환경 변수에서 자기 ID를 읽을 수 있다. PR 본문·커밋 메시지·Slack 글·생성 리포트에 추적 가능한 링크를 넣어 리뷰어가 그 산출물을 만든 실행을 열 수 있게 한다.

변수 값은 `cse_` prefix를 쓰지만 트랜스크립트 URL 경로는 같은 ID에 `session_` prefix를 쓴다. prefix를 치환해 링크를 만든다.

```bash
echo "https://claude.ai/code/${CLAUDE_CODE_REMOTE_SESSION_ID/#cse_/session_}"
```

#### 테스트 실행·서비스 시작·패키지 추가

- 테스트: 프롬프트에 "fix the failing tests in `tests/`" 또는 "run pytest after each change"처럼 요청. pytest·jest·cargo test 등은 사전 설치되어 바로 동작.
- **PostgreSQL/Redis는 설치되어 있으나 기본 실행 안 됨** — 세션 중 시작 요청:
  ```bash
  service postgresql start
  ```
  ```bash
  service redis-server start
  ```
- **Docker**: 컨테이너 서비스용. `docker compose up` 요청 가능. 이미지 pull은 환경 access level을 따르며 Trusted 기본 도메인에 Docker Hub 등이 포함됨. 큰 이미지는 setup script에 `docker compose pull`/`docker compose build`를 넣으면 [캐시](#환경-캐싱-environment-caching)되어 매 세션 디스크에 존재(단 실행 프로세스는 캐시 안 됨 — 매 세션 컨테이너 시작 필요).
- **패키지 추가**: 미리 설치되지 않은 건 setup script로. 출력이 캐시되어 매 세션 재설치 없이 사용 가능. 세션 중 설치 요청도 가능하나 다른 세션엔 안 이어진다.

#### 환경 구성 (Configure your environment)

환경은 네트워크 접근, 환경 변수, 세션 시작 전 도는 setup script를 제어한다. 웹 인터페이스 또는 터미널에서 관리한다.

| 작업 | 방법 |
|------|------|
| 환경 추가 | 현재 환경을 선택해 셀렉터를 열고 **Add environment**. 다이얼로그에 이름, 네트워크 접근 수준, 환경 변수, setup script 포함 |
| 환경 편집 | 현재 환경 이름이 표시된 cloud 아이콘 선택 → 셀렉터에서 환경 위에 hover → 우측 settings 아이콘 클릭 |
| 환경 보관 | 편집 화면에서 **Archive**. 보관된 환경은 셀렉터에서 숨겨지나 기존 세션은 계속 실행 |
| `--remote` 기본값 설정 | 터미널에서 `/remote-env`. 환경이 하나면 현재 구성 표시. `/remote-env`는 기본값 선택만 함; 추가·편집·보관은 웹에서 |

```text

LOG_LEVEL=debug
DATABASE_URL=postgres://localhost:5432/myapp
```

### Setup scripts

setup script는 새 클라우드 세션이 시작될 때, Claude Code 실행 **전에** 도는 Bash 스크립트다. 의존성 설치·도구 구성·사전 미설치 항목 fetch에 쓴다. **root로 Ubuntu 24.04에서** 실행되므로 `apt install`과 대부분의 패키지 매니저가 동작한다. 환경 설정 다이얼로그의 **Setup script** 필드에 입력한다.

```bash
#!/bin/bash
apt update && apt install -y gh
```

- 스크립트가 비-제로로 종료하면 세션이 시작되지 않는다. 비-치명 명령엔 `|| true`를 붙여 일시적 설치 실패가 세션을 막지 않게 한다.
- 총 실행 시간을 **약 5분 이내**로 유지해야 [환경 캐시](#환경-캐싱-environment-caching)가 빌드된다. 독립 설치는 `&` + `wait`로 병렬화. 단일 다운로드가 5분을 못 맞추면 [SessionStart hook](#setup-script-vs-sessionstart-hook)으로 옮겨 백그라운드 실행.

> [!note] 네트워크 필요
> 패키지를 설치하는 setup script는 레지스트리 접근을 위해 네트워크가 필요하다. 기본 **Trusted**는 npm·PyPI·RubyGems·crates.io 등 공통 레지스트리를 허용한다. **None**에서는 패키지 설치가 실패한다.

#### 환경 캐싱 (Environment caching)

setup script는 한 환경에서 세션을 **처음** 시작할 때 실행된다. 완료 후 Anthropic이 파일시스템을 스냅샷해 이후 세션의 시작점으로 재사용한다. 새 세션은 의존성·도구·Docker 이미지가 디스크에 이미 있는 상태로 시작하고 setup script 단계는 건너뛴다.

- 캐시는 **파일만** 캡처하며 실행 프로세스는 안 됨. setup script가 디스크에 쓴 건 이어지지만, 시작한 서비스/컨테이너는 안 이어진다 → 세션마다 Claude에게 요청하거나 SessionStart hook으로 시작.
- setup script나 허용 네트워크 호스트를 변경하면, 그리고 캐시 만료(약 7일) 시 캐시를 다시 빌드하려고 재실행된다.
- 기존 세션 resume은 setup script를 **절대 재실행하지 않는다**.
- 캐싱은 직접 활성화/관리할 필요 없다.

#### Setup script vs. SessionStart hook

언어 런타임·CLI 도구처럼 "클라우드엔 필요하지만 내 노트북엔 이미 있는 것"은 setup script로. `npm install`처럼 클라우드·로컬 모두에서 돌아야 하는 프로젝트 셋업은 [SessionStart hook]([[09 훅]])으로.

|  | Setup scripts | SessionStart hooks |
|--|---------------|--------------------|
| 부착 위치 | 클라우드 환경 | 내 저장소 |
| 설정 위치 | 클라우드 환경 UI | 저장소의 `.claude/settings.json` |
| 실행 시점 | Claude Code 실행 전, 캐시된 환경이 없을 때 | Claude Code 실행 후, resume 포함 모든 세션 |
| 범위 | 클라우드 환경 전용 | 로컬·클라우드 둘 다 |

SessionStart hook은 로컬 `~/.claude/settings.json`에도 정의할 수 있으나, **사용자 레벨 설정은 클라우드로 이어지지 않는다**. 클라우드에선 저장소에 커밋된 훅만 실행된다.

#### SessionStart hook으로 의존성 설치

클라우드 세션에서만 의존성을 설치하려면 저장소의 `.claude/settings.json`에 SessionStart hook을 추가한다.

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/install_pkgs.sh"
          }
        ]
      }
    ]
  }
}
```

`scripts/install_pkgs.sh`를 만들고 `chmod +x`. 클라우드 세션에선 `CLAUDE_CODE_REMOTE=true`이므로 이를 검사해 로컬 실행을 건너뛴다.

```bash
#!/bin/bash

if [ "$CLAUDE_CODE_REMOTE" != "true" ]; then
  exit 0
fi

npm install
pip install -r requirements.txt
exit 0
```

클라우드 세션에서 SessionStart hook의 제약:

- **클라우드 전용 스코핑 없음**: 로컬·클라우드 모두에서 실행. `CLAUDE_CODE_REMOTE`로 분기.
- **네트워크 필요**: install 명령이 레지스트리에 닿아야 함. **None**이면 실패. **Trusted** 기본 allowlist가 npm·PyPI·RubyGems·crates.io 커버.
- **Proxy 호환성**: 모든 아웃바운드가 [보안 프록시](#보안-프록시-security-proxy)를 지남. 일부 패키지 매니저는 미작동(Bun 알려진 예).
- **시작 지연 추가**: setup script와 달리 캐시 이점 없이 매 세션/resume마다 실행. 이미 설치됐는지 검사해 빠르게 유지.

후속 Bash 명령에 환경 변수를 영속하려면 `$CLAUDE_ENV_FILE`에 쓴다([[09 훅]] SessionStart 참고). 베이스 이미지를 자기 Docker 이미지로 교체하는 것은 아직 미지원 — setup script로 제공 이미지 위에 설치하거나 `docker compose`로 컨테이너 실행.

### 네트워크 접근 (Network access)

네트워크 접근은 클라우드 환경의 아웃바운드 연결을 제어한다. 각 환경은 하나의 access level을 지정하고 커스텀 허용 도메인으로 확장할 수 있다. 기본값은 **Trusted**(패키지 레지스트리 등 allowlist 도메인 허용). 환경을 편집해 다이얼로그의 **Network access** 셀렉터로 변경한다.

> [!note] MCP connector 트래픽
> MCP connector 트래픽은 Anthropic 서버를 경유하므로, 세션/routine에서 활성화한 connector는 **Allowed domains**에 호스트를 추가하지 않아도 동작한다. connector는 세션/routine 단위로 설정되며 불필요한 건 제거해 도달 가능한 도구를 제한하라.

#### Access levels

| 레벨 | 아웃바운드 연결 |
|------|-----------------|
| **None** | 아웃바운드 네트워크 접근 없음 |
| **Trusted** | allowlist 도메인만: 패키지 레지스트리, GitHub, cloud SDK |
| **Full** | 모든 도메인 |
| **Custom** | 자체 allowlist (기본 목록 포함 선택 가능) |

GitHub 작업은 이 설정과 무관한 별도 [GitHub proxy](#github-proxy)를 쓴다.

#### 특정 도메인 허용

Trusted에 없는 도메인을 허용하려면 환경 네트워크 설정에서 **Custom**을 선택하면 **Allowed domains** 필드가 나타난다. 한 줄에 한 도메인.

```text
api.example.com
*.internal.example.com
registry.example.com
```

`*.`는 와일드카드 서브도메인 매칭. **"Also include default list of common package managers"**를 체크하면 Trusted 도메인을 커스텀 항목과 함께 유지하고, 체크 해제하면 나열한 것만 허용.

#### GitHub proxy

보안을 위해 모든 GitHub 작업이 전용 프록시를 거친다. 샌드박스 내부의 git 클라이언트는 커스텀 빌드된 scoped credential로 인증하며, 프록시가 이를 검증해 실제 GitHub 토큰으로 변환한다.

- GitHub 인증을 안전하게 관리 (scoped credential ↔ 실제 토큰 변환)
- `git push`를 현재 작업 브랜치로만 제한 (안전)
- 보안 경계를 유지하며 clone·fetch·PR 작업 허용

#### 보안 프록시 (Security proxy)

환경은 보안·남용 방지를 위해 HTTP/HTTPS 네트워크 프록시 뒤에서 돈다. 모든 아웃바운드 트래픽이 이를 지난다.

- 악성 요청 차단
- rate limiting·남용 방지
- content filtering

### 작업을 웹↔터미널 간 이동

이 워크플로우들은 같은 claude.ai 계정으로 로그인된 [Claude Code CLI]([[01 시작하기]])가 필요하다. 터미널에서 새 클라우드 세션을 시작하거나, 클라우드 세션을 터미널로 끌어와 로컬에서 이어갈 수 있다.

> [!note] 핸드오프는 한 방향
> CLI에서 세션 핸드오프는 한 방향이다: `--teleport`로 클라우드 세션을 터미널로 끌어올 수는 있어도, 기존 터미널 세션을 웹으로 밀어 올릴 수는 없다. `--remote`는 현재 저장소의 **새** 클라우드 세션을 만든다. Desktop 앱([[14 IDE와 데스크톱]])의 **Continue in** 메뉴는 로컬 세션을 웹으로 보낼 수 있다.

#### 터미널 → 웹 (`--remote`)

```bash
claude --remote "Fix the authentication bug in src/auth/login.ts"
```

claude.ai에 새 클라우드 세션을 만든다. 세션은 현재 디렉터리의 GitHub remote를 현재 브랜치로 클론한다 — VM은 내 머신이 아닌 GitHub에서 클론하므로 로컬 커밋이 있으면 **먼저 push**할 것. `--remote`는 한 번에 단일 저장소.

> [!note] `--remote` ≠ `--remote-control`
> `--remote`는 클라우드 세션을 만든다. `--remote-control`은 무관하다 — 로컬 CLI 세션을 웹에서 모니터링하도록 노출한다 (Remote Control, [[13 자동화와 스케줄링]]).

CLI에서 `/tasks`로 진행 확인, 또는 claude.ai/모바일 앱에서 직접 조향·피드백·질문 응답.

**팁:**
- **로컬 계획, 원격 실행**: 복잡한 작업은 plan mode로 접근을 협의한 뒤 클라우드로 보낸다.
  ```bash
  claude --permission-mode plan
  ```
  plan을 저장소에 저장·커밋·push한 뒤:
  ```bash
  claude --remote "Execute the migration plan in docs/migration-plan.md"
  ```
- **ultraplan으로 클라우드에서 계획**: 웹 세션에서 계획 자체를 초안·리뷰. Claude가 웹에서 계획 생성 → 브라우저에서 섹션별 코멘트 → 원격 실행 또는 터미널로 회송.
- **병렬 실행**: 각 `--remote`는 독립 세션. 여러 작업을 동시에.
  ```bash
  claude --remote "Fix the flaky test in auth.spec.ts"
  claude --remote "Update the API documentation"
  claude --remote "Refactor the logger to use structured output"
  ```
  `/tasks`로 전체 모니터링.

#### GitHub 없이 로컬 저장소 전송

GitHub에 연결되지 않은 저장소에서 `claude --remote`를 실행하면, Claude Code가 로컬 저장소를 번들로 묶어 클라우드 세션에 직접 업로드한다. 번들엔 모든 브랜치의 전체 히스토리 + 추적 파일의 미커밋 변경이 포함된다. GitHub 접근이 없으면 자동 작동. GitHub가 연결돼 있어도 강제하려면 `CCR_FORCE_BUNDLE=1`:

```bash
CCR_FORCE_BUNDLE=1 claude --remote "Run the test suite and fix any failures"
```

번들 제약:
- git 저장소이고 커밋이 1개 이상
- 100 MB 미만 (초과 시 현재 브랜치만 → 단일 squash 스냅샷 순으로 폴백, 그래도 크면 실패)
- 추적되지 않는 파일 미포함 — `git add` 필요
- 번들로 만든 세션은 GitHub 인증이 별도로 없으면 remote로 push 불가

#### 웹 → 터미널 (teleport)

클라우드 세션을 터미널로 끌어오는 방법:

| 방법 | 동작 |
|------|------|
| `claude --teleport` | 대화형 세션 피커 |
| `claude --teleport <session-id>` | 특정 세션 직접 resume |
| `/teleport` (또는 `/tp`) | 기존 CLI 세션 안에서 재시작 없이 피커 열기 |
| `/tasks` → `t` | 백그라운드 세션 목록에서 teleport |
| 웹 인터페이스 **Open in CLI** | 터미널에 붙여넣을 명령 복사 |

teleport 시 Claude는 올바른 저장소인지 확인하고, 클라우드 세션의 브랜치를 fetch·checkout하며, 전체 대화 히스토리를 터미널에 로드한다. 미커밋 변경이 있으면 stash 안내. `--teleport`는 `--resume`과 다르다 — `--resume`은 이 머신의 로컬 히스토리에서 대화를 재개(클라우드 세션 미표시), `--teleport`는 클라우드 세션과 그 브랜치를 끌어온다.

**Teleport 요건:**

| 요건 | 상세 |
|------|------|
| Clean git state | 작업 디렉터리에 미커밋 변경 없음 (필요 시 stash 안내) |
| Correct repository | 같은 저장소의 체크아웃에서 실행 (fork 아님) |
| Branch available | 클라우드 세션의 브랜치가 remote에 push되어 있어야 함 (자동 fetch·checkout) |
| Same account | 클라우드 세션과 동일 claude.ai 계정으로 인증 |

**`--teleport` 불가 시:** teleport는 claude.ai 구독 인증이 필요하다. API key / Bedrock / Vertex AI / Microsoft Foundry로 인증돼 있으면 `/login`으로 claude.ai 계정 로그인. 이미 claude.ai로 로그인했는데도 불가하면 조직이 클라우드 세션을 비활성화했을 수 있다. ([[17 클라우드 프로바이더]] 참고)

### 세션 작업 (Work with sessions)

세션은 claude.ai/code 사이드바에 나타난다. 변경 리뷰, 팀원과 공유, 보관, 영구 삭제가 가능하다.

#### 컨텍스트 관리

클라우드 세션은 텍스트 출력을 내는 내장 명령([[02 CLI 레퍼런스]])을 지원한다. `/model`, `/config`처럼 대화형 터미널 피커를 여는 명령은 사용 불가.

| 명령 | 클라우드 세션 | 비고 |
|------|:---:|------|
| `/compact` | Yes | 대화를 요약해 컨텍스트 확보. 포커스 지시 가능: `/compact keep the test output` |
| `/context` | Yes | 현재 컨텍스트 윈도우 내용 표시 |
| `/clear` | No | 대신 사이드바에서 새 세션 시작 |

Auto-compaction은 컨텍스트가 한계에 가까워지면 CLI와 동일하게 자동 실행된다. 더 일찍 트리거하려면 환경 변수에 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 설정(예: `=70`이면 기본 ~95% 대신 70%에서 compact). compaction 계산용 윈도우 크기는 `CLAUDE_CODE_AUTO_COMPACT_WINDOW`로 변경. (컨텍스트 관리 일반은 [[03 메모리와 컨텍스트]])

[Subagents]([[08 서브에이전트와 에이전트 팀]])는 로컬과 동일하게 동작한다. 저장소의 `.claude/agents/`에 정의된 것은 자동 인식. **Agent teams**는 기본 off이며 환경 변수에 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 추가로 활성화.

#### 변경 리뷰 / 공유 / 보관 / 삭제

- **변경 리뷰**: 각 세션은 `+42 -18` 같은 diff 인디케이터를 보여준다. 선택해 diff 뷰를 열고 특정 라인에 인라인 코멘트를 남겨 다음 메시지와 함께 Claude에게 전송. (전체 워크플로우는 위 [리뷰 및 반복](#리뷰-및-반복-review-and-iterate))
- **공유**: 가시성을 토글한 뒤 링크를 공유. 수신자는 여는 시점의 최신 상태를 보며 실시간 갱신은 안 됨.
  - **Enterprise/Team**: **Private** / **Team**. Team은 claude.ai 조직 멤버에게 공개. 저장소 접근 검증이 기본 활성(수신자 GitHub 계정 기준). 내 표시 이름이 모든 수신자에게 보임. [Claude in Slack]([[16 CI-CD와 팀 통합]]) 세션은 자동으로 Team 가시성으로 공유.
  - **Max/Pro**: **Private** / **Public**. Public은 claude.ai 로그인 사용자 누구에게나 공개. 저장소 접근 검증이 기본 비활성 → 공유 전 민감 콘텐츠(private repo 코드·자격증명) 확인. 수신자에게 저장소 접근을 요구하거나 이름을 숨기려면 **Settings > Claude Code > Sharing settings**.
- **보관(Archive)**: 사이드바에서 세션 위에 hover → archive 아이콘. 기본 목록에서 숨겨지나 보관 필터로 조회 가능.
- **삭제(Delete)**: 영구 삭제, 되돌릴 수 없음. (1) 사이드바에서 보관 필터 → hover → delete 아이콘, 또는 (2) 세션 메뉴에서 제목 옆 드롭다운 → **Delete**. 확인 후 삭제.

### Auto-fix pull requests

Claude가 PR을 감시하며 CI 실패와 리뷰 코멘트에 자동 대응한다. PR의 GitHub 활동을 구독해, 체크 실패나 리뷰어 코멘트가 생기면 조사하고 명확한 수정이 있으면 push한다.

> [!note] Auto-fix는 Claude GitHub App 필수
> PR webhook 수신용. 아직 없으면 [GitHub App 페이지](https://github.com/apps/claude)에서 설치.

켜는 방법:
- **웹에서 만든 PR**: CI status bar를 열고 **Auto-fix** 선택
- **터미널에서**: PR 브랜치에서 `/autofix-pr` 실행. `gh`로 열린 PR을 감지해 웹 세션을 spawn하고 한 번에 auto-fix 활성화
- **모바일 앱에서**: "watch this PR and fix any CI failures or review comments"처럼 요청
- **임의의 기존 PR**: PR URL을 세션에 붙여넣고 auto-fix 요청

per-PR 토글이다. 중지하려면 웹 세션 CI status bar에서 **Auto-fix** 토글 해제, 또는 Claude에게 감시 중지 요청.

#### Claude가 PR 활동에 대응하는 방식

- **Clear fixes**: 확신이 있고 이전 지시와 충돌 없으면 변경·push 후 세션에 설명
- **Ambiguous requests**: 다의적이거나 아키텍처적으로 중요한 코멘트면 행동 전에 질문
- **Duplicate/no-action**: 중복이거나 변경 불필요하면 세션에 기록하고 넘어감

base 브랜치 전진으로 인한 머지 충돌은 GitHub가 webhook을 안 보내므로 auto-fix가 스스로 대응 못 함 → 세션을 열어 rebase 요청. Claude는 리뷰 코멘트 스레드에 답글을 달 수 있는데, **내 GitHub 계정으로** 게시되어 내 username 아래 나타나지만 각 답글엔 Claude Code 작성 라벨이 붙는다.

> [!warning] 코멘트 트리거 자동화 주의
> 저장소가 Atlantis, Terraform Cloud, `issue_comment` 이벤트로 도는 커스텀 GitHub Actions 등 코멘트 트리거 자동화를 쓴다면, Claude가 내 대신 답글을 달아 그 워크플로우를 트리거할 수 있다. auto-fix 활성화 전 자동화를 검토하고, PR 코멘트가 인프라를 배포하거나 권한 작업을 실행하는 저장소에서는 비활성화를 고려하라. (팀 통합은 [[16 CI-CD와 팀 통합]])

### 보안과 격리 (Security and isolation)

각 클라우드 세션은 여러 계층으로 내 머신·다른 세션과 분리된다. (전체 샌드박스 모델은 [[18 보안과 샌드박스]])

- **격리된 VM**: 각 세션은 격리된 Anthropic 관리 VM에서 실행
- **네트워크 접근 제어**: 기본적으로 제한되며 비활성화 가능. 단 비활성화해도 Claude Code는 여전히 Anthropic API와 통신할 수 있어 데이터가 VM을 나갈 수 있음
- **자격증명 보호**: git credential·서명 키 등 민감 자격증명은 샌드박스 안의 Claude Code와 함께 두지 않음. 인증은 scoped credential을 쓰는 보안 프록시로 처리
- **안전한 분석**: 코드는 PR 생성 전 격리된 VM 안에서 분석·수정

### 트러블슈팅 (클라우드 고유)

런타임 API 에러는 [[25 트러블슈팅]] 참고. 클라우드 세션 고유 이슈:

| 증상 | 해결 |
|------|------|
| `Session creation failed` / provisioning 정체 | 환경 할당 실패. status.claude.com 확인 → 1분 후 재시도 → 저장소 도달 가능 여부 확인(연결 계정이 GitHub에서 접근 가능해야 함; App 설치는 불필요) |
| `Remote Control session expired` / `Access denied` (teleport) | teleport는 Remote Control 인프라를 거쳐 해당 문구가 뜸. `/login`으로 자격증명 갱신 후 재연결 → 세션 소유 계정과 동일한지 확인. `Remote Control may not be available for this organization`이면 관리자가 원격 세션 미활성화 |
| `Environment expired` | 비활성 기간 후 환경 회수. 로컬에선 `Could not resume session ... its environment has expired. Creating a fresh session instead.`로, 웹에선 expired로 표시. claude.ai/code에서 재오픈하면 대화 히스토리가 복원된 새 환경 프로비저닝 |

### 제약 (Limitations)

- **Rate limits**: 웹은 계정 내 다른 모든 Claude/Claude Code 사용과 rate limit을 공유. 병렬 작업이 많을수록 비례 소비. 클라우드 VM에 대한 별도 컴퓨트 과금은 없음. (비용은 [[19 비용과 성능]])
- **Repository authentication**: 동일 계정 인증 시에만 웹 → 로컬 이동 가능
- **Platform restrictions**: 저장소 클론·PR 생성은 GitHub 필요. 자체 호스팅 GitHub Enterprise Server는 Team/Enterprise 플랜에서 지원. GitLab/Bitbucket 등 비-GitHub 저장소는 로컬 번들로 보낼 수 있으나 결과를 remote로 push 불가
- **Organization IP allowlist**: 클라우드 세션은 내 네트워크가 아닌 Anthropic 관리 인프라에서 Anthropic API를 호출. 조직이 IP allowlisting을 켜면 모든 클라우드 세션이 인증 에러로 실패(Code Review·Routines도 동일). Anthropic 지원에 문의해 Anthropic 호스팅 서비스를 allowlist에서 제외

## 모바일 (mobile)

Claude 앱(iOS·Android)은 코드가 실제로 도는 곳이 아니라 Claude Code 세션의 **클라이언트**다. 폰에서 닿을 수 있는 대상은 셋: (1) 위에서 다룬 Anthropic 클라우드 세션, (2) 자기 머신에서 도는 세션을 이어가는 **Remote Control**, (3) Desktop 앱에 세션을 띄우는 **Dispatch**([[14 IDE와 데스크톱]]). 셋 다 앱의 **Code** 탭에서 접근하며 별도 모바일 전용 앱은 없다.

### 앱 받기

[iOS](https://apps.apple.com/us/app/claude-by-anthropic/id6473753684) / [Android](https://play.google.com/store/apps/details?id=com.anthropic.claude) 설치(iPad는 iOS 앱 그대로). Claude Code 세션에서 `/mobile`(별칭 `/ios`·`/android`)을 실행하면 다운로드 QR 코드가 뜬다. claude.ai와 같은 계정으로 로그인 — 클라우드 세션·Remote Control은 claude.ai 계정이 필요해 Anthropic Console API key나 Bedrock 등 서드파티 provider로는 접근 불가.

### 세 가지 접속 방식 비교

| 기능 | 연결 대상 | 언제 쓰나 |
|------|-----------|-----------|
| 클라우드 세션 | Anthropic 관리 인프라 | 저장소가 GitHub에 있고 폰을 내려놔도 계속 돌아야 할 때. [web-quickstart](#web-quickstart)로 셋업 |
| Remote Control | 내 컴퓨터에서 도는 세션 | 로컬 파일시스템·도구·MCP 서버가 필요할 때 |
| Dispatch | Desktop 앱 | task만 메시지로 던지고 실행 방식은 Dispatch가 결정하게 할 때. Pro·Max 플랜 필요 |

컴퓨터가 꺼져 있을 거면 클라우드 세션(랩탑 닫아도 계속 실행). Remote Control·Dispatch는 내 머신을 구동하므로 Claude Code·Desktop 앱이 켜져 있어야 한다(Remote Control 세션 중 머신이 잠들면 깨어날 때 재연결).

### Remote Control로 로컬 세션 이어가기

`claude remote-control` 실행(또는 이미 연 세션에서 `/remote-control`) 후 터미널의 QR 코드를 스캔하거나, 앱에서 **Code** → 세션 목록에서 선택. 코드 실행·파일 접근은 로컬에 머물고 UI만 원격이다. 앱에서 첨부한 이미지·파일은 로컬 세션으로 전달되어 `@` 파일 참조로 전달된다.

### 푸시 알림

Remote Control 활성 중엔 장시간 작업 완료나 결정이 필요할 때 폰으로 푸시 알림을 보낼 수 있다(`notify me when the tests finish`처럼 프롬프트로 요청 가능). Dispatch는 완료·승인 필요 시 자체 알림을 보낸다.

### 제약

- **로컬 전용 명령**: `/plugin`·`/resume`처럼 터미널 UI 전용 명령은 앱에서 미동작.
- **권한 모드**: 클라우드 세션은 Auto accept edits·Plan·Auto만, Remote Control은 Manual·Auto accept edits·Plan만 — 둘 다 **Bypass permissions 선택 불가**, Remote Control은 Auto도 불가. ([[05 권한]])
- **Dispatch**: Pro·Max 플랜 전용, Team·Enterprise 미제공.

## Artifacts — 세션 산출물을 라이브 페이지로 공유 (artifacts)

> [!note] 가용성
> Pro·Max·Team·Enterprise 플랜에서 `/login`으로 인증한 세션에 한함. Anthropic API 전용 — Amazon Bedrock·Google Cloud's Agent Platform·Microsoft Foundry에서는 불가. Agent SDK·GitHub Action·MCP-server 컨텍스트와 `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` 설정 시에는 기본 off.

artifact는 Claude Code가 세션에서 claude.ai의 **private URL**로 발행하는 라이브 인터랙티브 페이지다. 브라우저로 열면 세션이 계속되는 동안 그 자리에서 갱신된다. Team·Enterprise 플랜은 페이지 헤더의 Share로 동료에게 공유 가능. 터미널 텍스트로는 보기 힘든 결과 — 주석 단 diff로 PR을 설명하거나, 세션 데이터로 대시보드를 만들거나, 진행 중인 조사 타임라인을 채워가는 용도에 적합.

### 만들기

Claude가 출력이 페이지에 적합하다고 판단하면 스스로 발행하거나, 직접 요청할 수 있다.

```text
Make an artifact that walks through this PR with the diff annotated inline.
```

Claude가 프로젝트에 HTML/Markdown 파일을 작성한 뒤 발행 — 새 artifact 발행 전 항상 권한을 묻는다(`Claude wants to publish "..." to a private page on claude.ai`). 이미 승인한 artifact를 재발행할 땐 다시 묻지 않는다. `Ctrl+]`로 언제든 가장 최근 artifact를 재오픈. 브라우저 자동 오픈을 끄려면 `CLAUDE_CODE_ARTIFACT_AUTO_OPEN=0`.

### 갱신과 공유

수정을 요청하거나 장시간 작업이 진행되며 재발행하면 같은 URL이 갱신된다. 발행마다 버전이 생기고 페이지 헤더 **Share**에서 뷰어가 볼 버전을 고를 수 있다. 다른 세션에서 갱신하려면 artifact URL을 주고 수정을 요청 — URL 없이는 항상 새 artifact가 생성된다.

Pro·Max는 본인에게만 비공개. Team·Enterprise는 헤더의 **Share**로 조직 내 특정 인원 또는 전체에 접근 권한 부여 가능(조직 경계를 넘어 공유 불가). 뷰어는 각 버전을 볼 수 있지만 편집은 못 한다 — 작성자는 항상 나 하나.

### 페이지 제약

artifact는 자체 완결된 단일 페이지다. Claude Code가 엄격한 CSP로 HTML 문서 셸에 감싸 서빙한다.

| 제약 | 내용 |
| --- | --- |
| 외부 요청 불가 | 다른 호스트의 스크립트·스타일시트·폰트·이미지, `fetch`/XHR/WebSocket 모두 CSP가 차단 — CSS·JS는 인라인, 이미지는 data URI로 임베드 |
| 백엔드 없음 | 정적 페이지 — 폼 입력 저장·뷰 시점 인증·API 호출 불가 |
| 단일 페이지 | 상대 링크 미작동(같이 배포되는 파일 없음) — 여러 섹션은 페이지 내 앵커로 |
| 소스 파일 타입 | `.html`/`.htm`/`.md`만(Markdown은 스타일된 HTML로 렌더) |
| 렌더 크기 | 16 MiB 이하 — 큰 임베드 이미지가 실패의 주 원인 |

인라인 CSS·인터랙션용 JS·특히 data URI 이미지가 토큰 비용을 키운다. 다이어그램은 래스터 대신 SVG/HTML+CSS, 불필요한 인터랙션 생략, 큰 데이터셋은 전체 인라인 대신 요약으로 비용을 줄인다.

### 끄기 / 조직 관리

| 방법 | 설정 |
| --- | --- |
| Settings 파일 | `"disableArtifact": true` |
| 환경변수 | `CLAUDE_CODE_DISABLE_ARTIFACT=1` |
| 권한 규칙 | `permissions.deny`에 `Artifact` 추가 |

Team·Enterprise Owner는 claude.ai admin settings(**Settings > Claude Code > Capabilities**)에서 조직 전체 on/off, Enterprise RBAC은 역할별로도 제한 가능. 보관 기간은 **Settings > Data & privacy controls**에서 비공개/공유 버전 각각 설정. 발행·공유·삭제는 `claude_artifact_*` 감사 로그 이벤트로 기록되고, Compliance API(`GET/DELETE /v1/compliance/code/artifacts`)로 목록·조회·삭제 가능. 조직이 아웃바운드를 제한하면 뷰어가 로드하는 `*.claudeusercontent.com` 오리진을 `claude.ai`와 함께 allowlist에 추가([[18 보안과 샌드박스]] network-config 참고).

## 관련 리소스

웹 클라우드 세션과 함께 쓰는 인접 기능 (대부분 [[13 자동화와 스케줄링]]에서 상세히 다룸):

- **Ultraplan**: 클라우드 세션에서 계획을 초안하고 브라우저에서 리뷰
- **Ultrareview**: 클라우드 샌드박스에서 멀티 에이전트 심층 코드 리뷰
- **Routines**: 스케줄·API 호출·GitHub 이벤트로 작업 자동화 (`/schedule`)
- **Hooks**: 세션 라이프사이클 이벤트에 스크립트 실행 → [[09 훅]]
- **Settings**: 전체 설정 옵션 → [[04 설정]]
- **GitHub Enterprise Server**: Team/Enterprise 자체 호스팅 지원 → [[16 CI-CD와 팀 통합]]
