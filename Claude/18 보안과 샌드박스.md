# 18 보안과 샌드박스

Claude Code의 보안 모델, 샌드박스 격리(Bash 단위 → 컨테이너 → VM), 보안 가이던스 플러그인, 데이터 사용·보존 정책, 그리고 법적·컴플라이언스 사항을 한 곳에 정리한 실전 레퍼런스다. 허브는 [[Claude]], 권한 세부는 [[05 권한]], 설정 키는 [[04 설정]]을 참고하라.

> [!note] 이 노트의 큰 그림
> 보안은 **방어 심층(defense in depth)** 으로 설계되어 있다. (1) 권한 시스템([[05 권한]])이 "어떤 도구가 실행될지"를 결정하고, (2) 샌드박스가 "실행된 명령이 무엇에 접근할 수 있는지"를 OS 레벨에서 제한하며, (3) 보안 가이던스 플러그인이 "Claude가 작성한 코드의 취약점"을 잡고, (4) 데이터/컴플라이언스 정책이 "전송·보존되는 데이터"를 규정한다. 어느 하나도 완전한 격리가 아니다.

## 1. 보안 모델 개요 (security)

### 1.1 보안 토대와 권한 기반 아키텍처

Claude Code는 Anthropic의 보안 프로그램에 따라 설계되었으며 SOC 2 Type 2, ISO 27001 등의 인증을 [Anthropic Trust Center](https://trust.anthropic.com)에서 확인할 수 있다.

핵심은 **기본 읽기 전용 권한(read-only by default)** 이다. 파일 편집·테스트 실행·명령 실행 등 추가 동작이 필요하면 Claude Code가 명시적 승인을 요청한다. Bash 명령은 실행 전에 승인을 요구하는 것이 기본이다. 자세한 권한 구성은 [[05 권한]] 참고.

### 1.2 내장 보호 장치 (Built-in protections)

| 보호 장치 | 동작 |
|---|---|
| Sandboxed bash tool | `/sandbox`로 파일시스템·네트워크 격리. 권한 프롬프트를 줄이면서 자율 작업 영역을 정의 (아래 3절) |
| Write access restriction | Claude Code는 **시작한 폴더와 그 하위 폴더에만** 쓰기 가능. 상위 디렉터리 수정은 명시적 권한 필요. 읽기는 작업 디렉터리 밖도 가능(시스템 라이브러리·의존성 접근용) |
| Prompt fatigue mitigation | 자주 쓰는 안전한 명령을 per-user / per-codebase / per-organization 단위로 allowlist |
| Accept Edits mode | 파일 편집과 `mkdir`, `touch`, `rm`, `mv`, `cp`, `sed` 같은 고정 집합의 파일시스템 Bash 명령을 작업 디렉터리 경로에 한해 자동 승인. 그 외 명령·범위 밖 경로는 여전히 프롬프트 |

> [!warning] 사용자 책임
> Claude Code는 **당신이 부여한 권한만** 가진다. 제안된 코드·명령의 안전성을 승인 전에 검토할 책임은 사용자에게 있다.

### 1.3 프롬프트 인젝션 방어 (Prompt injection)

프롬프트 인젝션은 악성 텍스트를 삽입해 AI의 지시를 덮어쓰거나 조작하는 기법이다. 핵심 보호:

- **Permission system**: 민감한 작업은 명시적 승인 필요
- **Context-aware analysis**: 전체 요청을 분석해 유해 지시 탐지
- **Input sanitization**: 사용자 입력 처리로 명령 인젝션 방지
- **Command blocklist**: `curl`, `wget`처럼 웹에서 임의 콘텐츠를 가져오는 위험 명령을 기본 차단. 명시적으로 허용할 때는 [권한 패턴의 한계](https://code.claude.com/docs/en/permissions)를 인지할 것

추가 안전장치:

| 안전장치 | 설명 |
|---|---|
| Network request approval | 네트워크 요청 도구는 기본적으로 사용자 승인 필요 |
| Isolated context windows | Web fetch는 별도 컨텍스트 윈도우 사용 → 악성 프롬프트 주입 방지 |
| Trust verification | 처음 실행하는 코드베이스·신규 MCP 서버는 신뢰 확인 필요 |
| Command injection detection | 의심스러운 bash 명령은 이전에 allowlist되어 있어도 수동 승인 요구 |
| Fail-closed matching | 매칭되지 않는 명령은 기본적으로 수동 승인 요구 |
| Natural language descriptions | 복잡한 bash 명령에 설명 첨부 |
| Secure credential storage | API 키·토큰은 암호화 저장 ([[04 설정]] / 인증) |

신뢰 확인 관련 주의:
- `-p` 플래그로 비대화식 실행 시 신뢰 확인은 비활성화된다. 단 `--worktree`는 예외로, 디렉터리에 대한 신뢰 수락이 여전히 필요하다.
- **홈 디렉터리에서 직접 실행**하면 신뢰 수락이 디스크에 기록되지 않고 현재 세션에만 유지되어 매 실행마다 프롬프트가 다시 뜬다. 이를 영구화하는 설정은 없으니, 프로젝트 하위 디렉터리에서 시작하라(거기서는 디렉터리별로 저장됨).

> [!warning] Windows WebDAV 보안 위험
> Windows에서 Claude Code 실행 시 WebDAV 활성화나 `\\*` 같은 경로 접근 허용을 권장하지 않는다. WebDAV는 Microsoft가 보안 위험으로 deprecated했으며, 활성화 시 권한 시스템을 우회해 원격 호스트로 네트워크 요청을 트리거할 수 있다.

**신뢰할 수 없는 콘텐츠 작업 베스트 프랙티스**:
1. 승인 전 제안 명령 검토
2. 신뢰할 수 없는 콘텐츠를 Claude에 직접 파이프하지 말 것
3. 중요 파일의 변경 제안 검증
4. 외부 웹 서비스 상호작용 시 VM에서 스크립트·도구 호출 실행
5. 의심스러운 동작은 `/feedback`로 보고

### 1.4 MCP / IDE / 클라우드 실행 보안

- **MCP 보안**: 허용 MCP 서버 목록은 소스 코드(체크인된 설정)에 구성한다. 직접 작성하거나 신뢰하는 제공자의 서버를 쓰라. Anthropic은 Directory 등재 전 listing criteria로 커넥터를 검토하지만 **MCP 서버를 보안 감사하거나 관리하지 않는다**. MCP 일반은 [[10 MCP]] 참고.
- **IDE 보안**: VS Code에서의 보안·프라이버시는 [[14 IDE와 데스크톱]] 참고.
- **클라우드 실행 보안** ([[15 웹과 모바일]]):
  - Isolated VM: 각 클라우드 세션은 Anthropic 관리 격리 VM에서 실행
  - Network access controls: 기본 제한, 비활성화 또는 특정 도메인만 허용 설정 가능
  - Credential protection: 보안 프록시가 샌드박스 내부에서 scoped credential을 사용하고 실제 GitHub 토큰으로 변환
  - Branch restrictions: Git push는 현재 작업 브랜치로 제한
  - Audit logging: 모든 작업 로깅
  - Automatic cleanup: 세션 완료 후 자동 종료
  - **Remote Control** 세션은 다르게 동작한다: 웹 인터페이스가 로컬 머신의 Claude Code 프로세스에 연결되어, 코드 실행·파일 접근이 모두 로컬에 머문다. 클라우드 VM·샌드박스가 관여하지 않고, 짧은 수명의 좁게 스코프된 자격증명 여러 개를 사용해 단일 자격증명 유출의 영향 범위를 제한한다. ([[13 자동화와 스케줄링]])

### 1.5 보안 베스트 프랙티스 & 취약점 보고

**민감한 코드 작업**: 변경 승인 전 검토 / 민감 저장소에 프로젝트별 권한 설정 / dev container로 추가 격리 / `/permissions`로 정기 권한 감사.

**팀 보안**: managed settings로 조직 표준 강제 / 승인된 권한 구성을 버전 관리로 공유 / 팀원 교육 / OpenTelemetry 메트릭으로 모니터링 ([[16 CI-CD와 팀 통합]]) / `ConfigChange` 훅으로 세션 중 설정 변경 감사·차단 ([[09 훅]]).

**취약점 보고**: 공개 금지 → [HackerOne 프로그램](https://hackerone.com/4f1f16ba-10d3-4d09-9ecc-c721aad90f24/embedded_submissions/new)으로 보고 → 상세 재현 단계 포함 → 공개 전 수정 시간 허용.

## 2. 보안 가이던스 플러그인 (security-guidance)

Claude가 자기 코드 변경을 작성하는 동안 일반 취약점(인젝션, 안전하지 않은 역직렬화, 안전하지 않은 DOM API 등)을 스스로 리뷰하고 같은 세션에서 고치게 하는 플러그인이다. PR에 도달하기 전 문제를 줄여 인간 리뷰 부담을 낮춘다. 설치 후 자동 실행되며 호출할 명령이 없다. 플러그인 일반은 [[11 플러그인]], 훅 기반 동작은 [[09 훅]] 참고.

### 2.1 전제 조건 (Prerequisites)

- Claude Code CLI **2.1.144 이상**
- `PATH`에 Python 3.8 이상 (`python3` → `python` → `py -3` 순으로 시도)
- 작업 디렉터리가 git 저장소 (end-of-turn·commit 리뷰는 git 상태와 diff함. 저장소 밖이면 조용히 skip. per-edit 패턴 체크는 어디서나 동작)
- 최초 실행 시 `~/.claude/security/` 아래 가상환경을 만들고 Claude Agent SDK를 설치(`pip`·네트워크 필요). 실패 시 commit 리뷰는 single-shot 리뷰로 폴백. Windows는 가상환경 단계를 건너뛴다.

### 2.2 설치 & 활성화

```text
/plugin install security-guidance@claude-plugins-official
```

설치 시 scope를 묻는다. **user scope**를 선택하면 이 머신의 모든 새 로컬 세션에서 로드된다. 마켓플레이스를 못 찾으면 먼저 `/plugin marketplace add anthropics/claude-plugins-official` 실행.

현재 세션에 즉시 적용:
```text
/reload-plugins
```

**클라우드 세션·공유 저장소에서 활성화** — user-scoped 플러그인은 [[15 웹과 모바일]] 세션으로 이어지지 않으므로, 프로젝트 체크인 설정에 선언한다:
```json
// .claude/settings.json
{
  "enabledPlugins": {
    "security-guidance@claude-plugins-official": true
  }
}
```
관리자는 managed settings의 `enabledPlugins`로 조직 전체 활성화 가능 ([[04 설정]]).

### 2.3 무엇을 검사하나 — 3개 레이어

| 레이어 | 시점 | 깊이 / 비용 |
|---|---|---|
| On each file edit | Claude가 파일에 쓸 때 | 빠른 패턴 매칭, 모델 호출 없음 → 비용 0 |
| At the end of each turn | 각 턴 종료 후 | 백그라운드 모델 리뷰 (해당 턴의 git diff 전체) |
| On each commit/push Claude makes | Claude가 `git commit`/`git push` 실행 시 | 더 깊은 agentic 리뷰 (주변 코드 읽음) |

**On each file edit** — 새 콘텐츠를 알려진 위험 패턴으로 스캔. 예시 카테고리:
- 동적 코드 실행: `eval(`, `new Function`, `os.system`, `child_process.exec`
- 안전하지 않은 역직렬화: `pickle`
- DOM 인젝션: `dangerouslySetInnerHTML`, `.innerHTML =`, `document.write`
- 워크플로 파일: `.github/workflows/` 아래 편집 (저장소 레벨 권한 부여 가능)

경고는 패턴·파일·세션당 한 번만 발생해 대화를 도배하지 않는다.

**At the end of each turn** — 한 턴(메시지→작업→응답)이 끝나면 작업 트리의 git diff(편집 도구·Bash·서브에이전트 변경 포함)를 별도 Claude 리뷰에 보낸다. 백그라운드 실행이라 응답이 지연되지 않는다. 문자열 매칭으로 못 잡는 것(인가 우회, IDOR, 인젝션, SSRF, 약한 암호)을 잡는다. 턴당 **최대 30개 변경 파일**, **연속 최대 3회**까지.

**On each commit/push** — Claude가 Bash 도구로 commit/push할 때만 발생. 자기 셸이나 `!` 셸 이스케이프로 한 커밋은 리뷰되지 않는다. **롤링 시간당 20회** 상한. end-of-turn 리뷰와 중복되면 재프롬프트하지 않음.

> [!note] 리뷰 독립성과 한계
> 코드를 작성한 Claude 인스턴스에게 자기 채점을 시키지 않는다. per-edit는 결정론적 문자열 매칭(모델 없음), end-of-turn·commit 리뷰는 **새 컨텍스트**의 별도 Claude 호출로 diff에서 시작해 문제만 찾도록 지시받는다. **어느 레이어도 쓰기·커밋을 차단하지 않으며** 리뷰 모델은 문제를 놓칠 수 있다. 방어 심층의 한 레이어로만 취급하라.

### 2.4 자체 규칙 추가 (Add your own rules)

두 확장 지점이 있고 모두 **추가만** 가능(내장 체크 비활성화 불가).

**모델 기반 리뷰용 가이던스** — `.claude/claude-security-guidance.md`에 위협 모델·체크리스트를 평문으로 작성:
```markdown
# Security guidance for this repo
- Do not log `customer_id` or `account_number` at INFO level or above.
- All routes under `/admin` must call `require_role("admin")` before any database read.
- Use `crypto.timingSafeEqual` for token comparison instead of `===`.
```
이는 결정론적 가드레일이 아니라 리뷰어 가이던스다. "취약점 클래스를 무시하라"는 규칙은 그 findings를 억제하지 못한다(추가 전용). 강한 강제는 [편집 차단 훅](https://code.claude.com/docs/en/hooks-guide)이나 CI 체크와 병행하라 ([[09 훅]]).

**per-edit 커스텀 패턴** — `.claude/security-patterns.yaml`:
```yaml
patterns:
  - rule_name: internal_api_key
    substrings: ["sk_live_", "AKIA"]
    reminder: "Hardcoded API key prefix. Load credentials from the secret manager."
  - rule_name: tenant_unfiltered_query
    regex: "\\.objects\\.all\\(\\)"
    paths: ["**/src/tenants/**"]
    reminder: "Multi-tenant code must filter by org_id."
```

| 필드 | 타입 | 설명 |
|---|---|---|
| `rule_name` | string | 경고에 표시되는 식별자 |
| `reminder` | string | Claude 컨텍스트에 추가되는 경고 텍스트, 최대 1 KB |
| `regex` | string | 편집된 콘텐츠에 매칭되는 Python 정규식 |
| `substrings` | list | 리터럴 substring; `regex`와 둘 중 하나 제공 |
| `paths` | list | 선택 glob; 매칭 파일에만 적용. 전체 경로에 매칭되므로 프로젝트 상대 패턴은 `**/`로 시작 |
| `exclude_paths` | list | 건너뛸 glob 패턴 |

`.yml`, `.json` 형식도 같은 스키마로 읽는다. JSON은 모든 Python에서 동작하나 YAML은 PyYAML이 importable해야 한다(플러그인이 설치해주지 않음). 최대 50개 규칙 로드, catastrophic backtracking 우려 정규식은 skip.

**규칙 파일 조회 위치**:

| Scope | 경로 | 비고 |
|---|---|---|
| User | `~/.claude/claude-security-guidance.md` | 머신의 모든 프로젝트에 적용 |
| Project | `.claude/claude-security-guidance.md` | 저장소에 체크인 |
| Project local | `.claude/claude-security-guidance.local.md` | gitignore, 개인 오버라이드 |

존재하는 모든 위치를 로드·연결하며 가이던스 파일은 합산 **8 KB** 상한. `security-patterns.yaml`도 동일 경로 적용.

### 2.5 비용 / 비활성화 / 통합

**비용**: per-edit 체크는 비용 0. end-of-turn·commit 리뷰는 일반 Claude 요청처럼 [usage]([[19 비용과 성능]])에 계상. commit 리뷰는 agentic이라 커밋당 여러 턴(시간당 20회 상한). 두 모델 리뷰 모두 기본 **Claude Opus 4.7** 사용. `SECURITY_REVIEW_MODEL`(end-of-turn), `SG_AGENTIC_MODEL`(commit)로 변경. 모든 플랜에서 사용 가능.

**레이어별 비활성화 환경변수**:

| 변수 | 효과 |
|---|---|
| `ENABLE_PATTERN_RULES=0` | per-edit 패턴 체크 비활성화 |
| `ENABLE_STOP_REVIEW=0` | end-of-turn diff 리뷰 비활성화 |
| `ENABLE_COMMIT_REVIEW=0` | commit/push 리뷰 비활성화 |
| `ENABLE_CODE_SECURITY_REVIEW=0` | 모든 모델 기반 리뷰 한 번에 비활성화 |
| `SECURITY_GUIDANCE_DISABLE=1` | 제거 없이 플러그인 전체 비활성화 |

```text
/plugin disable security-guidance@claude-plugins-official   # user scope에서 일시중지
/plugin uninstall security-guidance@claude-plugins-official # 제거
```
프로젝트 `.claude/settings.json`로 활성화된 경우 `/plugin disable`은 체크인 파일 대신 `.claude/settings.local.json`에 오버라이드를 써서 팀원에게 영향 없이 본인만 끈다. managed settings로 활성화된 경우 관리자만 비활성화 가능.

**훅 통합** — 플러그인은 전부 [[09 훅]] 위에 구현:

| Hook event | 용도 |
|---|---|
| `SessionStart` | Python 환경 부트스트랩 |
| `UserPromptSubmit` | end-of-turn 리뷰가 diff할 작업 트리 baseline 캡처 |
| `PostToolUse` on `Edit`/`Write`/`NotebookEdit` | per-edit 패턴 매칭 |
| `Stop` | 백그라운드 end-of-turn diff 리뷰 |
| `PostToolUse` on `Bash` (git commit/push 필터) | 백그라운드 commit/push 리뷰 |

### 2.6 다른 보안 도구와의 조합 (defense in depth)

| 단계 | 도구 | 범위 |
|---|---|---|
| In session | Security guidance plugin | Claude가 작성한 코드의 일반 취약점, 같은 세션에서 수정 |
| On demand | `/security-review` | 현재 브랜치 1회 보안 패스 |
| On pull request | Code Review (Team/Enterprise) | 멀티 에이전트 정확성·보안 리뷰, 전체 코드베이스 컨텍스트 ([[16 CI-CD와 팀 통합]]) |
| In CI | 기존 정적 분석·의존성 스캐너 | 언어별 규칙·공급망 검사 |

**트러블슈팅**: 런타임 진단은 `~/.claude/security/log.txt`. 조용히 skip하는 흔한 이유 — git 저장소 아님 / Anthropic 인증 없음(per-edit만 동작) / `security-patterns.yaml`이 있으나 PyYAML 없음(`.json` 사용).

## 3. 샌드박스 Bash 도구 (sandboxing)

Bash 샌드박스는 대부분의 셸 명령을 권한 질문 없이 실행하게 해준다. 명령마다 승인하는 대신 **어떤 파일·네트워크 도메인에 접근 가능한지**를 정의하면, OS가 모든 Bash 명령과 그 자식 프로세스에 대해 경계를 강제한다. 권한·권한 모드는 [[05 권한]] 참고.

### 3.1 시작하기 (`/sandbox`)

macOS·Linux·WSL2에서 동작. **네이티브 Windows는 미지원** → WSL2 안에서 실행. macOS는 내장 Seatbelt를 쓰므로 설치 불필요.

```text
/sandbox
```
패널 3개 탭:
- **Mode**: 샌드박스 명령 승인 방식 선택
- **Overrides**: 샌드박스에서 실패한 명령이 비샌드박스로 폴백 가능한지 (`allowUnsandboxedCommands` 설정)
- **Config**: 해석된 샌드박스 설정 확인

패널에서 모드 선택 시 `.claude/settings.local.json`(현재 프로젝트, git 비체크인)에 기록. 모든 프로젝트에 적용하려면 `~/.claude/settings.json`에 `sandbox.enabled: true`. 조직 전체 강제는 managed settings(3.6절).

> [!warning] 폴백 동작
> 기본적으로 의존성 누락·미지원 플랫폼으로 샌드박스가 시작 못 하면 경고 후 비샌드박스로 실행한다. 이를 hard failure로 만들려면 `sandbox.failIfUnavailable: true` (보안 게이트가 필요한 managed 배포용).

### 3.2 Linux·WSL2 셋업

두 패키지 필요:
- [`bubblewrap`](https://github.com/containers/bubblewrap): 파일시스템 격리를 강제하는 비특권 샌드박싱 도구
- [`socat`](http://www.dest-unreach.org/socat/): 트래픽을 샌드박스 프록시로 라우팅하는 릴레이

```bash
# Ubuntu/Debian
sudo apt-get install bubblewrap socat
# Fedora
sudo dnf install bubblewrap socat
```
`/sandbox`의 Dependencies 탭이 `ripgrep`, `bubblewrap`, `socat`, seccomp 필터 가용성을 표시한다. ripgrep은 네이티브 바이너리에 번들. seccomp 필터(Unix 도메인 소켓 차단, 선택)는 누락 시 `npm install -g @anthropic-ai/sandbox-runtime`로 설치. 의존성 체크는 시작 시 실행되므로 설치 후 재시작 필요.

**Ubuntu 24.04+** — 기본 AppArmor 정책이 bubblewrap의 user namespace 생성을 막는다. `sysctl kernel.apparmor_restrict_unprivileged_userns`가 `1`이면 프로필 추가:
```bash
sudo tee /etc/apparmor.d/bwrap > /dev/null <<'EOF'
abi <abi/4.0>,
include <tunables/global>

profile bwrap /usr/bin/bwrap flags=(unconfined) {
  userns,
  include if exists <local/bwrap>
}
EOF
sudo systemctl reload apparmor
```

**WSL2** — `wsl -l -v`로 버전 확인. `Sandboxing requires WSL2`가 보이면 WSL1이니 업그레이드. WSL2에서 샌드박스 명령은 `cmd.exe`, `powershell.exe`, `/mnt/c/` 등 Windows 바이너리를 실행할 수 없다(Unix 소켓 핸드오프를 샌드박스가 차단) → 필요하면 `excludedCommands`에 추가.

### 3.3 샌드박스 모드 (Sandbox modes)

| 모드 | 동작 |
|---|---|
| **Auto-allow** | Bash 명령이 샌드박스 안에서 자동 허용(프롬프트 없음). 샌드박스 불가 명령(미허용 호스트 네트워크 등)은 일반 권한 플로우로 폴백 |
| **Regular permissions** | 모든 Bash 명령이 샌드박스되더라도 일반 권한 플로우 통과. 통제 ↑ 승인 ↑ |

auto-allow에서도 적용되는 것: explicit deny rules는 항상 존중 / `rm`·`rmdir`가 `/`·홈·중요 시스템 경로 대상이면 여전히 프롬프트 / 폴백 명령에 ask rules 적용.

**escape hatch** — 샌드박스 제약으로 명령이 실패하면 Claude가 실패를 분석해 `dangerouslyDisableSandbox` 파라미터로 재시도할 수 있다(샌드박스 밖 실행 → 일반 권한 플로우 통과·승인 필요). `"allowUnsandboxedCommands": false`로 비활성화하면(Overrides 탭의 **Strict sandbox mode**) 이 파라미터가 완전 무시되고 모든 명령은 샌드박스되거나 `excludedCommands`에 명시되어야 한다.

> [!info] auto-allow는 권한 모드와 독립
> "accept edits" 모드가 아니어도 auto-allow가 켜져 있으면 샌드박스 Bash 명령은 자동 실행된다. 즉 샌드박스 경계 내 파일 수정 Bash 명령은 파일 편집 도구가 보통 승인을 요구하는 상황에서도 프롬프트 없이 실행된다.

### 3.4 샌드박싱 구성 (`settings.json`)

기본적으로 샌드박스 명령은 현재 작업 디렉터리에만 쓸 수 있다. `kubectl`, `terraform`, `npm` 등이 프로젝트 밖에 써야 하면 `sandbox.filesystem.allowWrite`로 경로 부여:
```json
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["~/.kube", "/tmp/build"]
    }
  }
}
```
OS 레벨 강제라 자식 프로세스까지 적용된다. 도구 전체를 `excludedCommands`로 제외하는 것보다 권장된다. 같은 filesystem 배열이 여러 scope에 정의되면 **합산**된다.

**경로 prefix 해석**:

| Prefix | 의미 | 예시 |
|---|---|---|
| `/` | 파일시스템 루트 기준 절대 경로 | `/tmp/build` → `/tmp/build` |
| `~/` | 홈 디렉터리 기준 | `~/.kube` → `$HOME/.kube` |
| `./` 또는 prefix 없음 | 프로젝트 설정이면 프로젝트 루트, user 설정이면 `~/.claude` 기준 | `.claude/settings.json`의 `./output` → `<project-root>/output` |

> [!note] 권한 규칙과 다른 경로 문법
> Read/Edit 권한 규칙은 절대 `//path`, 프로젝트 상대 `/path`를 쓰지만, 샌드박스 filesystem 경로는 표준 관례(`/tmp/build`가 절대)를 쓴다. ([[05 권한]])

읽기 차단 후 프로젝트만 재허용 (프로젝트 `.claude/settings.json`에 둘 때만 `.`이 프로젝트 루트로 해석됨):
```json
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "denyRead": ["~/"],
      "allowRead": ["."]
    }
  }
}
```

### 3.5 샌드박싱 동작 원리

**파일시스템 격리**:
- 기본 쓰기: 현재 작업 디렉터리와 하위
- 기본 읽기: 거부된 디렉터리 제외 컴퓨터 전체. **주의 — 기본값은 `~/.aws/credentials`, `~/.ssh/` 같은 자격증명 파일도 읽을 수 있다.** 차단하려면 `denyRead`에 추가
- 차단: 작업 디렉터리 밖 수정 불가(`~/.bashrc`, `/bin/` 등)
- Git worktrees: linked worktree일 때 메인 저장소의 공유 `.git`에 쓰기 허용(`git commit`이 ref·index 갱신). `hooks/`·`config` 쓰기는 여전히 거부 ([[02 CLI 레퍼런스]] worktree)

**네트워크 격리** — 샌드박스 밖 프록시 서버로 제어:
- Domain restrictions: 사전 허용 도메인 없음. 첫 도메인 필요 시 프롬프트. `allowedDomains`로 사전 허용
- Managed lockdown: `allowManagedDomainsOnly` 설정 시 미허용 도메인은 프롬프트 없이 자동 차단, managed의 `allowedDomains`만 존중
- Custom proxy: 고급 사용자는 커스텀 규칙 구현 가능

> 내장 프록시는 요청 hostname 기반으로 allowlist를 강제하며 **TLS 트래픽을 종료·검사하지 않는다**(3.7 보안 한계 참고).

**OS 레벨 강제**: macOS → Seatbelt, Linux → bubblewrap, WSL2 → bubblewrap(WSL1 미지원). 같은 primitive가 standalone [`@anthropic-ai/sandbox-runtime`](https://github.com/anthropic-experimental/sandbox-runtime) 패키지로도 제공되어 Claude Code 프로세스 전체를 감쌀 수 있다(4절).

### 3.6 권한·권한 모드와의 관계

| 레이어 | 무엇을 제어 | 평가 시점 |
|---|---|---|
| Permission rules | Claude Code가 어떤 도구를 쓸 수 있는지 (모든 도구) | 도구 실행 전 |
| Sandboxing | Bash 명령·자식 프로세스가 파일시스템·네트워크에서 무엇에 접근하는지 | OS가 실행 중 프로세스에 강제 |

샌드박스는 모델이 무엇을 실행하든, 허용된 명령이 이름보다 더 많은 일을 해도 경계를 유지한다.

**관련 설정·규칙 매핑**:

| 설정/규칙 | 동작 |
|---|---|
| `sandbox.filesystem.allowWrite` | 작업 디렉터리 밖 경로에 subprocess 쓰기 허용 |
| `sandbox.filesystem.denyWrite` / `denyRead` | 특정 경로 접근 차단 |
| `sandbox.filesystem.allowRead` | `denyRead` 영역 내 특정 경로 읽기 재허용 |
| `Edit` allow rules | 특정 경로 쓰기 부여 (`allowWrite`와 동일 효과) |
| `Read`/`Edit` deny rules | 특정 파일·디렉터리 접근 차단 |
| `WebFetch` allow/deny rules | 도메인 접근 제어 |
| Sandbox `allowedDomains` | Bash 명령이 도달 가능한 도메인 |
| Sandbox `deniedDomains` | `allowedDomains` 와일드카드로 허용될 도메인도 차단 |

`/sandbox`는 권한 모드가 아니다. 권한 모드는 도구 호출 실행 여부·프롬프트 여부를 결정한다:

| | 제어 대상 | 프롬프트 대체 |
|---|---|---|
| `/sandbox` | Bash 명령 실행 후 접근 범위 | auto-allow 모드의 샌드박스 경계 자체 |
| Auto mode | 각 도구 호출 실행 여부 | 동작을 검토하는 classifier |
| `--dangerously-skip-permissions` | 각 도구 호출 실행 여부 | 없음. Protected path 체크도 skip(오직 `/`·홈 삭제만 프롬프트) |

샌드박스의 auto-allow와 권한 모드의 auto mode는 독립적이며 조합 가능하다.

### 3.7 조직 강제 & 한계

**managed settings로 강제** ([[16 CI-CD와 팀 통합]] / [[04 설정]]):
```json
{
  "sandbox": {
    "enabled": true,
    "failIfUnavailable": true,
    "allowUnsandboxedCommands": false
  }
}
```
boolean 키(`enabled`, `failIfUnavailable`)는 managed 값이 로컬을 무시한다. 배열 키(`excludedCommands`, `allowRead`)는 합산되어 개발자가 정책을 넓힐 수 있다 → `allowManagedReadPathsOnly: true`(managed `allowRead`만 존중), `allowManagedDomainsOnly`(네트워크 잠금)로 제어. `excludedCommands`는 managed-only 잠금이 없으니 목록을 좁게 유지. 자격증명 디렉터리(`~/.aws`, `~/.ssh`)에 `denyRead` 추가 권장.

**커스텀 프록시** (HTTPS 검사·필터·로깅):
```json
{
  "sandbox": {
    "network": {
      "httpProxyPort": 8080,
      "socksProxyPort": 8081
    }
  }
}
```

**트러블슈팅**:
- host-not-allowed 오류 → 프롬프트에서 허용하면 호스트가 allowlist에 추가됨
- `jest` 멈춤/실패 → `watchman` 비호환, `jest --no-watchman`
- macOS Go CLI(`gh`, `gcloud`, `terraform`) TLS 검증 실패 → `excludedCommands`에 추가. MITM 프록시+커스텀 CA 사용 시 `enableWeakerNetworkIsolation: true`
- `docker` 명령 실패 → 비호환, `docker *`를 `excludedCommands`에 추가
- 컨테이너 안에서 bubblewrap 시작 실패 → 비특권 컨테이너는 `/proc`을 mount 못 함. `enableWeakerNestedSandbox: true`(외부 컨테이너가 이미 격리 제공할 때만)
- Linux seccomp 필터 누락 → `npm install -g @anthropic-ai/sandbox-runtime`
- `--dangerously-skip-permissions`가 root에서 실패 → root·sudo에서 차단됨. 인식된 샌드박스 안에서는 자동 skip. 컨테이너 자율 실행은 비root 사용자로 도는 [dev container](#5-개발-컨테이너-devcontainer) 사용

**보안 한계** (하드 컨트롤로 의존 전 검토):
- Network filtering: TLS 검사를 안 함 → `github.com` 같은 넓은 도메인 허용은 [domain fronting]으로 데이터 유출 경로가 될 수 있다. 강한 보장이 필요하면 TLS 종료·검사하는 커스텀 프록시 + 내부 CA 인증서 설치
- Unix sockets로 권한 상승: `allowUnixSockets`로 `/var/run/docker.sock` 허용 시 호스트 접근 부여 효과
- 파일시스템 권한 상승: `$PATH` 실행 파일 디렉터리·시스템 설정·`.bashrc`/`.zshrc`에 쓰기 허용은 다른 보안 컨텍스트에서 코드 실행 유발
- Linux 샌드박스 강도: `enableWeakerNestedSandbox`는 보안을 상당히 약화 — 추가 격리가 따로 강제될 때만 사용
- Settings files protected: 샌드박스는 모든 scope의 `settings.json`과 managed 디렉터리 쓰기를 자동 거부 → 샌드박스 명령이 자기 정책을 못 고침

**범위(Scope)** — 샌드박스는 Bash 서브프로세스만 격리:
- 내장 파일 도구(Read/Edit/Write)는 권한 시스템 직접 사용 ([[06 내장 도구 레퍼런스]])
- Computer use는 실제 데스크톱에서 실행, 앱별 프롬프트로 게이트 ([[21 브라우저와 컴퓨터 사용]])
- 환경변수: 샌드박스 Bash는 부모 환경(자격증명 포함) 상속. Anthropic·클라우드 자격증명 제거는 `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB`
- 서브에이전트: 부모와 같은 프로세스·샌드박스 구성 사용 ([[08 서브에이전트와 에이전트 팀]])

> [!warning] 파일시스템 + 네트워크 둘 다 필요
> 네트워크 격리 없으면 손상된 에이전트가 SSH 키를 유출하고, 파일시스템 격리 없으면 시스템 리소스를 backdoor해 네트워크 접근을 얻을 수 있다. 기본값을 넓힐 때 한쪽 제한이 다른 쪽을 무력화하지 않는지 확인하라.

## 4. 샌드박스 환경 선택 (sandbox-environments)

Claude Code 격리는 경량 per-command 샌드박스부터 완전 분리 VM까지 여러 종류가 있다. 더 적은 권한 프롬프트로 작업하거나, 무인 실행하거나, 완전히 신뢰하지 않는 코드를 다룰 때 중요하다.

### 4.1 격리 접근법 비교

| 접근법 | 격리 대상 | Docker 필요 | 셋업 노력 |
|---|---|---|---|
| Sandboxed Bash tool | Bash 명령과 자식 프로세스 | No | macOS 최소; Linux/WSL2 낮음 |
| Sandbox runtime | Claude Code 프로세스 전체 (파일 도구·MCP·훅 포함) | No | 낮음 |
| Dev container | 전체 개발 환경 | Yes | 중간 |
| Custom container | 전체 개발 환경 | Yes | 중간~높음 |
| Virtual machine | 전체 OS | No | 높음 |
| Claude Code on the web | 전체 OS, Anthropic 호스팅 | No | 없음; Claude 구독+GitHub 필요 |

앞의 두 개는 컨테이너 없이 호스트 OS에서 실행된다. 나머지는 Claude Code를 컨테이너·VM 안에 둔다. **Bash 샌드박스만 Bash를 제한**하고, 나머지 접근법은 프로세스 전체(파일 도구·MCP·훅 포함)를 경계 안에 넣는다.

> [!warning] 격리는 위험을 줄이지만 제거하지 않는다
> 네트워크 egress를 허용하는 어떤 접근법도 에이전트가 읽을 수 있는 데이터를 유출할 수 있고, 프로젝트 디렉터리를 쓰기 가능하게 마운트하면 코드를 수정할 수 있다. 또한 격리는 **모델로 전송되는 것을 바꾸지 않는다** — 프롬프트와 Claude가 읽는 파일은 샌드박스 유무와 무관하게 API로 전송된다(6절 data-usage).

### 4.2 목표별 선택

| 하고 싶은 것 | 시작점 |
|---|---|
| 자기 머신 일상 작업에서 권한 프롬프트 줄이기 | Sandboxed Bash tool (`/sandbox`) |
| `--dangerously-skip-permissions`·auto mode로 무인 작업 | 사전구성 dev container, 임의 컨테이너/VM, 또는 sandbox runtime |
| Bash뿐 아니라 MCP·훅도 격리(Docker 없이) | Sandbox runtime |
| 신뢰할 수 없는 저장소 작업 | 전용 VM, 또는 Claude Code on the web |
| 팀 전체에 샌드박스 환경 표준화 | 사전구성 dev container를 저장소에 복사 |
| 로컬 셋업 없는 기기에서 사용 | Claude Code on the web |
| 조직 모든 개발자에게 격리 강제 | 4.5 조직 강제 |
| 네이티브 Windows 호스트에서 작업 | 컨테이너/VM, 또는 WSL2 안의 Bash 샌드박스 |

### 4.3 격리와 권한 모드의 관계

권한 모드는 도구 호출 실행 여부·프롬프트 여부를 결정하고, 격리는 실행 후 접근 범위를 제한한다. `--dangerously-skip-permissions`는 per-action 검토를 완전히 제거하므로 **격리 경계가 유일한 제한** → 항상 컨테이너·VM·sandbox runtime 안에서 실행해 파일 도구·MCP·훅도 경계 안에 둘 것. auto mode는 classifier(per-action 통제)로 동작을 검토하므로 격리는 무인 실행의 방어 심층을 더할 뿐 필수는 아니다. Bash 샌드박스 단독으로는 어느 모드에서도 완전 무인 실행에 불충분하다. ([[05 권한]])

### 4.4 각 접근법 상세

- **Sandbox runtime** — `@anthropic-ai/sandbox-runtime`가 프로세스 전체를 Seatbelt/bubblewrap 격리로 감싼다. 세션의 모든 도구·훅·MCP 서버를 제약. **베타 리서치 프리뷰**, 설정 형식 변경 가능. 기본 모든 쓰기·네트워크 거부 → `~/.srt-settings.json`(또는 `--settings`)에 프로젝트 디렉터리, `~/.claude`·`~/.claude.json`, 필요 도메인(`api.anthropic.com` 등) 허용:
  ```bash
  npx @anthropic-ai/sandbox-runtime claude
  ```
- **Dev containers** — VS Code 등이 관리하는 Docker 컨테이너에서 실행, 프로젝트를 마운트. `.devcontainer/` 디렉터리로 정의. claude-code 저장소의 예제 dev container는 default-deny iptables 방화벽 포함 → `--dangerously-skip-permissions` 무인 작업 지원. (5절)
- **Custom container** — 임의 Docker/OCI 이미지에 자체 네트워크 정책·마운트·seccomp 프로필. 기존 컨테이너 인프라·CI 러너가 있는 조직에 일반적. 마운트·도달 가능 자격증명·egress 정책을 검토하라. 내부에 Bash 샌드박스 중첩 가능(비특권 컨테이너는 nested-sandbox 설정 필요).
- **Virtual machine** — 자체 커널, 클라우드/microVM은 가상화 하드웨어까지. 가장 강한 분리. 신뢰할 수 없는 코드 평가·커널 레벨 분리 요구 시. Firecracker 같은 microVM, Docker Desktop sandboxes 등.
- **Claude Code on the web** — Anthropic 관리 격리 VM. 네트워크 프록시가 기본 allowlist 강제, 별도 프록시가 GitHub 토큰을 샌드박스 밖에 두고 scoped credential 발급. Claude 구독+GitHub 필요, 저장소를 GitHub에서 클론. ([[15 웹과 모바일]])

### 4.5 조직 전체 격리 강제

- **Built-in Bash sandbox**: Claude Code가 스스로 강제하는 유일한 접근법. managed settings(MDM 또는 Claude.ai server-managed settings)로 `sandbox` 키 전달 (3.7절)
- **Dev containers**: 예제 dev container를 저장소에 커밋해 표준화 → 단 컨테이너를 요구하지 않으므로 **관례일 뿐 강제 경계 아님**. 외부 실행을 막으려면 조직 device management·소프트웨어 allowlisting 사용
- **Custom containers/VMs**: 승인 이미지로 Claude Code 배포 + device management로 외부 설치 방지

## 5. 개발 컨테이너 (devcontainer)

[dev container](https://containers.dev/)는 팀 모두가 동일한 격리 환경을 실행하게 해준다. Claude Code를 그 컨테이너에 설치하면 Claude의 명령은 호스트가 아닌 컨테이너 안에서 실행되고, 프로젝트 파일 편집은 로컬 저장소에 그대로 나타난다. 격리 접근법 비교는 4절, IDE 통합은 [[14 IDE와 데스크톱]] 참고.

> [!warning] dev container의 한계
> `--dangerously-skip-permissions`로 실행 시 dev container는 악성 프로젝트가 컨테이너 내 접근 가능한 모든 것(`~/.claude`에 저장된 Claude Code 자격증명 포함)을 유출하는 것을 막지 못한다. **신뢰하는 저장소에서만** 사용하고 활동을 모니터링하라. 호스트 시크릿(`~/.ssh`·클라우드 자격증명 파일) 마운트를 피하고 저장소 스코프·단기 토큰을 선호하라.

### 5.1 Claude Code를 dev container에 추가

[Claude Code Dev Container Feature](https://github.com/anthropics/devcontainer-features/tree/main/src/claude-code)로 설치한다(VS Code, GitHub Codespaces, JetBrains 등 Dev Containers 스펙 지원 도구).

```json
// .devcontainer/devcontainer.json
{
  "image": "mcr.microsoft.com/devcontainers/base:ubuntu",
  "features": {
    "ghcr.io/anthropics/devcontainer-features/claude-code:1.0": {}
  }
}
```
끝의 `:1.0` 태그는 feature의 install 스크립트를 핀할 뿐 Claude Code 릴리스가 아니다. feature는 최신 Claude Code를 설치하고 기본적으로 컨테이너 안에서 자동 업데이트한다.

이후 **Dev Containers: Rebuild Container**(`Cmd/Ctrl+Shift+P`) → 터미널에서 `claude` 실행 → 인증.

인증 프롬프트는 provider에 따라 다르다:
- **Anthropic**: 브라우저로 Claude/Anthropic Console 계정 로그인
- **Amazon Bedrock / Google Vertex AI / Microsoft Foundry**: 클라우드 자격증명 사용, 브라우저 프롬프트 없음 ([[17 클라우드 프로바이더]])

클라우드 provider는 자격증명 파일 마운트 대신 `containerEnv`·Codespaces secret·workload identity로 환경변수 전달. 브라우저 콜백이 컨테이너에 안 닿으면 코드를 복사해 `Paste code here if prompted`에 붙여넣기.

### 5.2 재빌드 간 인증·설정 유지

기본적으로 컨테이너 홈은 rebuild 시 폐기된다. Claude Code는 인증 토큰·user 설정·세션 기록을 `~/.claude`에 저장하므로 named volume을 마운트:
```json
// devcontainer.json
"mounts": [
  "source=claude-code-config,target=/home/node/.claude,type=volume"
]
```
`/home/node`를 컨테이너 `remoteUser`의 홈으로 교체. `~/.claude` 외에 마운트하면 `CLAUDE_CONFIG_DIR`로 경로 지정. 프로젝트별 격리는 source 이름에 `${devcontainerId}` 변수 사용(`claude-code-config-${devcontainerId}`).

GitHub Codespaces에서 `~/.claude`는 stop/start 간 유지되나 rebuild 시 지워진다 → volume 마운트 적용. codespace 간 인증 유지는 `ANTHROPIC_API_KEY` 또는 `claude setup-token`의 `CLAUDE_CODE_OAUTH_TOKEN`을 Codespaces secret으로 저장.

### 5.3 조직 정책 강제

dev container는 동일 이미지·구성이 모든 머신에서 돌기에 정책 적용에 좋다. Claude Code는 Linux에서 `/etc/claude-code/managed-settings.json`을 읽어 settings 계층 최상위로 적용한다:
```dockerfile
# Dockerfile
RUN mkdir -p /etc/claude-code
COPY managed-settings.json /etc/claude-code/managed-settings.json
```
단 Dockerfile은 저장소에 있어 쓰기 권한자가 변경·제거할 수 있다 → 우회 불가 정책은 server-managed settings나 MDM으로 전달 ([[16 CI-CD와 팀 통합]] / [[04 설정]]).

세션 환경변수는 `containerEnv`에 추가:
```json
// devcontainer.json
"containerEnv": {
  "CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC": "1",
  "DISABLE_AUTOUPDATER": "1"
}
```
feature는 항상 최신 릴리스를 설치하므로 버전 핀이 필요하면 Dockerfile에서 `npm install -g @anthropic-ai/claude-code@X.Y.Z`로 설치하고 `DISABLE_AUTOUPDATER` 설정. MCP 서버는 저장소 루트의 `.mcp.json`(project scope)에 정의 ([[10 MCP]]).

### 5.4 네트워크 egress 제한 & 권한 프롬프트 없이 실행

reference 컨테이너의 [`init-firewall.sh`](https://github.com/anthropics/claude-code/blob/main/.devcontainer/init-firewall.sh)는 Claude Code·개발 도구가 필요로 하는 도메인 외 모든 outbound를 차단한다. 컨테이너 내 방화벽은 추가 권한이 필요해 `runArgs`로 `NET_ADMIN`·`NET_RAW` capability를 추가한다(필수 아님, 자체 네트워크 통제로 대체 가능). 필요 도메인은 network-config의 access requirements와 telemetry services 참고(6절).

컨테이너가 비root 사용자로 실행되고 명령 실행을 컨테이너에 가두므로 무인 운영에 `--dangerously-skip-permissions`를 쓸 수 있다(CLI는 root 실행 시 이 플래그를 거부 → `remoteUser`가 비root인지 확인). 프롬프트 검토 기회가 사라지고 Claude가 마운트된 워크스페이스의 모든 파일(호스트에 직접 나타남)을 수정하고 네트워크 정책이 허용하는 것에 도달할 수 있으니 네트워크 egress 제한과 병행하라. 안전 점검을 끄지 않고 프롬프트를 줄이려면 auto mode가 대안. 엔지니어가 이 플래그를 못 쓰게 하려면 managed settings에 `permissions.disableBypassPermissionsMode: "disable"`. ([[05 권한]])

### 5.5 reference 컨테이너

[`anthropics/claude-code`](https://github.com/anthropics/claude-code/tree/main/.devcontainer)의 예제는 CLI·egress 방화벽·persistent volume·Zsh 셸을 결합한 **동작 예시**(유지되는 base 이미지 아님). 3개 파일:

| 파일 | 용도 |
|---|---|
| `devcontainer.json` | volume 마운트, `runArgs` capability, VS Code 확장, `containerEnv` |
| `Dockerfile` | base 이미지, 개발 도구, Claude Code 설치 |
| `init-firewall.sh` | 허용 도메인 외 모든 outbound 차단 |

VS Code + Dev Containers 확장 설치 → 저장소 클론 → **Reopen in Container** → `claude` 실행. 자기 프로젝트에 쓰려면 `.devcontainer/`를 복사 후 Dockerfile을 자기 toolchain에 맞게 조정.

## 6. 데이터 사용 (data-usage)

### 6.1 데이터 학습 정책 (Data training)

- **Consumer (Free/Pro/Max)**: 데이터를 모델 개선에 쓸지 **선택 가능**. 설정이 켜져 있으면 이들 계정 데이터로 새 모델을 학습한다(Claude Code 사용 포함). [privacy settings](https://claude.ai/settings/privacy)에서 언제든 변경.
- **Commercial (Team/Enterprise/API/3rd-party/Claude Gov)**: Anthropic은 상업 약관 하의 코드·프롬프트로 생성 모델을 **학습하지 않는다**(고객이 Development Partner Program 등으로 데이터 제공을 선택하지 않는 한).
- **Development Partner Program**: 조직 admin이 명시적 opt-in. Anthropic first-party API에서만 가능(Bedrock/Vertex 불가).

### 6.2 피드백 & 세션 품질 설문

- `/feedback`: 보낸 피드백을 제품 개선에 사용 가능. **공유된 transcript는 5년 보존**.
- **세션 품질 설문** ("How is Claude doing this session?"): "Dismiss" 포함 응답은 **평점만 기록**, transcript·입출력·세션 데이터를 수집·저장하지 않는다.
  - 이후 별도 follow-up "Can Anthropic look at your session transcript...": **Yes** → 대화·서브에이전트 transcript·원본 세션 로그 업로드(알려진 API 키·토큰 패턴은 redacted, 소스 코드·파일 내용은 그대로). **공유 transcript는 최대 6개월 보존**. **No**/**Don't ask again** → 전송 없음.
  - ZDR 조직, 제품 피드백이 정책으로 비활성화된 곳, `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` 설정 시 follow-up이 보이지 않는다. 설문 응답·제출된 transcript는 데이터 학습 선호도에 영향을 주지 않고 모델 학습에 쓸 수 없다.

| 설문 제어 | 변수/설정 |
|---|---|
| 설문 비활성화 | `CLAUDE_CODE_DISABLE_FEEDBACK_SURVEY=1` (또는 `DISABLE_TELEMETRY`/`DO_NOT_TRACK`/`CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` 설정 시 자동) |
| OTel 수집기로 다시 켜기 | `CLAUDE_CODE_ENABLE_FEEDBACK_SURVEY_FOR_OTEL=1` (평점만 수집기에 로그, transcript-share follow-up은 비활성 유지) |
| 빈도 조절 | settings의 `feedbackSurveyRate` (0~1 확률) |

### 6.3 데이터 보존 (Data retention)

| 계정 | 보존 기간 |
|---|---|
| Consumer, 모델 개선 허용 | **5년** |
| Consumer, 비허용 | **30일** |
| Commercial 표준 | **30일** |
| Commercial + ZDR | 서버 측 보존 없음 (7절) |
| 로컬 캐시 | `~/.claude/projects/`에 평문 세션 transcript 기본 **30일** (세션 재개용). `cleanupPeriodDays`로 조정 |

Consumer 프라이버시 설정은 [claude.ai/settings/data-privacy-controls](https://claude.ai/settings/data-privacy-controls)에서 변경. Claude Code on the web 세션은 개별 삭제 가능(이벤트 데이터 영구 삭제). 로컬 `.claude` 디렉터리는 [[03 메모리와 컨텍스트]]·[[04 설정]]과 관련.

### 6.4 데이터 흐름 & 암호화

Claude Code는 로컬 실행이지만 LLM 상호작용을 위해 모든 user 프롬프트·모델 출력을 **TLS 1.2+** 로 암호화해 네트워크 전송한다. 대부분 VPN·LLM 프록시와 호환. Remote Control 세션은 모든 실행이 로컬에서 일어나므로 로컬 데이터 흐름을 따른다. (remote Claude Code는 세션을 시작한 저장소에만 접근, 연결만 하고 시작 안 한 저장소엔 접근 안 함)

**저장 시 암호화(at rest)** — 모델 provider에 따라:

| Provider | Encryption at rest |
|---|---|
| Anthropic API | 인프라 레벨 디스크 암호화 (AES-256). ZDR 활성화 시 서버 측 영속 없음 |
| Amazon Bedrock | AES-256 (AWS 관리 키). AWS KMS로 고객 관리 키 가능 |
| Google Cloud Vertex AI | Google 관리 키. CMEK 가능 |
| Microsoft Foundry | Anthropic 인프라로 라우팅, AES-256 디스크 암호화 |

클라우드 실행(Claude Code on the web): 저장소를 격리 VM에 클론, GitHub 인증은 보안 프록시(자격증명이 샌드박스에 안 들어감), 모든 outbound는 보안 프록시 경유(감사 로깅·남용 방지). ([[17 클라우드 프로바이더]])

### 6.5 텔레메트리 서비스 (Telemetry services)

| 서비스 | 내용 | opt-out |
|---|---|---|
| Anthropic 메트릭 | latency·reliability·usage 패턴 (코드·파일 경로 미포함, 전송·저장 암호화) | `DISABLE_TELEMETRY` |
| Sentry 에러 로깅 | 운영 에러 (TLS + 256-bit AES) | `DISABLE_ERROR_REPORTING` |
| `/feedback` | 대화 기록(코드 포함) 전송, 범위 선택 가능(현재 세션 기본 / 24h / 7d), 선택적 공개 GitHub 이슈 생성 | `DISABLE_FEEDBACK_COMMAND=1` |

3rd-party provider(Bedrock/Vertex) 사용 또는 Anthropic 자격증명 없을 때 `/feedback`는 `~/.claude/feedback-bundles/`에 로컬 아카이브로 쓴다(API 키·토큰 패턴 redacted). 보낼 때까지 머신을 떠나지 않는다.

**Provider별 기본 동작** — Bedrock/Vertex/Foundry/Claude Platform on AWS에서는 에러 리포팅·텔레메트리·버그 리포팅이 기본 비활성. 세션 품질 설문·WebFetch 도메인 안전 체크는 예외(provider 무관 실행). `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC`로 비필수 트래픽 일괄 opt-out(단 WebFetch 체크엔 영향 없음).

| 서비스 | Claude API | Vertex/Bedrock/Foundry/AWS |
|---|---|---|
| Anthropic (Metrics) | 기본 on, `DISABLE_TELEMETRY=1` | 기본 off (`CLAUDE_CODE_USE_VERTEX`/`_BEDROCK`/`_FOUNDRY`/`_ANTHROPIC_AWS`=1 필요) |
| Sentry (Errors) | 기본 on, `DISABLE_ERROR_REPORTING=1` | 기본 off |
| `/feedback` | 기본 on, `DISABLE_FEEDBACK_COMMAND=1` | 기본 off |
| Session quality surveys | 기본 on, `CLAUDE_CODE_DISABLE_FEEDBACK_SURVEY=1` | 기본 on, 동일 |
| WebFetch domain safety check | 기본 on, settings의 `skipWebFetchPreflight: true` | 기본 on, 동일 |

모든 환경변수는 `settings.json`에 체크인 가능 ([[04 설정]]). v2.1.126부터 호스트가 `CLAUDE_CODE_PROVIDER_MANAGED_BY_HOST`를 설정하면 Vertex/Bedrock/Foundry에서 메트릭이 기본 on(표준 `DISABLE_TELEMETRY` opt-out 따름), Sentry·`/feedback`는 off 유지.

**WebFetch 도메인 안전 체크** — URL fetch 전 WebFetch가 요청 hostname을 `api.anthropic.com`에 보내 안전 blocklist와 대조한다(**hostname만**, 전체 URL·경로·내용 미전송, hostname당 5분 캐시). 모든 provider에서 실행되고 `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC`에 영향받지 않는다. 네트워크가 `api.anthropic.com`을 차단하면 allowlist하거나 `skipWebFetchPreflight: true` 설정(이 경우 WebFetch 권한 규칙과 병행 권장). WebFetch 도구는 [[06 내장 도구 레퍼런스]] 참고.

## 7. Zero Data Retention (zero-data-retention)

ZDR은 **Claude for Enterprise**를 통한 Claude Code에서 사용 가능하다. 활성화 시 Claude Code 세션의 프롬프트·모델 응답이 실시간 처리되고 응답 반환 후 Anthropic이 **저장하지 않는다**(법 준수·남용 대응 필요 시 제외). 사용 모델과 무관하게 적용된다.

ZDR + Claude for Enterprise는 관리 기능도 제공: 사용자별 비용 통제 / Analytics 대시보드 / Server-managed settings / Audit logs. ([[16 CI-CD와 팀 통합]])

ZDR은 Anthropic 직접 플랫폼에만 적용된다. Bedrock/Vertex/Foundry 배포는 해당 플랫폼의 보존 정책을 따른다 ([[17 클라우드 프로바이더]]).

> [!warning] 조직별 활성화
> ZDR은 **per-organization** 으로 활성화된다. 새 조직마다 Anthropic 계정 팀이 별도로 켜야 하며, 같은 계정의 신규 조직에 자동 적용되지 않는다.

### 7.1 ZDR 범위 & 비포함

ZDR은 Claude for Enterprise의 Claude Code 추론 호출(터미널에서 보낸 프롬프트·생성 응답)을 커버한다. **커버하지 않는 것**(표준 보존 정책 따름):

| 기능 | 상세 |
|---|---|
| Chat on claude.ai | Enterprise 웹 인터페이스 대화는 ZDR 미적용 |
| Cowork | ZDR 미적용 |
| Claude Code Analytics | 프롬프트·응답은 저장 안 하나 account email·usage 통계 같은 생산성 메타데이터 수집. ZDR 조직은 contribution metrics 미제공, usage metrics만 표시 |
| User and seat management | account email·seat 할당 등 관리 데이터는 표준 정책 보존 |
| Third-party integrations | 외부 도구·MCP 서버 처리 데이터는 ZDR 미적용 — 해당 서비스 정책 별도 검토 ([[10 MCP]]) |

### 7.2 ZDR 하에서 비활성화되는 기능

프롬프트·완성을 저장해야 하는 기능은 백엔드 레벨에서 자동 비활성화된다:

| 기능 | 이유 |
|---|---|
| Claude Code on the Web | 대화 기록의 서버 측 저장 필요 ([[15 웹과 모바일]]) |
| Remote sessions (Desktop) | 프롬프트·완성 포함 영속 세션 데이터 필요 ([[14 IDE와 데스크톱]]) |
| Feedback submission (`/feedback`) | 대화 데이터를 Anthropic에 전송 |

백엔드에서 차단되므로 시도하면 정책상 불허 오류를 반환한다. 향후 프롬프트·완성 저장이 필요한 기능도 비활성화될 수 있다.

### 7.3 정책 위반 데이터 보존 & 요청

ZDR이 켜져 있어도 법 요구·Usage Policy 위반 대응을 위해 데이터를 보존할 수 있다. 세션이 정책 위반으로 플래그되면 관련 입출력을 **최대 2년** 보존할 수 있다.

ZDR 요청: [contact sales](https://www.anthropic.com/contact-sales) 또는 계정 팀. 계정 팀이 내부 제출 → Anthropic이 적격성 확인 후 조직에 활성화(모든 활성화는 audit-log됨). pay-as-you-go API 키로 ZDR을 쓰던 경우 Claude for Enterprise로 전환하면 관리 기능을 얻으면서 ZDR 유지 가능.

### 8.2 컴플라이언스 — Healthcare (BAA)

고객이 Anthropic과 BAA(Business Associate Agreement)를 맺고 Claude Code를 쓰려면, BAA를 체결하고 **ZDR을 활성화**한 경우 BAA가 Claude Code를 자동 커버한다(해당 고객의 Claude Code 경유 API 트래픽에 적용). ZDR은 per-organization이므로 각 조직이 별도 활성화해야 BAA 커버된다. (7절)

### 8.3 Usage Policy

- **Acceptable use**: [Anthropic Usage Policy](https://www.anthropic.com/legal/aup)를 따른다. Pro/Max의 광고 사용 한도는 Claude Code·Agent SDK의 통상적·개인적 사용을 전제.
- **인증·자격증명 사용**:
  - **OAuth 인증**: Free/Pro/Max/Team/Enterprise 구독 구매자 전용, Claude Code 등 네이티브 Anthropic 앱의 통상 사용 지원
  - **개발자**(제품·서비스 구축, Agent SDK 사용 포함): Claude Console 또는 지원 클라우드 provider를 통한 **API 키 인증** 사용. Anthropic은 third-party 개발자가 Claude.ai 로그인을 제공하거나 사용자를 대신해 Free/Pro/Max 플랜 자격증명으로 요청을 라우팅하는 것을 **불허**한다. 이 제한을 사전 통보 없이 강제할 수 있다.
