# 05 권한

Claude Code가 무엇을 할 수 있고 무엇을 할 수 없는지를 세밀하게 통제하는 권한 시스템 레퍼런스다. 권한 규칙(allow/ask/deny)과 권한 모드(permission mode)는 별개의 두 축이며, 둘을 겹쳐서 방어 계층을 만든다. 권한 설정은 버전 관리에 체크인해 조직 전체에 배포할 수 있고, 개발자별로 커스터마이즈할 수도 있다. 설정 파일과 우선순위의 전체 그림은 [[04 설정]], OS 수준 격리는 [[18 보안과 샌드박스]]를 참고하라.

> [!important] 모델이 아니라 하네스가 강제한다
> 권한 규칙은 Claude Code(하네스)가 강제하는 것이지 모델이 강제하는 게 아니다. 프롬프트나 `CLAUDE.md`의 지시는 Claude가 *무엇을 하려 하는지*를 형성할 뿐, *무엇이 허용되는지*는 바꾸지 못한다. 접근을 허용/취소하려면 `/permissions`, 권한 규칙, 권한 모드, 또는 PreToolUse 훅을 써라.

---

## 권한 시스템 (Permission system)

Claude Code는 능력과 안전 사이의 균형을 위해 도구를 계층으로 나눈다.

| Tool type | Example | Approval required | "Yes, don't ask again" 동작 |
| :--- | :--- | :--- | :--- |
| Read-only | File reads, Grep | No | N/A |
| Bash commands | Shell execution | Yes | 프로젝트 디렉터리 + 명령어 단위로 영구 저장 |
| File modification | Edit/write files | Yes | 세션 종료까지 |

읽기 전용 도구는 승인 없이 실행되고, Bash와 파일 수정 도구는 승인을 요구한다. Bash는 "다시 묻지 않기"가 영구적(프로젝트 디렉터리+명령어 단위), 파일 수정은 세션 단위라는 점이 다르다.

## 권한 관리 (Manage permissions)

`/permissions` UI로 모든 권한 규칙과 그것이 어느 settings.json에서 왔는지 확인·관리한다.

- **Allow** 규칙: 수동 승인 없이 지정한 도구를 사용하게 한다.
- **Ask** 규칙: 지정한 도구를 쓰려 할 때마다 확인을 요구한다.
- **Deny** 규칙: 지정한 도구 사용을 막는다.

### 평가 순서 — deny → ask → allow

규칙은 **deny → ask → allow** 순으로 평가되고 **첫 매칭 규칙이 이긴다**. 따라서 deny 규칙이 항상 우선한다(deny-first). 이 우선순위는 [[09 훅]]의 PreToolUse 훅보다도 강하다(아래 "훅으로 권한 확장" 참고).

### deny 규칙의 두 가지 동작

deny 규칙은 도구를 이름으로 지목하느냐, 패턴으로 범위를 좁히느냐에 따라 다르게 동작한다.

- 맨이름(bare) `Bash` → 도구 자체를 Claude의 컨텍스트에서 **완전히 제거**. Claude는 그 도구를 아예 보지 못한다.
- 범위 지정 `Bash(rm *)` → 도구는 그대로 두고, 매칭되는 호출만 차단한다.

## 권한 모드 개요 (Permission modes)

권한 모드는 도구 승인 방식을 통째로 바꾼다. 설정 파일의 `defaultMode`로 기본 모드를 지정한다.

| Mode | 설명 |
| :--- | :--- |
| `default` | 표준 동작. 각 도구 첫 사용 시 권한을 물음 |
| `acceptEdits` | 작업 디렉터리/`additionalDirectories` 경로의 파일 편집과 일반 파일시스템 명령(`mkdir`, `touch`, `mv`, `cp` 등)을 자동 수락 |
| `plan` | Plan 모드. 파일을 읽고 읽기 전용 셸 명령으로 탐색하되 소스를 편집하지 않음 |
| `auto` | 백그라운드 안전 검사와 함께 도구 호출을 자동 승인. 현재 research preview |
| `dontAsk` | `/permissions`나 `permissions.allow`로 사전 승인된 것 외의 도구는 자동 거부 |
| `bypassPermissions` | 모든 권한 프롬프트를 건너뜀. `rm -rf /` 같은 루트/홈 삭제는 안전 차단기로 여전히 물음 |

> [!warning] bypassPermissions 위험성
> `bypassPermissions`는 `.git`, `.config/git`, `.claude`, `.vscode`, `.idea`, `.husky`, `.cargo`, `.devcontainer`, `.yarn`, `.mvn`로의 쓰기를 포함해 모든 프롬프트를 건너뛴다. 파일시스템 루트나 홈을 겨냥한 삭제(`rm -rf /`, `rm -rf ~`)만 모델 오류 방지용 차단기로 여전히 물음. 컨테이너/VM처럼 Claude Code가 손상을 줄 수 없는 격리 환경에서만 사용하라. 관리자는 managed settings에서 `permissions.disableBypassPermissionsMode`를 `"disable"`로 설정해 이 모드를 막을 수 있다.

`bypassPermissions`나 `auto` 모드를 막으려면 어느 설정 파일에서든 `permissions.disableBypassPermissionsMode` 또는 `permissions.disableAutoMode`를 `"disable"`로 설정한다. 덮어쓸 수 없는 managed settings에 두는 것이 가장 효과적이다.

각 모드의 상세한 사용법과 트레이드오프는 아래 "[권한 모드 상세](#권한-모드-상세)"에서 다룬다.

## 권한 규칙 문법 (Permission rule syntax)

권한 규칙은 `Tool` 또는 `Tool(specifier)` 형식을 따른다.

### 도구의 모든 사용 매칭

괄호 없이 도구 이름만 쓰면 그 도구의 모든 사용을 매칭한다.

| Rule | Effect |
| :--- | :--- |
| `Bash` | 모든 Bash 명령 매칭 |
| `WebFetch` | 모든 web fetch 요청 매칭 |
| `Read` | 모든 파일 읽기 매칭 |

`Bash(*)`는 `Bash`와 동등하며 모든 Bash 명령을 매칭한다. deny 규칙으로 쓰면 두 형식 모두 도구를 컨텍스트에서 제거한다.

### 세밀한 제어를 위한 specifier

| Rule | Effect |
| :--- | :--- |
| `Bash(npm run build)` | 정확히 `npm run build` 명령 매칭 |
| `Read(./.env)` | 현재 디렉터리의 `.env` 파일 읽기 매칭 |
| `WebFetch(domain:example.com)` | example.com으로의 fetch 요청 매칭 |

### 와일드카드 패턴 (Wildcard patterns)

Bash 규칙은 `*` 글로브 패턴을 지원하며, 와일드카드는 명령의 어느 위치에든 올 수 있다. 아래는 npm과 git commit은 허용하고 git push는 막는 예시다.

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git commit *)",
      "Bash(git * main)",
      "Bash(* --version)",
      "Bash(* --help *)"
    ],
    "deny": [
      "Bash(git push *)"
    ]
  }
}
```

`*` 앞의 공백이 중요하다. `Bash(ls *)`는 `ls -la`는 매칭하지만 `lsof`는 매칭하지 않고, `Bash(ls*)`는 둘 다 매칭한다. `:*` 접미사는 끝부분의 트레일링 와일드카드와 동등해서 `Bash(ls:*)`는 `Bash(ls *)`와 같은 명령을 매칭한다.

명령 접두사에 대해 "Yes, don't ask again"을 선택하면 권한 다이얼로그는 공백으로 구분한 형식을 기록한다. `:*` 형식은 패턴의 **끝**에서만 인식된다. `Bash(git:* push)` 같은 패턴에서 콜론은 리터럴 문자로 취급되어 git 명령을 매칭하지 못한다.

## 도구별 권한 규칙 (Tool-specific permission rules)

### Bash

Bash 권한 규칙은 `*` 와일드카드 매칭을 지원하며, 와일드카드는 시작·중간·끝 어디에든 올 수 있다.

- `Bash(npm run build)` — 정확히 `npm run build`
- `Bash(npm run test *)` — `npm run test`로 시작하는 명령
- `Bash(npm *)` — `npm `으로 시작하는 모든 명령
- `Bash(* install)` — ` install`로 끝나는 모든 명령
- `Bash(git * main)` — `git checkout main`, `git log --oneline main` 같은 명령

단일 `*`는 공백 포함 임의의 문자열을 매칭하므로 와일드카드 하나가 여러 인자를 걸칠 수 있다. `Bash(git *)`는 `git log --oneline --all`을, `Bash(git * main)`은 `git push origin main`과 `git merge main`을 매칭한다.

끝에 공백을 두고 `*`를 쓰면(`Bash(ls *)`) 단어 경계를 강제해 접두사 뒤에 공백 또는 문자열 끝이 와야 한다. `Bash(ls *)`는 `ls -la`는 매칭, `lsof`는 비매칭. 반면 `Bash(ls*)`는 단어 경계 제약이 없어 둘 다 매칭한다.

#### 복합 명령 (Compound commands)

> [!tip] 셸 연산자 인식
> Claude Code는 셸 연산자를 인식하므로 `Bash(safe-cmd *)` 규칙은 `safe-cmd && other-cmd` 실행 권한을 주지 않는다. 인식하는 명령 구분자는 `&&`, `||`, `;`, `|`, `|&`, `&`, 그리고 줄바꿈이다. 규칙은 각 하위 명령을 독립적으로 매칭해야 한다.

복합 명령을 "Yes, don't ask again"으로 승인하면 전체 복합 문자열에 대한 단일 규칙이 아니라, 승인이 필요한 각 하위 명령에 대해 별도 규칙을 저장한다. 예: `git status && npm test`를 승인하면 `npm test` 규칙이 저장되어 앞에 무엇이 오든 이후 `npm test` 호출이 인식된다. 하위 디렉터리로의 `cd` 같은 하위 명령은 그 경로에 대한 Read 규칙을 따로 생성한다. 단일 복합 명령에 대해 최대 5개 규칙까지 저장된다.

#### 프로세스 래퍼 (Process wrappers)

Bash 규칙 매칭 전에 Claude Code는 고정된 프로세스 래퍼 집합을 벗겨낸다. 그래서 `Bash(npm test *)` 규칙은 `timeout 30 npm test`도 매칭한다. 인식하는 래퍼는 `timeout`, `time`, `nice`, `nohup`, `stdbuf`다.

맨이름 `xargs`도 벗겨내므로 `Bash(grep *)`는 `xargs grep pattern`을 매칭한다. 단, `xargs`에 플래그가 없을 때만 벗겨낸다. `xargs -n1 grep pattern`은 `xargs` 명령으로 매칭되어 내부 명령용 규칙으로는 커버되지 않는다.

이 래퍼 목록은 빌트인이고 설정 불가다. `direnv exec`, `devbox run`, `mise exec`, `npx`, `docker exec` 같은 개발 환경 러너는 목록에 없다. 이들은 인자를 명령으로 실행하므로 `Bash(devbox run *)` 규칙은 `run` 뒤의 무엇이든 매칭한다 — `devbox run rm -rf .`까지 포함. 환경 러너 안의 작업을 승인하려면 러너와 내부 명령을 함께 명시한 구체 규칙을 써라(예: `Bash(devbox run npm test)`). 허용할 내부 명령마다 규칙을 하나씩 추가한다.

`watch`, `setsid`, `ionice`, `flock` 같은 exec 래퍼는 항상 프롬프트하며 `Bash(watch *)` 같은 접두사 규칙으로 자동 승인할 수 없다. `find`의 `-exec`/`-delete` 형식도 마찬가지로 `Bash(find *)` 규칙이 커버하지 못한다. 특정 호출을 승인하려면 전체 명령 문자열에 대한 정확 매칭 규칙을 작성하라.

#### 읽기 전용 명령 (Read-only commands)

Claude Code는 빌트인 Bash 명령 집합을 읽기 전용으로 인식해 모든 모드에서 프롬프트 없이 실행한다: `ls`, `cat`, `echo`, `pwd`, `head`, `tail`, `grep`, `find`, `wc`, `which`, `diff`, `stat`, `du`, `cd`, 그리고 `git`의 읽기 전용 형식들. 이 집합은 설정 불가이며, 프롬프트를 강제하려면 해당 명령에 `ask`/`deny` 규칙을 추가한다.

모든 플래그가 읽기 전용인 명령에는 따옴표 없는 글로브가 허용되므로 `ls *.ts`, `wc -l src/*.py`는 프롬프트 없이 실행된다. `find`, `sort`, `sed`, `git`처럼 쓰기/exec 가능 플래그를 가진 명령은 따옴표 없는 글로브가 있으면 여전히 프롬프트한다(글로브가 `-delete` 같은 플래그로 확장될 수 있어서다).

작업 디렉터리나 [추가 디렉터리](#작업-디렉터리-working-directories) 안 경로로의 `cd`도 읽기 전용이다. `cd packages/api && ls` 같은 복합 명령은 각 부분이 단독으로 자격을 갖추면 프롬프트 없이 실행된다. 단, `cd`와 `git`을 한 복합 명령에 결합하면 대상 디렉터리와 무관하게 항상 프롬프트한다.

> [!warning] Bash 인자 제약 패턴은 취약하다
> `Bash(curl http://github.com/ *)`처럼 인자를 제약하려는 패턴은 다음 변형을 못 잡는다:
> - URL 앞 옵션: `curl -X GET http://github.com/...`
> - 다른 프로토콜: `curl https://github.com/...`
> - 리다이렉트: `curl -L http://bit.ly/xyz` (github으로 리다이렉트)
> - 변수: `URL=http://github.com && curl $URL`
> - 추가 공백: `curl  http://github.com`
>
> 더 신뢰할 수 있는 URL 필터링:
> - **Bash 네트워크 도구 제한**: deny 규칙으로 `curl`, `wget` 등을 막고, 허용 도메인은 `WebFetch(domain:github.com)` 권한으로 WebFetch 도구를 쓴다.
> - **PreToolUse 훅 사용**: Bash 명령의 URL을 검증해 비허용 도메인을 막는 훅을 구현한다.
> - **CLAUDE.md 가이드 추가**: 허용 curl 패턴을 `CLAUDE.md`에 적는다. 단 이는 Claude가 *시도하는* 것을 형성할 뿐 경계를 강제하지 않으므로 위 옵션과 병행하라.
>
> WebFetch만 쓴다고 네트워크 접근이 막히는 건 아니다. Bash가 허용되면 Claude는 `curl`, `wget` 등으로 임의 URL에 도달할 수 있다.

### PowerShell

PowerShell 규칙은 Bash 규칙과 같은 형식을 쓴다. `*` 와일드카드는 어느 위치에서든 매칭하고, `:*` 접미사는 트레일링 ` *`와 동등하며, 맨이름 `PowerShell`이나 `PowerShell(*)`는 모든 명령을 매칭한다. 아래는 `Get-ChildItem`과 `git commit`은 허용, `Remove-Item`은 막는 예시다.

```json
{
  "permissions": {
    "allow": [
      "PowerShell(Get-ChildItem *)",
      "PowerShell(git commit *)"
    ],
    "deny": [
      "PowerShell(Remove-Item *)"
    ]
  }
}
```

매칭 전에 흔한 별칭(alias)이 정규화된다. cmdlet 이름으로 쓴 규칙은 별칭도 매칭하므로 `PowerShell(Get-ChildItem *)`는 `gci`, `ls`, `dir`도 매칭한다. 매칭은 대소문자 구분 없음. Claude Code는 PowerShell AST를 파싱해 복합 명령의 각 명령을 독립 검사한다. 파이프라인 `|`, 문장 구분자 `;`, PowerShell 7+의 체인 연산자 `&&`/`||`가 복합 명령을 하위 명령으로 분할한다. 복합 명령이 허용되려면 규칙이 모든 하위 명령을 매칭해야 한다. PowerShell 도구 자체는 [[06 내장 도구 레퍼런스]] 참고.

### Read와 Edit

`Edit` 규칙은 파일을 편집하는 모든 빌트인 도구에 적용된다. `Read` 규칙은 Grep·Glob 같은 파일 읽기 빌트인 도구, 프롬프트의 `@file` 멘션, 연결된 [[14 IDE와 데스크톱]]가 공유하는 선택/열린 파일 컨텍스트에 best-effort로 적용된다.

> [!warning] 빌트인 파일 도구에만 적용된다
> Read/Edit deny 규칙은 Claude의 빌트인 파일 도구와, Bash에서 Claude Code가 인식하는 파일 명령(`cat`, `head`, `tail`, `sed`)에 적용된다. 파이썬·Node 스크립트가 직접 파일을 여는 식의 임의 하위 프로세스에는 적용되지 않는다. 모든 프로세스의 경로 접근을 막는 OS 수준 강제가 필요하면 [[18 보안과 샌드박스]]의 샌드박스를 켜라.

Read/Edit 규칙은 [gitignore](https://git-scm.com/docs/gitignore) 명세를 따르며 네 가지 패턴 타입을 가진다.

| Pattern | 의미 | Example | Matches |
| :--- | :--- | :--- | :--- |
| `//path` | 파일시스템 루트로부터의 **절대** 경로 | `Read(//Users/alice/secrets/**)` | `/Users/alice/secrets/**` |
| `~/path` | **홈** 디렉터리로부터의 경로 | `Read(~/Documents/*.pdf)` | `/Users/alice/Documents/*.pdf` |
| `/path` | **프로젝트 루트** 기준 상대 경로 | `Edit(/src/**/*.ts)` | `<project root>/src/**/*.ts` |
| `path` 또는 `./path` | **현재 디렉터리** 기준 상대 경로 | `Read(*.env)` | `<cwd>/*.env` |

> [!warning] `/Users/alice/file`은 절대 경로가 아니다
> `/Users/alice/file` 같은 패턴은 절대 경로가 아니라 프로젝트 루트 기준 상대 경로다. 절대 경로는 `//Users/alice/file`처럼 슬래시 두 개로 쓴다.

Windows에서는 경로가 매칭 전 POSIX 형식으로 정규화된다. `C:\Users\alice`는 `/c/Users/alice`가 되므로 그 드라이브의 `.env`를 매칭하려면 `//c/**/.env`를 쓴다. 모든 드라이브를 가로지르려면 `//**/.env`.

예시:
- `Edit(/docs/**)`: `<project>/docs/`의 편집 (`/docs/`도 아니고 `<project>/.claude/docs/`도 아님)
- `Read(~/.zshrc)`: 홈 디렉터리의 `.zshrc` 읽기
- `Edit(//tmp/scratch.txt)`: 절대 경로 `/tmp/scratch.txt` 편집
- `Read(src/**)`: `<current-directory>/src/`에서 읽기

규칙은 자신의 앵커 아래 파일만 매칭하므로 앵커가 deny 규칙의 도달 범위를 결정한다. 맨파일명은 gitignore 시맨틱을 따라 임의 깊이에서 매칭되므로 `Read(.env)`와 `Read(**/.env)`는 동등하다.

| Deny rule | Blocks | Does not block |
| :--- | :--- | :--- |
| `Read(.env)` 또는 `Read(**/.env)` | 현재 디렉터리 이하의 모든 `.env` | 상위 디렉터리나 다른 프로젝트의 `.env` |
| `Read(//**/.env)` | 파일시스템 전체의 모든 `.env` | 없음. 규칙이 파일시스템 루트에 앵커됨 |

> [!note] `*` vs `**`
> gitignore 패턴에서 `*`는 단일 디렉터리 내 파일을, `**`는 디렉터리를 가로지르는 재귀 매칭을 한다. 모든 파일 접근을 허용하려면 괄호 없이 도구 이름만 쓴다: `Read`, `Edit`, `Write`.

#### 심볼릭 링크 처리

Claude가 심볼릭 링크에 접근하면 권한 규칙은 두 경로를 검사한다 — 링크 자체와 그것이 해석되는 대상 파일. allow와 deny가 이 쌍을 다르게 다룬다:

- **Allow 규칙**: 링크 경로와 대상이 **둘 다** 매칭할 때만 적용. 허용 디렉터리 안의 링크가 바깥을 가리키면 여전히 프롬프트한다.
- **Deny 규칙**: 링크 경로 **또는** 대상 중 하나만 매칭해도 차단. 거부된 파일을 가리키는 링크는 그 자체가 거부된다.

예: `Read(./project/**)` 허용 + `Read(~/.ssh/**)` 거부 상태에서 `~/.ssh/id_rsa`를 가리키는 `./project/key` 링크는 차단된다(대상이 allow를 통과 못 하고 deny에 매칭).

### WebFetch

- `WebFetch(domain:example.com)` — example.com으로의 fetch 요청 매칭. (자세한 도구 동작은 [[10 MCP]]·[[06 내장 도구 레퍼런스]])

### MCP

- `mcp__puppeteer` — `puppeteer` 서버가 제공하는 모든 도구 매칭(서버 이름은 Claude Code에서 설정)
- `mcp__puppeteer__*` — 와일드카드 문법, 역시 `puppeteer` 서버의 모든 도구 매칭
- `mcp__puppeteer__puppeteer_navigate` — `puppeteer` 서버의 `puppeteer_navigate` 도구 매칭

MCP 서버 설정 전반은 [[10 MCP]] 참고.

### Agent (서브에이전트)

`Agent(AgentName)` 규칙으로 Claude가 쓸 수 있는 [[08 서브에이전트와 에이전트 팀]]을 통제한다.

- `Agent(Explore)` — Explore 서브에이전트 매칭
- `Agent(Plan)` — Plan 서브에이전트 매칭
- `Agent(my-custom-agent)` — 커스텀 서브에이전트 `my-custom-agent` 매칭

이 규칙을 `deny` 배열에 넣거나 `--disallowedTools` CLI 플래그로 특정 에이전트를 비활성화한다. Explore 에이전트 비활성화 예:

```json
{
  "permissions": {
    "deny": ["Agent(Explore)"]
  }
}
```

## 훅으로 권한 확장 (Extend permissions with hooks)

[[09 훅]]은 런타임에 권한 평가를 수행하는 커스텀 셸 명령을 등록하는 방법을 제공한다. Claude Code가 도구 호출을 하면 PreToolUse 훅이 권한 프롬프트 *전에* 실행된다. 훅 출력은 도구 호출을 거부하거나, 프롬프트를 강제하거나, 프롬프트를 건너뛰어 호출을 진행시킬 수 있다.

**훅 결정이 권한 규칙을 우회하지는 않는다.** deny와 ask 규칙은 PreToolUse 훅이 무엇을 반환하든 평가되므로, 매칭되는 deny 규칙은 호출을 막고, 매칭되는 ask 규칙은 훅이 `"allow"`/`"ask"`를 반환해도 여전히 프롬프트한다. 이로써 deny-first 우선순위가 managed settings의 deny 규칙까지 포함해 유지된다.

차단 훅은 allow 규칙보다도 우선한다. 종료 코드 2로 종료하는 훅은 권한 규칙 평가 전에 도구 호출을 멈추므로, allow 규칙이 통과시킬 상황에서도 차단이 적용된다. 몇 개만 빼고 모든 Bash를 프롬프트 없이 돌리려면 allow 리스트에 `"Bash"`를 넣고, 특정 명령만 거부하는 PreToolUse 훅을 등록하라.

## 작업 디렉터리 (Working directories)

기본적으로 Claude는 실행된 디렉터리의 파일에만 접근한다. 접근을 확장하는 세 방법:

| 시점 | 방법 |
| :--- | :--- |
| 시작 시 | `--add-dir <path>` CLI 인자 |
| 세션 중 | `/add-dir` 명령 |
| 영구 설정 | settings 파일의 `additionalDirectories` |

추가 디렉터리의 파일은 원래 작업 디렉터리와 같은 권한 규칙을 따른다 — 프롬프트 없이 읽기 가능해지고, 편집 권한은 현재 권한 모드를 따른다. CLI 인자 전반은 [[02 CLI 레퍼런스]] 참고.

### 추가 디렉터리는 파일 접근만 부여한다 (설정 루트가 아님)

디렉터리를 추가하면 Claude가 읽고 편집할 수 있는 범위만 넓어진다. 그 디렉터리가 완전한 설정 루트가 되지는 않는다 — 대부분의 `.claude/` 설정은 추가 디렉터리에서 발견되지 않으며, 일부 타입만 예외로 로드된다.

아래 예외는 `--add-dir` 플래그나 `/add-dir` 명령으로 추가한 디렉터리에만 적용된다. settings 파일의 `permissions.additionalDirectories`에 나열된 디렉터리는 파일 접근만 부여하고 아래 설정을 전혀 로드하지 않는다.

| Configuration | `--add-dir`에서 로드 |
| :--- | :--- |
| `.claude/skills/`의 [[07 스킬]] | Yes, 라이브 리로드 포함 |
| `.claude/settings.json`의 플러그인 설정 | `enabledPlugins`, `extraKnownMarketplaces`만 |
| [[03 메모리와 컨텍스트]] 파일(`CLAUDE.md`), `.claude/rules/`, `CLAUDE.local.md` | `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1`이 설정된 경우만. `CLAUDE.local.md`는 추가로 `local` 설정 소스 필요(기본 활성) |

서브에이전트, 명령, 출력 스타일은 현재 작업 디렉터리와 그 상위, 사용자 디렉터리 `~/.claude/`, managed settings에서 발견된다. 훅과 그 외 `settings.json` 키는 현재 작업 디렉터리의 `.claude/` 폴더(상위 폴백 없음), 사용자 `~/.claude/settings.json`, managed settings에서 로드된다. 이 설정을 프로젝트 간 공유하려면:

- **사용자 레벨 설정**: `~/.claude/agents/`, `~/.claude/output-styles/`, `~/.claude/settings.json`에 두어 모든 프로젝트에서 쓰이게 한다.
- **플러그인**: 설정을 [[11 플러그인]]으로 패키징해 팀이 설치하게 한다.
- **설정 디렉터리에서 실행**: 원하는 `.claude/` 설정이 있는 디렉터리에서 Claude Code를 실행한다.

## 샌드박스와의 상호작용 (How permissions interact with sandboxing)

권한과 [[18 보안과 샌드박스]]의 샌드박스는 상호 보완적인 보안 계층이다.

- **Permissions**: Claude Code가 어떤 도구를 쓰고 어떤 파일/도메인에 접근할 수 있는지 통제. 모든 도구(Bash, Read, Edit, WebFetch, MCP 등)에 적용.
- **Sandboxing**: Bash 도구의 파일시스템·네트워크 접근을 제한하는 OS 수준 강제. Bash 명령과 그 자식 프로세스에만 적용.

심층 방어(defense-in-depth)를 위해 둘 다 사용:

- 권한 deny 규칙은 Claude가 제한 리소스에 접근을 **시도조차** 못 하게 막는다.
- 샌드박스 제한은 프롬프트 인젝션이 Claude의 판단을 우회해도 Bash 명령이 경계 밖에 닿지 못하게 한다.
- 샌드박스의 파일시스템 제한은 `sandbox.filesystem` 설정과 Read/Edit deny 규칙을 결합 — 둘이 최종 샌드박스 경계로 병합된다.
- 네트워크 제한은 WebFetch 권한 규칙과 샌드박스의 `allowedDomains`/`deniedDomains` 리스트를 결합한다.

샌드박스가 `autoAllowBashIfSandboxed: true`(기본값)로 켜지면, 샌드박스된 Bash 명령은 권한에 `ask: Bash(*)`가 있어도 프롬프트 없이 실행된다. 샌드박스 경계가 명령별 프롬프트를 대체하기 때문이다. 명시적 deny 규칙은 여전히 적용되고, `/`·홈·기타 핵심 시스템 경로를 겨냥한 `rm`/`rmdir`은 여전히 프롬프트한다.

## Managed settings (관리형 설정)

중앙 통제가 필요한 조직을 위해 관리자는 사용자/프로젝트 설정으로 덮어쓸 수 없는 managed settings를 배포할 수 있다. 일반 설정 파일과 같은 형식이며 MDM/OS 정책, managed settings 파일, server-managed settings로 전달한다. 전달 메커니즘과 파일 위치는 [[04 설정]] 참고.

### Managed-only 설정

아래 설정은 **managed settings에서만** 읽힌다. 사용자/프로젝트 설정에 두면 효과가 없다.

| Setting | 설명 |
| :--- | :--- |
| `allowAllClaudeAiMcps` | `true`면 claude.ai 커넥터가 배포된 `managed-mcp.json`과 함께 로드된다(억제되지 않음) |
| `allowedChannelPlugins` | 메시지를 푸시할 수 있는 채널 플러그인 허용 목록. 설정 시 기본 Anthropic 허용 목록을 대체. `channelsEnabled: true` 필요 |
| `allowManagedHooksOnly` | `true`면 managed 훅·SDK 훅·managed `enabledPlugins`로 강제 활성화된 플러그인 훅만 로드. 사용자·프로젝트·그 외 플러그인 훅은 차단 |
| `allowManagedMcpServersOnly` | `true`면 managed settings의 `allowedMcpServers`만 인정. `deniedMcpServers`는 모든 소스에서 병합 |
| `allowManagedPermissionRulesOnly` | `true`면 사용자/프로젝트 설정이 `allow`/`ask`/`deny` 규칙을 정의하지 못함. managed 규칙만 적용. MCP 서버 허용 목록에는 영향 없음(그건 `allowManagedMcpServersOnly`) |
| `blockedMarketplaces` | 마켓플레이스 소스 차단 목록. 다운로드 전에 검사되어 파일시스템에 닿지 않음 |
| `channelsEnabled` | 조직의 channels 허용 |
| `forceRemoteSettingsRefresh` | `true`면 원격 managed settings를 새로 가져올 때까지 CLI 시작을 막고, 가져오기 실패 시 종료 (fail-closed) |
| `pluginTrustMessage` | 설치 전 표시되는 플러그인 신뢰 경고에 덧붙이는 커스텀 메시지 |
| `sandbox.filesystem.allowManagedReadPathsOnly` | `true`면 managed settings의 `filesystem.allowRead` 경로만 인정. `denyRead`는 모든 소스에서 병합 |
| `sandbox.network.allowManagedDomainsOnly` | `true`면 managed settings의 `allowedDomains`와 `WebFetch(domain:...)` allow 규칙만 인정. 비허용 도메인은 프롬프트 없이 자동 차단. 거부 도메인은 모든 소스에서 병합 |
| `strictKnownMarketplaces` | 사용자가 추가·설치할 수 있는 플러그인 마켓플레이스 소스를 통제 |
| `strictPluginOnlyCustomization` | 사용자/프로젝트 소스의 스킬·에이전트·훅·MCP 서버를 차단해 플러그인이나 managed settings에서만 오게 함. `true`는 네 표면 모두 잠금, `["skills","hooks"]` 같은 배열은 명명한 것만 잠금 |
| `wslInheritsWindowsSettings` | Windows HKLM 레지스트리 키 또는 `C:\Program Files\ClaudeCode\managed-settings.json`에서 `true`면 WSL이 `/etc/claude-code`에 더해 Windows 정책 체인에서도 managed settings를 읽음 |

`disableBypassPermissionsMode`는 보통 조직 정책 강제를 위해 managed settings에 두지만 어느 스코프에서든 동작한다. 사용자가 자기 설정에 넣어 스스로 bypass 모드를 잠글 수도 있다.

> [!note] Remote Control / Web sessions
> Team·Enterprise 플랜에서 관리자는 [[15 웹과 모바일]]의 Remote Control과 web session을 [Claude Code admin settings](https://claude.ai/admin-settings/claude-code)에서 조직 전체로 켜고 끈다. Remote Control은 추가로 `disableRemoteControl` managed setting으로 기기별 비활성화 가능. web session에는 기기별 managed settings 키가 없다.

## 설정 우선순위 (Settings precedence)

권한 규칙은 다른 모든 Claude Code 설정과 같은 우선순위를 따른다([[04 설정]]).

1. **Managed settings**: 명령행 인자를 포함해 어떤 레벨로도 덮어쓸 수 없음
2. **Command line arguments**: 임시 세션 오버라이드
3. **Local project settings** (`.claude/settings.local.json`)
4. **Shared project settings** (`.claude/settings.json`)
5. **User settings** (`~/.claude/settings.json`)

**어느 레벨에서든 도구가 deny되면 다른 레벨이 allow할 수 없다.** managed deny는 `--allowedTools`로 덮을 수 없고, `--disallowedTools`는 managed settings가 정의한 것 너머로 제한을 추가할 수 있다.

임베딩 호스트는 [`parentSettingsBehavior`](/en/settings#settings-precedence)가 `"merge"`일 때 SDK `managedSettings` 옵션으로 추가 managed 정책을 공급할 수 있다([[22 Agent SDK — 시작]]). 임베더 값은 정책을 조일 수 있으나 풀 수는 없다.

예: 사용자 설정이 어떤 권한을 allow하고 프로젝트 설정이 deny하면 deny가 막는다. 역도 성립 — 사용자 레벨 deny가 프로젝트 레벨 allow를 막는다. deny 규칙은 어느 스코프든 allow보다 먼저 평가되기 때문이다.

### 예제 구성 (Example configurations)

Anthropic의 [예제 리포지토리](https://github.com/anthropics/claude-code/tree/main/examples/settings)에 흔한 배포 시나리오용 스타터 설정이 있다. 이를 출발점으로 삼아 조정하라.

## 권한 모드 상세

`permission-modes` 페이지는 각 모드를 언제·어떻게 쓰는지를 다룬다. Claude가 파일 편집, 셸 명령, 네트워크 요청을 하려 할 때 일시 정지하고 승인을 요청하는데, 모드는 그 정지 빈도를 통제한다. 민감한 작업엔 더 많은 감독을, 방향을 신뢰할 땐 더 적은 중단을 고른다.

### 가용 모드와 트레이드오프

| Mode | 프롬프트 없이 실행되는 것 | 적합한 용도 |
| :--- | :--- | :--- |
| `default` | 읽기만 | 시작 단계, 민감한 작업 |
| `acceptEdits` | 읽기 + 파일 편집 + 일반 파일시스템 명령(`mkdir`, `touch`, `mv`, `cp` 등) | 리뷰하며 코드 반복 작업 |
| `plan` | 읽기만 | 변경 전 코드베이스 탐색 |
| `auto` | 모든 것 + 백그라운드 안전 검사 | 긴 작업, 프롬프트 피로 감소 |
| `dontAsk` | 사전 승인된 도구만 | 잠긴 CI와 스크립트 |
| `bypassPermissions` | 모든 것 | 격리된 컨테이너·VM 전용 |

`bypassPermissions`를 제외한 모든 모드에서 [보호 경로](#보호-경로-protected-paths) 쓰기는 절대 자동 승인되지 않아 리포지토리 상태와 Claude 자신의 설정을 우발적 손상에서 지킨다. 모드가 베이스라인을 잡고, 그 위에 권한 규칙을 얹어 어느 모드에서든 특정 도구를 사전 승인/차단한다(권한 계층을 통째로 건너뛰는 `bypassPermissions` 제외).

### 권한 모드 전환 (Switch permission modes)

모드는 세션 중, 시작 시, 영구 기본값으로 설정할 수 있다. 모드는 아래 컨트롤로 설정하지 채팅에서 Claude에게 부탁해 바꾸는 게 아니다.

**CLI**:
- **세션 중**: `Shift+Tab`으로 `default` → `acceptEdits` → `plan` 순환. 현재 모드는 상태줄에 표시([[12 출력 스타일과 상태줄]]). 모든 모드가 기본 순환에 있는 건 아니다:
  - `auto`: 계정이 auto 모드 요건을 충족하면 등장. auto로 순환하면 수락 전까지 opt-in 프롬프트가 뜨거나, **No, don't ask again**으로 순환에서 제거
  - `bypassPermissions`: `--permission-mode bypassPermissions`, `--dangerously-skip-permissions`, `--allow-dangerously-skip-permissions`로 시작한 뒤 등장. `--allow-` 변형은 활성화 없이 순환에만 추가
  - `dontAsk`: 순환에 절대 안 나타남. `--permission-mode dontAsk`로 설정
  - 활성화된 선택 모드는 `plan` 뒤에 끼어들고, `bypassPermissions`가 먼저·`auto`가 마지막. 둘 다 켜면 auto로 가는 길에 bypassPermissions를 거친다.
- **시작 시**: 플래그로 모드 전달. `claude --permission-mode plan`
- **기본값**: settings의 `defaultMode` 설정.

```json
{
  "permissions": {
    "defaultMode": "acceptEdits"
  }
}
```

`--permission-mode` 플래그는 [비대화 실행](/en/headless)의 `-p`와도 동작한다([[13 자동화와 스케줄링]]).

**VS Code**: 세션 중엔 프롬프트 박스 하단 모드 인디케이터 클릭. 기본값은 VS Code 설정의 `claudeCode.initialPermissionMode` 또는 확장 설정 패널. UI 라벨 매핑:

| UI label | Mode |
| :--- | :--- |
| Ask before edits | `default` |
| Edit automatically | `acceptEdits` |
| Plan mode | `plan` |
| Auto mode | `auto` |
| Bypass permissions | `bypassPermissions` |

`claudeCode.initialPermissionMode`는 `auto`를 받지 않는다. auto로 기본 시작하려면 사용자 settings의 `defaultMode`를 설정하라. Claude Code는 프로젝트/로컬 설정의 `defaultMode: "auto"`를 무시한다. Bypass permissions는 확장 설정의 **Allow dangerously skip permissions** 토글이 켜져야 인디케이터에 등장. 상세는 [[14 IDE와 데스크톱]].

**JetBrains**: IDE 터미널에서 Claude Code를 실행하므로 모드 전환은 CLI와 동일(`Shift+Tab` 순환, `--permission-mode` 전달).

**Desktop**: 보내기 버튼 옆 모드 셀렉터. Auto와 Bypass permissions는 Desktop 설정에서 활성화해야 등장.

**Web and mobile** ([[15 웹과 모바일]]): claude.ai/code나 모바일 앱의 프롬프트 박스 옆 모드 드롭다운. 등장하는 모드는 세션 위치에 따라 다름:
- **Cloud sessions** (Claude Code on the web): Auto accept edits, Plan mode만. Ask permissions·Auto·Bypass permissions 불가.
- **Remote Control sessions** (로컬 머신): Ask permissions, Auto accept edits, Plan mode. Auto·Bypass permissions 불가.

Remote Control은 호스트 실행 시 시작 모드도 지정 가능: `claude remote-control --permission-mode acceptEdits`

### acceptEdits 모드

`acceptEdits`는 작업 디렉터리에서 프롬프트 없이 파일을 생성·편집하게 한다. 상태줄에 `⏵⏵ accept edits on`이 표시된다.

파일 편집에 더해 일반 파일시스템 Bash 명령을 자동 승인한다: `mkdir`, `touch`, `rm`, `rmdir`, `mv`, `cp`, `sed`. 안전 환경변수(`LANG=C`, `NO_COLOR=1`) 접두나 프로세스 래퍼(`timeout`, `nice`, `nohup`) 접두 시에도 자동 승인된다. 파일 편집처럼 자동 승인은 작업 디렉터리/`additionalDirectories` 안 경로에만 적용된다. 그 범위 밖 경로, 보호 경로 쓰기, 그 외 모든 Bash 명령은 여전히 프롬프트한다.

PowerShell 도구가 켜져 있으면 `acceptEdits`는 범위 내 경로의 `Set-Content`, `Add-Content`, `Clear-Content`, `Remove-Item`과 그 흔한 별칭도 자동 승인한다. 같은 범위·보호 경로 규칙이 적용된다.

편집을 인라인으로 승인하는 대신 에디터나 `git diff`로 사후 리뷰하고 싶을 때 쓴다. default에서 `Shift+Tab` 한 번으로 진입하거나 `claude --permission-mode acceptEdits`로 직접 시작.

### plan 모드

Plan 모드는 Claude에게 변경을 만들지 말고 리서치해 변경을 제안하라고 지시한다. 파일을 읽고 탐색용 셸 명령을 돌리고 계획을 쓰지만 소스를 편집하지 않는다. 권한 프롬프트는 default와 동일하게 적용된다.

`Shift+Tab`으로 진입하거나 단일 프롬프트 앞에 `/plan`을 붙인다. CLI 시작: `claude --permission-mode plan`. 계획 승인 없이 나가려면 `Shift+Tab`을 다시 누른다.

#### 계획 리뷰와 승인

계획이 준비되면 Claude가 제시하고 진행 방식을 묻는다:
- 승인 후 auto 모드로 시작
- 승인 후 편집 수락(accept edits)
- 승인 후 각 편집을 수동 리뷰
- 피드백 주며 계속 계획
- [Ultraplan](/en/ultraplan)으로 브라우저 기반 리뷰하며 정제

계획을 승인하면 plan 모드를 나가 각 승인 옵션이 기술하는 권한 모드로 전환되어 Claude가 편집을 시작한다. 다시 계획하려면 `Shift+Tab`으로 plan으로 순환하거나 다음 프롬프트에 `/plan`을 붙인다. `Ctrl+G`로 제안된 계획을 기본 텍스트 에디터에서 직접 열어 편집할 수 있다. `showClearContextOnPlanAccept`가 켜지면 각 승인 옵션이 먼저 계획 컨텍스트를 비울지 제안한다. 계획 수락은 `--name`/`/rename`으로 이미 이름을 정하지 않았다면 계획 내용에서 세션 이름을 자동으로 짓는다.

#### plan 모드를 기본값으로

프로젝트 기본을 plan으로: `.claude/settings.json`에 `"permissions": {"defaultMode": "plan"}`.

### auto 모드

> [!note] 요건
> auto 모드는 Claude Code v2.1.83 이상 필요.

auto 모드는 권한 프롬프트 없이 Claude가 실행하게 한다. 별도 분류기(classifier) 모델이 실행 전 행동을 검토해, 요청을 넘어 에스컬레이션하거나, 미인식 인프라를 겨냥하거나, Claude가 읽은 적대적 콘텐츠에 이끌린 것으로 보이는 행동을 차단한다. auto 모드는 또한 Claude가 명확화 질문으로 멈추지 않고 계속 작업하게 유도한다(단 프롬프트나 스킬이 명시적으로 요구하면 여전히 묻는다). 프롬프트는 유지하면서 더 강한 자율 행동을 원하면 [[12 출력 스타일과 상태줄]]의 Proactive 출력 스타일을 대신 설정하라.

> [!warning] research preview
> auto 모드는 research preview다. 프롬프트를 줄이지만 안전을 보장하지 않는다. 방향을 신뢰하는 작업에 쓰고, 민감한 작업의 리뷰 대체로 쓰지 마라.

요건(모두 충족해야 가용):
- **Plan**: 모든 플랜
- **Admin**: Team·Enterprise에선 관리자가 admin settings에서 켜야 사용자가 켤 수 있음. managed settings에서 `permissions.disableAutoMode`를 `"disable"`로 잠글 수도 있음
- **Model**: Anthropic API에선 Claude Opus 4.6 이상 또는 Sonnet 4.6. Amazon Bedrock·Google Cloud Vertex AI·Microsoft Foundry에선 Claude Opus 4.7과 Opus 4.8만. Sonnet 4.5, Opus 4.5, Haiku, claude-3는 어느 프로바이더에서도 미지원
- **Provider**: Anthropic API에선 기본 가용. Bedrock·Vertex AI·Foundry에선 `CLAUDE_CODE_ENABLE_AUTO_MODE`를 설정하기 전까지 꺼짐

auto 모드가 가용 불가로 보고되면 요건 중 하나가 미충족이며 일시적 장애가 아니다. 모델명을 대며 행동의 "safety를 결정할 수 없다"는 별도 메시지는 일시적 분류기 장애다([[25 트러블슈팅]]).

`defaultMode: "auto"`를 설정했는데 세션이 에러 없이 `default`로 시작하면, 그 설정이 `.claude/settings.json`이나 `.claude/settings.local.json`에 있을 가능성이 높다. Claude Code v2.1.142 이상은 그 파일들의 `auto`를 무시한다(리포지토리가 스스로 auto를 부여하지 못하게). `~/.claude/settings.json`으로 옮겨라.

#### Bedrock·Vertex AI·Foundry에서 auto 모드 켜기

이 프로바이더에선 `CLAUDE_CODE_ENABLE_AUTO_MODE`를 `1`로 설정하기 전까지 auto가 `Shift+Tab` 순환에 안 나타난다. 변수는 v2.1.158 이상에서 동작. Claude Opus 4.7과 Opus 4.8만 지원. 한 개발자용:

```json
{
  "env": {
    "CLAUDE_CODE_ENABLE_AUTO_MODE": "1"
  }
}
```

조직용은 같은 `env` 블록을 managed settings에 추가. 변수가 설정되면 auto가 매 세션 순환에 등장. 기본 시작 모드로 하려면 사용자/managed settings에 `"permissions": {"defaultMode": "auto"}`도 설정. 이 프로바이더에선 `CLAUDE_CODE_ENABLE_AUTO_MODE`도 설정되지 않으면 `defaultMode: "auto"`를 무시한다. 개발자가 auto를 켜지 못하게 하려면 managed settings에 `disableAutoMode`를 `"disable"`로(enable 변수를 덮어씀). `ANTHROPIC_BASE_URL`로 구성된 [[10 MCP]]/LLM 게이트웨이를 통하면 게이트웨이가 Anthropic API로 라우팅하므로 enable 변수 없이도 auto가 닿을 수 있다. 프로바이더 설정 전반은 [[17 클라우드 프로바이더]].

#### 분류기가 기본 차단/허용하는 것

분류기는 작업 디렉터리와 리포의 구성된 remote를 신뢰한다. 그 외는 트러스트 인프라를 구성하기 전까지 외부로 취급한다.

**기본 차단**:
- 코드 다운로드 후 실행, 예: `curl | bash`
- 외부 엔드포인트로 민감 데이터 전송
- 프로덕션 배포·마이그레이션
- 클라우드 스토리지 대량 삭제
- IAM/리포 권한 부여
- 공유 인프라 수정
- 세션 전 존재하던 파일을 되돌릴 수 없게 파괴
- Force push, 또는 `main`에 직접 push

**기본 허용**:
- 작업 디렉터리 내 로컬 파일 작업
- lock 파일/매니페스트에 선언된 의존성 설치
- `.env` 읽기와 매칭되는 API로의 자격 증명 전송
- 읽기 전용 HTTP 요청
- 시작한 브랜치나 Claude가 만든 브랜치로 push

샌드박스 네트워크 접근 요청은 기본 허용이 아니라 분류기로 라우팅된다. 전체 규칙 목록은 `claude auto-mode defaults`로 본다. 일상 행동이 막히면 관리자가 `autoMode.environment` 설정으로 신뢰 리포·버킷·서비스를 추가할 수 있다([Configure auto mode](/en/auto-mode-config)).

#### 대화에서 진술하는 경계 (Boundaries)

분류기는 대화에서 진술한 경계를 차단 신호로 취급한다. "push하지 마"나 "리뷰 전까지 배포 기다려"라고 하면 기본 규칙이 허용할 행동도 차단한다. 경계는 이후 메시지에서 해제하기 전까지 유효하며, Claude 자신이 조건이 충족됐다고 판단해도 해제되지 않는다.

경계는 규칙으로 저장되지 않는다. 분류기는 매 검사마다 트랜스크립트에서 다시 읽으므로, 컨텍스트 압축이 그 메시지를 제거하면 경계가 사라질 수 있다([[19 비용과 성능]]). 확실한 보장이 필요하면 deny 규칙을 추가하라.

#### auto 모드가 폴백할 때

차단된 행동마다 알림이 뜨고 `/permissions`의 Recently denied 탭에 나타나며, 거기서 `r`을 눌러 수동 승인으로 재시도할 수 있다. 분류기가 연속 3회 또는 총 20회 차단하면 auto 모드가 멈추고 Claude Code가 프롬프트를 재개한다. 프롬프트된 행동을 승인하면 auto가 재개된다. 이 임계값은 설정 불가. 허용된 행동은 연속 카운터를 리셋하고, 총 카운터는 세션 동안 유지되어 자체 한계가 폴백을 일으킬 때만 리셋된다. 비대화 모드(`-p`)에선 반복 차단이 세션을 중단시킨다(프롬프트할 사용자가 없으므로). 반복 차단은 보통 분류기가 인프라 컨텍스트를 모른다는 뜻이다. `/feedback`으로 거짓 양성을 보고하거나 관리자가 트러스트 인프라를 구성하게 하라.

#### 분류기 동작 상세

**평가 순서** (첫 매칭 단계 승리):
1. allow/deny 규칙에 매칭되는 행동은 즉시 해결. 단 보호 경로 쓰기는 allow 매칭이어도 분류기로 라우팅
2. 읽기 전용 행동과 작업 디렉터리 내 파일 편집은 자동 승인. 단 보호 경로 쓰기 제외
3. 그 외 모든 것은 분류기로
4. 분류기가 차단하면 Claude가 이유를 받고 대안을 시도

auto 모드 진입 시 임의 코드 실행을 부여하는 광범위 allow 규칙은 드롭된다: 무차별 `Bash(*)`/`PowerShell(*)`, 와일드카드 인터프리터 `Bash(python*)`, 패키지 매니저 run 명령, `Agent` allow 규칙. `Bash(npm test)` 같은 좁은 규칙은 유지. 드롭된 규칙은 auto를 나가면 복원된다.

분류기는 사용자 메시지·도구 호출·CLAUDE.md 내용을 본다. 도구 결과는 제거되므로 파일/웹페이지의 적대적 콘텐츠가 분류기를 직접 조작할 수 없다. 별도 서버사이드 프로브가 들어오는 도구 결과를 스캔해 Claude가 읽기 전 의심 콘텐츠를 표시한다.

**서브에이전트 처리** ([[08 서브에이전트와 에이전트 팀]]): 분류기는 세 지점에서 검사 — (1) 서브에이전트 시작 전 위임 태스크 설명 평가(위험해 보이는 태스크는 spawn 시점에 차단), (2) 실행 중 각 행동을 부모 세션과 같은 규칙으로 검사하며 서브에이전트 frontmatter의 `permissionMode`는 무시, (3) 완료 시 전체 행동 이력 검토(우려 발견 시 결과 앞에 보안 경고 prepend).

**비용·지연**: 분류기는 `/model` 선택과 무관한 서버 구성 모델에서 돈다(모델을 바꿔도 분류기 가용성 불변). 분류기 호출은 토큰 사용량에 포함된다. 각 검사는 트랜스크립트 일부 + 대기 행동을 보내 실행 전 왕복을 추가한다. 보호 경로 밖의 읽기와 작업 디렉터리 편집은 분류기를 건너뛰므로, 오버헤드는 주로 셸 명령과 네트워크 작업에서 온다.

### dontAsk 모드

`dontAsk`는 프롬프트할 도구 호출을 모두 자동 거부한다. `permissions.allow` 규칙에 매칭되는 행동과 [읽기 전용 Bash 명령](#bash)만 실행되고, 명시적 `ask` 규칙은 프롬프트 대신 거부된다. 이로써 CI 파이프라인이나 Claude가 할 수 있는 것을 정확히 사전 정의하는 제한 환경에서 완전 비대화가 된다. 시작 시: `claude --permission-mode dontAsk`

### bypassPermissions 모드

`bypassPermissions`는 권한 프롬프트와 안전 검사를 비활성화해 도구 호출이 즉시 실행된다. v2.1.126부터 보호 경로 쓰기도 포함된다(이전 버전은 프롬프트했음). 파일시스템 루트나 홈을 겨냥한 삭제(`rm -rf /`, `rm -rf ~`)만 모델 오류 방지용 차단기로 여전히 프롬프트한다. 인터넷 없는 컨테이너·VM·dev container처럼 Claude Code가 호스트를 손상시킬 수 없는 격리 환경에서만 쓴다.

enable 플래그 없이 시작한 세션에선 `bypassPermissions`로 진입할 수 없다. 활성화하려면 플래그와 함께 재시작: `claude --permission-mode bypassPermissions`. `--dangerously-skip-permissions` 플래그가 동등하다.

Linux·macOS에선 root나 `sudo` 하에서 이 모드로 시작을 거부한다:

```text
--dangerously-skip-permissions cannot be used with root/sudo privileges for security reasons
```

인식된 샌드박스 안에선 이 검사가 자동 스킵된다. 컨테이너에서 자율 실행하려면 [dev container](/en/devcontainer) 구성을 쓴다(비-root 사용자로 Claude Code 실행).

> [!warning] 프롬프트 인젝션 무방비
> `bypassPermissions`는 프롬프트 인젝션이나 의도치 않은 행동에 아무 보호를 제공하지 않는다. 프롬프트 없는 백그라운드 안전 검사를 원하면 auto 모드를 대신 써라. 관리자는 managed settings에서 `permissions.disableBypassPermissionsMode`를 `"disable"`로 막을 수 있다.

## 보호 경로 (Protected paths)

일부 경로로의 쓰기는 `bypassPermissions`를 제외한 모든 모드에서 절대 자동 승인되지 않는다. 리포지토리 상태와 Claude 자신의 설정의 우발적 손상을 막는다.

| Mode | 보호 경로 쓰기 |
| :--- | :--- |
| `default`, `acceptEdits`, `plan` | Prompted |
| `auto` | 분류기로 라우팅 |
| `dontAsk` | Denied |
| `bypassPermissions` | Allowed |

**보호 디렉터리**: `.git`, `.config/git`, `.vscode`, `.idea`, `.husky`, `.cargo`, `.devcontainer`, `.yarn`, `.mvn`, `.claude`(단 Claude가 자체 git worktree를 두는 `.claude/worktrees`는 예외).

**보호 파일**:
- `.gitconfig`, `.gitmodules`
- `.bashrc`, `.bash_profile`, `.bash_login`, `.bash_aliases`, `.bash_logout`, `.zshrc`, `.zprofile`, `.zshenv`, `.zlogin`, `.zlogout`, `.profile`, `.envrc`
- `.npmrc`, `.yarnrc`, `.yarnrc.yml`, `.pnp.cjs`, `.pnp.loader.mjs`, `.pnpmfile.cjs`, `bunfig.toml`, `.bunfig.toml`
- `.bazelrc`, `.bazelversion`, `.bazeliskrc`
- `.pre-commit-config.yaml`, `lefthook.yml`, `lefthook.yaml`, `.lefthook.yml`, `.lefthook.yaml`
- `gradle-wrapper.properties`, `maven-wrapper.properties`
- `.devcontainer.json`
- `.ripgreprc`, `pyrightconfig.json`
- `.mcp.json`, `.claude.json`
