# 25 트러블슈팅

Claude Code의 설치/로그인 실패, 런타임 에러, 성능·안정성·검색 문제를 진단하고 복구하는 종합 레퍼런스다. 허브: [[Claude]].

> [!tip] 가장 먼저 할 일
> Claude Code 안에서 `/doctor`를 실행하면 설치 상태, 설정 유효성, MCP 서버, 컨텍스트 사용량을 한 번에 자동 점검한다. `claude`가 아예 시작되지 않으면 쉘에서 `claude doctor`를 실행하라.

세 페이지의 역할 분담:
- **성능/안정성/검색** (CPU·메모리·행업·검색 문제) → 이 노트의 `## 성능과 안정성`
- **설치/로그인 실패** (`command not found`, PATH, 권한, 네트워크, 인증) → `## 설치 및 로그인 문제 해결`
- **런타임 에러** (`API Error: 5xx`, `429`, 모델·요청 에러) → `## 런타임 에러 레퍼런스`

런타임 이후 어떤 문제가 어느 페이지(섹션)에 속하는지 빠르게 찾는 색인이다.

| 증상 | 참고 위치 |
| :--- | :--- |
| `command not found`, 설치 실패, PATH 문제, `EACCES`, TLS 에러 | `## 설치 및 로그인 문제 해결` |
| 로그인 루프, OAuth 에러, `403 Forbidden`, "organization disabled", Bedrock/Vertex/Foundry 자격증명 | `### 로그인과 인증` |
| 설정이 적용 안 됨, 훅이 안 걸림, MCP 서버 미로드 | [[04 설정]] / [[09 훅]] / [[10 MCP]] (debug-your-config) |
| `API Error: 5xx`, `529 Overloaded`, `429`, 요청 검증 에러 | `## 런타임 에러 레퍼런스` |
| `model not found` 또는 `you may not have access to it` | `### 선택한 모델에 문제가 있음` |

> 어느 쪽인지 모르겠으면 무조건 `/doctor`(또는 쉘에서 `claude doctor`) 먼저.

## 성능과 안정성 (troubleshooting)

Claude Code가 실행된 이후의 리소스 사용량, 응답성, 검색 동작 관련 문제를 다룬다.

### 높은 CPU 또는 메모리 사용

대규모 코드베이스를 처리할 때 리소스를 많이 소비할 수 있다. 성능 문제가 있다면 순서대로:

1. `/compact`를 주기적으로 실행해 컨텍스트 크기를 줄인다.
2. 주요 작업 사이에 Claude Code를 닫고 재시작한다.
3. 큰 빌드 디렉터리를 `.gitignore`에 추가한다.

이후에도 메모리가 높게 유지되면 `/heapdump`를 실행한다. JavaScript 힙 스냅샷과 메모리 분해(breakdown)를 `~/Desktop`에 기록한다. 데스크톱 폴더가 없는 Linux에서는 홈 디렉터리에 기록된다.

| 항목 | 설명 |
| :--- | :--- |
| `/heapdump` | 힙 스냅샷(`.heapsnapshot`) + 메모리 분해 파일 생성 |
| 분해 내용 | resident set size, JS heap, array buffers, 미집계 native 메모리 |
| 분석 방법 | `.heapsnapshot`을 Chrome DevTools → Memory → Load로 열어 retainer 검사 |

메모리 이슈를 [GitHub](https://github.com/anthropics/claude-code/issues)에 보고할 때 두 파일을 모두 첨부하라. JS 객체 증가인지 native 코드 증가인지 식별하는 데 쓰인다.

### 자동 압축이 thrashing 에러로 멈춤

`Autocompact is thrashing: the context refilled to the limit...`가 보이면, 자동 압축은 성공했지만 파일이나 도구 출력이 곧바로 컨텍스트 윈도를 여러 번 연속으로 다시 채운 것이다. Claude Code는 진전 없는 루프에 API 호출을 낭비하지 않으려고 재시도를 멈춘다.

복구 방법:

1. 큰 파일을 통째로 읽지 말고 특정 줄 범위나 함수처럼 작은 단위로 읽게 한다.
2. 큰 출력을 버리는 포커스로 압축: `/compact keep only the plan and the diff`.
3. 큰 파일 작업을 [[08 서브에이전트와 에이전트 팀]]으로 옮겨 별도 컨텍스트 윈도에서 실행한다.
4. 앞선 대화가 더 필요 없으면 `/clear`.

### 명령이 행업하거나 멈춤

응답이 없어 보이면:

1. Ctrl+C로 현재 작업 취소를 시도한다.
2. 여전히 응답 없으면 터미널을 닫고 재시작한다.

재시작해도 대화는 잃지 않는다. 같은 디렉터리에서 `claude --resume`으로 세션을 이어받는다.

### 에디터 통합 터미널에서 글자가 깨짐

VS Code, Cursor, Devin Desktop 통합 터미널에서 글자가 박스·번짐·잘못된 글리프로 렌더링되면 터미널의 GPU 렌더러가 원인일 가능성이 크다. Claude Code 안에서 `/terminal-setup`을 실행하면 `terminal.integrated.gpuAcceleration`을 `"off"`로 설정한다. 또는 에디터 설정에서 직접 끄고 창을 리로드한다. `/terminal-setup`이 쓰는 다른 설정은 Terminal 설정 문서를 참조.

| 명령 | 효과 |
| :--- | :--- |
| `/terminal-setup` | `terminal.integrated.gpuAcceleration: "off"` 등 터미널 설정 기록 |

### 검색과 탐색 문제

Search 도구, `@file` 멘션, 커스텀 에이전트, 커스텀 스킬이 파일을 못 찾으면 번들된 `ripgrep` 바이너리가 시스템에서 실행되지 않는 것일 수 있다. 플랫폼별 `ripgrep` 패키지를 설치하고 그것을 쓰도록 지시하라.

| 플랫폼 | 설치 명령 |
| :--- | :--- |
| macOS | `brew install ripgrep` |
| Ubuntu/Debian | `sudo apt install ripgrep` |
| Alpine | `apk add ripgrep` |
| Arch | `pacman -S ripgrep` |
| Windows | `winget install BurntSushi.ripgrep.MSVC` |

그런 다음 [[04 설정]]에서 환경변수 `USE_BUILTIN_RIPGREP=0`을 설정한다.

### WSL에서 느리거나 불완전한 검색 결과

WSL에서 파일 시스템을 넘나들 때의 디스크 읽기 성능 패널티로 인해, 기대보다 적은 매치가 나올 수 있다. 검색은 작동하지만 native 파일 시스템보다 결과가 적다.

> [!note]
> 이 경우 `/doctor`는 Search를 OK로 표시한다.

해결책:
1. **더 구체적으로 검색**: 디렉터리/파일 타입을 지정해 검색 대상 파일 수를 줄인다. 예: "Search for JWT validation logic in the auth-service package", "Find use of md5 hash in JS files".
2. **프로젝트를 Linux 파일 시스템으로 이동**: 가능하면 `/mnt/c/`가 아니라 `/home/`에 둔다.
3. **네이티브 Windows 사용**: WSL 대신 Windows에서 직접 실행하면 파일 시스템 성능이 더 좋다.

### 추가 도움 받기

여기에 없는 문제라면:
1. `/doctor` — 설치 상태, 설정 유효성, MCP 설정, 컨텍스트 사용량을 한 번에 점검.
2. `/feedback` — Anthropic에 직접 문제 보고.
3. [GitHub 저장소](https://github.com/anthropics/claude-code) — 알려진 이슈 확인.
4. Claude에게 직접 기능을 물어보기 (Claude는 자체 문서에 내장 접근 가능).

## 설치 및 로그인 문제 해결 (troubleshoot-install)

설치가 실패하거나 로그인이 안 될 때의 진단·복구 레퍼런스다. 설치 이후 런타임 문제는 `## 성능과 안정성`, 설정 적용/훅 문제는 debug-your-config를 보라.

> [!tip] 터미널을 건너뛰고 싶다면
> [[14 IDE와 데스크톱]]의 Claude Code Desktop 앱으로 GUI를 통해 설치·사용할 수 있다.

### 에러 메시지 → 해결책 색인

| 보이는 메시지 | 해결책(이 노트 섹션) |
| :--- | :--- |
| `command not found: claude` / `'claude' is not recognized` | 설치 후 command not found |
| `syntax error near unexpected token '<'` | 설치 스크립트가 HTML 반환 |
| `curl: (22) ... returned error: 403` | 설치 스크립트 403 |
| `curl: (23)` / `curl: (56) Failure writing output...` | 연결 확인 또는 대체 설치 |
| 설치 중 `Killed` (Linux) | 저메모리 서버에 swap 추가 |
| `TLS connect error` / `SSL/TLS secure channel` | CA 인증서 업데이트 |
| `Failed to fetch version` | 네트워크/프록시 설정 확인 |
| `irm is not recognized` / `&& is not valid` | 쉘에 맞는 명령 사용 |
| `'bash' is not recognized as the name of a cmdlet` | Windows 설치 명령 사용 |
| `Claude Code on Windows requires either Git for Windows ... or PowerShell` | 쉘 설치 |
| `Claude Code does not support 32-bit Windows` | x86 아닌 PowerShell 열기 |
| `The process cannot access the file ... used by another process` | downloads 폴더 비우고 재시도 |
| `Error loading shared library` | 잘못된 바이너리 변종 (musl/glibc) |
| `Illegal instruction` | 아키텍처/CPU 명령어 셋 불일치 |
| `cannot execute binary file: Exec format error` (WSL) | WSL1 native-binary 회귀 |
| 설치 후 `claude` 미발견 또는 구버전 | 터미널 재시작 + PATH 확인 |
| `dyld: cannot load` / `Symbol not found` / `Abort trap` (macOS) | 바이너리 비호환 |
| `Invoke-Expression: Missing argument in parameter list` | 설치 스크립트 HTML 반환 |
| `App unavailable in region` | 지원 국가 아님 ([supported countries](https://www.anthropic.com/supported-countries)) |
| `unable to get local issuer certificate` | 사내 CA 인증서 구성 |
| `OAuth error` / `403 Forbidden` | 인증 수정 |
| `Could not load the default credentials` / `Could not load credentials from any providers` | Bedrock/Vertex/Foundry 자격증명 |
| `ChainedTokenCredential authentication failed` / `CredentialUnavailableError` | Bedrock/Vertex/Foundry 자격증명 |
| 위에 없는 4xx/5xx | `## 런타임 에러 레퍼런스` |

### 진단 체크 (Run diagnostic checks)

#### 네트워크 연결 확인

설치 프로그램은 `downloads.claude.ai`에서 다운로드한다. 도달 가능한지 확인:

```bash
curl -sI https://downloads.claude.ai/claude-code-releases/latest
```

> PowerShell에서는 `curl.exe -sI`를 써라. PowerShell의 `curl` 별칭(`Invoke-WebRequest`)은 `-sI` 플래그를 거부한다.

`HTTP/2 200` 줄이 보이면 서버에 도달한 것이다. 출력이 없거나 `Could not resolve host`, 타임아웃이면 네트워크가 차단 중이다. 흔한 원인: 사내 방화벽/프록시의 `downloads.claude.ai` 차단, 지역 제한(VPN/다른 네트워크 시도), TLS/SSL 이슈(CA 인증서 업데이트 또는 `HTTPS_PROXY` 확인).

사내 프록시 뒤라면 설치 전 프록시 변수를 설정한다:

| OS | 설정 + 설치 |
| :--- | :--- |
| macOS/Linux | `export HTTP_PROXY=http://proxy.example.com:8080`<br>`export HTTPS_PROXY=http://proxy.example.com:8080`<br>`curl -fsSL https://claude.ai/install.sh \| bash` |
| Windows PowerShell | `$env:HTTP_PROXY = 'http://proxy.example.com:8080'`<br>`$env:HTTPS_PROXY = 'http://proxy.example.com:8080'`<br>`irm https://claude.ai/install.ps1 \| iex` |

#### PATH 확인 (Verify your PATH)

설치는 성공했는데 `claude` 실행 시 `command not found`가 나오면 설치 디렉터리가 PATH에 없는 것이다. 설치 위치:
- macOS/Linux: `~/.local/bin/claude`
- Windows: `%USERPROFILE%\.local\bin\claude.exe`

PATH에 있는지 확인:

| 쉘 | 확인 명령 |
| :--- | :--- |
| macOS/Linux | `echo $PATH \| tr ':' '\n' \| grep -Fx "$HOME/.local/bin"` |
| Windows PowerShell | `$env:PATH -split ';' \| Select-String '\.local\\bin'` |
| Windows CMD | `echo %PATH% \| findstr /i "local\bin"` |

출력이 없으면 PATH에 추가:

```bash
# Zsh (macOS 기본)
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc

# Bash (대부분의 Linux)
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc
```

```powershell
# Windows PowerShell — User PATH에 추가
$currentPath = [Environment]::GetEnvironmentVariable('PATH', 'User')
[Environment]::SetEnvironmentVariable('PATH', "$currentPath;$env:USERPROFILE\.local\bin", 'User')
# 이후 터미널 재시작
```

Windows CMD는 시스템 설정 → 환경 변수에서 `%USERPROFILE%\.local\bin`을 User PATH에 추가하고 터미널 재시작. 검증은 `claude --version`. fish/Nushell은 각 쉘 문법으로 추가.

#### 충돌하는 설치 확인

여러 Claude Code 설치가 버전 불일치나 예기치 못한 동작을 일으킬 수 있다. `claude` 바이너리가 올 수 있는 세 위치:

| 위치 | 의미 |
| :--- | :--- |
| `~/.local/bin/claude` | native 설치 프로그램 (권장) |
| `~/.claude/local/` | 구버전이 만든 레거시 local npm 설치 |
| `npm -g ls @anthropic-ai/claude-code` 결과 | `-g` 글로벌 설치 |

확인 명령:

```bash
# macOS/Linux
which -a claude
ls -la ~/.local/bin/claude
ls -la ~/.claude/local/
npm -g ls @anthropic-ai/claude-code 2>/dev/null
```

```powershell
# Windows PowerShell
where.exe claude
Test-Path "$env:USERPROFILE\.local\bin\claude.exe"
```

여럿 발견되면 하나만 남긴다 (native 권장). 제거 명령:

| 대상 | 제거 명령 |
| :--- | :--- |
| npm 글로벌 | `npm uninstall -g @anthropic-ai/claude-code` |
| 레거시 local npm (macOS/Linux) | `rm -rf ~/.claude/local` |
| 레거시 local npm (Windows) | `Remove-Item -Recurse -Force "$env:USERPROFILE\.claude\local"` |
| Homebrew (macOS) | `brew uninstall --cask claude-code` |
| WinGet (Windows) | `winget uninstall Anthropic.ClaudeCode` |

#### 디렉터리 권한 확인

설치 프로그램은 macOS/Linux에서 `~/.local/bin/`과 `~/.claude/`에 쓰기 권한이 필요하다. (Windows는 `%USERPROFILE%` 아래라 보통 해당 없음.)

```bash
test -w ~/.local/bin && echo "writable" || echo "not writable"
test -w ~/.claude && echo "writable" || echo "not writable"
# 쓰기 불가 시:
sudo mkdir -p ~/.local/bin
sudo chown -R $(whoami) ~/.local
```

#### 바이너리 동작 확인

`claude --version`은 버전을 출력하는데 시작 시 크래시/행업하면:

```bash
ls -la "$(command -v claude)"          # 존재·실행 권한 확인
ldd "$(command -v claude)" | grep "not found"   # Linux: 누락된 공유 라이브러리
claude --version                        # 실행 확인
```

```powershell
Get-Command claude | Select-Object Source   # Windows
```

musl 기반(Alpine)은 setup 문서의 Alpine Linux 섹션 참조.

### 흔한 설치 문제

#### 설치 스크립트가 HTML/에러를 반환

```text
bash: line 1: syntax error near unexpected token `<'
bash: line 1: `<!DOCTYPE html>'
```
PowerShell: `Invoke-Expression: Missing argument in parameter list.` / 또는 본문 없는 403: `curl: (22) The requested URL returned error: 403`.

설치 URL이 스크립트 대신 HTML 페이지/에러 상태를 반환한 것이다. HTML이 "App unavailable in region"이면 지원 국가가 아니다. 본문 없는 403은 사내 프록시/방화벽 차단일 수도 있으니 지원 국가인데도 403이면 먼저 네트워크 연결 확인을 거쳐라.

해결책 — 대체 설치 방법:

| OS | 명령 |
| :--- | :--- |
| macOS | `brew install --cask claude-code` |
| Windows | `winget install Anthropic.ClaudeCode` |

아니면 몇 분 후 원래 명령 재시도(일시적 문제인 경우가 많음).

#### 설치 후 `command not found: claude`

플랫폼별 에러:

| 플랫폼 | 메시지 |
| :--- | :--- |
| macOS | `zsh: command not found: claude` |
| Linux | `bash: claude: command not found` |
| Windows CMD | `'claude' is not recognized as an internal or external command` |
| PowerShell | `claude : The term 'claude' is not recognized as the name of a cmdlet` |

설치 디렉터리가 PATH에 없는 것 → 위의 PATH 확인 참조.

#### `curl: (56) Failure writing output to destination`

`curl ... | bash`는 스크립트를 다운로드해 Bash에 파이프한다. 이 에러(및 `curl: (23)`)는 Bash가 완전한 스크립트를 못 받은 것이다. 56은 다운로드 자체가 중단, 23은 curl이 받은 것을 파이프에 못 쓴 경우(보통 Bash가 일찍 종료).

해결: 네트워크 안정성 확인(`curl -sI https://downloads.claude.ai/claude-code-releases/latest` → `HTTP/2 200`이면 일시적, 재시도), 또는 대체 설치(`brew install --cask claude-code` / `winget install Anthropic.ClaudeCode`).

#### TLS 또는 SSL 연결 에러

`curl: (35) TLS connect error`, `schannel: next InitializeSecurityContext failed`, `Could not establish trust relationship for the SSL/TLS secure channel`는 TLS 핸드셰이크 실패다.

해결책:

| 상황 | 조치 |
| :--- | :--- |
| 시스템 CA 인증서 업데이트 (Ubuntu/Debian) | `sudo apt-get update && sudo apt-get install ca-certificates` |
| macOS | 시스템 curl은 Keychain trust store 사용 — macOS 업데이트가 루트 인증서 갱신 |
| Windows에서 TLS 1.2 활성화 | `[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12`<br>`irm https://claude.ai/install.ps1 \| iex` |
| 사내 CA bundle 지정 (설치 단계) | `curl --cacert /path/to/corporate-ca.pem -fsSL https://claude.ai/install.sh \| bash` |
| 설치 후 Claude Code 자체 | `export NODE_EXTRA_CA_CERTS=/path/to/corporate-ca.pem` |
| Windows 인증서 폐기 검사 우회 (`CRYPT_E_NO_REVOCATION_CHECK 0x80092012` 등) | `curl --ssl-revoke-best-effort -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd` |

TLS 인스펙션을 하는 사내 프록시는 `unable to get local issuer certificate`, `SELF_SIGNED_CERT_IN_CHAIN`도 유발한다. 네트워크/CA 관련 자세한 설정은 [[18 보안과 샌드박스]] 및 network-config 참조.

#### `Failed to fetch version from downloads.claude.ai`

`downloads.claude.ai`가 네트워크에서 차단된 경우다. (1) `curl -sI ...`로 직접 연결 테스트, (2) 프록시 뒤면 `export HTTPS_PROXY=http://proxy.example.com:8080` 후 설치, (3) 제한된 네트워크면 다른 네트워크/VPN 또는 brew/winget 대체 설치.

#### Windows에서 잘못된 설치 명령

| 보이는 에러 | 원인 → 해결 |
| :--- | :--- |
| `'irm' is not recognized` | CMD에서 PowerShell 명령 실행 → PowerShell 열고 `irm https://claude.ai/install.ps1 \| iex` 또는 CMD 설치 `curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd` |
| `The token '&&' is not valid` | PowerShell에서 CMD 명령 실행 → `irm https://claude.ai/install.ps1 \| iex` |
| `'bash' is not recognized` | Windows에서 macOS/Linux 설치 실행 → `irm https://claude.ai/install.ps1 \| iex` |

#### Windows 설치 중 `The process cannot access the file`

`Failed to download binary: The process cannot access the file ... because it is being used by another process`는 `%USERPROFILE%\.claude\downloads`에 쓰지 못한 것이다. 이전 설치가 아직 실행 중이거나 백신이 부분 다운로드 바이너리를 스캔 중인 경우다.

```powershell
Remove-Item -Recurse -Force "$env:USERPROFILE\.claude\downloads"
irm https://claude.ai/install.ps1 | iex
```

#### 저메모리 Linux 서버에서 설치가 `Killed`

```text
bash: line 142: 34803 Killed    "$binary_path" install ${TARGET:+"$TARGET"}
```

Linux OOM killer가 메모리 부족으로 프로세스를 죽인 것. Claude Code는 **최소 4GB 가용 RAM**이 필요하다.

해결책 — swap 추가:

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
# 재시도
curl -fsSL https://claude.ai/install.sh | bash
```

또는 다른 프로세스 종료로 메모리 확보, 더 큰 인스턴스 사용.

#### Docker에서 설치가 행업

root로 `/`에 설치하면 행업할 수 있다. 설치 프로그램이 `/`에서 실행되면 전체 파일 시스템을 스캔해 메모리를 과도하게 쓴다.

```dockerfile
WORKDIR /tmp
RUN curl -fsSL https://claude.ai/install.sh | bash
```

Docker Desktop이면 메모리 한도 상향: `docker build --memory=4g .`. 관련 컨테이너 설정은 [[18 보안과 샌드박스]] 참조.

#### Claude Desktop이 Windows에서 `claude` 명령을 가로챔

구버전 Claude Desktop이 `WindowsApps`에 `Claude.exe`를 등록해 PATH 우선순위를 차지하면, `claude` 실행 시 CLI 대신 Desktop 앱이 열린다. → Claude Desktop을 최신 버전으로 업데이트하면 해결.

#### Git for Windows 또는 PowerShell 필요

`Claude Code on Windows requires either Git for Windows (for bash) or PowerShell`은 두 쉘 모두 없을 때 나온다. Git for Windows는 선택사항이며, Git Bash가 없으면 Claude Code는 [[06 내장 도구 레퍼런스]]의 PowerShell tool을 쓴다.

- **PowerShell이 PATH에 없으면**: 기본 위치 `C:\Windows\System32\WindowsPowerShell\v1.0\`를 PATH에 추가하거나 [PowerShell 7](https://aka.ms/powershell) 설치(`pwsh` 제공).
- **Git for Windows 설치**: [git-scm.com/downloads/win](https://git-scm.com/downloads/win)에서 받아 "Add to PATH" 선택, 터미널 재시작. Bash 기반 스크립트/도구 작업에 유용.
- **Git은 있는데 못 찾으면** [[04 설정]]의 settings.json에 경로 지정:

```json
{
  "env": {
    "CLAUDE_CODE_GIT_BASH_PATH": "C:\\Program Files\\Git\\bin\\bash.exe"
  }
}
```

다른 위치면 `where.exe git`로 찾아 `bin\bash.exe` 경로 사용.

#### `Claude Code does not support 32-bit Windows`

시작 메뉴의 `Windows PowerShell (x86)`은 32비트 프로세스로 실행돼 64비트 머신에서도 이 에러를 낸다.

```powershell
[Environment]::Is64BitOperatingSystem
```

`True`면 OS는 정상 — 창을 닫고 x86 접미사 없는 `Windows PowerShell`을 열어 재설치. `False`면 32비트 Windows라서 미지원(64비트 OS 필요).

#### Linux musl/glibc 바이너리 불일치

설치 후 `libstdc++.so.6`, `libgcc_s.so.1` 같은 공유 라이브러리 누락 에러:

```text
Error loading shared library libstdc++.so.6: No such file or directory
```

glibc 시스템에 musl 크로스컴파일 패키지가 있어 설치 프로그램이 musl로 오탐할 때 발생.

```bash
ldd --version 2>&1 | head -1   # GNU libc/GLIBC → glibc, musl → musl
```

- **glibc인데 musl 바이너리를 받았으면**: 제거 후 재설치. 또는 manifest로 올바른 바이너리 수동 다운로드: `https://downloads.claude.ai/claude-code-releases/{VERSION}/manifest.json`. `ldd --version`과 `ls /lib/libc.musl*` 출력으로 GitHub 이슈 제출.
- **실제 musl(Alpine)이면**: `apk add libgcc libstdc++ ripgrep`.

#### `Illegal instruction`

native 바이너리가 CPU가 지원하지 않는 명령어를 쓴 것. 두 가지 원인:

| 원인 | 진단 | 조치 |
| :--- | :--- | :--- |
| 아키텍처 불일치 (ARM 서버에 x86 등) | `uname -m` (macOS/Linux), `$env:PROCESSOR_ARCHITECTURE` (PowerShell) | 결과가 받은 바이너리와 다르면 GitHub 이슈 제출 |
| AVX 명령어 셋 누락 (대략 2013년 이전 Intel/AMD, AVX 미전달 VM) | `grep -m1 -ow avx /proc/cpuinfo` (빈 결과면 AVX 없음) | native-binary 우회 없음 — [issue #50384](https://github.com/anthropics/claude-code/issues/50384) 추적 |

보고 시 CPU 모델 포함: `grep -m1 "model name" /proc/cpuinfo` (Linux) 또는 `sysctl -n machdep.cpu.brand_string` (macOS). 대체 설치 방법도 같은 native 바이너리를 받으므로 해결되지 않는다.

#### macOS `dyld: cannot load`

`dyld: cannot load`, `dyld: Symbol not found`, `Abort trap: 6`은 바이너리가 macOS 버전/하드웨어와 비호환인 경우다. `libicucore`를 참조하는 `Symbol not found`는 macOS가 바이너리 지원 버전보다 오래된 신호.

- Claude Code는 **macOS 13.0 이상** 필요. Apple 메뉴 → About This Mac에서 확인.
- 오래됐으면 macOS 업데이트. Homebrew 등 대체 설치도 같은 바이너리라 해결 안 됨.

#### WSL1 `Exec format error`

`cannot execute binary file: Exec format error`는 WSL1의 known 회귀([issue #38788](https://github.com/anthropics/claude-code/issues/38788))다. 가장 깔끔한 해결은 WSL2 변환:

```powershell
wsl --set-version <DistroName> 2
```

WSL1을 유지해야 하면 동적 링커로 바이너리를 호출하는 함수를 `~/.bashrc`에 추가:

```bash
claude() {
  /lib64/ld-linux-x86-64.so.2 "$(readlink -f "$HOME/.local/bin/claude")" "$@"
}
```

이후 `source ~/.bashrc` 후 재시도.

#### WSL에서 npm 설치 에러

`npm install -g`로 설치한 경우에만 해당(native 설치면 건너뜀).

| 문제 | 해결 |
| :--- | :--- |
| OS/플랫폼 감지 (WSL이 Windows npm 사용) | `npm config set os linux` 후 `npm install -g @anthropic-ai/claude-code --force` (sudo 금지) |
| `exec: node: not found` (Windows Node 사용 중) | `which npm`/`which node` 경로가 `/mnt/c/`면 Windows. Linux 패키지 매니저나 [nvm](https://github.com/nvm-sh/nvm)으로 Node 설치 |
| nvm 버전 충돌 (WSL이 Windows PATH·nvm 우선) | `~/.bashrc`/`~/.zshrc`에 nvm 로더 추가 |

nvm 로더:

```bash
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"
[ -s "$NVM_DIR/bash_completion" ] && \. "$NVM_DIR/bash_completion"
# 또는 현재 세션만: source ~/.nvm/nvm.sh
```

nvm은 로드됐는데 Windows 경로가 여전히 우선이면 Linux Node 경로를 앞에 붙인다:

```bash
export PATH="$HOME/.nvm/versions/node/$(node -v)/bin:$PATH"
```

> [!warning]
> `appendWindowsPath = false`로 Windows PATH 임포트를 끄지 마라 — WSL에서 Windows 실행 파일 호출이 깨진다. Windows 개발에 쓴다면 Windows Node.js도 제거하지 마라.

#### 설치 중 권한 에러

native 설치 프로그램이 권한 에러로 실패하면 대상 디렉터리가 쓰기 불가일 수 있다(위 디렉터리 권한 확인). 이전에 npm으로 설치해 npm 특유의 권한 에러를 만나면 native 설치로 전환: `curl -fsSL https://claude.ai/install.sh | bash`.

#### npm 설치 후 native 바이너리 미발견

`@anthropic-ai/claude-code` 패키지는 플랫폼별 optional dependency(예: `@anthropic-ai/claude-code-darwin-arm64`)로 native 바이너리를 가져온다. `Could not find native binary package "@anthropic-ai/claude-code-<platform>"`가 나오면:

- **optional dependency 비활성화**: npm `--omit=optional`, pnpm `--no-optional`, yarn `--ignore-optional` 제거, `.nprc`의 `optional=false` 확인 후 재설치. native 바이너리는 optional로만 배포되며 JS 폴백이 없다.
- **미지원 플랫폼**: prebuilt는 `darwin-arm64`, `darwin-x64`, `linux-x64`, `linux-arm64`, `linux-x64-musl`, `linux-arm64-musl`, `win32-x64`, `win32-arm64`만 제공.
- **사내 npm 미러에 플랫폼 패키지 누락**: 메타 패키지 외에 8개 `@anthropic-ai/claude-code-*` 플랫폼 패키지 모두 미러링 필요.

`--ignore-scripts`로 설치하면 이 에러는 안 나지만 postinstall 링크 단계가 생략돼 매 실행 시 래퍼가 바이너리를 찾아 spawn한다(느림). 직접 실행하려면 scripts 활성화로 재설치.

### 로그인과 인증

#### 로그인 재설정 (Reset your login)

로그인 실패 원인이 불분명할 때 깨끗한 재인증으로 대부분 해결된다:
1. `/logout`으로 완전 로그아웃.
2. Claude Code 종료.
3. `claude`로 재시작해 인증 과정 다시 완료.

브라우저가 자동으로 안 열리면 `c`를 눌러 OAuth URL을 클립보드에 복사해 수동으로 붙여넣는다(좁은/SSH 터미널에서 URL이 줄바꿈돼 클릭 불가일 때도 유용).

#### OAuth error: Invalid code

`OAuth error: Invalid code. Please make sure the full code was copied`는 로그인 코드가 만료됐거나 복붙 중 잘렸다는 의미. (1) Enter로 재시도해 브라우저 열린 뒤 빠르게 완료, (2) 브라우저가 안 열리면 `c`로 전체 URL 복사, (3) 원격/SSH면 브라우저가 다른 머신에서 열릴 수 있으니 터미널의 URL을 로컬 브라우저에서 열기.

#### 로그인 후 403 Forbidden

`API Error: 403 {"error":{"type":"forbidden","message":"Request not allowed"}}`:
- **Claude Pro/Max**: [claude.ai/settings](https://claude.ai/settings)에서 구독 활성 확인.
- **Anthropic Console**: 계정에 "Claude Code" 또는 "Developer" 역할 확인(관리자가 Settings → Members에서 부여).
- **프록시 뒤**: 사내 프록시가 API 요청을 방해할 수 있음 ([[18 보안과 샌드박스]] / network-config).

#### 구독이 있는데 조직이 비활성화됨

`API Error: 400 ... "This organization has been disabled"`가 활성 구독에도 나오면, `ANTHROPIC_API_KEY` 환경변수가 구독을 덮어쓴 것이다(이전 직장/프로젝트의 옛 키가 쉘 프로필에 남은 경우가 흔함). `ANTHROPIC_API_KEY`가 있고 승인됐으면 OAuth 대신 그 키를 쓰며, `-p` 비대화형 모드에서는 키가 있으면 항상 사용된다(인증 우선순위는 authentication 문서).

```bash
unset ANTHROPIC_API_KEY
claude
```

영구 제거는 `~/.zshrc`, `~/.bashrc`, `~/.profile`의 `export ANTHROPIC_API_KEY=...`를 삭제. Windows는 PowerShell 프로필(`$PROFILE`)과 User 환경변수 확인. `/status`로 활성 인증 방식 확인.

#### WSL2/SSH/컨테이너에서 OAuth 로그인 실패

브라우저가 다른 호스트에서 열려 로컬 콜백 서버에 redirect가 못 닿는다. 로그인 후 브라우저에 코드가 표시되면 `Paste code here if prompted` 프롬프트에 붙여넣어 완료.

WSL2에서 브라우저가 아예 안 열리면:

```bash
export BROWSER="/mnt/c/Program Files/Google/Chrome/Application/chrome.exe"
claude
```

또는 `c`로 OAuth URL 복사, 혹은 `claude auth login`이 출력하는 URL을 로컬 브라우저에서 연다. 코드를 붙여넣어도 반응이 없으면 터미널의 paste 바인딩(우클릭/Shift+Insert)을 시도하거나, 표준입력에서 코드를 읽는 `claude auth login`을 쓴다:

```bash
claude auth login
```

#### 로그인 안 됨 또는 토큰 만료

세션 후 다시 로그인하라고 하면 OAuth 토큰이 만료된 것. `/login`으로 재인증. 자주 발생하면 토큰 검증이 정확한 타임스탬프에 의존하므로 **시스템 시계 정확성**을 확인.

macOS에서는 Keychain이 잠겼거나 비밀번호가 계정 비밀번호와 어긋나면 자격증명 저장이 막혀 로그인이 실패할 수 있다. `claude doctor`로 Keychain 접근 점검. 수동 잠금 해제:

```bash
security unlock-keychain ~/Library/Keychains/login.keychain-db
```

해제로도 안 되면 Keychain Access에서 `login` 키체인 선택 → Edit > Change Password로 계정 비밀번호와 재동기화.

#### Bedrock/Vertex/Foundry 자격증명 미로드

클라우드 프로바이더 사용 설정 후 `Could not load credentials from any providers`(Bedrock), `Could not load the default credentials`(Vertex), `ChainedTokenCredential authentication failed`(Foundry)가 나오면 현재 쉘에서 프로바이더 CLI가 인증되지 않은 것이다.

| 프로바이더 | 확인/인증 |
| :--- | :--- |
| Bedrock | `aws sts get-caller-identity` |
| Vertex AI | `ANTHROPIC_VERTEX_PROJECT_ID`, `CLOUD_ML_REGION` 설정 확인 후 `gcloud auth application-default login` |
| Foundry | `ANTHROPIC_FOUNDRY_API_KEY` 설정, 또는 `az login` |

터미널에선 되는데 VS Code/JetBrains 확장에서 안 되면 IDE 프로세스가 쉘 환경을 상속하지 못한 것이다. IDE 자체 설정에 프로바이더 환경변수를 넣거나, 변수가 export된 터미널에서 IDE를 실행. 전체 설정은 [[17 클라우드 프로바이더]] 참조.

### 여전히 막혔다면 (Still stuck)

1. [GitHub 저장소](https://github.com/anthropics/claude-code/issues) 알려진 이슈 확인 또는 OS, 실행한 설치 명령, 전체 에러 출력과 함께 새 이슈 등록.
2. `claude --version`은 되는데 다른 게 문제면 `claude doctor`로 자동 진단.
3. 세션을 시작할 수 있으면 `/feedback`으로 보고.

## 런타임 에러 레퍼런스 (errors)

Claude Code가 표시하는 런타임 에러와 각 복구 방법, 그리고 에러 없이 응답이 이상할 때 확인할 것들이다. 설치 에러(`command not found`, 설치 중 TLS 실패 등)는 `## 설치 및 로그인 문제 해결`.

> [!note]
> 이 에러·복구 명령은 CLI, [[14 IDE와 데스크톱]] Desktop 앱, [[15 웹과 모바일]] Claude Code 웹 모두에 적용된다(셋 다 같은 CLI를 감쌈). Claude Code는 모델 응답을 Claude API로 호출하므로 대부분의 런타임 에러는 기저 API 에러 코드로 매핑된다. 원시 HTTP 상태 코드 정의는 [Claude Platform error reference](https://platform.claude.com/docs/en/api/errors).

### 자동 재시도 (Automatic retries)

Claude Code는 에러를 보여주기 전에 일시적 실패를 재시도한다. 서버 에러, 과부하 응답, 요청 타임아웃, 일시적 429 throttle, 끊긴 연결은 모두 **지수 백오프로 최대 10회** 재시도한다. 재시도 중 스피너에 `Retrying in Ns · attempt x/y` 카운트다운이 표시된다. 이 페이지의 에러가 보였다는 건 이미 재시도가 소진됐다는 뜻이다.

| 환경변수 | 기본값 | 효과 |
| :--- | :--- | :--- |
| `CLAUDE_CODE_MAX_RETRIES` | 10 | 재시도 횟수. 스크립트에서 빨리 실패시키려면 낮추고, 긴 장애를 견디려면 높임 |
| `API_TIMEOUT_MS` | 600000 | 요청당 타임아웃(ms). 느린 네트워크/프록시면 상향 |

환경변수 상세는 [[04 설정]] 참조.

### 서버 에러 (Server errors)

계정/요청이 아닌 추론 프로바이더 측 문제다(Anthropic API면 Anthropic 인프라, Bedrock/Vertex/Foundry/커스텀 게이트웨이면 해당 프로바이더 인프라).

#### `API Error: 500 Internal server error`

```text
API Error: 500 Internal server error. This is a server-side issue, usually temporary — try again in a moment. If it persists, check https://status.claude.com.
```

API 내부의 예기치 못한 실패로, 프롬프트·설정·계정과 무관하다. 끝 문장의 상태 페이지는 프로바이더별로 다르다(커스텀 `ANTHROPIC_BASE_URL`이면 게이트웨이 호스트). 조치: [status.claude.com](https://status.claude.com) 확인, 1분 후 재전송(원문이 대화에 남아 있어 긴 프롬프트는 `try again`만 입력 가능), 게시된 장애 없이 지속되면 `/feedback`.

#### `API Error: Repeated 529 Overloaded errors`

```text
API Error: Repeated 529 Overloaded errors. The API is at capacity — this is usually temporary. Try again in a moment. If it persists, check https://status.claude.com.
```

전체 사용자에 걸친 일시적 용량 부족이다. 사용 한도가 아니며 쿼터에 카운트되지 않는다. 조치: status 확인, 몇 분 후 재시도, **용량은 모델별로 추적되므로 `/model`로 다른 모델로 전환**(`Opus is experiencing high load, please use /model to switch to Sonnet`처럼 안내됨).

#### `Request timed out`

API가 연결 마감 전에 응답하지 못함. 높은 부하나 매우 큰 응답 생성 시 발생. 기본 요청 타임아웃은 10분. 조치: 재시도, 긴 작업은 작은 프롬프트로 분할, 느린 네트워크/프록시면 `API_TIMEOUT_MS` 상향. 인터넷 연결 언급 메시지면 `### 네트워크와 연결 에러` 참조.

#### auto mode가 액션의 안전성을 판단 못함

[[05 권한]]의 auto mode가 액션을 분류하는 모델이 결정을 못 내려 자동 승인하지 않은 것. 작업 디렉터리 내 읽기·검색·편집은 분류기를 건너뛰므로 이 모든 경우에 계속 작동한다.

| 메시지 | 의미 | 조치 |
| :--- | :--- | :--- |
| `<model> is temporarily unavailable, so auto mode cannot determine the safety of <tool> right now...` | 분류기 모델 과부하 | 몇 초 후 재시도(Claude가 보통 알아서 재시도). 실패 지속 시 읽기 전용 작업 진행 후 나중에. 설정 변경 불필요 |
| `Auto mode could not evaluate this action and is blocking it for safety — run with --debug for details` | 분류기가 파싱 불가 응답 | 재시도(보통 다음 시도 성공). `claude --debug`로 분류기 응답 확인 |
| `Auto mode classifier transcript exceeded context window — falling back to manual approval (try /compact...)` | 대화가 분류기 컨텍스트 초과 | 대화형: 일반 권한 프롬프트로 폴백 → 수동 승인/거부. [[13 자동화와 스케줄링]] 비대화형(headless): transcript가 계속 커져 재시도 불가하므로 run 중단. `/compact`로 대화 축소 |

### 사용 한도 (Usage limits)

계정/플랜에 묶인 쿼터에 도달한 것. 모두에게 영향을 주는 서버 에러와는 다르다. 비용·플랜은 [[19 비용과 성능]] 참조.

#### `You've hit your session limit`

```text
You've hit your session limit · resets 3:45pm
You've hit your weekly limit · resets Mon 12:00am
You've hit your Opus limit · resets 3:45pm
```

리셋 시간까지 추가 요청이 차단된다. 조치:
- 메시지의 리셋 시간까지 대기.
- `/usage` — 플랜 한도와 리셋 시점 확인.
- `/usage-credits` — Pro/Max에서 추가 사용량 구매, Team/Enterprise는 관리자에게 요청.
- 기본 한도 상향은 [claude.com/pricing](https://claude.com/pricing).

남은 한도를 미리 보려면 [[12 출력 스타일과 상태줄]] 커스텀 상태줄에 `rate_limits` 필드 추가, 또는 Desktop 앱의 usage ring 클릭.

#### `Server is temporarily limiting requests`

```text
API Error: Server is temporarily limiting requests (not your usage limit)
```

플랜 쿼터와 무관한 단기 throttle이며 자동 재시도된다. 잠시 후 재시도, 지속되면 status 확인.

#### `Request rejected (429)`

API 키, Bedrock 프로젝트, Vertex AI 프로젝트에 설정된 rate limit에 도달.

```text
API Error: Request rejected (429) · this may be a temporary capacity issue. If it persists, check https://status.claude.com.
```

조치: `/status`로 활성 자격증명이 의도한 것인지 확인(떠도는 `ANTHROPIC_API_KEY`가 구독 대신 저티어 키로 라우팅할 수 있음), 프로바이더 콘솔에서 한도 확인·상향 요청, Anthropic API 키는 rate limits 레퍼런스 참조, 동시성 축소(`CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY` 하향, 병렬 서브에이전트 자제, 고볼륨 스크립트는 `/model`로 더 작은 모델).

#### `Credit balance is too low`

Console 조직의 선불 크레딧 소진. 조치: [platform.claude.com/settings/billing](https://platform.claude.com/settings/billing)에서 크레딧 추가(auto-reload 권장), Pro/Max/Team/Enterprise면 `/login`으로 구독 인증 전환, 워크스페이스별 spend cap 설정. 비용 관리는 [[19 비용과 성능]].

### 인증 에러 (Authentication errors)

Claude Code가 API에 신원을 증명하지 못한 것. 언제든 `/status`로 현재 활성 자격증명을 확인하라. 권한·인증 개념은 [[05 권한]] 참조.

#### `Not logged in`

```text
Not logged in · Please run /login
```

조치: `/login`으로 인증, 환경변수 인증을 기대했다면 `ANTHROPIC_API_KEY`가 `claude`를 실행한 쉘에 설정·export됐는지 확인, CI/자동화면 시작 시 키를 가져오는 [[04 설정]]의 `apiKeyHelper` 스크립트 구성. 반복 로그인 요구면 `### 로그인 안 됨 또는 토큰 만료`의 시계·Keychain 확인.

#### `Invalid API key`

```text
Invalid API key · Fix external API key
```

`ANTHROPIC_API_KEY` 또는 `apiKeyHelper`가 API가 거부한 키를 반환. 조치: 오타·폐기 여부 확인([Console](https://platform.claude.com/settings/keys)), `env | grep ANTHROPIC` 실행(direnv, dotenv 플러그인, IDE 터미널이 `.env`에서 stale 키를 로드할 수 있음), `ANTHROPIC_API_KEY` unset 후 `/login`, `apiKeyHelper`면 스크립트를 직접 실행해 stdout 확인, `/status`로 실제 사용 중인 자격증명 소스 확인.

#### `This organization has been disabled`

```text
Your ANTHROPIC_API_KEY belongs to a disabled organization · Unset the environment variable to use your other credentials
API Error: 400 ... This organization has been disabled.
```

비활성 Console 조직의 stale `ANTHROPIC_API_KEY`가 `/login`을 덮어쓴 것(환경변수가 `/login`보다 우선, `-p`에서는 항상 사용). 조치: 현재 쉘과 쉘 프로필에서 `ANTHROPIC_API_KEY` 제거 후 재시작, `/status`로 확인. 변수가 없는데도 지속되면 `/login` 계정에 묶인 조직이 비활성화된 것이니 지원에 문의하거나 다른 계정으로 로그인.

#### `Your organization has disabled Claude subscription access`

```text
Your organization has disabled Claude subscription access for Claude Code · Use an Anthropic API key instead, or ask your admin to enable access
```

조직이 구독 로그인을 허용하지 않는 서버 측 설정이라 로컬 설정·환경변수·CLI 플래그로 못 덮어쓴다. Agent SDK와 `-p` 모드에서는 `oauth_org_not_allowed` 에러 코드로 표시. 조치: 관리자에게 활성화 요청, Console API 키로 인증, 관리자인데 옵션이 없으면 [Anthropic 지원](https://support.claude.com).

#### `Routines are disabled by your organization's policy`

Team/Enterprise 관리자가 조직 레벨에서 routine을 끈 것. `/schedule`과 [[13 자동화와 스케줄링]]의 Routines UI에서 생성/실행 시 나온다. 서버 측 설정이라 로컬에서 못 덮어쓴다. 조치: 관리자에게 [claude.ai/admin-settings/claude-code](https://claude.ai/admin-settings/claude-code)의 **Routines** 토글 활성화 요청, 일회성 스케줄 작업은 scheduled tasks 참조.

#### `OAuth token revoked or expired`

```text
OAuth token revoked · Please run /login
OAuth token has expired · Please run /login
API Error: 401 ... authentication_error
```

저장된 로그인이 더 이상 유효하지 않음(revoked = 전체 로그아웃/관리자 접근 제거, expired = 자동 갱신 실패). 조치: `/login`, 같은 세션에서 재인증 후에도 반복되면 `/logout` 먼저 후 `/login`, 런치마다 반복되면 시계·Keychain 확인.

#### `OAuth scope requirement`

```text
OAuth token does not meet scope requirement: user:profile
```

저장된 토큰이 신기능이 필요로 하는 권한 scope 이전 것. `/usage`와 상태줄 usage 표시에서 자주 발생. 조치: `/login`으로 현재 scope의 새 토큰 발급(로그아웃 불필요).

### 네트워크와 연결 에러 (Network and connection errors)

Claude Code의 네트워크 요청이 목적지에 못 닿은 것. 보통 로컬 네트워크·프록시·방화벽 또는 클라우드 환경의 네트워크 정책에서 비롯된다.

#### `Unable to connect to API`

```text
Unable to connect to API. Check your internet connection
Unable to connect to API (ECONNREFUSED)
Unable to connect to API (ECONNRESET)
Unable to connect to API (ETIMEDOUT)
fetch failed
Request timed out. Check your internet connection and proxy settings
```

원인: 인터넷 없음, `api.anthropic.com`을 막는 VPN, 미구성된 사내 프록시. 조치:
- 같은 쉘에서 `curl -I https://api.anthropic.com` (Windows PowerShell은 `curl.exe -I https://api.anthropic.com`).
- 사내 프록시면 `HTTPS_PROXY` 설정 후 실행 (network-config).
- LLM 게이트웨이/릴레이 경유면 `ANTHROPIC_BASE_URL`을 그 주소로 설정 (llm-gateway).
- 방화벽이 network access requirements의 호스트를 허용하는지 확인.

`curl`은 되는데 Claude Code가 실패하면 런타임과 네트워크 사이가 원인:
- Linux/WSL: `/etc/resolv.conf`의 도달 불가 nameserver 확인(WSL은 깨진 resolver를 상속하기 쉬움).
- macOS: 끊기거나 제거된 VPN이 남긴 tunnel 인터페이스·라우팅 룰 — `ifconfig`로 stale `utun` 확인, 시스템 설정에서 VPN network extension 제거.
- Docker Desktop 등 컨테이너 런타임이 아웃바운드를 가로챌 수 있음 — 종료 후 재시도로 배제.

#### SSL certificate errors

```text
Unable to connect to API: SSL certificate verification failed. Check your proxy or corporate SSL certificates
Unable to connect to API: Self-signed certificate detected
```

프록시/보안 어플라이언스가 자체 인증서로 TLS를 가로채는데 Claude Code가 신뢰하지 않는 것. 조치: 조직 CA bundle을 export해 `NODE_EXTRA_CA_CERTS=/path/to/ca-bundle.pem`으로 지정, custom CA 설정은 [[18 보안과 샌드박스]] / network-config 참조.

> [!warning]
> `NODE_TLS_REJECT_UNAUTHORIZED=0`은 인증서 검증을 완전히 끄므로 설정하지 마라.

#### 클라우드 세션에서 `Host not allowed`

```text
HTTP 403
x-deny-reason: host_not_allowed
```

클라우드 세션/routine의 아웃바운드 HTTP 요청이 환경의 네트워크 정책에 차단된 것. 목적지의 실제 인증서와 안 맞는 TLS 인증서도 보일 수 있다(프록시가 연결을 종료했다는 신호). 클라이언트 측 문제가 아니다. 클라우드 세션과 routine은 샌드박스 환경에서 실행되고 아웃바운드가 allowlist로 필터된다. **Default** 환경은 **Trusted** 접근으로 패키지 레지스트리·클라우드 API·컨테이너 레지스트리·일반 개발 도메인의 default allowlist만 허용한다.

조치: routine 편집 또는 클라우드 세션 시작 → 환경 이름(예 **Default**) 클라우드 아이콘 → 환경 위 settings 아이콘 → **Update cloud environment** 대화상자에서 **Network access**를 **Trusted**에서 **Custom**으로 변경, 차단 도메인을 **Allowed domains**에 한 줄씩 추가(**Also include default list of common package managers** 체크로 default allowlist 유지, 무제한은 **Full**) → **Save changes**. 자세한 접근 레벨과 default allowlist는 [[15 웹과 모바일]] 참조. 로컬 CLI 세션은 이 정책의 영향을 받지 않는다.

### 요청 에러 (Request errors)

API가 요청을 받았지만 내용을 거부한 것. 컨텍스트 관리는 [[03 메모리와 컨텍스트]] 참조.

#### `Prompt is too long`

대화 + 첨부 파일이 모델의 컨텍스트 윈도를 초과. 조치:
- `/compact`(이전 턴 요약) 또는 `/clear`(새 시작).
- `/context` — 시스템 프롬프트, 도구, 메모리 파일, 메시지가 윈도를 얼마나 차지하는지 분해.
- `/mcp disable <name>` — 안 쓰는 [[10 MCP]] 서버의 도구 정의 제거.
- 큰 `CLAUDE.md` 메모리 파일 정리, 또는 관련 시에만 로드되는 path-scoped rules로 이동.
- 서브에이전트는 부모의 모든 MCP 도구 정의를 상속하므로 첫 턴 전에 컨텍스트가 찰 수 있다 — 스폰 전 안 쓰는 MCP 서버 비활성화.
- 자동 압축은 기본 on이라 보통 이 에러를 막는다. `DISABLE_AUTO_COMPACT`를 설정했다면 재활성화하거나 윈도가 차기 전 수동 `/compact`.

#### `Error during compaction: Conversation too long`

```text
Error during compaction: Conversation too long. Press esc twice to go up a few messages and try again.
```

`/compact` 자체가 실패 — 요약을 담을 여유 컨텍스트가 부족한 것. 윈도가 이미 꽉 찬 상태에서 auto-compact가 트리거되거나 `Prompt is too long` 후 `/compact`를 실행할 때 발생. 조치: Esc 두 번으로 메시지 목록 열어 몇 턴 뒤로(최근 메시지를 컨텍스트에서 제거) 후 `/compact` 재시도. 그래도 부족하면 `/clear`(이전 대화는 보존돼 `/resume`으로 재개 가능).

#### `Request too large`

```text
Request too large (max 30 MB). Double press esc to go back and remove or shrink the attached content.
```

raw 요청 본문이 토큰화 전 API의 바이트 한도(30MB)를 초과 — 보통 큰 파일/첨부 붙여넣기. 컨텍스트 윈도 한도와는 별개의 HTTP 요청 크기 제한이다. 조치: Esc 두 번으로 그 턴 이전으로, 큰 파일은 붙여넣지 말고 경로로 참조(Claude가 청크로 읽음), 이미지는 아래 참조.

#### `Image was too large`

```text
Image was too large. Double press esc to go back and try again with a smaller image.
API Error: 400 ... image dimensions exceed max allowed size
```

이미지가 API의 크기/치수 한도 초과. 에러 후에도 이미지가 히스토리에 남아 제거 전까지 이후 모든 메시지가 같은 에러로 실패한다. 조치: Esc 두 번으로 이미지 추가 턴 이전으로, 붙여넣기 전 리사이즈(단일 이미지는 가장 긴 변 8000px까지, 여러 이미지면 2000px), 전체 화면 대신 관련 영역만 타이트하게 캡처.

#### `Unable to resize image`

```text
Unable to resize image — image processing is unavailable and dimensions could not be read from the file header. Please convert the image to PNG, JPEG, GIF, or WebP.
Unable to resize image — dimensions exceed the 2000x2000px limit and image processing failed. Please resize the image to reduce its pixel dimensions.
Unable to resize image (… raw, … base64). The image exceeds the … API limit and compression failed. Please resize the image manually or use a smaller image.
Unable to resize image — could not verify image dimensions are within the 2000x2000px API limit.
```

Claude Code가 전송 전 다운스케일을 못한 것(native 이미지 프로세서 로드 실패/에러). 조치: 변환 요청 메시지면 PNG/JPEG/GIF/WebP로 변환 후 재첨부(이 포맷은 프로세서 없이 치수 검증 가능), 치수/크기 한도 보고면 그 이하로 리사이즈/재압축.

#### PDF errors

```text
PDF too large (max 100 pages, 32 MB). Try splitting it or extracting text first.
PDF is password protected. Try removing protection or extracting text first.
The PDF file was not valid. Try converting to a different format first.
```

조치: 큰 PDF는 [[06 내장 도구 레퍼런스]] Read 도구로 페이지 범위만 읽게 하거나 `pdftotext`로 텍스트 추출 후 경로 참조, 보호/무효 PDF는 비밀번호 제거하거나 원본 앱에서 재내보내기.

#### `Extra inputs are not permitted`

```text
API Error: 400 ... Extra inputs are not permitted ... context_management
API Error: 400 ... Extra inputs are not permitted ... tools.0.custom.input_examples
API Error: 400 ... Unexpected value(s) for the `anthropic-beta` header
```

프록시/LLM 게이트웨이가 `anthropic-beta` 요청 헤더를 제거해 그에 의존하는 필드를 API가 거부한 것. Claude Code는 `context_management`, `effort`, tool `input_examples` 같은 beta 전용 필드를 헤더와 함께 보내는데 게이트웨이가 본문만 전달하고 헤더를 떨어뜨리면 발생. 조치: 게이트웨이가 `anthropic-beta` 헤더를 전달하도록 구성(llm-gateway), 폴백으로 실행 전 `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1` 설정(beta 헤더 필요 기능 비활성화).

#### 선택한 모델에 문제가 있음 (There's an issue with the selected model)

```text
There's an issue with the selected model (claude-...). It may not exist or you may not have access to it. Run /model to pick a different model.
```

설정된 모델명이 인식 안 되거나 계정에 접근 권한 없음. v2.1.160부터 끝의 힌트는 surface별로 다르다.

| Surface | 조치 |
| :--- | :--- |
| 대화형 CLI | `/model`로 계정에서 사용 가능한 모델 선택 |
| 비대화형 (`-p`) | `--model`에 유효한 alias/ID 전달 또는 `ANTHROPIC_MODEL` 설정 (힌트는 `Run --model`) |
| Agent SDK | TS `model` on `Options`, Python `ClaudeAgentOptions(model=...)`, 구조화된 `model_not_found` 에러 처리 — [[24 Agent SDK — 고급과 레퍼런스]] |

가급적 full 버전 ID 대신 `sonnet`/`opus` alias 사용(유지되는 default로 resolve돼 stale 안 됨). CLI에서 잘못된 모델이 계속 나오면 어딘가에 stale ID가 있는 것 — 우선순위 순으로 확인: `--model` 플래그 → `ANTHROPIC_MODEL` 환경변수 → `.claude/settings.local.json`의 `model` → 프로젝트 `.claude/settings.json` → `~/.claude/settings.json`. stale 값을 제거하면 계정 default로 폴백. Vertex AI는 [[17 클라우드 프로바이더]] 트러블슈팅 참조.

#### `Claude Opus is not available with the Claude Pro plan`

```text
Claude Opus is not available with the Claude Pro plan · Select a different model in /model
```

활성 구독 플랜이 선택 모델을 포함하지 않음. 조치: `/model`로 플랜 포함 모델 선택, 최근 업그레이드했는데도 나오면 `/logout` 후 `/login`(저장 토큰이 로그인 당시 플랜을 반영하므로 웹에서 업그레이드해도 기존 세션엔 재인증 전까지 미반영), 플랜별 모델은 [claude.com/pricing](https://claude.com/pricing).

#### `thinking.type.enabled is not supported for this model`

```text
API Error: 400 ... "thinking.type.enabled" is not supported for this model. Use "thinking.type.adaptive" and "output_config.effort" to control thinking behavior.
```

Claude Code 버전이 Opus 4.7/4.8 최소 버전보다 오래됨(모델이 더 이상 받지 않는 thinking 구성을 보냄). 조치: `claude update` 후 재시작(Opus 4.7은 v2.1.111+, Opus 4.8은 v2.1.154+), 업그레이드 불가면 `/model`로 Opus 4.6/Sonnet 선택. Agent SDK는 [[22 Agent SDK — 시작]] 트러블슈팅 참조.

#### Thinking budget이 출력 한도를 초과

```text
API Error: 400 ... max_tokens must be greater than thinking.budget_tokens
```

확장 thinking budget이 최대 응답 길이를 초과해 답변 공간이 없음. Anthropic API에서는 자동 조정되지만, Bedrock/Vertex에서 `MAX_THINKING_TOKENS`가 프로바이더 출력 한도보다 높거나 plan mode가 budget을 올릴 때 발생. 조치: `MAX_THINKING_TOKENS` 하향 또는 `CLAUDE_CODE_MAX_OUTPUT_TOKENS`를 budget 위로 상향. extended thinking은 model-config 참조.

#### Tool use 또는 thinking block 불일치

```text
API Error: 400 due to tool use concurrency issues. Run /rewind to recover the conversation.
API Error: 400 ... unexpected `tool_use_id` found in `tool_result` blocks
API Error: 400 ... thinking blocks ... cannot be modified
```

대화 히스토리가 API에 불일치 상태로 도달한 것(보통 tool call이 중단되거나 턴이 중간 편집됨). 세 변종 모두 history의 `tool_use`/`tool_result`/`thinking` 블록 순서가 API 기대와 안 맞는 것. 조치: Opus 4.7/4.8 사용 중이면 `claude update` 먼저(v2.1.156 이전은 정상 도구 사용 중에도 이 에러를 유발하고 `/rewind`로 안 지워짐), `/rewind` 또는 Esc 두 번으로 손상 턴 이전 체크포인트로 복귀. checkpointing은 [[20 베스트 프랙티스와 워크플로우]] 참조.

#### Usage Policy 거부

```text
API Error: Claude Code is unable to respond to this request, which appears to violate our Usage Policy (https://www.anthropic.com/legal/aup). Please double press esc to edit your last message or start a new session for Claude Code to assist with a different task.
```

대화 내용이 [Usage Policy](https://www.anthropic.com/legal/aup) 검사를 트리거해 API가 응답을 거부. 메시지에 Request ID가 포함돼 부당하다고 보면 지원에 인용 가능. 검사는 최신 프롬프트가 아닌 전체 대화를 평가하므로 같은 세션에서 새 메시지를 보내면 보통 같은 거부가 재발하고, `--continue`/`--resume`으로 재개해도 디스크 transcript에 트리거 내용이 남아 마찬가지다. 조치: Esc 두 번 또는 `/rewind`로 트리거 턴 이전 체크포인트로 복귀 후 다른 접근, 어느 턴인지 모르면 `/clear`(이전 대화는 디스크에 보존돼 `/resume` 가능), `-p` 비대화형은 rewind 불가하므로 rephrase하거나 `--continue` 없이 새 세션.

### 응답 품질이 평소보다 낮아 보일 때

에러 없이 답변이 덜 유능해 보이면 원인은 보통 모델 자체가 아니라 대화 상태다. Claude Code는 조용히 모델 버전을 바꾸지 않지만, Opus 쿼터 도달이나 Bedrock/Vertex 리전에 모델이 없는 등 특정 경우 폴백 모델로 전환할 수 있다(아래 Model selection 체크가 둘 다 잡아냄). 먼저 확인할 것:

| 체크 | 명령 | 설명 |
| :--- | :--- | :--- |
| Model selection | `/model` | 기대한 모델인지 확인. 이전 `/model` 선택이나 `ANTHROPIC_MODEL`이 더 작은 모델로 만들었을 수 있음 |
| Effort level | `/effort` | 현재 추론 레벨 확인, 어려운 디버깅/설계는 상향. default는 모델별로 다름. `ultrathink` 단축 포함 |
| Context pressure | `/context` | 윈도가 얼마나 찼는지. 한계 근처면 자연스러운 분기점에서 `/compact` 또는 `/clear` |
| Stale instructions | `/doctor`, `/context` | 크고 오래된 `CLAUDE.md`와 MCP 도구 정의가 컨텍스트를 먹고 응답을 왜곡. `/doctor`가 oversized 메모리·서브에이전트를 플래그, `/context`가 MCP 토큰 사용 표시 |

응답이 잘못되면 in-thread로 정정하기보다 **rewind가 더 잘 통한다**(Esc 두 번 또는 `/rewind`로 나쁜 턴 이전으로 가서 구체적으로 rephrase). 정정은 잘못된 시도를 컨텍스트에 남겨 이후 답변을 거기에 anchor시킬 수 있다. 그래도 이상하면 `/feedback`(transcript 포함, 실제 regression 진단에 가장 빠름). 모델·effort·컨텍스트 개념은 [[03 메모리와 컨텍스트]], [[19 비용과 성능]], [[20 베스트 프랙티스와 워크플로우]] 참조.

### 에러 보고 (Report an error)

이 페이지는 Claude API 에러를 다룬다. 다른 컴포넌트 에러:

| 에러 출처 | 참고 |
| :--- | :--- |
| MCP 서버 연결/인증 실패 | [[10 MCP]] |
| 훅 스크립트 실패/도구 차단 | [[09 훅]] (debug hooks) |
| 설치 중 권한 거부/파일 시스템 에러 | `## 설치 및 로그인 문제 해결` |

목록에 없거나 제안된 수정이 안 통하면:
- `/feedback` — transcript + 설명을 Anthropic에 전송(prefilled GitHub 이슈 제안도 함). Bedrock/Vertex/Foundry 등 서드파티에서는 로컬 아카이브를 저장해 계정 담당자에게 전달.
- `/doctor` — 로컬 구성 문제 점검.
- [status.claude.com](https://status.claude.com) — 활성 장애 확인.
- [GitHub 이슈](https://github.com/anthropics/claude-code/issues) 검색.
