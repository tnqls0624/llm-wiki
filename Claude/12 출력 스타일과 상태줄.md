# 12 출력 스타일과 상태줄

이 노트는 Claude Code의 두 가지 UX 커스터마이징 기능을 다룬다. **출력 스타일(output styles)** 은 Claude가 *어떻게* 응답하는지(역할·톤·포맷)를 시스템 프롬프트 수준에서 바꾸고, **상태줄(status line)** 은 화면 하단에 컨텍스트 사용량·비용·git 상태 등을 상시 표시한다. 둘 다 허브 [[Claude]]에 연결된다.

---

## 출력 스타일 (Output styles)

출력 스타일은 Claude가 **무엇을 아는지가 아니라 어떻게 응답하는지**를 바꾼다. 시스템 프롬프트를 수정해 역할·톤·출력 포맷을 지정한다. 매 턴 같은 목소리/포맷을 얻으려고 반복해서 프롬프트를 다시 쓰게 될 때, 또는 Claude를 소프트웨어 엔지니어가 아닌 다른 역할(작문 보조, 데이터 분석가 등)로 쓰고 싶을 때 사용한다.

커스텀 출력 스타일은 사용자 지시를 시스템 프롬프트에 추가하면서, Claude Code의 **내장 소프트웨어 엔지니어링 지시문**을 유지할지 말지를 선택하게 해준다.
- **유지(keep)**: 의사소통 방식만 바꾸고 여전히 코딩은 그대로 시키고 싶을 때 (예: 항상 다이어그램으로 설명 시작).
- **제외(leave out)**: Claude가 소프트웨어 엔지니어링을 전혀 하지 않을 때 (예: 작문 보조, 데이터 분석가).

> [!note] 출력 스타일 vs CLAUDE.md
> 프로젝트·컨벤션·코드베이스에 관한 지시는 출력 스타일이 아니라 [[03 메모리와 컨텍스트]]의 CLAUDE.md를 사용한다. 출력 스타일은 "어떻게 말하느냐"를, CLAUDE.md는 "프로젝트에 대해 무엇을 아느냐"를 담당한다.

### 내장 출력 스타일 (Built-in output styles)

기본 **Default** 스타일은 기존 시스템 프롬프트로, 소프트웨어 엔지니어링 작업을 효율적으로 수행하도록 설계됐다. 추가로 3개의 내장 스타일이 있다.

| 스타일 | 동작 |
| :--- | :--- |
| **Default** | 기존 시스템 프롬프트. 소프트웨어 엔지니어링 작업 최적화 |
| **Proactive** | 즉시 실행하고, 일상적 결정에서 멈추지 않고 합리적 가정을 내리며, 계획보다 행동을 선호. [[05 권한]]의 auto mode보다 강한 자율 실행 가이드지만 권한 모드를 바꾸지 않으므로 도구 실행 전 권한 프롬프트는 그대로 뜬다 |
| **Explanatory** | 작업 중간중간 교육적 "Insights"를 제공. 구현 선택과 코드베이스 패턴을 이해하도록 도움 |
| **Learning** | 협업형 learn-by-doing 모드. "Insights"를 공유할 뿐 아니라 사용자에게 작고 전략적인 코드 조각을 직접 작성하도록 요청하며, 코드에 `TODO(human)` 마커를 남겨 사용자가 구현하게 함 |

> [!warning] 토큰 사용량 주의
> Explanatory와 Learning은 설계상 Default보다 긴 응답을 생성하므로 출력 토큰이 증가한다. 비용 영향은 [[19 비용과 성능]] 참고.

### 출력 스타일 변경하기

`/config`를 실행하고 **Output style**을 선택해 메뉴에서 고른다. 선택은 [[04 설정]]의 로컬 프로젝트 레벨인 `.claude/settings.local.json`에 저장된다.

메뉴 없이 설정 파일에서 `outputStyle` 필드를 직접 지정할 수도 있다.

```json
{
  "outputStyle": "Explanatory"
}
```

> [!note] 적용 시점과 캐싱
> 출력 스타일은 시스템 프롬프트의 일부이며, Claude Code는 이를 **세션 시작 시 한 번** 읽는다. 변경은 `/clear` 후 또는 새 세션부터 적용된다. 출력 스타일 변경이 프롬프트 캐시에 미치는 영향은 [[19 비용과 성능]]을 참고.

> [!warning] `/output-style` 명령 폐지
> 독립 명령 `/output-style`은 v2.1.73에서 deprecated, v2.1.91에서 제거됐다(max-version 2.1.90 기준). 대신 `/config`를 쓰거나 `outputStyle` 설정을 직접 편집한다. CLI/슬래시 명령 전반은 [[02 CLI 레퍼런스]] 참고.

### 커스텀 출력 스타일 만들기

커스텀 출력 스타일은 frontmatter(메타데이터) + 시스템 프롬프트에 추가할 지시문으로 구성된 Markdown 파일이다.

1. **Markdown 파일 생성** — 세 레벨 중 한 곳에 저장. 파일명이 스타일 이름이 되지만 frontmatter의 `name`으로 덮어쓸 수 있다.

| 레벨 | 경로 |
| :--- | :--- |
| User | `~/.claude/output-styles` |
| Project | `.claude/output-styles` |
| Managed policy | managed settings 디렉터리 내부의 `.claude/output-styles` |

2. **frontmatter + 지시문 추가** — 코딩 지시문 유지 여부를 결정. 의사소통 방식만 바꾸고 코딩은 유지하려면 `keep-coding-instructions: true`. 엔지니어링을 안 할 거면 생략.

다음 예시는 코딩 동작은 유지하면서 모든 설명을 다이어그램으로 시작하게 한다.

```markdown
---
name: Diagrams first
description: Lead every explanation with a diagram
keep-coding-instructions: true
---

When explaining code, architecture, or data flow, start with a Mermaid diagram showing the structure, then explain in prose.

## Diagram conventions

Use `flowchart TD` for control flow and `sequenceDiagram` for request paths. Keep diagrams under 15 nodes.
```

3. **스타일로 전환** — `/config`를 실행해 **Output style** 아래에서 선택. `/clear` 후 또는 다음 세션 시작 시 적용.

[[11 플러그인]]은 `output-styles/` 디렉터리에 출력 스타일을 함께 배포할 수도 있다.

### Frontmatter 필드

| Frontmatter | 용도 | 기본값 |
| :--- | :--- | :--- |
| `name` | 출력 스타일 이름 (파일명 대신 사용) | 파일명에서 상속 |
| `description` | `/config` 선택기에 표시되는 설명 | None |
| `keep-coding-instructions` | Claude Code 내장 소프트웨어 엔지니어링 지시문 유지 | `false` |
| `force-for-plugin` | 플러그인 출력 스타일 전용. 플러그인이 활성화되면 사용자가 선택하지 않아도 이 스타일을 자동 적용하고 사용자의 `outputStyle` 설정을 덮어씀. 여러 플러그인이 설정하면 먼저 로드된 것을 사용 | `false` |

### 동작 방식 (How output styles work)

출력 스타일은 Claude Code의 시스템 프롬프트를 **직접 수정**한다.

- 모든 출력 스타일은 자체 커스텀 지시문을 시스템 프롬프트 **끝**에 추가한다.
- 모든 출력 스타일은 대화 중 스타일 지시문을 지키도록 Claude에게 리마인더를 트리거한다.
- 커스텀 출력 스타일은 `keep-coding-instructions: true`가 아닌 한, Claude Code의 내장 엔지니어링 지시문(변경 범위 산정, 주석 작성법, 작업 검증 방법 등)을 **제외**한다.

토큰 사용량은 스타일에 따라 다르다. 시스템 프롬프트에 지시를 추가하면 입력 토큰이 늘지만, 세션 첫 요청 이후에는 프롬프트 캐싱이 이 비용을 줄여준다. 커스텀 스타일의 출력 토큰은 지시문이 무엇을 생성하라고 했는지에 달려 있다.

### 관련 기능과의 비교

여러 기능이 Claude Code의 동작을 커스터마이즈한다. 출력 스타일은 시스템 프롬프트를 직접 수정하고 모든 응답에 적용된다. 나머지는 기본 시스템 프롬프트를 바꾸지 않고 지시를 추가하거나, 특정 작업으로 범위를 한정한다.

| 기능 | 작동 방식 | 사용 시점 |
| :--- | :--- | :--- |
| **출력 스타일** | 시스템 프롬프트를 수정 | 매 턴 다른 역할·톤·기본 응답 포맷을 원할 때 |
| **CLAUDE.md** ([[03 메모리와 컨텍스트]]) | 시스템 프롬프트 뒤에 user 메시지를 추가 | Claude가 항상 프로젝트 컨벤션·코드베이스 컨텍스트를 알아야 할 때 |
| `--append-system-prompt` | 아무것도 제거하지 않고 시스템 프롬프트에 추가 ([[02 CLI 레퍼런스]]) | 단일 호출에 일회성 추가를 원할 때 |
| **에이전트** ([[08 서브에이전트와 에이전트 팀]]) | 자체 시스템 프롬프트·모델·도구를 가진 서브에이전트 실행 | 집중된 작업을 위한 별도 범위의 헬퍼를 원할 때 |
| **스킬** ([[07 스킬]]) | 호출되거나 관련될 때 작업별 지시를 로드 | 재사용 가능한 워크플로우가 있을 때 |

> [!note] 관련 리소스
> 출력 스타일이 안 먹힐 때 진단은 [[25 트러블슈팅]], 설정 우선순위와 `outputStyle` 위치는 [[04 설정]], Proactive 스타일 vs auto mode는 [[05 권한]] 참고.

## 상태줄 (Status line)

상태줄은 Claude Code 화면 하단의 커스터마이즈 가능한 바로, 사용자가 설정한 **임의의 쉘 스크립트**를 실행한다. stdin으로 JSON 세션 데이터를 받아 스크립트가 출력하는 것을 그대로 표시하므로, 컨텍스트 사용량·비용·git 상태 등을 한눈에 상시 모니터링할 수 있다.

상태줄이 유용한 경우:
- 작업하면서 컨텍스트 윈도우 사용량 모니터링
- 세션 비용 추적
- 여러 세션을 오가며 구분
- git 브랜치·상태를 항상 표시

> [!note] 상태줄은 로컬에서 실행되며 API 토큰을 소비하지 않는다.
> 자동완성 제안, 도움말 메뉴, 권한 프롬프트 등 일부 UI 상호작용 중에는 일시적으로 숨겨진다.

### 상태줄 설정하기

두 가지 방법: `/statusline` 명령으로 Claude가 스크립트를 생성하게 하거나, 직접 스크립트를 만들어 설정에 추가한다.

#### `/statusline` 명령 사용

`/statusline` 명령은 표시할 내용을 자연어로 받는다. Claude Code가 `~/.claude/`에 스크립트 파일을 생성하고 설정을 자동 업데이트한다.

```text
/statusline show model name and context percentage with a progress bar
```

#### 수동 설정 (`statusLine` 필드)

user 설정(`~/.claude/settings.json`) 또는 프로젝트 설정([[04 설정]])에 `statusLine` 필드를 추가한다. `type`을 `"command"`로, `command`를 스크립트 경로 또는 인라인 쉘 명령으로 지정한다.

```json
{
  "statusLine": {
    "type": "command",
    "command": "~/.claude/statusline.sh",
    "padding": 2
  }
}
```

`command`는 쉘에서 실행되므로 스크립트 파일 대신 인라인 명령도 가능하다. 다음은 `jq`로 JSON을 파싱해 모델명과 컨텍스트 퍼센트를 표시한다.

```json
{
  "statusLine": {
    "type": "command",
    "command": "jq -r '\"[\\(.model.display_name)] \\(.context_window.used_percentage // 0)% context\"'"
  }
}
```

**`statusLine` 옵션 필드**

| 필드 | 설명 | 기본값 |
| :--- | :--- | :--- |
| `type` | `"command"` (쉘 명령 실행) | (필수) |
| `command` | 스크립트 경로 또는 인라인 쉘 명령 | (필수) |
| `padding` | 상태줄 내용에 추가 수평 여백(문자 단위). 인터페이스 내장 여백에 *더해지는* 상대 들여쓰기 | `0` |
| `refreshInterval` | 이벤트 기반 업데이트에 더해 N초마다 명령 재실행. 최소 `1`. 시계 등 시간 기반 데이터나, 메인 세션이 idle인 동안 백그라운드 서브에이전트가 git 상태를 바꾸는 경우에 설정 | (미설정 시 이벤트에만 실행) |
| `hideVimModeIndicator` | 프롬프트 아래 내장 `-- INSERT --` 텍스트를 숨김. 스크립트가 `vim.mode`를 직접 렌더링할 때 중복 표시 방지용으로 `true` 설정 | `false` |

#### 상태줄 비활성화

`/statusline`을 실행하고 제거를 요청한다(예: `/statusline delete`, `/statusline clear`, `/statusline remove it`). 또는 settings.json에서 `statusLine` 필드를 직접 삭제한다.

### 단계별로 만들기 (Build step by step)

> [!note] `/statusline`에 원하는 바를 설명하면 아래 전부를 자동 구성해준다. 다음은 내부 동작 이해용 수동 과정이다.
> 이 예시들은 macOS·Linux용 Bash 스크립트다. Windows는 아래 [Windows 설정](#windows-설정) 참고.

1. **JSON을 읽고 출력하는 스크립트 작성** — Claude Code가 stdin으로 JSON을 보낸다. `jq`(CLI JSON 파서, 별도 설치 필요할 수 있음)로 모델명·디렉터리·컨텍스트 퍼센트를 추출. `~/.claude/statusline.sh`에 저장.

```bash
#!/bin/bash
# Read JSON data that Claude Code sends to stdin
input=$(cat)

# Extract fields using jq
MODEL=$(echo "$input" | jq -r '.model.display_name')
DIR=$(echo "$input" | jq -r '.workspace.current_dir')
# The "// 0" provides a fallback if the field is null
PCT=$(echo "$input" | jq -r '.context_window.used_percentage // 0' | cut -d. -f1)

# Output the status line - ${DIR##*/} extracts just the folder name
echo "[$MODEL] 📁 ${DIR##*/} | ${PCT}% context"
```

2. **실행 권한 부여**

```bash
chmod +x ~/.claude/statusline.sh
```

3. **설정에 추가** — `~/.claude/settings.json`에 `statusLine`을 추가. 설정은 자동 reload되지만 변경은 다음 상호작용부터 반영된다.

```json
{
  "statusLine": {
    "type": "command",
    "command": "~/.claude/statusline.sh"
  }
}
```

### 동작 원리 (How status lines work)

Claude Code가 스크립트를 실행하고 stdin으로 JSON 세션 데이터를 파이프한다. 스크립트는 JSON을 읽어 필요한 것을 추출하고 stdout에 텍스트를 출력하면, Claude Code가 그것을 표시한다.

#### 업데이트 시점

스크립트는 다음 시점에 실행된다.
- 새 assistant 메시지가 나온 후
- `/compact`가 끝난 후
- 권한 모드가 바뀔 때
- vim 모드가 토글될 때

업데이트는 **300ms 디바운스**되어 빠른 변경이 묶여 안정되면 한 번 실행된다. 스크립트 실행 중 새 업데이트가 트리거되면 진행 중인 실행은 취소된다. 스크립트를 편집해도 다음 상호작용이 업데이트를 트리거할 때까지 변경이 보이지 않는다.

메인 세션이 idle일 때(예: coordinator가 백그라운드 서브에이전트를 기다림) 이 트리거가 멈출 수 있다. idle 동안 시간 기반·외부 소스 세그먼트를 최신으로 유지하려면 `refreshInterval`로 고정 타이머에서도 재실행하게 한다.

#### 스크립트가 출력할 수 있는 것

| 출력 | 방법 |
| :--- | :--- |
| **여러 줄** | 각 `echo`/`print` 문이 별도 행으로 표시 (아래 멀티라인 예시) |
| **색상** | ANSI escape code(예: `\033[32m` 초록). 터미널 지원 필요 |
| **링크** | OSC 8 escape 시퀀스로 클릭 가능(macOS Cmd+클릭, Windows/Linux Ctrl+클릭). iTerm2·Kitty·WezTerm 등 하이퍼링크 지원 터미널 필요 |

#### 터미널 크기에 맞추기

Claude Code는 스크립트 출력을 터미널에 직접 연결하지 않고 캡처하므로, 스크립트 안에서 `tput cols`나 언어 레벨 너비 감지로 터미널 크기를 읽을 수 없다. 대신 `COLUMNS`·`LINES` 환경변수를 읽는다. Claude Code가 스크립트 실행 전에 현재 터미널 치수를 이 변수에 설정한다. **(Claude Code v2.1.153 이상 필요)**

#### 컨텍스트 윈도우 필드

`context_window` 객체는 가장 최근 API 응답의 라이브 컨텍스트 윈도우를 설명한다. v2.1.132부터 `total_input_tokens`·`total_output_tokens`는 누적 세션 합계가 아니라 **현재** 컨텍스트 사용량을 반영한다.

- **결합 합계** (`total_input_tokens`, `total_output_tokens`): 현재 컨텍스트 윈도우의 토큰. `total_input_tokens`는 `input_tokens` + `cache_creation_input_tokens` + `cache_read_input_tokens`의 합, `total_output_tokens`는 최근 응답의 output 토큰. 첫 API 응답 전엔 둘 다 `0`.
- **컴포넌트별 사용량** (`current_usage`): 카테고리별로 분해한 동일 토큰 수. cache hit를 fresh input과 분리해야 할 때 사용.

`current_usage` 객체 내용:

| 필드 | 의미 |
| :--- | :--- |
| `input_tokens` | 현재 컨텍스트의 input 토큰 |
| `output_tokens` | 생성된 output 토큰 |
| `cache_creation_input_tokens` | 캐시에 쓰인 토큰 |
| `cache_read_input_tokens` | 캐시에서 읽은 토큰 |

cache 필드의 의미와 과금은 [[19 비용과 성능]]의 프롬프트 캐싱을 참고. `used_percentage`는 **input 토큰만**으로 계산된다: `input_tokens + cache_creation_input_tokens + cache_read_input_tokens`. output_tokens는 포함하지 않는다. 직접 계산할 때도 같은 input-only 공식을 써야 `used_percentage`와 일치한다.

### 예시 (Examples)

각 예시는 (1) `~/.claude/statusline.sh`(또는 `.py`/`.js`)에 저장, (2) `chmod +x`, (3) [설정](#수동-설정-statusline-필드)에 경로 추가로 사용한다. Bash는 `jq`로 JSON 파싱, Python·Node.js는 내장 파싱을 쓴다.

#### 컨텍스트 윈도우 사용량 (진행 바)

`used_percentage`를 읽어 10칸 진행 바를 만든다.

```bash
#!/bin/bash
input=$(cat)

MODEL=$(echo "$input" | jq -r '.model.display_name')
PCT=$(echo "$input" | jq -r '.context_window.used_percentage // 0' | cut -d. -f1)

# Build progress bar: printf -v creates a run of spaces, then
# ${var// /▓} replaces each space with a block character
BAR_WIDTH=10
FILLED=$((PCT * BAR_WIDTH / 100))
EMPTY=$((BAR_WIDTH - FILLED))
BAR=""
[ "$FILLED" -gt 0 ] && printf -v FILL "%${FILLED}s" && BAR="${FILL// /▓}"
[ "$EMPTY" -gt 0 ] && printf -v PAD "%${EMPTY}s" && BAR="${BAR}${PAD// /░}"

echo "[$MODEL] $BAR $PCT%"
```

```python
#!/usr/bin/env python3
import json, sys

data = json.load(sys.stdin)
model = data['model']['display_name']
# "or 0" handles null values
pct = int(data.get('context_window', {}).get('used_percentage', 0) or 0)

filled = pct * 10 // 100
bar = '▓' * filled + '░' * (10 - filled)
print(f"[{model}] {bar} {pct}%")
```

```javascript
#!/usr/bin/env node
let input = '';
process.stdin.on('data', chunk => input += chunk);
process.stdin.on('end', () => {
    const data = JSON.parse(input);
    const model = data.model.display_name;
    // Optional chaining (?.) safely handles null fields
    const pct = Math.floor(data.context_window?.used_percentage || 0);
    const filled = Math.floor(pct * 10 / 100);
    const bar = '▓'.repeat(filled) + '░'.repeat(10 - filled);
    console.log(`[${model}] ${bar} ${pct}%`);
});
```

#### git 상태 (색상 포함)

브랜치와 staged/modified 파일 수를 ANSI 색상(`\033[32m` 초록, `\033[33m` 노랑, `\033[0m` 리셋)으로 표시.

```bash
#!/bin/bash
input=$(cat)

MODEL=$(echo "$input" | jq -r '.model.display_name')
DIR=$(echo "$input" | jq -r '.workspace.current_dir')

GREEN='\033[32m'; YELLOW='\033[33m'; RESET='\033[0m'

if git rev-parse --git-dir > /dev/null 2>&1; then
    BRANCH=$(git branch --show-current 2>/dev/null)
    STAGED=$(git diff --cached --numstat 2>/dev/null | wc -l | tr -d ' ')
    MODIFIED=$(git diff --numstat 2>/dev/null | wc -l | tr -d ' ')

    GIT_STATUS=""
    [ "$STAGED" -gt 0 ] && GIT_STATUS="${GREEN}+${STAGED}${RESET}"
    [ "$MODIFIED" -gt 0 ] && GIT_STATUS="${GIT_STATUS}${YELLOW}~${MODIFIED}${RESET}"

    echo -e "[$MODEL] 📁 ${DIR##*/} | 🌿 $BRANCH $GIT_STATUS"
else
    echo "[$MODEL] 📁 ${DIR##*/}"
fi
```

(Python·Node.js 버전도 동일 로직으로 원문에 제공된다 — `subprocess`/`execSync`로 git 명령을 호출해 staged/modified 줄 수를 센다.)

#### 비용·소요 시간 추적

`cost.total_cost_usd`(세션 전체 API 비용 누적)와 `cost.total_duration_ms`(세션 시작 후 경과 시간)를 통화·분초 포맷으로.

```bash
#!/bin/bash
input=$(cat)

MODEL=$(echo "$input" | jq -r '.model.display_name')
COST=$(echo "$input" | jq -r '.cost.total_cost_usd // 0')
DURATION_MS=$(echo "$input" | jq -r '.cost.total_duration_ms // 0')

COST_FMT=$(printf '$%.2f' "$COST")
DURATION_SEC=$((DURATION_MS / 1000))
MINS=$((DURATION_SEC / 60)); SECS=$((DURATION_SEC % 60))

echo "[$MODEL] 💰 $COST_FMT | ⏱️ ${MINS}m ${SECS}s"
```

#### 여러 줄 표시 (Multi-line)

각 `echo`/`print`가 별도 행이 된다. 이 예시는 임계값 기반 색상(70% 미만 초록, 70~89% 노랑, 90%+ 빨강), 진행 바, git 브랜치를 결합한다.

```bash
#!/bin/bash
input=$(cat)

MODEL=$(echo "$input" | jq -r '.model.display_name')
DIR=$(echo "$input" | jq -r '.workspace.current_dir')
COST=$(echo "$input" | jq -r '.cost.total_cost_usd // 0')
PCT=$(echo "$input" | jq -r '.context_window.used_percentage // 0' | cut -d. -f1)
DURATION_MS=$(echo "$input" | jq -r '.cost.total_duration_ms // 0')

CYAN='\033[36m'; GREEN='\033[32m'; YELLOW='\033[33m'; RED='\033[31m'; RESET='\033[0m'

# Pick bar color based on context usage
if [ "$PCT" -ge 90 ]; then BAR_COLOR="$RED"
elif [ "$PCT" -ge 70 ]; then BAR_COLOR="$YELLOW"
else BAR_COLOR="$GREEN"; fi

FILLED=$((PCT / 10)); EMPTY=$((10 - FILLED))
printf -v FILL "%${FILLED}s"; printf -v PAD "%${EMPTY}s"
BAR="${FILL// /█}${PAD// /░}"

MINS=$((DURATION_MS / 60000)); SECS=$(((DURATION_MS % 60000) / 1000))

BRANCH=""
git rev-parse --git-dir > /dev/null 2>&1 && BRANCH=" | 🌿 $(git branch --show-current 2>/dev/null)"

echo -e "${CYAN}[$MODEL]${RESET} 📁 ${DIR##*/}$BRANCH"
COST_FMT=$(printf '$%.2f' "$COST")
echo -e "${BAR_COLOR}${BAR}${RESET} ${PCT}% | ${YELLOW}${COST_FMT}${RESET} | ⏱️ ${MINS}m ${SECS}s"
```

#### 클릭 가능한 링크 (OSC 8)

git remote URL을 읽어 SSH→HTTPS로 변환(`sed`)하고 repo 이름을 OSC 8 escape로 감싼다. Cmd(macOS)/Ctrl(Win·Linux)+클릭으로 브라우저에서 연다. Bash는 `printf '%b'`가 `echo -e`보다 escape를 셸 간 더 안정적으로 해석한다.

```bash
#!/bin/bash
input=$(cat)
MODEL=$(echo "$input" | jq -r '.model.display_name')

# Convert git SSH URL to HTTPS
REMOTE=$(git remote get-url origin 2>/dev/null | sed 's/git@github.com:/https:\/\/github.com\//' | sed 's/\.git$//')

if [ -n "$REMOTE" ]; then
    REPO_NAME=$(basename "$REMOTE")
    # OSC 8 format: \e]8;;URL\a then TEXT then \e]8;;\a
    printf '%b' "[$MODEL] 🔗 \e]8;;${REMOTE}\a${REPO_NAME}\e]8;;\a\n"
else
    echo "[$MODEL]"
fi
```

#### Rate limit 사용량

`rate_limits.five_hour`(5시간 롤링)·`seven_day`(주간) 윈도우의 `used_percentage`(0~100)·`resets_at`(리셋 epoch 초)를 표시. Claude.ai Pro/Max 구독자가 첫 API 응답 후에만 존재하므로, `jq`의 `// empty`로 부재를 우아하게 처리한다.

```bash
#!/bin/bash
input=$(cat)
MODEL=$(echo "$input" | jq -r '.model.display_name')
# "// empty" produces no output when rate_limits is absent
FIVE_H=$(echo "$input" | jq -r '.rate_limits.five_hour.used_percentage // empty')
WEEK=$(echo "$input" | jq -r '.rate_limits.seven_day.used_percentage // empty')

LIMITS=""
[ -n "$FIVE_H" ] && LIMITS="5h: $(printf '%.0f' "$FIVE_H")%"
[ -n "$WEEK" ] && LIMITS="${LIMITS:+$LIMITS }7d: $(printf '%.0f' "$WEEK")%"

[ -n "$LIMITS" ] && echo "[$MODEL] | $LIMITS" || echo "[$MODEL]"
```

#### 비싼 연산 캐싱 (Cache expensive operations)

상태줄 스크립트는 활성 세션 동안 자주 실행되므로 큰 저장소에서 `git status`·`git diff`가 느릴 수 있다. git 정보를 임시 파일에 캐시하고 5초마다만 갱신한다.

> [!warning] 캐시 파일명에 `$$`/`os.getpid()`/`process.pid`를 쓰지 말 것
> 프로세스 기반 식별자는 매 호출마다 바뀌어 캐시를 무력화한다. 대신 JSON 입력의 `session_id`를 사용한다 — 세션 수명 동안 안정적이고 세션별로 고유하므로, 서로 다른 저장소의 동시 세션이 캐시를 섞어 읽지 않는다.

```bash
#!/bin/bash
input=$(cat)

MODEL=$(echo "$input" | jq -r '.model.display_name')
DIR=$(echo "$input" | jq -r '.workspace.current_dir')
SESSION_ID=$(echo "$input" | jq -r '.session_id')

CACHE_FILE="/tmp/statusline-git-cache-$SESSION_ID"
CACHE_MAX_AGE=5  # seconds

cache_is_stale() {
    [ ! -f "$CACHE_FILE" ] || \
    # stat -f %m is macOS, stat -c %Y is Linux
    [ $(($(date +%s) - $(stat -f %m "$CACHE_FILE" 2>/dev/null || stat -c %Y "$CACHE_FILE" 2>/dev/null || echo 0))) -gt $CACHE_MAX_AGE ]
}

if cache_is_stale; then
    if git rev-parse --git-dir > /dev/null 2>&1; then
        BRANCH=$(git branch --show-current 2>/dev/null)
        STAGED=$(git diff --cached --numstat 2>/dev/null | wc -l | tr -d ' ')
        MODIFIED=$(git diff --numstat 2>/dev/null | wc -l | tr -d ' ')
        echo "$BRANCH|$STAGED|$MODIFIED" > "$CACHE_FILE"
    else
        echo "||" > "$CACHE_FILE"
    fi
fi

IFS='|' read -r BRANCH STAGED MODIFIED < "$CACHE_FILE"

if [ -n "$BRANCH" ]; then
    echo "[$MODEL] 📁 ${DIR##*/} | 🌿 $BRANCH +$STAGED ~$MODIFIED"
else
    echo "[$MODEL] 📁 ${DIR##*/}"
fi
```

### Windows 설정

Windows에서 Claude Code는 Git Bash가 설치돼 있으면 Git Bash로, 없으면 PowerShell로 상태줄 명령을 실행한다.

> [!warning] Windows 경로는 forward slash로
> Git Bash는 따옴표 없는 백슬래시를 escape 문자로 취급한다. `C:\Users\username\script.mjs` 같은 경로는 구분자가 제거된 채 스크립트 러너에 도달해 **에러 표시 없이 실패**한다. `command` 문자열의 파일 경로는 forward slash(`/`)로 쓴다. `~` 단축도 동작하며 Windows 홈 디렉터리로 확장된다.

PowerShell 스크립트를 상태줄로 실행하려면 `powershell`로 호출한다(Git Bash 경유든 PowerShell 경유든 동작).

```json
{
  "statusLine": {
    "type": "command",
    "command": "powershell -NoProfile -File C:/Users/username/.claude/statusline.ps1"
  }
}
```

```powershell
$input_json = $input | Out-String | ConvertFrom-Json
$cwd = $input_json.cwd
$model = $input_json.model.display_name
$used = $input_json.context_window.used_percentage
$dirname = Split-Path $cwd -Leaf

if ($used) {
    Write-Host "$dirname [$model] ctx: $used%"
} else {
    Write-Host "$dirname [$model]"
}
```

Git Bash가 설치돼 있으면 Bash 스크립트를 직접 실행할 수도 있다(`"command": "~/.claude/statusline.sh"`). 이 경우 `grep -o`로 `cwd`·`display_name`을 추출하는 경량 파싱이 예시로 제공된다.

### 서브에이전트 상태줄 (Subagent status lines)

`subagentStatusLine` 설정은 프롬프트 아래 agent 패널에 표시되는 각 [[08 서브에이전트와 에이전트 팀]] 행의 본문을 커스텀 렌더링한다. 기본 `name · description · token count` 행을 사용자 포맷으로 교체할 때 쓴다.

```json
{
  "subagentStatusLine": {
    "type": "command",
    "command": "~/.claude/subagent-statusline.sh"
  }
}
```

- 명령은 refresh tick마다 **한 번** 실행되며, 보이는 모든 서브에이전트 행이 stdin에 단일 JSON 객체로 전달된다.
- 입력에는 [[09 훅]]의 base hook 공통 필드 + `columns`(사용 가능 행 너비) + `tasks` 배열이 포함된다. 각 task는 `id`, `name`, `type`, `status`, `description`, `label`, `startTime`, `tokenCount`, `tokenSamples`, `cwd`를 가진다.
- 덮어쓸 행마다 stdout에 한 줄씩 JSON을 쓴다: `{"id": "<task id>", "content": "<row body>"}`. `content`는 ANSI 색상·OSC 8 하이퍼링크 포함 그대로 렌더링된다. task의 `id`를 생략하면 해당 행은 기본 렌더링을 유지하고, 빈 `content`를 내보내면 숨겨진다.
- `statusLine`과 동일한 신뢰(trust)·`disableAllHooks` 게이트가 적용된다. [[11 플러그인]]은 자체 `settings.json`에 기본 `subagentStatusLine`을 배포할 수 있다.

### 팁 (Tips)

- **mock 입력으로 테스트**:
  ```bash
  echo '{"model":{"display_name":"Opus"},"workspace":{"current_dir":"/home/user/project"},"context_window":{"used_percentage":25},"session_id":"test-session-abc"}' | ./statusline.sh
  ```
- **출력은 짧게**: 상태바 너비가 제한돼 긴 출력은 잘리거나 어색하게 줄바꿈된다.
- **느린 연산은 캐시**: 자주 실행되므로 `git status` 등이 랙을 유발할 수 있다(위 캐싱 예시).
- 커뮤니티 프로젝트 [ccstatusline](https://github.com/sirmalloc/ccstatusline), [starship-claude](https://github.com/martinemde/starship-claude)가 테마·추가 기능이 있는 사전 구성을 제공한다.

### 트러블슈팅

[[25 트러블슈팅]]에 일반 진단이 있고, 상태줄 특화 항목은 다음과 같다.

| 증상 | 점검/해결 |
| :--- | :--- |
| **상태줄이 안 나옴** | 스크립트 실행 권한(`chmod +x`) 확인 · 출력이 stderr가 아닌 stdout인지 확인 · 스크립트를 수동 실행해 출력 검증 · Windows Git Bash면 경로 백슬래시가 escape로 소비되니 forward slash 사용 · `disableAllHooks: true`면 상태줄도 비활성화되므로 제거하거나 `false`로 · `claude --debug`로 첫 호출의 exit code·stderr 로깅 · Claude에게 settings를 읽고 `statusLine` 명령을 직접 실행시켜 에러 확인 |
| **`--`/빈 값 표시** | 첫 API 응답 전엔 필드가 `null`일 수 있음 · jq `// 0` 등 fallback 처리 · 여러 메시지 후에도 비면 Claude Code 재시작 |
| **컨텍스트 퍼센트가 이상** | 가장 단순·정확한 상태엔 `used_percentage` 사용 · 계산 시점 차이로 `/context` 출력과 다를 수 있음 |
| **OSC 8 링크가 클릭 안 됨** | 터미널의 OSC 8 지원 확인(iTerm2·Kitty·WezTerm 지원, Terminal.app 미지원) · 자동 감지 실패 시(Windows Terminal 등) `FORCE_HYPERLINK=1 claude`로 강제(PowerShell: `$env:FORCE_HYPERLINK = "1"; claude`) · SSH·tmux는 OSC 시퀀스를 strip할 수 있음 · `\e]8;;` 같은 리터럴 텍스트가 보이면 `echo -e` 대신 `printf '%b'` 사용 |
| **escape 시퀀스 표시 깨짐** | 복잡한 ANSI·OSC 8이 다른 UI 업데이트와 겹치면 깨질 수 있음 · 깨지면 plain text로 단순화 · 멀티라인+escape가 단일라인 plain보다 렌더링 문제에 취약 |
| **워크스페이스 trust 필요** | `statusLine`은 쉘 명령을 실행하므로 hook과 동일한 trust 수락이 필요. 현재 디렉터리의 trust 대화를 수락해야만 실행됨. 미수락 시 출력 대신 `statusline skipped · restart to fix` 알림. 재시작 후 trust 수락 ([[18 보안과 샌드박스]] 참고) |
| **스크립트 에러/행** | non-zero exit나 무출력이면 상태줄이 빈칸이 됨 · 느린 스크립트는 완료까지 업데이트를 막음(빠르게 유지) · 실행 중 새 업데이트 트리거 시 진행 중 스크립트는 취소 · mock 입력으로 독립 테스트 권장 |
| **알림이 같은 행 공유** | MCP 서버 에러·자동 업데이트 등 시스템 알림은 상태줄과 같은 행 오른쪽에 표시 · verbose 모드는 토큰 카운터를 이 영역에 추가 · 좁은 터미널에선 알림이 상태줄 출력을 자를 수 있음 |
